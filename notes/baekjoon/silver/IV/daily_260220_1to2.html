    문제 1120 — 문자

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    문자열 (백준 1120)는
    길이가 같은 두 문자열 A, B의 차이를 최소화하기 위해, 짧은 문자열 X의 앞뒤에 임의의 문자를 붙여
    X와 Y의 대응되는 위치에서 다른 문자의 개수(차이)를 최소화하는 문제다.

    • X의 길이는 항상 Y의 길이 이하이며, X의 앞뒤에 문자를 붙여 Y와 길이를 맞춘다.
    • 앞뒤에 붙이는 문자는 자유롭게 선택할 수 있으므로, Y의 대응 구간과 항상 같게 만들 수 있다. 즉 앞뒤에 붙인 문자는 차이에 영향을 주지 않는다.
    • 따라서 X를 Y의 어느 위치에 정렬(슬라이딩)하느냐에 따라 X의 각 문자와 Y의 대응 문자를 비교하면 된다.

    전략 개요

    • 핵심 알고리즘: 슬라이딩 윈도우 + 브루트포스 비교
    • 선택 이유: X의 길이와 Y의 길이가 최대 50으로 매우 작다. X를 Y 위에서 슬라이딩하며 가능한 모든 정렬 위치에 대해 일일이 비교해도 최대 50 × 50 = 2,500번의 연산으로 충분히 해결된다.
    • 세부 전략:
      • X를 Y 위에서 왼쪽부터 오른쪽으로 이동시킬 수 있는 횟수는 Y.length() - X.length()번이다.
      • 각 정렬 위치 i에 대해, X의 각 문자와 Y의 idx + i 위치 문자를 비교해 다른 문자의 수 cnt를 센다.
      • 모든 위치에서의 cnt 중 최솟값을 출력한다.

    코드가 하는 일

    • 1) 입력 및 이동 가능 횟수 계산 
      String X = st.nextToken();
      String Y = st.nextToken();
      
      int canMove = Y.length() - X.length();
      int min = Integer.MAX_VALUE;
      • X와 Y를 입력받는다. 문제 조건상 X의 길이 ≤ Y의 길이가 보장된다.
      • canMove는 X를 Y 위에서 슬라이딩할 수 있는 최대 이동 횟수다. 예를 들어 X의 길이가 3, Y의 길이가 5이면 시작 위치는 0, 1, 2로 총 3가지다.
      • min은 최솟값을 저장할 변수로, 비교 전 충분히 큰 값으로 초기화한다.
    • 2) 슬라이딩하며 위치별 차이 계산 
      for (int i = 0; i <= canMove; i++) {
          int cnt = 0;
      
          for (int idx = 0; idx < X.length(); idx++) {
              if (X.charAt(idx) != Y.charAt(idx + i)) {
                  cnt++;
              }
          }
      
          if (cnt < min) {
              min = cnt;
          }
      }
      • 외부 반복문의 i는 X의 시작 위치(오프셋)를 나타낸다. i = 0이면 X가 Y의 맨 왼쪽에 정렬된 상태다.
      • 내부 반복문에서 X의 각 문자 X.charAt(idx)와 Y의 대응 문자 Y.charAt(idx + i)를 비교해 다를 때마다 cnt를 증가시킨다.
      • 해당 위치에서의 cnt가 현재 최솟값 min보다 작으면 갱신한다.
      • 예시: X = "abc", Y = "xabcd"일 때, i=0이면 "abc"와 "xab"를 비교(cnt=2), i=1이면 "abc"와 "abc"를 비교(cnt=0), i=2이면 "abc"와 "bcd"를 비교(cnt=3). 최솟값은 0.
    • 3) 결과 출력 
      System.out.println(min);
      • 모든 슬라이딩 위치를 탐색한 뒤, 가장 작은 차이 값 min을 출력한다.

    시간 · 공간 복잡도

    • 시간 복잡도
      • O(|Y| × |X|): 외부 반복문은 최대 |Y| - |X|번, 내부 반복문은 |X|번 실행된다. |X|, |Y|가 최대 50이므로 최악의 경우에도 50 × 50 = 2,500번의 연산으로 매우 효율적이다.
    • 공간 복잡도
      • O(|X| + |Y|): 두 문자열 X, Y를 저장하는 공간 외에 추가적인 자료구조를 사용하지 않으므로 입력 문자열 크기에 비례하는 공간만 사용한다.

    문제 4134 — 다음 소수

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    다음 소수 (백준 4134)는
    주어진 정수 n에 대해 n 이상의 소수 중 가장 작은 소수를 찾아 출력하는 문제다.
    n은 0 이상 1,000,000,000 이하의 정수이며, 테스트 케이스는 최대 1,000개다.

    • n이 최대 10억이므로 소수 판별 시 오버플로우를 방지하기 위해 long 타입을 사용해야 한다.
    • n이 0이나 1인 경우 2가 n 이상의 가장 작은 소수이므로 별도로 처리한다.
    • 소수 판별은 √n까지만 나눠보는 방식을 사용한다. n이 최대 10억일 때 √n ≈ 31,623이므로 각 수에 대한 판별이 충분히 빠르다.

    전략 개요

    • 핵심 알고리즘: 소수 판별 (Trial Division) + 선형 탐색
    • 선택 이유: n 이상의 수를 하나씩 증가시키며 소수인지 판별한다. 소수 간격이 평균적으로 크지 않고, 각 수의 소수 판별이 O(√n)으로 빠르기 때문에 1,000개의 테스트 케이스도 충분히 처리할 수 있다.
    • 세부 전략:
      • n이 0 또는 1이면 즉시 2를 반환한다.
      • n부터 시작해 2 이상 √temp 이하의 수로 나누어 떨어지지 않으면 소수로 판별하고 반환한다.
      • 소수가 아니면 temp를 1 증가시켜 다음 수에 대해 동일하게 반복한다.

    코드가 하는 일

    • 1) 테스트 케이스 반복 처리 
      int T = Integer.parseInt(br.readLine());
      
      for (int tc = 0; tc < T; tc++) {
          long n = Long.parseLong(br.readLine());
          long prime = isPrime(n);
          sb.append(prime).append("\n");
      }
      • 테스트 케이스 수 T를 읽은 뒤, 각 n에 대해 isPrime(n)을 호출해 n 이상의 가장 작은 소수를 구한다.
      • n이 최대 10억이므로 long 타입으로 입력받는다.
      • 결과를 StringBuilder에 누적해 마지막에 한 번에 출력함으로써 I/O 비용을 줄인다.
    • 2) 0, 1 예외 처리 
      if (num == 0 || num == 1) {
          return 2;
      }
      • 0과 1은 소수가 아니므로, 이 경우 n 이상의 가장 작은 소수인 2를 즉시 반환한다.
    • 3) 소수 판별 및 탐색 
      long temp = num;
      
      while (true) {
          boolean flag = true;
      
          for (int i = 2; (long) i * i <= temp; i++) {
              if (temp % i == 0) {
                  flag = false;
                  break;
              }
          }
      
          if (flag) {
              return temp;
          }
      
          temp++;
      }
      • temp를 n으로 초기화한 뒤, 소수를 찾을 때까지 1씩 증가시키며 판별한다.
      • 2부터 √temp까지의 수로 나눠보아 하나라도 나누어 떨어지면 소수가 아니므로 flag = false로 설정하고 즉시 내부 반복을 종료한다.
      • (long) i * i <= temp와 같이 캐스팅하는 이유는 i * i 연산 시 int 오버플로우를 방지하기 위함이다. temp가 최대 10억에 근접하면 √temp ≈ 31,623이므로 i * i가 최대 약 10억에 달해 int 범위를 초과할 수 있다.
      • flag가 true이면 소수이므로 temp를 반환한다.

    시간 · 공간 복잡도

    • 시간 복잡도
      • O(T × k × √n): T는 테스트 케이스 수, k는 n 이상에서 소수까지의 거리(평균적으로 ln(n)에 비례), √n은 각 수에 대한 소수 판별 비용이다. n이 최대 10억일 때 √n ≈ 31,623이고, T ≤ 1,000이므로 충분히 빠르게 동작한다.
    • 공간 복잡도
      • O(T): 결과를 누적하는 StringBuilder가 테스트 케이스 수에 비례하는 공간을 사용하며, 그 외 변수들은 O(1)의 상수 공간을 사용한다.
