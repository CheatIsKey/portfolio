    문제 25192 — 인사성 밝은 곰곰이

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    인사성 밝은 곰곰이 (백준 25192)는
    채팅방 로그에서 곰곰이가 ENTER 이후 인사를 건넨 적 있는 유저들에게
    총 몇 번 인사를 받았는지 구하는 문제다.
    ENTER 이후 같은 유저가 여러 번 채팅해도 인사는 1번으로만 카운트한다.

    • ENTER는 새로운 유저가 채팅방에 입장하는 이벤트로, ENTER가 등장하면 이전까지 인사한 유저 목록을 초기화한다.
    • 같은 ENTER 구간 내에서 동일한 유저가 여러 번 채팅을 보내도 인사는 1회로 취급한다.
    • 로그의 첫 번째 줄은 반드시 ENTER임이 보장된다.

    전략 개요

    • 핵심 알고리즘: HashSet을 활용한 중복 제거
    • 선택 이유: ENTER와 ENTER 사이 구간에서 인사한 유저를 중복 없이 세야 하므로, 삽입과 조회가 O(1)인 HashSet이 가장 적합하다. ENTER가 등장할 때마다 set을 비워 다음 구간을 새로 시작하면 된다.
    • 세부 전략:
      • ENTER가 등장하면 현재 set의 크기(해당 구간에서 인사한 고유 유저 수)를 totalCnt에 더하고 set을 초기화한다.
      • 일반 채팅 메시지는 유저 이름을 set에 추가한다. 이미 존재하면 HashSet이 자동으로 중복을 무시한다.
      • 모든 로그를 처리한 뒤, 마지막 ENTER 이후 구간의 set 크기를 별도로 totalCnt에 더한다.

    코드가 하는 일

    • 1) 입력 및 초기화 
      int N = Integer.parseInt(br.readLine());
      HashSet set = new HashSet<>();
      int totalCnt = 0;
      • 채팅 로그의 수 N을 입력받는다.
      • set은 현재 ENTER 구간에서 인사한 유저를 중복 없이 저장하는 역할을 한다.
      • totalCnt는 전체 인사 횟수를 누적하는 변수다.
    • 2) ENTER 처리 — 구간 집계 및 초기화 
      if (chat.equals("ENTER")) {
          totalCnt += set.size();
          set.clear();
          continue;
      }
      • ENTER가 등장하면 지금까지 set에 모인 고유 유저 수를 totalCnt에 더한다.
      • set.clear()로 set을 비워 다음 구간을 새로 시작할 준비를 한다.
      • 첫 번째 줄이 반드시 ENTER이므로, 첫 ENTER에서는 빈 set의 크기인 0이 더해져 아무런 영향이 없다.
    • 3) 일반 채팅 처리 — 유저 등록 
      set.add(chat);
      • ENTER가 아닌 일반 채팅은 유저 이름을 set에 추가한다.
      • HashSet의 특성상 동일한 유저가 여러 번 채팅해도 중복 없이 1개만 저장된다.
    • 4) 마지막 구간 처리 및 출력 
      totalCnt += set.size();
      System.out.println(totalCnt);
      • 반복문은 ENTER가 등장할 때만 집계하므로, 마지막 ENTER 이후의 구간은 반복문 안에서 처리되지 않는다. 반복문 종료 후 남아 있는 set의 크기를 한 번 더 더해준다.
      • 최종 누적된 totalCnt를 출력한다.

    시간 · 공간 복잡도

    • 시간 복잡도
      • O(N): N개의 로그를 한 번씩 순회하며, 각 로그에 대해 HashSet의 삽입·조회가 평균 O(1)이므로 전체 O(N)이다.
    • 공간 복잡도
      • O(U): U는 하나의 ENTER 구간 내 고유 유저 수의 최댓값이다. set은 한 구간의 고유 유저만 저장하며 ENTER마다 초기화되므로, 전체 유저 수가 아닌 한 구간의 최대 고유 유저 수에 비례하는 공간만 사용한다.

    문제 3036 — 링

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    링 (백준 3036)은
    첫 번째 링이 한 바퀴 돌 때, 나머지 각 링이 몇 바퀴 도는지를 기약분수로 출력하는 문제다.
    링의 둘레는 반지름에 비례하므로, 두 링의 반지름 비율로 회전 수를 구할 수 있다.

    • 첫 번째 링(기준 링)이 한 바퀴 돌면, 맞닿은 링은 첫 번째 링의 반지름 / 해당 링의 반지름만큼 회전한다.
    • 결과는 기약분수로 출력해야 하므로, 두 반지름의 최대공약수(GCD)로 나누어 약분한다.
    • 링의 수 N은 최대 100, 반지름은 최대 1,000이므로 유클리드 호제법으로 충분히 빠르게 처리된다.

    전략 개요

    • 핵심 알고리즘: 유클리드 호제법(GCD)을 이용한 기약분수 변환
    • 선택 이유: 첫 번째 링과 각 링의 반지름 비율이 곧 회전 수이므로, 이 비율을 기약분수로 나타내기 위해 두 값의 GCD를 구해 각각 나누면 된다. 유클리드 호제법은 O(log n)으로 매우 빠르다.
    • 세부 전략:
      • 기준 링의 반지름 mainRing을 먼저 읽고, 나머지 링을 순서대로 읽는다.
      • 각 링에 대해 mainRing과 해당 링의 반지름의 GCD를 구한다.
      • 분자(mainRing / gcd)와 분모(ring / gcd)를 /로 이어 출력한다.

    코드가 하는 일

    • 1) 입력 및 기준 링 설정 
      int N = Integer.parseInt(br.readLine());
      StringTokenizer st = new StringTokenizer(br.readLine());
      int mainRing = Integer.parseInt(st.nextToken());
      • 링의 수 N을 읽은 뒤, 한 줄에 공백으로 주어지는 반지름들을 StringTokenizer로 파싱한다.
      • 첫 번째 토큰이 기준 링의 반지름 mainRing이다. 이 값이 회전 비율 계산의 분자 역할을 한다.
    • 2) 각 링의 회전 수 계산 및 기약분수 변환 
      for (int i = 0; i < N - 1; i++) {
          int ring = Integer.parseInt(st.nextToken());
      
          int value = gcd(mainRing, ring);
      
          sb.append((mainRing / value)).append('/').append((ring / value)).append('\n');
      }
      • 나머지 N-1개의 링을 순서대로 읽는다.
      • gcd(mainRing, ring)으로 두 반지름의 최대공약수를 구한다.
      • 분자 mainRing / value와 분모 ring / value를 각각 GCD로 나눠 기약분수로 만든 뒤, /로 이어 StringBuilder에 누적한다.
      • 예시: mainRing = 6, ring = 4이면 GCD = 2, 결과는 3/2. 즉 기준 링이 한 바퀴 돌 때 해당 링은 1.5바퀴 회전한다.
    • 3) 유클리드 호제법으로 GCD 계산 
      private static int gcd(int num1, int num2) {
          if (num2 == 0) {
              return num1;
          }
          return gcd(num2, num1 % num2);
      }
      • 재귀적 유클리드 호제법으로 두 수의 GCD를 구한다.
      • num2가 0이 되면 num1이 GCD이므로 반환한다. 그렇지 않으면 gcd(num2, num1 % num2)를 재귀 호출한다.

    시간 · 공간 복잡도

    • 시간 복잡도
      • O(N log R): N은 링의 수, R은 반지름의 최댓값이다. N-1개의 링 각각에 대해 유클리드 호제법 O(log R)을 수행하므로 전체 O(N log R)이다. N ≤ 100, R ≤ 1,000이므로 매우 빠르게 동작한다.
    • 공간 복잡도
      • O(N): 결과를 누적하는 StringBuilder가 링의 수에 비례하는 공간을 사용하며, 재귀 호출 스택은 O(log R)로 매우 작다.
