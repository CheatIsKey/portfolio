    문제 17521 — Byte Coin

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Byte Coin (백준 17521)는
    n일간의 코인 가격이 주어질 때, 초기 현금 W로 코인을 사고팔아 최종 보유 현금을 최대화하는 문제다.
    다음 날 가격이 오를 때 전량 매수, 다음 날 가격이 내릴 때 전량 매도하는 그리디 전략으로 최적 수익을 얻을 수 있다.

    • 내일 가격이 오늘보다 높으면 오늘 살 수 있는 만큼 전량 매수한다.
    • 내일 가격이 오늘보다 낮고 코인을 보유 중이라면 오늘 전량 매도한다.
    • 가격과 현금이 int 범위를 초과할 수 있으므로 long 타입으로 처리해야 한다.

    전략 개요

    • 핵심 알고리즘: 그리디 (내일 가격 기준 전량 매수·매도)
    • 선택 이유: 미래 가격을 모두 알고 있는 상황에서, 오르기 직전에 사고 내리기 직전에 파는 전략이 항상 최적임이 보장된다. 연속 상승 구간에서는 첫날 매수 후 마지막 날 매도하는 것과 매일 사고파는 것이 동일한 수익을 내므로, 단순히 다음 날 가격과 비교하는 O(n) 순회로 해결된다.
    • 세부 전략:
      • 오늘 가격(cur)과 내일 가격(next)을 비교한다.
      • cur < next이면 현금으로 살 수 있는 최대 코인 수를 매수하고 잔돈을 남긴다.
      • cur > next이고 보유 코인이 있으면 전량 매도하여 현금화한다.
      • 마지막 날 이후에도 코인을 보유 중이면 마지막 가격으로 전량 매도한다.

    코드가 하는 일

    • 1) 입력 및 초기화 
      int n = Integer.parseInt(st.nextToken());
      long W = Long.parseLong(st.nextToken());
      
      long coin = 0;
      long cur = Long.parseLong(br.readLine());
      • 날 수 n과 초기 현금 W를 입력받는다. 보유 코인 수 coin을 0으로 초기화하고, 첫째 날 가격 cur을 읽어 둔다. 현금과 코인 수가 int 범위를 초과할 수 있으므로 모두 long으로 선언한다.
    • 2) 일별 매수·매도 결정 
      for (int i = 0; i < n - 1; i++) {
          long next = Long.parseLong(br.readLine());
      
          if (cur < next) {
              coin += W / cur;
              W %= cur;
          } else if (coin > 0 && cur > next) {
              W += coin * cur;
              coin = 0;
          }
      
          cur = next;
      }
      • cur < next: 내일 가격이 더 높으므로 오늘 전량 매수한다. W / cur개의 코인을 사고 W % cur의 잔돈을 남긴다.
      • cur > next && coin > 0: 내일 가격이 더 낮고 코인을 보유 중이므로 오늘 전량 매도한다. coin * cur을 현금에 더하고 코인을 0으로 초기화한다.
      • 순회가 끝나면 cur을 next로 갱신하여 다음 날 비교를 준비한다.
    • 3) 마지막 날 잔여 코인 처리 및 출력 
      if (coin > 0) {
          W += coin * cur;
      }
      
      System.out.println(W);
      • 루프 종료 후 코인을 아직 보유 중이라면 마지막 날 가격 cur로 전량 매도하여 현금화한다. 최종 현금 W를 출력한다.

    시간 · 공간 복잡도

    • 시간 복잡도
      • O(n): n일간의 가격을 한 번 순회하며 각 날에 O(1) 연산만 수행하므로 전체 O(n)이다.
    • 공간 복잡도
      • O(1): 현금 W, 코인 수 coin, 현재 가격 cur, 다음 가격 next 등 고정된 수의 변수만 사용하므로 상수 공간만 필요하다.

    문제 9322 — 철벽 보안 알고리즘

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    철벽 보안 알고리즘 (백준 9322)는
    세 줄의 문자열 배열이 주어질 때, 첫 번째 배열을 기준 순서로 삼아 두 번째 배열의 순서를 첫 번째에 맞게 재배열한 뒤, 세 번째 배열을 그 순서로 출력하는 문제다.
    문자열과 인덱스의 대응 관계를 HashMap으로 관리하여 효율적으로 순서를 복원한다.

    • 첫 번째 줄의 각 문자열이 원래 위치(인덱스)를 정의하는 기준이 된다.
    • 두 번째 줄은 첫 번째 줄과 동일한 문자열이 섞인 순서로 주어지며, 이를 통해 각 위치가 원래 어느 인덱스에 대응하는지 파악할 수 있다.
    • 세 번째 줄의 문자열을 두 번째 줄의 순서 정보를 역추적하여 첫 번째 줄 기준의 올바른 위치에 배치한 뒤 출력한다.

    전략 개요

    • 핵심 알고리즘: HashMap을 이용한 인덱스 매핑
    • 선택 이유: 첫 번째 줄의 문자열과 그 인덱스를 HashMap에 저장해두면, 두 번째 줄의 각 문자열이 원래 몇 번째 위치에 있었는지를 O(1)에 조회할 수 있다. 이 대응 관계를 이용해 세 번째 줄의 문자열을 정답 배열의 올바른 위치에 삽입하면 별도의 정렬 없이 O(n)에 해결된다.
    • 세부 전략:
      • 첫 번째 줄의 문자열 → 인덱스를 HashMap map에 저장한다.
      • 두 번째 줄의 i번째 문자열을 map에서 조회해 원래 인덱스를 number[i]에 저장한다.
      • 세 번째 줄의 i번째 문자열을 answer[number[i]] 위치에 배치한다.
      • answer 배열을 순서대로 출력한다.

    코드가 하는 일

    • 1) 첫 번째 줄: 문자열 → 인덱스 매핑 
      Map<String, Integer> map = new HashMap<>();
      StringTokenizer st = new StringTokenizer(br.readLine());
      
      for (int i = 0; i < n; i++) {
          String s = st.nextToken();
          map.put(s, i);
      }
      • 첫 번째 줄의 각 문자열을 키로, 그 인덱스를 값으로 HashMap에 저장한다. 이후 문자열이 원래 몇 번째 위치였는지를 O(1)에 조회할 수 있다.
    • 2) 두 번째 줄: 섞인 순서에서 원래 인덱스 복원 
      st = new StringTokenizer(br.readLine());
      for (int i = 0; i < n; i++) {
          String s = st.nextToken();
          number[i] = map.get(s);
      }
      • 두 번째 줄의 i번째 문자열을 map에서 조회하여 해당 문자열의 원래 인덱스를 number[i]에 저장한다. 즉, 현재 i번째 위치가 원래 number[i]번째였음을 기록한다.
    • 3) 세 번째 줄: 정답 배열에 올바른 위치로 배치 
      st = new StringTokenizer(br.readLine());
      for (int i = 0; i < n; i++) {
          String s = st.nextToken();
          int idx = number[i];
          answer[idx] = s;
      }
      • 세 번째 줄의 i번째 문자열을 answer[number[i]] 위치에 삽입한다. 두 번째 줄의 i번째 항목이 원래 number[i]번째였으므로, 세 번째 줄의 같은 위치 문자열도 동일한 인덱스로 이동시키면 첫 번째 줄 기준의 올바른 순서가 복원된다.
    • 4) 결과 누적 및 출력 
      for (String s : answer) {
          sb.append(s).append(" ");
      }
      
      sb.append('\n');
      
      System.out.println(sb);
      • 복원된 answer 배열을 순서대로 StringBuilder에 누적하고, 모든 테스트케이스가 끝난 뒤 한 번에 출력하여 I/O 비용을 줄인다.

    시간 · 공간 복잡도

    • 시간 복잡도
      • O(T · n): T개의 테스트케이스마다 세 줄을 각각 n번 순회하며 HashMap 조회와 배열 접근은 O(1)이므로 전체 O(T · n)이다.
    • 공간 복잡도
      • O(n): HashMap, number 배열, answer 배열이 각각 O(n)의 공간을 사용하며, 테스트케이스마다 초기화되므로 동시에 사용되는 최대 공간은 O(n)이다.
