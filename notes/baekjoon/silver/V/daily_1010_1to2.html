    문제 2167 — 2차원 배열의 합

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 — 이 문제는 여러 번 주어지는 부분 영역 (x₁, y₁) ~ (x₂, y₂)의 합을 빠르게 구해야 한다. 단순히 매번 합을 구하면 최대 300×300×10,000번 덧셈이 필요하므로 O(NMK)으로 매우 비효율적이다. 이를 해결하기 위해 2차원 누적합(Prefix Sum)을 사용한다.

    2차원 누적합 배열 ps[i][j]은 “왼쪽 위 (1,1)부터 현재 좌표 (i,j)까지의 모든 원소 합”을 저장한다. 이 배열을 한 번 구해두면, 임의의 사각형 합은 다음 한 줄 공식으로 바로 계산된다:

    sum = ps[x₂][y₂] - ps[x₁-1][y₂] - ps[x₂][y₁-1] + ps[x₁-1][y₁-1];

    이는 ‘전체 영역 합 - 위쪽 영역 - 왼쪽 영역 + 겹친 영역 복구’의 원리다. 이때 0행과 0열은 항상 0으로 초기화하여 인덱스 범위를 벗어나지 않게 한다.

    입력 파싱 — 첫 줄에 행과 열 크기 N, M이 주어진다 (1 ≤ N, M ≤ 300). 다음 N줄에는 M개의 정수가 주어지고, 그 후 질의 개수 K와 K개의 구간 좌표 x₁, y₁, x₂, y₂가 입력된다. 빠른 입출력을 위해 BufferedReader와 StringTokenizer를 사용한다.

    알고리즘

    • 1️⃣ 입력 배열 ints를 1-based 인덱스로 저장한다.
    • 2️⃣ 누적합 배열 ints1을 계산한다: 
      ints1[i][j] = ints1[i-1][j] + ints1[i][j-1] - ints1[i-1][j-1] + ints[i][j];
      위 + 왼쪽 - 겹친 부분 + 현재값 으로 (1,1)~(i,j)까지 합을 구한다.
    • 3️⃣ 각 질의에 대해 누적합 공식을 이용해 즉시 결과를 계산한다. 
      sum = ints1[x2][y2] - ints1[x1-1][y2] - ints1[x2][y1-1] + ints1[x1-1][y1-1];
    • 4️⃣ 출력은 StringBuilder에 누적 후 한 번에 출력한다.

    예시

    입력:
      3 3
      1 2 3
      4 5 6
      7 8 9
      1
      2 2 3 3
      
      출력:
      28

    해설: 2행 2열~3행 3열의 합은 5+6+8+9=28. 공식으로 계산하면 ps[3][3]-ps[1][3]-ps[3][1]+ps[1][1] = 45-6-12+1=28이다.

    시간·공간 복잡도 — 누적합 계산이 O(N×M), 각 질의가 O(1)이므로 전체 O(N×M + K). 공간 복잡도도 O(N×M).

    정리

    • 2차원 누적합의 정의: ps[i][j]=(1,1)~(i,j)까지의 합.
    • 누적합 공식: ps[i][j]=ps[i-1][j]+ps[i][j-1]-ps[i-1][j-1]+a[i][j].
    • 부분 구간 합 공식: ps[x₂][y₂]-ps[x₁-1][y₂]-ps[x₂][y₁-1]+ps[x₁-1][y₁-1].
    • 모든 질의를 상수 시간(O(1))에 처리 가능.
    • 전형적인 2차원 Prefix Sum 응용 문제.

    문제 2238 — 경매

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 — 경매에서 여러 사람이 가격을 부르는 상황을 시뮬레이션한다. 모든 입찰 내역 중에서 “가장 적은 인원이 부른 금액”을 찾고, 그런 금액이 여러 개면 “가장 낮은 금액”을 선택한다. 최종적으로 그 금액을 가장 먼저 제시한 사람이 낙찰된다.

    입력 파싱 — 첫 줄에는 두 정수 U, N이 주어진다.

    • U — 입찰 가능한 금액의 상한 (1 ≤ U ≤ 10,000)
    • N — 입찰 횟수 (1 ≤ N ≤ 100,000)
    이후 N개의 줄에는 각 입찰자의 이름과 금액이 주어진다. 한 사람이 여러 번 등장할 수 있으며, 모든 줄은 독립된 입찰로 처리된다.

    알고리즘

    • 가격(1~U)을 인덱스로 하여 통계를 계산한다.
    • freq[price]: 해당 가격을 제시한 횟수
    • firstName[price]: 해당 가격을 가장 먼저 부른 사람
    • firstOrder[price]: 그 가격이 처음 등장한 시점(줄 번호)
    • 입력을 순서대로 읽으며:
      • freq[price]++
      • 아직 등장하지 않은 가격이면 firstName과 firstOrder 기록
    • 모든 입력 후:
      • freq[p] > 0인 가격 중 최솟값 minFreq를 찾는다.
      • freq[p] == minFreq인 가격 중 가장 작은 price를 선택한다.
      • 그 가격을 가장 먼저 부른 firstName[price]을 함께 출력한다.

    예시

          입력:
      10 5
      amy 3
      bob 8
      cindy 3
      david 5
      eric 8
      
      과정:
      가격별 빈도 → 3(2회), 5(1회), 8(2회)
      → 가장 낮은 빈도는 1, 해당 금액은 5
      → 5원을 가장 먼저 부른 사람은 david
      
      출력:
      david 5

    시간·공간 복잡도 — 모든 입력을 한 번씩만 처리하므로 O(N + U), 추가 메모리 또한 O(U)로 충분하다.

    정리

    • “가장 낮은 유일한 금액”이 아니라 “가장 적은 인원이 부른 금액”이 규칙의 핵심.
    • 동일 가격이 여러 번 등장하더라도, 가장 먼저 제시한 사람이 낙찰.
    • 가격 상한 U은 인덱스 크기 한도로만 사용된다.
    • 전체 복잡도는 O(N) 수준으로 충분히 효율적이다.
