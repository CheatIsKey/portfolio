    문제 3863 — 행복한 전화 통화

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 —
    각 통화는 [cs, cs + cd) 구간, 각 문의는 [qs, qs + qd) 구간으로 본다. 두 구간이 한 번이라도 겹치면(교집합이 비어있지 않으면) 해당 통화가 카운트된다.
    구간 겹침 판정은 반열린구간 기준으로 cs < qe 그리고 qs < ce (여기서 qe = qs + qd, ce = cs + cd) 이면 참이다. 끝점만 맞닿는 경우(ce == qs 또는 qe == cs)는 겹치지 않는 것으로 처리된다.

    제약 정리 —
    입력: 여러 테스트 케이스. 각 케이스마다 N M (통화 수, 문의 수). N = 0, M = 0이면 종료.
    다음 N줄: a b cs cd (발신 번호, 수신 번호, 통화 시작 시각, 통화 길이). 번호 두 값은 계산에 사용하지 않음.
    다음 M줄: qs qd (문의 시작 시각, 길이).
    출력: 각 문의마다 해당 구간과 겹치는 통화의 개수.

    코드 핵심 (변수/구조 매핑) —

    • long[][] callInfo = new long[N][4]: 통화 정보 저장. 실제 계산에는 callInfo[i][2] = cs, callInfo[i][3] = cd만 사용.
    • 문의 처리:
      • qs, qd → qe = qs + qd
      • 통화별 cs, cd → ce = cs + cd
      • 겹침 조건: if (cs < qe && qs < ce) cnt++;
    • 입력 루프: (N, M) = (0, 0) 또는 EOF까지 반복.
    • 출력 누적: StringBuilder sb에 줄 단위로 카운트 결과 추가 후 마지막에 일괄 출력.

    동작 과정 —
    1) 한 케이스의 N과 M을 읽는다.
    2) N개의 통화에 대해 cs, cd를 저장한다(번호 두 값은 건너뜀).
    3) M개의 문의에 대해:
      • qe = qs + qd를 계산하고,
      • 모든 통화에 대해 ce = cs + cd를 만든 뒤,
      • cs < qe 그리고 qs < ce면 카운트.
    4) 각 문의의 카운트를 출력 버퍼에 추가한다.
    5) 다음 케이스를 이어서 처리한다.

    정확성 근거 —
    반열린구간 [s, e)에서 두 구간 [a, b), [c, d)의 겹침은 a < d 그리고 c < b와 동치다. 이에 따라 끝점만 맞닿는 경우는 제외되고, 실제로 시간이 공유되는 경우만 카운트된다.

    엣지 케이스

    • 끝점 접촉: ce == qs 또는 qe == cs → 겹침 아님.
    • 문의/통화 길이가 0인 입력: 반열린구간에서 길이 0이면 실질 겹침 없음(카운트 증가 없음).
    • 여러 케이스/EOF: 0 0이나 EOF 처리로 안전하게 종료.
    • 큰 값: 합산 시 long 사용으로 범위 여유.

    시간·공간 복잡도 —
    각 케이스에서 문의마다 모든 통화를 선형 스캔하므로 O(NM). 추가 메모리는 통화 저장에 O(N).

    정리

    • 통화/문의 모두 반열린구간 [start, start+duration)으로 통일.
    • 겹침 판정은 cs < qe 그리고 qs < ce.
    • 문의마다 모든 통화를 검사해 카운트하고, 케이스별로 이어서 처리.

    문제 3974 — Absurd prices

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 —
    가격 c의 “기괴함(absurdity)”을 먼저 계산한다. 규칙은 다음과 같다:

    1. c에서 뒤쪽의 0(자릿수 말단의 0)을 모두 제거해 양의 정수 x를 만든다.
    2. x의 자릿수를 a라 하고, 마지막 자리(최하위) 숫자를 d라 한다.
    3. d = 5이면 기괴함은 2·a − 1, 그 외에는 2·a.
    이제 닫힌 구간 [0.95·c, 1.05·c] 안에 기괴함이 c보다 작은 정수 e가 하나라도 존재하면 c는 “absurd”, 없으면 “not absurd”.
    구간 경계는 정수로 올림/내림해 low = ⌈19c/20⌉, high = ⌊21c/20⌋로 잡는다(low = (19c+19)/20, high = (21c)/20).

    제약 정리 —
    입력: 테스트케이스 수 t, 이어서 각 줄에 가격 c(cent 단위), 1 ≤ c ≤ 109.
    출력: 각 c에 대해 absurd 또는 not absurd.

    코드 핵심 (변수/구조 매핑) —

    • static int absurd(long x): 위 규칙대로 기괴함을 계산.
      • 뒤 0 제거: while (clean % 10 == 0) clean /= 10;
      • 마지막 자리: lastDigit = clean % 10
      • 자릿수: cnt를 증가시키며 clean을 10으로 나눠 0이 될 때까지 반복
      • 반환: (lastDigit == 5) ? 2*cnt - 1 : 2*cnt
    • long[] p10: 10^k 프리컴퓨트(k=0..11)
    • 구간 경계:
      • low = (19L * c + 19) / 20 (올림)
      • high = (21L * c) / 20 (내림)
    • 후보 생성/검사:
      • step1 = 10^k와 step2 = 5·10^k를 단위로, c 근처의 배수들을 확인
      • m = c / step 근방의 m + d (d ∈ [-2..2])만 검사 → 후보 num = (m+d)·step
      • 후보가 [low, high]에 있고 양수이면, 기괴함 비교: absurd(num) < absurd(c)면 “absurd” 판정
    • 출력: 후보가 발견되면 즉시 종료하고 absurd, 끝까지 못 찾으면 not absurd.

    동작 과정 —
    1) t를 읽고, 10^k 테이블을 준비한다.
    2) 각 c에 대해:
      • absurdity = absurd(c) 계산
      • low, high 산출
      • 각 k에 대해 step ∈ {10^k, 5·10^k}:
        – m = c / step 근방의 배수 5개(±2)를 생성
        – 구간 포함 & 양수 체크 후 기괴함 비교
      • 더 작은 기괴함 후보를 찾으면 즉시 “absurd”, 없으면 “not absurd”.

    정확성 근거 —
    기괴함은 뒤 0 제거 후의 자릿수(a)에 비례하고, 마지막 자리가 5면 1만큼 더 유리하다. 따라서 “작은 기괴함”을 갖는 수는 10^k 또는 5·10^k의 배수처럼 뒤에 많은 0을 갖는 수(혹은 5로 끝나는 수) 근방에서 잘 나타난다. c를 해당 단위로 나눠 근처 배수(±2)만 검사해도 [0.95c, 1.05c] 구간 내에서 필요한 후보를 충분히 커버한다.

    엣지 케이스

    • 경계 포함: low와 high는 각각 올림/내림으로 계산하므로 닫힌 구간을 정확히 반영.
    • 후보의 유효성: num > 0 확인으로 0 이하 후보 배제.
    • 큰 값 처리: c ≤ 10^9, step ≤ 5·10^11 범위에서 long 연산 안전.
    • 즉시 종료: 더 작은 기괴함을 찾는 순간 탐색을 중단해 불필요한 계산 제거.

    시간·공간 복잡도 —
    각 테스트에서 k는 최대 12, 각 k당 후보는 최대 10개(두 단위 × 5개)이며, 후보 하나의 기괴함 계산은 자릿수 길이에 비례. 따라서 시간 복잡도는 대략 O(120 · log₁₀ c) 수준(상수에 가까움), 공간은 O(1).

    정리

    • 기괴함 계산: 뒤 0 제거 → 자릿수/마지막 자리로 2a 또는 2a−1.
    • 구간: low = ⌈19c/20⌉, high = ⌊21c/20⌋.
    • 후보: 10^k 및 5·10^k 배수의 근방(±2)만 검사.
    • 더 작은 기괴함이 구간 내에 존재하면 absurd, 아니면 not absurd.
