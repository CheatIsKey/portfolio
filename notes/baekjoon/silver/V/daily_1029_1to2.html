    문제 4158 — CD

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 —
    두 사람이 가진 CD 번호 목록이 각각 오름차순으로 주어지므로, 두 포인터로 한 번씩만 훑으며 공통 원소의 개수를 센다.
    각 포인터가 가리키는 값이 같으면 공통 개수(cnt)를 증가시키고 둘 다 한 칸 전진, 값이 다르면 더 작은 쪽 포인터만 전진한다.
    입력은 여러 테스트 케이스이며, 마지막에 0 0이 나오면 종료한다.

    제약 정리 —
    입력: 여러 테스트 케이스. 각 케이스는 N, M (각각 최대 1,000,000).
    다음 줄부터 N개의 상근이 CD 번호(오름차순), 이어서 M개의 선영이 CD 번호(오름차순).
    CD 번호는 1 이상 1,000,000,000 이하의 정수. 한 사람 내에서는 중복 없음. 종료 조건은 마지막 줄 0 0.
    출력: 각 케이스마다 두 사람이 동시에 가지고 있는 CD 개수.

    코드 핵심 (변수/구조 매핑) —

    • BufferedReader br: 대용량 입력 처리.
    • StringBuilder sb: 각 케이스 결과 누적 후 일괄 출력.
    • int N, M: 각 사람의 CD 개수.
    • int[] A, B: 상근이/선영이의 CD 번호 배열(오름차순 가정).
    • int idx1, idx2: 두 포인터(각 배열의 현재 위치).
    • int cnt: 공통 CD 개수 카운트.

    동작 과정 —
    1) N M을 읽어 N==0 && M==0이면 루프 종료.
    2) N개의 줄을 읽어 배열 A를 채운 뒤, M개의 줄로 배열 B를 채운다(입력 자체가 오름차순).
    3) 두 포인터 초기화(idx1=0, idx2=0), cnt=0으로 시작.
    4) while (idx1 < N && idx2 < M):

    • A[idx1] == B[idx2] → cnt++, 두 포인터 모두 증가.
    • A[idx1] < B[idx2] → idx1++.
    • 그 외(A[idx1] > B[idx2]) → idx2++.
    5) 루프 종료 후 sb.append(cnt).append('\n')로 결과 누적.
    6) 모든 케이스 처리 뒤 System.out.print(sb)로 한 번에 출력.

    정확성 근거 —
    두 배열이 정렬되어 있으므로, 두 포인터 방식은 각 원소를 최대 한 번씩만 비교하며 공통 원소를 빠짐없이 세어낼 수 있다.
    한 사람 내 중복이 없으므로 같은 값을 만났을 때 둘 다 한 칸 전진해도 누락/중복 카운트가 발생하지 않는다.
    정렬 가정과 비교 규칙에 의해 모든 공통 항목을 정확히 1회씩 집계한다.

    엣지 케이스

    • N==0 또는 M==0: 바로 0 출력.
    • 두 배열이 전혀 겹치지 않는 경우: cnt=0.
    • 모든 원소가 일치하는 경우: cnt = min(N, M) (문제 조건상 각자 중복 없음).
    • 입력 끝 0 0을 읽으면 즉시 종료.
    • 대용량 입력: BufferedReader, StringBuilder 사용으로 I/O 오버헤드 완화.

    시간·공간 복잡도 —
    두 포인터 비교는 각 케이스당 O(N+M). 입력 저장을 위해 배열 A, B가 필요하므로 공간은 O(N+M).
    (입력 자체가 정렬되어 들어오므로 추가 정렬 비용 없음.)

    정리

    • 정렬된 두 목록을 두 포인터로 선형 스캔.
    • 같으면 카운트 후 둘 다 전진, 다르면 작은 쪽만 전진.
    • 대용량에서도 O(N+M)로 효율적이며, 각 케이스 결과를 누적 출력.

    문제 4333 — Major Scales

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 —
    12개의 반음계 음(C, C#, D, D#, E, F, F#, G, G#, A, A#, B)을 기준으로, 각 음을 시작음으로 하는 장조(Major Scale)는 일정한 간격 패턴(tone, tone, semi-tone, tone, tone, tone, semi-tone)으로 구성된다.
    이를 반음 단위로 환산하면 [2, 2, 1, 2, 2, 2, 1]이다. 각 시작음을 기준으로 이 간격을 적용해 장조 음계 집합을 미리 계산하고, 주어진 입력의 모든 음이 특정 장조의 음계 안에 포함되는지를 검사한다. 포함되는 모든 장조의 이름을 오름차순(C, C#, D, …, B)으로 출력한다.

    제약 정리 —
    입력: 여러 테스트 케이스. 각 줄은 공백으로 구분된 음 이름들의 시퀀스이며, 마지막 줄은 END로 종료된다.
    한 줄의 길이는 최대 1000자. 각 음은 12개의 반음계 이름 중 하나이다.
    출력: 각 테스트 케이스마다 가능한 모든 장조(Key)를 공백으로 구분하여 출력한다. 가능한 장조가 없으면 빈 줄을 출력한다.

    코드 핵심 (변수/구조 매핑) —

    • String[] notes: 12개의 반음계 이름을 순서대로 저장.
    • int[] steps: 장조 스케일의 음 간격 패턴([2, 2, 1, 2, 2, 2, 1]).
    • Map<String, Set<String>> map: 각 시작음을 key로, 해당 장조의 음계 집합을 value로 저장.
    • Set<String> group: 입력 줄에 등장한 모든 음의 집합.
    • StringBuilder sb: 각 테스트 케이스 결과를 누적 후 일괄 출력.

    동작 과정 —
    1) 12개의 반음계 이름을 배열 notes에 저장한다.
    2) 장조 간격 패턴 steps를 이용해, 각 음(notes[i])을 시작으로 하는 장조의 7개 음을 계산한다.
    3) 계산된 각 장조 음계는 Set으로 저장해 map.put(notes[i], scale)에 등록한다.
    4) 이후 입력을 한 줄씩 읽는다. END가 나오면 종료.
    5) 입력 줄의 모든 음을 group 집합에 저장한다.
    6) 12개의 장조 중, map의 해당 음계 집합이 group을 모두 포함(containsAll())하는 경우를 찾는다.
    7) 조건을 만족하는 모든 키 이름을 공백으로 구분하여 출력, 아무 키도 없으면 빈 줄 출력.

    정확성 근거 —
    각 장조는 12개의 반음계 중 일정한 간격으로 생성되는 고정된 집합이므로, 모든 가능한 장조를 미리 구성해두면 완전 탐색이 가능하다.
    입력된 음이 특정 장조의 음계 집합에 모두 포함되는지 containsAll()로 판정하면, 그 장조에서만 연주 가능한 조합인지 정확히 판단할 수 있다.
    모든 장조(12개)를 검사하므로 누락 없이 가능한 모든 키를 탐색한다.

    엣지 케이스

    • 입력이 빈 줄인 경우: 출력도 빈 줄.
    • 모든 음이 동일한 경우: 대부분의 장조에서 포함되므로 여러 키가 출력될 수 있다.
    • 어떤 장조에도 속하지 않는 음 조합: 빈 줄 출력.
    • 입력 종료 조건 END: 루프 즉시 종료.

    시간·공간 복잡도 —
    장조 생성: 12개의 시작음 × 7개의 간격 → O(12×7) = O(1).
    각 입력 줄 처리: 최대 12개의 장조 검사 × 음 개수 n → O(12n).
    전체적으로 O(n) 선형 시간에 매우 효율적이며, 공간 복잡도는 O(12×7) = O(1).

    정리

    • 12개의 반음계 음을 기반으로 각 장조 음계를 미리 생성한다.
    • 입력된 음 집합이 어떤 장조 음계의 부분집합인지 검사한다.
    • 모든 조건을 만족하는 장조를 공백으로 구분해 출력한다.
    • 없으면 빈 줄, 마지막 입력 END에서 종료한다.
