    문제 4359 — Forests

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 —
    각 사람(1..P)이 들었다고 한 나무 번호들의 집합을 만들고, 이 집합이 완전히 같은 사람들끼리는 동일한 의견으로 본다.
    따라서 “서로 다른 의견 수” = {사람별 나무집합}의 서로 다른 원소 개수이다. 집합 동등성은 원소 기준이므로 순서 무관.

    제약 정리 —
    입력: 여러 테스트케이스가 빈 줄로 구분되어 이어서 주어진다. 각 케이스는 첫 줄에 P T (사람 수, 나무 수) — 주의: 문제 페이지 문구 오류로 보일 수 있으나 실제 입력은 P, T 순 — 가 주어지고,
    이어지는 줄들에 i j가 주어지며 “사람 i가 나무 j의 넘어짐을 들었다”를 뜻한다. 동일한 쌍이 중복될 수 있다.
    출력: 해당 케이스에서 서로 다른 의견(=서로 다른 나무 집합) 개수.

    코드 핵심 (변수/구조 매핑) —

    • BufferedReader br: 표준 입력 처리.
    • Map<Integer, Set<Integer>> map: 사람 번호 → 들었다고 한 나무들의 집합.
    • map.computeIfAbsent(person, ...): 사람별 집합 생성/갱신.
    • for (1..P) map.computeIfAbsent: “아무 것도 못 들은 사람”도 ∅ 집합으로 포함.
    • Set<Set<Integer>> cnt: 모든 사람의 집합들을 모아 중복 제거(자바의 Set#equals는 원소 기준이므로 적합).

    동작 과정 —
    1) 한 케이스의 첫 줄을 읽어 P, T를 파싱한다. 이때 순서가 P, T임에 유의.
    2) 이어지는 줄을 EOF 또는 빈 줄이 나올 때까지 읽으며 각 (person, tree)를 해당 사람의 집합에 추가한다.
    3) 1..P를 순회하며 입력이 없던 사람에게도 빈 집합을 할당한다(“아무 것도 듣지 못함”이라는 의견 포함).
    4) new HashSet<>(map.values())로 서로 다른 집합의 개수를 구해 출력한다.
    5) EOF 전까지 위 과정을 반복한다.

    정확성 근거 —
    의견의 동치성은 “들었다고 한 나무들의 집합”의 완전 일치로 정의된다. 자바의 Set은 순서와 무관하게 원소 동등성으로 equals/hashCode를 정의하므로,
    Set<Set<Integer>>로 중복을 제거하면 서로 다른 의견 수가 정확히 계산된다. 입력에 중복 쌍이 있어도 집합 특성상 자동으로 제거된다.
    또한 아무 것도 듣지 못한 사람도 ∅라는 의견으로 포함되므로 정의에 부합한다.

    엣지 케이스

    • 동일 쌍 다수 등장: 집합이므로 중복 무시.
    • 어떤 사람은 입력에 전혀 등장하지 않음: 빈 집합(∅) 의견으로 반드시 포함.
    • 빈 줄로 케이스 구분 / EOF로 종료: 코드에서 빈 줄을 만나면 해당 케이스 마감.
    • 입력 첫 줄 파싱: P, T 순서로 읽지 않으면 오답 유발(문제 페이지 표기 혼동 주의).

    시간·공간 복잡도 —
    각 케이스에서 입력된 쌍 수를 M이라 하면, 집합 삽입이 평균 O(1)이므로 전체 O(M + P).
    공간은 사람별 집합과 서로 다른 집합 보관에 O(M + P).

    정리

    • 사람별로 “들었다고 한 나무 번호”의 집합을 만든다.
    • 등장하지 않은 사람은 빈 집합(∅) 의견으로 포함한다.
    • Set<Set<Integer>>로 서로 다른 집합의 개수를 세면 “서로 다른 의견 수”.
    • 입력 첫 줄은 P T 순 — 문제 페이지 문구와 달라 헤매기 쉬우니 반드시 주의.

    문제 4378 — 트ㅏㅊ;

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 —
    사용자가 실제로는 오른쪽으로 한 칸 밀린 손 위치로 타이핑했으므로, 입력의 각 문자에 대해 쿼티 자판에서의 왼쪽 이웃 키로 치환하면 원문이 복원된다.
    스페이스(공백)는 손이 밀려도 위치 영향을 받지 않으므로 그대로 출력한다.

    제약 정리 —
    입력: 여러 줄이며, 각 줄은 숫자, 공백, 알파벳 대문자, 그리고 쿼티 상단에 있는 기호로만 구성된다. Q, A, Z 또는 `(back-quote)처럼 맨 왼쪽 열의 키는 입력에 등장하지 않는다(문제 보장). 스페이스는 예외적으로 등장 가능하며 오타가 없다.
    출력: 각 줄마다, 각 문자를 해당 자판의 왼쪽 이웃 키로 치환(공백은 그대로)하여 복원된 문자열을 출력.

    코드 핵심 (변수/구조 매핑) —

    • BufferedReader br: 표준 입력을 EOF까지 라인 단위로 읽는다.
    • StringBuilder sb: 변환 결과 누적 후 한 번에 출력.
    • String[] rows: 쿼티 자판의 4개 행을 왼→오 순서로 하드코딩.
    • Map<Character, Character> leftKey: 현재 키 → 왼쪽 이웃 키 매핑 테이블.

    동작 과정 —
    1) 자판 각 행에 대해 인덱스 1..len-1을 순회하며 leftKey[row[i]] = row[i-1]로 매핑을 채운다(맨 왼쪽 키는 매핑하지 않음).
    2) 입력을 한 줄씩 읽어 각 문자 c에 대해:
      • c == ' '이면 그대로 ' '를 추가.
      • 아니면 leftKey.getOrDefault(c, c)로 변환하여 추가. (문제 조건상 대부분 매핑 존재)
    3) 각 줄 처리 후 개행을 추가한다.
    4) 모든 줄을 처리하면 누적된 결과를 출력한다.

    정확성 근거 —
    문제의 “손이 오른쪽으로 한 칸 이동”은 자판 상에서 입력된 문자가 실제 의도 문자보다 오른쪽에 위치함을 의미하므로, 역변환은 “왼쪽 이웃으로 이동”이다.
    각 행에서 왼쪽 이웃을 매핑하면 행 내에서의 치환이 정확히 일치하며, 맨 왼쪽 열 키는 입력에 등장하지 않으므로 미정의 매핑이 사용되지 않는다. 집합적 규칙(한 문자당 독립 치환)이므로 라인 길이 및 줄 수와 무관하게 올바르게 복원된다.

    엣지 케이스

    • 스페이스: 그대로 유지(변환 없음).
    • 중복/연속 기호(예: ]]]]): 각 문자를 독립적으로 왼쪽 이웃으로 치환.
    • 맨 왼쪽 열 키: 문제에서 입력으로 주어지지 않는다고 보장(매핑 필요 없음).
    • EOF: 입력이 끝날 때까지 모든 줄을 변환(빈 줄도 개행 처리).

    시간·공간 복잡도 —
    매핑 테이블 구성은 자판 크기에 상수 시간. 변환은 전체 문자 수를 N이라 할 때 O(N), 추가 공간은 출력 버퍼와 매핑에 O(N) 및 O(1)(자판 크기 상수).

    정리

    • 쿼티 자판 4개 행을 하드코딩하고 “현재 키 → 왼쪽 이웃” 매핑을 만든다.
    • 각 문자를 매핑으로 치환(공백은 그대로)하여 한 줄씩 복원한다.
    • 문제 보장: 맨 왼쪽 열 키는 입력에 등장하지 않으므로 미매핑 문자로 인한 오류가 없다.
