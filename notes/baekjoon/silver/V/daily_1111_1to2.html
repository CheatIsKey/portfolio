    문제 4754 — Context-Free Clock

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Context-Free Clock (BOJ 4754)는 시계의 "위쪽" 기준이 사라진 상황에서, 시침과 분침 사이의 각도와 어떤 시각 이후라는 조건만으로 가능한 실제 시각을 역으로 계산하는 수학/구현 문제다.

    • 입력:
      • A : 시침에서 분침까지 시계 방향(clockwise)으로 잰 각도 (0 ~ 359)
      • HH:MM:SS : "이 시각 이후"라는 기준 시각 (24시간 형식)
    • 출력:
      • 조건을 만족하는 시각 중, 기준 시각 이상이면서 가장 처음 등장하는 시각을 HH:MM:SS (24시간 형식, 0-padding) 으로 출력
      • 각도는 시침에서 분침까지 시계 방향으로 간 값이며, 초 단위에서 내림(floor)한 시각이어야 한다.
    • 시계 바늘은 연속적으로 움직인다(초 단위로만 뛰지 않음).

    전략 개요

    • 핵심은 시침·분침의 각도 공식을 이용해 각도 조건을 만족하는 모든 시각을 수식으로 표현하고, 그 중에서 기준 시각 이상인 가장 작은 시각을 찾는 것이다.
    • 각도 정의:
      • 초 단위 시각 t (0 ≤ t < 86400)에 대해:
        • 분침 각도: 하루 기준이 아닌, 한 바퀴(60분) 기준이므로 minute = 0.1 × t (deg)
        • 시침 각도: 12시간(43200초)에 360도 회전이므로 hour = (1 / 120) × t (deg)
      • 시침에서 분침까지 시계 방향 각도: ((minute - hour) % 360 + 360) % 360
    • 상대 각속도:
      • minute - hour = (0.1 - 1/120) × t = (11/120) × t
      • 따라서 조건 ((11/120) × t) mod 360 = A 를 만족해야 한다.
    • 이를 방정식으로 적으면:
      • (11/120) × t = A + 360k (k는 정수)
      • 즉, t = (120/11) × (A + 360k)
      • 코드에서 쓰는 상수: FACT = 120.0 / 11.0 이다.
    • 관찰:
      • 각도 패턴은 12시간(43200초)마다 반복된다.
      • 우리는 기준 시각 start 이상인 해 중에서 가장 빠른 것을 찾고, 그 결과를 24시간(86400초) 안으로 나머지 연산 하여 출력하면 된다.

    핵심: 코드가 하는 일

    • 상수 정의:
      • DAY = 24 * 3600: 하루 초 단위 (86400)
      • FACT = 120.0 / 11.0: 위에서 유도한 식의 계수
      • 해당 시간은 항상 t = FACT * (A + 360k) 형태를 가진다.
    • 입력 처리:
      • 각 줄에서 A와 HH:MM:SS를 읽는다.
      • A == -1이면 전체 입력 종료.
      • HH:MM:SS를 초 단위로 변환: start = HH * 3600 + MM * 60 + SS
    • k의 초기값 추정:
      • 조건: t = FACT * (A + 360k) >= start
      • 이를 정리하면 k >= (start / FACT - A) / 360
      • 코드:
        • double kStart = (start / FACT - A) / 360.0;
        • long k = (long) Math.ceil(kStart - 1e-12);
          • 부동소수점 오차를 줄이기 위해 1e-12를 빼고 ceil 적용.
        • k < 0이면 가능한 최소는 0이므로 k = 0으로 보정.
    • 실제 t 계산 및 보정:
      • t = FACT * (A + 360.0 * k)로 후보 시각 계산.
      • 만약 부동소수 오차로 인해 t가 여전히 start보다 조금 작으면:
        • while (t + 1e-9 < start) 인 동안 k++ 후 다시 계산.
      • 이 과정을 거치면 조건을 만족하는 첫 번째 t ≥ start를 얻는다.
    • 초 단위 내림 및 24시간 범위 조정:
      • 문제에서 "각도는 해당 시각에서 내림"이므로:
        • answerSec = (long) Math.floor(t + 1e-9);
      • 24시간 형식으로 맞추기 위해:
        • answerSec = ((answerSec % DAY) + DAY) % DAY;
        • 음수 가능성을 모두 안전하게 제거.
      • 이후 answerH, answerM, answerS로 분해하여 %02d:%02d:%02d 형식으로 출력.
    • I/O 구현:
      • BufferedReader와 StringTokenizer를 사용해 빠른 입력 처리.
      • StringBuilder에 결과를 모아 마지막에 한 번에 출력.

    시간 복잡도

    각 테스트 케이스마다 상수 번의 연산만 수행하므로 시간 복잡도는 O(1). 전체 입력에 대해선 O(T) (T는 테스트 케이스 수).

    공간 복잡도

    상수 개의 변수만 사용하므로 O(1).

    주의

    • 부동소수점 오차:
      • k 계산 시 - 1e-12, t 비교 시 + 1e-9, floor 시 + 1e-9 등으로 경계값 오차를 방지한다.
      • 이 처리가 없으면 근접한 해를 놓치거나, start보다 작은 t를 선택하는 오답이 나올 수 있다.
    • 각도 정의 착각 금지:
      • A는 항상 시침 → 분침을 시계 방향으로 잰 값이다.
      • 일반적인 "두 바늘 사이의 최소 각도"가 아니다.
    • 반복 주기:
      • 해는 12시간마다 반복되지만, 출력은 24시간 형식이므로 % DAY 처리로 0~23시 안에 넣어줘야 한다.

    포인트 요약

    • 시침·분침의 상대 각속도는 11/120 deg/s 이고, 이를 이용해 해의 형태 t = FACT * (A + 360k)를 만든다.
    • 기준 시각 이상이 되도록 하는 최소의 정수 k를 계산해 그에 해당하는 t를 선택한다.
    • 각도는 해당 순간에서 초 단위 버림.
    • 결과 시각은 24시간(0~23시) 범위로 맞춘 뒤 HH:MM:SS 형식으로 출력.
    • 부동소수점 경계 처리(1e-9, 1e-12)가 실전 구현에서 매우 중요하다.

    문제 4775 — Spelling Be

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Spelling Be (BOJ 4775)는 주어진 사전(dictionary)을 기준으로 여러 개의 이메일에서 등장하는 단어들이 모두 올바른 단어인지 검사하고, 잘못된 단어가 있다면 어떤 단어들이 잘못되었는지 출력하는 문제다.

    • 입력 개요:
      • N : 사전에 포함된 단어의 개수
      • 다음 N줄 : 사전에 들어 있는 올바른 단어들
      • M : 검사해야 할 이메일의 개수
      • 각 이메일:
        • 여러 줄에 걸쳐 단어들이 주어지다가, "-1" 한 줄이 나오면 해당 이메일 입력 종료
    • 출력:
      • 각 이메일에 대해:
        • 모든 단어가 사전에 있으면:
          Email i is spelled correctly.
        • 하나라도 사전에 없는 단어가 있으면:
          Email i is not spelled correctly.
          이어서 그 이메일에서 사전에 없는 단어들을 줄마다 출력
      • 모든 이메일 처리 후 마지막 줄에:
        End of Output

    전략 개요

    • 사전을 빠르게 조회하기 위해 모든 단어를 HashSet에 저장한다.
    • 각 이메일에 대해:
      • "-1"이 나올 때까지 단어를 한 줄씩 읽으면서 사전에 없는 단어만 따로 모은다.
      • 모은 리스트가 비어 있으면 "spelled correctly" 문구 출력,
      • 비어 있지 않으면 "not spelled correctly" + 리스트 출력.
    • 구조적으로는 그대로 구현하는 단순한 검증 문제이며, 핵심은 입력 형식과 이메일 구분(-1)을 정확히 처리하는 것.

    핵심: 코드가 하는 일

    • 입력 루프:
      • while ((line = br.readLine()) != null) 로 EOF까지 처리.
      • 각 블록에서 먼저 N을 읽고, 그에 맞춰 사전과 이메일들을 처리한다.
    • 사전 구성:
      • int N = Integer.parseInt(st.nextToken());
      • Set<String> dict = new HashSet<>(N);
      • 다음 N줄의 단어를 dict에 삽입.
      • HashSet 사용으로 포함 여부 확인이 O(1)에 가능.
    • 이메일 처리 준비:
      • int M = Integer.parseInt(br.readLine().trim()); : 이메일 개수.
      • List<String> list = new ArrayList<>(); : 해당 이메일에서 사전에 없는 단어들을 모을 리스트.
    • 각 이메일 처리:
      • for (int i = 1; i <= M; i++): 이메일 번호를 1부터 M까지 순회.
      • 이메일 내용 읽기:
        • while (true) 안에서 한 줄씩 읽고, "-1"이면 현재 이메일 종료.
        • 그 외의 문자열 s에 대해:
          • if (!dict.contains(s)) list.add(s);
          • 사전에 없는 단어만 리스트에 저장한다.
      • 검사 결과 출력:
        • 오타 없음:
          • list.isEmpty() 이면
            Email i is spelled correctly.
        • 오타 있음:
          • Email i is not spelled correctly.
          • 이후 list에 쌓인 단어들을 등장 순서대로 한 줄씩 출력.
        • 마지막에 list.clear();로 다음 이메일을 위해 비운다.
    • 출력 마무리:
      • 모든 입력 처리가 끝난 후 sb.append("End of Output");
      • System.out.print(sb.toString());으로 한 번에 출력.
    • I/O:
      • BufferedReader로 빠르고 안정적인 입력 처리.
      • StringBuilder로 출력 문자열을 누적해 불필요한 flush/println 호출을 줄인다.

    시간 복잡도

    사전 단어 수를 N, 모든 이메일에 등장하는 단어 수의 합을 K라 하면, 각 단어에 대해 HashSet 조회를 한 번 수행하므로 전체 시간 복잡도는 O(N + K).

    공간 복잡도

    사전 저장을 위한 HashSet이 O(N), 각 이메일에 대해 오타 단어를 저장하는 리스트가 최대 그 이메일 단어 수에 비례하므로 전체적으로 O(N + 최대 이메일 단어 수).

    주의

    • 입력 구분:
      • "-1"은 이메일 종료를 의미하며, 실제 단어로 취급하면 안 된다.
    • 공백 처리:
      • 각 줄 입력 시 trim()을 사용해 불필요한 공백을 제거하는 것이 안전하다.
    • 자료구조 선택:
      • 사전은 HashSet을 사용해야 빠른 포함 여부 확인이 가능하다.
    • 출력 형식:
      • 문제에서 요구하는 문구와 줄바꿈을 정확히 맞추지 않으면 오답 처리된다.

    포인트 요약

    • 사전 단어들을 HashSet에 넣고, 각 이메일의 단어를 순차적으로 검사한다.
    • 사전에 없는 단어가 하나도 없으면 Email i is spelled correctly.
    • 사전에 없는 단어가 있으면 Email i is not spelled correctly. 후, 그 단어들을 한 줄씩 출력한다.
    • 이메일은 "-1"로 구분되며, 모든 이메일 처리 후 End of Output을 출력한다.
    • 문제는 입력 파싱과 조건에 맞는 출력 형식만 정확히 지키면 해결되는 구현 문제다.
