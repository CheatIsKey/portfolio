    문제 4809 — Stock Prices

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Stock Prices (BOJ 4809)는 여러 날에 걸친 주가 기록이 주어졌을 때, 가장 싼 날들(k1개)과 가장 비싼 날들(k2개)의 인덱스를 출력하는 문제다.
    단순히 값만 찾는 것이 아니라, 동일한 가격이 여러 번 등장할 때 어떤 날들을 선택할지에 대한 “사전식(lexicographical)” 규칙까지 맞춰야 한다.

    • 입력:
      • n : 가격이 주어지는 일 수 (≤ 1,000,000)
      • k1, k2 : 각각 가장 낮은 가격을 뽑을 개수, 가장 높은 가격을 뽑을 개수 (≤ 100, k1 + k2 ≤ n)
      • 다음 줄: 길이 n의 정수열, i번째 값은 i번째 날의 주가
      • n = k1 = k2 = 0인 줄이 나오면 입력 종료
    • 출력 (각 테스트케이스마다):
      • 첫 줄: Case X (X는 1부터 시작하는 케이스 번호)
      • 둘째 줄: k1개의 가장 낮은 가격이 등장하는 “날”들의 인덱스들
        • 인덱스는 1-based (문제 정의)
        • 날짜는 오름차순으로 출력
        • 동점(같은 가격)이 많아 가능한 답이 여러 개면, 사전식으로 가장 작은 목록을 선택
      • 셋째 줄: k2개의 가장 높은 가격이 등장하는 “날”들의 인덱스들
        • 날짜는 내림차순으로 출력
        • 동점이 많아 가능한 답이 여러 개면, 사전식으로 가장 큰 목록을 선택

    전략 개요

    • 각 날을 (가격, 날 번호) 쌍으로 다룬 뒤 정렬하는 방식으로 해결할 수 있다.
    • 핵심은:
      • k1개의 최소 가격을 뽑을 때, (가격 오름차순, 같은 가격이면 날 번호 오름차순)으로 정렬 후 앞에서부터 k1개를 선택하면, 조건을 만족하는 “사전식 최소” 날짜 집합을 얻을 수 있다.
      • k2개의 최대 가격을 뽑을 때, (가격 내림차순, 같은 가격이면 날 번호 내림차순)으로 정렬 후 앞에서부터 k2개를 선택하면, 조건을 만족하는 “사전식 최대” 날짜 집합을 얻을 수 있다.
    • 그 뒤, 선택한 날 번호들을 문제에서 요구하는 정렬 순서에 맞게(최솟값 쪽은 오름차순, 최댓값 쪽은 내림차순) 다시 정렬해 출력하면 된다.

    핵심: 왜 이렇게 하면 “사전식 최소/최대”가 되는가?

    • k1 최소 가격 쪽 (오름차순, 사전식 최소)
      • 각 날을 (price, day)로 보고, 전체를 price 오름차순, price가 같으면 day 오름차순으로 정렬한다.
      • 앞에서부터 k1개를 고르면:
        • 먼저 “가격이 더 작은 날”이 항상 우선적으로 뽑힌다.
        • 동일 가격에서는 “날 번호가 더 작은 날”이 우선적으로 뽑힌다.
      • 이렇게 뽑힌 k1개의 day들만 따로 모아서 오름차순으로 정렬해 출력하면, 가능한 모든 해 중에서 사전식(lexicographical)으로 가장 작은 오름차순 리스트가 된다.
        • 직관적으로, “가능한 후보들 중 가장 앞날부터 채우는” 전략이라 생각하면 된다.
    • k2 최대 가격 쪽 (내림차순, 사전식 최대)
      • 각 날을 (price, day)로 보고, 전체를 price 내림차순, price가 같으면 day 내림차순으로 정렬한다.
      • 앞에서부터 k2개를 고르면:
        • 먼저 “가격이 더 큰 날”이 우선적으로 뽑힌다.
        • 동일 가격에서는 “날 번호가 더 큰 날”이 우선적으로 뽑힌다.
      • 이렇게 뽑힌 k2개의 day를 내림차순으로 정렬해 출력하면, 가능한 모든 해 중에서 사전식으로 가장 큰 내림차순 리스트가 된다.

    핵심: 코드가 하는 일

    • 입력 처리 & 루프
      • caseNo = 1부터 시작.
      • 매 줄마다 n, k1, k2를 읽고, n = k1 = k2 = 0이면 전체 루프 종료.
      • 그 다음 줄에서 길이 n의 가격 배열 price[1..n]를 읽는다.
    • 데이터 구조 준비
      • 각 날에 대해 구조체/클래스/튜플 (price, day)를 만든다. (day는 1-based 인덱스)
      • 예: Java라면 class Node { int price, day; } 형태로 배열을 만들 수 있다.
      • 혹은 최소/최대용으로 각각 따로 배열을 복사해 써도 되지만, 한 배열로 재활용해도 충분하다.
    • k1 최소 가격용 정렬 & 선택
      • 전체 (price, day) 배열을 가격 오름차순, 날짜 오름차순으로 한 번 정렬한다.
        • Comparator: (a, b) → a.price != b.price ? a.price - b.price : a.day - b.day
      • 정렬된 배열에서 앞쪽 k1개의 원소를 선택하고, 이들의 day 값만 모아서 별도의 리스트에 저장한다.
      • 그 리스트를 오름차순으로 정렬한 뒤, 공백으로 구분해 출력한다.
        • 이 줄이 “k1개의 최소 가격이 나타난 날” 리스트가 된다.
    • k2 최대 가격용 정렬 & 선택
      • 같은 데이터에서 다시 정렬하되, 이번에는 가격 내림차순, 날짜 내림차순으로 정렬한다.
        • Comparator: (a, b) → a.price != b.price ? b.price - a.price : b.day - a.day
      • 앞쪽 k2개의 원소를 선택해서 그들의 day 값을 리스트에 모은다.
      • 이 리스트를 내림차순으로 정렬한 뒤, 공백으로 구분해 출력한다.
        • 이 줄이 “k2개의 최대 가격이 나타난 날” 리스트가 된다.
    • 출력 포맷
      • 각 테스트케이스마다: 
        Case X
      [d1 d2 ... dk1]
      [e1 e2 ... ek2]
      • 사이에 빈 줄 없이 연속해서 출력한다.

    시간 복잡도

    한 케이스당:

    • n개의 (price, day) 쌍 생성: O(n)
    • 최소 가격용 정렬: O(n log n)
    • 최대 가격용 정렬: O(n log n) (같은 배열을 다시 정렬한다고 가정)
    • k1, k2는 최대 100이므로, 나머지 선택 및 출력에 드는 비용은 O(k1 + k2)로 무시 가능

    따라서 한 케이스의 전체 시간 복잡도는 O(n log n) 정도이며, n ≤ 1,000,000에서도 충분히 통과 가능한 수준이다 (언어/구현에 따라 적절한 입출력 최적화 필요).

    공간 복잡도

    (price, day) 쌍을 저장하는 배열이 O(n) 필요하며,
    k1, k2 개수만큼의 보조 리스트는 최대 200개 정도라 전체 공간 복잡도는 O(n)이다.

    주의

    • 인덱스는 1-based
      • i번째 입력 가격은 i번째 날 가격이므로, day를 저장할 때 1부터 시작해야 한다.
    • 사전식(lexicographical) 조건 처리
      • 최소 가격 쪽:
        • (price 오름차순, day 오름차순)으로 정렬 후 앞에서 k1개를 뽑는다.
        • 그 후 day를 오름차순 정렬 → “사전식 최소” 조건을 만족.
      • 최대 가격 쪽:
        • (price 내림차순, day 내림차순)으로 정렬 후 앞에서 k2개를 뽑는다.
        • 그 후 day를 내림차순 정렬 → “사전식 최대” 조건을 만족.
    • 입출력 성능
      • n이 최대 1,000,000 이고 테스트케이스 여러 개일 수 있으므로, Scanner 대신 BufferedReader + StringTokenizer (Java 기준) 같은 빠른 I/O를 쓰는 것이 좋다.
      • 출력도 StringBuilder에 모아서 한 번에 출력하는 편이 안전하다.

    포인트 요약

    • 날짜를 (가격, 인덱스) 쌍으로 관리한다.
    • k1 최저가:
      • 가격↑, 날↑ 정렬 → 앞에서 k1개 선택 → day만 뽑아서 오름차순 출력.
      • 이렇게 하면 동점이 있을 때 사전식으로 가장 이른 날짜들이 선택된다.
    • k2 최고가:
      • 가격↓, 날↓ 정렬 → 앞에서 k2개 선택 → day만 뽑아서 내림차순 출력.
      • 이렇게 하면 동점이 있을 때 사전식으로 가장 늦은 날짜들이 선택된다.
    • 시간 복잡도는 각 케이스당 O(n log n), 공간은 O(n).
    • 입출력 최적화와 1-based 인덱스, 정렬 기준(오름/내림)만 정확히 맞추면 구현 난이도는 비교적 낮은 편이다.

    문제 4818 — Ropes

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Ropes (BOJ 4818)는 하나의 등반 루트(여러 개의 피치)와 세 종류의 로프 길이(50, 60, 70m)가 주어졌을 때,
    각 로프를 사용할 수 있다면 최대 몇 명의 클라이머가 이 로프로 등반 & 하강할 수 있는지를 구하는 문제다.

    • 입력:
      • 각 줄: N P1 P2 ... PN
      • N: 피치 개수 (1 ≤ N ≤ 100)
      • Pk: k번째 피치 길이 (1 ≤ Pk ≤ 100)
      • 마지막 줄은 0 하나, 입력 종료
    • 출력:
      • 각 루트(입력 한 줄)에 대해, 50, 60, 70m 로프 각각에 대해 사용 가능하다면 최대 클라이머 수, 사용 불가능하면 0
      • 세 값을 공백으로 구분해서 출력

    등반 시에는 각 피치마다:

    • 안전하게 등반하려면 적어도 2명이어야 하고,
    • 로프 길이는 피치 길이 이상이어야 한다.

    하강(래펠) 시에는 전체 루트에 대해:

    • 로프를 중간에서 걸어 두 갈래로 나누어 내려가므로,
      로프 길이 L가 전체 높이의 두 배 이상이어야 한다. (L ≥ 2 × (P1 + ... + PN))

    전략 개요

    • 각 로프 길이 L ∈ {50, 60, 70}에 대해 독립적으로:
      1. 이 루트에 해당 로프가 적합한지 판단한다.
        • 모든 피치에 대해 L ≥ max(Pk) (등반 가능)
        • 그리고 L ≥ 2 × sum(Pk) (하강 가능)
        • 둘 중 하나라도 안 되면 → 이 로프는 0 출력
      2. 적합하다면, 해당 로프를 쓸 수 있는 최대 클라이머 수를 계산한다.

    핵심은 “한 피치를 몇 명까지 동시에 이 로프로 처리할 수 있는가?”를 수식으로 만드는 것이다.

    한 피치에서 가능한 최대 클라이머 수

    문제 설명의 예시:

    • 피치 길이: 10m
    • 로프 길이: 50m
    • 최대 6명까지 가능, 마지막 사람은 로프 끝에 묶인다.

    직관적으로 보면, 서로 이웃한 클라이머 사이의 로프 간격이 피치 길이만큼 떨어져 있어야 안전하게 올라갈 수 있다고 볼 수 있다. (선두는 상단 확보지점에 묶여 있고, 그 아래로 간격마다 한 명씩 묶여 있는 형태)

    • 클라이머 수를 m이라 하면, 클라이머들 사이의 “간격”은 m - 1개.
    • 각 간격마다 pitch 길이의 로프가 필요하므로:
      • (m - 1) × pitch ≤ L
    • 따라서:
      • m - 1 ≤ L / pitch
      • m ≤ L / pitch + 1
      • m_max = ⌊L / pitch⌋ + 1

    예시 확인:

    • L = 50, pitch = 10 → m_max = ⌊50/10⌋ + 1 = 5 + 1 = 6

    그리고 문제 조건상 최소 2명 이상이어야 하는데,
    L ≥ pitch이면 ⌊L/pitch⌋ ≥ 1이므로 항상 m_max ≥ 2가 된다.

    여러 피치가 있는 경우, 모든 피치에 대해 이 조건을 동시에 만족해야 하므로:

    • 각 피치 Pk마다 m ≤ ⌊L / Pk⌋ + 1
    • 전체 루트에 대해 가능한 최대 인원: m_max = min_k (⌊L / Pk⌋ + 1)

    단, 이 전에 이미 로프 적합성 조건: L ≥ max(Pk), L ≥ 2 × sum(Pk)를 만족해야 한다.

    핵심: 코드가 하는 일

    • 입력 처리
      • 반복해서 한 줄을 읽어 N과 P1..PN을 받는다.
      • N == 0이면 전체 종료.
      • 각 케이스마다:
        • sumP = P1 + ... + PN (전체 높이)
        • maxP = max(Pk) (가장 긴 피치)
    • 각 로프 길이에 대한 처리 (L ∈ {50, 60, 70})
      • 1단계: 로프 적합성 검사
        • 등반 가능: L ≥ maxP
        • 하강 가능: L ≥ 2 × sumP
        • 둘 중 하나라도 실패하면: 결과는 0
      • 2단계: 최대 클라이머 수 계산 (적합한 경우)
        • 초기값: m_max = 매우 큰 값
        • 모든 피치 Pk에 대해:
          • curr = L / Pk + 1 (정수 나눗셈)
          • m_max = min(m_max, curr)
        • 최종적으로 구한 m_max를 이 로프의 결과로 사용
    • 출력
      • 각 케이스마다 ans50 ans60 ans70을 한 줄에 공백으로 구분해 출력.

    시간 복잡도

    한 케이스당:

    • 피치 합/최댓값 계산: O(N)
    • 각 로프 길이에 대해:
      • 피치 전체를 한 번씩 훑으며 m_max 계산: O(N)
      • 로프가 3개이므로 총 O(3N) = O(N)

    전체 시간 복잡도는 O(T × N) (T는 케이스 수)이며,
    N ≤ 100이므로 매우 여유롭다.

    공간 복잡도

    피치 길이 배열과 약간의 변수만 사용하므로 O(N) (실질적으로 매우 작다).

    주의

    • 등반 + 하강 조건 둘 다 필요
      • L ≥ max(Pk)만 봐서는 안 되고,
        반드시 L ≥ 2 × sum(Pk)도 동시에 만족해야 해당 로프 사용 가능.
    • 정수 나눗셈
      • L / Pk는 정수 나눗셈(몫)이다.
      • 공식: m_max = ⌊L / Pk⌋ + 1을 그대로 구현하면 된다.
    • 최소 2명 조건
      • 로프 적합성 조건에서 이미 L ≥ maxP를 확인하므로, m_max ≥ 2는 자동으로 보장된다.

    포인트 요약

    • 로프 길이 L가 루트에 적합하려면:
      • L ≥ max(Pk) (모든 피치 등반 가능)
      • L ≥ 2 × sum(Pk) (전체 루트 래펠 가능)
      • 둘 중 하나라도 실패하면 결과는 0
    • 적합하다면, 각 피치별 최대 인원:
      • m_k = ⌊L / Pk⌋ + 1
      • 전체 루트에 대해 가능한 인원: m_max = min_k m_k
    • 위 과정을 50, 60, 70m에 대해 각각 수행하고, 세 값을 한 줄에 출력한다.
    • 수학 자체는 단순하지만, “등반 + 하강” 두 조건을 모두 만족해야 한다는 점과,
      “모든 피치에 대해 동시에 가능한 최대 인원”이 min_k라는 점만 정확히 잡으면 구현은 간단하다.
