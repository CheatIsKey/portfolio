    문제 5043 — 정말 좋은 압축

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    정말 좋은 압축 (BOJ 5043)은
    서로 다른 1000비트짜리 파일이 N개 주어졌을 때,
    이 파일들을 모두 압축해서 각 파일의 길이가 최대 b비트 이하가 되도록 만드는 압축 알고리즘이 “이론적으로” 존재할 수 있는지를 판별하는 문제다.

    핵심 질문은 다음 한 줄로 정리된다.

    “길이가 최대 b비트인 서로 다른 비트열을
    최대 몇 개까지 만들 수 있는가?”

    이 값을 모든 압축 결과(서로 다른 코드) 개수의 최댓값이라고 보면,
    그 개수 이상으로 파일들을 구분할 수 있어야만 압축이 가능하다.

    핵심 조건을 정리하면:

    • 원본 파일:
      • 각 파일은 1000비트이며, 문제에서 주어지는 N개는 서로 다르다.
      • 하지만 전체 21000개의 모든 파일을 압축해야 하는 것이 아니라, 입력에서 주어진 N개만 구분해서 압축하면 된다.
    • 압축 결과:
      • 각 파일은 길이가 0비트부터 b비트까지인 비트열 하나로 표현된다.
      • 따라서 “서로 다른 압축 결과 코드”의 개수는 2^0 + 2^1 + 2^2 + ... + 2^b개가 될 수 있다.
    • 필요 충분 조건:
      • 압축 결과로 쓸 수 있는 서로 다른 비트열이 충분히 많아야 한다.
      • 즉, N ≤ (길이가 최대 b인 비트열 개수의 최댓값)이면 이론적으로는 항상 압축이 가능하다.

    따라서, “길이가 최대 b인 서로 다른 비트열의 개수”를 구해
    그 값이 N 이상인지 비교만 해 주면 된다.

    전략 개요

    • 길이가 정확히 i비트인 비트열의 개수는 2^i개다.
    • 길이가 최대 b비트인 비트열의 개수는 2^0 + 2^1 + ... + 2^b가 된다.
    • 이 합은 등비수열의 합 공식으로 2^(b+1) - 1이지만,
      코드에서는 굳이 공식을 쓰지 않고 직접 i = 0..b를 순회하며 더한다.
    • 이렇게 구한 “가능한 서로 다른 압축 코드의 최대 개수”를 file이라고 두고,
      file ≥ N이면 “yes”, 아니면 “no”를 출력한다.

    결국 이 문제는 압축 알고리즘을 직접 설계하는 것이 아니라,
    서로 다른 코드의 개수가 N개 이상 나올 수 있는지만 확인하는 단순한 수학/카운팅 문제가 된다.

    핵심: 코드가 하는 일

    • 입력 처리
      • BufferedReader로 한 줄씩 읽는다.
      • 각 줄에서 N, b를 공백 기준으로 분리해 읽는다. 
        String line;
while ((line = br.readLine()) != null) {
    line = line.trim();
    StringTokenizer st = new StringTokenizer(line);

    long N = Long.parseLong(st.nextToken());
    int b = Integer.parseInt(st.nextToken());
    ...
}
      • 문제는 한 줄 입력이지만, 이 코드는 여러 줄이 들어와도 EOF까지 반복 처리할 수 있도록 작성되어 있다.
    • 가능한 코드 수 계산
      • file 변수에 “길이 ≤ b인 서로 다른 비트열의 최대 개수”를 누적한다. 
        long file = 0;
for (int i = 0; i <= b; i++) file += Math.pow(2, i);
      • i를 0부터 b까지 돌면서 2^i를 모두 더하므로
        최종적으로 file = 2^0 + 2^1 + ... + 2^b가 된다.
      • 최대 b = 50이므로, 이 값은 2^51 - 1 수준이고,
        long 범위(약 9 × 1018) 안에 충분히 들어간다.
    • 압축 가능 여부 판별
      • 서로 다른 코드의 수가 N개 이상이면, N개 파일을 서로 다르게 매핑하는 것이 가능하다.
      • 따라서 다음과 같이 비교 후 출력한다. 
        if (file >= N) sb.append("yes").append('\n');
else sb.append("no").append('\n');
      • 모든 줄을 처리한 뒤, 마지막에 한 번만 System.out.print(sb);로 출력한다.

    시간 복잡도

    각 테스트케이스(각 줄)에 대해:

    • for (int i = 0; i <= b; i++) 루프만 보면, 반복 횟수는 최대 b+1 ≤ 51회로 상수에 가깝다.
    • 각 반복에서 하는 일은 Math.pow와 덧셈 한 번뿐인 상수 시간 연산이다.

    따라서 한 줄(한 케이스)에 대한 시간 복잡도는 사실상
    O(b)이고, 여기서 b ≤ 50인 상수이므로 전체적으로 O(1)에 가깝다.

    여러 줄 입력을 처리하더라도, 각 줄마다 상수 시간에 해결 가능하므로
    전체 프로그램의 시간 복잡도 역시 매우 작다.

    공간 복잡도

    사용되는 추가 메모리는:

    • 입력 파싱용 문자열/토크나이저
    • N, b, file 같은 몇 개의 정수 변수
    • 출력 누적용 StringBuilder

    입력 크기 N과 상관없이 고정된 크기의 변수만 사용하므로,
    공간 복잡도는 O(1)이다.

    주의

    • 길이 0비트 코드:
      • b = 0일 때도 비트열은 1개(길이 0인 코드) 존재한다고 보는 것이 핵심이다.
      • 예제 N = 1, b = 0에서 답이 yes인 이유가 바로 이것이다.
    • 타입 범위:
      • 2^0 + 2^1 + ... + 2^b는 최대 2^51 - 1 정도이므로
        반드시 long으로 저장해야 한다.
      • int로 두면 오버플로가 발생해 잘못된 결과를 낼 수 있다.
    • Math.pow 사용:
      • Math.pow는 double을 반환하지만,
        여기서는 2의 거듭제곱을 더하는 형태라 long으로 캐스팅해도 안전하게 동작한다.
      • 보다 깔끔하게 작성하려면 (1L << i)를 사용하는 방법도 있다.
    • 문제 해석:
      • 이 문제는 “모든 1000비트 파일을 항상 압축할 수 있는가?”가 아니라,
        “주어진 서로 다른 N개 파일만 구분해 압축하면 되는가?”를 묻는 것이다.
      • 따라서 N과 “가능한 서로 다른 코드 개수”만 비교하면 충분하다.

    포인트 요약

    • 길이가 정확히 i비트인 비트열은 2^i개, 길이가 최대 b비트인 비트열은 2^0 + ... + 2^b개이다.
    • 이 개수(서로 다른 코드의 최대 개수)가 N 이상이면, N개 파일을 서로 다른 코드로 압축하는 것이 이론적으로 가능하다.
    • 코드에서는 단순히 i = 0..b에 대해 2^i를 모두 더해 file에 저장한 뒤,
      file ≥ N인지 비교해 “yes/no”를 출력한다.
    • 시간/공간 복잡도 모두 사실상 상수 수준으로 매우 작다.
    • 길이 0 비트 코드까지 포함해 세야 예제 N = 1, b = 0를 올바르게 처리할 수 있다.

    문제 5077 — Maps

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Maps (BOJ 5077)는
    눈밭 위 건물 배치를 나타내는 작은 패턴(이미지)을 하나 주고,
    더 큰 지도(map) 안에서 이 패턴이 정확히 일치하는 위치가 몇 번 등장하는지를 세는 문제다.

    각 위치는 다음 조건을 만족할 때만 하나의 “발견된 비밀 기지”로 센다.

    • 패턴 이미지와 지도에서 선택한 부분이 모양 그대로 완전히 일치해야 한다.
    • 회전, 대칭(미러링)은 인정되지 않는다. 즉, 90도 회전/좌우 반전/상하 반전 등은 모두 다른 모양이다.
    • 눈은 ., 건물은 X로 표현되며, X/.가 하나라도 다르면 그 위치는 불일치로 본다.

    입력으로 여러 개의 시나리오가 주어지며,
    각 시나리오에 대해 패턴이 지도에 몇 번 나타나는지 하나의 정수로 출력하면 된다.

    전략 개요

    • 패턴의 크기를 Li × Ci, 지도의 크기를 Lm × Cm이라고 하자.
    • 패턴이 지도에 들어갈 수 있는 시작 좌표(sr, sc)의 범위는:
      • 행 시작 sr: 0 ≤ sr ≤ Lm - Li
      • 열 시작 sc: 0 ≤ sc ≤ Cm - Ci
    • 각 시작 좌표마다, 패턴 전체를 지도에서 겹쳐보면서 모든 칸이 일치하는지 검사한다.
    • 해당 위치에서 Li × Ci 모든 칸이 완전히 같으면 카운트를 1 증가시킨다.
    • 이 과정을 모든 가능한 시작 좌표에 대해 반복하면, 전체 등장 횟수를 구할 수 있다.

    입력 크기가 작기 때문에, 브루트포스 2차원 패턴 매칭만으로도 충분히 통과된다.

    핵심: 코드가 하는 일

    • 전체 시나리오 수 읽기
      • 첫 줄에서 정수 하나를 읽어 scenarios에 저장한다. 
        int scenarios = Integer.parseInt(line);
for (int testcase = 0; testcase < scenarios; testcase++) {
    ...
}
    • 패턴(이미지) 입력
      • Li, Ci를 읽고, Li × Ci 크기의 문자 배열을 만든다. 
        StringTokenizer st = new StringTokenizer(br.readLine());
int Li = Integer.parseInt(st.nextToken());
int Ci = Integer.parseInt(st.nextToken());

char[][] scenario = new char[Li][Ci];
for (int i = 0; i < Li; i++) {
    String row = br.readLine();
    for (int j = 0; j < Ci; j++) {
        scenario[i][j] = row.charAt(j);
    }
}
      • scenario 배열에 패턴 이미지(비밀 기지 도면)를 그대로 저장한다.
    • 지도(map) 입력
      • Lm, Cm을 읽고, Lm × Cm 크기의 문자 배열을 만든다. 
        st = new StringTokenizer(br.readLine());
int Lm = Integer.parseInt(st.nextToken());
int Cm = Integer.parseInt(st.nextToken());

char[][] map = new char[Lm][Cm];
for (int i = 0; i < Lm; i++) {
    String row = br.readLine();
    for (int j = 0; j < Cm; j++) {
        map[i][j] = row.charAt(j);
    }
}
      • map 배열에 실제 눈밭 지도를 통째로 저장한다.
    • 2차원 패턴 매칭
      • 패턴이 들어갈 수 있는 모든 시작 좌표(sr, sc)를 순회한다. 
        int count = 0;
for (int sr = 0; sr <= Lm - Li; sr++) {
    for (int sc = 0; sc <= Cm - Ci; sc++) {
        boolean ok = true;
        for (int i = 0; i < Li && ok; i++) {
            for (int j = 0; j < Ci; j++) {
                if (map[sr + i][sc + j] != scenario[i][j]) {
                    ok = false;
                    break;
                }
            }
        }

        if (ok) count++;
    }
}
      • sr, sc는 각각 “지도에서 패턴을 놓는 시작 행/열”이다.
      • i, j는 패턴 내부 좌표이며,
        map[sr + i][sc + j]와 scenario[i][j]를 하나씩 비교한다.
      • 하나라도 다르면 ok = false로 바꾸고 그 시작 좌표에 대한 검사는 중단한다.
      • 끝까지 모두 같다면 ok == true 상태로 유지되며, 이 경우 count++로 발견 횟수를 올린다.
    • 결과 출력
      • 각 시나리오마다 구한 count를 한 줄에 하나씩 출력 버퍼에 쌓는다. 
        sb.append(count).append('\n');
      • 모든 시나리오를 처리한 뒤, 마지막에 한 번만 출력한다. 
        System.out.print(sb);

    시간 복잡도

    한 시나리오에 대해:

    • 시작 좌표(sr, sc)의 개수는 (Lm - Li + 1) × (Cm - Ci + 1)개이다.
    • 각 시작 좌표마다 패턴 전체(Li × Ci)를 검사한다.

    따라서 한 시나리오의 시간 복잡도는 대략 O((Lm - Li + 1) × (Cm - Ci + 1) × Li × Ci)이다.
    최악의 경우를 단순화하면 O(Lm × Cm × Li × Ci)라고 볼 수 있다.

    문제에서 주어지는 상한은:

    • 1 ≤ Li, Ci ≤ 12
    • 1 ≤ Lm, Cm ≤ 32이며 항상 Lm ≥ Li, Cm ≥ Ci

    즉, 최악의 경우에도 32 × 32 × 12 × 12 ≈ 150,000 번 정도의 문자 비교만 수행하므로,
    테스트케이스가 여러 개 있어도 충분히 빠르게 동작한다.

    공간 복잡도

    사용되는 주요 메모리는:

    • 패턴 배열 scenario[Li][Ci]
    • 지도 배열 map[Lm][Cm]
    • 그 외 작은 정수 변수들과 StringBuilder

    따라서 한 시나리오당 공간 복잡도는 O(Lm × Cm + Li × Ci)이고,
    입력 상한에서 봐도 최대 32 × 32 + 12 × 12 정도로 매우 작다.

    주의

    • 회전/대칭을 고려하지 않는다:
      • 문제에서 “exact replica”라고 했으므로,
        코드는 패턴을 그대로 겹쳐 비교만 한다.
      • 90도 회전한 모양이나 좌우/상하 반전된 모양은 세지 않는다.
    • 인덱스 범위:
      • 시작 좌표의 범위를 0 ≤ sr ≤ Lm - Li, 0 ≤ sc ≤ Cm - Ci로 두지 않으면
        배열 인덱스가 범위를 벗어나기 쉽다.
    • 불일치 시 빠른 중단:
      • 내부 루프에서 한 칸이라도 다르면 ok = false로 만들고 바로 빠져나와
        불필요한 비교를 줄인다.
    • 여러 시나리오 처리:
      • 첫 줄의 scenarios만큼 반복하면서,
        각 시나리오에 대해 독립적으로 패턴/지도를 읽고 카운트한 뒤 출력한다.

    포인트 요약

    • 패턴(비밀 기지 이미지)을 scenario 2차원 배열에, 지도를 map 2차원 배열에 저장한다.
    • 패턴이 들어갈 수 있는 모든 시작 위치(sr, sc)를 브루트포스로 돌면서,
      scenario와 map이 완전히 같은지 확인한다.
    • 한 칸이라도 다르면 즉시 해당 위치를 버리고, 끝까지 같으면 등장 횟수 count를 증가시킨다.
    • 회전/대칭은 고려할 필요 없고, 오직 “그대로의 모양”만 세면 된다.
    • 입력 크기가 작아서 단순한 2차원 패턴 매칭으로도 시간/메모리 모두 여유 있게 통과된다.
