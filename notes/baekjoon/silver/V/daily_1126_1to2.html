    문제 5092 — Air Old Zeeland

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Air Old Zeeland (BOJ 5092)는
    항공사에서 포인트로 구매 가능한 상품 목록과 고객들의 주문 정보를 바탕으로,
    각 고객이 실제로 구매한 상품 포인트 합을 계산하고,
    배송 지연 때문에 원하는 상품을 다 받지 못한 불만족 고객 수까지 세는 시뮬레이션 문제다.

    상품마다 다음 정보가 주어진다.

    • 상품 이름 (공백 없음, 최대 20자)
    • 필요 포인트 (≤ 1000)
    • 배송 지연 일수 (≤ 100)

    각 고객에 대해서는:

    • 고객 번호
    • 구매를 희망하는 상품 개수 P
    • 기다릴 수 있는 최대 일수 M
    • 이후 P개의 줄에 각 상품 이름

    고객이 원하는 각 상품에 대해:

    • 해당 상품의 배송 일수가 M일 이하라면 → 구매 성공, 포인트 합산
    • M일보다 크면 → 그 상품은 사지 못함, 불만족 표시 대상

    출력은 고객마다:

    • 고객번호 총포인트 형식으로 출력
    • 만약 원한 상품 중 하나라도 배송일 초과로 못 산 경우, 포인트 뒤에 *를 붙인다.
      • 예: 1860 0 *
    • 시나리오 마지막 줄에는
      Number of discontented customers is: X 형태로
      별표가 붙은(불만족) 고객 수를 출력한다.

    전략 개요

    • 각 상품 이름으로 바로 가격/배송일을 찾기 위해
      상품 이름 → (포인트, 배송일) 구조를 HashMap에 저장한다.
    • 각 고객에 대해:
      • 주문하려는 상품 P개를 순회하면서
        • 상품 정보를 맵에서 조회
        • 배송일 ≤ M이면 포인트 합계에 더함
        • 배송일 > M이면 불만족 플래그를 true로 설정
      • 포인트 합과 불만족 여부(별표)를 출력 버퍼에 기록한다.
    • 시나리오별로 불만족 고객 수를 카운트하여 마지막에 한 줄로 출력한다.
    • 상품 수 N이 0인 시나리오를 만나면 전체 입력을 종료한다.

    핵심: 코드가 하는 일

    • 시나리오 반복 처리 및 종료 조건
      • 한 줄씩 읽어 N (상품 수)을 구한다. 
        String line;
while ((line = br.readLine()) != null) {
    line = line.trim();

    int N = Integer.parseInt(line);
    if (N == 0) break;
    ...
}
      • N == 0이면 더 이상 시나리오가 없으므로 break로 전체 종료.
    • 상품 정보 입력 및 맵 구성
      • 상품 수 N만큼 반복해서 상품 정보를 읽는다. 
        Map<String, ProductInfo> map = new HashMap<>();
StringTokenizer st;

for (int i = 0; i < N; i++) {
    st = new StringTokenizer(br.readLine());

    String name = st.nextToken();
    int point = Integer.parseInt(st.nextToken());
    int delivery = Integer.parseInt(st.nextToken());

    map.put(name, new ProductInfo(point, delivery));
}
      • ProductInfo는 포인트와 배송일을 담는 간단한 내부 클래스다. 
        private static class ProductInfo {
    int point, delivery;

    public ProductInfo(int point, int delivery) {
        this.point = point;
        this.delivery = delivery;
    }
}
    • 고객 처리 및 불만족 카운트
      • 시나리오별 불만족 고객 수를 저장하는 cnt를 0으로 초기화한다.
      • 고객 수 C를 읽고, 각 고객을 차례대로 처리한다. 
        int cnt = 0;
int C = Integer.parseInt(br.readLine());
for (int i = 0; i < C; i++) {
    st = new StringTokenizer(br.readLine());
    int clientNum = Integer.parseInt(st.nextToken());
    int P = Integer.parseInt(st.nextToken());
    int M = Integer.parseInt(st.nextToken());

    int sum = 0;
    boolean missed = false;
    for (int j = 0; j < P; j++) {
        String name = br.readLine();
        ProductInfo info = map.get(name);

        if (info.delivery <= M) {
            sum += info.point;
        } else {
            missed = true;
        }
    }

    sb.append(clientNum).append(" ").append(sum);
    if (missed){
        sb.append(" *");
        cnt++;
    }
    sb.append('\n');
}
      • 각 고객에 대해:
        • sum: 실제로 구매 가능한 상품 포인트 합
        • missed: 배송 지연으로 구매하지 못한 상품이 하나라도 있었는지 여부
      • missed == true이면 고객 출력 줄에 *를 붙이고,
        불만족 고객 수 cnt를 1 증가시킨다.
    • 시나리오 요약 출력
      • 모든 고객 처리가 끝나면, 시나리오 마지막 줄에 
        sb.append("Number of discontented customers is: ").append(cnt).append('\n');
        를 출력 버퍼에 추가한다.
      • 루프가 모두 끝난 뒤 System.out.print(sb);로 전체 결과를 한 번에 출력한다.

    시간 복잡도

    한 시나리오에 대해:

    • 상품 정보 입력 및 맵 구성: 상품 수를 N이라 하면 O(N)
    • 고객 처리:
      • 고객 수를 C, 각 고객이 주문하는 상품 수를 Pi라고 하면,
        전체 주문 라인 수는 ∑ Pi이다.
      • 각 주문에서 맵 조회는 O(1)이므로
        전체는 O(∑ Pi)

    따라서 시나리오 전체 시간 복잡도는 대략
    O(N + ∑ Pi)이고,
    문제 제한(N ≤ 50, C ≤ 500, P ≤ N)에서는 매우 여유롭다.

    여러 시나리오가 있어도, 각 시나리오는 독립적으로 처리되므로
    전체 입력에 대해서도 충분히 빠르게 동작한다.

    공간 복잡도

    한 시나리오에서 사용하는 주요 추가 메모리는:

    • 상품 맵 map: 최대 N ≤ 50개의 상품 정보 저장
    • 출력 누적용 StringBuilder sb (전체 시나리오 공통)
    • 몇 개의 정수/불리언 변수

    상품 수에 비례하는 맵만 추가로 필요하므로
    공간 복잡도는 O(N)이다.
    상한이 매우 작으므로 메모리 사용량은 거의 미미한 수준이다.

    주의

    • 배송일 기준 비교:
      • info.delivery <= M일 때만 포인트를 합산해야 한다.
      • <와 <=를 혼동하면 결과가 달라지므로 주의.
    • 부분 구매 가능:
      • 어떤 고객이 3개를 원했는데 2개만 기한 내 배송 가능하면,
        2개 포인트만 합산하고, 나머지 1개 때문에 불만족(*)으로 처리한다.
    • 불만족 고객 수 카운트:
      • 각 고객마다 missed == true인 경우에만 cnt++ 해야 한다.
      • 한 고객이 여러 상품을 놓쳤더라도 고객 수는 1명만 증가한다.
    • 입력 형식:
      • 문제에서 상품 이름은 항상 사전에 주어진 목록 안에서만 사용되므로,
        map.get(name)가 null이 되는 케이스는 없다.

    포인트 요약

    • 상품 정보를 이름 → (포인트, 배송일) 맵에 저장한다.
    • 각 고객에 대해, 원하는 상품별로 배송일 ≤ 최대 대기일 M인지 확인하고 가능한 것만 포인트 합산한다.
    • 배송일 초과로 못 산 상품이 하나라도 있으면 그 고객은 불만족 처리(* 출력).
    • 시나리오별로 불만족 고객 수를 세어 마지막에 Number of discontented customers is: X 형태로 출력한다.
    • 맵 조회는 O(1), 전체 알고리즘은 O(N + ∑ P)로 충분히 빠르다.

    문제 5093 — Letter Replacement

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Letter Replacement (BOJ 5093)는
    단어 안에서 중복해서 등장하는 알파벳들을
    등장 순서에 따라 특수 기호로 치환하는 문자열 처리 문제다.

    규칙은 다음과 같다.

    • 대소문자는 구분하지 않는다. (예: R과 r는 같은 문자)
    • 단어를 왼쪽에서 오른쪽으로 보면서, 이미 등장한 적 있는 글자가 다시 등장하면 “반복 문자”로 본다.
    • 반복 문자를 처음으로 만나게 되는 글자 종류 순서대로 다음 기호를 배정한다.
      • 1번째 반복 문자 종류 → *
      • 2번째 반복 문자 종류 → ?
      • 3번째 반복 문자 종류 → /
      • 4번째 반복 문자 종류 → +
      • 5번째 반복 문자 종류 → !
    • 같은 글자가 여러 번 반복되면, 그 글자의 2번째 이후 등장부터는
      항상 처음 배정된 기호로 치환된다.
    • 단어의 첫 등장 글자는 원래 글자를 그대로 출력한다. (대소문자 유지)

    예를 들어 Reindeer의 경우:

    • R / r, e 등이 대소문자 구분 없이 같은 글자로 취급된다.
    • 왼쪽에서 오른쪽으로 보며 반복 여부를 체크하면, e가 먼저 반복되고 그 다음 r이 반복된다.
    • 따라서 반복된 e는 *, 반복된 r은 ?로 치환되어
      최종 결과가 Reind**?가 된다.

    입력은 여러 줄의 단어로 이루어지며, 마지막 줄의 #는 종료를 의미하므로 처리하지 않는다.

    전략 개요

    • 각 단어에 대해 다음과 같은 상태를 유지한다.
      • count[26]: 각 알파벳이 지금까지 몇 번 등장했는지 (대소문자 무시)
      • symbol[26]: 해당 알파벳이 반복 문자로 판정되었을 때 배정된 기호
    • 반복 문자 종류에 할당할 기호 풀: 
      char[] pool = {'*', '?', '/', '+', '!'};
    • poolIdx를 0부터 시작해서, 새로운 반복 문자 종류가 나올 때마다 pool[poolIdx]를 배정하고 poolIdx를 1 증가시킨다.
    • 문자를 하나씩 보면서:
      • 해당 문자의 알파벳 인덱스를 구해(toLowerCase 후 'a' 기준)
      • count[idx]++로 등장 횟수를 올린다.
      • count[idx] == 1이면 첫 등장 → 원래 문자(대소문자 그대로) 출력.
      • count[idx] >= 2이면 반복 문자:
        • 이 문자에 아직 기호가 배정되지 않았다면 → symbol[idx]에 새 기호 배정.
        • 이미 기호가 있다면 → 그 기호를 그대로 사용.
    • 각 줄 처리 후 개행 문자를 추가하고, 다음 단어를 처리한다.

    핵심: 코드가 하는 일

    • 입력 반복 및 종료 조건
      • 한 줄씩 읽어 #가 나오면 종료한다. 
        String line;
while ((line = br.readLine()) != null) {
    line = line.trim();
    if (line.equals("#")) break;

    ...
}
    • 알파벳 카운트 및 기호 배열 초기화
      • 각 단어마다 새로 초기화한다. 
        int[] count = new int[26];
char[] symbol = new char[26];

char[] pool = {'*', '?', '/', '+', '!'};
int poolIdx = 0;
    • 문자별 처리
      • 단어의 각 문자를 순회하며, 대소문자를 무시한 인덱스를 구해 카운트한다. 
        for (int i = 0; i < line.length(); i++) {
    char origin = line.charAt(i);
    char c = Character.toLowerCase(origin);
    int idx = c - 'a';

    count[idx]++;

    if (count[idx] == 1) {
        // 첫 등장: 원래 문자 출력
        sb.append(origin);
    } else {
        // 반복 등장
        if (symbol[idx] == 0) {
            symbol[idx] = pool[poolIdx++];
        }

        sb.append(symbol[idx]);
    }
}
      • count[idx] == 1이면 첫 등장 → 원래 문자를 그대로 결과에 붙인다.
      • count[idx] > 1이면 이미 등장했던 문자:
        • symbol[idx]가 비어 있다면(0) 새 기호 배정.
        • 그 후 symbol[idx]에 저장된 기호를 출력한다.
    • 줄 마무리 및 출력
      • 단어 처리가 끝나면 개행 문자를 추가한다. 
        sb.append('\n');
      • 모든 단어 처리가 끝난 뒤 System.out.print(sb);로 한 번에 출력한다.

    시간 복잡도

    각 단어의 길이를 L이라고 하면:

    • 문자 처리 루프는 길이 L에 비례 → O(L)
    • L ≤ 10이므로, 매우 작은 상수 시간에 가깝다.

    전체 입력에 대해 여러 단어가 주어져도,
    단어 수를 W라 하면 전체 시간 복잡도는 O(∑ L) ≤ O(10W)로
    충분히 빠르게 동작한다.

    공간 복잡도

    한 단어 처리에 사용하는 추가 메모리는:

    • count[26] 정수 배열
    • symbol[26] 문자 배열
    • 기호 풀 pool[5]와 인덱스 변수 하나

    모두 알파벳 개수(26)에 비례하는 상수 크기 배열이므로
    공간 복잡도는 O(1)이다.

    주의

    • 대소문자 처리:
      • 반복 여부 판정과 기호 배정은 소문자로 변환한 기준으로 처리해야 한다.
      • 하지만 첫 등장 문자를 출력할 때는 원래 문자(origin)를 그대로 써서
        대소문자를 유지한다.
    • 기호 배정 순서:
      • 기호는 “반복 문자 종류가 처음 등장한 순서”에 따라 배정된다.
      • 즉, 단어를 왼쪽에서 오른쪽으로 스캔하면서
        처음으로 count[idx] == 2가 되는 시점에 그 문자에게 기호를 배정한다고 보면 된다.
    • 반복 횟수 제한:
      • 문제에서 한 단어의 반복 문자 종류는 최대 5개라고 보장하므로,
        기호 풀 {'*','?','/','+','!'}로 충분하다.
    • 종료 입력 처리:
      • 마지막 줄의 #는 단순 종료 마커이므로
        이 줄은 결과를 생성하지 않고 바로 루프를 빠져나와야 한다.

    포인트 요약

    • 각 단어에 대해 알파벳별 등장 횟수를 세며 왼쪽에서 오른쪽으로 스캔한다.
    • 첫 등장 문자는 그대로 출력하고, 두 번째 이후 등장부터는 기호로 치환한다.
    • 반복 문자 종류가 처음 등장할 때만 기호 풀에서 순서대로 하나씩 배정한다.
    • 대소문자 구분 없이 반복을 판단하지만, 출력할 때는 원래 문자의 대소문자를 유지한다.
    • 배열 크기가 작고, 단어 길이도 최대 10이라 구현과 성능 모두 단순한 문제다.
