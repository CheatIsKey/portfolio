    문제 4903 — Relax! It’s just a game

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Relax! It’s just a game (BOJ 4903)는
    전반전 스코어 A : B가 주어졌을 때,
    그 스코어가 만들어질 수 있는 골 순서의 경우의 수를 구하고,
    이 경우의 수가 A + B와 같은지 여부를 판단하는 문제다.

    전반전 스코어가 A : B라면, 전반전에 총 A + B골이 나왔다.
    각 골이 어느 팀의 골인지에 따라 골 순서를 나열하면, 가능한 모든 시나리오의 수는

    • 총 골 수 = A + B
    • 그 중 B골(또는 A골)의 위치를 선택 → 이항 계수 C(A+B, B)

    따라서 이 문제에서 말하는 “number of possibilities”는
    comb(A+B, B) (또는 comb(A+B, A))가 된다.
    이 값을 A + B와 비교해서 같으면 =, 다르면 !=로 출력한다.

    출력 형식은 정확히

    • A+B=C 또는 A+B!=C 형태
    • C는 단순 합 A + B

    입력의 마지막은 -1 -1이며, 이 줄은 처리하지 않고 종료한다.

    전략 개요

    • 각 줄에서 A, B를 읽는다.
      • A == -1 && B == -1이면 입력 종료.
    • sum = A + B를 계산한다.
    • 골 순서 경우의 수 ways = C(sum, B)를 계산한다.
      • C(n, k)는 조합 nCk를 의미한다.
    • sum == ways이면 A+B=sum 형식으로 출력.
    • 그렇지 않으면 A+B!=sum 형식으로 출력.

    조합 계산 C(n, k)는 직접 팩토리얼을 쓰면 오버플로/비효율 문제가 생길 수 있으므로,
    곱셈과 나눗셈을 섞어가며 점진적으로 계산한다.

    핵심: 코드가 하는 일

    • 입력 처리 및 종료 조건 
      String line;
while ((line = br.readLine()) != null) {
    line = line.trim();

    StringTokenizer st = new StringTokenizer(line);
    long scoreOne = Long.parseLong(st.nextToken());
    long scoreTwo = Long.parseLong(st.nextToken());
    if (scoreOne == -1 && scoreTwo == -1) break;

    long sum = scoreOne + scoreTwo;
    long ways = comb(sum, scoreTwo);

    sb.append(scoreOne).append("+").append(scoreTwo);

    if (sum == ways) {
        sb.append("=").append(sum).append('\n');
    } else {
        sb.append("!=").append(sum).append('\n');
    }
}
      • scoreOne → A, scoreTwo → B.
      • sum은 합 A + B.
      • ways는 C(A+B, B) (골 순서 경우의 수).
      • sum == ways인지 비교하여 출력 문자열 형태 결정.
    • 조합 함수 구현: comb(sum, num) 
      private static long comb(long sum, long num) {
    // C(sum, num)에서 num과 sum-num 중 더 작은 쪽 선택
    num = Math.min(num, sum - num);

    long res = 1;
    for (int i = 1; i <= num; i++) {
        res = res * (sum - i + 1) / i;
    }

    return res;
}
      • C(n, k)는 C(n, n-k)와 같으므로,
        k 대신 min(k, n-k)를 사용해 곱셈 범위를 줄인다.
      • 반복문에서는 수식을 
        res = res * (n - i + 1) / i;
        형태로 누적 계산한다.
        • 수학적으로는 C(n, k) = (n × (n-1) × ... × (n-k+1)) / (k!)를 전개한 것.
      • 문제 조건에서 A, B ≤ 10이므로 n = A+B ≤ 20이고,
        최대값 C(20, 10) = 184756 정도라 long에 안전하게 들어간다.
    • 출력 형식
      • 각 테스트케이스마다 한 줄 출력:
        • 예: 2+1=3, 3+2!=5 등
      • A, B, C 사이에 다른 공백은 넣지 않고,
        등호/부등호 앞뒤 형식도 문제 예시 그대로 맞춘다.

    시간 복잡도

    한 테스트케이스에 대해:

    • comb(sum, num) 계산은 num = min(B, sum-B)에 대해 O(num) 반복.
    • 스코어 범위상 sum ≤ 20이므로, num ≤ 10 → 사실상 상수 시간.

    따라서 전체 입력에 대해 시간 복잡도는
    테스트케이스 수를 T라 할 때 O(T) 수준으로 매우 가볍다.

    공간 복잡도

    추가로 사용하는 공간은:

    • 몇 개의 정수/long 변수(scoreOne, scoreTwo, sum, ways, res 등)
    • 출력 누적용 StringBuilder sb

    입력 크기와 무관하게 일정하므로, 공간 복잡도는 O(1)이다 (출력 문자열은 어차피 필요).

    주의

    • -1 -1 종료 처리:
      • -1 -1은 테스트케이스가 아니므로, 이 줄은 출력하지 않아야 한다.
    • 조합 계산 시 정수 나눗셈 순서:
      • res = res * (sum - i + 1) / i; 순서로 계산하면,
        매 단계에서 나눗셈이 정확히 떨어지도록 되어 있다.
      • 곱셈과 나눗셈의 순서를 잘못 섞으면 오버플로나 오차가 날 수 있어 주의해야 한다.
    • 대칭성 활용:
      • C(n, k) == C(n, n-k)를 사용해 min(k, n-k)만큼만 반복하면,
        불필요한 반복을 줄이고 곱셈 범위도 줄일 수 있다.
    • 스코어 범위:
      • 문제 조건상 각 팀 골 수는 0~10이므로, sum은 최대 20이다.
      • 이 범위 안에서는 long 타입으로 모든 이항 계수를 안전하게 표현할 수 있다.

    포인트 요약

    • 전반전 스코어 A:B가 만들어질 수 있는 골 순서 경우의 수는 C(A+B, B)이다.
    • 이 값이 A+B와 같은지 비교해서 = 또는 !=를 출력한다.
    • 조합은 팩토리얼 대신 반복 곱셈/나눗셈 방식으로 안전하게 계산한다.
    • 입력은 -1 -1이 나올 때까지 처리하고, 마지막 줄은 테스트에 포함하지 않는다.
    • 출력 형식은 A+B=C 또는 A+B!=C 형태를 정확히 지켜야 한다.

    문제 5134 — Grocery Shopping

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Grocery Shopping (BOJ 5134)는
    마트의 재고 정보와 장바구니 목록이 주어졌을 때,
    멤버십 카드(클럽 카드)를 사용해서 절약한 총 금액을 계산하는 문제다.

    각 상품에 대해:

    • ni: 현재 재고 수량
    • pi: 일반 가격
    • ci: 클럽 카드 가격
    • si: 상품 이름 (대소문자, 공백 포함 가능)

    장바구니에는:

    • mj: 구매를 희망하는 수량
    • sj: 상품 이름

    이름은 대소문자를 구분하지 않고 비교한다.
    장바구니에 있는 상품이 마트 재고에 없으면 그 상품은 무시한다.

    어떤 상품에 대해 실제로 x개를 구매했다면,
    이 상품에서 절약한 금액은

    • x × (일반 가격 - 클럽 가격)

    재고가 부족하면:

    • 실제 구매 수량은 min(요청 수량, 재고)가 된다.
    • 재고는 그만큼 줄어들고, 절약 금액도 그 수량에 대해서만 계산한다.

    각 데이터셋마다 절약한 금액을 모두 합산해
    $x.xx 형식으로 출력한다.

    전략 개요

    • 각 데이터셋에 대해:
      1. 마트 재고 정보 N개를 읽어 상품 이름(소문자 처리) → (재고, 일반가, 클럽가)를 Map에 저장한다.
      2. 장바구니 항목 M개에 대해:
        • 요청 수량과 이름을 읽고, 이름을 소문자로 변환해 맵에서 상품 정보를 찾는다.
        • 없는 상품이면 건너뛴다.
        • 재고와 요청 수량 중 작은 값만큼만 실제 구매하고,
          그 수량에 대해 (일반가 - 클럽가)만큼 절약 금액을 누적한다.
        • 사용한 수량만큼 재고를 차감한다.
      3. 누적된 절약 금액(센트 기준)을 달러로 바꾸어 $%.2f 형식으로 출력한다.

    핵심: 코드가 하는 일

    • 데이터셋 개수 K 읽기 
      String line;
while ((line = br.readLine()) != null) {
    line = line.trim();

    int K = Integer.parseInt(line);

    for (int caseNum = 1; caseNum <= K; caseNum++) {
        ...
    }
}
      • 첫 줄에서 K를 읽고, 각 데이터셋에 대해 반복한다.
    • N, M 읽기 
      StringTokenizer st = new StringTokenizer(br.readLine());
int N = Integer.parseInt(st.nextToken());
int M = Integer.parseInt(st.nextToken());
      • N: 마트 상품 개수
      • M: 장바구니 상품 개수
    • 마트 재고 정보 파싱 및 저장 
      Map<String, ProductInfo> store = new HashMap<>();

for (int j = 0; j < N; j++) {
    line = br.readLine();
    st = new StringTokenizer(line);

    int ni = Integer.parseInt(st.nextToken());
    String pi = st.nextToken();
    String ci = st.nextToken();

    StringBuilder nameBuilder = new StringBuilder();
    while (st.hasMoreTokens()) {
        if (nameBuilder.length() > 0) nameBuilder.append(' ');
        nameBuilder.append(st.nextToken());
    }

    String name = nameBuilder.toString().toLowerCase();
    int price = parsePrice(pi);
    int club = parsePrice(ci);

    store.put(name, new ProductInfo(ni, price, club));
}
      • 각 줄 형식: ni pi ci si...
      • 앞의 3개 토큰은 수량과 가격 2개, 나머지 모든 토큰을 이어 붙여 상품 이름으로 만든다.
      • 이름은 소문자로 변환해 맵의 키로 사용한다.
      • parsePrice를 사용해 "$6.00" 같은 문자열을 정수 센트(예: 600)로 변환한다.
    • 장바구니 처리 및 절약 금액 누적 
      long saving = 0L;

for (int j = 0; j < M; j++) {
    line = br.readLine();
    st = new StringTokenizer(line);

    int mj = Integer.parseInt(st.nextToken());

    StringBuilder nameBuilder = new StringBuilder();
    while (st.hasMoreTokens()) {
        if (nameBuilder.length() > 0) nameBuilder.append(' ');
        nameBuilder.append(st.nextToken());
    }

    String name = nameBuilder.toString().toLowerCase();
    ProductInfo info = store.get(name);

    if (info == null) continue;

    int actual = info.changeStock(mj);
    saving += (long) actual * (info.price - info.club);
}
      • 각 장바구니 줄: mj sj...
      • 마찬가지로 이름을 토큰 이어 붙여 만든 뒤 소문자 변환.
      • 마트에 없는 상품이면 info == null이므로 그냥 무시.
      • changeStock(mj)는 실제 구매 가능한 수량을 반환하고 재고를 줄인다. 
        int changeStock(int request) {
    int accept = Math.min(request, stock);
    stock -= accept;
    return accept;
}
      • 절약 금액은 
        saving += (long) actual * (info.price - info.club);
        로 센트 단위로 누적된다.
    • 결과 출력 
      sb.append("Data Set ").append(caseNum).append(":\n");
sb.append(String.format("$%.2f\n\n", saving / 100.0));
      • 각 데이터셋마다
        • Data Set x: 한 줄 출력
        • 다음 줄에 절약 금액을 $와 소수 둘째 자리까지 출력
        • 그 다음 줄에 빈 줄 추가
      • saving은 센트 단위이므로 / 100.0으로 달러로 변환한다.

    가격 파싱 로직

    • 문자열 가격을 센트 단위 정수로 변환 
      private static int parsePrice(String s) {
    s = s.trim();
    if (s.charAt(0) == '$') {
        s = s.substring(1);
    }
    String[] parts = s.split("\\.");

    int dollars = Integer.parseInt(parts[0]);
    int cents = 0;
    if (parts.length > 1) {
        String centStr = parts[1];

        if (centStr.length() == 1) centStr = centStr + "0";
        cents = Integer.parseInt(centStr);
    }

    return dollars * 100 + cents;
}
      • "$6.00"에서 '$'를 제거 후 "6.00"을 "6", "00"으로 나눈다.
      • 달러와 센트를 정수로 바꾼 다음 dollars * 100 + cents로 센트 값으로 만들고 반환한다.
      • 센트 부분이 1자리일 경우(예: "4.5") 뒤에 0을 붙여 "50"처럼 맞춘다.

    시간 복잡도

    각 데이터셋에 대해:

    • 마트 재고 N개를 읽어 맵에 저장: O(N)
    • 장바구니 항목 M개를 처리:
      • 각 항목당 맵 조회는 O(1), 이름 파싱도 입력 길이에 비례하지만 길이가 한정적이다.
      • 전체적으로 O(M).

    따라서 한 데이터셋당 시간 복잡도는 대략 O(N + M)이고,
    1 ≤ N, M ≤ 100이라 전체 입력에서도 충분히 빠르다.

    공간 복잡도

    주요 추가 공간 사용은:

    • 마트 재고 맵 store:
      • 최대 N개의 ProductInfo를 저장 → O(N)
    • 출력 누적용 StringBuilder sb
    • 그 외에는 단순 변수와 보조 문자열만 사용

    전체적으로 추가 공간 복잡도는 O(N) 수준이다.

    주의

    • 이름 비교 시 대소문자 무시:
      • 재고, 장바구니 모두 소문자로 변환해서 비교해야 한다.
      • 예제처럼 "TV dinner"와 "tv dinner"는 같은 상품으로 인식해야 한다.
    • 재고보다 많이 주문하는 경우:
      • min(요청 수량, 재고)만큼만 구매해야 하며, 재고는 그만큼 줄여야 한다.
      • 여러 번 같은 상품을 장바구니에서 요청할 수도 있으므로 재고 관리가 중요하다.
    • 존재하지 않는 상품:
      • 장바구니에 있지만 재고 목록에 없는 경우(info == null)는 무시한다.
      • 절약 금액에도 영향을 주지 않는다.
    • 출력 형식:
      • 각 데이터셋 사이에 빈 줄 한 줄을 꼭 넣어야 한다.
      • 금액은 항상 $ 기호와 소수 둘째 자리까지 출력해야 한다 (예: $3.00).

    포인트 요약

    • 상품 이름을 키로 하는 맵에 재고, 일반 가격, 클럽 가격을 저장한다.
    • 장바구니를 돌면서 재고 내에서만 실제 구매 수량을 정하고,
      그 수량에 대해 (일반가 - 클럽가) 만큼 절약 금액을 누적한다.
    • 이름 비교는 대소문자를 무시해야 하므로 모두 소문자로 통일한다.
    • 가격은 센트 단위의 정수로 처리하고, 출력 시에만 달러/소수점 형식으로 변환한다.
    • 각 데이터셋마다 Data Set x:와 절약 금액, 그리고 빈 줄을 정확히 출력해야 한다.
