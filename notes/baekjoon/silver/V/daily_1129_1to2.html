    문제 5144 — Ecosystem

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Ecosystem (BOJ 5144) 문제는
    먹이사슬 구조와 각 종의 초기 개체 수, 그리고 각 종이 생존을 위해 필요로 하는 먹이를 바탕으로,
    일정 시간이 지난 뒤 각 종이 몇 마리 살아남는지를 계산하는 시뮬레이션 문제다.

    모델은 다음과 같다.

    • 종은 총 n개, 번호는 1 ~ n (코드에서는 0 ~ n-1 인덱스로 사용).
    • 각 종은 자신보다 번호가 작은 종만 먹을 수 있다.
    • 각 종 i에 대해:
      • 현재 개체 수: init[i]
      • 먹이 요구량: need[i][j] (종 i의 개체 하나가 생존하기 위해 종 j를 몇 마리 먹어야 하는지)
    • 먹이 요구량은 균형 잡힌 식단이다.
      • 예를 들어, need[seal][herring] = 1, need[seal][squid] = 1이면
        청어 1마리 + 오징어 1마리를 정확히 먹어야 생존할 수 있고,
        청어 2마리 + 오징어 0마리 같은 식으로 대체할 수 없다.
    • 어떤 먹이 종이 이미 0마리라면, 그 먹이를 필요로 하는 상위 종의 개체는 모두 굶어 죽는다.
    • 처리는 항상 종 번호 오름차순으로 진행한다.
      • 즉, 번호가 작은 종이 먼저 먹이를 소비하고, 그 이후 남은 개체 수를 기반으로 더 높은 종이 생존 가능 개체 수를 계산한다.

    목표는, 주어진 한 단위 시간 동안 이 과정을 한 번 시뮬레이션한 뒤,
    최종적으로 각 종별 생존 개체 수를 구하는 것이다.

    전략 개요

    • 입력 구조
      • 데이터셋 개수 K.
      • 각 데이터셋마다:
        • n: 종의 개수.
        • 각 종 i (1..n)에 대해 한 줄:
          • 첫 정수: init[i] (현재 개체 수)
          • 다음 i-1개 정수: need[i][1..i-1] (먹이 요구량)
    • 시뮬레이션 개요
      • surv[i]: 시뮬레이션 중 현재 남아 있는 종 i의 개체 수.
        • 시작 시점에는 surv[i] = init[i].
      • 종을 번호 순서대로 1번부터 n번까지 처리한다.
        • 종 1은 먹이가 필요 없으므로, 단순히 surv[0] = init[0] 유지.
        • 종 i (i > 1)에 대해:
          • 자신이 필요로 하는 먹이 종들(번호 < i)의 남은 개체 수를 보고,
            실제로 최대 몇 마리까지 생존할 수 있는지 계산한다.
          • 생존이 가능하다고 판단된 개체 수만큼, 먹이 종의 개체 수를 소비한다.
          • 나머지는 굶어 죽었으므로 surv[i]를 생존 개체 수로 덮어쓴다.
    • 생존 가능 개체 수 계산 핵심
      • 종 i가 종 j를 need[i][j]마리씩 먹어야 한다면,
        현재 남은 먹이 surv[j]로부터 최대 
        surv[j] / need[i][j]
        마리의 종 i만 생존할 수 있다.
      • 먹이가 여러 종이라면, 각 먹이 종마다 이런 상한이 생기고,
        실제로는 이 상한들의 최솟값이 종 i가 생존할 수 있는 최대 개체 수가 된다.
      • 이 값을 maxFood로 두고, 실제 생존 개체 수는 
        alive = min(init[i], maxFood)
        로 결정한다.

    핵심: 코드가 하는 일

    • 기본 입력 처리 및 데이터셋 루프 
      String line;
while ((line = br.readLine()) != null) {
    line = line.trim();

    int K = Integer.parseInt(line);

    for (int caseNum = 1; caseNum <= K; caseNum++) {
        int n = Integer.parseInt(br.readLine());

        int[] init = new int[n];
        int[][] need = new int[n][n];

        ...
    }
}
      • 첫 줄에서 K를 읽고, 그만큼 데이터셋을 처리한다.
      • 각 데이터셋마다 n을 읽고, 초기 개체 수와 먹이 요구량을 위한 배열을 만든다.
    • 각 종의 초기 정보 입력 
      for (int i = 0; i < n; i++) {
    StringTokenizer st = new StringTokenizer(br.readLine());
    init[i] = Integer.parseInt(st.nextToken());

    for (int j = 0; j < i; j++) {
        need[i][j] = Integer.parseInt(st.nextToken());
    }
}
      • 종 인덱스는 0부터 사용하지만, 입력은 1번 종부터 순서대로 들어온다고 생각하면 된다.
      • i번째 줄:
        • 첫 토큰: init[i] (현재 개체 수)
        • 다음 i개의 먹이 요구량 중, 실제로는 번호가 더 작은 종만 먹기 때문에
          j < i인 구간만 읽는다.
    • 생존 개체 수 배열 초기화 
      long[] surv = new long[n];
for (int i = 0; i < n; i++) {
    surv[i] = init[i];
}
      • surv[i]는 시뮬레이션 중 해당 종의 현재 개체 수를 나타낸다.
      • 초기에는 모두 init 값으로 시작.
      • long을 사용해 나중에 먹이 소비 계산 시 곱셈으로 인한 오버플로 위험을 줄인다.
    • 종별 생존 계산 
      for (int i = 0; i < n; i++) {
    if (i == 0) continue;

    long maxFood = Long.MAX_VALUE;

    for (int j = 0; j < i; j++) {
        int needCnt = need[i][j];
        if (needCnt == 0) continue;

        if (surv[j] == 0) {
            maxFood = 0;
            break;
        }

        long possible = surv[j] / needCnt;
        if (possible < maxFood) {
            maxFood = possible;
        }
    }

    long alive;
    if (maxFood == Long.MAX_VALUE) {
        alive = init[i];
    } else {
        alive = Math.min((long) init[i], maxFood);
    }

    for (int j = 0; j < i; j++) {
        if (need[i][j] > 0) {
            surv[j] -= alive * need[i][j];
        }
    }

    surv[i] = alive;
}
      • i == 0인 경우(1번 종)는 먹이 요구량이 없으므로 그대로 둔다.
      • maxFood 계산:
        • 처음에 Long.MAX_VALUE로 두어, “아직 먹이 요구가 없다”는 의미로 사용.
        • 각 먹이 종 j < i에 대해:
          • needCnt = need[i][j]가 0이면, 해당 종은 먹지 않으므로 건너뜀.
          • 필요량이 있고, surv[j] == 0이면:
            • 필수 먹이 종이 이미 멸종했으므로, 종 i 전체가 굶어 죽음.
            • maxFood = 0으로 설정하고 바로 루프 종료.
          • 그렇지 않다면, possible = surv[j] / needCnt가
            종 i가 최대 몇 마리까지 생존 가능한지를 제한하는 상한이 된다.
          • maxFood는 모든 먹이 종에 대한 상한의 최솟값으로 갱신.
      • alive 결정:
        • 만약 maxFood == Long.MAX_VALUE라면, 종 i는 먹이가 전혀 필요 없다는 뜻이므로
          alive = init[i] (초기 개체 수 그대로 생존).
        • 그 외에는: 
          alive = Math.min((long) init[i], maxFood);
          • 초기 개체 수보다 더 많이 살 수는 없고, 먹이 상한을 넘어서도 안 되므로 두 값 중 작은 것을 생존 개체 수로 선택.
      • 먹이 소비:
        • 생존한 개체 수 alive만큼, 각 먹이 종 j에서 
          alive * need[i][j]
          개체를 먹는다.
        • 즉, 
          surv[j] -= alive * need[i][j];
          로 먹힌 만큼 줄어든다.
      • 마지막에 surv[i] = alive;로 종 i의 최종 생존 개체 수를 갱신한다.
    • 출력 
      sb.append("Data Set ").append(caseNum).append(":\n");
for (int i = 0; i < n; i++) {
    sb.append(surv[i]).append('\n');
}
sb.append('\n');
      • 각 데이터셋마다:
        • Data Set x: 헤더 출력
        • 각 종의 최종 생존 개체 수를 한 줄에 하나씩 출력
        • 마지막에 빈 줄 한 줄 추가

    시간 복잡도

    각 데이터셋에 대해:

    • 입력으로 need를 채우는 부분:
      • 각 종 i에 대해 최대 i-1개의 값을 읽으므로, 전체적으로 O(n^2).
    • 시뮬레이션 루프:
      • 바깥 루프 i = 0..n-1에 대해, 안쪽 루프 j = 0..i-1을 돈다.
      • 이 역시 전체적으로 O(n^2)이다.

    따라서 한 데이터셋당 시간 복잡도는 O(n^2)이며,
    n ≤ 100이므로 충분히 빠르게 동작한다.

    공간 복잡도

    • init[n], surv[n], need[n][n]를 사용한다.
    • 따라서 공간 복잡도는 O(n^2)이다.

    주의

    • 처리 순서:
      • 항상 종 번호가 작은 종부터 먹고, 남은 개체 수를 바탕으로 더 큰 종을 계산해야 한다.
      • 코드에서도 for (int i = 0; i < n; i++)로 순서대로 진행한다.
    • 필수 먹이 부족 시 전멸 처리:
      • 필요량이 need[i][j] > 0인데 surv[j] == 0이면, 해당 먹이 종이 이미 없는 상태다.
      • 이 경우 종 i는 전부 굶어 죽어야 하므로, maxFood = 0으로 설정하고 이후 계산을 생략한다.
    • 먹이 요구가 전혀 없는 종:
      • 모든 need[i][j] == 0인 경우, maxFood는 Long.MAX_VALUE 그대로 유지된다.
      • 이때는 생존 개체 수를 init[i]로 그대로 유지한다.
    • 오버플로 방지:
      • surv와 alive 계산에 long을 사용하는 것은
        alive * need[i][j] 곱셈에서 오버플로를 피하기 위함이다.

    포인트 요약

    • 종 번호가 낮은 것부터 순서대로, 먹이 요구량과 남은 먹이를 바탕으로 생존 가능 개체 수를 계산한다.
    • 각 먹이 종마다 가능한 상한 surv[j] / need[i][j]을 구하고, 그 최솟값이 상위 종이 생존할 수 있는 최대 개체 수가 된다.
    • 필수 먹이 종 중 하나라도 이미 멸종했다면, 해당 종은 전부 굶어 죽는다.
    • 생존한 개체 수만큼 먹이를 소비하여, 이후에 등장하는 더 상위 종들이 사용할 수 있는 먹이량을 줄인다.
    • 이 과정을 모든 종에 대해 반복한 뒤, 각 종의 최종 surv[i]를 그대로 출력하면 된다.

    문제 5210 — Campus Buildings

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Campus Buildings (BOJ 5210) 문제는
    여러 개의 건물 이름과 하나의 건물 약어(abbreviation)가 주어졌을 때,
    이 약어가 어떤 건물 이름의 부분 수열로 등장하는지를 판별해서
    조건을 만족하는 건물 이름들을 모두 출력하는 문제다.

    약어와 건물 이름 매칭 규칙은 다음과 같다.

    • 약어의 문자들이 주어진 순서대로 건물 이름 안에 등장해야 한다.
      • 예: SAL은
        • SALvatori
        • studENt aeRoSpace laBoratory (대소문자, 공백 무시)
        • univerSity of southern cALifornia
        에는 매칭된다.
      • 하지만 angeles에는 S, A, L이 순서대로 나타나지 않으므로 매칭되지 않는다.
    • 대소문자는 구분하지 않는다. 즉, 'S'와 's'는 같은 문자로 본다.
    • 약어의 각 문자는 건물 이름에서 서로 다른 위치에 매칭되어야 한다.
      • 한 위치의 문자를 약어의 두 문자에 재사용할 수는 없다 (일반적인 부분 수열 조건).

    각 데이터셋마다, 약어와 매칭되는 모든 건물 이름을
    입력으로 주어진 순서 그대로 출력하면 된다.

    전략 개요

    • 각 데이터셋에 대해:
      1. 건물 개수 n을 읽는다.
      2. 건물 이름 n개를 그대로 배열에 저장한다 (대소문자·공백 포함, 원본 출력용).
      3. 약어 코드 문자열을 읽고, 소문자로 변환해 code에 저장한다.
      4. 각 건물 이름에 대해:
        • 건물 이름을 소문자로 변환한 문자열 lower를 만든다.
        • code의 각 문자들을 순서대로, lower 안에서 앞에서부터 차례대로 찾아간다.
        • 모든 문자를 순서대로 찾을 수 있으면 매칭 성공 → 원래 건물 이름을 출력.
        • 하나라도 찾지 못하면 매칭 실패 → 출력하지 않음.
    • 데이터셋마다
      • Data Set x: 헤더를 출력한 뒤,
      • 매칭된 건물 이름들을 순서대로 출력한다.

    핵심: 코드가 하는 일

    • 데이터셋 개수 K 및 반복 
      String line;
while ((line = br.readLine()) != null) {
    line = line.trim();

    int K = Integer.parseInt(line);

    for (int caseNum = 1; caseNum <= K; caseNum++) {
        int n = Integer.parseInt(br.readLine());

        String[] origin = new String[n];
        for (int i = 0; i < n; i++) {
            origin[i] = br.readLine();
        }

        sb.append("Data Set ").append(caseNum).append(":\n");

        String code = br.readLine().toLowerCase();

        ...
    }
}
      • 첫 줄에서 데이터셋 개수 K를 읽는다.
      • 각 데이터셋마다:
        • n: 건물 이름의 개수.
        • origin[i]: 입력 그대로의 건물 이름(나중에 그대로 출력).
        • code: 약어를 소문자로 변환한 문자열.
      • 각 데이터셋 시작 시 "Data Set x:"를 출력 버퍼에 추가한다.
    • 부분 수열(약어 매칭) 검사 
      for (String s : origin) {
    String lower = s.toLowerCase();
    int pos = -1;
    boolean ok = true;

    for (int i = 0; i < code.length(); i++) {
        char c = code.charAt(i);
        pos = lower.indexOf(c, pos + 1);

        if (pos == -1) {
            ok = false;
            break;
        }
    }

    if (ok) {
        sb.append(s).append("\n");
    }
}
      • lower = s.toLowerCase():
        • 현재 건물 이름을 모두 소문자로 만들어, 대소문자 구분 없이 비교할 준비를 한다.
      • pos:
        • 이전까지 찾은 약어 문자 위치를 기억하는 인덱스.
        • 처음에는 -1로 두고, 항상 pos + 1 이후에서만 다음 문자를 찾는다.
      • for (int i = 0; i < code.length(); i++):
        • 약어의 각 문자 code[i]를 순서대로 처리.
        • pos = lower.indexOf(c, pos + 1);:
          • lower 문자열에서 이전 위치 이후에 있는 문자 c의 첫 등장 위치를 찾는다.
          • 찾지 못하면 -1을 반환한다.
        • 만약 pos == -1이면:
          • 해당 문자를 순서대로 찾을 수 없다는 뜻 → 이 건물 이름은 약어와 매칭되지 않으므로 ok = false 후 루프 종료.
      • 전체 문자를 다 찾았으면 ok = true 그대로 유지되므로:
        • 원래 이름인 s를 그대로 출력 버퍼에 추가한다.
    • 최종 출력 
      System.out.print(sb);
      • 모든 데이터셋을 처리한 뒤, 한 번에 출력한다.
      • 각 데이터셋 사이에 별도의 공백 줄을 출력하라는 조건은 없으므로,
        예제 출력처럼 바로 이어서 Data Set x: 줄이 나오게 된다.

    시간 복잡도

    한 데이터셋에서:

    • 건물 개수를 n, 각 건물 이름의 평균 길이를 L, 약어 길이를 m이라고 하면,
      • 각 건물에 대해 code.length()번 반복하면서, 내부에서 indexOf로 남은 부분을 검색한다.
      • 한 건물 이름당 시간 복잡도는 대략 O(L)~O(L × m) 사이지만,
        indexOf를 누적해서 사용하는 구조라 실제로는 전체 문자열을 한 번쯤 훑는 수준이다.
    • 따라서 전체는 대략 O(n × L) 수준으로 볼 수 있다.

    문제 제한에서 n ≤ 100, 각 이름 길이도 크게 길지 않으므로
    충분히 빠르게 동작한다.

    공간 복잡도

    • 건물 이름 배열 origin[n]과 문자열 몇 개(lower, code)만 추가로 사용한다.
    • 따라서 한 데이터셋에 대한 추가 공간은 O(n × L) (입력 저장 공간) 정도이고,
      알고리즘 자체의 추가 공간은 O(1)에 가깝다.

    주의

    • 대소문자 무시:
      • 건물 이름과 약어 모두 소문자로 변환해서 비교해야 한다.
      • 코드에서는 s.toLowerCase(), code.toLowerCase()로 처리한다.
    • 부분 수열(subsequence) 매칭:
      • 단순히 각 문자 존재 여부만 확인하는 것이 아니라,
        이전 위치 이후에서만 다음 문자를 찾으면서 순서를 유지해야 한다.
      • 이 부분이 indexOf(c, pos + 1)로 구현되어 있다.
    • 출력 형식:
      • 각 데이터셋마다 반드시 
        Data Set x:
        를 먼저 출력해야 한다.
      • 그 뒤에 매칭되는 건물 이름들을 그대로 출력하고,
        매칭되는 것이 하나도 없으면 예제의 Data Set 2:처럼 헤더만 출력된다.

    포인트 요약

    • 약어가 건물 이름의 부분 수열로 등장하는지(순서 유지, 건너뛰기는 허용)를 검사하는 문제다.
    • 대소문자를 무시하기 위해 모두 소문자로 변환한 뒤 비교한다.
    • 각 건물 이름에 대해 indexOf와 시작 위치 인덱스를 활용해, 약어의 문자를 앞에서부터 차례대로 찾는다.
    • 모든 문자를 순서대로 찾으면 해당 건물 이름을 결과에 포함하고, 하나라도 실패하면 버린다.
    • 각 데이터셋마다 Data Set x: 헤더 뒤에 매칭된 건물 이름들을 입력 순서대로 출력한다.
