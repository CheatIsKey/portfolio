    문제 5263 — samba

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    samba (BOJ 5263) 문제는
    각 삼바 학교의 ID가 적힌 무용수들의 목록이 주어졌을 때,
    모든 학교는 자교 무용수들을 각 행에 정확히 k명씩 배치해야 하지만
    단 한 학교만 이 조건을 만족하지 못했다고 할 때,
    그 예외 학교의 ID를 찾는 문제다.

    조건을 수식으로 정리하면:

    • 같은 학교의 무용수 수를 cnt(ID)라고 할 때,
      • 정상인 학교는 모두 cnt(ID) % k == 0을 만족한다.
      • 단 하나의 학교만 cnt(ID) % k != 0 이다.
    • 입력으로는 무용수 수 n, 한 줄에 배치해야 하는 인원 k,
      그리고 길이 n의 ID 목록이 주어진다.
    • 우리는 이 중에서 개수가 k로 나누어떨어지지 않는 ID를 찾아 출력하면 된다.

    핵심은 각 ID가 몇 번 나왔는지 센 뒤, 그 개수의 나머지를 체크하는 것이다.

    전략 개요

    • 입력
      • 첫 줄: n (무용수 수), k (한 줄 인원 수).
      • 다음 n줄: 각 줄마다 무용수 한 명의 schoolId (정수).
    • 아이디어
      • 각 학교 ID별로 등장 횟수를 해시맵으로 센다.
        • map[id] = 해당 ID의 무용수 수
      • 모든 입력을 다 센 후, map을 순회하면서
        • cnt % k != 0인 ID를 찾는다.
        • 문제에서 오직 하나만 존재한다고 했으므로, 처음 찾는 ID가 곧 답이다.
    • 주의할 점
      • n이 최대 1,000,000이므로, 입력 읽기와 카운팅은 O(n)으로 처리해야 한다.
      • ID 범위가 0 ~ 1,000,000,000이기 때문에, 단순 배열로 카운트할 수 없고
        HashMap<Integer, Integer> 같은 맵 구조를 사용해야 한다.
      • 문제 보장 상 하나의 ID만 조건에서 벗어나므로, 맵을 한 번만 순회해도 된다.

    핵심: 코드가 하는 일

    • 입력 루프 및 첫 줄 파싱 
      String line;
      while ((line = br.readLine()) != null) {
          line = line.trim();
      
          StringTokenizer st = new StringTokenizer(line);
          int n = Integer.parseInt(st.nextToken());
          int k = Integer.parseInt(st.nextToken());
      • EOF까지 여러 테스트 케이스가 있을 수 있는 입력 형식이라, 바깥을 while ((line = br.readLine()) != null)로 감싼 구조다.
      • 각 테스트 케이스의 첫 줄에서 n, k를 읽는다.
    • ID별 개수 세기 
          Map<Integer, Integer> map = new HashMap<>();
          for (int i = 0; i < n; i++) {
              int schoolId = Integer.parseInt(br.readLine());
              map.put(schoolId, map.getOrDefault(schoolId, 0) + 1);
          }
      • map의 key: 학교 ID (정수)
      • map의 value: 해당 ID의 등장 횟수
      • 각 무용수 ID를 한 줄씩 읽어오면서, getOrDefault로 기존 값에 1을 더해 저장한다.
      • 결과적으로 학교별 무용수 수를 모두 카운트하게 된다.
    • 조건 위반 학교 ID 찾기 
          for (int id : map.keySet()) {
              if (map.get(id) % k != 0) {
                  sb.append(id).append('\n');
                  break;
              }
          }
      • map.keySet()으로 모든 학교 ID를 순회한다.
      • 각 ID에 대해:
        • map.get(id)가 해당 학교의 총 인원 수.
        • map.get(id) % k != 0이면,
          • 해당 학교 인원 수가 k의 배수가 아니다 → 지정된 행에 딱 맞게 나누어 떨어지지 않는다.
          • 즉, 이 학교가 바로 조건을 만족하지 못한 예외 학교다.
      • 문제 보장상 이런 학교는 단 하나뿐이므로,
        첫 번째로 발견되는 ID를 출력 버퍼에 담고 break로 루프를 종료한다.
    • 최종 출력 
      System.out.print(sb);
      • 모든 테스트 케이스에서 찾은 ID들을 누적해 한 번에 출력한다.
      • 예제 입력의 경우:
        • k = 2이고, 각 ID별 개수는
          • 123: 6명 → 6 % 2 == 0 (정상)
          • 1678: 2명 → 2 % 2 == 0 (정상)
          • 43: 3명 → 3 % 2 == 1 (예외)
        • 따라서 출력은 43이 된다.

    시간 복잡도

    • ID를 읽으면서 HashMap에 카운트하는 부분:
      • 무용수 수 n에 대해, 각 삽입/갱신이 평균 O(1)이므로 전체 O(n).
    • 맵을 순회하며 조건을 체크하는 부분:
      • 서로 다른 학교 수를 m이라 하면, O(m).
      • 최대 m ≤ n이므로 전체적으로는 여전히 O(n).

    따라서 한 테스트 케이스의 시간 복잡도는 O(n)이며,
    n ≤ 1,000,000에서도 충분히 빠르게 동작한다.

    공간 복잡도

    • HashMap<Integer, Integer>에 서로 다른 학교 ID만큼 저장한다.
    • 서로 다른 학교 수를 m이라 할 때, 공간 복잡도는 O(m)이며, 최악의 경우 O(n)까지 갈 수 있다.
    • 추가로 사용하는 변수들은 상수 개이므로, 전체 공간 복잡도는 O(n) 수준이다.

    주의

    • 정렬이 필요 없다:
      • 정답은 “딱 하나뿐인 예외 학교 ID”이고,
        그 ID만 정확히 찾아 출력하면 되므로, 별도의 정렬 과정이 필요 없다.
      • HashMap의 순회 순서는 일정하지 않지만, 어차피 예외 ID가 하나뿐이므로
        어떤 순서로 순회하더라도 첫 번째로 발견되는 그 ID가 정답이다.
    • 메모리와 입출력:
      • n이 1,000,000까지 가능하므로, BufferedReader와 StringBuilder를 이용해
        입출력을 최적화한 점이 중요하다.
      • 단순 Scanner + System.out.println 조합은 시간 초과 위험이 있다.
    • 여러 테스트케이스 처리:
      • 코드는 EOF까지 반복해서 n, k를 읽는 구조라, 입력에 여러 케이스가 들어와도 모두 처리할 수 있다.
      • 한 줄마다 하나의 정답 ID를 출력 버퍼에 추가한다.

    포인트 요약

    • 각 학교 ID별로 무용수 수를 HashMap으로 센다.
    • cnt(ID) % k != 0인 ID가 곧 조건을 만족하지 못한 삼바 학교다.
    • 문제에서 예외 학교는 정확히 하나라고 보장하므로, 첫 번째로 발견되는 ID를 출력하면 된다.
    • 시간 복잡도는 O(n), 공간 복잡도는 O(n) (서로 다른 ID 수에 비례)로, 대용량 입력에도 적합한 풀이다.

    문제 5294 — Pirates On Parade

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Pirates On Parade (BOJ 5294) 문제는
    해적들의 이름과 키가 주어졌을 때, 해적들을 키 순으로 정렬한 뒤
    키 차이가 최대 2인 해적끼리만 앞에서부터 순서대로 짝을 만들어 출력하는 문제다.

    규칙을 정리하면:

    • 모든 해적은 이름과 키(정수, 인치 단위)를 가진다.
    • 해적들은 두 명씩 짝을 지어 줄 세운다.
      • 전체 줄은 가장 작은 키부터 가장 큰 키까지 오름차순이다.
      • 각 짝 내부에서도 더 키가 작은 해적 이름이 먼저 와야 한다.
    • 같은 짝에 속한 두 해적의 키 차이는 최대 2여야 한다.
      • 차이가 2를 초과하면 그 둘은 짝이 될 수 없다.
    • 정렬된 순서에서 앞에서부터 해적을 하나씩 보면서 짝을 만들고, 적절한 파트너를 찾지 못한 해적은 “plank”로 보내져 출력에서 제외된다.

    출력은 성공적으로 짝을 이룬 해적들만이며,
    각 줄에는 “키가 작은 해적 이름, 키가 큰 해적 이름” 순으로 출력한다.
    짝을 이루지 못한 해적은 출력하지 않는다.

    전략 개요

    • 모든 해적 정보를 먼저 입력받아 리스트에 저장한다.
    • 리스트를 키(height) 기준 오름차순으로 정렬한다.
    • 정렬된 리스트를 왼쪽에서 오른쪽으로 한 번 훑으면서:
      • 현재 인덱스 idx에서
        • p1 = list[idx], p2 = list[idx+1]를 후보로 잡는다.
        • 두 해적 간 키 차이가 2 이하이면 → 둘을 짝지어 출력하고 idx += 2.
        • 키 차이가 2보다 크면 → p1은 아무와도 짝을 못 만난 것이므로 버리고, idx += 1만 증가.
      • 마지막에 혼자 남는 해적(즉, idx + 1 == list.size()인 경우)은 자동으로 짝을 못 만난 것으로 처리된다.
    • 이 방식은 “가장 키가 작은 해적부터 가능한 한 앞에서 짝을 만들고, 안 되면 버린다”는 문제 설명과 그대로 일치하는 그리디 전략이다.

    핵심: 코드가 하는 일

    • 입력 및 해적 리스트 구성 
      BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
      StringBuilder sb = new StringBuilder();
      List<People> list = new ArrayList<>();
      
      String line;
      while ((line = br.readLine()) != null) {
          line = line.trim();
      
          StringTokenizer st = new StringTokenizer(line);
          String name = st.nextToken();
          int height = Integer.parseInt(st.nextToken());
      
          list.add(new People(name, height));
      }
      • EOF까지 한 줄씩 읽는다.
      • 각 줄은 이름, 키 형식이므로, 토크나이징해서
        • name: 문자열
        • height: 정수
      • 이를 People 객체로 만들어 리스트에 추가한다.
      • 입력 개수에 대한 별도의 수가 주어지지 않기 때문에, EOF까지 모두 읽는 구조가 맞다.
    • 키 기준 정렬 
      list.sort((a, b) -> Integer.compare(a.height, b.height));
      • height 기준으로 오름차순 정렬한다.
      • 이제 list.get(0)가 가장 키가 작은 해적, 그 다음이 두 번째로 작은 해적 … 식이 된다.
      • 문제에서 요구한 “shortest to tallest” 조건을 만족시키는 정렬이다.
    • 그리디하게 짝 만들기 
      int idx = 0;
      while (idx + 1 < list.size()) {
          People p1 = list.get(idx);
          People p2 = list.get(idx + 1);
      
          if (p2.height - p1.height <= 2) {
              sb.append(p1.name).append(" ").append(p2.name).append('\n');
              idx += 2;
          } else {
              idx += 1;
          }
      }
      • idx + 1 < list.size():
        • 현재 해적 p1과 그 바로 다음 해적 p2가 둘 다 존재해야 짝을 검사할 수 있으므로,
          인덱스가 리스트 끝을 넘지 않게 조건을 잡는다.
      • p1 = list.get(idx), p2 = list.get(idx + 1):
        • 정렬된 상태에서 항상 더 작은 키를 가진 해적이 p1이므로,
          나중에 이름을 출력할 때도 p1.name → p2.name 순으로 출력하면 된다.
      • if (p2.height - p1.height <= 2):
        • 키 차이가 2 이하이면, 문제에서 허용하는 한 쌍이므로 짝을 만든다.
        • 출력 버퍼에 p1.name p2.name을 추가하고, idx += 2로 두 해적을 모두 소비한다.
      • 그 외의 경우:
        • p2.height - p1.height > 2라면, 현재 가장 작은 해적 p1은
          그보다 큰 어떤 해적과도 키 차이가 최소 3 이상이라는 의미가 된다.
          • 키가 더 큰 해적들과의 차이는 더 커질 뿐이므로, p1은 아무와도 짝을 이룰 수 없다.
        • 따라서 p1은 “plank”로 보내고, idx += 1로 다음 해적으로 넘어간다.
      • 마지막에 해적이 한 명만 남는 경우(idx + 1 == list.size())도 자연스럽게 짝을 못 만들고 루프 종료 → 출력되지 않는다.
    • 출력 
      System.out.print(sb);
      • 모든 가능한 짝을 StringBuilder에 쌓아 두었다가, 한 번에 출력한다.
      • 예제 1:
        • 입력: Sam(60), Dan(52), Bob(54), Art(61)
        • 키 기준 정렬: Dan(52), Bob(54), Sam(60), Art(61)
        • 짝:
          • (52, 54) → 차이 2 → Dan Bob
          • (60, 61) → 차이 1 → Sam Art
        • 출력: 문제 예제와 동일.

    시간 복잡도

    • 입력으로 해적 수를 n이라고 하면:
      • 리스트에 해적 추가: O(n)
      • 키 기준 정렬: O(n log n)
      • 정렬된 리스트를 한 번 훑으면서 짝 만들기: O(n)

    전체 시간 복잡도는 O(n log n)이다.
    정렬 비용이 지배적이고, 그 이후의 짝 만들기 과정은 선형 시간에 끝난다.

    공간 복잡도

    • 해적 정보를 담는 리스트 list에 n개의 People 객체를 저장한다.
    • 추가로 사용하는 것은 StringBuilder와 몇 개의 지역 변수 뿐이다.

    따라서 공간 복잡도는 O(n)이다.

    주의

    • 입력 종료 조건:
      • 입력에 해적 수 n이 따로 주어지지 않으므로, EOF까지 읽는 구조(while ((line = br.readLine()) != null))가 필요하다.
    • 정렬 후 그리디:
      • 항상 “가장 키가 작은 해적부터” 짝을 시도해야 하므로,
        정렬 후 앞에서부터 그리디하게 처리하는 방식이 문제 조건과 정확히 맞는다.
      • 키 차이가 2보다 크면, 그 해적은 뒤에 있는 어떤 해적과도 짝을 이룰 수 없다는 점을 이용해 바로 버린다.
    • 오름차순 출력:
      • 정렬된 리스트에서 p1, p2를 순서대로 출력하면
        전체적으로도 “shortest to tallest” 순서를 그대로 유지하게 된다.

    포인트 요약

    • 입력된 모든 해적을 (이름, 키) 쌍으로 리스트에 저장한다.
    • 키 기준 오름차순으로 정렬해 전체 줄을 구성한다.
    • 정렬된 리스트를 앞에서부터 보면서,
      • 연속한 두 해적의 키 차이가 2 이하이면 → 짝으로 출력하고 둘 다 소비한다.
      • 2를 초과하면 → 현재 해적은 누구와도 짝을 못 이룬다고 보고 버린다.
    • 짝을 이루지 못한 해적은 출력하지 않는다 (walk the plank).
    • 정렬 + 한 번의 선형 스캔으로 O(n log n)에 해결 가능한 깔끔한 그리디 문제다.
