    문제 5345 — PLU Count

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    PLU Count (BOJ 5345) 문제는
    하나의 문자열 안에서 대소문자를 구분하지 않고 P → L → U 순서로 등장하는 패턴이
    얼마나 많이, 서로 겹치지 않게(non-interleaved) 나오는지를 세는 문제다.

    조건을 정리하면:

    • P, L, U는 이 순서대로만 등장해야 한다.
    • 문자 사이에 다른 문자가 껴도 상관없지만,
      • 한 번 P, L, U를 묶어서 하나의 PLU를 만들면, 그 문자는 다른 PLU와 공유할 수 없다.
      • 즉, 서로 PLU 묶음이 겹치거나 교차(interleave)되면 안 된다.
    • 대소문자는 구분하지 않는다. (p, P, l, L, u, U 모두 합쳐서 같은 글자 취급)

    예를 들어, 문자열 "pppxLLLxuuu"를 보자.

    • p, p, p → 여러 개의 P 후보가 있지만 결국 하나의 P만 사용된다.
    • L, L, L → 마찬가지로 하나의 L만 사용된다.
    • u, u, u → 그 중 하나의 U를 사용해 PLU 1개를 만든다.
    • 사용하고 남은 P, L, U들이 있어도, 이미 한 번 PLU로 묶이는 과정에서 비연속적으로 사용했기 때문에
      새롭게 non-interleaved PLU를 더 만들 수 없다.
    • 결과: 1개.

    전략 개요

    • 각 문자열에 대해 왼쪽부터 한 번만 순회하면서
      현재까지의 진행 상태를 나타내는 간단한 상태값을 유지하면 된다.
    • 상태는 다음 세 단계로 보면 된다:
      1. 아직 P를 찾지 못한 상태
      2. P는 찾았고, L을 기다리는 상태
      3. P와 L을 모두 찾았고, U를 기다리는 상태
    • 코드에서는 이 상태를 정수 플래그로 관리한다.
      • p: P를 이미 찾았는지 여부 (0 or 1)
      • l: L을 이미 찾았는지 여부 (0 or 1, 단 P를 찾은 뒤에만 의미 있음)
    • 문자를 하나씩 보면서:
      • 아직 P를 못 찾았고(p == 0), 현재 문자가 p면 → P를 찾았다고 표시.
      • P는 찾았고(p == 1), L은 아직이고(l == 0), 현재 문자가 l면 → L을 찾았다고 표시.
      • P와 L을 모두 찾은 상태에서(l == 1), 현재 문자가 u면 → PLU 하나 완성 → 개수 증가 후 상태 초기화.
    • 이렇게 하면 항상 가장 앞쪽에서부터 차례대로 PLU를 만들어 사용하므로, 자연스럽게 non-interleaved 조건을 만족시킨다.

    핵심: 코드가 하는 일

    • 입력 처리 
      BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
      StringBuilder sb = new StringBuilder();
      
      String line;
      while ((line = br.readLine()) != null) {
          line = line.trim();
      
          int n = Integer.parseInt(line);
          for (int i = 0; i < n; i++) {
              line = br.readLine().toLowerCase();
              int p = 0, l = 0;
              int cnt = 0;
      • 첫 줄에서 문자열 개수 n을 읽는다.
      • 그 다음 n개의 줄을 각각 하나의 텍스트 문자열로 처리한다.
      • line.toLowerCase()로 모두 소문자로 바꿔서 대소문자 구분 없이 비교할 수 있게 한다.
      • p, l은 상태 플래그, cnt는 완성된 PLU 개수다.
    • 문자 단위 순회 및 상태 갱신 
              for (char c : line.toCharArray()) {
                  if (c == 'p' && p == 0) p++;
                  if (c == 'l' && p == 1 && l == 0) l++;
                  if (c == 'u' && l == 1) {
                      p = 0;
                      l = 0;
                      cnt++;
                  }
              }
      • 각 문자 c를 왼쪽에서 오른쪽으로 한 번씩만 본다.
      • if (c == 'p' && p == 0)
        • 아직 P를 찾지 못한 상태에서 처음 'p'를 만나면 p = 1로 설정한다.
        • 추가로 등장하는 P들은 p == 1이므로 무시된다 (하나의 PLU에서 P는 한 번만 필요).
      • if (c == 'l' && p == 1 && l == 0)
        • 이미 P를 찾았고(p == 1), 아직 L은 못 찾았으며(l == 0) 현재 문자가 'l'이면
          l = 1로 설정해서 L을 찾았다고 표시한다.
      • if (c == 'u' && l == 1)
        • P와 L을 이미 찾은 상태(p == 1, l == 1를 거친 상태)에서 'u'를 만나면
          PLU 하나를 완성한 것이므로:
          • cnt++; → PLU 개수 1 증가
          • p = 0; l = 0; → 상태를 초기화해서 다음 PLU를 새로 찾을 준비를 한다.
        • 이렇게 상태를 리셋하기 때문에 이미 사용한 P, L, U는 다음 PLU에 재사용되지 않는다.
          → non-interleaved 조건을 자연스럽게 만족.
    • 문자열별 결과 저장 
              sb.append(cnt).append('\n');
          }
      }
      
      System.out.print(sb);
      • 각 문자열에 대해 구한 cnt(최대 non-interleaved PLU 개수)를 한 줄에 하나씩 출력 버퍼에 추가한다.
      • 마지막에 System.out.print(sb);로 한 번에 출력한다.

    시간 복잡도

    • 각 문자열의 길이를 L이라고 하면, 해당 문자열에 대해:
      • 각 문자를 딱 한 번씩만 검사 → O(L).
    • 첫 줄에서 주어지는 문자열 개수를 n이라고 하면,
      전체 시간 복잡도는 O(ΣL) (모든 문자열 길이의 합에 비례)이다.
    • 문제에서 각 문자열 길이가 최대 80이므로, 매우 여유로운 수준이다.

    공간 복잡도

    • 문자열을 한 줄씩 읽어 처리하므로, 각 시점에서 메모리에 있는 문자열은 한 개뿐이다.
    • 추가로 사용하는 것은 p, l, cnt 같은 정수 변수와 StringBuilder뿐이다.

    따라서 공간 복잡도는 문자열 길이에 비례하는 O(L), 전체적으로도 매우 작다.

    주의

    • 대소문자 처리:
      • line.toLowerCase()로 미리 소문자로 바꾸지 않으면 'P'와 'p'를 따로 취급하게 되어 오답이 된다.
    • 상태 초기화 시점:
      • 하나의 PLU를 완성했을 때(u를 만난 순간) 바로 p = 0; l = 0;로 초기화해야 한다.
      • 이 때문에 이미 사용한 P, L, U는 다시 쓸 수 없고, PLU들끼리 겹치지 않게 카운팅된다.
    • 여러 P, L, U가 연속으로 나오는 경우:
      • P 상태에서 P가 더 나와도 p == 0 조건 때문에 무시된다.
      • 마찬가지로 L 상태에서도 L이 더 나와도 한 번만 인정된다.
      • 이런 처리 덕분에 연속된 같은 문자들 속에서도 깔끔하게 PLU를 하나씩만 세게 된다.

    포인트 요약

    • 문자열을 소문자로 바꾼 뒤, 왼쪽에서 오른쪽으로 한 번만 훑는다.
    • p, l 두 개의 플래그로 현재까지 P, L이 나왔는지 상태를 관리한다.
    • P를 아직 못 찾았으면 P를 먼저 찾고, 그 다음에 L을 찾고, 그 다음에 U를 찾으면 PLU 하나 완성.
    • U까지 찾으면 PLU 개수를 1 증가시키고, 상태를 초기화해 다음 PLU를 찾는다.
    • 이 방식은 PLU를 앞에서부터 순서대로 소비하므로, non-interleaved 조건을 자동으로 만족한다.

    문제 5347 — LCM

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    LCM (BOJ 5347) 문제는
    여러 테스트 케이스에 대해 두 자연수 a, b가 주어졌을 때
    각 쌍에 대한 최소공배수(LCM, Least Common Multiple)를 구해 출력하는 문제다.

    수학적으로 두 수 a, b에 대해:

    • 최대공약수를 g = gcd(a, b)라고 하면,
    • 최소공배수는
      lcm(a, b) = a × b / gcd(a, b) 로 구할 수 있다.

    입력에서 a, b ≤ 1,000,000 이므로,
    a × b의 최대값은 1012 수준이고, 이는 long 범위 안에 들어간다.
    따라서 int 곱셈은 오버플로우 위험이 있으므로, 곱셈 결과는 long으로 처리하는 것이 안전하다.

    전략 개요

    • 첫 줄에서 테스트 케이스 개수 n을 입력받는다.
    • 각 테스트 케이스마다:
      • 두 자연수 a, b를 읽는다.
      • 유클리드 호제법으로 gcd(a, b)를 구한다.
      • lcm = (long)a * b / gcd(a, b) 공식을 이용해 최소공배수를 구한다.
      • 결과를 출력 버퍼에 쌓는다.
    • 모든 테스트 케이스 처리 후 한 번에 출력한다.

    핵심: 코드가 하는 일

    • 입력 루프 및 테스트 케이스 처리 
      BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
      StringBuilder sb = new StringBuilder();
      
      String line;
      while ((line = br.readLine()) != null) {
          line = line.trim();
      
          int n = Integer.parseInt(line);
          for (int i = 0; i < n; i++) {
              StringTokenizer st = new StringTokenizer(br.readLine());
              int a = Integer.parseInt(st.nextToken());
              int b = Integer.parseInt(st.nextToken());
      
              int g = gcd(a, b);
              long lcm = (long) a * b / g;
      
              sb.append(lcm).append('\n');
          }
      }
      
      System.out.print(sb);
      • while ((line = br.readLine()) != null):
        • EOF까지 읽는 구조지만, 일반적으로 입력에는 한 번만 n이 주어지므로 한 번만 돈다고 보면 된다.
      • int n = Integer.parseInt(line);
        • 테스트 케이스 개수 n을 읽는다.
      • 각 테스트 케이스에 대해:
        • a, b를 읽고,
        • gcd(a, b)를 구한 뒤,
        • lcm = (long) a * b / g로 최소공배수를 계산한다.
      • (long) a * b:
        • a, b는 int지만, 곱셈 결과가 int 범위를 넘을 수 있어 long으로 캐스팅한 뒤 계산한다.
      • 결과는 StringBuilder에 줄 단위로 쌓고, 마지막에 한 번에 출력한다.
    • 유클리드 호제법으로 gcd 구하기 
      private static int gcd(int a, int b) {
          if (b == 0) return a;
      
          return gcd(b, a % b);
      }
      • 유클리드 호제법의 재귀 구현이다.
      • 원리:
        • gcd(a, b) = gcd(b, a % b)
        • b == 0이 되는 순간의 a가 바로 최대공약수.
      • 입력 값이 자연수이기 때문에, 재귀는 반드시 종료된다.

    시간 복잡도

    • 각 테스트 케이스마다:
      • gcd(a, b) 계산: 유클리드 호제법은 O(log min(a, b))
      • 나머지 연산(a * b / g)은 상수 시간

    따라서 테스트 케이스 개수를 n이라고 하면,
    전체 시간 복잡도는 대략 O(n · log M) (M은 입력 숫자의 최대값, 여기선 ≤ 1,000,000)이다.

    공간 복잡도

    • 추가로 사용하는 것은 몇 개의 정수 변수와 StringBuilder뿐이다.
    • 입력 크기에 비례해 StringBuilder가 늘어나지만, 알고리즘 자체의 추가 메모리는 매우 작다.

    따라서 공간 복잡도는 O(1) 수준으로 볼 수 있다(출력 버퍼 제외).

    주의

    • 곱셈 시 오버플로우 방지:
      • a, b를 int로 받더라도 a * b 연산은 long으로 캐스팅해서 수행해야 한다.
      • 그래야 a * b가 1012까지 올라가는 경우에도 안전하다.
    • 공식 적용 순서:
      • lcm = (long) a * b / g처럼, 먼저 곱하고 나중에 나누는 것이 일반적이다.
      • 다른 방식으로는 lcm = a / g * b처럼 나눗셈을 먼저 해서 곱셈을 줄이는 방법도 있지만,
        현재 입력 범위에서는 long 캐스팅만 잘 해줘도 충분히 안전하다.
    • gcd 구현:
      • 재귀 호출이므로, b == 0일 때 a를 바로 반환하는 구조가 꼭 필요하다.
      • 이 부분이 잘못 구현되면 무한 재귀에 빠질 수 있다.

    포인트 요약

    • 두 수 a, b의 최소공배수는 lcm(a, b) = a × b / gcd(a, b)로 구한다.
    • 유클리드 호제법으로 gcd를 O(log min(a,b))에 계산한다.
    • a × b는 long으로 캐스팅해서 오버플로우를 방지한다.
    • 각 테스트 케이스마다 LCM을 구해서 줄마다 출력하면 끝나는 깔끔한 수학/구현 문제다.
