    문제 5690 — King’s Poker

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    King’s Poker (BOJ 5690)는
    3장의 카드로 이루어진 한 손(hand)이 주어졌을 때,
    그 손을 이길 수 있는 세트(set) 또는 페어(pair) 중에서 가장 낮은(가장 약한) 손을 찾아서 출력하는 문제다.

    King’s Poker에서 랭크는 다음과 같다.

    • 랭크: A(1), 2, 3, ..., 10, J(11), Q(12), K(13)
    • 무늬(suit)는 아예 고려하지 않는다.

    손의 유형은 세 가지다.

    • 세트(set): 세 장 모두 같은 랭크 (예: 5,5,5)
    • 페어(pair): 두 장만 같은 랭크이고 나머지 한 장은 다른 랭크 (예: 7,7,3)
    • 노페어(no-pair): 세 장 모두 서로 다른 랭크 (예: 2,5,9)

    손의 강약은 다음 규칙으로 정해진다.

    • 어떤 세트든 모든 페어와 모든 노페어보다 강하다.
    • 어떤 페어든 모든 노페어보다 강하다.
    • 세트 vs 세트: 셋을 이루는 카드 랭크가 더 큰 쪽이 승리.
    • 페어 vs 페어:
      • 먼저 페어를 이루는 랭크를 비교해서 더 큰 쪽이 승리.
      • 페어 랭크가 같다면, 킥커(kicker, 나머지 한 장) 랭크를 비교해서 더 큰 쪽이 승리.
      • 페어 랭크와 킥커 랭크까지 모두 같으면 동점(문제에서는 그런 경우가 안 나오도록 입력이 주어진다).

    우리의 목표는:

    • 입력으로 주어진 손을 엄밀히 이기는(단순 동점은 X) 세트 또는 페어들 중에서,
    • 위의 규칙으로 비교했을 때 가장 약한(순위가 가장 낮은) 손을 찾는 것이다.
    • 그런 손이 존재하지 않으면 *를 출력한다.

    전략 개요

    • 입력으로 받은 세 장의 카드 A, B, C를 정렬해서 cards[0] ≤ cards[1] ≤ cards[2] 형태로 만든다.
    • 정렬된 배열을 이용해 손의 유형을 판별한다.
      • cards[0] == cards[2] → 세 장 모두 같은 값 → 세트.
      • 그 외, cards[0]==cards[1] 또는 cards[1]==cards[2] → 페어.
      • 위 둘이 아니면 → 노페어.
    • 각 경우에 대해, 규칙에 따라 “가장 약한 이기는 세트/페어”를 수학적으로 구성한다:
      • 세트인 경우 → 더 높은 세트(트리플)을 만들 수 있으면 그 중 가장 낮은 것.
      • 페어인 경우 → 우선 같은 페어 랭크에 대해 킥커를 가능한 한 조금만 키워서 이기는 손을 만들고,
        그게 안 되면 페어 랭크를 올려서 가장 약한 이기는 페어를 만든다. 그마저도 불가능하면 세트로 넘어간다.
      • 노페어인 경우 → 어떤 페어든 반드시 이기므로, 가장 약한 페어인 1,1,2를 바로 답으로 사용한다.

    핵심: 코드가 하는 일

    • 입력 처리 및 종료 조건 
      String line;
      while ((line = br.readLine()) != null) {
          StringTokenizer st = new StringTokenizer(line);
      
          int A = Integer.parseInt(st.nextToken());
          int B = Integer.parseInt(st.nextToken());
          int C = Integer.parseInt(st.nextToken());
          if (A == 0 && B == 0 && C == 0) break;
      
          int[] cards = {A, B, C};
          Arrays.sort(cards);
      
          ... // 손 유형에 따른 처리
      }
      • 0 0 0이 입력되면 반복을 종료한다.
      • 각 테스트 케이스마다 cards를 정렬해 이후 로직을 단순하게 만든다.
    • 1) 세트인 경우 (cards[0] == cards[2]) 
      if (cards[0] == cards[2]) {
          int r = cards[0];
      
          if (r == 13) {
              sb.append("*");
          } else {
              int nr = r + 1;
              sb.append(nr).append(" ").append(nr).append(" ").append(nr).append(" ");
          }
      }
      • 현재 손이 r,r,r (세트)일 때, 이를 이기려면 랭크가 더 큰 세트여야만 한다.
      • r < 13이면 가장 약한 이기는 세트는 (r+1),(r+1),(r+1).
      • r == 13이면 이미 가장 강한 세트(KKK)이므로, 이길 수 있는 세트/페어가 없어서 *를 출력한다.
    • 2) 페어인 경우 
      else if (cards[0] == cards[1] || cards[1] == cards[2]) {
          int pair;
          int kick;
      
          if (cards[0] == cards[1]) {
              pair = cards[0];
              kick = cards[2];
          } else {
              pair = cards[1];
              kick = cards[0];
          }
      
          int newKick = -1;
          for (int i = kick + 1; i <= 13; i++) {
              if (i != pair) {
                  newKick = i;
                  break;
              }
          }
      
          if (newKick != -1) {
              int[] answer = {pair, pair, newKick};
              Arrays.sort(answer);
      
              sb.append(answer[0]).append(" ")
                .append(answer[1]).append(" ")
                .append(answer[2]).append(" ");
          } else {
              if (pair < 13) {
                  int newPair = pair + 1;
                  int[] answer = {newPair, newPair, 1};
                  Arrays.sort(answer);
      
                  sb.append(answer[0]).append(" ")
                    .append(answer[1]).append(" ")
                    .append(answer[2]).append(" ");
              } else {
                  sb.append("1 1 1");
              }
          }
      }

      이 부분이 로직의 핵심이다. 현재 손이 페어일 때, 가장 약한 이기는 페어 또는 세트를 찾는 과정이다.

      • 페어와 킥커 추출
        • cards[0] == cards[1]이면 페어는 cards[0], 킥커는 cards[2].
        • 그 외에는 cards[1] == cards[2]이므로 페어는 cards[1], 킥커는 cards[0].
      • 2-1. 같은 페어 랭크에서 킥커만 키워서 이기는 페어 만들기
        • 페어 랭크 pair는 그대로 두고, 킥커 kick보다 큰 카드 중에서 페어 랭크와 겹치지 않는 가장 작은 값을 찾는다. 
          for (int i = kick + 1; i <= 13; i++) {
          if (i != pair) {
              newKick = i;
              break;
          }
      }
        • 이렇게 찾은 newKick이 있다면, {pair, pair, newKick}이 현재 손보다 조금 더 강한(가장 약한) 페어가 된다.
      • 2-2. 킥커를 더 키울 수 없을 때 → 페어 랭크 올리기
        • newKick == -1인 경우는 “현재 킥커보다 더 크면서 페어 랭크와 겹치지 않는 카드”를 찾을 수 없는 경우다.
        • 이 때는 페어 랭크를 하나 키운 새로운 페어를 고려한다. 
          if (pair < 13) {
          int newPair = pair + 1;
          int[] answer = {newPair, newPair, 1};
          Arrays.sort(answer);
          ...
      }
        • 킥커로는 가능한 한 작은 카드(1)를 사용하고, 정렬해서 오름차순으로 출력한다.
      • 2-3. 현재 페어가 K 페어(13,13,x)이고 킥커도 더 못 키울 때
        • pair == 13이면서 newKick == -1인 경우, 더 강한 페어는 존재하지 않는다.
        • 이 경우, 다음으로 강한 손은 세트이므로, 가장 약한 세트인 1,1,1을 정답으로 사용한다. 
          else {
          sb.append("1 1 1");
      }
    • 3) 노페어인 경우 
      else {
          sb.append("1 1 2");
      }
      • 노페어는 어떤 페어에게도 항상 진다.
      • 따라서 노페어를 이길 수 있는 손들 중 가장 약한 손은, 전체 페어 중에서 가장 약한 것인 1,1,2이다.
      • 그래서 노페어인 경우는 항상 1 1 2를 출력하면 된다.

    시간 복잡도

    • 각 테스트 케이스에서 하는 일:
      • 카드 3장을 정렬: O(1) (상수 크기 배열).
      • 페어인 경우, 킥커를 키우는 루프는 최대 13 - 1번 → O(1).
    • 테스트 케이스 수를 T라고 하면 전체는 O(T) 수준이다.

    공간 복잡도

    • 카드 세 장을 저장하는 int[3]와 몇 개의 지역 변수만 사용 → O(1).
    • 출력용 StringBuilder는 출력 길이에 비례하지만, 이는 문제 요구 출력 자체이므로 추가적인 큰 메모리는 사용하지 않는다.

    주의

    • 항상 비기는 손이 아니라 “엄밀히 이기는” 손을 찾아야 한다.
    • 출력은 문제 요구대로 오름차순 정렬된 세 카드를 공백으로 구분해서 출력해야 한다.
      • 페어나 세트를 만들 때도 마지막에 Arrays.sort(answer)로 정렬하는 이유.
    • 13 13 13처럼 가장 강한 세트는 이길 수 있는 손이 없으므로 *를 출력해야 한다.

    포인트 요약

    • 세 장을 정렬해서 세트 / 페어 / 노페어를 간단히 판별한다.
    • 세트면 → 더 높은 세트(r+1,r+1,r+1)가 답, 단 13,13,13이면 *.
    • 페어면 → 동일 페어 랭크에서 킥커를 조금만 키워서 이기는 페어를 먼저 시도하고,
      안 되면 페어 랭크를 올린 페어, 그것도 안 되면 가장 약한 세트 1,1,1을 사용한다.
    • 노페어면 → 어떤 페어든 이기므로 가장 약한 페어 1,1,2가 정답이다.

    문제 5725 — Pole Position

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Pole Position (BOJ 5725)는
    레이스 도중 전광판(폴)에 표시된 현재 순위와
    각 차량이 스타트 그리드에서 몇 칸을 앞/뒤로 이동했는지(P) 정보가 주어졌을 때,
    가능한 시작 그리드(출발 순서)를 복원할 수 있는지를 판별하는 문제다.

    입력 형식은 다음과 같다.

    • 각 테스트 케이스마다 N (차량 수, 2 ≤ N ≤ 1000)이 주어짐.
    • 이후 N줄에 걸쳐, 폴에 표시된 현재 순서대로 차들이 주어진다.
      • 각 줄: C P
      • C는 차량 번호 (1 ≤ C ≤ 10,000, 서로 모두 다름)
      • P는 현재 위치에서 시작 그리드 기준으로 얼마나 이득(+)/손해(-) 봤는지를 나타내는 정수
      • 예: 현재 2위인 차가 P = +2라면, 스타트는 4위(두 칸 앞당김).
    • N = 0이 나오면 입력 종료.

    우리의 목표:

    • 현재 폴의 i번째 줄(1-based)이 현재 i등이라고 보고, 거기에 적힌 C, P를 이용해 스타트 그리드에서의 위치 S를 구한다.
    • 모든 차에 대해 서로 다른 S ∈ [1, N]을 만들 수 있으면 해당 위치에 차량 번호를 배치해 시작 그리드를 출력한다.
    • 만약:
      • 어떤 차의 S가 범위를 벗어나거나,
      • 두 차 이상이 같은 S를 갖게 된다면
      → 모순이므로 -1을 출력한다.

    전략 개요

    • 폴에 표시된 순서를 그대로 현재 순위라고 두고, 줄 번호 i (1 ≤ i ≤ N)를 현재 등수로 사용한다.
    • 각 차에 대해:
      • 현재 등수 = i
      • 이동한 칸 수 = P
      • 스타트 그리드에서의 등수 S = i + P
    • S에 차량 C를 배치하되:
      • S < 1 또는 S > N이면 → 유효하지 않은 시작 위치 → 불가능.
      • start[S]가 이미 채워져 있으면 → 중복 위치 → 불가능.
    • 한 번이라도 불가능 판정이 나면, 나머지 줄은 입력만 소비하고 로직은 건너뛴다.
    • 끝까지 모순이 없다면 start[1..N]이 완성된 시작 그리드가 된다.

    핵심: 코드가 하는 일

    • 입력 루프 + 종료 조건 
      String line;
      while ((line = br.readLine()) != null) {
          int N = Integer.parseInt(line);
          if (N == 0) break;
      
          boolean ok = true;
          int[] start = new int[N + 1];
          ...
      }
      • 각 테스트마다 N을 읽고, N == 0이면 전체 입력 종료.
      • start[1..N] 배열에 스타트 위치별 차량 번호를 채운다 (편의를 위해 1-based).
      • ok는 현재까지 유효한지 여부를 나타내는 플래그.
    • 각 차량 정보 처리 
      for (int i = 1; i <= N; i++) {
          StringTokenizer st = new StringTokenizer(br.readLine());
          int C = Integer.parseInt(st.nextToken());
          int P = Integer.parseInt(st.nextToken());
      
          if (!ok) continue;
      
          int S = i + P;
          if (S < 1 || S > N || start[S] != 0) {
              ok = false;
          } else {
              start[S] = C;
          }
      }
      • 반복 인덱스 i는 현재 폴 상에서의 순위(1등, 2등, ...)를 의미한다.
      • 차량 번호 C와 이동량 P를 읽어,
        스타트 그리드 위치 S = i + P를 계산한다.
      • 유효성 검사:
        • S가 1~N 범위를 벗어나면 → 불가능.
        • start[S] != 0이면 → 이미 다른 차가 그 자리에 있다는 뜻 → 불가능.
      • ok == false 이후에는 그대로 입력만 읽고, 계산은 건너뛴다. (다음 테스트 케이스로 넘어가기 위한 입력 소모)
    • 결과 출력 
      if (ok) {
          for (int i = 1; i <= N; i++) {
              if (i > 1) sb.append(' ');
              sb.append(start[i]);
          }
      } else {
          sb.append(-1);
      }
      sb.append('\n');
      • ok == true이면:
        • start[1]부터 start[N]까지를 공백으로 구분해 출력 → 복원된 스타트 그리드.
      • ok == false이면:
        • 어떤 방식으로도 일관된 스타트 그리드를 만들 수 없으므로 -1 출력.

    시간 복잡도

    • 각 테스트 케이스에서:
      • 차량 N개에 대해 한 번씩만 처리 → O(N).
      • 각 차량마다 하는 일은 배열 접근과 상수번 검사뿐.
    • 테스트 케이스를 여러 개 처리해도, 전체는 입력 크기에 선형이다.

    공간 복잡도

    • 각 테스트 케이스마다 start[0..N] 하나만 사용 → O(N).
    • 추가로 사용하는 변수들은 상수 개수이므로 전체 메모리 사용량은 매우 작다.

    주의

    • 현재 순위의 인덱스 i를 잘못 이해하면 안 된다.
      • 입력에서 N줄은 이미 현재 1등, 2등, ... 순서대로 주어지는 것으로 가정한다.
      • 그래서 S = i + P가 된다 (앞으로 v칸 당겼으면 시작 순위는 더 뒤의 번호).
    • 불가능 판정(ok = false) 이후에도 남은 줄은 반드시 입력만 읽어줘야 다음 테스트 케이스를 정상 처리할 수 있다.
      • 코드에서 if (!ok) continue;로 로직만 건너뛰고 읽기는 유지한다.
    • start 배열을 1-based로 사용하므로, 크기를 N + 1로 잡는 것에 주의한다.

    포인트 요약

    • 폴에 주어진 순서는 이미 현재 순위이며, 줄 번호 i를 그대로 사용한다.
    • 각 차량에 대해 시작 순위는 S = i + P로 계산한다.
    • S가 범위를 벗어나거나, 한 위치에 두 번 이상 배치하려 하면 즉시 불가능 → -1.
    • 끝까지 모순이 없으면 start[1..N]이 복원된 스타트 그리드이므로 그대로 출력한다.
