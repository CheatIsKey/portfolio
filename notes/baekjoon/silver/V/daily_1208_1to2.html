    문제 5747 — Odd or Even

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Odd or Even (BOJ 5747) 문제는
    존(John)과 메리(Mary)가 여러 번 한 게임의 기록을 가지고,
    메리가 최소 몇 판은 “무조건 이겼다고 확신할 수 있는지”를 구하는 문제다.

    게임 규칙은 다음과 같다.

    • 존은 항상 홀수(odds)를, 메리는 항상 짝수(evens)를 선택한다.
    • 각 게임에서 두 사람은 0~5 사이의 손가락 수를 낸다.
    • 두 수의 합이 짝수면 메리 승, 합이 홀수면 존 승이다.

    하지만 지금은 게임 순서가 섞여서
    “메리가 각 게임에서 낸 손가락 수 목록”과 “존이 각 게임에서 낸 손가락 수 목록”만 주어진다.
    어떤 게임에서 어떤 값이 짝이었는지는 모른다.

    우리는 가능한 모든 매칭(순서 재배치) 중에서
    메리의 승리 수가 가장 적게 되는 경우를 가정했을 때,
    그 최소 승리 횟수를 구해야 한다. (메리 입장에서 최악, 존 입장에서 최선인 경우)

    핵심 아이디어 (짝/홀 개수 세기)

    승패는 합의 짝/홀 여부만으로 결정된다.

    • 메리의 손가락 수: 0~5 중 어떤 값이든, 짝수/홀수만 중요하다.
    • 존의 손가락 수도 마찬가지.

    각 게임에서 두 수의 합이 짝수가 되는 경우는:

    • 짝수 + 짝수
    • 홀수 + 홀수

    즉, 두 사람이 동일한 parity(짝/홀)일 때 메리가 이기고,
    서로 다른 parity일 때 존이 이긴다.

    존이 메리의 승리를 최소로 만들고 싶다면, 가능한 한 많이 “서로 다른 짝/홀”끼리 매칭해서 본인이 이겨야 한다. 그 최대 개수를 구하고, 전체 게임 수에서 빼면
    메리가 최소한으로 이긴 판 수 = N - (존이 최대한 이길 수 있는 판 수)가 된다.

    • 메리 쪽 짝수 개수: maryEven
    • 메리 쪽 홀수 개수: maryOdd
    • 존 쪽 짝수 개수: johnEven
    • 존 쪽 홀수 개수: johnOdd

    서로 다른 parity로 매칭할 수 있는 최대 개수(= 존이 이기는 판 수, mis)는:

    • 메리 짝수 vs 존 홀수 조합: 최대 min(maryEven, johnOdd)개
    • 메리 홀수 vs 존 짝수 조합: 최대 min(maryOdd, johnEven)개

    따라서

    mis = min(maryEven, johnOdd) + min(maryOdd, johnEven)
    win = N - mis ← 메리가 최소한으로 이긴 판 수

    코드가 하는 일

    • 입력 루프
      • 각 테스트케이스마다 먼저 N을 읽고, N == 0이면 종료.
    • 메리 카드의 짝/홀 개수 세기 
      int maryEven = 0;
      int maryOdd = 0;
      
      StringTokenizer st = new StringTokenizer(br.readLine());
      for (int j = 0; j < N; j++) {
          int num = Integer.parseInt(st.nextToken());
          if (num % 2 == 0) maryEven++;
          else maryOdd++;
      }
      • 두 번째 줄에서 메리가 낸 손가락 수 N개를 읽고,
      • 짝수면 maryEven++, 홀수면 maryOdd++.
    • 존 카드의 짝/홀 개수 세기 
      int johnEven = 0;
      int johnOdd = 0;
      
      st = new StringTokenizer(br.readLine());
      for (int j = 0; j < N; j++) {
          int num = Integer.parseInt(st.nextToken());
          if (num % 2 == 0) johnEven++;
          else johnOdd++;
      }
    • 존이 이길 수 있는 최대 판 수 계산 
      int mis = Math.min(maryEven, johnOdd)
              + Math.min(maryOdd, johnEven);
      • 짝수-홀수, 홀수-짝수 매칭을 최대한 많이 만들어,
        존의 승리(메리의 패배)를 최대로 만든다.
    • 메리가 확실히 이겼다고 말할 수 있는 최소 판 수 
      int win = N - mis;
      sb.append(win).append('\n');
      • 남은 게임은 모두 짝+짝 또는 홀+홀 매칭이라 메리가 이기는 판.
      • 어떤 식으로 매칭해도 최소 win판은 반드시 메리가 이길 수밖에 없다.

    예시

    예제 1: 

    N = 3
      메리: 1 0 4
      존  : 3 1 2
    • 메리: 짝수 2개(0, 4), 홀수 1개(1)
    • 존: 짝수 1개(2), 홀수 2개(3, 1)

    짝/홀 매칭 최대 개수:

    • min(maryEven=2, johnOdd=2) = 2
    • min(maryOdd=1, johnEven=1) = 1

    따라서

    mis = 2 + 1 = 3 (존이 최대 3판까지 이길 수 있다)
    win = N - mis = 3 - 3 = 0

    어떤 식으로 짝을 맞춰도 메리가 확실히 이겼다고 말할 수 있는 판은 0판이다.
    → 출력: 0

    시간 복잡도

    • 각 테스트케이스에서
      • 메리 카드 N개, 존 카드 N개를 한 번씩만 순회한다.
    • 따라서 한 케이스당 시간 복잡도는 O(N)이다.
    • N ≤ 100이므로 전체 입력에 대해서도 충분히 빠르다.

    공간 복잡도

    • 짝/홀 개수와 몇 개의 정수 변수만 사용한다.
    • 추가 배열을 따로 만들지 않으므로, 추가 공간 복잡도는 O(1)이다.
    • 출력 누적을 위한 StringBuilder는 출력 길이에 비례하지만, 이는 필수 출력 공간이다.

    주의

    • 실제 “어떤 게임에서 누가 이겼는지”를 맞추는 것이 아니라,
      모든 가능한 재배치 중 메리 승리 수의 최솟값만 구하면 된다.
    • 승패는 오직 짝/홀 여부에 의해서만 결정되므로, 구체적인 숫자(0~5)가 아니라 짝수/홀수 개수만 세는 것이 핵심이다.
    • 존이 이길 수 있는 판 수를 최대화하면, 그만큼 메리의 승리는 최소가 된다.

    포인트 요약

    • 각 게임의 승패는 두 수의 합이 짝수냐 홀수냐에만 달려 있다.
    • 카드 순서가 섞였으므로, 메리·존 각각의 짝수/홀수 개수만 중요하다.
    • 존의 승리(서로 다른 짝/홀 매칭)를 최대로 만들면, 메리의 승리는 자동으로 최소가 된다.
    • mis = min(maryEven, johnOdd) + min(maryOdd, johnEven), win = N - mis를 계산하면 된다.
    • 시간 복잡도 O(N), 공간 복잡도 O(1)로 충분히 해결 가능하다.

    문제 5770 — Dice

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    Dice (BOJ 5770) 문제는
    여러 명의 플레이어가 주사위 합만큼 말(말 위치)을 전진시키면서,
    먼저 코스 끝(마지막 칸 이후)에 도달하는 플레이어의 번호를 구하는 시뮬레이션 문제다.

    규칙을 정리하면:

    • 플레이어 수: P (1 ≤ P ≤ 10)
    • 트랙 칸 수: S (3 ≤ S ≤ 10000)
      시작 위치는 1번 칸 앞으로, 처음 위치는 모두 0으로 보면 된다.
    • 트랩은 정확히 3칸이고, 위치는 T1, T2, T3 (1 ~ S 사이)로 주어진다.
    • 주사위를 두 개 굴려 합(D1 + D2)만큼 전진한다.
    • 말이 트랩 칸에 정확히 도착하면, 다음 차례를 한 번 건너뛴다 (한 라운드 쉼).
    • 플레이어는 항상 1번부터 P번까지 순서대로 차례를 가진다. (1 → 2 → … → P → 다시 1 → …)
    • 말의 위치가 S를 넘는 순간(pos > S), 그 플레이어가 즉시 승리한다.
    • 입력으로 주어지는 주사위 결과 개수는 어느 한 플레이어가 정확히 승리할 수 있을 만큼만 주어진다고 보장된다. (비길 일이 없다)

    전략 개요

    • 각 플레이어의 현재 위치를 저장하는 배열 pos[0..P-1]를 둔다. (초기값 0)
    • 각 플레이어가 다음 차례를 건너뛰어야 하는지 여부를 저장하는 배열 skip[0..P-1]를 둔다.
    • 트랩 위치를 빠르게 확인하기 위해 trap[1..S] 불리언 배열을 사용한다.
    • 주사위 합은 미리 dice[] 배열에 저장해 두고, 인덱스를 증가시키며 차례대로 사용한다.
    • 현재 플레이어 인덱스를 curPlayer로 두고, 승자가 나올 때까지 반복해서 시뮬레이션한다.

    핵심은 규칙 그대로 구현하는 단순 시뮬레이션이다.

    코드가 하는 일

    • 입력 처리 및 종료 조건 
      StringTokenizer st = new StringTokenizer(line);
      int P = Integer.parseInt(st.nextToken());
      int S = Integer.parseInt(st.nextToken());
      if (P == 0 && S == 0) break;
      • P, S가 둘 다 0이면 전체 입력 종료.
    • 트랩 정보 저장 
      st = new StringTokenizer(br.readLine());
      int T1 = Integer.parseInt(st.nextToken());
      int T2 = Integer.parseInt(st.nextToken());
      int T3 = Integer.parseInt(st.nextToken());
      
      boolean[] trap = new boolean[S + 1];
      trap[T1] = trap[T2] = trap[T3] = true;
      • trap[x] == true이면 x번 칸이 트랩이다.
    • 주사위 합 미리 저장 
      int diceCnt = Integer.parseInt(br.readLine());
      int[] dice = new int[diceCnt];
      for (int i = 0; i < diceCnt; i++) {
          st = new StringTokenizer(br.readLine());
          int D1 = Integer.parseInt(st.nextToken());
          int D2 = Integer.parseInt(st.nextToken());
      
          dice[i] = D1 + D2;
      }
      • 각 줄의 D1, D2를 읽어서 합만 배열에 저장.
    • 상태 배열 초기화 
      int[] pos = new int[P];
      boolean[] skip = new boolean[P];
      int curPlayer = 0;
      int diceIdx = 0;
      int winner = -1;
      • pos[i]: i번 플레이어의 위치 (0부터 시작)
      • skip[i]: 다음 라운드를 쉬어야 하면 true
      • curPlayer: 현재 차례인 플레이어 (0 ~ P-1)
      • diceIdx: 다음에 사용할 주사위 합 인덱스
      • winner: 승자 번호 (-1이면 아직 미정)
    • 게임 시뮬레이션 루프 
      while (winner == -1) {
          if (skip[curPlayer]) {
              skip[curPlayer] = false;
          } else {
              int move = dice[diceIdx++];
              pos[curPlayer] += move;
      
              if (pos[curPlayer] > S) {
                  winner = curPlayer + 1;
                  break;
              }
      
              if (trap[pos[curPlayer]]) {
                  skip[curPlayer] = true;
              }
          }
      
          curPlayer = (curPlayer + 1) % P;
      }
      • 1) 쉬는 차례 확인
        • skip[curPlayer] == true이면 이번 라운드는 그냥 건너뛴다.
        • 건너뛰면서 skip[curPlayer] = false로 되돌려 놓는다.
      • 2) 정상적으로 주사위를 굴리는 경우
        • move = dice[diceIdx++]로 주사위 합을 가져온다.
        • pos[curPlayer] += move 만큼 전진.
        • 만약 pos[curPlayer] > S가 되면, 트랙 끝을 넘어섰으므로 즉시 승리: 
          winner = curPlayer + 1;
          (플레이어 번호는 1부터이므로 +1)
        • 아직 끝나지 않았고, 현재 위치가 트랩이면: 
          if (trap[pos[curPlayer]]) {
          skip[curPlayer] = true;
      }
          다음 라운드를 한 번 쉰다.
      • 3) 다음 플레이어로 넘기기
        • curPlayer = (curPlayer + 1) % P;
        • 마지막 플레이어까지 끝났으면 다시 0번(=1번 플레이어)로 순환.
    • 결과 출력 
      sb.append(winner).append('\n');
      • 각 테스트케이스마다 승자 번호 한 줄 출력.

    시간 복잡도

    • 입력으로 주어지는 주사위 합의 개수를 N이라고 하면,
      • 각 테스트케이스에서, 시뮬레이션은 주사위 배열을 한 번 끝까지 순회한다.
      • 각 칸 이동과 트랩/스킵 체크는 모두 상수 시간.
    • 따라서 한 테스트케이스의 시간 복잡도는 O(N)이다.
    • P ≤ 10, S ≤ 10000 범위에서 충분히 빠르다.

    공간 복잡도

    • pos[P], skip[P], trap[S+1], dice[N] 정도만 사용한다.
    • 따라서 공간 복잡도는 O(P + S + N)이며, 문제 제약 상 매우 충분하다.

    주의

    • 트랩 조건은 “말이 해당 칸에 멈췄을 때 다음 라운드 한 번 쉰다”는 점을 정확히 구현해야 한다.
    • 위치가 S를 넘어가는 순간 바로 승리이므로, 트랩 칸을 지나쳐서 끝을 넘어갈 때는 트랩 효과를 보지 않는다.
    • skip 처리는 “쉬고 나면 바로 해제”가 되어야 한다. (두 번 연속 쉬지 않도록)
    • 플레이어 번호는 1부터, 배열 인덱스는 0부터이므로 출력 시 curPlayer + 1을 넣는 것을 잊지 말아야 한다.

    포인트 요약

    • 각 플레이어 위치 pos[]와 한 턴 건너뛸지 여부 skip[]를 관리한다.
    • 트랩은 trap[] 불리언 배열로 빠르게 체크한다.
    • 주사위 결과는 미리 합으로 저장하고 차례대로 사용한다.
    • 각 턴마다
      • skip이면 그냥 넘기고,
      • 아니면 전진 → 끝 도달 여부 → 트랩 여부를 순서대로 검사한다.
    • 어느 플레이어가 처음으로 pos > S가 되는 순간 그 플레이어가 승자가 된다.
