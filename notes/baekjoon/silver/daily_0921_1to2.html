    문제 1 — 다리 놓기

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 — 서쪽 N개, 동쪽 M개의 사이트가 있을 때 다리가 서로 겹치지 않으려면 서쪽의 순서가 동쪽에서도 유지되어야 합니다. 이는 결국 M개 중에서 N개를 고르는 조합과 같아서, 정답은 C(M, N)입니다.

    변수 매핑 — 입력에서 west는 서쪽(고르는 개수, r), east는 동쪽(전체 개수, n)을 의미합니다. 따라서 우리가 구하는 값은 nCr(west, east)라기보다 수학적으로는 C(n=east, r=west)입니다. 혼동을 줄이려고 함수 내부에서 이 의미를 정확히 반영해 계산합니다.

    왜 BigInteger인가 — 팩토리얼로 직접 계산하면 중간 값이 급격히 커져 long을 넘어섭니다(예: 30! ≈ 2.65×1032). 그래서 자바의 임의 정밀 정수형인 BigInteger로 연산합니다. BigInteger.ONE은 1을 나타내는 상수이고, 곱셈/나눗셈은 multiply/divide 메서드로 수행합니다.

    조합을 안전하게 계산하는 방법 — 팩토리얼을 모두 계산하지 않고, 조합의 등가식

    C(n, r) = ∏(i=1..r) (n - r + i) / i

    을 그대로 구현했습니다. 먼저 r을 더 작은 쪽(min(r, n-r))으로 바꿔 곱·나눗셈 횟수를 줄입니다(대칭성 C(n, r) = C(n, n-r)). 그다음 반복문에서 매 회차마다 분자 하나와 분모 하나를 처리합니다:

    • east - west + i → 분자 (n - r + i)
    • i → 분모 i
    • result = result.multiply(...).divide(...) → 누적 곱에 분자/분모를 차례로 반영

    이렇게 하면 매 단계가 result = result × (분자/분모) 형태로 누적되며, 루프가 끝나면 정확히 C(n, r)가 됩니다. 팩토리얼 전체를 만들지 않기 때문에 중간 값이 불필요하게 커지지 않고, 오버플로우 위험도 없습니다.

    입출력 최적화(StringBuilder) — 테스트 케이스가 여러 개일 때 System.out.println()을 반복 호출하면 I/O 비용이 커질 수 있습니다. 그래서 각 케이스의 결과를 StringBuilder로 이어 붙였다가(append) 마지막에 한 번에 출력합니다. 논리 결과는 같지만, 채점 환경에서 일반적으로 더 빠르고 안정적입니다.

    예시

    입력
      3
      2 2
      1 5
      13 29
      
      출력
      1
      5
      67863915

    요약

    • 문제는 결국 조합 C(M, N) 계산이다.
    • BigInteger로 곱하면서 즉시 나누는 방식(∏ (n - r + i) / i)을 구현해 오버플로우 없이 정확하게 구한다.
    • StringBuilder로 결과를 모아 한 번에 출력해 I/O 부담을 줄인다.
