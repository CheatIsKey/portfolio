    문제 3279 — DOLLARS

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 —
    N일 동안 주어지는 일일 환율 rate[i](정수, 소수 둘째 자리 고정: 예 123 → 1.23)로 달러(USD)와 마르크(MARK) 사이를 자유롭게 교환할 수 있을 때, 시작 자산 $100.00에서 출발해 마지막 날 이후 보유 가능한 달러의 최댓값을 구한다.
    핵심은 “지금 달러를 들고 있을 때의 최적값(USD 상태)”과 “지금 마르크를 들고 있을 때의 최적값(MARK 상태)”을 하루씩 갱신하는 2상태 DP이다.

    제약 정리 —
    입력: 첫 줄에 N. 이어서 N개의 줄에 각 날짜의 환율 rate (정수, 소수 둘째 자리까지).
    해석: rate는 “마르크/달러”를 100배 한 값이므로 실수 환율은 r = rate / 100.0.
    출력: 최대로 만들 수 있는 달러 금액을 소수 둘째 자리까지 출력.

    코드 핵심 (변수 매핑) —
    usd: i일까지 고려했을 때 달러를 보유하고 달성 가능한 최대 금액(달러 단위).
    mark: i일까지 고려했을 때 마르크를 보유하고 달성 가능한 최대 금액(마르크 단위).
    초기화: usd = 100.0, mark = 0.0 (처음엔 달러만 보유).
    하루 갱신 규칙(환율 r = rate/100):

    • mark = max(mark, usd * r) — 오늘 달러를 전부 마르크로 바꿔 보는 것이 이득이면 교환.
    • usd = max(usd, mark / r) — 오늘 마르크를 전부 달러로 바꿔 보는 것이 이득이면 교환.

    구현 상 정수 rate로 주어지므로 usd * rate / 100.0, mark * 100.0 / rate 형태로 계산한다.

    동작 과정 —
    1) N을 읽고 usd=100.0, mark=0.0으로 시작한다.
    2) 각 날짜의 rate를 읽어 mark를 먼저 갱신(달러→마르크), 이어 usd를 갱신(마르크→달러)한다.
    3) 모든 날짜를 처리한 뒤 usd가 “마지막에 달러로 보유할 때의 최댓값”이므로 이를 소수 둘째 자리까지 출력한다(System.out.printf("%.2f", usd)).

    엣지 케이스

    • 환율이 단조 하락(상승)만 해도 중간 교환이 손해라면 자연스럽게 교환이 발생하지 않는다(상태 최대값 유지).
    • 같은 날에 달러→마르크→달러 “왕복”이 이득이면 위의 두 식이 연쇄적으로 그 이득을 반영한다(수수료 없음 가정).
    • 부동소수점 오차를 피하려 중간 반올림을 하지 않고 마지막에만 %.2f로 출력한다.

    시간·공간 복잡도 —
    매일 상수 번 연산만 하므로 시간 복잡도 O(N).
    추가 메모리는 usd, mark 두 실수 변수뿐이므로 공간 복잡도 O(1).

    정리

    • 두 상태(USD/MARK) 최댓값을 하루씩 갱신하는 2상태 DP(또는 무제한 교환 그리디)로 해결.
    • mark = max(mark, usd * rate/100), usd = max(usd, mark * 100/rate)만 올바르게 누적하면 된다.
    • 최종 답은 usd를 소수 둘째 자리까지 출력.

    문제 3280 — CARDS

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 —
    카드를 3열로 row = N/3행에 깔고, 매 라운드마다 관찰자가 “정답 카드가 들어 있는 열”을 first/second/third로 알려 준다.
    매 라운드에 대해 “그 열에 있지 않은 카드”를 후보군에서 제거하고, 그 다음 라운드 진행을 위해 문제의 재배치 규칙(열 단위 집어서 다시 위에서부터 좌→우로 채우기)을 그대로 시뮬레이션한다.
    모든 라운드를 통과해 끝까지 “매번 지정된 열에 있었던 카드들”이 최종 후보가 된다.

    제약 정리 —
    입력: 첫 줄 N(카드 개수, 문제에서 3의 배수), 다음 줄 D(라운드 수), 이어서 D개의 줄에 각 라운드에서 정답 카드가 있는 열: "first", "second", "third".
    출력: 가능한 카드 번호(오름차순, 공백 구분).

    코드 핵심 (변수 매핑) —
    row = N/3: 행의 개수.
    field[row][3]: 현재 배치된 카드 번호(행-주기 저장). 초기에는 1부터 N까지를 위에서 아래, 왼→오 순으로 채움.
    check[1..N]: 각 카드가 “끝까지 후보인지” 여부(초깃값 true).
    line: 이번 라운드에서 정답 카드가 들어 있는 열 인덱스(0: first, 1: second, 2: third).

    • 후보 걸러내기: 각 행에 대해, 이번 라운드의 지정 열이 아닌 두 열의 카드 번호를 check[번호] = false로 표기.
    • 재배치 시뮬레이션: 문제의 규칙을 그대로 구현.
      • 열 단위로(왼→오, 각 열은 위→아래) 카드를 1차 배열 next로 이어 붙임.
      • next를 다시 위→아래, 왼→오 순으로 field에 채워 다음 라운드 배치를 만든다.

    동작 과정 —
    1) N, D를 읽고 field를 1..N으로 초기화, check는 모두 true로 설정.
    2) 각 라운드에 대해 열 이름을 line으로 매핑(0/1/2).
    3) 모든 행에 대해 지정 열이 아닌 칸들의 카드를 후보에서 제거(check[..]=false).
    4) 열→배열→행주입 순서로 재배치를 시뮬레이션.
    5) 모든 라운드 종료 후 check[i]==true인 카드 번호를 오름차순 출력.

    엣지 케이스

    • 같은 라운드에서 여러 행에 대해 같은 열이 지정되므로, 지정 열 외 카드는 그 라운드에서 전부 후보 탈락.
    • 라운드가 진행될수록 후보는 단조 감소하며(되살아나지 않음), 끝까지 모든 라운드에서 지정 열에 속했던 카드만 남음.
    • 부정확한 입력(예: N이 3의 배수가 아님)은 문제 보장으로 배제.

    시간·공간 복잡도 —
    라운드마다 후보 제거는 O(N), 재배치도 O(N)이므로 총 O(D·N).
    추가 배열 field(크기 N), next(크기 N), check(크기 N)를 사용하므로 공간은 O(N).

    정리

    • “지정 열에 없던 카드”는 라운드마다 즉시 탈락 → 필터링.
    • 문제의 재배치 규칙(열로 모아 이어 붙인 뒤 행-주입)을 정확히 시뮬레이션.
    • 최종적으로 끝까지 조건을 만족한 카드 번호를 출력한다.
