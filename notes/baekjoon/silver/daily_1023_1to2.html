    문제 3281 — T9

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 —
    휴대폰 T9 키패드에서 각 알파벳은 특정 숫자(2–9)에 매핑된다. 메시지는 1로 구분되는 단어들의 연속이며, 각 단어는 해당 숫자열(시그니처)과 사전에 있는 단어의 시그니처를 대조해 가장 먼저(사전순 첫 번째) 일치하는 단어로 치환된다. 일치하는 단어가 없으면 단어 길이만큼 '*'로 출력한다.
    주어진 사전이 이미 오름차순으로 정렬되어 있으므로, 같은 시그니처를 가진 여러 단어 중 첫 등장 단어만 저장해 두면 곧바로 “첫 매칭 우선” 규칙을 만족한다.

    제약 정리 —
    입력: 첫 줄 M(1 ≤ M ≤ 100) — 사전 단어 수. 다음 M줄 — 대문자(A–Z) 단어(길이 ≤ 100, 오름차순 정렬).
    다음 줄 N(1 ≤ N ≤ 100) — 키 입력 개수. 그 다음 줄 — N개의 키(1–9) 공백 구분.
    출력: T9 규칙으로 얻은 메시지 한 줄. 1은 단어 경계(공백)로 해석. 일치 단어가 없으면 해당 단어를 '*'로 대체.

    코드 핵심 (변수/구조 매핑) —
    키패드 매핑:

    • 2: ABC, 3: DEF, 4: GHI, 5: JKL, 6: MNO, 7: PQRS, 8: TUV, 9: WXYZ
    • Map<Character,Integer> map — 문자 → 숫자 키.

    사전 전처리:

    • 단어를 숫자 시그니처로 변환 후 Map<String,String> dict에 putIfAbsent로 저장.
    • 사전이 오름차순이므로 같은 시그니처의 첫 단어만 보존 ⇒ “첫 매칭” 보장.

    런타임 처리:

    • StringBuilder buf — 현재 단어의 숫자열 누적.
    • StringBuilder out — 최종 출력 메시지.
    • 키를 왼쪽부터 차례로 읽음:
      • key == 1: 단어 경계 — buf를 사전 조회해 출력(또는 * 반복), buf 초기화, 공백 추가.
      • key ∈ {2..9}: buf에 숫자 추가.
    • 모든 키 처리 후 buf에 남은 마지막 단어를 동일 규칙으로 출력.

    동작 과정 —
    1) 키패드 문자→숫자 매핑 테이블 구성.
    2) 사전의 각 단어를 숫자 시그니처로 변환해 dict[시그니처] = 첫 단어로 저장.
    3) 입력 키열을 순회하며 1에서 단어를 끊고 출력, 마지막에 buf 잔여 처리.
    4) 사전에 없으면 buf.length()만큼 '*'를 출력.

    정확성 근거 —
    사전이 오름차순 정렬이므로 동일 시그니처 최초 등장 단어가 T9의 “첫 단어” 규칙과 일치한다. putIfAbsent로 최초 단어만 보존하여 규칙을 그대로 구현한다.

    엣지 케이스

    • 사전에 없는 시그니처: 해당 길이만큼 * 출력.
    • 연속된 1: 코드상 ' '가 연속 추가되어 다중 공백이 생긴다. 입력이 “단어는 한 공백으로 분리”라면 연속 1이 등장하지 않는 것이 자연스럽다.
    • 사전에 동일 시그니처 다수: putIfAbsent로 첫 단어만 유지.
    • 마지막 키가 1이 아님: 루프 종료 후 buf 잔여 단어를 처리.

    시간·공간 복잡도 —
    사전 전처리: 각 단어 길이를 L이라 할 때 총 O(∑L).
    입력 처리: 키 수 N에 대해 O(N).
    공간: 시그니처→단어 해시맵 크기 O(사전 시그니처 수) + 보조 버퍼(미미) .

    정리

    • 알파벳→숫자 매핑으로 사전 단어를 시그니처화하고, 같은 시그니처의 첫 단어만 저장.
    • 입력 숫자열을 순회하며 1에서 단어를 끊고 매핑된 단어 또는 *로 출력.
    • 사전 정렬 특성을 이용해 T9의 “첫 매칭 우선”을 간결히 만족.

    문제 3288 — BEER

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 —
    병들은 아래에서 위로 한 줄씩 개수가 1씩 줄어드는 삼각형 형태다. 한 병을 90° 시계방향으로 돌리면 자기 자신은 시계방향(+1), 왼쪽 위/오른쪽 위 이웃은 동시에 반시계방향(−1)으로 회전한다. 따라서 아래 행부터 위로 진행하면, 현재 행의 병을 모두 ‘위(U)’로 맞출 때 그 영향은 위쪽으로만 전파되어 이미 처리한 아래 행은 다시 흐트러지지 않는다. 각 병은 상태가 4가지(U,R,D,L)이므로 현재 방향을 U로 만들기까지 시계방향 회전 횟수는 (4 - dir) % 4회 이내(최대 3회)이며, 전체 병 수가 최대 55개(N ≤ 10)이므로 총 연산 수는 165회를 넘지 않아 요구 조건(≤ 1000회)을 만족한다.

    제약 정리 —
    입력: 첫 줄 N(1 ≤ N ≤ 10). 이어서 N줄: 각 행의 화살표 방향들(U,L,D,R). 예시처럼 위에서부터 아래 순으로 주어지므로 코드에서 배열을 역으로 배치해 인덱스 0이 아래 행이 되도록 저장한다.
    출력: 각 단순 회전(시계 90°)을 “행 번호 열 번호”로 한 줄에 하나씩. 행은 아래=1부터 위로 증가, 열은 왼쪽=1부터 오른쪽으로 증가.

    코드 핵심 (변수/구조 매핑) —
    상태 인코딩: U=0, R=1, D=2, L=3 (시계 방향으로 +1씩).
    그리드: int[][] refrigerator — refrigerator[i][j]는 아래에서 i행, 왼쪽에서 j열 병의 방향 상태.
    회전 규칙(단순 회전 1회):

    • 자기 자신: dir = (dir + 1) % 4
    • 왼쪽 위: (i+1, j-1)가 존재하면 dir = (dir + 3) % 4
    • 오른쪽 위: (i+1, j)가 존재하면 dir = (dir + 3) % 4

    처리 순서: i=0(아래 행) → N-1(위 행)로, 각 칸 (i,j)에 대해 cnt = (4 - dir) % 4회 자기 병을 돌려 U로 만든다. 매회전 시 위 이웃의 반시계 회전을 즉시 반영하고, 해당 움직임을 답에 기록한다.

    동작 과정 —
    1) 입력을 받아 위→아래 순서의 문자열을 역인덱스로 저장해 refrigerator[0]가 아래 행이 되도록 준비.
    2) 아래 행부터 각 병을 순회하며 U가 될 때까지 회전((4 - dir) % 4회). 매 회전 시 위 이웃 두 곳을 반시계로 갱신.
    3) 수행한 모든 회전을 “행 열”로 누적 출력. 이미 처리한 아래쪽이 다시 변하지 않으므로 추가 조정이 필요 없다.

    정확성 근거 —
    연산이 자기 자신과 위쪽만 영향을 주므로, 하향식(아래→위) 그리디로 각 칸을 U로 확정한 뒤에는 되돌릴 필요가 없다. 4-원 환산(모듈러 4)에서 U가 되기까지 필요한 최소 시계 회전 수는 (4 - dir) % 4로 유일하며, 각 칸에 대해 독립적으로 적용해도 상부만 교란되므로 다음 단계에서 흡수 가능하다.

    엣지 케이스

    • N=1: 위 이웃이 없어 자기 자신만 돌리면 끝.
    • 가장 위 행: 위 이웃이 없으므로 자기 자신만 회전.
    • 입력 행 순서: 예제처럼 위→아래로 주어진 것을 코드에서 역배열로 저장해 행 번호(아래=1) 요구와 일치시킴.

    시간·공간 복잡도 —
    병 개수 M = N(N+1)/2 ≤ 55. 각 병 최대 3회 회전이므로 시간 O(M) (상수 작음), 공간은 그리드와 출력 버퍼로 O(M) .

    정리

    • 영향이 위쪽으로만 전파되므로 아래→위 순서로 각 병을 U로 고정.
    • 필요 회전 수는 (4 - dir) % 4, 위 이웃은 매 회전마다 반시계 보정.
    • 총 회전 수는 항상 1000 이하로 충분(최대 165회).
