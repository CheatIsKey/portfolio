    문제 3281 — T9

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 —
    길이가 정확히 n자리인 정수들 중에서 D로 나누어떨어지는 가장 작은 수를 찾는 문제다. n = 1이면 최솟값은 1, 그 외에는 최솟값 min = 10^(n-1), 상한(배제) max = 10^n으로 두고, min 이상에서의 D의 최초 배수 M을 계산한다. 이는 M = ceil(min / D) * D 와 동일하며, M < max면 답이고, 아니면 No solution이다. 큰 자릿수를 다루기 위해 Java의 BigInteger를 사용한다.

    제약 정리 —
    입력: 정수 n(자릿수), 정수 D.
    출력: D로 나누어떨어지는 가장 작은 n자리 수가 존재하면 그 수, 없으면 No solution.
    정확한 범위(예: n, D의 상한)는 원문을 따름 (확실하지 않음).

    코드 핵심 (변수/구조 매핑) —

    • BigInteger min: n==1이면 1, 아니면 10^(n-1)
    • BigInteger max: 10^n (상한, 배제)
    • BigInteger d: D의 BigInteger 표현
    • tmp = min.mod(d), M = (tmp==0) ? min : min + (d - tmp): min 이상 최초의 D 배수
    • 판정: M.compareTo(max) < 0 이면 M 출력, 아니면 No solution

    동작 과정 —
    1) min과 max를 설정한다(자릿수 경계값 계산).
    2) min을 D로 나눈 나머지 tmp를 구한다.
    3) tmp==0이면 min 자체가 답 후보, 아니면 d - tmp만큼 올려 최초 배수 M을 만든다.
    4) M < max면 M은 n자리이므로 출력, 아니면 존재하지 않으므로 No solution.

    정확성 근거 —
    정수의 배수 집합은 등차수열이므로, min 이상에서의 최초 배수는 나머지 보정 (d - (min mod d))를 더해 얻는 값과 일치한다. 또한 n자리 수의 범위는 [10^(n-1), 10^n)이므로, M이 이 구간에 속하는지(M < 10^n)만 확인하면 충분하다.

    엣지 케이스

    • n = 1: min = 1로 시작(0은 1자리 자연수로 취급하지 않음 가정).
    • D = 1: 항상 해 존재, 답은 min.
    • M = max가 되는 경우: max는 n+1자리의 경계값이므로 n자리가 아니어서 No solution이 맞다.
    • 아주 큰 n: 10^n 구성 및 출력 비용이 커질 수 있음 (환경 의존, 확실하지 않음).

    시간·공간 복잡도 —
    자릿수 길이를 L ≈ n이라 보면, 10^(n-1), 10^n 생성과 나머지/덧셈/비교 같은 BigInteger 연산은 자릿수에 선형~준선형으로 비례한다. 따라서 전체는 BigInteger 산술 기준으로 O(L)~O(L log L) 수준이며, 출력도 O(L)가 된다 (상수·구현 세부는 JVM/라이브러리 의존).

    정리

    • n자리 구간을 [10^(n-1), 10^n)로 잡고, 그 안에서의 D의 첫 배수 M을 계산.
    • M < 10^n이면 답, 아니면 존재하지 않으므로 No solution.
    • 큰 자릿수를 다루기 위해 BigInteger 연산으로 안전하게 처리.

    문제 3288 — BEER

    문제 링크

    백준 문제 보기

    코드 (IDE 스크린샷)

    해설

    핵심 아이디어 —
    병들은 아래에서 위로 한 줄씩 개수가 1씩 줄어드는 삼각형 형태다. 한 병을 90° 시계방향으로 돌리면 자기 자신은 시계방향(+1), 왼쪽 위/오른쪽 위 이웃은 동시에 반시계방향(−1)으로 회전한다. 따라서 아래 행부터 위로 진행하면, 현재 행의 병을 모두 ‘위(U)’로 맞출 때 그 영향은 위쪽으로만 전파되어 이미 처리한 아래 행은 다시 흐트러지지 않는다. 각 병은 상태가 4가지(U,R,D,L)이므로 현재 방향을 U로 만들기까지 시계방향 회전 횟수는 (4 - dir) % 4회 이내(최대 3회)이며, 전체 병 수가 최대 55개(N ≤ 10)이므로 총 연산 수는 165회를 넘지 않아 요구 조건(≤ 1000회)을 만족한다.

    제약 정리 —
    입력: 첫 줄 N(1 ≤ N ≤ 10). 이어서 N줄: 각 행의 화살표 방향들(U,L,D,R). 예시처럼 위에서부터 아래 순으로 주어지므로 코드에서 배열을 역으로 배치해 인덱스 0이 아래 행이 되도록 저장한다.
    출력: 각 단순 회전(시계 90°)을 “행 번호 열 번호”로 한 줄에 하나씩. 행은 아래=1부터 위로 증가, 열은 왼쪽=1부터 오른쪽으로 증가.

    코드 핵심 (변수/구조 매핑) —
    상태 인코딩: U=0, R=1, D=2, L=3 (시계 방향으로 +1씩).
    그리드: int[][] refrigerator — refrigerator[i][j]는 아래에서 i행, 왼쪽에서 j열 병의 방향 상태.
    회전 규칙(단순 회전 1회):

    • 자기 자신: dir = (dir + 1) % 4
    • 왼쪽 위: (i+1, j-1)가 존재하면 dir = (dir + 3) % 4
    • 오른쪽 위: (i+1, j)가 존재하면 dir = (dir + 3) % 4

    처리 순서: i=0(아래 행) → N-1(위 행)로, 각 칸 (i,j)에 대해 cnt = (4 - dir) % 4회 자기 병을 돌려 U로 만든다. 매회전 시 위 이웃의 반시계 회전을 즉시 반영하고, 해당 움직임을 답에 기록한다.

    동작 과정 —
    1) 입력을 받아 위→아래 순서의 문자열을 역인덱스로 저장해 refrigerator[0]가 아래 행이 되도록 준비.
    2) 아래 행부터 각 병을 순회하며 U가 될 때까지 회전((4 - dir) % 4회). 매 회전 시 위 이웃 두 곳을 반시계로 갱신.
    3) 수행한 모든 회전을 “행 열”로 누적 출력. 이미 처리한 아래쪽이 다시 변하지 않으므로 추가 조정이 필요 없다.

    정확성 근거 —
    연산이 자기 자신과 위쪽만 영향을 주므로, 하향식(아래→위) 그리디로 각 칸을 U로 확정한 뒤에는 되돌릴 필요가 없다. 4-원 환산(모듈러 4)에서 U가 되기까지 필요한 최소 시계 회전 수는 (4 - dir) % 4로 유일하며, 각 칸에 대해 독립적으로 적용해도 상부만 교란되므로 다음 단계에서 흡수 가능하다.

    엣지 케이스

    • N=1: 위 이웃이 없어 자기 자신만 돌리면 끝.
    • 가장 위 행: 위 이웃이 없으므로 자기 자신만 회전.
    • 입력 행 순서: 예제처럼 위→아래로 주어진 것을 코드에서 역배열로 저장해 행 번호(아래=1) 요구와 일치시킴.

    시간·공간 복잡도 —
    병 개수 M = N(N+1)/2 ≤ 55. 각 병 최대 3회 회전이므로 시간 O(M) (상수 작음), 공간은 그리드와 출력 버퍼로 O(M) .

    정리

    • 영향이 위쪽으로만 전파되므로 아래→위 순서로 각 병을 U로 고정.
    • 필요 회전 수는 (4 - dir) % 4, 위 이웃은 매 회전마다 반시계 보정.
    • 총 회전 수는 항상 1000 이하로 충분(최대 165회).
