    내일배움캠프 Spring 백엔드 3일차 TIL — SQL 기초 & JD 분석

    작성일 2025-12-10 · 분류: 부트캠프, TIL, SQL, JD

    3일차 개요 — SQL 복습과 서버 JD 리서치

    정리

    3일차에는 SQL 기본 문법 복습과 서버 개발자 JD 분석에 집중했다. 어제와 마찬가지로 예시 데이터를 표로 정리하고 직접 쿼리를 작성하면서 감각을 유지했고, 오후에는 커머스 백엔드와 공공데이터 기반 서비스에 가까운 토스증권·한울·카카오뱅크의 JD를 읽으면서 실제 현업에서 요구하는 역량을 정리했다.

    • 학습용 학생/프로젝트/게임 유저 테이블을 만들어 조건 조회, 집계, 정렬, 윈도 함수 등을 복습했다.
    • 관심 있는 서버 포지션 3개를 골라 기술 스택·요구 역량·도메인 특징을 비교했다.
    • JD를 기준으로 내가 잘하는 것과 부족한 부분을 나눠 보며, 앞으로의 공부 방향을 더 구체화했다.

    SQL 기초 퀘스트 — 학생 테이블(bootcamp_students)

    예시 데이터

    학생 정보를 담은 bootcamp_students 테이블 임시 데이터는 다음과 같다.

    id name course grade start_year
    1 르탄 Node.js A 2023
    2 배캠 Spring B 2022
    3 구구 Unity C 2021
    4 이션 Node.js B 2022

    풀이한 쿼리

    1. bootcamp_students 테이블에서 모든 학생의 이름(name)과 트랙(course)을 선택하는 쿼리

    SELECT name AS "이름", course AS "트랙"
FROM bootcamp_students;

    2. bootcamp_students 테이블에서 Unity 트랙 소속이 아닌 학생들을 선택하는 쿼리

    SELECT *
FROM bootcamp_students
WHERE course <> 'Unity';

    3. bootcamp_students 테이블에서 시작 연도(start_year)가 2021년인 학생과 2023년인 학생을 선택하는 쿼리

    SELECT *
FROM bootcamp_students
WHERE start_year IN (2021, 2023);

    4. bootcamp_students 테이블에서 Node.js 트랙 소속이고 학점이 ‘A’인 학생의 시작 연도(start_year)를 선택하는 쿼리

    SELECT name AS "이름", start_year AS "수강 시작년도"
FROM bootcamp_students
WHERE course = 'Node.js'
  AND grade = 'A';

    SQL 기초 퀘스트 — 프로젝트 테이블(bootcamp_projects)

    예시 데이터

    프로젝트 정보를 담은 bootcamp_projects 테이블 임시 데이터는 다음과 같다.

    id project_name started_at finished_at cloud_cost
    1 일조 2023-01-01 2023-01-07 30000
    2 꿈꾸는이조 2023-03-15 2023-03-22 50000
    3 보람삼조 2023-11-20 2023-11-30 80000
    4 사조참치 2022-07-01 2022-07-30 75000

    풀이한 쿼리

    1. bootcamp_projects 테이블에서 클라우드 예산(cloud_cost)이 40000 이상 들어간 프로젝트들의 이름을 선택하는 쿼리

    SELECT project_name AS "프로젝트 명"
FROM bootcamp_projects
WHERE cloud_cost >= 40000;

    2. bootcamp_projects 테이블에서 2022년에 시작된 프로젝트를 선택하는 쿼리 (단, started_at < '2023-01-01' 조건 사용 X)

    SELECT *
FROM bootcamp_projects
WHERE YEAR(started_at) = 2022;

    3. bootcamp_projects 테이블에서 현재 진행 중인 프로젝트를 선택하는 쿼리 (지금 시점의 날짜를 하드코딩하지 않기)

    SELECT *
FROM bootcamp_projects
WHERE CURDATE() BETWEEN started_at AND finished_at;

    4. bootcamp_projects 테이블에서 각 프로젝트의 지속 기간을 일 수로 계산하는 쿼리

    SELECT project_name,
       DATEDIFF(finished_at, started_at) AS duration_days
FROM bootcamp_projects;

    SQL 기초 퀘스트 — 게임 유저 테이블(summoner_users)

    예시 데이터

    게임 유저 정보를 담은 summoner_users 테이블 임시 데이터는 다음과 같다.

    id name region rank_score joined_at
    1 르탄이 한국 1300 2019-06-15
    2 배캠이 미국 1500 2020-09-01
    3 구구이 한국 1400 2021-01-07
    4 이션이 미국 1350 2019-11-15

    풀이한 쿼리

    1. summoner_users 테이블에서 각 유저의 레이팅(rank_score) 순위를 계산하는 쿼리 (전체 지역 기준, 점수가 높을수록 순위↑)

    SELECT *,
       RANK() OVER (ORDER BY rank_score DESC) AS "순위"
FROM summoner_users;

    2. summoner_users 테이블에서 가장 늦게 게임을 시작한(joined_at) 유저의 이름을 선택하는 쿼리

    SELECT name
FROM summoner_users
ORDER BY joined_at DESC
LIMIT 1;

    3. summoner_users 테이블에서 지역별로 레이팅이 높은 순으로 유저들을 정렬해서 나열하는 쿼리

    SELECT *
FROM summoner_users
ORDER BY region, rank_score DESC;

    4. summoner_users 테이블에서 지역별로 평균 레이팅을 계산하는 쿼리

    SELECT region,
       AVG(rank_score) AS "평균 점수"
FROM summoner_users
GROUP BY region;

    JD 분석 — 서버 개발자 포지션과 나의 방향

    공통적으로 보이는 핵심 역량

    토스증권, 한울, 카카오뱅크의 JD를 같이 놓고 보니, 세 회사 모두 Java 계열과 Spring 기반 서버 개발 역량을 기본 전제로 두고 있다는 점이 가장 먼저 보였다. Java/Kotlin + Spring Framework, Kotlin/Java + Spring Boot + JPA 등 표현은 조금씩 다르지만, 결국 “실무에서 통하는 백엔드 개발자”의 축은 Java와 Spring 라인이라는 걸 다시 확인했다.

    또 하나의 공통점은 서비스 전체를 보는 시야다. 단순히 API를 만드는 수준이 아니라, 아키텍처 개선·기술 부채 해결·대용량 트래픽에서의 트러블슈팅과 지표 개선까지 함께 맡길 수 있는 개발자를 찾고 있었다. 코드리뷰·형상관리·피드백 문화 등 협업 역량도 빠지지 않고 등장한다.

    각 JD의 차이점과 느낀 점

    • 토스증권: 금융 원장, 주식 매매·결제·환전, 실시간 데이터 처리, Kubernetes·ELK 등 분산 인프라가 핵심이다. “기술적으로 도전적인 환경에서 성장하고 싶은 서버 개발자”를 상정하고 있는 느낌을 받았다.
    • 한울: 출입통제·RFID·안전관리·IoT, 전자정부 프레임워크 기반 시스템 등 공공·SI 도메인이 중심이다. Android/iOS, 웹, 서버를 모두 아우르는 실무형 개발자 이미지가 강하고, 정보처리기사 등 전통적인 우대요건을 중시한다.
    • 카카오뱅크: 수신·여신·지급결제·개인사업자 상품을 담당하는 뱅킹 백엔드 포지션으로, 서버 개발 5년 이상을 요구한다. 대용량 트래픽과 복잡한 금융 도메인 경험을 가진 시니어급 개발자를 찾는 JD라는 느낌이었다.

    나의 커리어 방향과 부족한 부분

    지금까지의 경험을 정리해 보면, 나는 도메인 로직과 데이터 흐름을 설계하고, 예외·성능까지 함께 고민하는 영역에 제일 재미를 느낀다. 반수골 사람들에서 엑셀 매크로로 데이터 매핑·일괄 변환을 자동화할 때도 “어떻게 하면 더 안정적인 구조가 될까”를 먼저 생각했고, 음악 데이터 파이프라인에서는 파싱 오류 시 자동 복구·재시도, 로그 기반 검증 절차를 설계하면서 운영을 의식한 설계 경험을 쌓았다.

    JD를 보면서 특히 부족하다고 느낀 부분은 Redis·Kafka·Kubernetes·ELK 같은 분산 인프라 환경 경험과 금융·출입통제처럼 특정 도메인에서 수년 단위로 서비스를 운영해 본 경험이다. 단기간에 채울 수 있는 부분은 아니지만, 첫 커리어를 선택할 때 “어떤 도메인에서 장기적으로 깊이를 쌓을지”를 고민해야겠다는 생각이 들었다.

    오늘의 회고와 내일 계획

    오늘 느낀 점

    SQL 문제는 이미 한 번 본 유형이 많았지만, 테이블 이름과 컬럼명을 바꿔서 다시 만들어 보니 “문제를 통째로 외우는 공부”를 피할 수 있어서 좋았다. 앞으로도 예제를 이렇게 변형해 가면서 연습해야, 진짜로 문법이 몸에 남을 것 같다.

    JD 분석을 하면서는 “언젠가 가고 싶다”라고 막연하게 생각하던 회사들이 실제로는 어떤 기술과 경험을 요구하는지 조금 더 구체적으로 보였다. 내가 쌓아 온 자동화·파이프라인 경험이 장점이 될 수 있는 부분과, 아직 비어 있는 분산 인프라·대규모 운영 경험을 뚜렷하게 나눠 볼 수 있었던 하루였다.

    내일 계획

    • SQL에서 JOIN과 GROUP BY, HAVING을 활용한 예제를 직접 만들어 TIL에 이어서 정리하기.
    • 오늘 정리한 JD 공통 역량을 기준으로, 쇼핑몰 프로젝트에서 보여 줄 수 있는 포인트를 따로 문서로 정리하기.
    • Spring 핵심 개념(IoC/DI, 트랜잭션, 예외 처리)이 JD 요구사항과 어떻게 연결되는지 간단한 메모로 남기기.
    • 오늘 작성한 SQL 예제와 JD 분석 내용을 깃허브에 커밋해 학습 기록을 꾸준히 쌓아가기.
