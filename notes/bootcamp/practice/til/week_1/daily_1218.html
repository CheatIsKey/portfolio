    6일차 개요 — Java 문법을 “왜 쓰는지” 기준으로 정리

    정리

    오늘은 SQL 퀘스트 없이, Java 문법 중에서 자주 헷갈리거나 “알고는 있는데 설명이 애매한” 키워드들을 정리했다. 단순 정의 암기보다는 왜 필요한지, 언제 쓰는지, 어디서 실수하는지를 기준으로 묶어서 정리했다.

    • 래퍼 클래스(Wrapper Class)
    • 오토 박싱 / 언박싱(Auto Boxing / Unboxing)
    • static(정적 변수/메서드, static 중첩 클래스)
    • final(변수/메서드/클래스)
    • 인터페이스(Interface)

    래퍼 클래스(Wrapper Class) — 기본 타입을 객체로 다루기

    핵심 정의

    래퍼 클래스는 기본 타입(primitive)을 객체(Object)처럼 다루기 위해 제공되는 클래스다. 예를 들면 int는 Integer, double은 Double 같은 식이다.

    • 기본 타입: int, long, double, boolean 등
    • 래퍼 타입: Integer, Long, Double, Boolean 등

    언제 필요하나

    • 컬렉션(List, Map 등)은 객체만 담을 수 있어서 래퍼 타입이 필요하다.
    • null을 표현해야 할 때 기본 타입은 불가능하고, 래퍼 타입은 가능하다.
    • 제네릭을 쓸 때(예: List<Integer>)는 기본 타입을 그대로 넣을 수 없다.

    예시 

    // 기본 타입
int a = 10;

// 래퍼 타입
Integer b = 10;              // 오토 박싱
Integer c = Integer.valueOf(10);

// 컬렉션에는 래퍼 타입
java.util.List<Integer> list = new java.util.ArrayList<>();
list.add(10);

    주의할 점

    • null 가능: 래퍼 타입은 null이 될 수 있으니, 언박싱 시 NPE(NullPointerException) 위험이 있다.
    • 같은 숫자라도 == 비교는 조심해야 한다. 객체 비교는 보통 equals가 안전하다.
    Integer x = null;
// int y = x;  // 언박싱이 발생하면서 NullPointerException 위험

Integer p = 1000;
Integer q = 1000;
// System.out.println(p == q);      // 객체 참조 비교라 결과가 기대와 다를 수 있음
System.out.println(p.equals(q));    // 값 비교는 equals가 안전

    오토 박싱 / 언박싱 — 편하지만 “자동”이라서 실수도 난다

    핵심 정의

    • 오토 박싱: 기본 타입 → 래퍼 타입으로 자동 변환
    • 언박싱: 래퍼 타입 → 기본 타입으로 자동 변환
    // 오토 박싱
Integer boxed = 10;

// 언박싱
int unboxed = boxed;

// 연산 시에도 언박싱이 끼어들 수 있다
Integer n = 5;
int sum = n + 10; // n이 자동으로 언박싱된 뒤 연산됨

    주의할 점

    • null 언박싱은 바로 예외로 이어질 수 있다.
    • 반복문에서 과도한 박싱/언박싱이 발생하면 성능에 영향을 줄 수 있다(불필요한 객체 생성).
    Integer maybeNull = null;
// int v = maybeNull; // 언박싱 시도 → NullPointerException 위험

    static — “객체 없이도” 공유되는 영역

    핵심 정의

    static은 클래스에 속한다는 의미에 가깝다. 즉, 인스턴스를 만들지 않아도 접근 가능하고, 모든 인스턴스가 같은 값을 공유한다.

    static 변수/메서드 

    class Counter {
    static int count = 0;

    static void increase() {
        count++;
    }
}

// 인스턴스 없이 사용 가능
Counter.increase();
System.out.println(Counter.count);
    • static 메서드는 인스턴스 필드/인스턴스 메서드에 바로 접근할 수 없다.
    • 전역처럼 쓰기 쉬워서 편하지만, 남발하면 의존관계가 꼬이고 테스트가 어려워질 수 있다.

    static 중첩 클래스(정적 내부 클래스)

    Java에서 “static 클래스”는 보통 static 중첩 클래스를 의미한다. 외부 클래스의 인스턴스 없이도 생성할 수 있고, 외부 인스턴스에 묶이지 않는다. 

    class Outer {
    static class Inner {
        void hello() {
            System.out.println("hi");
        }
    }
}

Outer.Inner inner = new Outer.Inner();
inner.hello();

    final — “변경 불가”를 의도로 박아두는 키워드

    핵심 정리

    • final 변수: 재할당 불가
    • final 메서드: 오버라이딩 불가
    • final 클래스: 상속 불가

    예시 

    // final 변수: 재할당 불가
final int MAX = 100;
// MAX = 200; // 컴파일 에러

// final 클래스: 상속 불가
final class FinalService { }

// final 메서드: 오버라이딩 불가
class Parent {
    final void fixed() { }
}
class Child extends Parent {
    // void fixed() { } // 컴파일 에러
}

    주의할 점

    final 참조 변수는 “객체 자체가 불변”이라는 뜻이 아니다. 참조(주소)를 바꾸지 못한다는 뜻이고, 객체 내부 값은 바뀔 수 있다. 

    final java.util.List<String> names = new java.util.ArrayList<>();
// names = new java.util.ArrayList<>(); // 재할당 불가
names.add("A"); // 내부 변경은 가능

    인터페이스(Interface) — “규약”을 먼저 정하고 구현을 분리하기

    핵심 정의

    인터페이스는 “이 기능은 반드시 제공한다”라는 규약(약속)을 정의하는 역할이다. 구현체는 implements로 인터페이스를 구현하고, 호출하는 쪽은 구현체가 바뀌어도 같은 방식으로 사용할 수 있다.

    왜 쓰나

    • 결합도 낮추기: 구현이 바뀌어도 사용하는 코드는 덜 흔들리게 만든다.
    • 테스트 용이: 구현체 대신 가짜 구현(mock)을 끼워 넣기 쉬워진다.
    • 역할 분리: “무엇을 할지”와 “어떻게 할지”를 분리한다.

    예시 

    interface MessageSender {
    void send(String message);
}

class EmailSender implements MessageSender {
    public void send(String message) {
        System.out.println("Email: " + message);
    }
}

class SmsSender implements MessageSender {
    public void send(String message) {
        System.out.println("SMS: " + message);
    }
}

    // 사용하는 쪽은 구현체가 바뀌어도 동일하게 호출 가능
MessageSender sender1 = new EmailSender();
sender1.send("hello");

MessageSender sender2 = new SmsSender();
sender2.send("hello");

    알아두면 좋은 포인트

    • 인터페이스는 다중 구현이 가능하다(클래스 상속은 보통 단일 상속).
    • 필요하면 default 메서드로 기본 구현을 제공할 수도 있다.
    • static 메서드를 인터페이스 안에 둘 수도 있다.
    interface Calculator {
    int add(int a, int b);

    default int addOne(int x) {
        return add(x, 1);
    }

    static int triple(int x) {
        return x * 3;
    }
}

    오늘의 회고와 내일 계획

    오늘 느낀 점

    오늘 정리한 키워드들은 전부 “문법을 아는 것”보다 왜 쓰는지를 함께 알고 있어야 실수할 확률이 줄어든다. 특히 래퍼 타입과 언박싱은 null 때문에 사고가 나는 경우가 많아서, 코드를 작성할 때 더 의식적으로 보게 될 것 같다.
