    7일차 개요 — 객체지향 4대 개념을 “코드 설계” 기준으로 정리

    정리

    오늘은 SQL 없이 Java 객체지향의 핵심인 캡슐화·상속·추상화·다형성을 정리했다. “용어 설명”에서 끝내지 않고, 실제로 코드를 짤 때 어떤 선택이 유지보수에 영향을 주는지 중심으로 정리했다.

    • 캡슐화: 데이터 보호 + 변경 지점 통제
    • 상속: 공통 기능 재사용 + 확장(하지만 남발하면 구조가 꼬임)
    • 추상화: 중요한 것만 드러내서 복잡도를 낮춤
    • 다형성: 같은 메시지(호출)로 다른 동작(구현)을 실행

    캡슐화(Encapsulation) — “데이터를 숨기고, 변경은 통제한다”

    핵심 정의

    캡슐화는 객체의 내부 상태(필드)를 외부에서 직접 건드리지 못하게 하고, 필요한 기능만 메서드로 공개해서 변경 지점을 통제하는 개념이다. 결국 목적은 안전하게 유지보수하기다.

    접근 제어자 정리

    접근 제어자 접근 가능 범위 주로 쓰는 상황
    public 어디서든 접근 가능 외부에 공개할 API(컨트롤러/서비스 메서드 등)
    protected 같은 패키지 + 상속 받은 클래스 상속 구조에서 확장 포인트를 열어둘 때
    (default) 같은 패키지 패키지 내부에서만 쓰는 기능(의도적 범위 제한)
    private 해당 클래스 내부 필드 보호, 내부 구현 숨기기(캡슐화의 기본)

    Getter / Setter 예시

    캡슐화의 기본 패턴은 필드는 private로 숨기고, 필요한 값만 조회(Getter)하거나 상태 변경은 메서드로 통제하는 방식이다. 

    class Member {
    private String name;
    private int point;

    public Member(String name) {
        this.name = name;
        this.point = 0;
    }

    public String getName() {
        return name;
    }

    public int getPoint() {
        return point;
    }

    // 무분별한 setPoint 대신, 의도를 담은 메서드로 변경을 통제
    public void addPoint(int amount) {
        if (amount <= 0) throw new IllegalArgumentException("적립 포인트는 양수여야 합니다.");
        this.point += amount;
    }
}

    무분별한 Setter는 지양

    • Setter를 다 열어두면 외부에서 값을 아무 때나 바꿀 수 있어서 버그가 숨어들기 쉽다.
    • “상태 변경”은 보통 비즈니스 규칙을 포함하니까, 의도를 담은 메서드로 통제하는 게 안전하다.
    • 정말 필요한 경우에만 제한적으로 Setter를 두고, 검증 로직을 함께 넣는 편이 낫다.

    상속(Inheritance) — 공통 기능 재사용 + 확장

    핵심 정의

    상속은 기존 클래스의 필드/메서드를 물려받아 중복을 줄이고, 필요한 부분을 확장하는 구조다. 다만 상속은 강한 결합을 만들 수 있어서, “무조건 좋다”가 아니라 적절한 범위에서 쓰는 게 중요하다. 

    class Animal {
    protected String name;

    public Animal(String name) {
        this.name = name;
    }

    public void sound() {
        System.out.println("...");
    }
}

class Dog extends Animal {
    public Dog(String name) {
        super(name);
    }

    @Override
    public void sound() {
        System.out.println("멍멍");
    }
}

    오버라이딩(Overriding)

    • 상속 관계에서 부모의 메서드를 자식이 재정의하는 것.
    • 메서드 시그니처(이름/파라미터)가 동일해야 한다.
    • 동작을 바꾸는 확장 포인트로 자주 사용된다.

    추상화(Abstraction) — 중요한 것만 남기고 단순화

    핵심 정의

    추상화는 공통된 성질을 뽑아 “핵심만 보이게” 만드는 것이다. 구현 디테일을 숨기고 사용 관점에서 필요한 기능만 노출하면 코드가 단순해진다. 

    abstract class Payment {
    public abstract void pay(int amount);

    public void printReceipt(int amount) {
        System.out.println("결제 금액: " + amount);
    }
}

class CardPayment extends Payment {
    @Override
    public void pay(int amount) {
        System.out.println("카드 결제 진행");
        printReceipt(amount);
    }
}
    • 추상 클래스: 공통 기능(기본 구현) + 반드시 구현할 메서드(abstract)를 같이 둘 수 있다.
    • 공통 흐름을 잡아두고, 세부 구현만 자식에서 채우는 구조를 만들 때 유용하다.

    다형성(Polymorphism) — 같은 호출, 다른 동작

    핵심 정의

    다형성은 타입은 같게(부모/인터페이스) 보이지만, 실제 실행되는 구현은 달라지는 성질이다. 덕분에 코드는 유연해지고, 구현체 교체가 쉬워진다. 

    interface Notifier {
    void send(String message);
}

class EmailNotifier implements Notifier {
    public void send(String message) {
        System.out.println("Email: " + message);
    }
}

class SmsNotifier implements Notifier {
    public void send(String message) {
        System.out.println("SMS: " + message);
    }
}

class AlarmService {
    private final Notifier notifier;

    public AlarmService(Notifier notifier) {
        this.notifier = notifier;
    }

    public void notifyUser() {
        notifier.send("알림이 도착했습니다.");
    }
}

    위 구조는 AlarmService가 구체 구현(Email/SMS)에 묶이지 않아서, 나중에 구현체가 바뀌어도 서비스 코드를 크게 바꾸지 않아도 된다.

    오버로딩(Overloading)

    • 같은 이름의 메서드를 파라미터 구성(개수/타입)을 다르게 해서 여러 개 만드는 것.
    • 오버라이딩과 헷갈리기 쉬운데, 오버로딩은 상속이 없어도 가능하다.
    class Calculator {
    public int add(int a, int b) {
        return a + b;
    }

    public int add(int a, int b, int c) {
        return a + b + c;
    }
}

    오늘의 회고와 내일 계획

    오늘 느낀 점

    객체지향 4대 개념은 결국 “예쁘게 보이려고” 쓰는 게 아니라, 변경에 강한 코드를 만들기 위해 필요하다는 걸 다시 느꼈다. 특히 캡슐화는 작은 프로젝트에서도 바로 효과가 보이고, Setter를 막아두는 것만으로도 코드가 안전해진다는 게 인상적이었다.
