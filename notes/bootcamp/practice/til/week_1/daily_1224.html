    10일차 개요 — SQL 기능 정리 + AI 핵심 개념 잡기

    정리

    오늘은 SQL 문제 풀이보다는 자주 쓰이는 기능을 묶어서 정리하는 날이었다. SQL에서는 피벗(조건부 집계), 윈도우 함수, DATE_FORMAT을 중심으로 “언제/왜 쓰는지”를 정리했고, AI 파트에서는 학습 방식(지도/비지도/강화), 생성형 AI 개념, LLM/임베딩/트랜스포머, Fine-Tuning/RAG를 큰 흐름으로 연결해 보았다.

    • SQL은 “보고서 형태로 뽑기(피벗)”와 “순위/누적 같은 분석(윈도우)”이 핵심이었다.
    • DATE_FORMAT은 결과를 보기 좋게 만드는 용도이지만, 문자열로 변환된다는 점을 항상 의식해야 한다.
    • AI는 용어가 많아서, 개념을 “정의 → 용도 → 장단점” 구조로 정리했다.

    SQL 정리 — 피벗(조건부 집계)과 MAX 사용 패턴

    핵심 개념

    SQL에서 “피벗”은 행(row) 데이터를 열(column) 형태로 바꿔 보고서처럼 만드는 방식이다. MySQL에서는 전용 PIVOT 문법이 없어서 보통 조건부 집계로 구현한다.

    • SUM(CASE WHEN ... THEN 값 ELSE 0 END): 한 그룹 안에서 여러 건이 누적될 수 있을 때
    • MAX(CASE WHEN ... THEN 값 END): 한 그룹 안에서 “해당 값이 1개”라고 확신할 때(없으면 NULL)

    예시 데이터

    월별 매출을 담은 camp_sales 테이블 임시 데이터는 다음과 같다.

    product sales_month amount
    laptop 2025-11 1200
    laptop 2025-12 1800
    tablet 2025-11 400
    tablet 2025-12 600

    예시 쿼리

    1. 상품별 월 매출을 “열”로 펼쳐서 조회하기(피벗)

    SELECT
  product,
  SUM(CASE WHEN sales_month = '2025-11' THEN amount ELSE 0 END) AS "2025-11",
  SUM(CASE WHEN sales_month = '2025-12' THEN amount ELSE 0 END) AS "2025-12"
FROM camp_sales
GROUP BY product;

    2. “각 (product, sales_month)이 1건만 존재한다”면 MAX 패턴으로도 가능

    SELECT
  product,
  MAX(CASE WHEN sales_month = '2025-11' THEN amount END) AS "2025-11",
  MAX(CASE WHEN sales_month = '2025-12' THEN amount END) AS "2025-12"
FROM camp_sales
GROUP BY product;

    주의할 점

    • 월 컬럼을 고정으로 박으면(‘2025-11’, ‘2025-12’) 데이터가 늘 때마다 쿼리를 수정해야 한다.
    • MAX는 “그룹 내 값이 1개”라는 전제가 흔들리면, 의도치 않은 결과가 나올 수 있다. (그때는 SUM이 안전하다)

    SQL 정리 — 윈도우 함수(Window Function)

    핵심 개념

    윈도우 함수는 행을 줄이지 않고(=GROUP BY처럼 뭉개지 않고) 집계/순위/누적 값을 “붙여서” 보여준다. 즉, 원본 행을 유지한 채로 분석 컬럼을 추가하는 느낌이다.

    • RANK(), DENSE_RANK(), ROW_NUMBER(): 순위/번호
    • SUM() OVER(...): 누적합/구간합
    • PARTITION BY: “그룹을 나누는 기준”
    • ORDER BY: “그룹 안에서 정렬 기준”

    예시 쿼리

    1. 전체 매출 기준으로 주문 순위 매기기(동점 처리 포함)

    SELECT
  order_id,
  user_id,
  amount,
  RANK() OVER (ORDER BY amount DESC) AS amount_rank
FROM camp_orders;

    2. 유저별로 주문 날짜 순서대로 “누적 결제 금액” 붙이기

    SELECT
  order_id,
  user_id,
  order_date,
  amount,
  SUM(amount) OVER (
    PARTITION BY user_id
    ORDER BY order_date
  ) AS running_total
FROM camp_orders;

    3. “각 유저별 최신 주문 1건”만 뽑기(Top-N per group)

    SELECT *
FROM (
  SELECT
    order_id,
    user_id,
    order_date,
    amount,
    ROW_NUMBER() OVER (
      PARTITION BY user_id
      ORDER BY order_date DESC
    ) AS rn
  FROM camp_orders
) t
WHERE t.rn = 1;

    주의할 점

    • 윈도우 함수는 “행을 유지”하므로, 결과가 커지기 쉽다. (필요 컬럼만 조회하는 습관)
    • 대부분 PARTITION BY, ORDER BY에 걸리는 컬럼에 인덱스가 있으면 성능에 도움이 된다.

    SQL 정리 — DATE_FORMAT 함수(MySQL)

    핵심 개념

    DATE_FORMAT(날짜/시간, '포맷')은 날짜를 문자열로 보기 좋게 바꾸는 함수다. 리포트용(표시용)으로는 정말 편하지만, 문자열로 변환되므로 정렬/그룹핑 기준을 뭘로 잡는지는 항상 의식해야 한다.

    자주 쓰는 포맷

    목표 포맷 예시
    연-월-일 %Y-%m-%d 2025-12-24
    연-월 %Y-%m 2025-12
    시:분 %H:%i 13:05
    연월일(숫자) %Y%m%d 20251224

    예시 쿼리

    1. 날짜별 평균 포인트(일 단위 그룹핑)

    SELECT
  DATE_FORMAT(created_at, '%Y-%m-%d') AS created_date,
  ROUND(AVG(point)) AS average_points
FROM point_users
GROUP BY DATE_FORMAT(created_at, '%Y-%m-%d');

    2. 월 단위 리포트(‘2025-12’ 형태)

    SELECT
  DATE_FORMAT(created_at, '%Y-%m') AS ym,
  COUNT(*) AS cnt
FROM users
GROUP BY DATE_FORMAT(created_at, '%Y-%m')
ORDER BY ym;

    주의할 점

    • DATE_FORMAT은 문자열이므로 “정확한 날짜 연산”이 필요하면 DATE(), TIMESTAMP 비교를 우선 고려한다.
    • 그룹핑은 SELECT와 GROUP BY에 같은 표현식을 쓰는 습관을 들인다.

    AI 정리 — 학습 방식, 생성형 AI, LLM/임베딩, Fine-Tuning과 RAG

    AI 한 문장 정의

    AI는 데이터를 바탕으로 분류/예측/추천/의사결정 같은 작업을 수행하는 소프트웨어/모델의 묶음이다. 단순 규칙(if-else)만으로 처리하기 어려운 복잡한 패턴을 학습해서 다룰 수 있다는 점이 핵심이다.

    AI 학습 방식 3가지

    구분 학습 데이터 목표 대표 예시
    지도 학습 정답(라벨)이 있는 데이터 입력 → 정답으로 매핑 학습 스팸 분류, 가격 예측
    비지도 학습 정답(라벨)이 없는 데이터 데이터 구조/패턴 발견 군집화, 이상치 탐지
    강화 학습 환경과 상호작용 + 보상 보상을 최대화하는 정책 학습 게임 플레이, 로봇 제어

    생성형 AI는 무엇이 다른가

    생성형 AI는 새로운 콘텐츠(텍스트/이미지/음성/코드 등)를 생성하는 모델을 말한다. “AI가 결과를 낸다 = 생성형”은 아니고, 예를 들어 분류 모델은 ‘스팸/정상’ 같은 라벨을 내지만 이건 생성형 AI라고 부르지 않는다.

    • 생성형 예시: 문장 생성, 요약/번역 생성, 이미지 생성, 코드 생성
    • 비생성형 예시: 점수 예측(회귀), 라벨 분류(분류), 이상치 여부 판단

    실무에서 이미 쓰이는 AI 사례 1가지

    대표적으로 금융/결제 분야에서는 이상 거래(부정 사용) 탐지에 AI가 널리 쓰인다. 카드/계좌 거래는 건수가 많고 패턴이 복잡해서, 단순 규칙만으로는 놓치는 경우가 생긴다. 그래서 과거 거래 데이터(금액, 시간대, 지역, 사용처, 빈도 등)를 기반으로 “평소와 다른 패턴”을 빠르게 감지해 추가 인증을 요청하거나 거래를 보류하는 방식으로 운영된다.

    LLM, 임베딩, 트랜스포머를 한 번에 정리

    • LLM(Large Language Model)
      방대한 텍스트로 학습되어, 입력 문맥을 바탕으로 다음 토큰(단어 조각)을 예측하는 방식으로 언어를 다룬다. 그 결과로 답변 생성, 요약, 번역, 코드 작성 같은 작업이 가능해진다.
    • 임베딩(Embedding)
      문장/단어를 숫자 벡터로 바꾸는 것. 의미가 비슷한 문장은 벡터 공간에서도 가깝게 위치한다.
      검색, 추천, 군집화, 유사도 비교(코사인 유사도 등)에 자주 쓰인다.
    • 트랜스포머(Transformer)
      문장 안에서 어떤 단어가 어떤 단어와 관련이 큰지 “주의(attention)”를 주는 방식으로 문맥을 처리한다. LLM은 트랜스포머 구조를 기반으로 동작한다.

    정리하면, 임베딩은 “의미를 숫자로 옮겨 담는 방법”이고, 트랜스포머는 그 숫자(토큰 임베딩)를 이용해 문맥을 계산하는 방식이며, LLM은 트랜스포머 기반으로 다음 토큰을 예측해서 언어를 생성/이해하는 모델이다.

    Fine-Tuning vs RAG

    구분 핵심 아이디어 장점 단점
    Fine-Tuning 모델 가중치를 추가 데이터로 미세 조정 특정 도메인/말투/작업에 강해짐 데이터 준비/평가가 필요, 비용 발생, 잘못하면 성능이 흔들릴 수 있음
    RAG 외부 문서를 검색해(임베딩 기반) 답변 근거로 넣고 생성 최신/사내 문서 반영 가능, 별도 재학습 없이 업데이트 쉬움 검색 품질이 성능을 좌우, 인프라(벡터DB/파이프라인) 비용·복잡도 증가

    실무 관점에서 한 줄 결론

    • Fine-Tuning: “모델 자체를 특정 업무에 맞게 바꾸고 싶다”에 강함
    • RAG: “근거가 되는 최신/사내 데이터를 참고하게 하고 싶다”에 강함
    • 그리고 Temperature는 모델을 재학습하는 게 아니라, 답변을 “더 보수적으로/더 창의적으로” 뽑는 추론(생성) 설정 값이다.

    오늘의 회고와 내일 계획

    오늘 느낀 점

    SQL은 “쿼리를 아는 것”과 “리포트 형태로 뽑아내는 것”이 느낌이 다르다는 걸 다시 느꼈다. 특히 피벗과 윈도우 함수는 실무 데이터에서 바로 쓰이는 패턴이라, 오늘처럼 개념을 분리해서 정리해 두는 게 도움이 될 것 같다. AI는 용어가 많아서 헷갈리기 쉬운데, 학습 방식 → 생성형 → LLM 구성 요소 → Fine-Tuning/RAG 순서로 연결하니 흐름이 잡혔다.

    내일 계획

    • 피벗/윈도우 함수를 실제 데이터(내 프로젝트 테이블)에도 한 번씩 대입해 보기.
    • DATE_FORMAT과 DATE/시간 비교를 섞어서 “월별/주별 리포트” 쿼리 템플릿 만들기.
    • RAG 흐름(문서 수집 → 청크 → 임베딩 → 검색 → 프롬프트)에 대해 간단한 다이어그램으로 정리해 보기.
