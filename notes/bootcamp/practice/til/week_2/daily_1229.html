    내일배움캠프 Spring 백엔드 11일차 TIL — SQL 퀘스트 & 프롬프트 엔지니어링

    작성일 2025-12-26 · 분류: 부트캠프, TIL, SQL, AI

    11일차 개요 — SQL 조인·집계 문제풀이와 프롬프트 기법 정리

    정리

    오늘은 SQL 퀘스트(조인·집계·서브쿼리/윈도우 함수)를 먼저 풀이하고, 이어서 프롬프트와 프롬프트 엔지니어링의 핵심 개념을 정리했다. 실무 관점에서는 “정답 쿼리”도 중요하지만, 데이터가 빠져도(LEFT JOIN), 중복이 생겨도(집계), 결과가 흔들려도(정렬/조건) 안정적으로 원하는 결과를 뽑아내는 습관이 더 중요하다고 느꼈다.

    • 고객/주문/상품 데이터를 조인해 총 구매금액과 주문 건수 집계를 만들었다.
    • 카테고리별 “최다 판매 상품”을 구하면서 윈도우 함수를 활용했다.
    • AI 파트에서는 프롬프트 구성 요소와 대표 프롬프팅 기법(Zero/Few-shot 등)을 정리했다.

    SQL 퀘스트 — 고객·주문·상품 조인(캠프 커머스 데이터)

    예시 데이터

    고객 정보를 담은 camp_customers 테이블 임시 데이터는 다음과 같다.

    customer_id customer_name country
    1AliceUSA
    2BobUK
    3CharlieUSA

    상품 정보를 담은 camp_products 테이블 임시 데이터는 다음과 같다.

    product_id product_name category price
    1LaptopElectronics1000
    2SmartphoneElectronics800
    3HeadphonesElectronics150
    4Coffee MakerHome200
    5BlenderHome100

    주문 정보를 담은 camp_orders 테이블 임시 데이터는 다음과 같다.

    order_id product_id order_date quantity customer_id
    10112024-02-0121
    10232024-02-0212
    10322024-02-0311
    10442024-02-0433
    10512024-02-0512
    10652024-02-0623

    풀이한 쿼리

    1. 고객 이름, 총 구매 금액(가격×수량 합), 고객이 실제 주문한 주문서 수를 출력하는 쿼리

    SELECT
  c.customer_name AS "CustomerName",
  COALESCE(SUM(p.price * o.quantity), 0) AS "TotalAmount",
  COUNT(o.order_id) AS "OrderCount"
FROM camp_customers c
LEFT JOIN camp_orders o ON c.customer_id = o.customer_id
LEFT JOIN camp_products p ON o.product_id = p.product_id
GROUP BY c.customer_id, c.customer_name;

    2. 각 제품 카테고리별로 가장 많이 팔린 제품의 이름과 총 판매량을 조회하는 쿼리(윈도우 함수 활용)

    SELECT
  t.category AS "Category",
  t.product_name AS "Top_Product",
  t.total_sold AS "TotalSold"
FROM (
  SELECT
    p.category,
    p.product_name,
    SUM(o.quantity) AS total_sold,
    ROW_NUMBER() OVER (
      PARTITION BY p.category
      ORDER BY SUM(o.quantity) DESC, p.product_id ASC
    ) AS rn
  FROM camp_products p
  JOIN camp_orders o ON p.product_id = o.product_id
  GROUP BY p.category, p.product_id, p.product_name
) t
WHERE t.rn = 1;

    SQL 퀘스트 — 부서별 최고 연봉 & 프로젝트 예산 필터

    예시 데이터

    직원 정보를 담은 camp_employees 테이블 임시 데이터는 다음과 같다.

    employee_id name department salary
    1AliceHR5000
    2BobIT7000
    3CharlieIT6000
    4DavidHR4500
    5EveSales5500
    6FrankIT7200

    프로젝트 정보를 담은 camp_projects 테이블 임시 데이터는 다음과 같다.

    project_id project_name budget
    101Alpha10000
    102Beta15000
    103Gamma12000
    104Delta8000

    직원-프로젝트 매핑을 담은 camp_employee_projects 테이블 임시 데이터는 다음과 같다.

    employee_id project_id
    2102
    3102
    4103
    6102
    6103

    풀이한 쿼리

    3. 각 직원이 속한 부서에서 가장 높은 월급을 받는 직원들만 조회하는 쿼리

    SELECT
  e.name AS "Name",
  e.department AS "Department",
  e.salary AS "Salary"
FROM camp_employees e
WHERE e.salary = (
  SELECT MAX(e2.salary)
  FROM camp_employees e2
  WHERE e2.department = e.department
);

    4. 직원이 참여한 프로젝트 중 예산이 10,000 이상인 프로젝트만 조회하는 쿼리(직원 이름/프로젝트/예산)

    SELECT
  e.name AS "Name",
  p.project_name AS "ProjectName",
  p.budget AS "Budget"
FROM camp_employees e
JOIN camp_employee_projects ep ON e.employee_id = ep.employee_id
JOIN camp_projects p ON ep.project_id = p.project_id
WHERE p.budget >= 10000
ORDER BY p.budget DESC, e.name ASC;

    AI 정리 — 프롬프트와 프롬프트 엔지니어링

    핵심 개념

    • 프롬프트(Prompt)
      LLM에게 입력하는 모든 메시지(지시/질문/데이터/맥락)를 의미한다. 모델의 역할(정체성), 목표(주 임무), 출력 방식을 정해주는 “설계도”로 볼 수 있다.
    • 토큰(Token)
      문장이 처리되는 단위(단어보다 더 작은 조각일 수 있음)다. 모델마다 한 번에 처리 가능한 최대 토큰(컨텍스트 길이)이 다르며, 입력이 길어질수록 비용/지연/오류 가능성이 커질 수 있다.
    • Temperature(자유도)
      출력의 “랜덤성/창의성”을 조절하는 값으로, 높을수록 다양한 표현이 나오지만 사실 기반 작업에서는 불필요한 추측이 늘 수 있다. 낮을수록 더 보수적이고 일관된 답을 내는 경향이 있다.
    • Prompt Engineering
      원하는 답을 얻기 위해 프롬프트를 구조화/개선/실험하는 활동이다. (명령을 더 명확히, 예시를 추가, 출력 포맷을 강제, 검증 단계를 포함 등)

    프롬프트 구성 요소(필요할 때 선택)

    요소 의미 예시(요약)
    지시(Instruction) 모델이 해야 하는 명령 “아래 표를 요약해줘”
    입력값(Input) 모델이 참고할 데이터 텍스트/표/코드/이미지
    문맥(Context) 추가 정보, 어조/목표 “면접용으로 간결하게”
    출력 지시자(Output) 출력 형식/틀 “JSON으로, 키는 …”
    예시(Examples) 원하는 답의 샘플 “예시 2개를 따라 작성”

    대표 프롬프팅 기법

    기법 한 줄 정의 장점 주의/단점
    Zero-shot 예시 없이 지시만으로 해결 빠르고 간단 모호하면 결과 흔들림
    Few-shot 소수의 예시로 패턴을 학습시키기 출력 품질/일관성 상승 예시 설계가 어렵고 길어질 수 있음
    Chain-of-Thought 단계적 추론을 유도 복잡한 문제에서 실수 감소 장황해질 수 있어 출력 규칙이 필요
    Tree of Thoughts 여러 경로를 탐색해 최적안을 선택 대안 탐색에 강함 검증/선택 기준이 없으면 흔들림
    Structured Prompting 절차/형식을 강제해 수행 업무 자동화에 유리 사람이 ‘기계가 읽기 좋게’ 써야 함
    Generated Knowledge 답변 전에 관련 지식을 먼저 생성 추론 기반이 생겨 정확도↑ 잘못된 지식이면 오류가 증폭될 수 있음

    Hallucination(환각) & Red Flag

    환각(Hallucination)은 모델이 “그럴듯하지만 사실이 아닌 내용”을 생성하는 현상이다. 오작동이라기보다, 다음 토큰을 예측하는 과정에서 근거가 부족한데도 답을 만들어내는 성향 때문에 발생한다.

    • Red Flag(위험 신호) 예시
      • 근거/출처가 필요한데도 “단정”하는 문장
      • 숫자/날짜/고유명사에서 말이 바뀌거나 모순이 생김
      • 요구한 형식과 다른 결과(예: JSON 요청했는데 서술형)
      • “정확히는 모르지만…” 없이 추측을 사실처럼 말함
    • 대응 방법
      • 검증 단계 포함: “근거를 함께 제시해” / “모르는 건 모른다고 말해”
      • 출력 형식 강제: 표/JSON/목록 등으로 구조를 고정
      • 중요한 사실은 재질문/교차검증(다른 자료/문서로 확인)
      • 사실 기반 작업은 Temperature를 낮추고, 추측 금지 규칙을 추가

    오늘의 회고와 내일 계획

    오늘 느낀 점

    SQL은 “정답 쿼리”를 외우는 게 아니라, 데이터가 부족한 경우(LEFT JOIN), 집계 기준(GROUP BY), 최댓값 추출(윈도우/서브쿼리)처럼 실무에서 자주 부딪히는 상황을 안정적으로 처리하는 훈련이 핵심이라는 걸 다시 느꼈다. 특히 카테고리별 1등을 뽑는 문제는, 윈도우 함수를 알면 훨씬 깔끔하게 표현할 수 있었다.

    AI 파트는 “프롬프트를 잘 쓰는 것”이 단순히 말 잘하는 게 아니라, 역할/입력/문맥/출력 규칙을 설계해서 결과를 통제하는 기술이라는 점이 인상적이었다. 앞으로는 업무형 질문일수록 구조를 먼저 잡고(Structured Prompt), 검증 단계까지 포함하는 습관을 들여야겠다.

    내일 계획

    • 윈도우 함수(ROW_NUMBER, RANK, DENSE_RANK) 차이를 예제로 다시 정리하기.
    • LEFT JOIN + 집계 조합에서 COUNT, SUM, COALESCE 패턴을 더 연습하기.
    • 프롬프트를 “업무 템플릿”으로 만들어 보고(출력 포맷 고정), 결과 품질 비교하기.
