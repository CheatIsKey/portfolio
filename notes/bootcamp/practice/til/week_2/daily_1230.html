    내일배움캠프 Spring 백엔드 12일차 TIL — 웹 개발 기초(Frontend/Backend, HTTP, REST)

    작성일 2025-12-29 · 분류: 부트캠프, TIL, Web, HTTP, REST, API

    12일차 개요 — 웹이 동작하는 전체 흐름을 큰 그림으로 정리

    정리

    오늘은 웹 서비스를 레스토랑에 비유해서 프론트엔드/백엔드의 역할을 구분하고, 웹의 기본 구성 요소(HTML/CSS/JavaScript), 서버와 클라우드, HTTP 요청/응답 구조, 그리고 REST API 설계/명세의 핵심을 흐름 중심으로 정리했다.

    • 사용자(클라이언트) → HTTP 요청 → 서버 처리 → 응답(HTML/JSON 등)이라는 흐름을 한 번에 연결했다.
    • 정적/동적 처리, Web Server/Application Server의 역할을 구분했다.
    • REST 원칙과 API 명세 작성법(메서드/엔드포인트/파라미터/응답)을 표로 정리했다.

    프론트엔드와 백엔드 — 레스토랑 비유로 이해하기

    큰 그림

    웹 서비스를 레스토랑에 비유하면, 손님이 메뉴판을 보고 주문하는 공간이 프론트엔드이고, 주문이 접수되어 조리/재료관리/서빙 준비를 하는 주방이 백엔드다. 프론트엔드는 사용자가 직접 보고 조작하는 영역(UI/UX)을 책임지고, 백엔드는 요청을 받아 핵심 로직 실행, 데이터 저장/조회, 결과 응답을 담당한다.

    구분 어디서 실행? 핵심 책임 예시
    프론트엔드(Frontend) 클라이언트(브라우저/앱) UI 구성, 사용자 입력 처리, 화면 갱신 상품 목록 화면, 버튼 클릭, 폼 입력
    백엔드(Backend) 서버(항상 켜진 환경) 인증/인가, 비즈니스 로직, DB 처리, API 응답 주문 생성, 결제 검증, 재고 차감
    API 클라이언트 ↔ 서버 사이 요청/응답 규칙(계약) GET /items, POST /orders

    웹(Web)과 인터넷(Internet) — URL/HTTP/HTML로 정의하기

    핵심 정의

    개념 한 줄 정의 포인트
    인터넷(Internet) 전 세계 컴퓨터를 연결한 거대한 네트워크 물리/논리적으로 “연결” 자체
    웹(Web) 인터넷 위에서 데이터를 “주소로 찾고 규약으로 주고받는” 서비스 URL + HTTP + (HTML/JSON 같은 표현)
    • URL: 웹 자원을 가리키는 고유 주소
    • HTTP: 요청(Request)과 응답(Response)을 주고받는 통신 규약
    • HTML: 브라우저가 화면을 그릴 수 있게 “구조”를 표현하는 문서 형식

    클라이언트와 웹 브라우저 — 화면을 그리는 주체

    클라이언트란?

    클라이언트는 서버에 요청을 보내는 프로그램이다. 대표적으로 웹 브라우저와 모바일 앱이 있다. 웹 브라우저는 HTML/CSS/JavaScript를 해석해 화면을 그리며, 모바일 앱도 HTTP로 서버와 통신하지만 화면을 그리는 방식은 네이티브 UI(또는 WebView 등)일 수 있다.

    HTML — 문서의 구조를 만드는 언어

    핵심 개념

    • HTML은 HyperText Markup Language로, 브라우저에서 웹 페이지의 구조를 정의한다.
    • HyperText: 링크를 통해 다른 문서/자원으로 이동할 수 있는 텍스트
    • Markup: 태그로 의미/구조를 표시해 브라우저가 해석하도록 하는 방식
    용어 의미 예시
    태그(Tag) 문서의 특정 부분을 의미/구조로 표시 <h1>제목</h1>
    속성(Attribute) 태그의 동작/정보를 세부 설정 <img src="a.png" alt="...">
    요소(Element) 태그 + 속성 + 내용(필요 시)을 합친 단위 <a href="...">링크</a>

    <head>에는 화면에 직접 보이지 않는 설정(메타 정보, CSS/JS 연결 등)이 들어가고, <body>에는 사용자에게 보이는 콘텐츠가 들어간다.

    <!doctype html>
<html lang="ko">
<head>
  <meta charset="UTF-8" />
  <title>페이지 제목</title>
  <link rel="stylesheet" href="/styles.css" />
  <script src="/app.js" defer></script>
</head>
<body>
  <h1>Hello</h1>
</body>
</html>

    CSS — 디자인과 배치를 담당하는 규칙

    핵심 정리

    • CSS(Cascading Style Sheets)는 스타일(색/크기/여백/레이아웃)을 정의한다.
    • 일반적으로 유지보수에 유리한 외부 파일 연결을 권장한다.
    <link rel="stylesheet" href="/style.css" />

    CSS 기본 구조는 “누구를(선택자), 어떻게(선언)” 꾸밀지 정의하는 규칙이다.

    /* 선택자 { 속성: 값; } */
.card {
  padding: 12px;
  border: 1px solid #ddd;
}

    박스 모델 & 레이아웃

    구성 의미 기억 포인트
    Content 실제 내용 영역 텍스트/이미지
    Padding 안쪽 여백 내용과 테두리 사이
    Border 테두리 경계선
    Margin 바깥쪽 여백 다른 요소와의 거리
    속성 의미 대표 사용
    display: block 한 줄을 차지 div, p
    display: inline 자기 크기만큼 span, a
    display: flex 자식 요소 정렬에 강함 가로 배치/정렬
    display: none 화면에서 제거 조건부 UI

    JavaScript — 동작(상호작용)과 통신을 담당

    역할

    • 사용자 이벤트 처리(클릭/입력 등)
    • DOM 제어(요소 생성/수정/삭제)
    • 서버 통신(fetch 등)으로 API 호출

    브라우저에서는 클라이언트 사이드에서 실행되며, Node.js 같은 런타임을 사용하면 서버 사이드에서도 실행할 수 있다.

    <script src="/app.js" defer></script>

    // 이벤트 리스너 예시
const btn = document.querySelector('#save-btn');
btn.addEventListener('click', () => {
  console.log('clicked');
});

    서버와 정적/동적 처리 — Web Server와 Application Server

    서버란?

    서버는 “항상 켜져 있으면서 요청을 처리하는 환경”이다. 하드웨어 관점의 서버(컴퓨터) 위에서, 소프트웨어 관점의 서버(웹/애플리케이션 프로그램)가 동작한다. 실제 서비스에서는 고정 IP 자체보다 도메인(고정된 접점)을 통해 안정적으로 접근하도록 구성하는 경우가 많다.

    구분 무엇을 응답? 예시
    정적(Static) 가공 없이 파일 그대로 HTML/CSS/JS, 이미지
    동적(Dynamic) 로직 실행 후 결과 생성 JSON API, 서버 렌더링 결과

    일반적으로 정적 콘텐츠는 Web Server(Nginx 등)가 빠르게 처리하고, 동적 요청은 Application Server(Spring Boot 등)가 비즈니스 로직과 DB 처리를 수행한다.

    HTTP — 요청/응답 규약과 메시지 구조

    동작 원칙

    • 클라이언트가 먼저 요청(Request)을 보낸다.
    • 서버는 요청을 처리하고 응답(Response)을 보낸다.

    HTTP 특징

    특징 의미 왜 필요한가
    무상태성(Stateless) 요청 간 이전 상태를 기본적으로 저장하지 않음 확장성/부하 관리에 유리
    비연결성(Connectionless) 요청-응답 후 연결을 정리하는 방식 서버 자원 효율

    요청/응답 메시지 구성

    구성 설명 예시
    Start Line 메서드/경로/HTTP 버전 등의 시작 정보 GET /items HTTP/1.1
    Headers 부가 정보(형식, 인증, 캐시 등) Content-Type, Authorization
    Body(선택) 전송 데이터(주로 POST/PUT/PATCH) JSON 본문

    메서드 & 상태 코드

    메서드 의미 대표 용도
    GET조회목록/단건 조회
    POST생성리소스 생성
    PUT / PATCH수정전체/부분 수정
    DELETE삭제리소스 삭제
    범위 의미 예시
    2xx성공200 OK, 201 Created
    3xx리다이렉트301, 302
    4xx클라이언트 오류400 Bad Request, 401/403, 404
    5xx서버 오류500 Internal Server Error

    데이터 전송과 JSON — Query/Path/Body 구분하기

    전송 방식

    방식 형태 주로 쓰는 상황
    Path /items/123 특정 자원 식별
    Query /items?sort=new&page=2 검색/필터/정렬/페이징
    Body (본문 JSON 등) 생성/수정처럼 데이터가 많을 때

    HTML form은 기본적으로 application/x-www-form-urlencoded 또는 multipart/form-data로 전송할 수 있다. JSON으로 보내고 싶으면, 제출을 JavaScript로 가로채서 fetch로 JSON 본문을 전송하는 방식이 흔하다. 

    // JSON 요청 예시(fetch)
const API_URL = 'https://example.com/posts';

const payload = { title: '제목', content: '내용' };

const response = await fetch(API_URL, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(payload),
});

    JSON(JavaScript Object Notation)은 텍스트 기반의 가벼운 데이터 형식으로, 클라이언트-서버 간 데이터 교환에서 가장 흔히 사용된다.

    API와 REST — 설계 원칙과 RESTful

    API란?

    API(Application Programming Interface)는 프로그램들이 서로 소통하기 위한 접점(규약)이다. 프론트엔드/모바일 앱/외부 시스템 등 소비자가 예측 가능한 방식으로 서버 기능을 사용하도록 “계약”을 제공한다.

    REST 핵심

    REST(Representational State Transfer)는 웹 자원을 URI로 식별하고, HTTP 표준에 맞게 자원의 표현(Representation)을 주고받는 설계 원칙이다.

    원칙 요약
    균일한 인터페이스자원은 URI로 식별, 표현으로 조작, 메시지는 자체 설명적
    클라이언트-서버 분리UI/UX와 데이터/로직의 책임 분리
    무상태성서버는 요청 간 클라이언트 상태를 기본적으로 저장하지 않음
    캐시 가능응답이 캐시 가능한지 명시해 성능 개선
    계층형 시스템중간 계층(프록시/게이트웨이)을 두어도 일관된 인터페이스 유지
    Code on Demand(선택)필요 시 실행 가능한 코드를 내려줄 수 있음(옵션)

    API 설계와 명세서 — “계약서”를 먼저 만든다

    왜 명세가 필요한가

    • 일관성(Consistency)과 예측 가능성(Predictability)을 확보한다.
    • 프론트/백이 서로 기다리지 않고 동시에 개발할 수 있다.
    • 테스트(Postman)와 협업(공유/문서화)이 쉬워진다.

    명세 체크리스트

    항목 설명 예시
    Method 무엇을 할 것인가(행동) GET / POST / PUT / DELETE
    Endpoint 어떤 자원인가(대상) /posts, /orders
    Params 필요한 입력(경로/쿼리/본문) /posts/1, ?page=2, JSON body
    Headers 부가 정보/인증/형식 Content-Type, Authorization
    Response 상태 코드 + 응답 형식 200 + JSON, 404

    REST API 예시(블로그 게시물) 

    GET    /posts         // 목록 조회
GET    /posts/{id}     // 단건 조회
POST   /posts         // 생성
PUT    /posts/{id}     // 수정(전체)
PATCH  /posts/{id}     // 수정(부분)
DELETE /posts/{id}     // 삭제
    • 엔드포인트는 동사가 아니라 명사(자원)로 표현하고, 보통 복수형을 사용한다.
    • 생성 성공은 보통 201 Created, 삭제 성공은 보통 204 No Content를 사용한다.

    개발 도구 — Postman, Mock Server, Collection

    실무에서 쓰는 이유

    도구 역할 언제 유용한가
    Postman HTTP 요청 테스트/공유 프론트 없이도 API 검증
    Mock Server 명세대로 응답하는 가짜 서버 백엔드 준비 전 프론트 개발
    Collections 요청 묶음(폴더) 자원 단위로 정리/재사용

    오늘의 회고

    오늘 느낀 점

    오늘 정리는 “용어를 아는 것”보다 요청-처리-응답 흐름을 머릿속에서 연결하는 데 도움이 됐다. 특히 API 명세를 계약서로 보고 먼저 정리해두면, 프론트/백 협업에서 기다림이 줄고 결과물도 예측 가능해진다는 점이 확실히 와닿았다.
