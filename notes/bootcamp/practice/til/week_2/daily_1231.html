    13일차 개요 — “맛집 관리 API 설계하기” 과제 진행

    정리

    오늘은 웹개발 1차 과제(맛집 관리 API 설계)를 진행했다. 실제 백엔드 구현 전에 API 명세서를 먼저 작성하고, Postman으로 Mock Server까지 구성해 “명세 기반 개발(Contract First)” 흐름을 경험하는 데 집중했다.

    • API 3개를 설계했다: 전체 조회(GET), 등록(POST), 삭제(DELETE)
    • Postman Collection을 만들고, 요청별 Example로 응답을 정의해 Mock Server를 만들었다.
    • 추가로 v0를 활용해 “맛집 관리 웹페이지” UI 프로토타입을 생성했다(프론트 챌린지).

    맛집 관리 API 설계 — 요구사항과 데이터 모델

    요구사항

    • 전체 조회: 등록된 맛집 목록을 배열로 응답한다.
    • 맛집 등록: 이름/주소/전화번호/카테고리/평점 정보를 받아 등록 결과를 응답한다.
    • 맛집 삭제: 특정 id를 기준으로 삭제 요청을 받고, 처리 결과를 응답한다.

    데이터 모델 (Place)

    필드 설명 예시
    id 맛집 식별자 112
    name 맛집 이름 돌솥비빔밥
    address 주소 서울 종로구 효...
    call 전화번호 02-123-4567
    category 카테고리 한식
    rating 평점 4.8

    Postman Mock Server — Collection과 Example 기반 응답 구성

    구성 포인트

    • Collection을 만들고, 각 API 요청(GET/POST/DELETE)을 등록했다.
    • 각 요청에 대해 Examples를 추가해 “요청이 들어오면 어떤 응답을 줄지”를 미리 정의했다.
    • Mock Server를 Collection과 연결해, 실제 서버 없이도 동일한 URL로 호출 테스트가 가능하도록 만들었다.

    Example 작성 팁 (Mock에서 자주 쓰는 값)

    • {{$randomInt}} : 임의 정수 생성(등록 응답 id 등에 활용)
    • {{$body '필드명' ''}} : 요청 바디의 특정 값을 응답에 그대로 반영

    POST /api/places Example 바디 예시

    {
  "id": {{$randomInt}},
  "name": "{{$body 'name' ''}}",
  "address": "{{$body 'address' ''}}",
  "call": "{{$body 'call' ''}}",
  "category": "{{$body 'category' ''}}",
  "rating": {{$body 'rating' ''}}
}

    Mock Server는 “명세대로 응답을 돌려주는 서버”이기 때문에, 등록(POST) 후 목록(GET)에 자동으로 누적되는 형태의 영속 저장은 기본적으로 제공되지 않는다. 따라서 실제 서비스처럼 동작시키려면, 이후 단계에서 백엔드 구현/DB 연동이 필요하다.

    프론트엔드 챌린지 — v0로 UI 생성하고 연동 포인트 정리

    v0 프롬프트 요약

    • “맛집 관리 웹페이지”를 만들고, 목록 조회/등록/삭제가 가능한 UI를 요청했다.
    • API는 baseUrl 환경변수를 사용하고, 아래 3개만 호출하도록 제한했다.
    GET    ${baseUrl}/api/places
POST   ${baseUrl}/api/places
DELETE ${baseUrl}/api/places/{id}

    산출물 정리

    • Postman Collection: 맛집관리_API.postman_collection.json
    • 프론트엔드 소스: restaurant-management-frontend.zip

    요청-응답 흐름 — 조회/등록/삭제 시나리오로 정리

    1) 조회 흐름 (GET)

    • 클라이언트가 GET /api/places 요청
    • 서버(또는 Mock)가 맛집 목록(JSON 배열)을 응답
    • 프론트는 응답을 받아 리스트 UI로 렌더링

    2) 등록 흐름 (POST)

    • 사용자가 폼에 값 입력 후 등록 버튼 클릭
    • 클라이언트가 POST /api/places + JSON 바디 전송
    • 서버(또는 Mock)가 등록 결과(JSON)를 응답
    • 성공 시, 다시 GET을 호출하거나 화면 상태를 갱신해 목록을 최신화

    3) 삭제 흐름 (DELETE)

    • 사용자가 특정 맛집의 삭제 버튼 클릭
    • 클라이언트가 DELETE /api/places/{id} 요청
    • 서버(또는 Mock)가 성공 여부를 응답
    • 성공 시, 목록 재조회 또는 화면 상태 갱신

    오늘의 회고

    오늘 느낀 점

    오늘은 “구현”보다 “설계”를 먼저 해보면서, API 명세가 곧 프론트-백엔드 계약서라는 감각이 확실해졌다. Postman Mock Server 덕분에 실제 서버가 없어도 요청/응답 흐름을 빠르게 검증할 수 있었고, UI(클라이언트)에서 어떤 데이터가 필요하고 어떤 응답 형식이 편한지도 동시에 점검할 수 있었다.
