    프로그래머스 118666 — 성격 유형 검사하기

    구분 내용
    키워드 구현, 누적 점수, 선택지 점수 매핑, 동점 처리
    핵심 접근 diff = choices[i] - 4로 (-3..+3) 변환 후, diff<0이면 첫 글자 점수에 -diff 누적, diff>0이면 둘째 글자 점수에 diff 누적. 마지막에 (R/T, C/F, J/M, A/N) 쌍 비교로 결과 문자열 생성(동점이면 알파벳 빠른 쪽).
    자주 터지는 실수 점수 배열을 직접 만들다가 5~7점 매핑이 꼬이는 버그가 나기 쉬움 → choices - 4가 가장 안전. 동점 처리(사전순 선택)를 반대로 쓰기 쉬움.
    깨달은 점 “매핑 규칙이 보이면 수식으로 줄여라.” → 구현 실수 가능성을 크게 낮춘다.

    프로그래머스 92334 — 신고 결과 받기

    구분 내용
    키워드 중복 신고 제거, id_list 순서 고정, Set/Map
    핵심 접근 (1) reporter -> Set<target>으로 “누가 누구를 신고했는지” 저장
    (2) Set에 “새로 추가될 때만” target 신고 수 증가
    (3) cnt[target] >= k인 target은 정지 대상
    (4) reporter별로 “정지 대상에게 한 신고 수”를 세어 메일 수 결정
    (5) 정답은 반드시 id_list 순서대로 채우기
    자주 터지는 실수 Set 중복인데도 신고 수를 올려버리면 오답. Map 순회로 결과를 만들면 순서가 흔들릴 수 있음 → for (i in id_list)로 채우기.
    깨달은 점 “중복 제거”가 문제의 본체. Set 추가 성공 여부를 기준으로 카운트하는 습관이 중요.

    프로그래머스 87389 — 나머지가 1이 되는 수 찾기

    구분 내용
    키워드 브루트포스, 최소값 탐색
    핵심 접근 x=2부터 증가시키며 n % x == 1인 첫 x를 즉시 반환
    자주 터지는 실수 x=1부터 시작(의미 없는 검사), “최소”인데 끝까지 다 돌고 min 갱신하는 불필요한 구현
    깨달은 점 최소값 문제는 “작은 것부터 탐색 + 첫 발견 즉시 return”이 가장 명확하고 안전하다.

    백준 1124 — 언더프라임

    구분 내용
    키워드 소인수분해, 소인수 개수(중복 포함), 개수의 소수 판정, SPF(최소 소인수)
    핵심 접근 수 x의 소인수 “개수” cnt(x)를 구하고, 그 값이 소수면 언더프라임.
    빠른 방식은 spf[](최소 소인수)를 체로 만들고 cnt[n] = cnt[n / spf[n]] + 1로 2..B까지 누적 계산 후 A..B를 검사.
    자주 터지는 실수 소인수분해 과정에서 루프용 변수를 직접 나눠버려 흐름이 꼬이는 실수. 남은 소인수 처리(마지막에 n>1이면 cnt++) 누락. cnt=1을 소수로 착각(소수는 2부터).
    깨달은 점 “판정 기준”이 숫자 자체가 아니라 “소인수 개수”라는 점을 놓치면 문제 전체가 흔들린다.

    백준 1141 — 접두사

    구분 내용
    키워드 정렬, 인접 비교, startsWith, 중복 제거
    핵심 접근 (1) 중복 제거(Set) → (2) 사전순 정렬 → (3) 인접한 두 단어만 비교.
    정렬 후에는 어떤 단어가 다른 단어의 접두사라면, 그 단어 바로 뒤쪽에 접두사를 공유하는 단어가 연속으로 온다.
    따라서 word[i+1].startsWith(word[i])이면 word[i]는 최대 집합에서 제외.
    자주 터지는 실수 substring으로 접두사 판정 시 경계 조건 실수 → startsWith가 가장 안전. “남긴 개수”를 출력해야 하는데 전체 개수를 출력하는 실수.
    깨달은 점 정렬 + 인접 비교로 O(N log N)에서 깔끔하게 끝난다(불필요한 전수 비교 제거).

    스프링 빈 조회 — ApplicationContext, 조회 방식, 예외/테스트

    기본 세팅 (컨테이너 생성)

    ApplicationContext(보통 변수명 ac)는 스프링 컨테이너를 의미한다. 아래 코드는 AppConfig.class 설정을 읽어 @Bean으로 등록된 빈을 컨테이너에 올리고 관리한다. 

    ApplicationContext ac =
    new AnnotationConfigApplicationContext(AppConfig.class);

    빈 조회 방법 정리 (가장 기본)

    방법 예시 특징
    이름 + 타입 ac.getBean("memberService", MemberService.class) 가장 명확하고 안전
    타입만 ac.getBean(MemberService.class) 동일 타입이 2개 이상이면 예외 발생 가능
    구현체 타입 조회 ac.getBean(MemberServiceImpl.class) 가능은 하지만 구현체 의존이 강해져 권장되지 않음

    조회 실패/중복 시 예외

    상황 예외 해결
    빈이 존재하지 않음 NoSuchBeanDefinitionException 등록 여부 확인, 이름/타입 확인
    타입으로 조회했는데 동일 타입이 2개 이상 NoUniqueBeanDefinitionException 빈 이름을 지정해서 조회하거나, 다건 조회로 변경

    테스트에서 검증하는 패턴 (isInstanceOf / assertThrows)

    1) 조회 결과 타입 검증(AssertJ)

    Object bean = ac.getBean("memberService");
org.assertj.core.api.Assertions.assertThat(bean)
    .isInstanceOf(MemberService.class);

    2) 예외 검증(JUnit)

    org.junit.jupiter.api.Assertions.assertThrows(
    org.springframework.beans.factory.NoSuchBeanDefinitionException.class,
    () -> ac.getBean("noBean", MemberService.class)
);

    특정 타입 “전체” 조회하기 (getBeansOfType)

    특정 타입의 빈을 모두 가져오고 싶다면 ac.getBeansOfType(타입)을 사용한다. 반환값은 Map<String, T> 형태(빈 이름 -> 빈 객체)이다. 

    Map<String, DiscountPolicy> beans =
    ac.getBeansOfType(DiscountPolicy.class);

// beans.forEach((name, bean) -> System.out.println(name + " = " + bean));

    “부모 타입”으로 조회하면 자식 타입도 함께 조회된다

    스프링의 타입 조회는 “할당 가능(다형성)” 기준이다. 예를 들어 인터페이스 타입으로 조회하면, 그 인터페이스를 구현한 구현체 빈도 함께 조회 대상이 된다. (주의: 동일 부모 타입의 빈이 여러 개면, 타입 단독 조회는 중복 예외가 날 수 있다.)

    ApplicationContext 계층 구조 (큰 그림) 

    <<interface>> BeanFactory
        ▲
        │
<<interface>> ApplicationContext
        ▲
        │
AnnotationConfigApplicationContext (구현체)

    BeanFactory는 스프링 컨테이너의 최상위 인터페이스로 “빈 관리/조회”의 핵심 기능을 제공한다. ApplicationContext는 여기에 더해 애플리케이션에 필요한 부가기능(국제화, 환경 정보, 이벤트, 리소스 조회 등)을 포함한다.

    스프링 빈 설정 메타 정보: BeanDefinition

    스프링은 빈을 만들 때 “설정 메타정보”인 BeanDefinition을 기반으로 생성한다. @Bean(자바 설정)이나 <bean>(XML 설정) 같은 “설정 방식”은 달라도, 컨테이너 내부에서는 BeanDefinition이라는 동일한 추상화로 관리된다.

    예를 들어 AnnotationConfigApplicationContext는 내부에서 설정 클래스를 읽어 BeanDefinition을 만들고, 그 정보를 토대로 빈을 등록/생성한다.

    프레임워크 vs 라이브러리 — 제어 흐름 관점

    핵심 구분

    구분 핵심 감각적인 예시
    프레임워크 프레임워크가 전체 흐름을 제어하고, 필요한 시점에 “내 코드”를 호출한다(IoC). 스프링이 요청 흐름을 잡고, 컨트롤러/서비스를 호출
    라이브러리 내 코드가 흐름을 제어하고, 필요할 때 라이브러리를 호출한다. 내가 필요할 때 문자열/컬렉션 유틸을 직접 호출
