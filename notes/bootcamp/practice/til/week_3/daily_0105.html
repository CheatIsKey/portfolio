    16일차 개요 — 스프링이 “객체를 관리하는 방식”을 중심으로 정리

    정리

    오늘은 싱글톤과 스프링 컨테이너가 싱글톤을 다루는 방식, 그리고 컴포넌트 스캔/자동 주입을 정리했다. “왜 생성자 주입을 권장하는지”, “빈이 여러 개일 때 어떻게 선택하는지”까지 함께 정리해 두었다.

    • 싱글톤 패턴의 구조와 한계, 그리고 스프링이 싱글톤을 관리하는 방식
    • @Configuration이 싱글톤을 안전하게 보장하는 이유(CGLIB 프록시 개념)
    • @ComponentScan + 의존관계 자동 주입(DI) 흐름
    • 주입 방식(생성자/Setter/필드/일반 메서드)과 “빈이 여러 개일 때” 해결법

    싱글톤 패턴 & 스프링 싱글톤

    싱글톤 패턴 개념과 기본 구현

    싱글톤 패턴(Singleton Pattern)은 클래스의 인스턴스가 1개만 생성됨을 보장하고 그 인스턴스를 공유하는 디자인 패턴이다. 전형적인 구현은 static 영역에 인스턴스를 하나 만들어 두고, 외부에서는 getInstance()로만 접근하게 만든다. 

    public class SingletonService {
    private static final SingletonService instance = new SingletonService();

    private SingletonService() { }

    public static SingletonService getInstance() {
        return instance;
    }
}

    싱글톤 패턴의 단점(실무 관점)

    • 구현 코드가 늘어난다(생성자 private, static 필드, getInstance 등)
    • 전역 접근 형태가 되기 쉬워 “사용하는 쪽”이 구현체에 강하게 묶일 수 있다
    • 테스트/초기화/상태 제어가 어려워질 수 있다(특히 전역 상태가 생기면 위험)
    • private 생성자 때문에 상속/확장이 제한된다
    • 유연성이 떨어지고, 설계를 바꾸기 어렵다

    스프링 컨테이너가 해결하는 부분

    스프링은 기본 스코프가 singleton이기 때문에, 별도의 싱글톤 패턴 코드를 직접 만들지 않아도 스프링 컨테이너가 “스프링 빈을 싱글톤으로 관리”한다. 즉, 컨테이너가 생성/관리/주입을 책임지기 때문에 설계가 더 단순해진다.

    싱글톤(공유 객체)에서 가장 중요한 주의점: 무상태(stateless)

    싱글톤 빈은 여러 요청(여러 사용자)이 같은 객체를 공유할 수 있다. 그래서 빈을 상태 유지(stateful)로 만들면 매우 위험하다. 가능한 한 무상태(stateless)로 만들고, 값은 지역변수/파라미터로 처리하는 습관이 필요하다.

    • 특정 클라이언트에 의존하는 필드(예: “현재 사용자”, “최근 주문 금액”)를 두지 않는다
    • 공유 필드에 값을 “저장”하지 않는다(저장해야 한다면 DB/캐시/세션 등 적절한 저장소로)
    • 요청마다 달라지는 값은 지역변수/파라미터로 처리한다
    // (권장) 무상태: 요청 값은 파라미터로 받고, 공유 필드에 저장하지 않는다.
public class OrderService {
    public int calculatePrice(int price, int quantity) {
        return price * quantity;
    }
}

    @Configuration과 싱글톤 보장 — 왜 같은 빈이 1개만 생성될까?

    핵심: CGLIB로 설정 클래스를 “강화(enhance)”한다

    @Configuration을 붙이면 스프링은 설정 클래스를 런타임에 CGLIB 기반으로 강화(프록시/서브클래싱)해서, @Bean 메서드를 호출할 때마다 “이미 생성된 빈이 있으면 그 빈을 반환”하도록 동작을 만든다. 그래서 여러 번 호출되어도 결과적으로 동일한 빈 인스턴스가 유지된다. 

    @Configuration
public class AppConfig {

    @Bean
    public MemberRepository memberRepository() {
        return new MemoryMemberRepository();
    }

    @Bean
    public MemberService memberService() {
        // memberRepository()를 다시 호출해도,
        // @Configuration 프록시가 같은 빈을 꺼내서 주도록 보장한다.
        return new MemberService(memberRepository());
    }
}

    정리하면, 스프링이 직접 싱글톤 패턴 코드를 요구하는 게 아니라, “컨테이너 + 설정 메타정보”로 싱글톤을 자연스럽게 유지하도록 만들어 준다.

    컴포넌트 스캔 & 자동 등록/자동 주입

    @ComponentScan이 하는 일

    @ComponentScan은 지정한 패키지 범위에서 @Component가 붙은 클래스를 찾아 스프링 빈으로 자동 등록한다. 또한 아래 애노테이션들도 모두 @Component를 포함(메타 애노테이션)하고 있어 스캔 대상이 된다.

    • @Component: 컴포넌트 스캔의 기본 대상
    • @Controller: 스프링 MVC 컨트롤러
    • @Service: 비즈니스 로직 계층(의미 부여 목적)
    • @Repository: 데이터 접근 계층(예외 변환 등 스프링 지원과 함께 사용됨)
    • @Configuration: 설정 정보(특히 @Bean 싱글톤 보장에 중요한 역할)

    자동 의존관계 주입(DI) — 생성자 주입 중심

    컴포넌트 스캔으로 빈이 등록되면, 스프링은 빈을 생성하는 과정에서 의존관계도 함께 주입할 수 있다. 보통 생성자 주입을 사용하며, 클래스에 생성자가 1개라면 @Autowired를 생략해도 동작한다. 

    @Component
public class OrderService {
    private final MemberRepository memberRepository;

    // 생성자 1개면 @Autowired 생략 가능
    public OrderService(MemberRepository memberRepository) {
        this.memberRepository = memberRepository;
    }
}

    빈 이름 충돌(동일 이름 등록) 시 동작

    상황 결과(대표 동작)
    자동 빈 등록 vs 자동 빈 등록 동일 이름이면 예외(충돌) 발생
    수동 빈 등록(@Bean) vs 자동 빈 등록(@Component) 환경 설정에 따라 다르지만, 스프링 부트에서는 “실수 방지”를 위해 기본적으로 충돌 시 오류가 나도록 설정되는 경우가 많다 (필요 시 별도 설정으로 오버라이딩을 허용할 수 있음).

    의존관계 주입 4가지 방식과 결론

    주입 방식 비교

    방식 특징 권장도
    생성자 주입 생성 시점에 1번 주입, 불변(final) 가능, 필수 의존관계에 적합 가장 권장
    Setter 주입 수정자 메서드로 주입(선택/변경 가능), 누락 가능성 관리 필요 옵션일 때만
    필드 주입 간결하지만 테스트/설계 유연성에 불리(순수 자바로 단독 테스트 어려움) 비권장
    일반 메서드 주입 한 번에 여러 의존관계를 받을 수 있지만 실사용 빈도가 낮음 드묾

    결론: 생성자 주입을 선택하는 이유

    • 대부분의 의존관계는 애플리케이션 실행 중 바뀌지 않는 편이다(불변 설계가 유리)
    • Setter를 열어두면 실수로 변경될 여지가 생긴다
    • final 필드로 강제하면 “누락”을 컴파일 단계에서 잡을 수 있다
    • 스프링이 없어도 순수 자바로 객체를 조립해서 테스트하기 쉽다

    롬복으로 생성자 주입 더 편하게 쓰기 (@RequiredArgsConstructor)

    핵심 사용법

    롬복의 @RequiredArgsConstructor는 final 필드(또는 @NonNull)를 기준으로 생성자를 자동 생성해 준다. 생성자 주입을 가장 깔끔하게 유지할 수 있는 패턴 중 하나다. 

    @Service
@lombok.RequiredArgsConstructor
public class MemberService {
    private final MemberRepository memberRepository;
}

    조회/주입 대상 빈이 2개 이상일 때 해결 방법

    해결 전략 3가지

    전략 설명 언제 쓰나
    1) @Autowired “이름 매칭” 타입 매칭 실패(동일 타입 여러 개) 시, 파라미터/필드 이름으로 빈 이름을 추가 매칭 간단히 이름으로 구분 가능한 경우
    2) @Qualifier 주입 지점에 “추가 구분자”를 명시해서 특정 빈을 선택 명확히 특정 구현을 선택해야 할 때
    3) @Primary 기본 우선순위를 지정(여러 빈 중 기본으로 선택) 대부분의 케이스는 기본 구현, 일부만 예외 처리할 때

    1) 이름 매칭 예시

    @Component
public class OrderService {
    private final DiscountPolicy rateDiscountPolicy;

    public OrderService(DiscountPolicy rateDiscountPolicy) {
        this.rateDiscountPolicy = rateDiscountPolicy;
    }
}

    2) @Qualifier 예시

    @Component
public class OrderService {
    private final DiscountPolicy discountPolicy;

    public OrderService(@org.springframework.beans.factory.annotation.Qualifier("rateDiscountPolicy")
                        DiscountPolicy discountPolicy) {
        this.discountPolicy = discountPolicy;
    }
}

    3) @Primary 예시

    @Component
@org.springframework.context.annotation.Primary
public class RateDiscountPolicy implements DiscountPolicy { }

    일반적으로 @Primary는 “기본값”을 정하는 용도, @Qualifier는 “명시적으로 특정 구현을 선택”하는 용도로 이해하면 편하다. 그리고 같은 상황이라면 @Qualifier가 @Primary보다 더 강하게 우선된다.

    오늘의 회고

    오늘 느낀 점

    “싱글톤 = 무조건 좋다”가 아니라, 싱글톤을 쓰는 순간부터 무상태 설계가 필수라는 점이 가장 중요했다. 또한 컴포넌트 스캔/자동 주입은 “편리함”이 핵심이지만, 빈이 여러 개일 때는 @Primary/@Qualifier처럼 선택 규칙을 명확히 해두는 습관이 필요하다고 느꼈다.
