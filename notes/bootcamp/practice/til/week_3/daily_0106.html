    17일차 개요 — 왜 Git/GitHub가 필요한가

    정리

    오늘은 Git/GitHub의 개념과 분산 버전관리(DVCS) 구조, 그리고 실무에서 자주 쓰는 명령어/용어/되돌리기 전략(reset·restore·revert)까지 한 번에 정리했다. “왜 필요한가”를 기준으로 큰 그림 → 구조 → 실전 명령어 순서로 정리했다.

    • Git: 변경 이력(버전)을 기록하고 협업을 돕는 도구(VCS)
    • GitHub: Git 저장소를 온라인(원격)에서 보관/공유/협업하는 플랫폼
    • DVCS: 저장소 전체 이력을 각자 로컬에 복제해, 네트워크/서버 장애에도 유연하게 작업

    Git과 GitHub — 역할 차이

    핵심 정의

    구분 의미 핵심 역할
    Git 버전 관리 도구(VCS) 로컬에서 변경 이력(커밋) 기록, 브랜치/병합 관리
    GitHub 원격 저장소 + 협업 플랫폼 저장소 공유, PR 기반 협업, 이슈/리뷰/액션 등 프로젝트 운영

    분산형 버전관리(DVCS) 포인트

    • 중앙 서버만 믿는 구조가 아니라, 로컬에도 전체 이력이 있다.
    • 네트워크가 불안정해도 로컬에서 커밋 단위로 작업을 이어갈 수 있다.
    • 원격은 “공유/백업/협업”의 중심이다.

    Git 작업 영역 — Working Directory / Staging Area / .git

    작업 흐름

    영역 의미 대표 명령
    Working Directory 실제 파일을 편집하는 공간(아직 기록 전) git status, 파일 편집
    Staging Area 커밋에 포함할 변경 사항을 “선별”해 올리는 공간 git add, git restore --staged
    .git (Local Repository) 커밋(스냅샷)과 이력이 저장되는 로컬 저장소 git commit, git log

    핵심 개념

    • commit: 변경 사항을 하나의 “스냅샷”으로 기록하는 단위
    • hash: 커밋을 식별하는 고유 값(보통 앞부분만 축약해 부른다)
    • HEAD: 현재 내가 바라보는(작업 중인) 커밋 위치
    • branch: 작업 단위를 분리해 충돌을 줄이는 “작업 줄기”
    • merge: 브랜치의 변경을 합치는 작업

    과거로 되돌리기 — reset / restore / revert

    차이점 요약

    명령 되돌리는 방식 협업 안전성 언제 쓰나
    reset 커밋 이력을 “되감기” (기록을 지울 수 있음) 낮음 로컬에서 커밋을 다시 정리할 때
    restore 파일 단위로 작업 내용을 복구 높음 특정 파일 변경만 되돌릴 때
    revert 되돌리는 “새 커밋”을 생성 (기록 유지) 가장 높음 원격에 이미 올라간 커밋을 안전하게 취소할 때

    자주 쓰는 예시 

    # 직전 커밋을 "커밋만" 취소하고, 스테이징은 유지
git reset --soft HEAD~

# 직전 커밋을 취소하고, 스테이징도 해제(파일 변경 내용은 남음)
git reset HEAD~

# 직전 커밋을 완전히 되감기(로컬 변경까지 삭제) - 매우 주의
git reset --hard HEAD~

# 작업 중인 파일을 최근 커밋 상태로 복원(변경 내용 삭제)
git restore path/to/file

# 스테이징만 취소(파일 변경 내용은 유지)
git restore --staged path/to/file

# 특정 커밋을 취소하는 "새 커밋" 생성(협업에 안전)
git revert <commit_hash>

    브랜치와 PR — 협업에서의 기본 흐름

    브랜치 기본 

    # 브랜치 목록 확인
git branch

# 브랜치 생성
git branch feature/login

# 브랜치 이동
git checkout feature/login

    PR(Pull Request) 의미

    • 내 변경 사항을 다른 사람에게 리뷰/합의 받기 위한 공유 단위
    • PR 설명에는 무엇을/왜/어떻게 바꿨는지, 테스트 방법을 적는 습관이 중요
    • 합의가 끝나면 Merge로 메인 브랜치에 반영

    원격 저장소 — remote / fork / clone / upstream

    개념 정리

    용어 의미 포인트
    remote 원격 저장소 연결 정보 원격을 추가/삭제하거나 추적 브랜치를 관리
    fork 다른 사람 저장소를 내 계정으로 복제 GitHub 상에서 “내 소유” 저장소가 생김
    clone 저장소를 로컬로 내려받기 내 PC에서 직접 작업 가능
    upstream fork의 원본 저장소 원본 업데이트를 받아오거나 PR로 기여

    터미널 기초 — 자주 쓰는 리눅스 명령어

    필수 명령어

    명령 설명 예시
    ls 목록 보기 ls -al
    cd 이동 cd ..
    pwd 현재 경로 pwd
    mkdir 폴더 생성 mkdir src
    touch 파일 생성 touch README.md
    rm 삭제(복구 어려움) rm file.txt
    cat 파일 내용 출력 cat a.txt
    less 페이지 단위로 보기 less a.txt
    grep 문자열 검색 grep "todo" a.txt

    오늘의 회고와 다음 액션

    오늘 느낀 점

    • “되돌리기”는 상황마다 정답이 다르다. 협업이면 revert를 우선으로 생각하자.
    • 작업을 쪼개서 커밋하면 PR 리뷰도 쉬워지고, 문제 발생 시 원인 추적도 빨라진다.

    내일 할 것

    • 브랜치 전략(예: feature/dev/main)과 PR 템플릿을 내 프로젝트에 맞춰 정리하기
    • 실제로 작은 기능 1개를 브랜치로 분리 → PR 작성 → 머지까지 연습하기
