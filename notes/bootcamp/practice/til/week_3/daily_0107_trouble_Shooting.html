    18일차 개요

    오늘 목표

    • 협업 중 자주 터지는 Git/GitHub 에러를 “원인 → 조치 → 확인” 흐름으로 빠르게 정리한다
    • 권한/푸시/브랜치 실수 케이스를 재현 없이도 설명할 수 있게 만든다
    • 팀 작업에서 안전한 선택(공유 브랜치 기준)을 기준으로 판단한다

    Trouble Shooting 1) Permission denied로 커밋/푸시가 막힐 때

    증상

    • 원격 저장소로 push 하는 순간 Permission denied / 403 류 에러가 발생한다
    • 내 로컬에서는 커밋이 정상적으로 만들어졌는데, 원격 반영이 되지 않는다

    원인/핵심 판단

    이 케이스는 토큰 문제라기보다, “해당 저장소에 내 계정이 커밋/푸시할 권한이 없는 상태”에서 가장 많이 발생한다. 특히 다른 사람 계정(또는 조직) 소유 저장소에 협업으로 붙는 단계에서 자주 터진다.

    조치 방법

    1. 저장소 소유자가 GitHub 저장소 Settings로 이동한다
    2. Collaborators(또는 Access 관리 메뉴)에서 팀원 계정 메일을 모두 초대한다
    3. 권한을 maintain 이상으로 부여해, 저장소에 커밋/푸시가 가능하도록 풀어준다
    4. 초대 수락 이후 다시 커밋/푸시를 시도한다

    체크 포인트

    확인 항목 왜 중요? 체크 방법
    내가 접근 권한을 받았는지 권한이 없으면 푸시는 구조적으로 불가능 GitHub 저장소에서 내 계정이 Collaborators에 보이는지
    권한 레벨(maintain 이상) 권한 레벨이 낮으면 브랜치/설정에 따라 제한 발생 Collaborators 권한 드롭다운 확인
    원격 저장소 주소가 맞는지 엉뚱한 저장소(origin)가 잡혀 있으면 계속 실패 git remote -v로 origin URL 확인

    Trouble Shooting 2) 팀장 저장소에서 PR이 안 보일 때

    핵심 원인

    git commit은 내 PC(로컬)에 “저장”만 한 상태다. GitHub(원격) 서버에 “업로드”를 하지 않으면, 원격 저장소는 아무 변화도 모른다. 그래서 팀장 입장에서도 PR 후보 브랜치가 잡히지 않고, “Compare & pull request” 버튼도 뜨지 않는다.

    해결 방법(가장 확실한 루트)

    • 내 로컬 브랜치(예: feature/내이름)의 커밋을 원격으로 올려야 한다
    • 푸시 성공 후 GitHub 페이지를 새로고침(F5) 한다
    # 내 작업 브랜치 확인
git branch

# 내 브랜치를 GitHub로 전송(본인 브랜치 이름으로 정확히 입력)
git push origin feature/내이름

    정리

    단계 상태 눈에 보이는 현상
    commit만 함 로컬에만 저장 GitHub에는 변화 없음, PR도 안 뜸
    push까지 함 원격 반영 완료 Compare & pull request 버튼이 활성화될 확률이 높음

    Trouble Shooting 3) 실수로 main 브랜치에 커밋했을 때

    전제(가장 깔끔한 케이스)

    이 조치는 아직 원격(main)에 푸시하지 않았고, 내 PC(로컬)에만 커밋이 있는 경우를 기준으로 한다. 이 상태라면 깔끔하게 “커밋만 취소 → 올바른 브랜치로 이동 → 다시 커밋”이 가능하다.

    해결 순서 

    # 1) main에서 방금 커밋을 취소(파일 변경은 워킹 디렉토리에 남김)
git checkout main
git reset HEAD~1

# 2) 원래 작업해야 할 브랜치로 이동(없으면 생성하면서 이동)
git checkout feature/내이름
# 또는
git checkout -b feature/내이름

# 3) 올바른 브랜치에서 다시 커밋
git add .
git commit -m "작업 내용"

    실수 방지 팁

    • 작업 시작 전에 git branch 또는 git status로 “내가 어디 브랜치에 있는지” 먼저 확인한다
    • main은 최대한 건드리지 않고, 기능 단위로 개인 브랜치에서만 커밋하는 습관을 만든다

    오늘의 회고

    오늘 느낀 점

    • 문제는 “명령어”보다 상태(로컬/원격, 권한, 브랜치)를 먼저 구분하는 게 절반이었다
    • 협업에서는 빠르게 고치는 것도 중요하지만, 공유 브랜치(main)는 안전하게 다루는 습관이 더 중요했다
    • 트러블슈팅은 결국 체크리스트화가 답이라, 다음부터는 “원인 후보”부터 빠르게 좁히기로 했다
