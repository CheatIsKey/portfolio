    내일배움캠프 Spring 백엔드 17일차 TIL — Git/GitHub 핵심 개념 & CLI 기본

    작성일 2026-01-06 · 분류: 부트캠프, TIL, Git, GitHub

    17일차 개요

    오늘 목표

    • Git과 GitHub의 역할을 명확히 구분한다
    • Working Directory → Staging Area → Local Repo(.git) → Remote Repo 흐름을 이해한다
    • 되돌리기(restore/reset/revert) 차이를 정리하고 “협업에서 안전한 선택”을 기준으로 판단한다
    • CLI(터미널) 기본 명령과 Git 명령을 실무 관점으로 정리한다

    Git vs GitHub

    핵심 정의

    구분 정의 한 줄 요약
    Git 파일/코드 변경 이력을 저장하고 관리하는 분산 버전 관리 시스템(VCS) 내 PC에서 버전 기록을 관리하는 도구
    GitHub Git 저장소를 원격(인터넷)에서 보관/공유/협업할 수 있는 웹 서비스(플랫폼) Git 저장소를 올려두고 협업하는 공간

    로컬에서만 작업하면 “내 컴퓨터가 고장났을 때”, “협업자가 내 이력을 봐야 할 때”, “여러 장소에서 작업해야 할 때” 문제가 커진다. 그래서 원격 저장소(리모트)를 두고 백업 + 협업을 같이 해결한다.

    분산형 버전 관리(DVCS)가 중요한 이유

    DVCS 관점에서 “로컬”은 부가물이 아니라 핵심

    • 중앙 서버만 망가지면 끝나는 구조가 아니라, 각자 로컬에 저장소(이력)가 존재한다
    • 네트워크가 불안정해도 로컬에서 커밋/브랜치 작업이 가능하다
    • 원격 저장소는 협업과 공유를 위한 “합류 지점” 역할을 한다

    Git 기반 대표 플랫폼(실무 관점)

    플랫폼 특징 주요 사용 맥락
    GitHub 오픈소스 생태계/PR 기반 협업에 강함 공개/사내 협업, 포트폴리오
    GitLab 사내 구축(Self-hosted) 지원이 강하고 DevOps 기능이 풍부한 편 기업 내부망/보안 환경에서 운영
    Gerrit 코드 리뷰(변경 단위) 중심의 엄격한 검토 흐름에 강함 리뷰 프로세스가 매우 엄격한 조직
    Bitbucket Atlassian 제품군(Jira 등)과 연동이 좋은 편 Jira 중심으로 업무 관리하는 팀

    Git의 내부 구조: Working Directory → Staging Area → Local Repo → Remote

    4단계 흐름 

    Working Directory(내가 파일 편집하는 곳)
  └─ git add
Staging Area(커밋에 포함할 변경만 “선별”)
  └─ git commit
Local Repository(.git)(내 PC의 커밋 이력 저장소)
  └─ git push
Remote Repository(GitHub 등 원격 저장소)
    영역 의미 대표 명령
    Working Directory 실제 파일을 수정/생성하는 작업 공간 git status
    Staging Area “이번 커밋에 포함할 변경”을 고르는 중간 단계 git add, git restore --staged
    Local Repo(.git) 커밋(스냅샷) 히스토리가 저장되는 로컬 저장소 git commit, git log
    Remote Repo 원격에 올려 공유/백업/협업하는 저장소 git push, git pull, git fetch

    핵심 용어 정리

    협업에서 자주 나오는 단어들

    용어 의미 핵심 포인트
    commit 프로젝트 상태를 기록한 “스냅샷(버전)” 원격에 올리기 전에도 로컬에서 반드시 생성해야 함
    snapshot 특정 시점의 코드 상태(버전) 문제 생기면 특정 커밋으로 되돌아갈 수 있음
    hash 커밋을 식별하는 값(보통 앞부분만 축약해 사용) 로그에서 커밋을 찾는 기준
    HEAD 현재 내가 바라보는 커밋(대개 현재 브랜치의 최신 커밋) HEAD~1은 직전 커밋을 의미
    branch 작업 흐름을 분리하는 “커밋 포인터” 기능 개발/버그 수정/배포 단계 분리에 필수
    merge 브랜치의 변경을 합치는 작업 충돌(conflict)이 나면 수동으로 해결 필요
    remote 원격 저장소(예: origin, upstream) push/pull/fetch 대상
    fork 다른 사람 저장소를 “내 GitHub 계정”으로 복제 계정에 복제될 뿐, 로컬로 내려오는 것은 아님
    clone 원격 저장소를 “내 로컬 PC”로 복제 로컬 개발 시작 단계
    PR(Pull Request) 변경 사항을 공유하고 코드 리뷰/병합을 요청하는 단위 협업에서 변경 공유 + 합의의 표준 흐름

    실전 Git 명령 흐름

    가장 기본: 로컬에서 커밋하고 원격에 올리기 

    # 1) 저장소 초기화(또는 clone으로 시작)
git init

# 2) 변경 확인
git status

# 3) 스테이징(이번 커밋에 포함할 변경 선택)
git add .

# 4) 커밋(스냅샷 생성)
git commit -m "docs: add til day17"

# 5) 원격 저장소 연결(처음 1회)
git remote add origin <REMOTE_URL>

# 6) 원격에 업로드
git push -u origin main

    브랜치 작업(기능 개발의 표준) 

    # 브랜치 생성
git branch feature/til-day17

# 브랜치 이동(최근에는 switch를 더 자주 사용)
git checkout feature/til-day17
# 또는
git switch feature/til-day17

# 작업 후 커밋 → PR → merge(보통 GitHub에서)

    브랜치는 “파일을 복사하는 기능”이 아니라, 커밋 흐름을 분리하는 포인터다. 기능 단위로 브랜치를 나눠 두면 리뷰/테스트/롤백이 훨씬 쉬워진다.

    되돌리기 전략: restore vs reset vs revert

    협업 기준으로 정리

    명령 무엇을 되돌리나 히스토리(커밋 로그) 협업 안전성
    restore 파일 단위(Working Directory / Staging) 상태 복구 유지됨 안전
    reset 커밋 기준으로 “현재 위치(HEAD)”를 이동 커밋이 사라질 수 있음 주의(공유 브랜치에서 위험)
    revert 특정 커밋의 변경을 “취소하는 새 커밋” 생성 유지됨(취소 기록도 남음) 가장 안전

    자주 쓰는 예시 

    # (1) 워킹 디렉토리 파일을 마지막 커밋 상태로 되돌리기(수정 내용 폐기)
git restore README.md

# (2) 스테이징만 취소(파일 수정 내용은 유지)
git restore --staged README.md

# (3) 직전 커밋을 취소하되, 변경 내용을 스테이징에 유지(커밋만 다시 만들고 싶을 때)
git reset --soft HEAD~1

# (4) 직전 커밋을 취소하고, 스테이징도 취소(변경 내용은 워킹에 남음)
git reset HEAD~1

# (5) 직전 커밋을 완전히 삭제(히스토리/작업까지 날림) - 협업 브랜치에서는 매우 위험
git reset --hard HEAD~1

# (6) 특정 커밋을 “취소하는 커밋” 생성(협업에서 권장)
git revert <COMMIT_HASH>

    터미널/리눅스 기초 명령 (Git Bash 포함)

    자주 쓰는 명령 모음

    명령 의미 예시
    pwd 현재 디렉토리 절대 경로 출력 pwd
    ls 목록 보기 ls -al
    cd 디렉토리 이동 cd ..
    mkdir 폴더 생성 mkdir til
    touch 빈 파일 생성(또는 타임스탬프 갱신) touch README.md
    cat 파일 내용 출력 cat README.md
    less 페이지 단위로 파일 보기(q로 종료) less README.md
    head/tail 앞/뒤 일부 출력 tail -n 20 app.log
    mv 이동/이름 변경 mv a.txt b.txt
    cp 복사 cp a.txt backup/
    rm 삭제(복구 어려움) rm a.txt
    clear 화면 지우기 clear
    history 입력 명령 이력 history
    whoami 현재 사용자 whoami
    df -h 디스크 사용량(사람이 읽기 쉬운 단위) df -h
    du -sh 특정 경로 용량 du -sh .

    문자열 검색: grep와 git grep 

    # 파일 내용에서 문자열 찾기
grep "TODO" README.md

# 특정 폴더 하위에서 재귀 검색(-r)
grep -r "TODO" .

# Git 저장소(추적 파일)에서 검색할 때
git grep "TODO"

    협업 흐름: fork/clone/upstream/PR

    fork 기반 협업의 기본 흐름

    1. 원본 저장소를 내 계정으로 fork한다
    2. 내 fork 저장소를 로컬로 clone한다
    3. 기능 브랜치를 만들고 커밋한다
    4. 내 fork 저장소로 push 후 Pull Request(PR)를 만든다
    5. 리뷰/합의 후 원본(upstream)에 merge 된다

    upstream 최신 반영(대표 예시) 

    # 원본 저장소를 upstream으로 추가(처음 1회)
git remote add upstream <UPSTREAM_URL>

# upstream 변경 가져오기
git fetch upstream

# 내 main에 반영(상황에 따라 merge 또는 rebase 선택)
git checkout main
git merge upstream/main

    오늘의 회고

    오늘 느낀 점

    • Staging Area는 “커밋 단위를 깔끔하게 만들기 위한 선별 공간”이라는 감각이 중요했다
    • 협업에서는 reset보다 revert가 안전한 경우가 많다(히스토리를 지우지 않기 때문)
    • CLI는 결국 “파일/폴더/텍스트를 빠르게 다루는 기술”이라, Git과 함께 익히면 생산성이 확 올라간다
