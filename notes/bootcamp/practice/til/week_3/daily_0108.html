    19일차 개요

    오늘 목표

    • 검사 예외와 비검사 예외의 차이를 컴파일 단계 관점으로 정리한다
    • Optional로 null을 안전하게 다루는 흐름을 익힌다
    • 제네릭 클래스와 제네릭 메서드가 언제 분리되는지 감을 잡는다

    검사 예외 vs 비검사 예외

    핵심 차이 정리

    구분 컴파일러 강제 대표 계열 처리 방식
    검사 예외(Checked) 필수(처리 또는 선언) Exception 하위 중 RuntimeException 제외 try-catch 또는 throws
    비검사 예외(Unchecked) 강제하지 않음 RuntimeException 및 하위 필수는 아니지만, 필요하면 try-catch 가능

    핵심은 발생 시점이 아니라, 컴파일러가 예외 처리를 강제하느냐다. 검사 예외는 처리하지 않으면 컴파일 에러로 막혀서, 실수로 누락되는 일을 줄여준다.

    왜 모든 예외를 검사 예외로 하지 않나

    • 모든 걸 강제로 처리하게 만들면, 예외 처리 코드가 과도하게 늘어나면서 흐름이 복잡해진다
    • 특히 프로그래밍 오류(예: 잘못된 인덱스 접근, null 역참조)까지 전부 강제하면 생산성이 크게 떨어질 수 있다
    • 그래서 자바는 복구 가능한 상황은 검사 예외로, 코드 결함에 가까운 상황은 비검사 예외로 두는 방향을 많이 사용한다

    짧은 예시 

    // 검사 예외: 처리(try-catch) 또는 선언(throws) 없으면 컴파일 단계에서 막힘
public void readFile() throws java.io.IOException {
  // ...
}

// 비검사 예외: 처리 강제는 아니지만, 상황에 따라 방어 코드나 예외 처리를 한다
public int divide(int a, int b) {
  if (b == 0) throw new IllegalArgumentException("b must not be 0");
  return a / b;
}

    Optional로 null 안전하게 다루기

    Optional 정의

    Optional은 값이 있을 수도 없을 수도 있는 상태를 표현하는 컨테이너다. null을 직접 다루는 대신 Optional을 사용하면 NullPointerException을 줄이고, 값이 없을 때의 흐름을 코드로 명확히 드러낼 수 있다.

    자주 쓰는 API 흐름 

    // null이 들어올 수 있는 값을 감쌀 때
java.util.Optional<String> opt = java.util.Optional.ofNullable(getNickname());

// 값이 존재하는지 확인
if (opt.isPresent()) {
  System.out.println(opt.get());
}

// 값이 있으면 실행(더 자주 쓰는 방식)
opt.ifPresent(System.out::println);

// 값이 없을 때 기본값 제공
String nickname = opt.orElseGet(() -> "guest");

    orElse vs orElseGet 차이

    메서드 언제 기본값이 계산되나 추천 상황
    orElse(value) 값이 있든 없든 value가 먼저 평가될 수 있음 기본값이 가볍고 단순할 때
    orElseGet(supplier) 값이 없을 때만 supplier가 실행됨 기본값 생성이 무겁거나, 지연 실행이 필요할 때

    제네릭 클래스 vs 제네릭 메서드

    개념 정리

    • 제네릭 클래스의 일반 메서드는 클래스에 선언된 타입 파라미터(T 등)를 그대로 따른다
    • 제네릭 메서드는 메서드 자체가 별도의 타입 파라미터를 선언한다(예: <S>)
    • 그래서 클래스 타입과 무관하게, 메서드 단위로 타입을 유연하게 받고 싶으면 제네릭 메서드를 쓴다

    짧은 예시 

    // 제네릭 클래스: T는 클래스가 결정
class Box<T> {
  private final T value;

  Box(T value) { this.value = value; }

  // 일반 메서드: 클래스의 T를 사용
  public T get() { return value; }

  // 제네릭 메서드: 메서드가 별도의 타입 S를 선언
  public <S> S convert(java.util.function.Function<T, S> mapper) {
    return mapper.apply(value);
  }
}

    오늘의 회고

    오늘 느낀 점

    • 예외는 종류를 외우는 것보다, 컴파일러 강제 여부로 먼저 분류하니까 정리가 빨랐다
    • Optional은 null을 숨기는 게 아니라, 없을 수 있음을 코드에서 드러내는 게 핵심이었다
    • 제네릭은 클래스/메서드 구분만 확실히 해도, 타입 흐름이 덜 헷갈렸다
