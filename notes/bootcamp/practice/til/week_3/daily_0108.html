    GitHub Issues로 작업을 계획하고 추적하기

    Issues가 해결해 주는 문제

    GitHub Issues는 리포지토리에서 해야 할 일(작업/버그/아이디어/피드백)을 하나의 “문제 단위”로 만들고, 토론과 진행 상태를 추적할 수 있게 해준다. 즉, 코드를 바꾸기 전에 “무엇을, 왜, 어떻게” 할지를 합의하고 기록으로 남기는 도구다.

    실무에서 자주 쓰는 구성 요소

    기능 의미 활용 예시
    Labels 이슈 분류(성격/우선순위/영역) bug, feature, docs, priority:high
    Assignees 담당자 지정 누가 처리하는지 명확히 하기
    Milestone 버전/스프린트 단위로 묶기 v1.1.0 릴리스에 포함될 이슈 묶기
    Link to PR 이슈 ↔ PR 연결 PR 본문에 Fixes #12를 적으면 merge 시 자동 닫힘

    이슈 작성 최소 템플릿 

    제목: [bug] 로그인 시 500 발생

1) 현상(What)
- 특정 조건에서 로그인 시 500 에러가 발생한다.

2) 재현 방법(How)
- /login 에서 잘못된 비밀번호 입력
- 다시 올바른 비밀번호 입력 후 로그인

3) 기대 결과(Expected)
- 정상적으로 로그인되고 홈으로 이동한다.

4) 참고(Reference)
- 서버 로그: NullPointerException at AuthService#...

    코드 검색: grep.app(코드 서치) 활용

    언제 도움이 되나

    • 공개 저장소에서 “이 에러/이 패턴을 실제로 어떻게 해결했는지” 빠르게 사례를 찾고 싶을 때
    • 특정 API/라이브러리의 사용 예제를 “코드 그대로” 보고 싶을 때
    • 정확한 키워드(함수명/클래스명/에러 메시지)로 전역 검색이 필요할 때

    주의할 점

    • 찾은 코드는 “그대로 복붙”보다 내 프로젝트 요구사항에 맞게 이해 후 적용이 우선이다
    • 라이선스/저작권(특히 큰 코드 덩어리)과 보안(비밀키/토큰 노출) 이슈를 항상 확인한다
    • 코드가 오래된 경우(버전 차이) 동작이 달라질 수 있어 공식 문서로 최종 검증한다

    Dev / Stage / Prod 브랜치로 환경을 분리하는 이유

    목적: 개발/검증/운영을 분리해서 안정성을 확보

    환경별 브랜치 구성을 쓰는 이유는 개발 중인 코드와 검증된 코드와 실제 운영 코드를 분리해서 “누가 봐도 지금 무엇이 배포 가능한지”를 명확히 하기 위함이다.

    브랜치 역할 대표 규칙
    Dev(또는 develop) 기능 개발/버그 수정이 합쳐지는 통합 개발 브랜치 feature 브랜치를 여기로 병합(코드 리뷰/테스트 포함)
    Stage(또는 staging/release) 배포 직전 최종 검증(QA/통합 테스트) 브랜치 운영과 유사한 설정에서 “배포 가능” 상태만 유지
    Prod(보통 main/master) 실 사용자에게 제공되는 운영 코드 항상 안정적이어야 하며, 배포는 이 브랜치 기준으로 수행 
    feature/*  ──>  dev(develop)  ──>  stage(staging)  ──>  prod(main)
(개별 기능)      (통합 개발)         (최종 검증)            (운영 배포)

    배포 리스크 키워드: Risky / Slow / Fat Release

    키워드 의미 대응 전략
    Risky 검증(스테이징/QA) 없이 운영에 바로 반영되어 장애 가능성이 커지는 상태 최소한의 QA 게이트 유지, 롤백 플랜/모니터링 준비
    Slow 변경량이 커져서 테스트/배포/원인 파악이 느려지는 상태 변경을 작게 쪼개고, 자동화 테스트/배포 파이프라인 강화
    Fat Release 한 번 배포에 너무 많은 변경이 포함되어 “문제 발생 시 원인 추적이 어려운” 릴리스 Small Release(점진 배포), 기능 플래그(Feature Flag) 활용

    feature / release / hotfix 브랜치의 역할

    기능/배포/긴급수정의 분리

    브랜치 분기 목적 병합
    feature/* dev(develop) 새 기능/개선 사항을 독립적으로 개발 dev(develop)로 PR/merge
    release/* dev(develop) 배포 직전 안정화(QA, 버그 수정) prod(main) + dev(develop)로 병합(정책에 따라)
    hotfix/* prod(main) 운영 장애/긴급 버그를 빠르게 수정 prod(main) + dev(develop)로 병합(재발 방지)

    Semantic Versioning(SemVer): x.y.z 버전 규칙

    MAJOR.MINOR.PATCH 의미

    구성 언제 올리나 예시
    MAJOR 호환되지 않는 변경(Breaking Change) 1.0.0 → 2.0.0
    MINOR 호환되는 기능 추가 1.0.0 → 1.1.0
    PATCH 호환되는 버그 수정 1.0.1 → 1.0.2

    브랜치 운영 전략: Git Flow vs GitHub Flow vs Trunk-based

    3가지 모델 비교

    전략 핵심 아이디어 잘 맞는 상황
    Git Flow feature/release/hotfix 등 브랜치를 세분화해 “버전/배포”를 엄격히 관리 릴리스 주기가 길고, 배포 안정화 단계가 중요한 조직
    GitHub Flow main은 항상 배포 가능, feature 브랜치 → PR → main 병합 후 바로 배포 자주 배포하고, CI/CD가 갖춰진 환경
    Trunk-based Development 하나의 메인(trunk/main)에 “아주 작은 변경”을 자주 통합(짧은 브랜치 수명) Small Release, 빠른 피드백, 자동화 테스트가 강한 팀

    충돌(conflict)과 Rebase

    충돌이 발생하는 이유

    충돌은 여러 사람이 같은 파일의 같은 구간을 서로 다르게 수정했을 때, Git이 “어느 변경이 맞는지” 자동으로 결정할 수 없어서 발생한다. 이 경우 최종 의도는 사람만이 판단할 수 있으므로, 사람이 직접 해결하도록 멈춰준다.

    Rebase 개념: 기록을 다시 쌓아 히스토리를 깔끔하게 만들기

    rebase는 한 브랜치의 커밋들을 다른 브랜치(기준) 위로 “다시 적용”하여 커밋 흐름을 일직선으로 정리하는 방식이다. 이 과정에서 커밋이 새로 만들어지므로 커밋 해시가 바뀐다(히스토리 재작성). 

    # 예시: feature 브랜치를 최신 main 위로 다시 쌓기
git checkout feature/login
git fetch origin
git rebase origin/main

    rebase 충돌 해결 흐름 

    # 1) 충돌 파일 수정(표시된 충돌 마커 해결)
# 2) 해결한 파일을 스테이징
git add <파일명>

# 3) rebase 계속 진행
git rebase --continue

# 중간에 그만두고 싶다면
git rebase --abort

    rebase 진행 중 에디터가 열리면, 안내에 따라 커밋 메시지를 확인/수정하고 저장 후 종료하면 된다 (vim 기준으로는 저장 후 종료: :wq).

    커밋 히스토리 다듬기: amend & interactive rebase

    git commit --amend: 마지막 커밋 수정

    git commit --amend는 가장 최근 커밋의 메시지를 바꾸거나, 빠뜨린 파일을 추가해서 “마지막 커밋 자체를 고치는” 명령이다. 결과적으로 마지막 커밋이 새로 만들어지므로 해시가 바뀐다. 

    # (1) 파일을 추가로 스테이징
git add <빠진파일>

# (2) 마지막 커밋에 포함 + 메시지 수정(또는 유지)
git commit --amend

    interactive rebase(-i): 커밋 합치기/수정/정리 

    # 최근 2개 커밋을 대상으로 편집
git rebase -i HEAD~2
    명령 의미
    pick 커밋 유지
    reword 커밋 메시지만 수정
    edit 커밋을 멈추고 내용 수정(추가 커밋/분할 등)
    squash 이전 커밋과 합치고 메시지를 통합/정리

    매우 중요한 원칙: 공유된 커밋은 rebase로 재작성하지 않는다

    • 이미 원격에 push되어 팀이 공유하는 브랜치(예: main/develop)의 커밋을 rebase하면 히스토리가 갈라져 혼란이 커진다
    • 정말 불가피하게 재작성해야 한다면, 팀 합의 후 --force-with-lease 같은 안전 장치를 고려한다
    • 공유 브랜치에서는 보통 merge 또는 revert가 더 안전한 선택이다

    pull 충돌/히스토리 꼬임 줄이기: git pull --rebase

    의미

    로컬 커밋이 있는 상태에서 원격 변경을 가져올 때, pull이 merge 커밋을 만들지 않고 “내 커밋을 위로 다시 쌓는 방식”으로 히스토리를 깔끔하게 유지하고 싶으면 git pull --rebase를 사용할 수 있다. 

    # 원격 main을 가져오면서 rebase로 정리
git pull --rebase origin main

    Tag와 Release Notes로 배포 이력 관리

    Tag란?

    tag는 특정 커밋을 “버전 스냅샷”처럼 고정해서, 그 시점으로 쉽게 돌아갈 수 있게 만든다. 보통 배포 버전(v1.2.0)을 태그로 남겨 릴리스 기준점을 만든다. 

    # annotated tag(메시지 포함)를 만드는 방식
git tag -a v1.1.0 -m "release v1.1.0"

# 태그를 원격에 푸시
git push origin v1.1.0

    Release Notes란?

    Release Notes는 새 버전을 배포할 때, 추가된 기능 / 수정된 버그 / 주의사항 / 업그레이드 힌트 등을 정리한 문서다. 변경이 많아질수록 “사용자/팀”에게 전달해야 하는 정보도 늘어나므로, 작은 릴리스와 함께 Release Notes를 꾸준히 관리하면 운영 품질이 올라간다.

    오늘의 회고

    오늘 느낀 점

    • Issues는 “할 일 목록”을 넘어, 팀이 같은 목표/상태를 공유하는 커뮤니케이션의 기준점이 된다
    • 브랜치 전략은 정답이 하나가 아니라, 배포 빈도/자동화 수준/팀 규모에 따라 선택해야 한다
    • rebase는 강력하지만 히스토리를 재작성하는 도구라서, “공유된 커밋에는 쓰지 않는다”는 원칙이 제일 중요했다
    • tag + release notes를 습관화하면 배포 이력이 명확해지고, 장애 대응 속도도 빨라진다
