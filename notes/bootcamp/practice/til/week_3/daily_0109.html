    인터페이스 vs 추상 클래스

    공통점: 계약을 제공한다

    인터페이스와 추상 클래스는 공통적으로 구현 클래스가 따라야 할 계약을 제공한다. 결국 핵심은 메서드 규격을 고정해서, 구현체가 바뀌어도 사용 코드가 안정적으로 동작하게 만드는 데 있다.

    차이점 정리

    구분 인터페이스 추상 클래스
    목적 기능 규격을 강제하는 계약 공통 상태/동작을 묶는 뼈대
    상속 형태 다중 구현 가능 단일 상속
    상태(필드) 상태를 들고 가는 설계는 보통 지양 공통 필드/생성자 설계 가능
    현업 감각 역할 분리, 확장 포인트 만들 때 강함 공통 코드 재사용, 템플릿 구조 만들 때 강함

    컬렉션과 접근 제어자, 필드 설계 감각

    컬렉션도 똑같이 캡슐화 대상이다

    컬렉션도 결국 객체라서, 필드로 둘 경우 접근 제어자를 붙여서 외부에서 마음대로 바꾸지 못하게 막는 게 기본이다. 계산기 프로젝트에서도 결과 목록을 private로 막고, 추가/삭제/조회 메서드로만 조작하도록 흐름을 만든 게 핵심 개선이었다.

    필드는 먼저 의심하고, 필요할 때만 둔다

    • 클래스를 만들 때 멤버 변수/필드가 꼭 필요한지 먼저 생각해본다
    • 입력값처럼 매번 바뀌는 값은, 필드보다 메서드 파라미터로 전달하는 편이 안전하다
    • 필드가 늘어날수록 객체 상태가 복잡해지고, 디버깅 포인트도 같이 늘어난다

    스코프와 객체 생성에서 터지는 실수 포인트

    전역 변수 vs 지역 변수 스코프

    구분 범위 주의점
    필드(전역에 가까운 상태) 객체가 살아있는 동안 유지 상태가 쌓이거나, 의도치 않게 공유될 수 있음
    지역 변수 블록이 끝나면 소멸 반복문/조건문 안에서만 유효하니 흐름을 잘 봐야 함

    new를 반복문 안에서 호출하면 데이터가 끊길 수 있다

    반복문 안에서 new로 객체를 계속 생성하면, 기존 객체가 들고 있던 데이터는 새 객체로 이어지지 않는다. 계산기 프로젝트 기준으로는, 계산 결과 목록을 들고 있는 Calculator를 반복마다 새로 만들면 결과 리스트가 초기화되는 문제가 생길 수 있다.

    계산기 프로젝트 정리

    프로젝트 상태 및 링크

    계산기 프로젝트는 현재 개발 진행 중이고, 코드는 GitHub에 정리해 두었다. CalculatorProject Repository

    핵심 흐름

    1. 숫자 입력은 정수만 받도록 검증한다
    2. 연산자는 사칙연산 기호만 통과시킨다
    3. 계산 결과를 출력하고, 결과를 리스트에 누적 저장한다
    4. 사용자 입력에 따라 종료하거나, 저장된 결과를 삭제한다

    내가 잡아낸 실수 포인트와 바로잡은 점

    실수 포인트 왜 문제였나 어떻게 바꿨나
    반복문 안에서 Calculator를 새로 생성할 뻔함 새 인스턴스가 만들어지면 기존 결과 리스트가 끊길 수 있음 Calculator는 한 번만 만들고, 입력값은 calculate(a, b, operator)로 전달
    숫자 입력 검증이 약했음 문자 입력 시 프로그램이 깨질 수 있음 InputMismatchException을 잡고, 잘못된 입력은 버린 뒤 다시 입력 받도록 분리
    연산자 검증이 분산될 수 있었음 유효하지 않은 연산자가 들어오면 예외가 흔들림 연산자 입력을 inputOperator로 고정하고, 통과 조건을 한 곳으로 모음
    나눗셈 분모 0 처리 누락 위험 실제 실행 중 예외 상황이 발생할 수 있음 Calculator에서 분모 0이면 IllegalArgumentException을 던지고, Main에서 메시지 출력
    컬렉션 출력 방식이 불필요하게 복잡할 뻔함 배열로 옮겨 담으면 코드가 길어지고 관리가 어려움 list.toString()으로 결과 목록을 바로 확인하도록 단순화

    예외 처리와 입력 검증의 역할 분리

    흐름을 분리한 기준은 단순했다. 입력 단계에서 걸러낼 수 있는 건 입력 단계에서 막고, 계산 로직 자체에서 방어해야 하는 건 Calculator 내부에서 예외로 처리한다. 그래서 Main은 최대한 진행 흐름에 집중하고, Calculator는 계산 규칙과 데이터 보관에 집중하도록 역할이 정리됐다.

    다음에 같은 실수를 안 하려면

    • 반복문 안에서 new를 치기 전에, 그 객체가 들고 있어야 하는 데이터가 무엇인지 먼저 확인한다
    • 입력 검증은 한 번에 끝내지 말고, 입력 메서드로 분리해서 재사용한다
    • 컬렉션은 무조건 private로 막고, 외부에서는 메서드로만 건드리게 만든다
    • 삭제 기능처럼 상태가 바뀌는 기능은, 빈 컬렉션 처리를 먼저 넣고 시작한다

    오늘의 회고

    오늘 느낀 점

    • 코드를 늘리기 전에, 역할과 책임을 먼저 나누니까 수정이 훨씬 편했다
    • 입력 검증과 예외 처리를 분리하니, 어디서 막아야 하는지가 명확해졌다
    • 작은 프로젝트도 결국 스코프와 객체 생명주기를 이해하는 게 핵심이었다
