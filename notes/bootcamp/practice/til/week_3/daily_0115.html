    인터페이스 vs 추상 클래스

    정의

    구분 핵심 역할 가장 큰 차이
    인터페이스 구현체가 따라야 할 계약을 제공 주로 동작 규격 중심(다중 구현 가능)
    추상 클래스 공통 상태/동작을 묶는 뼈대 제공 필드/생성자로 공통 상태를 가질 수 있음(단일 상속)

    인터페이스를 무조건 완전 미완성, 추상 클래스를 일부 미완성으로 외우기보다는, 인터페이스는 역할(계약), 추상 클래스는 공통 구현 + 공통 상태라는 감각으로 보는 게 더 안전했다.

    인터페이스의 default 메서드와 private 메서드

    default 메서드가 나온 이유

    인터페이스에 메서드를 추가하면, 그 인터페이스를 구현하는 모든 클래스가 해당 메서드를 구현해야 한다. 이 부담을 줄이기 위해 default 메서드가 생겼고, 기본 구현을 제공해서 기존 구현체들이 깨지지 않게 해준다.

    private 메서드의 역할

    • 인터페이스 내부에서 default 또는 static 메서드가 공통으로 쓰는 로직을 분리하기 위해 사용한다
    • 반드시 구현부가 있어야 한다
    • static 메서드에서 호출할 private 메서드는 private static이어야 한다

    예시 코드 

    interface Loggable {
  default void log(String msg) {
    validate(msg);
    System.out.println("[LOG] " + msg);
  }

  static void staticLog(String msg) {
    staticValidate(msg);
    System.out.println("[STATIC] " + msg);
  }

  private void validate(String msg) {
    if (msg == null || msg.isBlank()) {
      throw new IllegalArgumentException("msg must not be blank");
    }
  }

  private static void staticValidate(String msg) {
    if (msg == null || msg.isBlank()) {
      throw new IllegalArgumentException("msg must not be blank");
    }
  }
}

    예외 되던지기와 연결된 예외

    예외 되던지기(rethrow)

    예외 되던지기는 예외를 한 번 처리(로그/정리/메시지 보강 등)한 뒤, 다시 던져서 호출자 쪽에서도 예외 처리를 하게 만드는 방식이다. 스프링에서도 하위 계층에서 상황을 정리하고, 상위 계층에서 최종 응답을 결정하는 흐름이 많아서 자주 만난다. 

    try {
  doSomething();
} catch (Exception e) {
  System.out.println("하위 로직에서 예외 로그를 남김: " + e.getMessage());
  throw e; // 다시 던져서 상위 로직에서도 처리하게 함
}

    연결된 예외(chained exception)

    • 예외 안에 다른 예외를 원인(cause)으로 저장하는 방식이다
    • 여러 예외를 상위 관점의 예외 하나로 묶어서 다루기 좋다
    • 검사 예외를 비검사 예외로 바꾸고 싶을 때, 원인을 유지한 채로 RuntimeException으로 감싸는 방식이 자주 쓰인다
    try {
  readFile();
} catch (java.io.IOException e) {
  throw new IllegalStateException("파일 처리 중 문제가 발생했습니다.", e);
}

    얕은 복사와 깊은 복사

    정의와 차이

    구분 무엇을 복사하나 특징
    얕은 복사 객체의 필드 값만 복사(참조 필드는 주소 공유) 내부 참조 객체를 수정하면 서로 영향이 갈 수 있음
    깊은 복사 참조하는 객체까지 새로 복사 공유가 끊겨서 독립적으로 수정 가능

    예시(얕은 복사 상황) 

    class Holder {
  int[] arr;
  Holder(int[] arr) { this.arr = arr; }
}

public class Main {
  public static void main(String[] args) {
    int[] origin = {1, 2, 3};
    Holder a = new Holder(origin);
    Holder b = new Holder(origin); // 같은 배열을 공유

    b.arr[0] = 999;

    System.out.println(a.arr[0]); // 999 (공유 영향)
  }
}

    String intern() 정리

    정의

    intern()은 문자열을 String 풀에서 관리하도록 만들고, 같은 내용이 이미 풀에 있으면 그 대표 참조를 돌려준다.

    주의점

    • == 비교가 항상 안전해지는 건 아니다. 둘 다 intern된 참조일 때만 같은 주소를 가리킨다
    • intern은 내부 풀을 건드리기 때문에, 상황에 따라 비용이 커질 수 있다
    • 문자열 비교는 기본적으로 equals()를 쓰는 게 안전하다

    예시 코드 

    public class Main {
  public static void main(String[] args) {
    String a = new String("hello");
    String b = new String("hello");

    System.out.println(a == b);          // false
    System.out.println(a.equals(b));     // true

    String ai = a.intern();
    String bi = b.intern();

    System.out.println(ai == bi);        // true (둘 다 풀의 대표 참조)
  }
}

    trim() vs strip()

    차이점

    둘 다 앞뒤 공백을 제거하지만, strip()은 유니코드 공백까지 더 넓게 처리한다. 반면 trim()은 비교적 제한된 범위의 공백만 제거한다.

    메서드 처리 범위 추천
    trim() 전통적인 공백 처리(제한적) 레거시 호환 필요할 때
    strip() 유니코드 공백 포함(더 넓음) 가능하면 strip() 쪽이 더 안정적

    join()과 split(), StringJoiner

    개념 정리

    • split(): 문자열을 구분 기준으로 쪼개서 배열로 만든다
    • join(): 배열/컬렉션을 구분자로 이어서 하나의 문자열로 만든다
    • StringJoiner: 구분자 + prefix/suffix까지 포함해 조합할 때 깔끔하다

    예시 코드 

    import java.util.StringJoiner;

public class Main {
  public static void main(String[] args) {
    StringJoiner sj = new StringJoiner(", ", "[", "]");
    String[] strArr = {"aaa", "bbb", "ccc"};

    for (String s : strArr) {
      sj.add(s.toUpperCase());
    }

    System.out.println(sj.toString()); // [AAA, BBB, CCC]
  }
}

    오늘의 회고

    오늘 느낀 점

    • 인터페이스는 결국 계약이라는 감각으로 잡으니까 default/private가 자연스럽게 이해됐다
    • 예외는 한 곳에서 끝내려 하기보다, 정보 보강하고 원인 유지하는 게 더 중요했다
    • String 쪽은 성능 이야기로 급하게 결론 내리기보다, 안전한 기본값을 먼저 지키는 게 맞았다
