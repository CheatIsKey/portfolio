    객체지향이란 무엇인가

    핵심 정의

    객체지향은 서로 관련 있는 데이터와 기능을 묶어 객체로 만들고, 객체들이 메시지를 주고받으며 협력하도록 설계하는 방식이다. 이때, 서로 관련 없어 보이는 클래스들도 공통 규격으로 묶을 수 있는데, 그 역할을 인터페이스가 담당한다.

    객체지향의 중심은 캡슐화

    객체지향의 감각은 결국 데이터를 가진 객체가 그 데이터를 직접 처리하게 만들고, 외부에서는 내부 구조를 몰라도 정해진 사용법만으로 동작하게 만드는 것이다. 이걸 캡슐화라고 한다.

    좋은 설계 기준: 결합도, 응집도, 데이터 무결성

    세 가지 기준을 한 번에 잡기

    기준 의미 실전 체크 포인트
    결합도 낮추기 서로를 너무 많이 알지 않게 만들기 구체 클래스 대신 인터페이스에 의존하는가
    응집도 높이기 하나의 클래스는 하나의 책임에 집중 한 클래스가 입력, 처리, 출력, 저장을 다 떠안지 않는가
    데이터 무결성 보장 객체의 상태가 규칙을 깨지 않게 유지 필드는 private로 숨기고, 검증된 메서드로만 변경하는가

    의존 관계와 협력: 카페 주문 예시

    의존 관계란

    어떤 객체가 자신의 일을 수행하기 위해 다른 객체의 도움이 필요한 상태를 의존 관계라고 본다. 예를 들어 주문 로직에서 캐셔가 바리스타에게 커피 제조를 맡긴다면, 캐셔는 바리스타에 의존한다. 

    public void order(String menuName, Barista barista) {
  // barista가 없으면 커피 제조를 진행할 수 없다
}

    손님은 바리스타를 몰라도 되게 만들기

    손님이 바리스타를 직접 호출하면, 손님이 커피 제조 과정을 알아야 한다. 그래서 손님 → 캐셔 → 바리스타 흐름으로 협력 구조를 만들면, 손님은 바리스타를 몰라도 주문이 가능해진다. 

    Customer → Cashier → Barista

    DI: 매개변수로 받을지, 필드로 가질지

    선택 기준

    상황 추천 방식 이유
    호출할 때만 잠깐 필요한 협력자 메서드 매개변수로 전달 생명주기가 짧고, 의존을 최소화하기 좋다
    함께 생성되고 함께 소멸하는 협력자 필드로 보관하고 생성자에서 주입 객체 생명주기 동기화가 자연스럽다

    생성자 주입 예시 

    class Cashier {
  private final CoffeeMaker coffeeMaker;

  public Cashier(CoffeeMaker coffeeMaker) {
    this.coffeeMaker = coffeeMaker;
  }

  public Coffee order(String menuName) {
    return coffeeMaker.make(menuName);
  }
}

    이 방식이 흔히 말하는 의존성 주입이다. 협력과 메시징을 더 깔끔하게 만들고, 결합도는 낮추고 응집도는 높이기 좋다.

    DIP와 OCP를 구조로 이해하기

    DIP: 고수준은 추상에 의존

    • High Level인 Cashier는 구체 클래스 Barista가 아니라 추상인 CoffeeMaker에 의존한다
    • Low Level인 Barista가 CoffeeMaker를 구현한다
    • 둘을 연결하는 조립 책임은 밖에서 수행한다
    interface CoffeeMaker {
  Coffee make(String menuName);
}

class Barista implements CoffeeMaker {
  public Coffee make(String menuName) {
    return new Coffee(menuName);
  }
}

class Config {
  public static Cashier cashier() {
    return new Cashier(new Barista());
  }
}

    OCP: 확장에는 열려 있고 변경에는 닫혀 있게

    새로운 바리스타 구현체를 추가해도, Cashier 코드는 바뀌지 않게 만드는 것이 핵심이다. 바뀌는 건 조립 코드만 바뀌고, 주문 로직은 그대로 유지된다. 

    class CatBarista implements CoffeeMaker {
  public Coffee make(String menuName) {
    return new Coffee(menuName);
  }
}

// 변경은 Config에서만
class Config {
  public static Cashier cashier() {
    return new Cashier(new CatBarista());
  }
}

    추상 클래스와 인터페이스를 언제 쓰나

    판단 기준

    구분 자연스러운 관계 추천 상황
    추상 클래스 IS-A 공통 상태와 공통 메서드를 공유해야 할 때
    인터페이스 CAN-DO 서로 다른 클래스에 공통 기능을 붙이고, 역할 중심으로 설계하고 싶을 때

    다형성: 하나의 타입으로 여러 객체를 다루기

    핵심 정의

    다형성은 부모 타입 하나로 여러 자식 타입 객체를 다루는 기술이다. 역할을 기준으로 코드를 작성하면, 구현체가 바뀌어도 코드가 덜 흔들린다.

    매개변수 다형성, 리턴 타입 다형성 

    // 매개변수 다형성: 부모 타입을 받으면 모든 자식을 받을 수 있다
public void serve(CoffeeMaker maker) {
  maker.make("latte");
}

// 리턴 타입 다형성: 부모 타입으로 반환하면, 내부에서 어떤 자식을 반환해도 된다
public CoffeeMaker selectMaker(boolean isMorning) {
  return isMorning ? new Barista() : new CatBarista();
}

    record 클래스란 무엇인가

    한 줄 정의

    record는 값만 담는 객체를 만들 때 반복되는 코드를 줄여주는 문법이다. 보통 DTO처럼 데이터 전달이 목적일 때 유용하다.

    왜 필요한가

    일반 클래스로 값을 담는 객체를 만들면 생성자, getter, equals, hashCode, toString을 매번 작성하게 된다. record를 쓰면 이런 반복을 줄이고, 의도가 더 명확해진다. 

    // 일반 클래스라면 이런 코드가 계속 늘어난다
class Menu {
  private final String name;
  private final int price;

  public Menu(String name, int price) {
    this.name = name;
    this.price = price;
  }

  public String getName() { return name; }
  public int getPrice() { return price; }
}

// record는 핵심만 적는다
record MenuRecord(String name, int price) { }

    머릿속 그림 

    record MenuRecord(name, price)

  자동으로 제공되는 것들
  - 생성자
  - name(), price() 접근 메서드
  - equals(), hashCode()
  - toString()

    오늘의 회고

    오늘 느낀 점

    • 객체지향은 클래스 개수 늘리는 게 아니라, 협력 구조를 잘 잡는 게 핵심이었다
    • DI는 편의 기능이 아니라, 결합도를 낮추는 설계 도구였다
    • record는 DTO를 만들 때 반복 코드를 줄여서, 의도를 더 선명하게 만들어줬다
