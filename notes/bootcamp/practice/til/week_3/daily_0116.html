    전략 패턴(Strategy Pattern)

    정의

    전략 패턴은 알고리즘(전략)을 별도의 클래스로 캡슐화하고, 필요할 때마다 교체해서 사용하는 패턴이다. 핵심은 무엇을 하는가(인터페이스)와 어떻게 하는가(구현체)를 완전히 분리하는 것이다.

    이점

    • OCP를 지키기 좋다: 새로운 전략이 추가되어도 기존 코드를 수정하지 않는다
    • 런타임 교체가 가능하다: 실행 중 상황에 맞게 알고리즘을 바꿀 수 있다
    • 단위 테스트가 편하다: 전략이 독립된 단위라 원하는 알고리즘만 분리해 검증 가능하다

    단점과 현실적인 해결

    전략이 늘어날수록 클래스 수도 같이 늘어나는 문제가 생긴다. 재사용이 필요 없거나 전략이 단순한 경우에는 익명 클래스 또는 람다로 전략을 바로 전달하는 방식이 유용했다.

    전략 패턴 구조: 인터페이스를 매개변수로 받기

    핵심 구조

    역할 의미 예시
    Context 전략을 사용해서 일을 수행하는 쪽 HamburgerChef
    Strategy 무엇을 할지 규격만 정의한 인터페이스 BurgerRecipe
    Concrete Strategy 실제 알고리즘 구현체 특정 레시피 구현

    매개변수를 인터페이스 타입으로 받으면, 구현체는 무엇이든 들어올 수 있고, Context는 구체 구현을 몰라도 된다. 이 구조가 전략 패턴을 성립시키는 기반이다.

    예시 코드(HamburgerChef) 

    public class HamburgerChef {
  // 전략 패턴을 쓸 때는 알고리즘이 변경될 수 있으므로 final을 두지 않는다
  private BurgerRecipe recipe;

  // 전략 변경 메서드(Setter)
  public void setRecipe(BurgerRecipe recipe) {
    this.recipe = recipe;
  }

  public void makeBurger() {
    System.out.println("--- 🍔 조리 시작 ---");
    System.out.println("# 번(빵)을 따뜻하게 굽습니다.");
    System.out.println("# 육즙 가득한 패티를 올립니다.");

    if (recipe != null) {
      recipe.cook(); // 어떤 전략이 들어왔는지는 모르지만, 규격대로 실행한다
    }

    System.out.println("# 빵을 덮고 포장합니다.");
    System.out.println("--- ✅ 조리 완료 ---\n");
  }
}

    전략을 주입하는 방법 1: 구현 클래스로 분리

    장점

    • 전략이 명확하게 이름을 가진다
    • 재사용이 쉽고 테스트하기 편하다

    단점

    전략이 많아지면 클래스 파일도 같이 늘어나서 관리 포인트가 늘어난다. 특히 재사용하지 않을 전략까지 모두 클래스로 만들면 오히려 피로도가 올라간다.

    전략을 주입하는 방법 2: 익명 클래스

    재사용하지 않을 전략을 빠르게 만들기

    특정 상황에서 한 번만 쓰는 전략이면, 익명 클래스로 바로 만들어 전달하면 된다. 필요하면 변수에 담아서 재사용할 수도 있다. 

    public class Main {
  public static void main(String[] args) {
    HamburgerChef chef = new HamburgerChef();

    BurgerRecipe hellRecipe = new BurgerRecipe() {
      @Override
      public void cook() {
        System.out.println("# 지옥에서 만든 특제 소스를 곁들입니다.");
      }
    };

    chef.setRecipe(hellRecipe);
    chef.makeBurger();

    chef.setRecipe(new BurgerRecipe() {
      @Override
      public void cook() {
        System.out.println("# 양상추 3장을 추가합니다.");
      }
    });
    chef.makeBurger();
  }
}

    한계

    익명 클래스는 코드가 길어지기 쉽다. 전략이 단순한데도 문법이 무거워져서, 읽는 입장에서 흐름이 끊긴다.

    전략을 주입하는 방법 3: Lambda

    짧은 전략은 람다가 깔끔하다 

    public class Main {
  public static void main(String[] args) {
    HamburgerChef chef = new HamburgerChef();

    BurgerRecipe hellRecipe = () -> {
      System.out.println("# 지옥에서 만든 특제 소스를 곁들입니다.");
    };

    chef.setRecipe(hellRecipe);
    chef.makeBurger();

    chef.setRecipe(() ->
      System.out.println("# 양상추 3장을 추가합니다.")
    );
    chef.makeBurger();
  }
}

    제약사항(중요)

    람다는 추상 메서드가 정확히 1개인 인터페이스에서만 사용할 수 있다. 그래서 이 조건을 만족하는 인터페이스를 함수형 인터페이스라고 부른다. 

    @FunctionalInterface
public interface BurgerRecipe {
  void cook();

  // 추상 메서드가 아닌 메서드는 추가 가능
  static void foo() {}
  default void bar() {}
}

    @FunctionalInterface는 기능을 강제로 만드는 게 아니라, 개발자가 실수로 추상 메서드를 2개 이상 추가하지 않도록 컴파일 단계에서 알려주는 역할을 한다.

    계산기 예제로 보는 전략 패턴

    핵심 아이디어

    계산도 결국 전략이다. 같은 입력을 받아도 덧셈, 뺄셈, 곱셈처럼 알고리즘만 바뀌는 구조라서, 인터페이스를 전략으로 두고 구현을 교체하는 방식이 자연스럽다. 

    // 덧셈(람다)
Calculator add = (num1, num2) -> num1 + num2;

// 뺄셈(익명 클래스)
Calculator sub = new Calculator() {
  @Override
  public int operate(int num1, int num2) {
    return num1 - num2;
  }
};

// 곱셈(람다)
Calculator multiply = (num1, num2) -> num1 * num2;

    오늘의 회고

    오늘 느낀 점

    • 전략 패턴은 결국 역할과 구현을 분리해서 유연함을 확보하는 방식이었다
    • 전략이 단순하고 재사용이 없으면 익명 클래스나 람다가 더 실용적이었다
    • 람다를 쓰려면 인터페이스가 함수형 인터페이스인지 먼저 확인하는 습관이 필요했다
