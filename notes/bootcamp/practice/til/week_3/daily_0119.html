    내일배움캠프 Spring 백엔드 26일차 TIL — 제네릭 복습과 Stream 기본 흐름

    작성일 2026-01-19 · 분류: 부트캠프, TIL, Java, Generic, Stream

    26일차 개요

    오늘 목표

    • 제네릭으로 타입 안정성을 확보하는 이유를 다시 잡는다
    • 제네릭 클래스와 제네릭 메서드를 구분해서 이해한다
    • Stream이 기존 반복문 코드와 무엇이 다른지, 흐름을 기준으로 정리한다
    • 자주 쓰는 Stream 연산(필터링, 변환, 정렬, 최종 연산)을 예제로 묶는다

    제네릭이 필요한 이유

    Object로 처리할 때의 문제

    제네릭이 없으면 모든 타입을 담기 위해 Object를 사용할 수 있다. 문제는 꺼낼 때 형변환이 필요해지고, 잘못된 다운캐스팅이 섞이면 실행 중 예외가 터진다는 점이다. 즉, 코드가 길어지고 안전하지도 않다.

    제네릭이 해결하는 것

    포인트 설명 결과
    타입 안정성 선언한 타입 외에는 들어오지 못하게 막는다 엉뚱한 타입으로 인한 예외 가능성이 크게 줄어든다
    형변환 제거 꺼낼 때 형변환을 하지 않는다 코드가 깔끔해지고 실수 포인트가 줄어든다

    제네릭 클래스 vs 제네릭 메서드

    왜 둘을 나눠서 쓰나

    • 유연함: 클래스 전체를 바꾸지 않고, 필요한 메서드만 제네릭으로 만들 수 있다
    • 범위: 제네릭 클래스의 타입 파라미터는 클래스 전체에 영향, 제네릭 메서드는 해당 메서드 안에서만 영향

    예시 코드 

    // 제네릭 메서드: 메서드 선언부에 타입 파라미터를 붙인다
public static <T> T scanAndReturn(T item) {
  return item;
}

// 타입 제한: Number 하위만 허용
public static <T extends Number> double toDouble(T number) {
  return number.doubleValue();
}

    타입 제한(extends)을 거는 이유

    제한이 없으면 생기는 문제

    타입 제한이 없으면 메서드 내부에서 해당 타입이 가진 기능을 확신하고 쓰기 어렵다. 반대로 제한을 걸면, 그 부모 타입의 기능을 전제로 코드 작성이 가능해진다.

    • 최소한의 안전장치: 관련 있는 타입만 통과하도록 제한한다
    • 부모의 능력 사용: 예를 들어 Number로 제한하면 doubleValue 같은 메서드를 안정적으로 쓸 수 있다

    Stream이 필요한 이유

    반복문 중심 코드의 한계

    기존 방식은 임시 리스트를 만들고, 반복문과 조건문으로 걸러내고, 정렬하고, 다시 반복문으로 출력하는 흐름이 많았다. 과정이 길어질수록 한 번에 읽기가 어렵고, 실수 포인트도 늘어난다.

    Stream이 바꾸는 관점

    Stream은 작업을 파이프라인으로 연결해서, 어떻게 반복할지보다 무엇을 할지에 집중하게 만든다. 그래서 흐름이 짧아지고, 의도가 더 잘 보인다.

    Stream 핵심 연산 정리

    중간 연산과 최종 연산

    분류 연산 설명
    필터링/슬라이싱 filter 조건이 true인 데이터만 통과
    필터링/슬라이싱 distinct 중복 제거( equals, hashCode 구현이 중요 )
    필터링/슬라이싱 limit 정해진 개수만 통과
    변환 map 데이터 형태 변환, 특정 필드만 추출
    정렬 sorted 정렬 기준을 지정해서 정렬
    최종 연산 collect 리스트/맵 등으로 수집
    최종 연산 reduce 누적 연산으로 하나의 결과로 만들기
    최종 연산 groupingBy 그룹별로 묶기
    최종 연산 partitioningBy true/false 두 그룹으로 나누기

    실수 포인트 메모

    NPE 방지 습관

    equals 비교는 null 가능성이 있는 쪽을 뒤로 보내는 습관이 안전했다. 

    // 위험: idol.getTeam()이 null이면 NullPointerException 가능
// idol.getTeam().equals("IVE");

// 안전: 앞쪽이 상수라 null이 아니다
"IVE".equals(idol.getTeam());

    distinct 사용 조건

    distinct는 같은 객체를 알아보려면 비교 기준이 필요하다. 객체 타입에서 중복 제거를 기대한다면 equals와 hashCode 구현이 전제다.

    예제 1: map으로 변환하기

    이름만 추출, 길이 변환, DTO로 변환 

    package kr.spartaclub.running.day03.step2;

import java.util.HashMap;
import java.util.List;
import java.util.Map;

import static kr.spartaclub.running.day03.step2.Menu.menuList;

public class Mapping {
  public static void main(String[] args) {

    // 요리 이름들만 추출
    List<String> dishNames = menuList.stream()
      .map(Dish::getName)
      .toList();

    System.out.println(dishNames);

    System.out.println("==========================================================");

    List<String> browsers = List.of("safari", "chrome", "ms edge", "opera", "firefox");

    browsers.stream()
      .map(String::length)
      .forEach(len -> System.out.printf("%d ", len));

    System.out.println();
    System.out.println("==========================================================");

    browsers.stream()
      .map(browser -> browser.charAt(0))
      .forEach(System.out::println);

    System.out.println("==========================================================");

    // 보기 좋게 조합(이름:칼로리)
    menuList.stream()
      .map(dish -> dish.getName() + ":" + dish.getCalories())
      .forEach(System.out::println);

    System.out.println("==========================================================");

    // 요리 이름과 칼로리만 Map으로 추출
    List<Map<String, Object>> results = menuList.stream()
      .map(dish -> {
        Map<String, Object> menuMap = new HashMap<>();
        menuMap.put("menuName", dish.getName());
        menuMap.put("calories", dish.getCalories());
        return menuMap;
      })
      .toList();

    results.forEach(System.out::println);

    System.out.println("==========================================================");

    // DTO로 변환해서 필요한 컬럼만 다루기
    List<SimpleDish> simpleDishList = menuList.stream()
      .map(SimpleDish::new)
      .toList();

    simpleDishList.forEach(System.out::println);

    System.out.println("==========================================================");

    // 500칼로리 이상인 요리 이름만 출력
    menuList.stream()
      .filter(dish -> dish.getCalories() >= 500)
      .map(Dish::getName)
      .forEach(System.out::println);

    System.out.println("==========================================================");

    // 500칼로리 이상인 요리 이름과 칼로리 출력(DTO)
    menuList.stream()
      .filter(dish -> dish.getCalories() >= 500)
      .map(SimpleDish::new)
      .forEach(System.out::println);
  }
}

    예제 2: 정렬과 limit 조합

    필터링 + 정렬 + 상위 N개 

    package kr.spartaclub.running.day03.step2;

import java.util.Comparator;
import java.util.List;

import static kr.spartaclub.running.day03.step2.DishType.MEAT;
import static kr.spartaclub.running.day03.step2.Menu.menuList;

public class Sorting {
  public static void main(String[] args) {

    // 육류 요리 중 칼로리가 낮은 순으로 정렬
    List<Dish> lowCalorieMeats = menuList.stream()
      .filter(menu -> menu.getType() == MEAT)
      .sorted(Comparator.comparing(Dish::getCalories))
      .toList();

    lowCalorieMeats.forEach(System.out::println);

    System.out.println("========================");

    // 메뉴들을 이름 내림차순으로 정렬
    menuList.stream()
      .sorted(Comparator.comparing(Dish::getName).reversed())
      .forEach(System.out::println);

    System.out.println("========================");

    // 300칼로리 이상인 요리 중 칼로리가 낮은 Top3
    menuList.stream()
      .filter(m -> m.getCalories() >= 300)
      .sorted(Comparator.comparing(Dish::getCalories))
      .limit(3)
      .forEach(System.out::println);
  }
}

    오늘의 회고

    오늘 느낀 점

    • 제네릭은 결국 타입 안정성을 얻기 위한 장치라는 감각이 더 선명해졌다
    • Stream은 기능 자체보다, 파이프라인으로 의도를 드러내는 방식이 더 강력했다
    • distinct와 equals, hashCode 관계는 앞으로도 실수하기 쉬운 포인트라 따로 기억해두기로 했다
