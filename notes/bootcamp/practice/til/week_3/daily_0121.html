    내일배움캠프 Spring 백엔드 27일차 TIL — 커스텀 예외와 Optional 실전 감각

    작성일 2026-01-20 · 분류: 부트캠프, TIL, Java, Exception, Optional

    27일차 개요

    오늘 목표

    • 예외를 의미 있게 던지는 방법을 정리한다
    • 커스텀 예외로 상황을 명확히 만드는 기준을 잡는다
    • Optional로 NPE를 줄이는 흐름을 예제로 익힌다

    왜 커스텀 예외가 필요할까

    정의

    커스텀 예외는 우리 서비스의 실패 상황을 예외 타입으로 명확하게 표현하는 방식이다. 단순히 Exception이나 RuntimeException을 그대로 던지면 로그를 봤을 때 어떤 문제가 터진 건지 한 번에 파악하기 어렵다.

    용도

    • 예외 타입만 봐도 의도가 드러나게 만든다
    • 에러 응답이 필요한 상황에서 상태코드, 발생 시각 같은 부가 정보를 같이 들고 갈 수 있다
    • 공통 처리(로깅, 응답 변환)에서 분기 기준을 예외 타입으로 세울 수 있다

    커스텀 예외 기본 형태

    예시 코드 

    import java.time.LocalDateTime;

public class BusinessException extends RuntimeException {

  private final int status;
  private final LocalDateTime timeStamp;

  public BusinessException(int status, String message) {
    super(message);
    this.status = status;
    this.timeStamp = LocalDateTime.now();
  }

  public int getStatus() {
    return status;
  }

  public LocalDateTime getTimeStamp() {
    return timeStamp;
  }
}

    예외에 필드를 추가하면, 단순 메시지보다 더 풍부한 정보를 함께 전달할 수 있다. 실제로는 프로젝트 스타일에 맞춰 필드 구성은 달라질 수 있다.

    검사 예외와 비검사 예외 선택 기준

    핵심 정리

    구분 대표 상속 특징 어울리는 상황
    검사 예외 Exception 컴파일 단계에서 처리 강제 호출자가 현실적으로 복구 로직을 가질 수 있는 경우
    비검사 예외 RuntimeException 처리 강제가 없고 흐름이 단순 프로그래밍 오류, 계약 위반, 입력 검증 실패처럼 보통은 호출자가 즉시 복구하지 않는 경우

    중요한 건 검사 예외가 무조건 더 좋은 것도 아니고, 비검사 예외가 무조건 대충 던지는 것도 아니라는 점이다. 팀 규칙과 처리 전략(로깅, 응답 변환, 재시도 정책)에 맞춰 일관되게 선택하는 게 더 중요했다.

    Optional로 NPE 줄이기

    정의

    Optional은 값이 있을 수도, 없을 수도 있다는 상태를 타입으로 표현하는 컨테이너다. 값이 없을 가능성이 있는 결과를 그냥 null로 던져두면, 결국 호출자는 방어 코드(!= null)를 계속 쓰게 되고 코드가 지저분해진다.

    생성 방식 

    // null 가능성이 있는 값을 감싸기
Optional<Coffee> coffeeOpt = Optional.ofNullable(coffee);

    Optional API 자주 쓰는 패턴

    핵심 메서드 정리

    메서드 의미 한 줄 감각
    ifPresent 값이 있으면 실행 있을 때만 동작
    orElse 없으면 기본값 사용 기본값 즉시 준비
    orElseGet 없으면 공급자 함수 실행 필요할 때만 생성
    orElseThrow 없으면 예외 발생 없으면 실패로 처리
    map 꺼내지 않고 내부에서 변환 변환을 체인으로

    예시 코드 

    // 1) 있으면 실행
coffeeOpt.ifPresent(c -> System.out.println(c));

// 2) 없으면 기본값
Coffee coffee = coffeeOpt.orElse(new Coffee("아메리카노"));

// 3) 없으면 기본값을 지연 생성
Coffee coffee2 = coffeeOpt.orElseGet(() -> new Coffee("아메리카노"));

// 4) 없으면 예외
Coffee coffee3 = coffeeOpt.orElseThrow(() -> new RuntimeException("값이 없습니다."));

// 5) 꺼내지 말고 안에서 변환하는 방식 권장
String name = coffeeOpt
  .map(Coffee::getCoffeeName)
  .orElse("커피 없음");

    Optional 사용 주의점

    체크리스트

    • Optional 자체를 null로 리턴하지 않는다. 값이 없으면 Optional.empty()가 기본이다
    • 필드나 파라미터에 Optional을 쓰는 건 일반적으로 권장되지 않는다. 보통은 리턴 타입에서 의미가 가장 선명하다
    • 무조건 Optional이 정답은 아니다. 리스트라면 Collections.emptyList()처럼 빈 컬렉션이 더 자연스러운 경우도 많다

    오늘의 회고

    오늘 느낀 점

    • 예외는 던지는 것보다, 읽히게 만드는 것이 더 중요하다는 감각이 생겼다
    • Optional은 null 방어를 줄여주지만, 상황에 맞게 쓰는 기준을 같이 가져가야 했다
    • orElse와 orElseGet 차이는 다음에 성능 이슈 잡을 때 바로 써먹을 수 있을 것 같다
