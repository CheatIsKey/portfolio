    28일차 개요

    오늘 목표

    • 객체 지향적 데이터 구조(Category와 Product의 관계)를 코드로 구현한다
    • Scanner와 반복문을 활용해 뎁스(Depth)가 있는 메뉴 구조를 만든다
    • 화면 출력 시 가독성을 높이기 위한 문자열 포맷팅을 적용한다

    객체 관계 설계 (1:N 구조)

    카테고리와 상품의 포함 관계

    이번 과제의 핵심은 데이터를 단순히 나열하는 것이 아니라, 계층 구조로 관리하는 것이었다. Category 클래스가 Product 리스트를 멤버 변수로 가지게 설계함으로써, 카테고리만 선택하면 그 안의 상품 목록을 바로 가져올 수 있는 구조를 만들었다. 

    public class Category {
    // 1:N 관계 - 하나의 카테고리는 여러 상품을 가짐
    private final List<Product> products = new ArrayList<>();
    private final String categoryName;

    public void addProduct(Product product) {
        products.add(product);
    }

    public List<Product> getProducts() {
        return products;
    }
}

    데이터 주입 방식 (Dependency Injection)

    CommerceSystem 내부에서 데이터를 직접 생성하지 않고, Main에서 생성한 데이터를 생성자를 통해 주입받는 방식을 사용했다. 이렇게 하면 로직(System)과 데이터(Category, Product)가 분리되어 관리가 쉬워진다. 

    // Main.java
CommerceSystem commerceSystem = new CommerceSystem(List.of(electronics, kitchen, foods));
commerceSystem.start();

// CommerceSystem.java
public CommerceSystem(List<Category> categories) {
    this.categories = categories;
}

    콘솔 UI 포맷팅과 사용자 경험

    printf를 이용한 정렬

    단순 println만 사용하면 상품명 길이에 따라 가격 위치가 들쑥날쑥해져 가독성이 떨어진다. System.out.printf를 활용해 고정된 너비를 확보하고, 숫자에 천 단위 콤마를 찍어주었다.

    포맷 지정자 의미 사용 예시 출력 결과
    %-14s 문자열 14칸 확보 (왼쪽 정렬) "만두" 만두
    %,9d 숫자 9칸 확보 + 3자리 콤마 3500 3,500

    실제 적용 코드 

    // 상품명은 왼쪽 정렬, 가격은 콤마 포함하여 출력
System.out.printf("%d. %-14s | %,9d원 | %s%n", 
    i + 1, name, price, description);

    트러블 슈팅: 입력값 검증과 인덱스

    문제 상황: 사용자 입력과 리스트 인덱스의 불일치

    사용자는 메뉴를 보고 1번부터 선택하지만, 실제 List는 0번 인덱스부터 시작한다. 이 차이를 처리하는 과정에서 IndexOutOfBoundsException이 발생할 위험이 있었다.

    해결 로직

    입력받은 숫자에서 1을 뺀 값(selectMenuNum - 1)을 인덱스로 사용하기 전에, 반드시 원본 입력값(1부터 시작하는 값) 기준으로 유효 범위를 먼저 검증해야 안전하다. 

    // 1. 범위 검증 (사용자 입력 기준)
if (selectMenuNum < 1 || selectMenuNum > categories.size()) {
    System.out.println("잘못된 입력입니다.");
    continue;
}

// 2. 안전하게 인덱스로 변환하여 접근
Category category = categories.get(selectMenuNum - 1);

    Scanner 버퍼 처리 주의점

    nextInt()는 숫자만 읽어가고 엔터(개행문자)를 버퍼에 남겨둔다. 만약 이후에 nextLine()을 호출하면 남겨진 엔터를 읽고 바로 넘어가버리는 문제가 발생할 수 있다. 이번 과제에서는 nextInt() 위주로 사용했지만, 문자열 입력을 섞어 쓸 때는 scan.nextLine()으로 버퍼를 비워주는 습관이 필요하다.

    오늘의 회고

    오늘 느낀 점

    • main 메서드는 데이터를 준비만 하고, 실제 로직은 객체에게 위임하니 코드가 훨씬 깔끔해졌다
    • printf 포맷팅을 적용하자마자 콘솔 창이 실제 서비스처럼 정돈되어 보이는 효과가 컸다
    • 다음 단계에서는 start() 메서드 안에 뭉쳐있는 while문과 출력 로직을 별도 메서드로 분리하는 리팩토링이 필요해 보인다
