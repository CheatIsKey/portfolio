    내일배움캠프 Spring 백엔드 30일차 TIL — 게임 로직과 분기 설계

    작성일 2026-01-23 · 분류: 부트캠프, TIL, Flowchart, Algorithm

    30일차 개요

    오늘 목표

    • 사용자 입력과 랜덤 값이 상호작용하는 게임 로직을 설계한다
    • 업다운(Up-Down) 로직과 상성(가위바위보) 로직의 차이를 이해한다
    • 반복문 내에서의 예외 처리와 상태 유지 전략을 흐름도로 정리한다

    1. 숫자 맞추기 게임

    입력/출력 정리

    • 입력: 1부터 100 사이의 숫자 input
    • 출력:
      • 정답 일치 시: "정답입니다!"
      • 정답보다 작을 때: "입력한 정수보다 작습니다!" (Up 힌트 필요 상황)
      • 정답보다 클 때: "입력한 정수보다 큽니다!" (Down 힌트 필요 상황)
    • 상태: 생성된 정답 숫자 randomInt (1~100)

    조건 정리

    • 종료 조건: 사용자가 randomInt를 맞추는 경우
    • 분기 조건:
      • 유효한 정수 입력 여부
      • 1~100 사이 범위 여부
      • 입력값과 정답의 대소 비교 (같음/작음/큼)
    • 예외 처리:
      • 정수가 아닐 때: "정수를 입력해주세요."
      • 범위 벗어날 때: "1과 100 사이만 입력가능합니다."

    흐름 설계

    1. 초기화: 게임 시작 시 randomInt를 1~100 사이 랜덤 값으로 한 번만 할당한다.
    2. 반복 구간:
      • 사용자로부터 input을 입력받는다.
      • 유효성 검사: 정수가 아니거나, 범위를 벗어나면 에러 메시지를 출력하고 다시 입력 단계로 돌아간다.
      • 정답 검사: input이 randomInt와 같은지 확인한다.
        • Yes: "정답입니다!" 출력 후 종료.
        • No: 대소 비교로 넘어간다.
      • 대소 비교:
        • input < randomInt이면 "입력한 정수보다 작습니다!" 출력.
        • input > randomInt이면 "입력한 정수보다 큽니다!" 출력.
        • 출력 후 다시 입력 단계로 돌아간다.
    3. 종료: 정답을 맞추면 프로그램이 끝난다.

    2. 가위바위보 게임

    입력/출력 정리

    • 입력: 가위(1), 바위(2), 보(3) 중 하나를 정수로 입력
    • 출력: "무승부입니다!", "사용자가 이겼습니다!", "컴퓨터가 이겼습니다!"
    • 상태: 컴퓨터의 선택값 computer (1~3 랜덤)

    조건 정리

    • 종료 조건: 승패가 결정되거나 무승부가 나는 즉시 종료 (단판 승부)
    • 승리 조건 (사용자 기준):
      • 가위(1) vs 보(3)
      • 바위(2) vs 가위(1)
      • 보(3) vs 바위(2)
    • 예외 처리: 정수가 아니거나 1~3 범위를 벗어난 경우 에러 메시지 출력 후 재입력

    흐름 설계

    1. 초기화: computer 변수에 1~3 사이 랜덤 값을 할당한다.
    2. 입력 및 검증 (반복):
      • 사용자 입력을 받는다.
      • 입력이 정수가 아니거나 1~3이 아니라면 에러 메시지를 출력하고 다시 입력을 받는다.
      • 유효한 입력이라면 승패 판정으로 넘어간다.
    3. 승패 판정:
      • 무승부: 입력값과 computer 값이 같은 경우.
      • 사용자 승리: 가위-보, 바위-가위, 보-바위의 상성 관계인 경우.
      • 컴퓨터 승리: 그 외의 모든 경우.
    4. 종료: 결과를 출력하고 프로그램을 종료한다.

    오늘의 회고

    오늘 느낀 점

    • 숫자 맞추기 게임에서 랜덤 값 생성 위치가 반복문 안인지 밖인지에 따라 게임의 성립 여부가 달라진다는 점을 확실히 깨달았다.
    • 가위바위보처럼 경우의 수가 정해진 로직은 if-else의 순서보다, 승리 조건을 먼저 정의하고 나머지를 패배로 처리하는 방식이 깔끔했다.
    • 예외 처리는 기능 구현의 마지막이 아니라, 입력 받는 즉시 수행해야 로직이 꼬이지 않는다.
