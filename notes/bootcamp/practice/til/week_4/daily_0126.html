    오늘 목표

    알고리즘과 복잡도

    알고리즘이란?

    💡 요리 레시피와 같다
    문제를 해결하기 위해 수행해야 하는 일련의 절차나 방법이다.
    "라면을 끓인다"는 목표를 위해 [물 끓이기 → 면 넣기 → 스프 넣기 → 기다리기] 순서로 진행하는 것 자체가 알고리즘이다.

    시간 복잡도 (Time Complexity)

    알고리즘이 문제를 해결하는 데 얼마나 많은 연산을 수행하는지를 나타낸 척도다. 단순히 "몇 초 걸린다"가 아니라, 입력 데이터(N)가 늘어날 때 시간이 얼마나 늘어나는가?를 따진다.

    표기법 (Big-O) 이름 비유 (직관적 이해) 실행 속도
    O(1) 상수 시간 서랍의 첫 번째 칸을 여는 것 (데이터가 100만 개여도 한 번에 찾음) 매우 빠름
    O(log N) 로그 시간 Up-Down 게임 (절반씩 버리면서 찾기) 빠름
    O(N) 선형 시간 책의 모든 페이지를 처음부터 끝까지 한 장씩 넘겨보기 보통
    O(N²) 이차 시간 구구단 외우기 (N단 * N개) 느림

    공간 복잡도 (Space Complexity)

    알고리즘을 실행하는 데 얼마나 많은 메모리(RAM)가 필요한지를 나타낸다. 변수 하나만 쓰면 적게 쓰는 것이고, 거대한 배열을 새로 만들면 많이 쓰는 것이다. 최근에는 하드웨어 성능이 좋아져서, 시간 복잡도를 줄이기 위해 공간(메모리)을 더 쓰는 전략(예: DP, 해시테이블)을 주로 취한다.

    핵심 알고리즘 5가지 패턴

    문제를 만났을 때 어떤 도구를 꺼내야 할지 결정하는 기준이 된다.

    1. 완전 탐색 (Brute Force)

    • 개념: 무식하게 모든 경우의 수를 다 해보는 방법.
    • 비유: 자물쇠 비밀번호 4자리를 풀기 위해 0000부터 9999까지 다 돌려보기.
    • 장단점: 답은 무조건 찾지만(정확성 100%), 시간이 너무 오래 걸린다.

    2. 그리디 (Greedy)

    • 개념: 현재 상황에서 당장 가장 좋아 보이는 것만 고르는 방법.
    • 비유: 거스름돈을 줄 때, 무조건 가장 큰 동전부터 꺼내서 주는 것.
    • 장단점: 속도가 매우 빠르지만, 항상 최적의 해를 보장하진 않는다. (마시멜로 실험에서 당장 1개를 먹는 것과 같음)

    3. 백트래킹 (Backtracking)

    • 개념: 가다가 아니면 되돌아오는 방식. 불필요한 경로는 미리 포기(가지치기, Pruning)한다.
    • 비유: 미로 찾기를 하다가 막힌 길이 나오면, 갈림길로 다시 돌아가서 다른 길로 가는 것.
    • 특징: 완전 탐색보다 효율적이다. (모든 길을 끝까지 가보지 않아도 되니까)

    4. 분할 정복 (Divide & Conquer)

    • 개념: 큰 문제를 작은 문제로 쪼개서(Divide) 해결하고, 나중에 합치는(Conquer) 방식.
    • 비유: 두꺼운 전화번호부를 반으로 찢고, 내가 찾는 이름이 있는 쪽만 남기는 과정을 반복하는 것.
    • 예시: 퀵 정렬, 병합 정렬.

    5. 동적 계획법 (Dynamic Programming, DP)

    • 개념: 한 번 푼 문제는 답을 적어두고(Memoization), 다시 풀지 않고 재활용하는 방식.
    • 비유: 수학 시험에서 12 * 34 계산 결과가 문제 1번에도 나오고 5번에도 나온다면, 5번 풀 때는 다시 계산 안 하고 아까 적어둔 답을 베끼는 것.
    • 특징: 점화식을 세우는 것이 핵심이다. (피보나치 수열이 대표적)

    Java Map 심화: 무엇을 골라야 할까?

    기본적인 HashMap 외에 상황에 따라 써야 하는 특수한 Map들이 있다.

    LinkedHashMap: 순서를 기억하는 Map 

    // 일반 HashMap: 넣은 순서와 상관없이 뒤섞임
Map<String, Integer> map = new HashMap<>();

// LinkedHashMap: 넣은 순서대로 차곡차곡 저장됨
Map<String, Integer> linkedMap = new LinkedHashMap<>();
    • 특징: HashMap은 순서가 뒤죽박죽이지만, LinkedHashMap은 입력된 순서(Insertion Order)를 보장한다.
    • 구조: 해시 테이블 + 더블 링크드 리스트(Doubly Linked List)가 결합된 형태다.
    • 용도: 최근 본 상품 목록, 캐시(LRU Cache) 구현 등 순서가 중요한 데이터를 저장할 때 쓴다.

    ConcurrentHashMap: 멀티쓰레드에 안전한 Map

    • 문제점: 일반 HashMap은 여러 쓰레드가 동시에 값을 넣거나 빼면 에러가 나거나 데이터가 깨진다.
    • 해결책: 동시성을 지원하는 ConcurrentHashMap을 사용한다.
    • 특징:
      • 옛날 방식인 Hashtable은 화장실 열쇠가 1개라 줄을 서야 했다(느림).
      • ConcurrentHashMap은 구역별로 잠금(Lock Striping)을 걸거나, CAS(Compare-And-Swap) 알고리즘을 써서 훨씬 빠르고 효율적이다.

    한 눈에 비교하기

    구분 HashMap LinkedHashMap ConcurrentHashMap
    순서 보장 X (무작위) O (입력 순서) X (무작위)
    동시성 이슈 발생함 (Not Safe) 발생함 (Not Safe) 안전함 (Thread Safe)
    주요 용도 일반적인 데이터 저장 순서가 필요한 캐시/히스토리 멀티쓰레드 환경/웹 서버 전역 변수

    오늘의 회고

    오늘 느낀 점

    • 알고리즘이 거창한 수학이 아니라, 효율적인 일 처리 방식이라는 점이 와닿았다
    • 코딩 테스트 문제에서 "N의 크기"를 보고 어떤 시간 복잡도를 써야 할지(O(N)인지 O(logN)인지) 역산하는 팁을 알게 되었다
    • Map도 무조건 HashMap만 쓰는 게 아니라, 순서가 필요한지, 동시성 처리가 필요한지에 따라 골라 써야 한다는 걸 배웠다
