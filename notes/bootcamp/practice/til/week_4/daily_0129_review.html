    34일차 개요 (1/2)

    오늘 목표

    • 라이브러리와 프레임워크의 제어권 차이를 명확히 이해한다.
    • 웹 통신의 기본인 HTTP 구조와 메서드 특징(멱등성, 안정성)을 정리한다.
    • 생산성을 높여주는 도구(Lombok, Annotation)와 서블릿의 역할을 파악한다.

    Spring과 개발 환경

    Spring vs Spring Boot

    • Spring Framework: 엔터프라이즈급 개발을 위한 강력한 기능을 제공하지만, 초기 환경 설정(XML, 의존성 등)을 개발자가 직접 하나하나 해줘야 하는 번거로움이 있다.
    • Spring Boot: Spring의 복잡한 초기 설정을 자동화(Auto Configuration)해주는 도구다. 자주 쓰이는 설정들을 미리 정의해두어 개발자가 비즈니스 로직에만 집중할 수 있게 해준다.

    라이브러리 vs 프레임워크

    구분 정의 제어권
    라이브러리 개발에 필요한 클래스, 함수들의 모음 개발자 → 라이브러리 (내가 필요할 때 가져다 씀)
    프레임워크 소프트웨어 개발을 위한 틀과 환경 프레임워크 → 개발자 (틀 안에서 내가 코드를 작성함)
    요약
    프레임워크는 라이브러리들의 모음이자 거대한 틀이다. 이 틀이 호환성을 미리 맞춰두었기에 팀 간의 코드 일관성이 높아지고 협업이 쉬워진다. 단, 프레임워크 사용법을 익히는 데는 학습 비용이 든다.

    HTTP 프로토콜 완전 정복

    웹에서 데이터를 주고받기 위한 통신 규약으로, 백엔드 개발자의 필수 상식이다.

    특징: 무상태 (Stateless)

    • 개념: 서버가 클라이언트의 이전 상태(State)를 기억하지 않는다.
    • 장점: 서버가 바뀌어도 상관없으므로 수평 확장(Scale Out)에 매우 유리하다. 요청량이 폭증하면 서버만 늘리면 된다.
    • 단점: 클라이언트가 매 요청마다 필요한 데이터(인증 정보 등)를 계속 담아서 보내야 한다.

    HTTP 메시지 구조

    요청(Request)과 응답(Response) 모두 크게 4부분으로 나뉜다.

    1. Start Line: 메서드(GET/POST), 경로(Path), 버전, 상태 코드(200 OK 등)
    2. Header: 데이터의 크기, 인증 토큰, 브라우저 정보 등 메타 데이터
    3. Empty Line: 헤더와 바디를 구분하는 공백 한 줄
    4. Message Body: 실제 전송할 데이터 (JSON, HTML 등)

    HTTP Method와 속성

    Method 용도 안정성 (Safety) 멱등성 (Idempotency) 캐시 가능
    GET 리소스 조회 O (데이터 변경 X) O (여러 번 호출해도 같음) O
    POST 리소스 생성 (등록) X X (호출할 때마다 계속 생성됨) △ (가능은 하나 잘 안 씀)
    PUT 전체 수정 (덮어쓰기) X O (결과가 항상 같음) X
    PATCH 부분 수정 X X (구현에 따라 다름) X
    DELETE 삭제 X O (삭제된 상태는 동일함) X
    멱등성(Idempotency)이 왜 중요한가?
    요청이 실패했을 때 "다시 요청(Retry)해도 되는가?"를 판단하는 기준이 된다. 멱등하다면 안심하고 재시도할 수 있지만, 그렇지 않다면(POST 등) 중복 결제 같은 사고를 막기 위해 별도의 검증 로직이 필요하다.

    API와 개발 생산성 도구

    REST API 디자인 원칙

    REST는 웹의 장점을 최대한 활용하기 위한 아키텍처 스타일이다.

    • 자원(Resource): URI로 식별한다. 동사(delete-user)보다는 명사(users)를, 단수보다는 복수를 사용한다.
    • 행위(Verb): 자원에 대한 행위는 HTTP Method(GET, POST 등)로 표현한다.
    • 표현(Representation): URI 마지막에 슬래시(/)를 넣지 않고, 밑줄(_) 대신 하이픈(-)을 사용하여 가독성을 높인다.

    Annotation과 Lombok

    어노테이션(Annotation)은 코드에 붙이는 라벨(메타 데이터)이다. 컴파일러나 런타임에게 "이 코드는 특별하게 처리해줘"라고 알리는 역할을 한다. Lombok은 이 어노테이션을 활용해 반복적인 코드(Boilerplate code)를 자동으로 생성해준다.

    • @NoArgsConstructor: 기본 생성자 생성
    • @AllArgsConstructor: 모든 필드를 받는 생성자 생성
    • @RequiredArgsConstructor: final이 붙은 필수 필드만 받는 생성자 생성 (의존성 주입 시 자주 사용)
    • @Slf4j: 로깅을 위한 Logger 객체 자동 생성

    서블릿과 컨테이너

    • 서블릿(Servlet): 클라이언트의 HTTP 요청을 받아 비즈니스 로직(조회, 계산 등)을 수행하고 응답을 돌려주는 자바 웹 프로그램이다.
    • 서블릿 컨테이너(Tomcat 등): 서블릿을 관리하는 매니저다. 서블릿의 생명주기(생성~소멸)를 관리하고, 동시에 여러 요청이 들어와도 멀티 쓰레드로 처리해준다.

    오늘의 회고 (Part 1)

    중간 점검

    • 프레임워크가 단순히 편한 도구가 아니라, '제어의 역전'을 통해 나를 틀 안에 가두고 보호해준다는 개념이 와닿았다.
    • HTTP 메서드의 멱등성은 단순히 이론이 아니라, 결제나 데이터 수정 시의 재시도 전략을 짤 때 핵심이 된다는 점을 배웠다.
    • 이제 Spring이 이 서블릿 컨테이너 위에서 어떻게 동작하는지 본격적으로 알아볼 준비가 되었다.
