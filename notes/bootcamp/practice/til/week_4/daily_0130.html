    35일차 개요

    오늘 목표

    • JPA를 활용하여 일정 관리 서버의 핵심 도메인과 리포지토리를 구현한다.
    • DTO(Data Transfer Object)를 사용하여 Controller와 Service 간의 데이터를 안전하게 주고받는다.
    • 개발 과정에서 마주친 데이터 바인딩 문제(@RequestParam vs @RequestBody)를 해결하고 정리한다.

    프로젝트 구현: 일정 관리 (진행 중)

    기본적인 CRUD 기능을 목표로 엔티티 설계부터 컨트롤러 구현까지 진행 중이다. 특히 반복되는 코드(생성일/수정일)를 제거하기 위해 JPA Auditing 기능을 적극 활용했다.

    1. BaseEntity로 공통 필드 관리

    모든 테이블에 공통으로 들어가는 createdAt(생성일), modifiedAt(수정일)을 매번 작성하는 것은 비효율적이다. 이를 해결하기 위해 @MappedSuperclass와 JPA Auditing을 적용했다. 

    @Getter
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {

    @CreatedDate
    @Column(updatable = false)
    private LocalDateTime createdAt;

    @LastModifiedDate
    private LocalDateTime modifiedAt;
}

    2. Repository의 메서드 명명 규칙

    JPA Repository의 강력한 기능인 쿼리 메서드(Query Method)를 사용하여, 복잡한 JPQL 작성 없이도 메서드 이름만으로 원하는 쿼리를 실행할 수 있었다. 

    public interface ScheduleRepository extends JpaRepository<Schedule, Long> {
    // 이름으로 검색하고 수정일 기준 내림차순 정렬
    List<Schedule> findByNameOrderByModifiedAtDesc(String name);

    // 전체 조회를 하되 수정일 기준 내림차순 정렬
    List<Schedule> findAllByOrderByModifiedAtDesc();
}

    트러블 슈팅: 데이터 바인딩의 차이

    문제 상황

    API를 개발하던 중, 클라이언트가 보내는 데이터를 컨트롤러에서 제대로 받지 못하는 경우가 있었다. 특히 GET 요청의 쿼리 파라미터와 POST 요청의 JSON 바디를 처리하는 방식이 혼동되었다.

    해결: @RequestParam vs @RequestBody 정리

    구분 @RequestParam @RequestBody
    데이터 위치 URL 뒤의 Query String
    (예: ?name=user&age=20)    		 HTTP 요청의 Body
    (주로 JSON 형태)
    주 사용 메서드 GET, DELETE
    (검색, 필터링, 단일 삭제)    		 POST, PUT, PATCH
    (데이터 생성, 수정)
    특징 단일 값이나 간단한 데이터를 받을 때 사용.
    required=false로 선택적 입력 가능.    		 복잡한 객체(DTO)를 통째로 받을 때 사용.
    MessageConverter가 JSON을 객체로 변환해줌.

    실제 코드 적용 예시 

    // 1. GET 요청: 검색 조건(name)은 쿼리 파라미터로 받음
@GetMapping("/api")
public List<ScheduleResponseDto> getSchedules(@RequestParam(required = false) String name) {
    return scheduleService.getSchedulesByName(name);
}

// 2. POST 요청: 생성할 일정 정보는 Body(JSON)로 받음
@PostMapping("/api")
public ScheduleResponseDto createSchedule(@RequestBody ScheduleRequestDto scheduleRequestDto) {
    return scheduleService.addSchedule(scheduleRequestDto);
}

    오늘의 회고

    오늘 느낀 점

    • JPA Auditing을 적용하니 createdAt 같은 반복 코드가 사라져 엔티티가 훨씬 깔끔해졌다. 실무에서도 필수일 것 같다.
    • DTO Record 타입을 처음 써봤는데, 불변(Immutable) 객체라 데이터 전달용으로 아주 적합하고 코드도 간결해서 좋았다.
    • Controller에서 데이터를 받을 때, "이 데이터가 URL에 있나? Body에 있나?"를 먼저 생각하고 어노테이션을 골라야 한다는 기준이 생겼다.
