    오늘 목표

    Spring MVC 아키텍처

    MVC 패턴이란?

    역할 분담의 기술
    하나의 서블릿이 모든 것을 처리하던 복잡함을 해결하기 위해 애플리케이션을 세 가지 역할로 나눈 설계 방식이다.
    • Model: 데이터와 비즈니스 로직을 담당한다.
    • View: 사용자에게 보여지는 화면(UI)을 담당한다.
    • Controller: 사용자의 요청(Request)을 받아 Model과 View를 연결해주는 중간 다리 역할을 한다.

    프론트 컨트롤러 패턴과 DispatcherServlet

    프론트 컨트롤러 패턴은 모든 클라이언트의 요청을 단일 진입점에서 받아 공통 처리를 수행하고, 세부 로직은 각 핸들러(Controller)에게 위임하는 방식이다. Spring MVC에서는 DispatcherServlet이 이 역할을 수행한다.

    • 장점: 공통 로직(보안, 인코딩 등)을 중앙에서 관리하여 코드 중복을 제거하고 유지보수성을 높인다.
    • ViewResolver: 컨트롤러가 반환한 논리적 뷰 이름(예: "home")을 실제 물리적 파일 경로(예: "/WEB-INF/views/home.html")로 변환해주는 역할을 한다.

    어댑터 패턴 (Adapter Pattern)

    • 개념: 서로 다른 인터페이스를 가진 클래스들을 연결해주는 패턴이다.
    • 특징: 다양한 방식의 컨트롤러를 통일된 방식으로 처리할 수 있게 해주며(다형성), 기존 코드를 수정하지 않고도 새로운 타입의 컨트롤러를 확장할 수 있다.

    렌더링 방식과 Thymeleaf

    SSR vs CSR 비교

    구분 SSR (Server Side Rendering) CSR (Client Side Rendering)
    방식 서버에서 HTML을 완성해서 브라우저에 전달한다. 서버는 빈 뼈대만 주고, 브라우저가 JS로 화면을 그린다.
    장점 초기 로딩이 빠르고 검색 엔진 최적화(SEO)에 유리하다. 화면 전환이 앱처럼 부드럽고 빠르다.
    단점 페이지 이동 시마다 새로고침이 발생할 수 있다. 초기 로딩 속도가 느릴 수 있다.
    대표 기술 Thymeleaf, JSP React, Vue, Angular

    Thymeleaf 기본 문법

    • 텍스트 출력 (th:text): 태그 안의 텍스트를 서버 데이터로 교체한다.
    • 속성 설정 (th:href, th:src): 링크나 이미지 경로 등의 속성을 동적으로 설정한다.
    • 반복 처리 (th:each): Java의 for-each문처럼 리스트 데이터를 반복해서 출력한다.
    • 조건 처리 (th:if, th:unless): 조건에 따라 태그를 렌더링하거나 제외한다.

    Spring Controller와 데이터 전달

    Controller 어노테이션

    • @Controller: 주로 View(화면)를 반환할 때 사용한다.
    • @RestController: @Controller + @ResponseBody가 합쳐진 형태다. View가 아니라 데이터(JSON 등)를 HTTP 응답 본문에 직접 담아 반환할 때 사용한다.
    • @RequestMapping: 특정 URL과 메서드를 매핑한다. 최근에는 @GetMapping, @PostMapping 등의 축약형을 주로 사용한다.

    데이터 요청 처리 (Parameter)

    어노테이션 용도 특징 및 예시
    @RequestParam 쿼리 파라미터
    (Query String)    		 /users?name=Steve 형태.
    required=false로 선택적 입력 가능.
    주로 검색 조건이나 필터링에 사용한다.
    @ModelAttribute 폼 데이터
    (Form Data)    		 HTML Form 태그의 데이터를 객체로 한 번에 바인딩한다.
    @PathVariable 경로 변수
    (Path Variable)    		 /users/123 형태.
    URL 경로 자체에 값이 포함되므로 필수 값이다.
    리소스 식별(ID)에 주로 사용한다.
    @RequestBody 요청 본문
    (Body)    		 JSON 데이터를 객체로 변환할 때 사용한다.
    REST API 통신에서 가장 많이 쓰인다.

    데이터베이스 (Database) 기초

    DB의 종류

    • RDBMS (관계형 DB): 데이터를 테이블 형태로 관리하며, SQL을 사용해 관계를 맺는다. 데이터의 정합성이 중요하다. (MySQL, Oracle)
    • NoSQL (비관계형 DB): 스키마가 고정되지 않고 Key-Value나 Document 형태로 자유롭게 저장한다. 확장성이 높다. (MongoDB, Redis)

    트랜잭션 (Transaction)

    데이터베이스의 논리적인 작업 단위이다. "모두 성공하거나, 하나라도 실패하면 모두 취소(Rollback)되어야 한다"는 ACID 원칙을 따른다.

    • A (Atomicity, 원자성): 작업은 모두 반영되거나 아예 반영되지 않아야 한다.
    • C (Consistency, 일관성): 작업 전후에 데이터베이스의 상태가 일관되어야 한다.
    • I (Isolation, 격리성): 하나의 트랜잭션은 다른 트랜잭션의 방해를 받지 않아야 한다.
    • D (Durability, 지속성): 성공한 트랜잭션의 결과는 영구적으로 저장되어야 한다.

    오늘의 회고 (Part 2)

    오늘 느낀 점

    • Spring MVC가 내부적으로 수많은 패턴(프론트 컨트롤러, 어댑터 등)의 집합체라는 것을 구조도를 통해 이해할 수 있었다.
    • @RequestParam과 @PathVariable은 비슷해 보이지만, '검색 조건(선택)'이냐 '자원의 식별자(필수)'냐에 따라 용도가 명확히 갈린다는 점을 배웠다.
    • DB 트랜잭션의 ACID 속성은 데이터 무결성을 지키기 위한 최후의 보루라는 생각이 들었다.
