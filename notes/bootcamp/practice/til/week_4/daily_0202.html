    ORM과 JPA의 세계

    패러다임의 불일치

    객체 vs 테이블
    Java 객체는 상속, 리스트 등을 자유롭게 쓰지만, DB 테이블은 오직 단일 값(String, Integer)만 저장할 수 있다. 이 차이를 패러다임의 불일치라고 하며, 이를 해결해 주는 기술이 ORM이다.

    기술 스택 용어 정리

    • JDBC: Java 프로그램이 DB와 통신하기 위한 가장 기초적인 도구(API)이다.
    • ORM (Object-Relational Mapping): 객체와 DB 테이블을 자동으로 연결해 주는 기술이다. SQL을 직접 짜지 않고도 객체를 조작해 DB를 다룰 수 있다.
    • JPA (Java Persistence API): Java 표준 ORM 인터페이스(설계도)이다.
    • Hibernate: JPA 인터페이스를 실제로 구현한 가장 유명한 구현체이다.

    Entity 매핑 필수 어노테이션

    Entity 클래스를 작성할 때는 아래 어노테이션들을 기본 세트처럼 사용한다.

    • @Entity: JPA가 관리할 객체임을 선언한다.
    • @Table: 매핑할 DB 테이블 이름을 지정한다. (예약어 충돌 방지를 위해 복수형 권장)
    • @Id: PK(기본 키)를 지정한다.
    • @GeneratedValue(strategy = GenerationType.IDENTITY): PK 자동 생성 전략(MySQL의 Auto Increment)을 사용한다.
    • @NoArgsConstructor(access = AccessLevel.PROTECTED): JPA가 객체를 생성할 때 필요한 기본 생성자를 만들되, 외부 접근은 막는다.

    3 Layer Architecture

    소프트웨어를 Controller, Service, Repository 세 계층으로 나누어 관리하는 아키텍처 패턴이다. 관심사를 분리하여 유지보수성과 재사용성을 높이는 것이 핵심이다.

    1. Controller Layer (Presentation)

    • 역할: 사용자의 요청(HTTP)을 받고 응답을 반환하는 창구.
    • 할 일: 요청 파라미터 검증, Service 호출, DTO 변환.
    • 하지 말 것: 비즈니스 로직 수행, DB 직접 접근.

    2. Service Layer (Business)

    • 역할: 핵심 비즈니스 로직을 수행하는 허리.
    • 할 일: 트랜잭션 관리(@Transactional), 데이터 조합 및 가공, DTO ↔ Entity 변환.
    • 하지 말 것: HTTP(Request/Response) 객체 직접 사용, SQL 쿼리 작성.

    3. Repository Layer (Data Access)

    • 역할: DB와 직접 대화하는 계층.
    • 할 일: 데이터 CRUD, 쿼리 최적화.
    • 하지 말 것: 비즈니스 로직 판단, 화면(UI) 관련 작업.

    영속성 컨텍스트 (Persistence Context)

    개념과 역할

    엔티티를 영구 저장하는 환경
    애플리케이션과 DB 사이에서 객체를 보관하는 '가상의 임시 저장소' 역할을 한다. 1차 캐시, 변경 감지(Dirty Checking), 쓰기 지연 등의 기능을 제공하여 성능을 최적화한다.

    엔티티의 생명주기

    • 비영속 (New): 객체만 생성하고 JPA와 전혀 관계없는 상태. (DB ID 없음)
    • 영속 (Managed): em.persist() 등으로 영속성 컨텍스트에 저장되어 관리되는 상태.
    • 준영속 (Detached): 영속 상태였다가 분리되어 관리를 받지 않는 상태.
    • 삭제 (Removed): 삭제하기로 표시된 상태.

    더티 체킹 (Dirty Checking)

    • 원리: 트랜잭션이 끝나는 시점에 엔티티의 최초 조회 상태(스냅샷)와 현재 상태를 비교한다.
    • 효과: 필드 값이 변경되었다면 개발자가 update 쿼리를 날리지 않아도, JPA가 알아서 UPDATE SQL을 생성하여 DB에 반영한다.

    JPA 편의 기능

    JPA Auditing

    데이터의 생성 시간과 수정 시간을 자동으로 관리해주는 기능이다.

    1. Application 클래스에 @EnableAuditing 추가
    2. BaseEntity 추상 클래스 생성 (@EntityListeners 추가)
    3. 각 엔티티에서 BaseEntity 상속

    @Transactional 옵션

    • readOnly = true: 읽기 전용으로 설정하여 변경 감지를 끄고 성능을 높인다.
    • rollbackFor: 특정 예외 발생 시 강제로 롤백하도록 지정한다. (기본적으로는 RuntimeException만 롤백됨)

    오늘의 회고

    오늘 느낀 점

    • 3 Layer Architecture에서 '계층 건너뛰기'가 왜 금지되는지, 각자의 역할에만 충실해야 유지보수가 쉽다는 점을 확실히 이해했다.
    • 더티 체킹 덕분에 save()를 호출하지 않아도 수정이 된다는 게 처음엔 마법 같았지만, 영속성 컨텍스트의 원리를 아니 납득이 되었다.
    • Entity에는 비즈니스 로직을 위한 Setter 사용을 지양하고, 생성자나 별도 메서드로 관리해야 안전하다는 점을 명심해야겠다.
