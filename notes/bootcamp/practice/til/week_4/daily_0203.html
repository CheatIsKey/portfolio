    37일차 개요 (2/2)

    오늘 목표

    • 일정 관리 프로젝트의 전체적인 아키텍처(Class Diagram)와 데이터 모델(ERD)을 정리한다.
    • JPA Auditing과 Dirty Checking 등 적용된 핵심 기술의 동작 원리를 재확인한다.
    • 계층별 역할(Controller, Service, Repository)과 데이터 검증 전략을 명확히 한다.

    프로젝트 소개 및 API

    Spring Boot와 JPA를 기반으로 한 일정 관리 REST API 서비스이다. 일정과 댓글에 대한 CRUD 기능을 제공하며, 객체 지향적 설계를 위해 DTO와 Entity를 철저히 분리했다.

    API 명세서

    상세한 요청/응답 예시(HTTP Request/Response)는 아래 링크에서 확인할 수 있다.

    상세 API 명세서 (Redoc)

    Method URI 기능
    POST /api/schedules 일정 생성
    GET /api/schedules 전체 일정 조회 (수정일 내림차순)
    GET /api/schedules/{id} 특정 일정 상세 조회 (댓글 포함)
    PUT /api/schedules/{id} 일정 수정 (비밀번호 검증)
    DELETE /api/schedules/{id} 일정 삭제 (비밀번호 검증)
    POST /api/comments/{scheduleId} 댓글 생성 (최대 10개 제한)

    핵심 기술 구현

    1. JPA Auditing (자동 시간 관리)

    • 구현: @MappedSuperclass를 적용한 BaseEntity를 만들어 createdAt, modifiedAt 필드를 추상화했다.
    • 효과: 모든 엔티티가 이를 상속받게 하여, 생성 및 수정 시간을 자동으로 관리하고 코드 중복을 제거했다.

    2. Dirty Checking (변경 감지)

    • 원리: updateSchedule 메서드에는 repository.save() 코드가 없다.
    • 동작: 트랜잭션 안에서 엔티티의 값을 변경(changeScheduleTitle 등)하면, 트랜잭션이 끝나는 시점에 JPA가 변경 사항을 감지하여 자동으로 UPDATE 쿼리를 날린다.

    3. 이중 데이터 검증 (Defense in Depth)

    • 1차 (DTO): Java Record의 생성자에서 null 체크나 길이 제한 등 기본적인 형식 검증을 수행하여, 잘못된 데이터가 서비스 계층으로 넘어오는 것을 막는다.
    • 2차 (Entity): Validator 유틸리티 클래스를 사용하여 엔티티 생성 시점에 데이터 무결성을 최종적으로 보장한다.

    계층별 역할과 예외 처리

    클래스 역할 분담

    계층 설명
    Controller 클라이언트 요청을 받아 Service로 전달하고, 결과를 DTO로 반환한다.
    Service 비즈니스 로직(비밀번호 검증, 댓글 개수 제한 등)을 수행하고 트랜잭션을 관리한다.
    Repository JPA를 통해 DB에 직접 접근하여 CRUD를 수행한다.
    Util (Validator) 검증 로직(checkLength 등)을 static 메서드로 분리하여 재사용성을 높였다.

    주요 예외 처리 전략

    • 404 Not Found: NoSuchScheduleException (일정 ID 조회 실패)
    • 401 Unauthorized: PasswordMismatchException (수정/삭제 시 비밀번호 불일치)
    • 400 Bad Request: CommentLimitExceededException (댓글 10개 초과), HttpMessageNotReadableException (Body 누락)

    오늘의 회고 (Part 2)

    프로젝트 정리 소감

    • 단순히 기능을 만드는 것을 넘어, "왜 이 기술(Auditing, Dirty Checking)을 썼는가?"를 설명할 수 있게 구조를 정리하니 프로젝트에 대한 이해도가 훨씬 깊어졌다.
    • Validator를 유틸리티로 분리하고, Record의 생성자 검증을 활용하는 등 객체지향적인 코드 설계를 위해 고민했던 흔적들을 문서로 남길 수 있어서 뿌듯하다.
    • API 명세서와 다이어그램 같은 시각 자료가 문서의 퀄리티를 결정짓는 중요한 요소임을 다시 한번 느꼈다.
