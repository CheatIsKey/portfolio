    API 문서화 도구 (Swagger & Redoc)

    API 명세서를 자동화하기 위해 Swagger를 도입했으나, Spring Boot 버전 차이와 배포 환경에서 여러 이슈를 마주쳤다.

    트러블 슈팅: Spring Boot 3.x 의존성 문제

    • 상황: Spring Boot 3.x 프로젝트에 2.x용 springdoc-openapi-ui를 추가하자 Swagger 접속 시 404 화이트라벨 에러가 발생했다.
    • 원인: Spring Boot 3.x부터는 javax 패키지가 jakarta로 변경되는 등 내부 구조가 달라져 구버전 라이브러리와 호환되지 않았다.
    • 해결: springdoc-openapi-starter-webmvc-ui (v2.x 이상) 의존성으로 교체하여 해결했다.

    트러블 슈팅: Redoc 배포 시 흰 화면 현상

    • 상황: GitHub Pages로 API 문서를 배포했으나, 제목만 뜨고 내용은 보이지 않는 빈 화면이 지속되었다.
    • 원인: HTML 파일이 외부의 openapi.json 파일을 비동기로 로드하는 과정에서 경로(Path)를 찾지 못했다.
    • 해결: JSON 데이터를 별도 파일 로딩 없이, HTML 내부 <script> 변수에 직접 할당(Embedding)하여 즉시 렌더링되도록 수정했다.

    Java Record와 DTO 검증

    데이터 전송 객체(DTO)로 Record를 도입하면서, 불변 객체의 특성과 생성자를 활용한 검증 로직을 학습했다.

    Compact Constructor (컴팩트 생성자)

    괄호 없는 생성자
    public RecordName { ... } 형태로 작성하며, 매개변수 선언과 this.val = val 할당 코드를 생략할 수 있다. 검증 로직만 깔끔하게 작성할 수 있는 Record만의 강력한 기능이다.

    트러블 슈팅: 검증 순서에 따른 NPE 

    // [문제 코드]
if (name.length() > 20 || name == null) { ... }
// name이 null일 경우 length()를 호출하다가 NullPointerException 발생

// [해결 코드]
if (name == null || name.length() > 20) { ... }
// null 체크를 먼저 수행하여(Short-circuit) 안전하게 검증

    Spring의 요청 처리 메커니즘

    트러블 슈팅: DELETE 요청과 @RequestBody

    • 상황: Postman에서 DELETE 요청 시 Body를 비우고 보냈더니 HttpMessageNotReadableException 에러가 발생했다.
    • 오해: "DELETE는 데이터를 지우는 거니까 Body가 없어도 DTO 검증만 안 하고 넘어가지 않을까?"라고 생각했다.
    • 실제 원인: 컨트롤러에 @RequestBody가 선언되어 있으면, 스프링은 내용을 읽기 위해 입력 스트림을 연다. 이때 Body가 아예 없으면 DTO를 생성하는 단계조차 가지 못하고 예외를 던진다.
    • 해결: 테스트 시 빈 JSON 객체 {}라도 보내야 DTO가 생성되고 내부 검증 로직이 작동함을 확인했다.

    엔티티 설계와 문서화

    이중 검증 구조 (Defense in Depth)

    • 1차 방어 (DTO): Record 생성자에서 형식(Format)과 필수값 여부를 검증한다.
    • 2차 방어 (Entity): 엔티티 생성자에서 데이터 무결성(Integrity)을 최종 검증한다.
    • Validator 분리: checkLength, isNotNull 같은 반복 로직을 별도 유틸 클래스로 분리해 가독성을 높였다.

    문서화 (README.md) 개선

    • 기존 텍스트 나열 방식에서 벗어나 ERD와 Class Diagram 같은 시각 자료를 적극 활용했다.
    • API 명세는 표로 요약하고, 상세 내용은 배포된 Redoc 링크로 연결하여 가독성을 확보했다.
    • 단순 기능 나열이 아니라, JPA Auditing이나 책임 분리 같은 핵심 기술 적용 이유를 서술하여 기술적 깊이를 더했다.

    오늘의 회고 (Part 1)

    트러블 슈팅 총평

    • 에러 로그를 만났을 때 단순히 코드를 고치는 것을 넘어, "프레임워크가 내부적으로 어떻게 동작하길래 이 에러가 났을까?"(예: DELETE Body 처리)를 고민하는 과정이 의미 있었다.
    • 문서화는 개발의 끝이 아니라, 내 코드의 설득력을 높이는 과정임을 깨달았다.
