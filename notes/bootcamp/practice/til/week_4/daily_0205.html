    39일차 개요

    오늘 목표

    • @RestControllerAdvice를 활용해 예외 처리를 전역적으로 관리하는 방법을 익힌다.
    • API 요청 데이터를 받는 3가지 방식(Path, Param, Body)의 차이점과 사용 목적을 명확히 구분한다.
    • NullPointerException(NPE)을 방지하는 안전한 객체 비교 메서드(ObjectUtils)를 학습한다.

    전역 예외 처리 (Global Exception Handling)

    각 컨트롤러마다 try-catch를 넣는 것은 비효율적이다. 스프링은 에러를 한 곳에서 잡아 처리하는 중앙 집중형 처리 방식을 제공한다.

    핵심 어노테이션

    • 클래스 단위: @RestControllerAdvice
      모든 컨트롤러에서 발생하는 예외를 감지한다. (@ControllerAdvice + @ResponseBody)
    • 메서드 단위: @ExceptionHandler(예외클래스.class)
      특정 예외가 발생했을 때 실행될 메서드를 지정한다.

    예외 처리 순서 (우선순위)

    "더 구체적인 자식 예외가 먼저 잡힌다."
    예를 들어 NullPointerException이 발생하면:

    1. @ExceptionHandler(NullPointerException.class)가 있는지 먼저 확인하고 실행한다.
    2. 없다면 부모인 @ExceptionHandler(RuntimeException.class)를 찾는다.
    3. 최후의 보루로 @ExceptionHandler(Exception.class)가 처리한다.

    구현 코드 예시 

    // 1. 에러 응답용 DTO (Record 활용)
public record ErrorResponse(
    int statusCode,       // HTTP 상태 코드 (예: 400, 404, 500)
    String errorMessage,  // 클라이언트에게 보여줄 에러 메시지
    String detail         // 디버깅용 상세 정보
) {}

// 2. 전역 예외 핸들러 클래스
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<ErrorResponse> handleBadRequest(IllegalArgumentException e) {
        return ResponseEntity
                .status(HttpStatus.BAD_REQUEST)
                .body(new ErrorResponse(
                        HttpStatus.BAD_REQUEST.value(),
                        e.getMessage(), 
                        "입력값을 다시 확인해주세요."
                ));
    }
}

    데이터 전달 방식 3대장 완벽 정리

    클라이언트가 서버로 데이터를 보낼 때 사용하는 대표적인 3가지 방식의 특징과 차이점이다.

    1. @PathVariable (경로 변수)

    • 형태: /users/10 (URL 경로 자체에 값이 포함됨)
    • 용도: 자원(Resource)의 식별. 특정 회원을 조회하거나 삭제할 때 ID 값으로 쓴다.
    • 특징: 캐싱에 유리하며, 값이 반드시 존재해야 한다.

    2. @RequestParam (쿼리 파라미터)

    • 형태: /users?name=steve&age=20 (URL 뒤에 ?로 붙음)
    • 용도: 검색, 필터링, 정렬.
    • 특징: required=false로 설정하여 값을 선택적으로 받을 수 있다.

    3. @RequestBody (요청 본문)

    • 형태: URL에는 안 보임. HTTP Body 안에 JSON 데이터로 숨겨져 있음.
    • 용도: 복잡하고 많은 데이터를 보낼 때 (주로 POST/PUT).
    • 특징: URL 길이 제한이 없어 대용량 전송이 가능하지만, URL 공유가 불가능하다.

    DB 전략과 튜닝

    PK 생성 전략: Auto Increment vs UUID

    구분 Auto Increment (Sequence) UUID (Universally Unique Identifier)
    형태 1, 2, 3... (숫자 증가) 550e8400-e29b-41d4... (랜덤 문자열)
    장점 데이터가 작아 빠르고, 순서 파악이 쉽다. 보안성 우수 (예측 불가), DB 병합 시 충돌 없음.
    단점 ID 예측 가능 (데이터 크롤링 위험). 데이터 크기가 커서 성능 저하 가능성.

    Null 안전한 비교

    개발을 하다 보면 DTO로 들어온 데이터나 DB에서 조회한 값이 null인 경우가 종종 발생한다. 특히 비밀번호 검증 로직에서 무심코 equals()를 썼다가 NPE(NullPointerException)가 터질 수 있다.

    위험한 코드 (NPE 발생 가능) 

    // requestPw가 null이면 즉시 NullPointerException 발생!
// (클라이언트가 실수로 password 필드를 안 보냈거나 null로 보낸 경우)
if (requestDto.getPassword().equals(entity.getPassword())) {
    // ...
}

    해결: ObjectUtils.nullSafeEquals()

    Spring 프레임워크가 제공하는 ObjectUtils를 사용하면 두 객체 중 하나(혹은 둘 다)가 null이어도 에러 없이 안전하게 비교할 수 있다. 

    import org.springframework.util.ObjectUtils;

// 1. requestPw가 null이어도 false를 반환할 뿐, 에러가 나지 않는다.
// 2. entity.getPassword()가 null이어도 안전하다.
if (ObjectUtils.nullSafeEquals(requestDto.getPassword(), entity.getPassword())) {
    // 비밀번호 일치 로직 실행
} else {
    // 비밀번호 불일치 예외 던지기
    throw new PasswordMismatchException();
}

    비교 결과 정리

    • nullSafeEquals(null, null) → true 반환
    • nullSafeEquals("1234", null) → false 반환 (에러 없음)
    • nullSafeEquals(null, "1234") → false 반환 (에러 없음)
    • nullSafeEquals("1234", "1234") → true 반환

    오늘의 회고

    오늘 느낀 점

    • 예외 처리를 @RestControllerAdvice로 한 곳에서 관리하니 비즈니스 로직에 try-catch가 사라져 코드가 훨씬 읽기 편해졌다.
    • @RequestBody는 URL 공유가 안 된다는 점을 깨달았다. 검색 페이지를 만들 땐 반드시 @RequestParam을 써야겠다.
    • ObjectUtils.nullSafeEquals()는 사소해 보이지만, 운영 환경에서 예기치 못한 NPE를 막아주는 강력한 습관이 될 것 같다.
