    전역 예외 처리 (Global Exception Handling)

    각 컨트롤러마다 try-catch를 넣는 것은 비효율적이다. 스프링은 에러를 한 곳에서 잡아 처리하는 중앙 집중형 처리 방식을 제공한다.

    핵심 어노테이션

    • 클래스 단위: @RestControllerAdvice
      모든 컨트롤러에서 발생하는 예외를 감지한다. (@ControllerAdvice + @ResponseBody)
    • 메서드 단위: @ExceptionHandler(예외클래스.class)
      특정 예외가 발생했을 때 실행될 메서드를 지정한다.

    예외 처리 순서 (우선순위)

    "더 구체적인 자식 예외가 먼저 잡힌다."
    예를 들어 NullPointerException이 발생하면:

    1. @ExceptionHandler(NullPointerException.class)가 있는지 먼저 확인하고 실행한다.
    2. 없다면 부모인 @ExceptionHandler(RuntimeException.class)를 찾는다.
    3. 최후의 보루로 @ExceptionHandler(Exception.class)가 처리한다.

    구현 코드 예시 

    // 1. 에러 응답용 DTO (Record 활용)
// record는 불변이며 getter, 생성자를 자동 생성해준다.
public record ErrorResponse(
    int statusCode,       // HTTP 상태 코드 (예: 400, 404, 500)
    String errorMessage,  // 클라이언트에게 보여줄 에러 메시지
    String detail         // (선택) 디버깅용 상세 정보
) {}

// 2. 전역 예외 핸들러 클래스
@RestControllerAdvice
public class GlobalExceptionHandler {

    /**
     * 잘못된 요청 (400 Bad Request) 처리
     * 검증 실패, 잘못된 인자 전달 등
     */
    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<ErrorResponse> handleBadRequest(IllegalArgumentException e) {
        return ResponseEntity
                .status(HttpStatus.BAD_REQUEST) // 400 상태 코드 설정
                .body(new ErrorResponse(
                        HttpStatus.BAD_REQUEST.value(),
                        e.getMessage(), // "잘못된 입력입니다" 등
                        "입력값을 다시 확인해주세요."
                ));
    }
}

    데이터 전달 방식 3대장 완벽 정리

    클라이언트가 서버로 데이터를 보낼 때 사용하는 대표적인 3가지 방식의 특징과 차이점이다.

    1. @PathVariable (경로 변수)

    • 형태: /users/10 (URL 경로 자체에 값이 포함됨)
    • 용도: 자원(Resource)의 식별. 특정 회원을 조회하거나 삭제할 때 ID 값으로 쓴다.
    • 특징:
      • 값이 반드시 존재해야 한다. (없으면 404 Not Found)
      • URL이 고유하므로 캐싱(Caching)에 매우 유리하다.
      • 링크 공유 시 상대방도 똑같은 결과를 볼 수 있다.

    2. @RequestParam (쿼리 파라미터)

    • 형태: /users?name=steve&age=20 (URL 뒤에 ?로 붙음)
    • 용도: 검색, 필터링, 정렬. 또는 게시글의 단일 삭제 같은 보조 옵션.
    • 특징:
      • 값이 없어도 가능하게 설정할 수 있다. (required=false)
      • @PathVariable과 마찬가지로 URL에 데이터가 보이므로 링크 공유가 가능하다.

    3. @RequestBody (요청 본문)

    • 형태: URL에는 안 보임. HTTP Body 안에 JSON 데이터로 숨겨져 있음.
    • 용도: 회원가입, 글쓰기 등 복잡하고 많은 데이터를 보낼 때 (주로 POST/PUT).
    • 특징:
      • 보안 및 깔끔함: URL에 정보가 노출되지 않아 PII(개인정보) 전송에 유리하고, URL 길이가 무제한이 아니므로 긴 데이터 전송에 필수적이다.
      • 공유 불가: URL에 데이터가 없으므로, 이 페이지의 링크를 복사해서 친구에게 주면 친구는 빈 화면이나 에러를 보게 된다. (내가 입력한 데이터가 친구에겐 없기 때문)

    DB 전략과 튜닝

    ORM이 있는데 왜 SQL을 알아야 할까?

    • 문제점: JPA(ORM)는 개발 편의성을 주지만, 복잡한 통계 쿼리나 조인이 많아지면 비효율적인 쿼리(N+1 문제 등)를 생성하여 성능을 떨어뜨릴 수 있다.
    • 해결책: 자동 생성된 쿼리가 느리다면, 직접 SQL(JPQL, QueryDSL)을 작성하여 튜닝해야 한다. 즉, ORM은 도구일 뿐 SQL을 몰라도 된다는 뜻이 아니다.

    PK 생성 전략: Auto Increment vs UUID

    구분 Auto Increment (Sequence) UUID (Universally Unique Identifier)
    형태 1, 2, 3... (숫자 증가) 550e8400-e29b-41d4... (랜덤 문자열)
    장점 - 데이터 크기가 작아 조회가 빠르다.
    - 눈으로 보고 순서를 알기 쉽다.     		- 전 세계적으로 유일한 값이 보장된다.
    - 보안성 우수 (다음 ID 예측 불가).
    - DB 병합 시 충돌이 없다.
    단점 - 보안 취약 (ID 예측하여 데이터 크롤링 가능).
    - 여러 DB 서버를 합칠 때 ID가 겹칠 수 있다.     		- 데이터 크기가 커서(16byte) 성능이 약간 떨어진다.
    - 정렬이나 인덱싱 효율이 숫자에 비해 낮다.

    오늘의 회고

    오늘 느낀 점

    • 예외 처리를 컨트롤러마다 하는 게 아니라 @RestControllerAdvice로 한 곳에서 관리하니 코드가 정말 깔끔해졌다.
    • @RequestBody를 쓴 요청은 URL 공유가 안 된다는 점을 확실히 이해했다. (검색 결과 페이지는 그래서 @RequestParam을 쓰는구나!)
    • 보안이 중요한 서비스라면 PK를 숫자로 1씩 올리는 게 위험할 수 있다는 점(예측 가능성)을 처음 알게 되었다.
