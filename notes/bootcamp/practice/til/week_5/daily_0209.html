    41일차 개요

    오늘 목표

    • 인증(Authentication)과 인가(Authorization)의 차이를 명확히 구분한다.
    • 세션(Session)과 JWT(Json Web Token)의 동작 방식과 장단점을 비교한다.
    • ArgumentResolver를 통해 컨트롤러 파라미터를 깔끔하게 처리하는 방법을 익힌다.

    인증과 인가, 그리고 세션 vs JWT

    인증(Authentication) vs 인가(Authorization)

    • 인증 (로그인): 사용자가 누구인지 확인하는 절차이다. "당신은 우리 회원이 맞습니까?"
    • 인가 (권한): 인증된 사용자가 특정 리소스에 접근할 수 있는지 확인하는 절차이다. "당신은 관리자 페이지에 들어갈 권한이 있습니까?"
    • 순서: 인증은 인가보다 항상 먼저 수행되어야 한다.

    세션 (Session) 방식

    서버가 사용자의 로그인 상태를 서버 메모리나 DB에 직접 기억하는 방식이다.

    1. 사용자가 로그인하면 서버는 고유한 Session ID를 생성하여 저장한다.
    2. 서버는 이 ID를 클라이언트에게 보내고, 브라우저는 쿠키(Cookie)에 저장한다.
    3. 이후 요청마다 쿠키에 담긴 Session ID를 보내면, 서버가 이를 확인하여 사용자를 식별한다.
    • 장점: 서버가 상태를 관리하므로 보안성이 높고, 강제 로그아웃 등의 제어가 쉽다.
    • 단점: 사용자가 많아지면 서버 메모리 부하가 커지고, 서버 확장이 어렵다.

    JWT (Json Web Token) 방식

    서버가 상태를 저장하지 않는(Stateless) 인증 방식이다. 토큰 자체에 인증 정보가 담겨있다.

    1. 로그인 성공 시, 서버는 사용자 정보를 암호화한 토큰(JWT)을 발급하여 클라이언트에게 준다. (서버는 저장 X)
    2. 클라이언트는 토큰을 로컬 스토리지나 쿠키에 저장한다.
    3. 요청 시 헤더에 토큰을 실어 보내면, 서버는 서명(Signature)을 검증하여 유효성을 확인한다.
    • 장점: 서버 부하가 적고 확장성이 뛰어나다. 모바일 앱 등 다양한 환경에서 쓰기 좋다.
    • 단점: 토큰이 탈취되면 만료될 때까지 막을 방법이 없다. (Access/Refresh Token 전략으로 보완)

    JWT 구성 요소

    • Header: 토큰 유형(JWT)과 알고리즘 정보.
    • Payload: 실제 정보(Claim). 누구나 디코딩 가능하므로 비밀번호 절대 금지.
    • Signature: 위변조 방지를 위한 서명. Header+Payload+SecretKey를 암호화한 값.

    HTTP Cookie와 보안

    쿠키는 Session ID나 JWT를 담는 운송 수단(상자/봉투) 역할을 한다.

    Set-Cookie 보안 옵션

    • Secure: HTTPS 통신일 때만 쿠키를 전송한다. (탈취 방지)
    • HttpOnly: 자바스크립트의 접근을 차단한다. (XSS 공격 방지)
    • SameSite: 다른 도메인에서의 요청 시 쿠키 전송을 제한한다. (CSRF 공격 방지)

    Spring의 세션 관리

    HttpSession vs HttpServletRequest

    구분 HttpSession HttpServletRequest
    사용법 파라미터로 선언하면 Spring이 알아서 주입해준다. request.getSession(false)로 직접 꺼내야 한다.
    생성 제어 없으면 무조건 생성한다. false 옵션으로 없을 때 생성을 막을 수 있다.
    용도 세션 기능만 필요할 때. 세션 외에 IP, 헤더 정보 등이 필요할 때.

    로그인/로그아웃 구현 예제 

    // [로그인] 세션 생성 및 저장
@PostMapping("/login")
public String login(@RequestBody LoginRequest dto, HttpSession session) {
    // ... 회원 검증 로직 ...
    
    // 세션에 사용자 정보 저장 (JSESSIONID 쿠키 자동 발급)
    session.setAttribute("loginUser", sessionUser);
    return "로그인 성공";
}

// [로그아웃] 세션 무효화
@PostMapping("/logout")
public String logout(HttpSession session) {
    // 세션 전체 삭제 (저장소 비우기)
    session.invalidate();
    return "로그아웃 성공";
}

// [정보 수정] @SessionAttribute로 세션 값 조회
@PatchMapping("/me")
public String update(@SessionAttribute(name = "loginUser", required = false) SessionUser user) {
    if (user == null) {
        throw new UnAuthorizedException(); // 로그인 안 된 경우
    }
    // ... 수정 로직 ...
}

    ArgumentResolver: 파라미터 자동 처리

    @LoginUser 같은 커스텀 어노테이션을 만들어, 컨트롤러 파라미터에 사용자 정보를 자동으로 주입해주는 마법 같은 기능이다.

    구현 3단계

    1. HandlerMethodArgumentResolver 구현:
      • supportsParameter(): 어떤 파라미터를 처리할지 조건 검사 (예: @LoginUser가 붙었는지).
      • resolveArgument(): 실제 주입할 값을 생성 (세션에서 User 꺼내기 등).
    2. WebMvcConfigurer 등록: addArgumentResolvers에 추가해야 동작한다.

    Spring 예외 처리와 로깅

    예외 처리 전략 (우선순위 순)

    1. @ExceptionHandler (컨트롤러 단위)
    2. @RestControllerAdvice (전역 단위, 표준)
    3. Spring 기본 예외 처리

    커스텀 에러와 다형성 활용 예시 

    // 부모 예외 (상태 코드 포함)
@Getter
public class ServiceException extends RuntimeException {
    private final HttpStatus status;
    public ServiceException(HttpStatus status, String message) {
        super(message);
        this.status = status;
    }
}

// 자식 예외 (구체적인 상황)
public class MovieNotFoundException extends ServiceException {
    public MovieNotFoundException(String message) {
        super(HttpStatus.NOT_FOUND, message); // 404 지정
    }
}

// 전역 핸들러에서 부모 타입으로 한 번에 처리
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(ServiceException.class)
    public ResponseEntity<String> handleServiceException(ServiceException ex) {
        return ResponseEntity.status(ex.getStatus()).body(ex.getMessage());
    }
}

    로깅 (Logging)

    • System.out.println 금지 이유: 성능 저하, 로그 레벨 제어 불가, 정보 부족.
    • 로그 레벨: TRACE → DEBUG → INFO(기본) → WARN → ERROR
    • 사용법: @Slf4j 어노테이션 사용 → log.info("메시지");

    그 외 핵심 개념들

    AppConfig의 역할 (DI) 

    @Configuration
public class AppConfig {
    @Bean
    public MemberService memberService() {
        // 생성자 주입: Service가 Repository를 사용하도록 연결
        return new MemberServiceImpl(memberRepository());
    }

    @Bean
    public MemberRepository memberRepository() {
        // 구현체 변경 시 여기만 수정하면 됨 (Memory -> DB)
        return new MemoryMemberRepository();
    }
}

    Bean Validation (데이터 검증)

    • @NotBlank, @Email, @Size, @Pattern(regexp=...) 등을 사용하여 DTO 검증.
    • 검증 실패 시 MethodArgumentNotValidException 발생 → 전역 핸들러에서 처리.

    오늘의 회고

    오늘 느낀 점

    • 인증(로그인)과 인가(권한)의 차이, 그리고 세션과 JWT의 장단점을 명확히 구분할 수 있게 되었다.
    • ArgumentResolver는 처음엔 복잡해 보였지만, 컨트롤러 코드를 획기적으로 줄여주는 강력한 도구임을 깨달았다.
    • System.out 대신 log를 써야 하는 이유와, 커스텀 예외를 상속 구조로 만들어 관리하는 패턴은 실무에서 꼭 써먹어야겠다.
