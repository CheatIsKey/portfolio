    41일차 개요

    오늘 목표

    • 좋은 객체 지향 설계를 위한 응집도와 결합도, 그리고 SOLID 원칙을 정리한다.
    • Spring의 핵심인 IoC/DI, Bean, 생명주기의 동작 원리를 파악한다.
    • 웹 애플리케이션의 필수 요소인 인증/인가(세션 vs JWT)와 예외 처리 전략을 마스터한다.

    객체 지향 설계 원칙

    응집도와 결합도

    • 응집도(Cohesion): 하나의 클래스가 하나의 목적을 위해 얼마나 똘똘 뭉쳐있는지를 나타낸다. 응집도가 높을수록 좋다. (예: `MemberService`는 회원 관련 로직만 담당)
    • 결합도(Coupling): 한 클래스가 다른 클래스에 얼마나 의존적인지를 나타낸다. 결합도가 낮을수록 변경의 여파가 적다.

    SOLID 원칙

    • S (SRP): 단일 책임 원칙. 클래스는 단 하나의 책임만 가져야 한다.
    • O (OCP): 개방-폐쇄 원칙. 확장은 열려있고, 수정은 닫혀있어야 한다.
    • L (LSP): 리스코프 치환 원칙. 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다.
    • I (ISP): 인터페이스 분리 원칙. 범용 인터페이스 하나보다 구체적인 여러 개가 낫다.
    • D (DIP): 의존관계 역전 원칙. 구체적인 것이 추상화(인터페이스)에 의존해야 한다.

    Spring Core: IoC와 Bean

    제어의 역전 (IoC)과 의존성 주입 (DI)

    전통적인 방식에서는 개발자가 객체를 직접 new로 생성했다. 하지만 Spring에서는 컨테이너가 객체의 생명주기를 관리하고 필요한 곳에 넣어준다(주입). 이것이 제어권이 넘어갔다는 의미의 IoC다.

    Bean 등록 방식

    1. 자동 등록 (@Component)

    @Service // @Component를 포함한 어노테이션
public class MemberService { ... }
    • @ComponentScan이 설정된 패키지 하위를 뒤져서 자동으로 등록한다.
    • @Controller, @Service, @Repository가 대표적이다.

    2. 수동 등록 (@Bean)

    @Configuration
public class AppConfig {
    @Bean
    public MemberService memberService() {
        return new MemberService(memberRepository());
    }
}
    • 외부 라이브러리 설정 등 개발자가 코드를 수정할 수 없는 경우에 주로 쓴다.

    DI 주입 방식 비교

    • 생성자 주입 (권장): 불변성(final) 보장, 테스트 용이, 순환 참조 방지.
    • Setter 주입: 변경 가능성이 있어 위험함.
    • 필드 주입 (@Autowired): 외부에서 수정이 불가능해 테스트가 어려움. 안티 패턴.

    인증(Authentication)과 인가(Authorization)

    개념 구분

    • 인증: "누구세요?" (로그인). 신원 확인 절차.
    • 인가: "권한이 있나요?" (접근 제어). 인증된 사용자만 접근 가능.

    세션 (Session) vs JWT

    구분 세션 (Session) JWT (Token)
    저장 위치 서버 메모리/DB 클라이언트 (브라우저 등)
    장점 보안성 높음 (서버가 제어 가능) 확장성 좋음 (Stateless, 서버 부하 적음)
    단점 사용자가 많으면 서버 메모리 부족 탈취 시 만료될 때까지 못 막음

    JWT 구성 요소

    • Header: 토큰 타입과 해싱 알고리즘 정보.
    • Payload: 실제 데이터(Claim). 비밀번호 절대 금지 (디코딩 쉬움).
    • Signature: 위변조 방지용 서명. (헤더+페이로드+서버비밀키) 조합.

    Spring MVC 심화 기능

    ArgumentResolver

    컨트롤러 파라미터에 값을 자동으로 채워주는 마법사다. 예를 들어 세션에서 사용자 정보를 꺼내 @LoginUser 파라미터에 넣어줄 수 있다.

    1. HandlerMethodArgumentResolver 인터페이스 구현 (supportsParameter, resolveArgument).
    2. WebMvcConfigurer에 등록.

    HttpSession 사용 예제 

    // [로그인] 세션 생성 및 값 저장
@PostMapping("/login")
public String login(HttpSession session, @RequestBody LoginDto dto) {
    // ...검증 로직...
    session.setAttribute("loginUser", user); // 세션 저장소에 저장
    return "ok";
}

// [로그아웃] 세션 삭제
@PostMapping("/logout")
public String logout(HttpSession session) {
    session.invalidate(); // 모든 데이터 삭제
    return "ok";
}

    예외 처리와 로깅

    예외 처리 전략

    • @ExceptionHandler: 컨트롤러 단위로 예외를 잡는다.
    • @RestControllerAdvice: 전역(Global) 단위로 예외를 잡는다. (가장 권장됨)

    로깅 (Logging)

    System.out.println은 성능 저하와 관리의 어려움 때문에 실무에서 절대 쓰지 않는다. 대신 @Slf4j 같은 로깅 프레임워크를 사용해 로그 레벨(INFO, WARN, ERROR)을 관리해야 한다.

    오늘의 회고

    오늘 느낀 점

    • Spring의 DI 컨테이너가 객체를 대신 관리해준다는 것이 얼마나 큰 편리함인지, 그리고 왜 생성자 주입을 써야 하는지 명확해졌다.
    • 세션과 JWT는 무조건 어느 하나가 좋은 게 아니라, 서비스의 규모와 특성(앱 여부 등)에 따라 선택해야 한다는 점을 배웠다.
    • 컨트롤러에서 중복되는 세션 조회 코드를 ArgumentResolver로 분리하면 코드가 훨씬 깔끔해질 것 같다.
