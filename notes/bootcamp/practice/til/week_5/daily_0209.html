    인증(Authentication) 아키텍처 비교

    사용자가 "나 로그인 했어"라고 증명하는 방식은 크게 두 가지로 나뉜다. 서버가 기억하느냐(Session), 클라이언트가 증명서를 들고 다니느냐(JWT)의 차이다.

    1. 세션 (Session) 방식

    핵심: "서버가 명부를 쥐고 있다."
    서버 메모리나 DB에 로그인한 사용자 정보를 저장하고, 클라이언트에게는 JSESSIONID라는 입장권 번호만 쿠키로 준다.
    • 흐름: 로그인 성공 → 서버에 세션 생성(ID발급) → 쿠키에 ID 담아 응답 → 이후 요청마다 쿠키 검사.
    • 장점:
      • 보안이 강력하다. (해킹 의심 시 서버에서 세션을 지워버리면 즉시 로그아웃됨)
      • 쿠키에는 의미 없는 ID만 들어있어 탈취되어도 정보 유출이 없다.
    • 단점:
      • 사용자가 100만 명이면 서버 메모리가 터진다. (Scale-out 시 세션 동기화 문제 발생)

    2. JWT (Json Web Token) 방식

    핵심: "클라이언트가 증명서를 들고 다닌다."
    서버는 아무것도 저장하지 않는다(Stateless). 대신 사용자의 정보와 위변조 방지 서명이 담긴 토큰(JWT)을 발급해준다.
    • 흐름: 로그인 성공 → 정보를 암호화해 토큰 생성 → 클라이언트에 전달 → 이후 요청마다 헤더(Header)에 토큰 첨부.
    • 장점:
      • 서버 부하가 없다. (서버를 100대로 늘려도 상관없음)
      • 앱(Mobile) 환경에서 사용하기 편하다.
    • 단점:
      • 토큰이 탈취되면 유효기간이 끝날 때까지 막을 방법이 없다. (Access/Refresh Token 전략 필수)
      • 세션 ID보다 데이터 크기가 커서 네트워크 비용이 조금 더 든다.

    Spring MVC 마법: ArgumentResolver

    컨트롤러 메서드에 @LoginUser 같은 어노테이션만 붙이면, 스프링이 알아서 세션이나 토큰에서 유저 정보를 꺼내 꽂아주는 기능이다.

    동작 원리 (파이프라인)

    1. 요청 수신: 클라이언트가 API를 호출한다.
    2. 파라미터 검사 (supportsParameter): 스프링이 "이 메서드 파라미터에 @LoginUser가 붙어있니?"라고 묻는다.
    3. 데이터 생성 (resolveArgument): "응, 붙어있어." -> 세션에서 유저 객체를 꺼내오거나 DB를 조회한다.
    4. 주입: 꺼낸 객체를 컨트롤러 메서드의 인자(Argument)로 넘겨준다.

    구현 코드 

    // 1. 리졸버 로직 구현
public class LoginUserArgumentResolver implements HandlerMethodArgumentResolver {
    @Override
    public boolean supportsParameter(MethodParameter parameter) {
        // 파라미터에 @LoginUser 어노테이션이 있고, 타입이 User 클래스인지 확인
        boolean hasAnnotation = parameter.hasParameterAnnotation(LoginUser.class);
        boolean isUserType = User.class.isAssignableFrom(parameter.getParameterType());
        return hasAnnotation && isUserType;
    }

    @Override
    public Object resolveArgument(...) {
        // 세션에서 사용자 정보 꺼내서 반환
        HttpServletRequest request = (HttpServletRequest) webRequest.getNativeRequest();
        HttpSession session = request.getSession(false);
        return session != null ? session.getAttribute("LOGIN_USER") : null;
    }
}

// 2. 설정에 등록 (필수)
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
        resolvers.add(new LoginUserArgumentResolver());
    }
}

    로깅 (Logging) 완벽 가이드

    System.out.println은 실무 금지어다. 성능을 잡아먹고 관리가 안 되기 때문이다. 반드시 @Slf4j를 써야 한다.

    로그 레벨별 사용 기준

    레벨 의미 언제 쓰는가? 운영 서버 설정
    ERROR 심각한 오류 기능 수행 불가, DB 장애, catch 블록 내부 필수 기록
    WARN 잠재적 위험 deprecated API 사용, 디스크 용량 부족 경고 기록함
    INFO 운영 정보 "로그인 성공", "결제 완료", "서버 시작됨" 기본값 (Default)
    DEBUG 개발 디버깅 SQL 쿼리 로그, 변수에 담긴 값 확인 출력 안 함 (성능 이슈)
    TRACE 상세 추적 모든 실행 경로 추적 (라이브러리 내부 등) 출력 안 함

    실무 적용 코드 

    @Slf4j // Lombok이 log 객체를 자동 생성해줌
@RestController
public class OrderController {

    @PostMapping("/orders")
    public String order(@RequestBody OrderDto dto) {
        // [INFO] 비즈니스 중요 정보
        log.info("주문 요청 발생 - 사용자: {}", dto.getUserId());

        try {
            // [DEBUG] 개발할 때만 보고 싶은 데이터 (운영에선 안 보임)
            log.debug("주문 데이터 상세: {}", dto);
            orderService.process(dto);
        } catch (Exception e) {
            // [ERROR] 에러는 스택트레이스(e)와 함께 남겨야 원인 파악 가능
            log.error("주문 처리 중 알 수 없는 오류", e);
            throw e;
        }
        return "ok";
    }
}

    설정 파일 (application.properties) 

    # 전체 로그 레벨 기본값 (보통 INFO)
logging.level.root=INFO

# 특정 패키지만 상세하게 보고 싶을 때
logging.level.com.myproject.payment=DEBUG

# 스프링 프레임워크 내부 로그는 WARN 이상만 보기
logging.level.org.springframework=WARN

    오늘의 회고

    오늘 느낀 점

    • 세션과 JWT의 흐름을 그림으로 보니, "누가 상태를 저장하느냐"가 가장 큰 차이점임이 명확해졌다.
    • ArgumentResolver는 처음엔 복잡해 보였지만, 한 번 만들어두면 컨트롤러의 중복 코드를 획기적으로 줄여주는 생산성 도구임을 깨달았다.
    • 로그 레벨을 구분하지 않고 막 찍으면, 정작 서버가 터졌을 때 로그 파일이 너무 커서 원인을 못 찾을 수도 있겠다는 생각이 들었다.
