    • 객체 지향 설계 원칙(SOLID, 응집도/결합도)을 재정립한다.

    1. 객체 지향 설계 원칙

    응집도와 결합도

    • 응집도(Cohesion):
      하나의 클래스(또는 모듈)가 하나의 목적을 위해 얼마나 긴밀하게 뭉쳐있는지 나타내는 척도이다.
      즉, 클래스 내의 메소드와 데이터들이 얼마나 서로 관련 있는 작업을 수행하는 지를 의미한다.
      응집도를 높이기 위해서는 하나의 클래스는 하나의 책임만 가져야 한다.
    • 결합도(Coupling):
      하나의 클래스(또는 모듈)가 다른 클래스와 얼마나 많이 엮어있는지, 즉 얼마나 의존적인지를 나타내는 척도이다.
      결합도를 낮추기 위해서는 하나의 클래스가 변경되어도 다른 클래스에 미치는 영향이 거의 없도록 설계해야 한다.

    SOLID 원칙

    • S (SRP): 클래스는 단 하나의 책임만 가져야 한다.
    • O (OCP): 확장에는 열려있고, 수정에는 닫혀있어야 한다.
    • L (LSP): 자식 클래스는 부모 클래스를 대체할 수 있어야 한다.
    • I (ISP): 클라이언트(이미 구현된 코드를 사용하는 코드)는 자신이 사용하지 않는 메서드에 의존하면 안된다. 인터페이스에도 책임을 많이 가지지 말고 분리해야 한다.
    • D (DIP): 고수준 모듈은 저수준 모듈에 의존하면 안된다. 둘 다 추상화에 의존해야 한다.

    의존성(Dependency)이란

    한 클래스가 다른 클래스(혹은 객체)를 사용하는 관계를 말한다.

    2. Spring Core: IoC와 Bean

    IoC와 DI의 개념

    • IoC (제어의 역전): 제어의 흐름이 바뀌는 것이다. 전통적인 방식에서는 개발자가 객체를 직접 생성/관리했으나, 스프링이 객체의 생성과 관리를 대신 해주는 것이다.
    • DI (의존성 주입): IoC 개념을 실제로 구현하는 대표적인 방법이다. 스프링이 객체를 생성하고 관리하는 곳에서 가져와서 사용(주입)하는 것이다.
    • IoC Container: Bean의 생성 및 생명주기 관리를 하는 컨테이너이다. (DI, Bean 설정 정보 관리, 의존 관계 설정)

    Bean과 싱글톤

    • Bean: 스프링이 개발자를 대신해서 생성하는 객체이다. 컨테이너에 의해 생성, 관리, 소멸되며 기본적으로 싱글톤 스코프로 관리된다.
    • 싱글톤(Singleton): 클래스의 인스턴스가 단 1개만 생성되는 것을 보장하는 디자인 패턴이다.
      • 최초 한 번만 생성하므로 메모리 낭비를 방지할 수 있다.
      • 시스템 전반의 설정 정보나 공통 자원을 관리할 때 데이터의 일관성을 유지하기 쉽다.

    사용 기준 (IoC/DI vs new)

    • IoC/DI 사용: 재사용되거나 교체 가능한 비즈니스 로직 혹은 인프라에 사용한다.
    • new 사용: 매번 새로운 상태를 가지는 데이터 객체 (DTO, 엔티티) 생성에 사용한다.

    어노테이션 합성

    여러 개의 어노테이션을 조합하여 하나의 새로운 어노테이션을 만드는 것이다. 스프링은 어노테이션을 분석할 때, 메타 어노테이션까지 재귀적으로 탐색하여 기능을 적용해준다.

    3. Bean 등록과 의존성 주입

    Bean 등록 방법 2가지

    1. Bean 자동 등록 (@Component)

    • @ComponentScan이 설정된 패키지 하위의 모든 @Component 클래스를 찾아 등록한다.
    • 합성 어노테이션: @Controller, @Service, @Repository

    2. Bean 수동 등록 (@Bean)

    • @Configuration 클래스 내부의 @Bean 메서드를 사용하여 명시적으로 등록한다.
    • 외부 라이브러리나 복잡한 초기화 로직이 필요한 경우 사용한다.
    AppConfig 예시 (DI 호출 흐름)
    역할과 구현을 명확히 분리하여, DB가 바뀌어도 설정 파일만 수정하면 된다. 
    @Configuration
public class AppConfig {
    @Bean
    public MemberService memberService() { // 역할(Interface)
        // 구현(Implementation) - 나중에 DB로 바꿀 때 여기만 수정!
        return new MemberServiceImpl(memberRepository());
    }

    @Bean
    public MemberRepository memberRepository() {
        return new MemoryMemberRepository();
    }
}

    DI 주입 방식 3가지 (상세)

    1. 필드 주입

    @Autowired를 필드에 직접 선언하여 의존성을 주입하는 방식이다.
    테스트 코드 말고는 사용하지 않는다. 여러 문제가 있지만, 대표적으로 final 키워드 사용이 불가능하기 때문에 변수에 할당된 객체를 null 혹은 다른 객체로 바꿀 수 있어 안전하지 않다. 

    @Service
public class OrderService {
    @Autowired
    private MemberRepository memberRepository; // final 사용 불가, 변경 위험 존재
}

    2. Setter 주입

    Setter 메서드를 통해 의존성을 주입받는 방식이다.
    필드 주입과 같은 문제가 있어 안전하지 않다. 따라서 사용하지 않는다. 

    @Service
public class OrderService {
    private MemberRepository memberRepository;

    @Autowired
    public void setMemberRepository(MemberRepository memberRepository) {
        this.memberRepository = memberRepository; // 외부에서 호출하여 변경 가능함
    }
}

    3. 생성자 주입 (Best)

    생성자를 통해 의존성을 주입하는 방식이다.
    생성자 앞에 @Autowired를 붙일 수 있지만, 일반적으로 생략(생성자가 1개만 있으면 생략 가능)한다. 불변성을 보장하여 가장 안전하다. 

    @Service
public class OrderService {
    // final 키워드 사용 가능 -> 값이 무조건 있어야 함을 컴파일러가 체크해준다.
    private final MemberRepository memberRepository;

    // 생성자가 1개면 @Autowired 생략 가능하다.
    public OrderService(MemberRepository memberRepository) {
        this.memberRepository = memberRepository;
    }
}

    Bean 우선순위와 스코프

    • 우선순위(충돌 해결):
      • @Primary: 우선적으로 선택될 기본 Bean 지정한다.
      • @Qualifier: 이름으로 특정 Bean을 직접 지정한다. (Primary보다 우선순위 높음, @RequiredArgsConstructor 불가)
    • Bean 스코프:
      • 싱글톤: 시작 시 1번 생성, 계속 재사용한다.
      • 프로토타입: 요청마다 새로 생성한다. 생성 후 관리 안 한다.
    • 생명주기 콜백:
      • @PostConstruct: 의존성 주입 완료 직후 초기화한다.
      • @PreDestroy: 컨테이너 종료 직전 자원 해제한다.

    4. Validation (데이터 검증)

    • 프론트엔드 검증: 보안 취약하지만 유저 사용성을 위해 필요하다.
    • 서버 검증: API 스펙 정의 및 필수 방어선이다.
    • DB 검증: Not Null 등 최종 방어선이다.

    Bean Validation & 정규식

    어노테이션으로 검증 로직을 표현한다. @Valid로 자동 검증한다.

    • @NotBlank, @Email, @Size, @Min/@Max
    • @Pattern: 정규표현식(Regex) 사용한다. (예: 핸드폰 번호 패턴)
    // 휴대폰 번호 정규식 예시
public boolean isValidPhone(String phone) {
    return phone.matches("^010-\\d{4}-\\d{4}$");
}

    5. 인증과 인가: 세션 vs JWT

    • 인증(Authentication): "누구냐?" (로그인). 인가보다 먼저 수행되어야 한다.
    • 인가(Authorization): "권한 있냐?" (접근 제어).

    1. 세션 (Session) 방식

    비유: "놀이공원 자유이용권 (팔찌)"
    놀이공원(서버)이 입장객 명단(세션 저장소)을 가지고 있고, 입장객(클라이언트)에게는 명단 번호가 적힌 팔찌(세션 ID)만 채워준다. 직원은 팔찌 번호를 보고 명단을 확인해 "아, 철수구나!" 하고 들여보내준다.
    • 개념: 서버가 사용자의 로그인 상태를 기억하는 방식이다.
    • 흐름: 로그인 → 세션 ID 생성 및 서버 저장 → 클라이언트에 쿠키로 전송 → 요청 시 쿠키 확인
    • 장점: 보안 유리, 강제 로그아웃 제어 용이하다.
    • 단점: 사용자 증가 시 서버 메모리/DB 부하, 확장성 문제가 발생한다.

    2. JWT (Json Web Token) 방식

    비유: "위조 불가능한 신분증 (주민등록증)"
    동사무소(서버)는 아무것도 기록하지 않는다. 대신 도장이 찍힌 신분증(JWT)을 발급해준다. 어디를 가든 신분증만 보여주면 "진짜 도장이 찍혀있네? 통과!" 하고 인정받는다.
    • 개념: 서버가 상태를 저장하지 않는(Stateless) 인증 방식이다. 토큰 자체에 인증 정보가 있다.
    • 흐름: 로그인 → 암호화된 토큰(JWT) 생성 → 클라이언트에 전송 → 요청 시 헤더에 토큰 첨부
    • 장점: 서버 부하 적음, 확장성 용이, 모바일 등 다양한 환경 적합하다.
    • 단점: 토큰 탈취 시 만료까지 악용 가능하다 (Refresh Token 전략 필요), 세션 ID보다 크기가 크다.

    세부 기술 요소

    • HTTP Cookie: 헤더에 포함되며 SessionId/JWT의 운송 수단이다.
      보안 설정: Secure(HTTPS만), HttpOnly(JS 접근 불가), SameSite(CSRF 방지).
    • JWT 구성: 헤더(알고리즘) . 페이로드(데이터, 비밀번호 금지) . 서명(무결성 보장)

    심화: Refresh Token 전략

    Access Token이 탈취되면 끝장 아닌가?
    그래서 유효기간이 짧은(예: 30분) Access Token과 유효기간이 긴(예: 2주) Refresh Token을 함께 발급한다. Access Token이 만료되면, Refresh Token을 서버에 보내 새 Access Token을 발급받는다. Refresh Token은 서버 DB 등에 저장하여 탈취 감지 시 삭제(무효화)할 수 있다. 
    // Refresh Token을 이용한 재발급 로직 예시
@PostMapping("/reissue")
public ResponseEntity<TokenResponse> reissue(HttpServletRequest request) {
    // 1. 헤더나 쿠키에서 Refresh Token 추출
    String refreshToken = extractToken(request);
    
    // 2. Refresh Token 유효성 검증 (DB 조회 등)
    if (!tokenProvider.validate(refreshToken)) {
        throw new InvalidTokenException("유효하지 않은 토큰입니다.");
    }
    
    // 3. 새로운 Access Token 발급
    Authentication auth = tokenProvider.getAuthentication(refreshToken);
    String newAccessToken = tokenProvider.createAccessToken(auth);
    
    return ResponseEntity.ok(new TokenResponse(newAccessToken));
}

    6. Spring MVC 심화

    ArgumentResolver

    컨트롤러 파라미터에 값을 자동으로 바인딩해주는 도구이다. (예: @LoginUser)

    1. HandlerMethodArgumentResolver 인터페이스 구현
    2. supportsParameter(): 실행 조건 검사 (어노테이션 등 확인)
    3. resolveArgument(): 실제 주입할 값 생성 (세션 조회 등)
    4. WebMvcConfigurer에 등록 (addArgumentResolvers)

    HttpServletRequest vs HttpSession 

    // HttpSession: Spring이 알아서 꺼내준다
public void login(HttpSession session) {
    session.setAttribute("key", "value");
}

// HttpServletRequest: 직접 꺼내야 한다 (세션 생성 제어 가능)
public void login(HttpServletRequest request) {
    HttpSession session = request.getSession(false);
    session.setAttribute("key", "value");
}
    • HttpSession: 세션 기능만 쓸 때. 무조건 생성한다.
    • HttpServletRequest: IP, 헤더 정보 필요할 때. getSession(false)로 생성 방지 가능하다.

    추가 기능 코드 

    // [로그아웃] 세션 무효화
session.invalidate();

// [@RequestHeader] 특정 헤더만 가져오기
public void method(@RequestHeader("Authorization") String token) { ... }

    7. 예외 처리와 로깅

    로깅 (Logging)

    System.out.println 사용이 금지되는 이유는 성능 저하와 로그 관리의 어려움 때문이다. 대신 @Slf4j를 사용해 log.info() 형태로 남겨야 한다.

    레벨 설명
    TRACE 가장 상세한 흐름 정보이다.
    DEBUG 개발 단계 디버깅 정보이다.
    INFO 중요한 비즈니스 흐름, 운영 정보 (기본값)이다.
    WARN 잠재적 위험 경고이다.
    ERROR 기능 수행 불가능한 심각한 오류이다.

    설정: logging.level.root=WARN (application.properties)

    전역 예외 처리 (@RestControllerAdvice)

    @ExceptionHandler를 사용하여 컨트롤러 전역의 예외를 한 곳에서 처리한다. (사실상 표준)

    // 커스텀 에러 다형성 활용
@ExceptionHandler(ServiceException.class)
public ResponseEntity<String> handleServiceException(ServiceException ex) {
    return ResponseEntity.status(ex.getStatus()).body(ex.getMessage());
}
    • MovieNotFoundException 같은 구체적인 에러도 부모인 ServiceException 핸들러가 처리 가능하다.
    • Bean Validation 실패 시 MethodArgumentNotValidException 발생 → 여기서 처리 가능하다.
