    1. 객체 지향 설계 원칙

    응집도와 결합도

    • 응집도(Cohesion): 하나의 클래스가 하나의 목적을 위해 얼마나 긴밀하게 뭉쳐있는지 나타내는 척도.
      → Good: MemberService는 회원 관련 로직만 있다.
      → Bad: MemberService 안에 갑자기 Order(주문) 계산 로직이 섞여 있다.
    • 결합도(Coupling): 한 클래스가 변경될 때 다른 클래스에 얼마나 영향을 주는가?
      → Good: 인터페이스(MemberRepository)에 의존하여 구현체가 바뀌어도 코드를 안 고친다.
      → Bad: new MySQLRepository() 처럼 구체적인 클래스를 직접 쓰면 DB 바꿀 때 다 뜯어고쳐야 한다.

    SOLID 원칙

    • S (SRP): 클래스는 단 하나의 책임만 가져야 한다.
    • O (OCP): 확장은 열려있고, 수정은 닫혀있어야 한다.
    • L (LSP): 자식 클래스는 부모 클래스를 대체할 수 있어야 한다.
    • I (ISP): 인터페이스를 기능별로 잘게 쪼개야 한다.
    • D (DIP): 구체적인 클래스가 아니라 추상화(인터페이스)에 의존해야 한다.

    2. Spring Core: IoC와 Bean

    IoC와 DI

    • IoC (제어의 역전): 개발자 대신 프레임워크(Spring)가 객체를 생성하고 관리하는 것.
    • DI (의존성 주입): Spring이 만든 객체(Bean)를 필요한 곳에 쏙쏙 넣어주는 기술.
    • Bean: Spring 컨테이너가 관리하는 객체. 기본적으로 싱글톤(1개만 생성)이다.

    Bean 등록 방법 비교

    1. 자동 등록 (@Component)

    @Service // 내부에 @Component가 포함됨
public class MemberService {
    // ...
}

    2. 수동 등록 (@Bean)

    AppConfig 예시 (DI 호출 흐름)
    역할과 구현을 명확히 분리하여, DB가 바뀌어도 설정 파일만 수정하면 된다. 외부 라이브러리 등록 시 필수다. 
    @Configuration
public class AppConfig {
    @Bean
    public MemberService memberService() {
        // 생성자를 통해 의존성을 주입 (DI)
        return new MemberServiceImpl(memberRepository());
    }

    @Bean
    public MemberRepository memberRepository() {
        // 나중에 DB로 바꿀 때 이 부분만 수정하면 됨!
        return new MemoryMemberRepository(); 
    }
}

    3. 의존성 주입 (DI) 방식 3가지

    실무에서는 생성자 주입을 권장한다. 왜 그런지 코드로 비교해보자.

    1. 필드 주입 (Field Injection) - 비추천

    코드는 짧지만, 외부에서 수정이 불가능해 테스트 코드를 짤 때 객체를 넣을 방법이 없다.

    @Service
public class OrderService {
    @Autowired
    private MemberRepository memberRepository; // final 사용 불가!
}

    2. Setter 주입 (Setter Injection) - 비추천

    public 메서드로 열려있어, 누군가 실행 중에 실수로 바꿀 위험이 있다.

    @Service
public class OrderService {
    private MemberRepository memberRepository;

    @Autowired
    public void setMemberRepository(MemberRepository memberRepository) {
        this.memberRepository = memberRepository; // 변경 가능성 존재
    }
}

    3. 생성자 주입 (Constructor Injection) - 추천 (Best)

    객체 생성 시점에 딱 1번만 호출되므로 불변성을 보장한다. final 키워드를 쓸 수 있어 안전하다.

    @Service
public class OrderService {
    // final 키워드 사용 가능 -> 값이 무조건 있어야 함을 컴파일러가 체크해줌
    private final MemberRepository memberRepository;

    // 생성자가 1개면 @Autowired 생략 가능
    public OrderService(MemberRepository memberRepository) {
        this.memberRepository = memberRepository;
    }
}

    4. Bean 우선순위와 스코프

    같은 타입 빈이 2개일 때 해결법

    DiscountPolicy 인터페이스를 구현한 빈이 2개(Fix, Rate) 있다고 가정하자.

    방법 1: @Primary (기본값 지정)

    @Component
@Primary // 얘가 대장이다. 중복되면 얘를 써라.
public class RateDiscountPolicy implements DiscountPolicy { ... }

    방법 2: @Qualifier (이름으로 콕 집기)

    // 사용하는 곳에서
public OrderService(@Qualifier("fixDiscountPolicy") DiscountPolicy discountPolicy) {
    this.discountPolicy = discountPolicy;
}

    주의: @Qualifier는 @RequiredArgsConstructor(Lombok)와 함께 쓰기 어렵다. 직접 생성자를 만들어야 한다.

    Bean 생명주기 콜백 

    @PostConstruct
public void init() {
    System.out.println("초기화: 의존성 주입 끝난 후 실행");
}

@PreDestroy
public void close() {
    System.out.println("종료: 컨테이너 죽기 직전 실행 (연결 종료 등)");
}

    5. 인증/인가: 세션 vs JWT

    1. 세션 (Session) 방식

    비유: "놀이공원 팔찌"
    놀이공원(서버)이 입장객 명단을 가지고 있고, 손님에겐 번호 적힌 팔찌(세션ID)만 채워준다. 팔찌를 잃어버려도 명단은 서버에 있어서 안전하다.
    • 흐름: 로그인 → 세션 ID 생성 및 서버 저장 → 쿠키로 ID 전송 → 요청마다 쿠키 확인.
    • 장점: 보안 우수 (서버가 강제 로그아웃 가능).
    • 단점: 사용자가 몰리면 서버 메모리가 터진다.

    2. JWT (Json Web Token) 방식

    비유: "위조 불가능한 신분증"
    관공서(서버)는 아무것도 적어두지 않는다. 대신 도장이 찍힌 신분증(JWT)을 발급해준다. 어디든 신분증만 보여주면 "진짜 도장이네?" 하고 통과시켜준다.
    • 흐름: 로그인 → 암호화된 토큰(JWT) 발급 → 클라이언트가 저장 → 요청마다 헤더에 토큰 제출.
    • 장점: 서버 부하 없음(Stateless), 확장성 좋음.
    • 단점: 토큰 털리면 만료될 때까지 못 막음 (그래서 Refresh Token 씀).

    6. Spring MVC 심화 구현

    로그인/로그아웃 (HttpSession) 

    // [로그인]
@PostMapping("/login")
public String login(@RequestBody LoginDto dto, HttpServletRequest request) {
    // 세션이 있으면 있는 거 반환, 없으면 신규 생성 (true)
    HttpSession session = request.getSession(true);
    // 세션에 회원 정보 보관
    session.setAttribute("loginUser", member);
    return "ok";
}

// [로그아웃]
@PostMapping("/logout")
public String logout(HttpServletRequest request) {
    HttpSession session = request.getSession(false);
    if (session != null) {
        session.invalidate(); // 세션 날려버리기
    }
    return "ok";
}

    ArgumentResolver (파라미터 자동 주입)

    컨트롤러에서 @LoginUser User user 처럼 쓰면 알아서 세션 값을 넣어주는 마법.

    // 1. 리졸버 구현
public class LoginUserArgumentResolver implements HandlerMethodArgumentResolver {
    @Override
    public boolean supportsParameter(MethodParameter parameter) {
        // @LoginUser 어노테이션이 붙어있는가?
        return parameter.hasParameterAnnotation(LoginUser.class);
    }

    @Override
    public Object resolveArgument(...) {
        // 세션에서 값 꺼내서 반환 -> 컨트롤러 인자로 쏙!
        HttpServletRequest request = (HttpServletRequest) webRequest.getNativeRequest();
        HttpSession session = request.getSession(false);
        return (session == null) ? null : session.getAttribute("loginUser");
    }
}

// 2. 설정 등록 (필수!)
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
        resolvers.add(new LoginUserArgumentResolver());
    }
}

    7. 로깅(Logging)과 예외 처리

    로깅 전략

    System.out.println은 성능 문제로 실무에서 절대 금지다. @Slf4j를 써야 한다. 

    @Slf4j
@RestController
public class LogController {
    @GetMapping("/log")
    public String log() {
        log.trace("가장 상세한 추적");
        log.debug("개발할 때 변수 값 확인용");
        log.info("운영에 필요한 정보 (로그인, 결제 등)"); // 기본값
        log.warn("에러는 아닌데 좀 위험함");
        log.error("심각한 에러! 당장 확인 필요"); 
        return "ok";
    }
}

    전역 예외 처리 (@RestControllerAdvice)

    컨트롤러마다 try-catch 하지 말고 한 곳에서 몰아서 처리한다.

    @RestControllerAdvice
public class GlobalExceptionHandler {

    // 내가 만든 커스텀 에러 처리
    @ExceptionHandler(ServiceException.class)
    public ResponseEntity<ErrorResponse> handleCustomException(ServiceException e) {
        return ResponseEntity
                .status(e.getStatus())
                .body(new ErrorResponse(e.getMessage()));
    }

    // Validation 실패 시 발생하는 에러 처리
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<String> handleValidationException(MethodArgumentNotValidException e) {
        return ResponseEntity.badRequest().body("입력값을 확인해주세요.");
    }
}
