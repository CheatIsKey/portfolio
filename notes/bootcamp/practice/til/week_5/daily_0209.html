    객체 지향 설계의 본질

    응집도(Cohesion)와 결합도(Coupling)

    • 응집도 (높을수록 좋음):
      하나의 클래스가 하나의 목적을 위해 얼마나 긴밀하게 뭉쳐있는가?
      예시: `MemberService`에 회원 가입, 조회 로직만 있으면 응집도가 높지만, 갑자기 `Order`(주문) 로직이 섞여 있으면 응집도가 낮다.
    • 결합도 (낮을수록 좋음):
      한 클래스가 변경될 때 다른 클래스에 얼마나 영향을 주는가?
      예시: `Service`가 구체적인 `MySQLRepository` 클래스에 직접 의존하면 결합도가 높다. 인터페이스(`Repository`)에 의존해야 결합도가 낮아진다.

    SOLID 원칙 (객체 지향 5계명)

    • S (SRP - 단일 책임 원칙): 클래스는 변경해야 할 이유가 단 하나여야 한다.
    • O (OCP - 개방 폐쇄 원칙): 기능 추가에는 열려있고(Open), 기존 코드 수정에는 닫혀있어야(Closed) 한다. (인터페이스 활용)
    • L (LSP - 리스코프 치환 원칙): 자식 클래스는 부모 클래스의 기능을 깨트리지 않고 대체 가능해야 한다.
    • I (ISP - 인터페이스 분리 원칙): 뚱뚱한 인터페이스 하나보다, 구체적인 여러 개가 낫다.
    • D (DIP - 의존관계 역전 원칙): 구체적인 클래스(구현체)가 아니라 추상화(인터페이스)에 의존해야 한다.

    Spring Core: IoC와 Bean 관리

    제어의 역전 (IoC)과 의존성 주입 (DI)

    개발자가 new로 객체를 직접 생성하는 것이 아니라, Spring 컨테이너가 객체(Bean)의 생명주기를 관리하고 필요한 곳에 주입(Injection)해주는 것이다.

    Bean 등록 방법 비교

    1. 자동 등록 (@Component)

    • @ComponentScan이 붙은 설정 정보 하위 패키지를 다 뒤져서 등록한다.
    • 주로 우리가 만드는 컨트롤러, 서비스에 사용한다.
    @Service // 내부적으로 @Component를 포함함
public class MemberService { ... }

    2. 수동 등록 (@Bean)

    • 외부 라이브러리(예: AWS SDK, PasswordEncoder)처럼 코드를 수정할 수 없을 때 사용한다.
    • 설정 클래스(@Configuration) 안에서 메서드로 등록한다.
    @Configuration
public class SecurityConfig {
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}

    DI 주입 방식과 우선순위

    • 생성자 주입 (Best): final 필드 사용 가능(불변성), 테스트 코드 작성 용이. 생성자가 1개면 @Autowired 생략 가능.
    • Bean 충돌 해결:
      • @Primary: 여러 빈 중 기본값으로 쓸 빈에 붙인다. (범용적)
      • @Qualifier("name"): 특정 이름을 가진 빈을 콕 집어서 주입받는다. (지엽적, 우선순위 높음)

    생명주기 콜백

    빈이 생성되거나 죽기 직전에 해야 할 일(DB 연결/종료 등)을 처리한다.

    @PostConstruct // 의존성 주입이 끝난 직후 실행 (초기화)
public void init() {
    System.out.println("초기화 완료");
}

@PreDestroy // 컨테이너 종료 직전 실행 (자원 해제)
public void close() {
    System.out.println("종료");
}

    인증(Authentication)과 인가(Authorization)

    세션 (Session) vs JWT

    구분 세션 (Session) JWT (Json Web Token)
    저장 위치 서버 메모리/DB (Stateful) 클라이언트 로컬스토리지/쿠키 (Stateless)
    장점 보안성 높음 (강제 로그아웃 가능) 서버 확장성 좋음, 모바일 친화적
    단점 사용자가 많으면 서버 터짐 (메모리 부족) 토큰 탈취 시 만료될 때까지 속수무책
    흐름 로그인 → SessionID 생성 → 쿠키로 전송 로그인 → 암호화 토큰 생성 → 헤더로 전송

    HTTP Cookie 보안 옵션

    • Secure: HTTPS 연결에서만 쿠키를 전송한다. (네트워크 감청 방지)
    • HttpOnly: 자바스크립트로 쿠키 접근 불가. (XSS 해킹 방지)
    • SameSite: 다른 사이트 요청 시 쿠키 전송 제한. (CSRF 공격 방지)

    Spring MVC 심화: 파라미터 자동화

    ArgumentResolver 활용

    컨트롤러에서 매번 request.getSession()이나 헤더를 파싱하는 건 귀찮다. @LoginUser 같은 어노테이션만 붙이면 알아서 객체를 꽂아주게 만들 수 있다. 

    // 1. HandlerMethodArgumentResolver 구현
public class LoginUserArgumentResolver implements HandlerMethodArgumentResolver {
    @Override
    public boolean supportsParameter(MethodParameter parameter) {
        // 파라미터에 @LoginUser 어노테이션이 있는지 확인
        return parameter.hasParameterAnnotation(LoginUser.class);
    }

    @Override
    public Object resolveArgument(...) {
        // 세션에서 사용자 정보 꺼내서 반환 -> 컨트롤러 파라미터에 쏙 들어감
        HttpServletRequest request = (HttpServletRequest) webRequest.getNativeRequest();
        HttpSession session = request.getSession(false);
        return session.getAttribute("LOGIN_USER");
    }
}

// 2. WebMvcConfigurer에 등록 필수!
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
        resolvers.add(new LoginUserArgumentResolver());
    }
}

    로깅 (Logging) 완전 정복

    개발할 땐 System.out.println이 편해 보이지만, 실제 운영 서버에서는 재앙이 될 수 있다. 반드시 로깅 프레임워크(SLF4J + Logback)를 써야 한다.

    왜 System.out.println을 쓰면 안 되나?

    • 성능 저하: 내부적으로 synchronized(동기화) 처리가 되어 있어, 요청이 몰리면 로그 찍다가 서버가 멈춘다.
    • 정보 부족: 언제, 어떤 쓰레드에서, 어떤 클래스에서 발생했는지 알 수 없다.
    • 제어 불가: "에러만 보고 싶은데.."와 같은 필터링이 불가능하다. 일일이 코드를 지워야 한다.

    로그 레벨 (Level)과 사용 기준

    우선순위: TRACE < DEBUG < INFO < WARN < ERROR

    레벨 의미 사용 상황 예시
    ERROR 요청 처리 불가, 시스템 에러 DB 연결 실패, NPE 발생, catch 블록 내부
    WARN 처리는 가능하나 잠재적 위험 사용되지 않는 API 호출, 디스크 용량 부족 경고
    INFO 중요한 비즈니스 진행 정보 "회원가입 완료", "주문 결제 성공", 서버 시작/종료 메시지
    DEBUG 개발용 디버깅 정보 SQL 실행 문장, 변수에 들어있는 값 확인 (운영 서버에선 끔)
    TRACE 가장 상세한 추적 정보 모든 실행 경로 추적 (거의 사용 안 함)

    실무 사용법 (Lombok @Slf4j) 

    @Slf4j // 1. Lombok 어노테이션으로 자동 주입
@RestController
public class OrderController {

    @PostMapping("/order")
    public String order() {
        try {
            // 2. 상황에 맞는 레벨 사용
            log.info("주문 요청이 들어왔습니다."); 
            
            // ... 비즈니스 로직 ...
            
            log.debug("현재 결제 금액: {}", 15000); // 변수값 확인용
            
        } catch (Exception e) {
            // 3. 에러 로그엔 반드시 예외 객체(e)를 함께 남김
            log.error("주문 처리 중 오류 발생", e);
        }
        return "ok";
    }
}

    로그 레벨 설정 (application.properties) 

    # 기본 레벨 설정 (보통 INFO)
logging.level.root=INFO

# 특정 패키지만 레벨 다르게 설정 가능
# com.example 패키지는 상세하게(DEBUG) 보고 싶다!
logging.level.com.example=DEBUG

    Spring 예외 처리 전략

    @RestControllerAdvice (전역 처리)

    개별 컨트롤러에서 try-catch 하지 말고, 에러 처리만 전담하는 클래스를 따로 만든다. 

    @RestControllerAdvice
public class GlobalExceptionHandler {

    // 1. 구체적인 예외 (커스텀 예외 등)
    @ExceptionHandler(ServiceException.class)
    public ResponseEntity<String> handleServiceException(ServiceException e) {
        return ResponseEntity.status(e.getStatus()).body(e.getMessage());
    }

    // 2. 나머지 모든 예외 (최후의 보루)
    @ExceptionHandler(Exception.class)
    public ResponseEntity<String> handleAll(Exception e) {
        log.error("알 수 없는 에러 발생", e); // 로깅 필수!
        return ResponseEntity.status(500).body("서버 내부 오류입니다.");
    }
}

    오늘의 회고

    오늘 느낀 점

    • 로깅은 단순히 콘솔에 글자를 찍는 게 아니라, "문제를 추적하는 블랙박스"라는 점을 깨달았다. 운영 서버에서는 DEBUG 로그를 꺼서 성능을 확보한다는 설정 전략이 인상 깊었다.
    • @RestControllerAdvice를 적용하니 컨트롤러 코드가 비즈니스 로직에만 집중할 수 있게 되어 가독성이 훨씬 좋아졌다.
    • ArgumentResolver를 직접 구현해보니, Spring이 내부적으로 얼마나 많은 편의 기능을 제공하고 있었는지 새삼 느꼈다.
