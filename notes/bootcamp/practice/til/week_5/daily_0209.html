    오늘 목표

    • 좋은 소프트웨어 설계를 위한 응집도와 결합도, SOLID 원칙을 이해한다.

    • 웹 보안의 핵심인 인증/인가(세션 vs JWT)의 흐름과 구현 방법을 상세히 정리한다.
    • 실무에서 필수적인 로깅(Logging) 전략과 예외 처리, ArgumentResolver 등을 마스터한다.

    객체 지향 설계 원칙

    응집도와 결합도

    • 응집도(Cohesion):
      하나의 클래스(또는 모듈)가 하나의 목적을 위해 얼마나 긴밀하게 뭉쳐있는지 나타내는 척도이다.
      클래스 내의 메소드와 데이터들이 얼마나 서로 관련 있는 작업을 수행하는 지를 의미한다.
      응집도를 높이기 위해서는 하나의 클래스는 하나의 책임만 가져야 한다.
    • 결합도(Coupling):
      하나의 클래스(또는 모듈)가 다른 클래스와 얼마나 많이 엮어있는지, 즉 얼마나 의존적인지를 나타내는 척도이다.
      결합도를 낮추기 위해서는 하나의 클래스가 변경되어도 다른 클래스에 미치는 영향이 거의 없도록 설계해야 한다.

    SOLID 원칙

    • S (SRP): 단일 책임 원칙. 클래스는 단 하나의 책임만 가져야 한다.
    • O (OCP): 개방 폐쇄 원칙. 확장에는 열려있고, 수정에는 닫혀있어야 한다.
    • L (LSP): 리스코프 치환 원칙. 자식 클래스는 부모 클래스를 대체할 수 있어야 한다.
    • I (ISP): 인터페이스 분리 원칙. 클라이언트는 자신이 사용하지 않는 메서드에 의존하면 안된다. 인터페이스에도 책임을 많이 가지지 말고 분리해야 한다.
    • D (DIP): 의존관계 역전 원칙. 고수준 모듈은 저수준 모듈에 의존하면 안된다. 둘 다 추상화에 의존해야 한다.

    의존성(Dependency)이란?

    한 클래스가 다른 클래스(혹은 객체)를 사용하는 관계를 말한다.

    Spring Core: IoC와 Bean

    IoC와 DI

    • IoC (Inversion of Control, 제어의 역전): 제어의 흐름이 바뀌는 것이다. 전통적인 프로그래밍 방식에서는 개발자가 객체를 직접 생성하고 관리했으나, 스프링이 객체의 생성과 관리를 대신 해주는 것이다.
    • DI (Dependency Injection, 의존성 주입): IoC 개념을 실제로 구현하는 대표적인 방법이다. 스프링이 객체를 생성하고 관리하는 곳에서 가져와서 사용(주입)하는 것이다.
    • IoC Container: Bean의 생성 및 생명주기 관리를 하는 컨테이너이다. (DI, 설정 정보 관리, 의존 관계 설정)

    Bean과 싱글톤

    • Bean: 스프링이 개발자를 대신해서 생성하는 객체이다. 컨테이너에 의해 생성, 관리, 소멸되며 애플리케이션 전역에서 재사용 가능하다. 기본적으로 싱글톤 스코프로 관리된다.
    • 싱글톤(Singleton): 클래스의 인스턴스가 단 1개만 생성되는 것을 보장하는 디자인 패턴이다.
      • 최초 한 번만 객체를 생성하므로 메모리 낭비를 방지할 수 있다.
      • 시스템 전반의 설정 정보나 공통 자원을 관리할 때 데이터의 일관성을 유지하기 쉽다.

    IoC/DI vs new 사용 기준

    • IoC/DI 사용: 재사용되거나 교체 가능한 비즈니스 로직/인프라 (Service, Repository 등)
    • new 사용: 매번 새로운 상태를 가지는 데이터 객체 (DTO, Entity)

    스프링 Bean 등록 방식

    1. Bean 자동 등록 (@Component)

    • Spring이 @Component 클래스를 찾아서 Bean으로 등록한다.
    • @ComponentScan에 설정되어 있는 패키지를 기준으로 하위의 모든 @Component 클래스를 탐색한다.
    • 관련 합성 어노테이션: @Controller, @Service, @Repository

    2. Bean 수동 등록 (@Bean)

    • @Configuration 클래스와 @Bean 메서드를 사용하여 명시적으로 등록한다.
    • 외부 라이브러리나 복잡한 초기화 로직이 필요한 객체를 등록할 때 유용하다.
    • 기본적으로 자동 등록이 우선이며, 불가피한 경우(외부 라이브러리 등)에만 수동 등록을 활용한다.

    AppConfig 예시 (DI 호출 흐름) 

    // AppConfig.class
@Bean
public MemberService memberService() { // 역할(Interface)
    // 구현(Implementation) - 나중에 DB로 바꿀 때 여기만 수정!
    // memberRepository() 메서드를 호출하여 의존성 주입
    return new MemberServiceImpl(memberRepository()); 
}

@Bean
public MemberRepository memberRepository() {
    return new MemoryMemberRepository();
}

    DI 주입 방식 3가지

    1. 필드 주입

    @Autowired를 필드에 직접 선언하여 주입하는 방식이다. 테스트 코드 외에는 사용하지 않는다. final 키워드 사용 불가로 객체가 변경될 위험이 있어 안전하지 않다.

    2. Setter 주입

    Setter 메서드를 통해 의존성을 주입받는 방식이다. 필드 주입과 마찬가지로 안전하지 않아 사용하지 않는다.

    3. 생성자 주입 (권장)

    생성자를 통해 의존성을 주입하는 방식이다. 생성자 앞에 @Autowired를 붙일 수 있지만, 생성자가 1개만 있으면 생략 가능하다. 불변성(final)을 보장하므로 가장 안전하다.

    Bean 우선순위와 스코프

    • Bean 충돌 해결: 똑같은 타입의 Bean이 2개 이상이면 에러가 발생한다.
      • @Primary: 우선적으로 선택될 기본 Bean 지정.
      • @Qualifier: 이름으로 특정 Bean을 직접 지정. (@Primary보다 우선순위 높음, @RequiredArgsConstructor 사용 불가)
    • Bean 스코프:
      • 싱글톤 (기본값): 컨테이너 시작 시 한 번만 생성되어 계속 재사용됨.
      • 프로토타입: @Scope("prototype"). 요청 시마다 새로운 객체 생성. 생성 이후 관리는 안 함.
    • 생명주기 콜백:
      • @PostConstruct: 의존성 주입 완료 직후 초기화 작업 수행.
      • @PreDestroy: 컨테이너 종료 직전 자원 해제 등 뒷정리 작업 수행.

    Validation (데이터 검증)

    검증의 종류와 역할

    • 프론트엔드 검증: 유저 사용성 증가, 보안 취약.
    • 서버 검증: 필수. API 스펙 정의 및 에러 응답 제공.
    • DB 검증: Not Null, Default 등 최종 방어선.

    Bean Validation 사용법

    어노테이션으로 검증 로직을 직관적으로 표현한다. 컨트롤러에서 @Valid만 붙이면 자동 검증된다.

    • @NotBlank: 공백이 아닌 문자 1개 이상 (null, "", " " 거부)
    • @Email: 이메일 형식
    • @Size, @Min, @Max: 크기 및 숫자 범위 제한
    • @Pattern: 정규표현식 (전화번호 등)
      예시: phone.matches("^010-\\d{4}-\\d{4}$")

    실패 시 MethodArgumentNotValidException 에러가 발생하며, GlobalExceptionHandler에서 처리 가능하다.

    인증(Authentication)과 인가(Authorization)

    개념 구분

    • 인증 (로그인): 사용자가 누구인지 확인하는 절차. (시스템 등록 사용자 증명)
    • 인가 (권한): 인증된 사용자가 특정 리소스에 접근할 수 있는지 확인하는 절차.
    • 순서: 인증은 인가보다 항상 먼저 수행되어야 한다.

    세션 (Session) vs JWT

    1. 세션 (Session)

    서버가 사용자의 로그인 상태를 기억하는 방식이다.

    • 흐름: 로그인 → 세션 ID 생성 및 서버 저장 → 클라이언트에 쿠키로 전송 → 요청 시 쿠키 확인
    • 장점: 보안 유리, 강제 로그아웃 제어 용이.
    • 단점: 사용자 증가 시 서버 메모리/DB 부하, 확장성 문제.

    2. JWT (Json Web Token)

    서버가 상태를 저장하지 않는(Stateless) 인증 방식이다. 토큰 자체에 인증 정보가 있다.

    • 흐름: 로그인 → 암호화된 토큰(JWT) 생성 → 클라이언트에 전송 → 요청 시 헤더에 토큰 첨부
    • 장점: 서버 부하 적음, 확장성 용이, 모바일 등 다양한 환경 적합.
    • 단점: 토큰 탈취 시 만료까지 악용 가능 (Refresh Token 전략 필요), 세션 ID보다 크기가 큼.

    JWT 구성 요소

    • 헤더(Header): 토큰 유형과 알고리즘 정보. Base64Url 인코딩.
    • 페이로드(Payload): 실제 정보(클레임). Base64Url 인코딩. (비밀번호 절대 금지)
    • 서명(Signature): 헤더+페이로드+비밀키를 암호화한 값. 무결성 보장.

    Access Token & Refresh Token 전략

    • Access Token: 유효기간 짧음. 실제 자원 접근용. 탈취 피해 최소화.
    • Refresh Token: 유효기간 김. Access Token 재발급용. 안전한 곳(DB 등)에 저장.

    HTTP Cookie와 보안

    쿠키는 헤더이며, SessionId와 JWT를 담는 상자(운송 수단)이다. Key-Value 형태이다.

    Set-Cookie 보안 설정

    • Secure: HTTPS 통신일 때만 전송 (탈취 방지).
    • HttpOnly: 자바스크립트 접근 차단 (XSS 방지).
    • SameSite: 타 도메인 요청 시 전송 제한 (CSRF 방지).

    Spring MVC 심화 기능

    ArgumentResolver

    컨트롤러 파라미터에 값을 자동으로 바인딩해주는 도구이다. (@LoginUser 등 구현 가능)

    1. HandlerMethodArgumentResolver 인터페이스 구현
    2. supportsParameter(): 실행 조건 검사 (어노테이션 등 확인)
    3. resolveArgument(): 실제 주입할 값 생성 (세션 조회 등)
    4. WebMvcConfigurer에 등록 (addArgumentResolvers)

    HttpServletRequest vs HttpSession

    HttpSession 사용 예제 (로그인)

    public void login(HttpSession session) {
    // Spring이 알아서 세션을 꺼내줌
    session.setAttribute("loginUser", user);
    // JSESSIONID 쿠키 자동 발급됨
}

    HttpServletRequest 사용 예제 (로그인)

    public void login(HttpServletRequest request) {
    // 직접 꺼내야 함. false 옵션으로 불필요한 생성 방지 가능
    HttpSession session = request.getSession(false);
    session.setAttribute("loginUser", user);
}

    차이점: HttpSession은 세션 기능만 쓸 때, HttpServletRequest는 IP, 헤더 등 추가 정보가 필요할 때 사용한다.

    로그아웃 기능 구현 

    // 세션 속성 조회 (@SessionAttribute)
public String update(@SessionAttribute(name="loginUser", required=false) User user) { ... }

// 로그아웃 (세션 무효화)
session.invalidate(); // 모든 데이터 삭제

    @RequestHeader

    HttpServletRequest 전체를 안 가져오고 특정 헤더만 쏙 빼올 수 있다.
    예시: @RequestHeader("Authorization") String token

    예외 처리와 로깅 전략

    스프링 예외 처리 전략 3가지

    1. @ExceptionHandler: 컨트롤러별 예외 처리. (중복 발생 가능)
    2. @RestControllerAdvice: 전역 예외 처리. (사실상 표준, 가장 많이 사용)
    3. Spring 기본 예외 처리

    커스텀 에러와 핸들링 예시 

    @Getter
public class ServiceException extends RuntimeException {
    private final HttpStatus status;
    public ServiceException(HttpStatus status, String message) {
        super(message);
        this.status = status;
    }
}

public class MovieNotFoundException extends ServiceException {
    public MovieNotFoundException(String message) {
        super(HttpStatus.NOT_FOUND, message); // 404 지정
    }
}

// GlobalExceptionHandler 내부
@ExceptionHandler(ServiceException.class)
public ResponseEntity<String> handleServiceException(ServiceException ex) {
    return ResponseEntity.status(ex.getStatus()).body(ex.getMessage());
}

    로깅 (Logging)

    System.out.println 금지 이유: 성능 저하, 로그 레벨 제어 불가, 정보 부족. 대신 @Slf4j를 사용해 log.info() 형태로 남겨야 한다.

    로그 레벨 (낮음 -> 높음)

    • TRACE: 가장 상세한 흐름 정보
    • DEBUG: 개발 단계 디버깅 정보
    • INFO: 중요한 비즈니스 흐름, 운영 정보 (기본값)
    • WARN: 잠재적 위험 경고
    • ERROR: 기능 수행 불가능한 심각한 오류

    설정: logging.level.root=WARN (application.properties)

    오늘의 회고

    오늘 느낀 점

    • DI가 단순히 '의존성 주입'인 줄만 알았는데, AppConfig 예제를 통해 객체 생성과 사용의 책임을 분리하는 거대한 설계 철학임을 깨달았다.
    • 로그인 구현 시 세션과 쿠키가 어떻게 상호작용하는지(JSESSIONID 등) 내부 흐름을 명확히 알게 되었다.
    • 예외 처리를 전역으로 관리하고 로깅을 체계적으로 남기는 것이 유지보수에 얼마나 중요한지 실감했다.
