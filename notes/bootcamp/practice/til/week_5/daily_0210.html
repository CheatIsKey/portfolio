    1. 영속성 컨텍스트 (Persistence Context)

    엔티티의 생명주기를 관리하고, 애플리케이션과 DB 사이에서 수많은 최적화 작업을 해주는 논리적인 공간(메모리 공간)이다.

    사용 이유와 특징

    1. 1차 캐시와 동일성(Identity) 보장 영속성 컨텍스트는 내부에 Map 형태의 캐시 저장소를 가지고 있다.
    조회 시 1차 캐시를 먼저 확인한다. 동일한 ID의 엔티티가 있으면 DB를 조회하지 않고 캐시 값을 즉시 반환한다.
    이 덕분에 같은 트랜잭션 내에서 조회한 엔티티는 항상 동일한 메모리 주소를 보장한다. 
    // 1차 캐시 예시
Member member1 = em.find(Member.class, 1L); // DB 조회
Member member2 = em.find(Member.class, 1L); // 1차 캐시에서 조회 (SQL 실행 X)

System.out.println(member1 == member2); // true (동일성 보장)
    2. 변경 감지 (Dirty Checking) 트랜잭션 커밋 직전, 1차 캐시의 최초 상태(스냅샷)와 현재 상태를 비교한다.
    다르면 JPA가 자동으로 UPDATE SQL을 생성하여 DB에 반영한다. 
    // 변경 감지 예시
Member member = em.find(Member.class, 1L);
member.setName("NewName"); // 데이터만 변경

// em.update(member) 같은 코드가 필요 없다.
// 트랜잭션 커밋 시점에 자동으로 UPDATE 쿼리가 나간다.
    3. 쓰기 지연 (Transactional Write-Behind) save() 호출 시 즉시 DB에 보내지 않고 INSERT SQL을 저장소에 모아둔다.
    트랜잭션이 커밋되는 순간(Flush), 모아둔 SQL을 한 번에 DB로 보낸다.
    → 네트워크 비용 감소 및 DB 상호작용 최소화로 성능 최적화.

    Flush와 EntityManager

    • Flush란: 영속성 컨텍스트의 변경 내용을 DB에 동기화하는 것이다. (SQL 전송). 트랜잭션 확정(Commit)은 아니다. 롤백 시 함께 롤백된다.
    • Persistence Unit (설계도): JPA 설정 정보 단위. DB 연결 정보 및 관리할 엔티티 명세.
    • EntityManagerFactory (공장): Persistence Unit을 기반으로 생성된다. 애플리케이션 전체에서 하나만 생성해 공유한다. (생성 비용 비쌈, Thread-Safe)
    • EntityManager (작업자): 실제 DB 작업을 수행한다(CRUD). DB 트랜잭션과 생명주기를 같이 하며, 요청마다 생성하고 닫아야 한다. (생성 비용 저렴, Not Thread-Safe)

    2. 트랜잭션 (Transaction)

    트랜잭션 전파 (Propagation)

    @Transactional(propagation = ...) 옵션으로 설정한다.

    전파 옵션 동작 방식 사용 예시 및 설명
    REQUIRED (Default) 부모 있으면 합류, 없으면 신규 생성. 서비스 핵심 로직 대부분에 사용한다.
    REQUIRED_NEW 무조건 새로운 트랜잭션 생성. 로그 기록: 핵심 로직이 실패해 롤백되어도, 에러 로그는 DB에 남아야 할 때 사용한다. 부모와 완전 별개다.
    NESTED 부모 있으면 중첩(SavePoint), 없으면 신규. 쿠폰 발급: 주문 중 쿠폰 실패 시 쿠폰만 롤백하고 주문은 유지하고 싶을 때. (부모 롤백 시 자식도 롤백됨)
    SUPPORTS 부모 있으면 합류, 없으면 트랜잭션 없이. 단순 읽기 전용 메서드에 적합하다.
    NOT_SUPPORTS 항상 트랜잭션 없이 실행. DB 커넥션 점유를 방지해야 하는 단순 로직.
    MANDATORY 부모 없으면 예외 발생. 반드시 기존 트랜잭션 내에서만 실행되어야 함을 강제할 때 (안전장치).
    NEVER 트랜잭션 있으면 예외 발생. 트랜잭션이 있으면 안 되는 로직 보호용.

    트랜잭션 격리 수준 (Isolation Level)

    • READ UNCOMMITTED: 커밋 전 데이터 읽기 가능. DIRTY READ 발생 (작업 미완료 데이터 조회). 데이터 정합성 문제로 권장 안 함.
    • READ COMMITTED: 커밋된 데이터만 읽음. RDB 기본값. NON-REPEATABLE READ 발생 (한 트랜잭션 내 같은 조회의 결과가 다름). Undo 영역 사용.
    • REPEATABLE READ: 트랜잭션 시작 전 커밋된 데이터만 읽음. MySQL 기본값. PHANTOM READ 발생 (없던 레코드가 갑자기 조회됨).
    • SERIALIZABLE: 가장 엄격함. 읽기 작업에도 Lock을 검. 성능이 매우 떨어짐.

    3. JPA 연관관계 매핑

    기본 키(PK)와 외래 키(FK)

    • Primary Key (PK): 행을 식별하는 고유 값. 유일성, NOT NULL, 불변성 보장.
    • Foreign Key (FK): 다른 테이블의 PK를 참조하는 값. 테이블 간 연결 고리. 중복/NULL 허용.

    매핑 어노테이션과 연관관계 주인

    연관관계 주인: 외래 키(FK)를 가진 테이블과 매핑된 엔티티가 주인이다.
    주인이 아닌 쪽은 mappedBy로 설정하며 조회만 가능하다.

    • @ManyToOne: N:1 관계. 연관관계의 주인. @JoinColumn으로 FK 관리. (가장 중요)
    • @OneToMany: 1:N 관계. @ManyToOne이 있는 곳에 사용.
    • @OneToOne: 1:1 관계. 주 사용 테이블에 FK를 두는 것을 권장한다.
    • @ManyToMany: 절대 사용 금지. 중간 테이블이 숨겨져 생기므로 @ManyToOne 2개로 풀어서 개발해야 한다.

    @JoinColumn 상세 옵션

    • name: FK 컬럼명 지정 (예: "team_id").
    • unique: true 시 유니크 제약 생성 (사실상 1:1이 됨).
    • nullable: FK의 NULL 허용 여부. optional과 짝을 맞춰야 한다.
    • foreignKey: 제약 조건 메타데이터.
      • ConstraintMode.NO_CONSTRAINT: 실제 FK 제약은 생성 안 하고 논리적 연관만 가짐. (MSA 환경 등에서 사용)

    4. 조회 전략과 최적화

    FetchType (조회 시점 결정)

    • EAGER (즉시 로딩): 부모 조회 시 자식까지 한 번에 다 가져온다. 쓰지 마라. 선택권이 없다.
    • LAZY (지연 로딩): 실제 데이터가 필요한 시점에 쿼리를 날린다. 무조건 이걸 써라.
      • 프록시(Proxy) 객체: LAZY 사용 시, 연관 객체 자리에 가짜(프록시)가 들어있다. 실제 사용 시(getXXX) 내부적으로 DB를 조회해 초기화된다.
    @ManyToOne(fetch = FetchType.LAZY) // 지연 로딩 설정
@JoinColumn(name = "team_id")
private Team team;

    영속성 전이 (Cascade)

    부모의 상태 변화를 자식에게 전파한다. @OneToMany(cascade = CascadeType.ALL) 형태로 쓴다.

    • ALL: 모든 옵션 적용. 생명주기가 일치할 때 사용.
    • PERSIST: 부모 저장 시 자식도 저장.
    • REMOVE: 부모 삭제 시 자식도 삭제.
    • MERGE: 부모 병합 시 자식도 병합.
    • REFRESH: 부모 다시 읽을 때 자식도 갱신.
    • DETACH: 부모 준영속 시 자식도 준영속.

    고아 제거 (Orphan Removal)

    부모와 연관관계가 끊어진 자식을 자동으로 삭제한다. (기본값 false)

    • 부모의 컬렉션에서 자식을 remove() 하면 DELETE 쿼리 실행.
    • 부모 컬렉션이 new ArrayList() 등으로 교체되면 기존 자식 전멸.
    • 부모 자체가 삭제되면 자식도 고아가 되므로 함께 삭제됨.

    5. JPQL과 프로젝션

    JPQL (Java Persistence Query Language)

    DB 테이블이 아닌 엔티티 객체를 대상으로 하는 객체지향 쿼리다. DB에 종속되지 않는다.

    // :username 파라미터 바인딩 예시
@Query("SELECT m FROM Member m WHERE m.username = :username")
List<Member> findUser(@Param("username") String username);

    프로젝션 (Projection)

    엔티티 전체가 아니라 필요한 필드만 골라서 조회하는 기술이다. 성능 최적화와 타입 안정성을 제공한다.

    1. 인터페이스 기반

    public interface UsernameOnly {
    String getUsername(); // Getter만 정의하면 구현체 자동 생성
}

    2. 클래스(DTO) 기반

    new 키워드와 패키지 경로를 포함해 생성자를 호출하는 방식이다.

    @Query("SELECT new com.example.dto.MemberDto(m.username, m.age) FROM Member m")
List<MemberDto> findMemberDto();

    6. JPA 테스트

    @DataJpaTest를 사용하면 Repository 계층만 슬라이스 테스트할 수 있다. 내부적으로 Transactional이 있어 테스트 후 자동 롤백된다.

    테스트 구성 요소

    • 의존성: testImplementation "com.h2database:h2" (테스트용 인메모리 DB)
    • Given-When-Then 패턴:
      • Given: 데이터 준비 (엔티티 생성 등)
      • When: 테스트 실행 (repository.save, find 등)
      • Then: 결과 검증 (Assertions.assertThat 사용)
    @DataJpaTest
class MemberRepositoryTest {
    
    @Autowired MemberRepository memberRepository;

    @Test
    void saveMember() {
        // Given
        Member member = new Member("userA");

        // When
        Member savedMember = memberRepository.save(member);

        // Then
        assertThat(savedMember.getId()).isNotNull();
        assertThat(savedMember.getName()).isEqualTo("userA");
    }
}
