    1. 영속성 컨텍스트 (Persistence Context)

    엔티티의 생명주기를 관리하고, 애플리케이션과 DB 사이에서 수많은 최적화 작업을 해주는 논리적인 공간(메모리 공간)이다.

    사용 이유와 특징

    1. 1차 캐시와 동일성(Identity) 보장
    영속성 컨텍스트는 내부에 Map 형태의 캐시 저장소를 가지고 있다.
    엔티티가 영속성 컨텍스트의 1차 캐시에 저장된다.
    조회 시 먼저 1차 캐시를 확인한다. 캐시에 동일한 ID의 엔티티 객체가 있으면, DB를 조회하지 않고 캐시의 엔티티를 즉시 반환한다.
    이 덕분에 같은 트랜잭션 안에서 같은 ID로 조회한 엔티티는 항상 동일한 메모리 주소의 인스턴스임이 보장된다. 
    // 1차 캐시 및 동일성 보장 예시
Member member1 = em.find(Member.class, 1L); // DB 조회
Member member2 = em.find(Member.class, 1L); // 1차 캐시에서 조회 (SQL 실행 X)

System.out.println(member1 == member2); // true (동일성 보장)
    2. 변경 감지 (Dirty Checking)
    트랜잭션이 커밋되기 직전에 영속성 컨텍스트는 1차 캐시에 있는 모든 엔티티의 최초 상태(스냅샷)와 현재 상태를 비교한다.
    만약 두 상태가 다르다면, JPA는 변경된 내용을 감지하고 자동으로 UPDATE SQL을 생성하여 DB에 반영한다. 
    // 변경 감지 예시
Member member = em.find(Member.class, 1L);
member.setName("NewName"); // 데이터만 변경

// em.update(member) 같은 코드가 필요 없다.
// 트랜잭션 커밋 시점에 자동으로 UPDATE 쿼리가 나간다.
    3. 쓰기 지연 (Transactional Write-Behind)
    memberRepository.save()가 호출되면, 해당 엔티티는 1차 캐시에 저장되고, JPA는 이 엔티티를 DB에 반영하기 위한 INSERT SQL을 생성하여 모아둔다.
    UPDATE나 DELETE 쿼리도 마찬가지로 이 저장소에 쌓인다.
    트랜잭션이 커밋되는 순간, JPA는 모아뒀던 모든 SQL들을 한 번에 DB로 보낸다(Flush).
    → SQL을 하나씩 보내지 않고 모았다가 한 번에 처리하므로, 네트워크 비용을 줄이고 DB와 상호작용하는 횟수를 최소화하여 성능을 최적화할 수 있다.

    Flush와 EntityManager

    • Flush란: 영속성 컨텍스트의 변경 내용을 DB에 동기화하는 것이다. DB에 SQL을 보내는 것일 뿐, 트랜잭션을 확정(Commit)하는 것은 아니다. 즉, Flush를 한다고 하더라도 여전히 트랜잭션으로 관리가 되고 있는 상태이기 때문에 트랜잭션 롤백 시 Flush 데이터도 롤백된다.
    • Persistence Unit (설계도): JPA 설정 정보의 단위이다. DB 연결 정보, 어떤 엔티티 클래스를 관리할 것인지, JPA 구현체의 특정 설정 등을 포함하는 설정파일이다. 즉, DB와 통신하기 위한 설계도 또는 명세서와 같다.
    • EntityManagerFactory (공장): Persistence Unit의 설정 정보를 바탕으로 EntityManager를 생성하는 팩토리(공장)이다. 애플리케이션 전체에서 단 하나만 생성되어 공유된다. 최초 생성 시 비용이 많이 들고(설정 읽기, 커넥션 풀 생성 등), 여러 스레드가 동시에 접근해도 안전하다(Thread-Safe).
    • EntityManager (작업자): 내부에 영속성 컨텍스트 공간을 가지고 있다. 실제 DB 작업을 처리하는 작업자이다. 엔티티를 저장(persist), 수정, 삭제(remove), 조회(find)하는 역할을 수행한다. 생성 비용이 저렴하며, DB 트랜잭션과 생명주기를 같이 한다. 여러 스레드가 동시에 접근하면 안 된다(Not Thread-Safe). 각 요청마다 개별 EntityManager를 사용해야 한다.

    JpaRepository는 내부적으로 EntityManager를 사용하고 있고 Spring Data JPA는 EntityManager를 더욱 편리하게 사용하도록 추상화했다.

    2. 트랜잭션 (Transaction)

    트랜잭션 전파 (Transaction Propagation)

    @Transactional(propagation = ...) 옵션으로 여러 가지 전략을 설정할 수 있다.

    전파 옵션 동작 방식 사용 예시 및 설명
    REQUIRED (Default) 부모 트랜잭션이 있으면 그 트랜잭션에 참여한다(합쳐진다). 부모가 없으면 새로운 트랜잭션을 생성한다. 서비스의 핵심 로직 대부분에 사용한다.
    REQUIRED_NEW 부모 트랜잭션의 존재 여부와 상관없이, 무조건 새로운 트랜잭션을 생성한다. (완전 별개) 로그 기록: 핵심 기능의 성공 여부와 관계없이, 에러 로그나 통계 데이터 등은 반드시 DB에 남아야 하기 때문이다.
    NESTED 부모 트랜잭션이 있으면, 그 안에 중첩된 트랜잭션(savePoint)을 만든다. 부모 없으면 REQUIRED처럼 동작한다. (부모에 종속적) 부분 실패 처리: 주문 처리 중 여러 개의 쿠폰을 적용하는데, 그 중 하나가 실패하더라도 전체 주문을 취소하지 않고 해당 쿠폰 적용만 롤백하고 싶을 때 유용하다.
    SUPPORTS 부모 트랜잭션이 있으면 참여한다. 부모가 없으면 트랜잭션 없이 그냥 실행된다. 주로 읽기 전용 메서드에 사용된다. 굳이 트랜잭션을 생성할 필요가 없을 때 적합하다.
    NOT_SUPPORTS 항상 트랜잭션 없이 실행된다. 트랜잭션과 전혀 관련 없는 특정 로직을 수행할 때, 불필요하게 DB 커넥션을 점유하는 것을 방지하기 위해 사용한다.
    MANDATORY 부모 트랜잭션이 반드시 있어야 한다. 없으면 예외(IllegalTransactionStateException)를 발생시킨다. 독립적으로 사용되어서는 안 되고, 반드시 어떤 트랜잭션의 일부로서만 동작해야 하는 메서드임을 강제하고 싶을 때 사용한다. (안전장치)
    NEVER 트랜잭션이 없는 상태에서만 실행될 수 있다. 부모 트랜잭션이 있으면 예외를 발생시킨다. 트랜잭션이 걸려있으면 오히려 문제가 되는 특정 메서드를 보호하고 싶을 때 사용한다.

    트랜잭션 격리 수준 (Transaction Isolation Level)

    동시에 실행되는 여러 트랜잭션들이 서로 어느 정도까지 격리되어 실행될지를 결정하는 DB 설정이다.

    • READ UNCOMMITTED: 가장 낮은 격리 수준이다. 커밋 전 데이터 읽기가 가능하다. DIRTY READ(작업 미완료 데이터 조회)가 발생하며, 데이터 정합성 문제로 사용을 권장하지 않는다.
    • READ COMMITTED: 커밋된 데이터만 읽는다. RDB에서 대부분 기본적으로 사용한다. NON-REPEATABLE READ(한 트랜잭션 내에서 같은 쿼리 두 번 실행 시 결과가 달라짐)가 발생한다. 실제 테이블 값이 아니라 Undo 영역(백업된 레코드)에서 값을 가져온다.
    • REPEATABLE READ: 자신의 트랜잭션이 생성되기 이전의 트랜잭션에서 커밋된 데이터만 읽는다. MySQL의 default 격리 수준이다. 트랜잭션 ID를 활용한다. PHANTOM READ(한 트랜잭션 내에서 없던 레코드가 갑자기 조회됨)가 발생한다.
    • SERIALIZABLE: 가장 엄격한 격리 수준이다. 데이터를 접근할 때 항상 Lock을 걸고 조회한다. 성능 문제(동시성 ↓)로 DB에서 거의 사용되지 않는다.

    * NON-REPEATABLE READ는 "기존 데이터의 값 변경" 문제이고, PHANTOM READ는 "데이터 행의 추가/삭제" 문제이다.

    3. JPA 연관관계 매핑

    연관관계란, 엔티티(테이블) 간의 관계를 의미한다.

    Primary Key vs Foreign Key

    • Primary Key (PK): 테이블에 있는 각 행(Row)을 유일하게 식별할 수 있는 고유한 값이다. 유일성, NOT NULL, 불변성을 가진다.
    • Foreign Key (FK): 한 테이블의 컬럼이 다른 테이블의 기본 키를 참조하는 것을 말한다. 테이블 간 관계를 맺어주는 연결 고리 역할을 하며, 중복과 NULL을 허용한다.

    연관관계의 주인과 매핑 어노테이션

    • 단방향 vs 양방향: 단방향은 한쪽에서만 참조, 양방향은 서로 참조하는 관계이다. 실제 DB에는 양방향이 없고 FK를 통한 단방향만 존재한다.
    • 연관관계 주인: 외래 키(FK)를 가진 테이블과 매핑된 엔티티가 주인이다. (@ManyToOne이 있는 쪽). 주인이 아닌 쪽은 mappedBy로 명시하며 조회만 가능하다.
    • @OneToOne: 1:1 매핑. 주도 테이블에 FK를 두는 것을 권장한다.
    • @ManyToOne: N:1 매핑. 연관관계의 주인이다. @JoinColumn으로 FK를 관리한다.
    • @OneToMany: 1:N 매핑. @ManyToOne이 반드시 매핑 관계에 있는 엔티티에 정의되어 있어야 사용할 수 있다.
    • @ManyToMany: N:M 매핑. 절대 사용하지 않는다. 중간 테이블이 숨겨져 생성되어(SideEffect) 유지보수가 어렵다. 연결 테이블용 엔티티를 만들어 @ManyToOne 2개로 풀어내야 한다.

    @JoinColumn 상세 옵션

    • name: FK 컬럼명 (ex. @JoinColumn(name = "team_id")).
    • unique: true로 설정하면 유니크 제약이 생성되어 사실상 1:1 제약(@OneToOne)과 같아진다 (default: false).
    • nullable: FK 컬럼의 NULL 허용 여부 (default: true). @ManyToOne(optional =)과 짝을 맞춰야 한다.
    • foreignKey: FK 제약 메타데이터.
      • ConstraintMode.CONSTRAINT: 실제 FK 제약 생성 (default).
      • ConstraintMode.NO_CONSTRAINT: 실제 FK 제약 미생성 (논리적 연관만 가짐). 최근 MSA 환경에서는 FK를 걸지 않는 경우가 늘고 있어 유용하다.

    4. 조회 전략과 최적화

    JPA 조회 전략 (FetchType)

    • FetchType.EAGER (즉시 로딩): 부모 엔티티를 조회하면, 연관된 자식 엔티티까지 한 번에 모두 조회한다. 반드시 항상 연관된 엔티티를 불러오기 때문에 선택권이 없다는 단점이 있어 쓰지 말아야 한다.
    • FetchType.LAZY (지연 로딩): 부모를 조회할 때는 자식을 가져오지 않고, 실제 데이터가 필요한 시점에 비로소 DB에서 조회한다. 필요하면 가져오는 선택권이 있다.
    프록시(Proxy) 객체
    LAZY를 사용하면 연관된 객체 자리에 프록시(가짜) 객체가 매핑되어 있다. (null이 아니다!)
    실제로 사용할 때 내부적으로 DB를 로드하여 마치 실제 객체처럼 행동하게 된다(Proxy Initialization). 
    @ManyToOne(fetch = FetchType.LAZY) // 지연 로딩 설정
@JoinColumn(name = "team_id")
private Team team;

    영속성 전이 (Cascade)

    부모 엔티티의 영속성 상태 변화를 자식 엔티티에게 그대로 전파하는 기능이다. @OneToMany(cascade = CascadeType.XXX) 형태로 사용한다.

    • ALL: 모든 Cascade 옵션을 적용한다. 부모와 자식의 생명주기가 완벽하게 일치할 때 사용한다.
    • PERSIST: 부모 저장 시 자식도 함께 저장된다.
    • REMOVE: 부모 삭제 시 자식도 함께 삭제된다.
    • MERGE: 부모 병합 시 자식도 병합된다.
    • REFRESH: 부모 다시 읽을 때 자식도 갱신된다.
    • DETACH: 부모 준영속 시 자식도 함께 준영속 상태로 만든다. (영속성 컨텍스트 관리 제외)

    고아 제거 (Orphan Removal)

    부모와 자식의 연관관계 자체가 끊어지는 것을 감지하여 자식을 삭제한다. (기본값 false)

    • 부모의 컬렉션에서 자식을 제거할 때: 트랜잭션 커밋 시점에 제거된 child에 대한 DELETE 쿼리가 실행된다.
    • 부모가 새로운 컬렉션으로 교체될 때: 기존의 모든 자식들은 참조를 잃고 고아가 되어 DELETE 된다.
    • 부모 엔티티가 삭제될 때: 자식들도 참조를 잃으므로 orphanRemoval 옵션에 의해 함께 삭제된다.

    5. JPQL과 프로젝션

    JPQL (Java Persistence Query Language)

    SQL은 테이블 기준이지만, JPQL은 엔티티 객체 기준으로 쿼리를 작성한다. 특정 DB에 종속되지 않으며, Spring Data JPA가 실행 시점에 적절한 SQL로 번역해준다. 

    // :username 파라미터 바인딩 예시
@Query("SELECT m FROM Member m WHERE m.username = :username")
List<Member> findUser(@Param("username") String username);

    프로젝션 (Projection)

    엔티티의 전체 필드가 아닌, 원하는 필드만 골라서 객체로 받아오는 기술이다. 성능 최적화와 타입 안정성을 제공한다.

    1. 인터페이스 기반 프로젝션

    조회할 필드의 getter 메서드만 선언한 인터페이스를 만들면, 구현 클래스를 자동 생성해준다.

    public interface UsernameOnly {
    String getUsername();
}

    2. 클래스(DTO) 기반 프로젝션

    조회 결과를 DTO로 직접 받아온다. new 키워드와 패키지 경로, 생성자가 필요하다.

    @Query("SELECT new com.example.dto.MemberDto(m.username, m.age) FROM Member m WHERE m.age > 20")
List<MemberDto> findMemberDto();

    6. JPA 테스트

    @DataJpaTest를 활용하여 JPA Repository 동작만 빠르고 정확하게 검증할 수 있다.

    테스트 구성 및 패턴

    • 의존성: testImplementation "com.h2database:h2" (테스트 시에만 사용하는 인메모리 DB)
    • Given-When-Then 패턴:
      • Given: 테스트 준비 (엔티티 생성 등)
      • When: 테스트 실행 (repository 메서드 호출)
      • Then: 결과 검증 (Assertions.assertThat() 사용)
    @DataJpaTest
class MemberRepositoryTest {
    
    @Autowired MemberRepository memberRepository;

    @Test
    void saveMember() {
        // Given
        Member member = new Member("userA");

        // When
        Member savedMember = memberRepository.save(member);

        // Then
        assertThat(savedMember.getId()).isNotNull();
        assertThat(savedMember.getName()).isEqualTo("userA");
    }
}
