    1. 영속성 컨텍스트 (Persistence Context)

    엔티티의 생명주기를 관리하고, 애플리케이션과 DB 사이에서 수많은 최적화 작업을 해주는 논리적인 공간(메모리 공간)이다.

    핵심 기능 3가지

    1. 1차 캐시와 동일성(Identity) 보장 영속성 컨텍스트는 내부에 Map 형태의 캐시 저장소를 가지고 있다. 조회 시 먼저 1차 캐시를 확인한다. 캐시에 동일한 ID의 엔티티 객체가 있으면, DB를 조회하지 않고 캐시의 엔티티를 즉시 반환한다. 이 덕분에 같은 트랜잭션 안에서 같은 ID로 조회한 엔티티는 항상 동일한 메모리 주소의 인스턴스임이 보장된다. 
    // 1차 캐시 예시
Member member1 = em.find(Member.class, 1L); // DB 조회
Member member2 = em.find(Member.class, 1L); // 1차 캐시에서 조회 (SQL 실행 X)

System.out.println(member1 == member2); // true (주소값 동일)
    2. 변경 감지 (Dirty Checking) 트랜잭션이 커밋되기 직전에 영속성 컨텍스트는 1차 캐시에 있는 모든 엔티티의 최초 상태(스냅샷)와 현재 상태를 비교한다. 만약 두 상태가 다르다면, JPA는 변경된 내용을 감지하고 자동으로 UPDATE SQL을 생성하여 DB에 반영한다. 
    // 변경 감지 예시
Member member = em.find(Member.class, 1L);
member.setName("NewName"); // 데이터만 변경

// em.update(member) 같은 코드가 필요 없다.
// 트랜잭션 커밋 시점에 자동으로 UPDATE 쿼리가 나간다.
    3. 쓰기 지연 (Transactional Write-Behind) memberRepository.save()가 호출되면, 해당 엔티티는 1차 캐시에 저장되고, JPA는 이 엔티티를 DB에 반영하기 위한 INSERT SQL을 저장소에 모아둔다. 트랜잭션이 커밋되는 순간(Flush), 모아둔 SQL을 한 번에 DB로 보낸다. 이를 통해 네트워크 비용을 줄이고 성능을 최적화한다.

    주요 용어 정리

    용어 설명 및 특징
    Flush 영속성 컨텍스트의 변경 내용을 DB에 동기화하는 것이다(SQL 전송).
    트랜잭션 확정(Commit)은 아니며, 롤백 시 함께 롤백된다.
    Persistence Unit
    (설계도)    		 JPA 설정 정보의 단위이다. DB 연결 정보 및 관리할 엔티티 명세가 담겨있다.
    EntityManagerFactory
    (공장)    		 애플리케이션 전체에서 단 하나만 생성되어 공유된다.
    생성 비용이 비싸며, 여러 스레드가 동시 접근해도 안전하다(Thread-Safe).
    EntityManager
    (작업자)    		 실제 DB 작업을 수행(CRUD)하는 객체이다.
    생성 비용이 저렴하지만, 스레드 간 공유가 절대 불가능하다(Not Thread-Safe).
    요청마다 생성하고 닫아야 한다.

    2. 트랜잭션 (Transaction)

    트랜잭션 전파 (Transaction Propagation)

    트랜잭션 전파란, 이미 진행 중인 트랜잭션이 있을 때, 새로운 트랜잭션이 어떻게 행동할지 결정하는 규칙이다.
    예를 들어, 부모 메서드(A)가 실행 중에 자식 메서드(B)를 호출했을 때, 자식 메서드(B)가 부모(A)의 트랜잭션에 합류할지, 아니면 자신만의 새로운 트랜잭션을 만들지를 정하는 것이다.

    @Transactional(propagation = ...) 옵션으로 설정한다.

    옵션 동작 방식 사용 예시 및 설명
    REQUIRED
    (Default)    		 부모 트랜잭션이 있으면 합류하고,
    없으면 새로 생성한다.    		 서비스의 핵심 로직 대부분에 사용한다.
    REQUIRED_NEW 부모 여부와 상관없이
    무조건 새로운 트랜잭션을 생성한다.
    (부모와 완전 별개)    		 로그 기록: 핵심 로직이 실패해 롤백되어도, 에러 로그나 통계 데이터는 반드시 DB에 남아야 할 때 사용한다.
    NESTED 부모가 있으면 중첩(SavePoint) 생성,
    없으면 새로 생성한다.    		 부분 롤백: 주문 처리 중 쿠폰 발급이 실패해도, 주문 전체를 취소하지 않고 쿠폰만 롤백하고 싶을 때 사용한다. (부모 롤백 시 자식도 롤백됨)
    SUPPORTS 부모가 있으면 합류하고,
    없으면 트랜잭션 없이 실행한다.    		 단순 읽기 전용 메서드처럼, 굳이 트랜잭션을 만들 필요는 없지만 있으면 쓰는 경우에 적합하다.
    NOT_SUPPORTS 항상 트랜잭션 없이 실행된다. 단순 메모리 연산이나 외부 API 호출 등 DB 트랜잭션과 묶이면 안 되는(커넥션 점유 방지) 로직에 사용한다.
    MANDATORY 부모 트랜잭션이 반드시 있어야 한다.
    없으면 예외 발생.    		 독립적으로 실행되면 안 되고, 반드시 상위 로직의 일부로만 동작해야 함을 강제할 때 사용한다. (안전장치)
    NEVER 트랜잭션이 없어야 한다.
    있으면 예외 발생.    		 트랜잭션이 적용되면 안 되는 로직을 보호할 때 사용한다.

    트랜잭션 격리 수준 (Isolation Level)

    동시에 실행되는 여러 트랜잭션들이 서로 데이터를 어디까지 볼 수 있는지(격리성)를 결정하는 설정이다.

    수준 (Level) 특징 발생 가능한 문제
    READ UNCOMMITTED
    (Lv.0)    		 커밋되지 않은 데이터도 읽을 수 있다.
    데이터 정합성 문제로 사용을 권장하지 않는다.    		 DIRTY READ
    (작업 중인 미완성 데이터를 읽음)
    READ COMMITTED
    (Lv.1)    		 커밋된 데이터만 읽는다. (RDB 기본값)
    Undo 영역(백업)의 데이터를 읽는다.    		 NON-REPEATABLE READ
    (한 트랜잭션 내에서 같은 조회를 반복했을 때 값이 달라짐)
    REPEATABLE READ
    (Lv.2)    		 트랜잭션 시작 시점의 데이터만 읽는다. (MySQL 기본값)
    자신보다 이후에 생긴 트랜잭션의 변경은 무시한다.    		 PHANTOM READ
    (한 트랜잭션 내에서 없던 데이터가 갑자기 조회됨 - Insert/Delete)
    SERIALIZABLE
    (Lv.3)    		 가장 엄격하다. 읽기 작업에도 Lock을 건다.
    성능이 매우 떨어져 거의 사용하지 않는다.    		 없음 (동시성이 매우 낮아짐)

    * 참고: NON-REPEATABLE READ는 "기존 데이터의 값 수정" 문제이고, PHANTOM READ는 "데이터 행의 추가/삭제" 문제이다.

    3. JPA 연관관계 매핑

    기본 키(PK)와 외래 키(FK)

    구분 설명 특징
    Primary Key (PK) 테이블의 각 행을 식별하는 고유한 값이다. 유일성, NOT NULL, 불변성 보장
    Foreign Key (FK) 다른 테이블의 PK를 참조하는 값이다.
    테이블 간 연결 고리 역할을 한다.    		 중복 허용, NULL 허용

    연관관계의 주인과 매핑 어노테이션

    • 연관관계 주인: 외래 키(FK)를 실질적으로 관리(등록, 수정)하는 엔티티이다. (보통 @ManyToOne 쪽)
    • 주인이 아닌 쪽: mappedBy 속성을 사용해 주인을 명시하며, 조회만 가능하다.
    어노테이션 관계 설명 및 주의사항
    @ManyToOne N : 1 연관관계의 주인이다. @JoinColumn으로 FK를 관리한다. 가장 많이 사용한다.
    @OneToMany 1 : N @ManyToOne이 있는 곳에 사용하며, mappedBy로 연결한다.
    @OneToOne 1 : 1 주로 사용하는(주도적인) 테이블에 FK를 두는 것을 권장한다.
    @ManyToMany N : M 절대 사용 금지. 중간 테이블이 숨겨져 생성되므로(SideEffect) 관리가 불가능하다. 연결 엔티티를 만들어 @ManyToOne 2개로 풀어야 한다.

    @JoinColumn 상세 옵션

    • name: 매핑할 FK 컬럼명을 지정한다. (예: "team_id")
    • unique: true 설정 시 유니크 제약조건이 생성되어 사실상 1:1 관계가 된다. (default: false)
    • nullable: FK의 NULL 허용 여부이다. (default: true)
    • foreignKey: FK 제약 조건을 상세 제어한다.
      • ConstraintMode.NO_CONSTRAINT: 실제 DB에 FK 제약조건은 생성하지 않고, 객체 간 논리적 연관만 맺는다. (MSA 환경이나 레거시 DB 연결 시 유용)

    4. 조회 전략과 최적화

    JPA 조회 전략 (FetchType)

    • EAGER (즉시 로딩): 부모를 조회하면 자식까지 한 번에 다 쿼리해온다. 불필요한 조인이 발생하므로 실무에서는 사용하지 않는다.
    • LAZY (지연 로딩): 실제 자식 데이터가 필요한 시점에 쿼리를 날린다. 무조건 이것을 사용한다.
    프록시(Proxy) 객체란?
    FetchType.LAZY를 사용하면 연관된 객체 자리에 프록시(가짜) 객체가 들어있다.
    이는 null이 아니며, 실제 데이터를 사용하는 시점(.getName() 등)에 DB를 조회하여 진짜 데이터로 채워진다(초기화). 
    @ManyToOne(fetch = FetchType.LAZY) // 지연 로딩 설정 (필수)
@JoinColumn(name = "team_id")
private Team team;

    영속성 전이 (Cascade)

    부모 엔티티의 상태 변화(저장, 삭제 등)를 자식 엔티티에게도 전파하는 옵션이다.
    @OneToMany(cascade = CascadeType.ALL) 형태로 사용한다.

    옵션 설명
    ALL 모든 변경 사항을 전파한다. (생명주기가 같을 때 사용)
    PERSIST 부모 저장 시 자식도 함께 저장한다.
    REMOVE 부모 삭제 시 자식도 함께 삭제한다.
    MERGE 부모 병합 시 자식도 병합한다.
    REFRESH 부모 다시 읽을 때 자식도 갱신한다.
    DETACH 부모 준영속 시 자식도 준영속 상태로 만든다.

    고아 제거 (Orphan Removal)

    부모와 연관관계가 끊어진 자식 엔티티를 자동으로 삭제한다. (orphanRemoval = true)

    • 컬렉션에서 제거 시: parent.getChildren().remove(0) 처럼 리스트에서 빼면 DB에서 DELETE 된다.
    • 부모 삭제 시: 부모가 지워지면 자식들은 고아가 되므로 함께 삭제된다. (Cascade.REMOVE와 유사)

    5. JPQL과 프로젝션

    JPQL (Java Persistence Query Language)

    SQL은 테이블을 대상으로 하지만, JPQL은 엔티티 객체를 대상으로 쿼리를 작성한다. 특정 DB에 종속되지 않으며, 실행 시점에 적절한 SQL로 변환된다. 

    // :username 파라미터 바인딩 예시
@Query("SELECT m FROM Member m WHERE m.username = :username")
List<Member> findUser(@Param("username") String username);

    프로젝션 (Projection)

    엔티티의 모든 필드가 아니라, 필요한 필드만 골라서 조회하는 기술이다. 성능과 가독성을 높인다.

    1. 인터페이스 기반

    Getter 메서드만 선언하면 Spring Data JPA가 구현체를 만들어준다.

    public interface UsernameOnly {
    String getUsername();
}

    2. 클래스(DTO) 기반

    JPQL에서 new 키워드를 사용해 DTO 생성자를 호출하여 반환받는다.

    @Query("SELECT new com.example.dto.MemberDto(m.username, m.age) FROM Member m")
List<MemberDto> findMemberDto();

    6. JPA 테스트

    @DataJpaTest를 사용하면 JPA 관련 설정만 로드하여 Repository 계층을 빠르고 정확하게 검증할 수 있다. 기본적으로 @Transactional이 포함되어 있어 테스트 종료 후 자동 롤백된다.

    테스트 구성 및 패턴

    • 의존성: testImplementation "com.h2database:h2" (테스트 전용 인메모리 DB)
    • Given-When-Then 패턴:
      • Given: 테스트 준비 (엔티티 생성 등)
      • When: 테스트 실행 (repository 메서드 호출)
      • Then: 결과 검증 (Assertions.assertThat() 사용)
    @DataJpaTest
class MemberRepositoryTest {
    
    @Autowired MemberRepository memberRepository;

    @Test
    void saveMember() {
        // Given
        Member member = new Member("userA");

        // When
        Member savedMember = memberRepository.save(member);

        // Then
        assertThat(savedMember.getId()).isNotNull();
        assertThat(savedMember.getName()).isEqualTo("userA");
    }
}
