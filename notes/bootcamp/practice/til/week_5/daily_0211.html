    43일차 개요

    오늘 목표

    • 일정 및 댓글 시스템 개발 중 발생한 페이징 조회 이슈(TotalElements 오류)를 해결한다.
    • N+1 문제를 방지하는 JPQL 설계와 DTO 반환 원칙을 확립한다.
    • 실무에서 권장하는 API 설계 원칙(목록 vs 상세)과 아키텍처 흐름을 정리한다.

    1. 트러블 슈팅: 복잡한 쿼리의 페이징

    [Issue] totalElements가 1로 나오는 현상

    상황: @Query 내부에 GROUP BY가 포함된 복잡한 조인 쿼리를 작성하여 페이징(Page) 객체를 반환받으려 했다.
    증상: 실제 데이터는 여러 개인데, Page.getTotalElements() 값이 1로 나오거나 계산 오류가 발생했다.

    원인 분석 Spring Data JPA는 페이징을 위해 자동으로 COUNT 쿼리를 생성한다.
    하지만 GROUP BY가 포함된 쿼리의 경우, JPA가 생성한 자동 카운트 쿼리가 그룹화된 결과의 첫 행만 인식하거나, 전체 그룹의 개수를 제대로 세지 못하는 문제가 발생한다.

    [Solved] countQuery 명시적 분리

    countQuery 속성을 사용하여 전체 데이터 개수만 조회하는 단순 쿼리를 명시적으로 분리해주어야 한다. 

    @Query(
    value = "SELECT s FROM Schedule s LEFT JOIN s.comments c GROUP BY s.id",
    countQuery = "SELECT COUNT(s) FROM Schedule s" // 카운트용 쿼리 별도 지정
)
Page<Schedule> findAllWithCommentCount(Pageable pageable);

    2. Pageable 동작 원리와 주의점

    Pageable 핵심 개념

    개념 설명
    Offset/Limit 애플리케이션 메모리에서 거르는 것이 아니라, DB 레벨에서 쿼리(Limit)를 통해 필요한 만큼만 가져오는 것이다.
    0-based Index 페이지 번호는 0부터 시작한다. (클라이언트가 1페이지를 요청하면 서버는 page=0으로 처리해야 한다.)
    @PageableDefault 컨트롤러 파라미터에서 기본 페이지 크기(size)나 정렬(sort) 조건을 손쉽게 설정할 수 있다. 
    // 컨트롤러 예시: 기본값으로 수정일 기준 내림차순, 10개씩 조회
@GetMapping
public ResponseEntity<Page<ReadScheduleResponse>> getSchedules(
    @PageableDefault(size = 10, sort = "modifiedAt", direction = Sort.Direction.DESC) Pageable pageable
) { ... }

    3. 설계 원칙과 성능 최적화

    N+1 문제의 함정과 해결

    상황: 일정 목록을 조회하면서, 각 일정에 달린 댓글 개수를 세기 위해 루프 안에서 Repository를 호출하면 쿼리가 폭발적으로 늘어난다(N+1 문제).

    해결 전략 JPQL의 JOIN과 집계 함수(COUNT)를 사용하여 한 번의 쿼리로 데이터를 가져온다.
    만약 연관된 엔티티를 리스트로 가져와야 한다면 @BatchSize 설정을 고려한다. 
    // Bad: 루프 돌며 조회 (N+1 발생)
for (Schedule s : schedules) {
    int count = commentRepository.countByScheduleId(s.getId()); // 쿼리 계속 발생
}

// Good: JPQL로 한 번에 조회 (Fetch Join 또는 DTO Projection)
@Query("SELECT new com.example.dto.ScheduleDto(s.title, COUNT(c)) " +
       "FROM Schedule s LEFT JOIN s.comments c GROUP BY s.id")
Page<ScheduleDto> findSchedulesWithCount(Pageable pageable);

    엔티티 노출 금지 (DTO 사용)

    컨트롤러에서 엔티티(Entity)를 직접 반환하면 안 된다. 스펙 변경 시 API가 깨질 수 있고, 민감한 정보가 노출될 수 있다. 반드시 DTO(Record)로 변환하여 반환한다. 

    // Service 계층에서의 변환 예시
public Page<ReadScheduleResponse> getSchedules(Pageable pageable) {
    Page<Schedule> schedules = scheduleRepository.findAll(pageable);
    
    // map() 함수를 이용해 Entity -> DTO 변환
    return schedules.map(ReadScheduleResponse::from); 
}

    4. 향후 설계 시 고려할 원칙

    페이징의 주인공 설정

    • 목록 조회 API: 메인 데이터(일정)를 기준으로 페이징하고, 연관 데이터(댓글)는 개수만 보여준다.
    • 상세 조회 API: 특정 일정을 선택했을 때, 해당 일정의 댓글을 별도로 페이징하여 보여준다.
    • 일정과 댓글을 한 번에 모두 페이징해서 가져오는 것은 성능과 UX 측면에서 좋지 않다.

    단방향 vs 양방향 관계

    • 현재 설계: 단방향 관계를 유지하여 엔티티 간 결합도를 낮추는 것이 권장된다.
    • 조회 전략: 단방향이라 하더라도 JPQL에서 LEFT JOIN을 활용하면 연관관계가 없는 것처럼 보이는 데이터도 효율적으로 묶어서 조회할 수 있다.

    최종 추천 아키텍처 (Best Practice)

    계층 역할 비고
    Controller @PageableDefault로 페이징 조건 수신 Page<DTO> 반환
    Service Repository 호출 및 비즈니스 로직 처리 엔티티 → DTO 변환 시점
    Repository @Query와 countQuery를 활용한 데이터 추출 성능 최적화의 핵심
    DTO Record 타입을 활용해 불변 객체로 데이터 전달 static from() 메서드 활용

    오늘의 회고

    오늘 느낀 점

    • countQuery를 분리하지 않으면 페이징 계산이 틀릴 수 있다는 점을 트러블 슈팅을 통해 깊이 이해했다.
    • 페이징은 단순히 잘라오는 것이 아니라, "무엇을 기준으로, 어떻게 성능을 잃지 않고 가져올 것인가"에 대한 고민이 필요하다.
    • 다음 단계로는 Pageable을 활용하면서 동적 쿼리(검색 조건 등)가 들어가는 기능을 구현해 볼 예정이다.
