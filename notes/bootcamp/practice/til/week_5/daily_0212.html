    1. 보안 설정 관리 — application.properties 숨기기

    application.properties 파일에는 데이터베이스 접속 정보, API 키, Secret Key 등 민감한 정보가 포함되어 있다. 이러한 정보가 Git 저장소에 노출되면 보안 사고로 이어질 수 있다. 따라서 민감 정보를 외부화하고 버전 관리에서 제외해야 한다.

    .gitignore에 등록하기

    가장 기본적인 방법은 .gitignore 파일에 application.properties를 추가하는 것이다.

    # .gitignore 파일
application.properties
application-*.properties
*.yml
!application-example.properties
    주의사항 이미 Git에 커밋된 파일은 .gitignore에 추가해도 추적이 계속된다. 이미 추적 중인 파일을 제거하려면 다음 명령어를 실행해야 한다. 
    git rm --cached src/main/resources/application.properties
git commit -m "Remove sensitive config file"

    환경 변수 사용하기

    Spring Boot는 환경 변수(Environment Variables)를 통해 설정값을 주입받을 수 있다. application.properties에 하드코딩하는 대신, ${} 문법으로 환경 변수를 참조한다. 

    # application.properties
spring.datasource.url=${DB_URL}
spring.datasource.username=${DB_USERNAME}
spring.datasource.password=${DB_PASSWORD}
jwt.secret=${JWT_SECRET}

    IntelliJ에서는 Run Configuration의 Environment Variables 섹션에서 설정할 수 있다.

    DB_URL=jdbc:mysql://localhost:3306/mydb
DB_USERNAME=root
DB_PASSWORD=1234
JWT_SECRET=mySecretKey123456789

    예시 설정 파일 제공

    팀 협업 시 다른 개발자가 프로젝트를 클론했을 때 설정 방법을 알 수 있도록, application-example.properties 파일을 제공하는 것이 좋다. 

    # application-example.properties (Git에 포함)
spring.datasource.url=jdbc:mysql://localhost:3306/your_database
spring.datasource.username=your_username
spring.datasource.password=your_password
jwt.secret=your_jwt_secret_key_at_least_256_bits

    다른 개발자는 이 파일을 복사하여 실제 값을 채운 후 사용하면 된다.

    2. 계층별 책임 분리 — 엔티티에서 검증하지 말기

    Spring Boot 애플리케이션은 계층(Layer)으로 구성되며, 각 계층은 명확한 책임을 가져야 한다. 엔티티(Entity)는 데이터베이스 테이블과 매핑되는 순수한 데이터 객체이며, 비즈니스 로직이나 검증 로직을 포함해서는 안 된다.

    잘못된 예시 — 엔티티에 검증 로직 포함 

    @Entity
public class Member {
    @Id @GeneratedValue
    private Long id;
    
    private String email;
    private String phoneNumber;
    
    // ❌ 나쁜 예: 엔티티에 검증 로직 추가
    public void validateEmail() {
        if (email == null || !email.contains("@")) {
            throw new IllegalArgumentException("유효하지 않은 이메일입니다.");
        }
    }
    
    public void validatePhoneNumber() {
        if (phoneNumber == null || phoneNumber.length() < 10) {
            throw new IllegalArgumentException("유효하지 않은 전화번호입니다.");
        }
    }
}
    왜 엔티티에서 검증하면 안 되는가? 1. 단일 책임 원칙(SRP) 위반: 엔티티는 데이터 저장 책임만 가져야 한다. 검증은 비즈니스 로직이므로 서비스 계층의 책임이다.
    2. 재사용성 저하: 검증 규칙이 엔티티에 고정되면, 다른 상황(회원가입 vs 정보수정)에서 다른 검증이 필요할 때 유연하게 대응할 수 없다.
    3. 테스트 어려움: 엔티티 테스트와 비즈니스 로직 테스트가 섞여서 복잡해진다.

    올바른 예시 — 서비스 계층에서 검증 

    // 엔티티 — 순수한 데이터 객체
@Entity
public class Member {
    @Id @GeneratedValue
    private Long id;
    
    private String email;
    private String phoneNumber;
    
    // Getter, Setter만 존재
}
    // 서비스 계층 — 비즈니스 로직과 검증 담당
@Service
@RequiredArgsConstructor
public class MemberService {
    
    private final MemberRepository memberRepository;
    
    public void registerMember(MemberRegisterRequest request) {
        // 검증 로직
        validateEmail(request.getEmail());
        validatePhoneNumber(request.getPhoneNumber());
        
        // 중복 확인
        if (memberRepository.existsByEmail(request.getEmail())) {
            throw new DuplicateMemberException("이미 존재하는 이메일입니다.");
        }
        
        // 엔티티 생성 및 저장
        Member member = Member.builder()
            .email(request.getEmail())
            .phoneNumber(request.getPhoneNumber())
            .build();
            
        memberRepository.save(member);
    }
    
    private void validateEmail(String email) {
        if (email == null || !email.matches("^[A-Za-z0-9+_.-]+@(.+)$")) {
            throw new InvalidEmailException("유효하지 않은 이메일 형식입니다.");
        }
    }
    
    private void validatePhoneNumber(String phoneNumber) {
        if (phoneNumber == null || !phoneNumber.matches("^01[0-9]{8,9}$")) {
            throw new InvalidPhoneNumberException("유효하지 않은 전화번호 형식입니다.");
        }
    }
}

    Nullable 제약 조건 설정 주의사항

    인스타그램과 같이 이메일 또는 휴대폰 번호 중 하나로 회원가입할 수 있는 시스템에서는, 두 필드 모두에 NOT NULL 제약 조건을 걸면 안 된다. 

    @Entity
public class Member {
    @Id @GeneratedValue
    private Long id;
    
    // ✅ nullable = true (기본값)
    // 이메일 또는 전화번호 중 하나만 있어도 가입 가능
    @Column(unique = true)
    private String email;
    
    @Column(unique = true)
    private String phoneNumber;
}
    // 서비스 계층에서 검증
public void registerMember(MemberRegisterRequest request) {
    // 둘 중 하나는 반드시 있어야 함
    if (request.getEmail() == null && request.getPhoneNumber() == null) {
        throw new InvalidInputException("이메일 또는 전화번호 중 하나는 필수입니다.");
    }
    
    // 이메일이 있으면 검증
    if (request.getEmail() != null) {
        validateEmail(request.getEmail());
        if (memberRepository.existsByEmail(request.getEmail())) {
            throw new DuplicateMemberException("이미 존재하는 이메일입니다.");
        }
    }
    
    // 전화번호가 있으면 검증
    if (request.getPhoneNumber() != null) {
        validatePhoneNumber(request.getPhoneNumber());
        if (memberRepository.existsByPhoneNumber(request.getPhoneNumber())) {
            throw new DuplicateMemberException("이미 존재하는 전화번호입니다.");
        }
    }
    
    // 엔티티 생성 및 저장
    Member member = Member.builder()
        .email(request.getEmail())
        .phoneNumber(request.getPhoneNumber())
        .build();
        
    memberRepository.save(member);
}

    3. DTO와 엔티티 변환 패턴

    DTO(Data Transfer Object)는 계층 간 데이터 전송을 담당하는 객체이다. 클라이언트와 서버 간 통신, 서비스 계층과 컨트롤러 계층 간 통신에 사용된다. 엔티티를 직접 외부에 노출하면 보안 문제와 강한 결합이 발생하므로, 반드시 DTO를 사용해야 한다.

    DTO → 엔티티 변환

    클라이언트로부터 받은 DTO를 엔티티로 변환하여 데이터베이스에 저장한다.

    // 요청 DTO
public record MemberRegisterRequest(
    String email,
    String phoneNumber,
    String name
) {
    // DTO → 엔티티 변환 메서드 (정적 팩토리 메서드)
    public Member toEntity() {
        return Member.builder()
            .email(this.email)
            .phoneNumber(this.phoneNumber)
            .name(this.name)
            .build();
    }
}
    // 서비스 계층에서 사용
@Service
@RequiredArgsConstructor
public class MemberService {
    
    private final MemberRepository memberRepository;
    
    public Long registerMember(MemberRegisterRequest request) {
        // 검증 로직 생략
        
        // DTO를 엔티티로 변환
        Member member = request.toEntity();
        
        // 저장
        Member savedMember = memberRepository.save(member);
        
        return savedMember.getId();
    }
}

    엔티티 → DTO 변환

    데이터베이스에서 조회한 엔티티를 DTO로 변환하여 클라이언트에 응답한다.

    // 응답 DTO
public record MemberResponse(
    Long id,
    String email,
    String phoneNumber,
    String name
) {
    // 엔티티 → DTO 변환 (정적 팩토리 메서드)
    public static MemberResponse from(Member member) {
        return new MemberResponse(
            member.getId(),
            member.getEmail(),
            member.getPhoneNumber(),
            member.getName()
        );
    }
}
    // 서비스 계층에서 사용
@Service
@RequiredArgsConstructor
public class MemberService {
    
    private final MemberRepository memberRepository;
    
    public MemberResponse getMember(Long id) {
        Member member = memberRepository.findById(id)
            .orElseThrow(() -> new MemberNotFoundException("회원을 찾을 수 없습니다."));
        
        // 엔티티를 DTO로 변환
        return MemberResponse.from(member);
    }
}
    왜 DTO와 엔티티를 분리하는가? 1. 보안: 엔티티에는 비밀번호, 권한 등 민감한 정보가 있을 수 있다. DTO는 필요한 정보만 선택적으로 노출한다.
    2. 유연성: API 스펙이 변경되어도 엔티티 구조는 유지할 수 있다. DTO만 수정하면 된다.
    3. 순환 참조 방지: 엔티티 간 양방향 관계가 있을 때 JSON 직렬화 시 무한 루프가 발생한다. DTO는 이를 방지한다.
    4. 단일 책임: 엔티티는 데이터 저장, DTO는 데이터 전송이라는 명확한 책임을 가진다.

    4. 빌더 패턴 (Builder Pattern)

    빌더 패턴은 복잡한 객체를 단계적으로 생성할 수 있게 해주는 디자인 패턴이다. 생성자의 파라미터가 많을 때 가독성을 높이고, 선택적 파라미터를 우아하게 처리할 수 있다. Lombok의 @Builder 어노테이션을 사용하면 자동으로 빌더 클래스가 생성된다.

    빌더 패턴의 장점

    장점 설명
    가독성 향상 어떤 값이 어느 필드에 들어가는지 명확하게 알 수 있다. 파라미터 순서를 외울 필요가 없다.
    선택적 파라미터 필수 필드만 설정하고, 선택 필드는 생략할 수 있다. 생성자 오버로딩이 필요 없다.
    불변 객체 생성 모든 필드를 final로 선언하고 빌더를 통해 한 번에 초기화하면, 변경 불가능한 안전한 객체를 만들 수 있다.
    유지보수성 새로운 필드가 추가되어도 기존 코드를 수정할 필요가 없다. 빌더 체이닝만 추가하면 된다.

    빌더 패턴의 단점과 해결 방법

    빌더 패턴의 가장 큰 단점 값을 입력하지 않은 필드는 자동으로 null이 들어간다. 일반 생성자를 사용하면 컴파일러가 파라미터 누락을 검사하지만, 빌더는 선택적이므로 필수 필드를 빼먹어도 컴파일 에러가 발생하지 않는다. 이로 인해 런타임에 NullPointerException이 발생할 수 있다. 
    // ❌ 문제 상황: 필수 필드(email)를 빼먹어도 컴파일 에러가 없음
@Entity
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class Member {
    @Id @GeneratedValue
    private Long id;
    
    private String email;    // 필수 필드인데 null이 들어갈 수 있음
    private String name;
    private String phoneNumber;
}

// 사용 시
Member member = Member.builder()
    .name("홍길동")
    // email을 깜빡했는데 컴파일 에러가 없음!
    .build();

memberRepository.save(member); // email = null로 저장됨

    해결 방법 1 — @NonNull 어노테이션 사용

    Lombok의 @NonNull 어노테이션을 필수 필드에 추가하면, 빌더가 해당 필드에 null을 허용하지 않는다.

    @Entity
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class Member {
    @Id @GeneratedValue
    private Long id;
    
    @NonNull  // ✅ 필수 필드 표시
    private String email;
    
    private String name;
    private String phoneNumber;
}

// 사용 시
Member member = Member.builder()
    .name("홍길동")
    // .email("test@example.com")  // 누락 시
    .build();  // ❌ 런타임에 NullPointerException 발생

    해결 방법 2 — 커스텀 생성자에 @Builder 적용 (권장)

    필수 필드만 받는 private 생성자를 만들고, 그 위에 @Builder를 적용하면 컴파일 시점에 필수 필드 누락을 방지할 수 있다. 

    @Entity
@NoArgsConstructor(access = AccessLevel.PROTECTED)  // JPA를 위한 기본 생성자
public class Member {
    @Id @GeneratedValue
    private Long id;
    
    private String email;
    private String name;
    private String phoneNumber;
    
    // ✅ 필수 필드만 받는 생성자에 @Builder 적용
    @Builder
    private Member(String email, String name) {
        this.email = email;
        this.name = name;
    }
    
    // phoneNumber는 선택적으로 setter로 설정
    public void setPhoneNumber(String phoneNumber) {
        this.phoneNumber = phoneNumber;
    }
}
    // 사용 시
Member member = Member.builder()
    .email("test@example.com")  // 필수
    .name("홍길동")              // 필수
    .build();  // ✅ email을 누락하면 컴파일 에러 발생!

member.setPhoneNumber("01012345678");  // 선택 사항
memberRepository.save(member);

    5. 정적 팩토리 메서드 (Static Factory Method)

    정적 팩토리 메서드는 객체 생성을 캡슐화하는 디자인 패턴이다. 생성자 대신 의미 있는 이름을 가진 정적 메서드를 통해 객체를 생성함으로써, 코드의 가독성과 유연성을 높일 수 있다.

    명명 규칙 (Naming Convention)

    메서드명 파라미터 개수 의미 및 사용 예시
    from 1개 형변환을 의미한다. 하나의 파라미터를 받아 해당 타입의 인스턴스를 반환한다.
    예: MemberResponse.from(member) — Member 엔티티를 MemberResponse DTO로 변환
    of 2개 이상 여러 파라미터를 집계하여 인스턴스를 생성한다.
    예: Member.of(email, name, phoneNumber) — 여러 값으로 Member 생성
    valueOf 1개 from과 유사하지만, 더 자세한 형변환을 의미한다.
    예: Integer.valueOf("123") — 문자열을 Integer로 변환
    getInstance
    / newInstance    		 0개 또는 여러 개 싱글톤 패턴이나 팩토리 메서드에서 사용된다.
    getInstance는 동일 인스턴스 반환, newInstance는 매번 새 인스턴스 반환

    실전 예시 — from과 of 활용 

    // DTO에서 from 사용 (엔티티 → DTO 변환)
public record MemberResponse(
    Long id,
    String email,
    String name
) {
    // 파라미터 1개: from 사용
    public static MemberResponse from(Member member) {
        return new MemberResponse(
            member.getId(),
            member.getEmail(),
            member.getName()
        );
    }
}
    // 엔티티에서 of 사용 (여러 파라미터로 생성)
@Entity
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class Member {
    @Id @GeneratedValue
    private Long id;
    
    private String email;
    private String name;
    private String phoneNumber;
    
    private Member(String email, String name, String phoneNumber) {
        this.email = email;
        this.name = name;
        this.phoneNumber = phoneNumber;
    }
    
    // 파라미터 2개 이상: of 사용
    public static Member of(String email, String name, String phoneNumber) {
        return new Member(email, name, phoneNumber);
    }
}
    // 사용 예시
// 엔티티 생성
Member member = Member.of("test@example.com", "홍길동", "01012345678");
memberRepository.save(member);

// DTO로 변환
MemberResponse response = MemberResponse.from(member);
    정적 팩토리 메서드의 장점 1. 이름을 가질 수 있다: from, of 같은 의미 있는 이름으로 생성 의도를 명확히 표현한다.
    2. 호출될 때마다 인스턴스를 새로 생성하지 않아도 된다: 캐싱이나 싱글톤 패턴 구현에 유리하다.
    3. 반환 타입의 하위 타입 객체를 반환할 수 있다: 인터페이스 기반 설계에서 구현체를 숨길 수 있다.
    4. 입력 파라미터에 따라 다른 클래스의 객체를 반환할 수 있다: 팩토리 패턴의 유연성을 제공한다.

    6. 성능 최적화 — existsBy 메서드

    데이터베이스에서 특정 조건을 만족하는 데이터의 존재 여부만 확인하고 싶을 때, count() 메서드를 사용하면 불필요한 연산이 발생한다. Spring Data JPA의 existsBy... 메서드는 내부적으로 SELECT EXISTS 쿼리를 사용하여 데이터를 하나만 찾으면 즉시 true를 반환하므로 훨씬 효율적이다.

    count vs existsBy 비교

    방법 쿼리 동작 방식 및 성능
    count() 사용 SELECT COUNT(*) FROM member WHERE email = ? 모든 데이터를 끝까지 세야 한다.
    데이터가 100만 건이면 100만 건을 모두 읽어야 결과를 반환한다.
    존재 여부만 확인하는 것이 목적이라면 불필요한 연산이다.
    existsBy 사용 SELECT EXISTS(SELECT 1 FROM member WHERE email = ? LIMIT 1) 조건에 맞는 데이터를 하나만 찾으면 즉시 true를 반환하고 중단한다.
    데이터가 100만 건이어도 첫 번째 데이터만 찾으면 끝난다.
    대규모 데이터에서 압도적으로 빠르다.

    실전 예시 

    // Repository 인터페이스
public interface MemberRepository extends JpaRepository {
    
    // ✅ 권장: existsBy 메서드 사용
    boolean existsByEmail(String email);
    boolean existsByPhoneNumber(String phoneNumber);
    
    // ❌ 비효율: count 사용
    long countByEmail(String email);
}
    // 서비스 계층에서 사용
@Service
@RequiredArgsConstructor
public class MemberService {
    
    private final MemberRepository memberRepository;
    
    public void registerMember(MemberRegisterRequest request) {
        // ✅ existsBy 사용: 데이터 하나만 찾으면 즉시 반환
        if (memberRepository.existsByEmail(request.getEmail())) {
            throw new DuplicateMemberException("이미 존재하는 이메일입니다.");
        }
        
        // ❌ count 사용: 모든 데이터를 세야 함 (비효율)
        if (memberRepository.countByEmail(request.getEmail()) > 0) {
            throw new DuplicateMemberException("이미 존재하는 이메일입니다.");
        }
        
        // 회원 저장 로직
        Member member = request.toEntity();
        memberRepository.save(member);
    }
}
    대규모 트래픽 환경에서의 성능 차이 회원 테이블에 1,000,000건의 데이터가 있고, 중복 체크를 한다고 가정하자.

    count 방식: 1,000,000건을 모두 읽어야 하므로 수백 밀리초 소요
    existsBy 방식: 인덱스를 통해 첫 번째 매칭 데이터만 찾으면 끝나므로 수 밀리초 소요

    대규모 트래픽 설계는 이런 작은 차이에서 시작된다. 하루에 10만 번의 회원가입 요청이 있다면, existsBy를 사용함으로써 수십 초의 응답 시간을 절약할 수 있다.

    7. 테스트 전략 — 단위 테스트와 TDD

    단위 테스트 (Unit Test)란?

    단위 테스트는 애플리케이션의 가장 작은 단위(주로 메서드)를 독립적으로 테스트하는 것이다. 외부 의존성(데이터베이스, 네트워크 등)을 Mock이나 Stub으로 대체하여, 순수하게 해당 로직만을 검증한다.

    단위 테스트의 장점

    장점 설명
    빠른 피드백 서버를 띄우지 않고 순수 로직만 테스트하므로 실행 속도가 매우 빠르다. 코드를 수정한 후 몇 초 안에 결과를 확인할 수 있다.
    리팩토링의 용기 코드를 개선하거나 구조를 변경했을 때, 테스트를 실행하여 다른 부분이 망가지지 않았는지 즉시 확인할 수 있다. 안전하게 코드를 진화시킬 수 있다.
    살아있는 문서 테스트 코드를 읽으면 해당 기능이 어떻게 동작해야 하는지, 어떤 예외 상황을 처리하는지 한눈에 알 수 있다. 주석보다 정확하고 항상 최신 상태를 유지한다.
    버그 조기 발견 배포 전에 잠재적인 버그를 발견하여 수정할 수 있다. 프로덕션 환경에서 발생할 치명적인 오류를 사전에 차단한다.

    TDD (Test Driven Development)

    TDD는 테스트를 먼저 작성하고, 그 테스트를 통과하는 최소한의 코드를 구현하는 개발 방법론이다. 검증되지 않은 코드는 서버에 올리지 않는다는 철학을 가지고 있다.

    TDD 사이클 — Red-Green-Refactor 1. Red (실패하는 테스트 작성): 아직 구현되지 않은 기능에 대한 테스트를 먼저 작성한다. 당연히 실패한다.
    2. Green (테스트를 통과하는 최소 코드 작성): 테스트를 통과하는 가장 간단한 코드를 작성한다. 완벽하지 않아도 된다.
    3. Refactor (코드 개선): 중복 제거, 구조 개선 등 코드를 리팩토링한다. 테스트가 여전히 통과하는지 확인한다. 
    // 1. Red: 실패하는 테스트 먼저 작성
@Test
void 이메일_중복_회원가입_실패() {
    // Given
    MemberRegisterRequest request = new MemberRegisterRequest(
        "test@example.com", 
        "01012345678", 
        "홍길동"
    );
    memberService.registerMember(request);
    
    // When & Then
    assertThatThrownBy(() -> memberService.registerMember(request))
        .isInstanceOf(DuplicateMemberException.class)
        .hasMessage("이미 존재하는 이메일입니다.");
}

// 2. Green: 테스트를 통과하는 코드 작성
public void registerMember(MemberRegisterRequest request) {
    if (memberRepository.existsByEmail(request.getEmail())) {
        throw new DuplicateMemberException("이미 존재하는 이메일입니다.");
    }
    Member member = request.toEntity();
    memberRepository.save(member);
}

// 3. Refactor: 검증 로직을 별도 메서드로 분리
private void validateDuplicateEmail(String email) {
    if (memberRepository.existsByEmail(email)) {
        throw new DuplicateMemberException("이미 존재하는 이메일입니다.");
    }
}

    GWT (Given-When-Then) 패턴

    GWT는 테스트 코드를 가독성 있게 구조화하는 패턴이다. BDD(Behavior Driven Development)에서 유래했으며, 테스트의 의도를 명확하게 표현한다.

    단계 역할 예시
    Given
    (준비)    		 테스트를 위해 필요한 데이터나 상황을 준비한다. Member member = new Member("test@example.com");
    memberRepository.save(member);
    When
    (실행)    		 실제로 테스트하고 싶은 동작을 실행한다. MemberResponse response = memberService.getMember(1L);
    Then
    (검증)    		 결과가 예상과 일치하는지 검증한다. assertThat(response.email()).isEqualTo("test@example.com"); 
    @Test
void 회원_조회_성공() {
    // Given: 테스트 데이터 준비
    Member member = Member.builder()
        .email("test@example.com")
        .name("홍길동")
        .phoneNumber("01012345678")
        .build();
    Member savedMember = memberRepository.save(member);
    
    // When: 조회 실행
    MemberResponse response = memberService.getMember(savedMember.getId());
    
    // Then: 검증
    assertThat(response.id()).isEqualTo(savedMember.getId());
    assertThat(response.email()).isEqualTo("test@example.com");
    assertThat(response.name()).isEqualTo("홍길동");
}

    8. Repository 계층 테스트 — @DataJpaTest

    @DataJpaTest는 JPA Repository를 테스트하기 위한 전용 어노테이션이다. 전체 애플리케이션 컨텍스트를 로드하지 않고, JPA 관련 설정만 로드하여 빠르고 정확한 테스트를 수행한다.

    @DataJpaTest의 특징

    • @Transactional 포함: 각 테스트 메서드는 트랜잭션 내에서 실행되고, 종료 후 자동 롤백된다. 테스트 간 데이터 격리가 보장된다.
    • 인메모리 데이터베이스 사용: 기본적으로 H2 데이터베이스를 사용하여 실제 DB에 영향을 주지 않는다.
    • 빠른 실행 속도: 전체 컨텍스트를 로드하지 않으므로 테스트 실행 시간이 짧다.
    • JPA 구성 요소만 로드: Repository, EntityManager 등 JPA 관련 빈만 등록된다. Service, Controller는 로드되지 않는다.
    @DataJpaTest
class MemberRepositoryTest {
    
    @Autowired
    private MemberRepository memberRepository;
    
    @Test
    void 회원_저장_성공() {
        // Given
        Member member = Member.builder()
            .email("test@example.com")
            .name("홍길동")
            .build();
        
        // When
        Member savedMember = memberRepository.save(member);
        
        // Then
        assertThat(savedMember.getId()).isNotNull();
        assertThat(savedMember.getEmail()).isEqualTo("test@example.com");
        assertThat(savedMember.getName()).isEqualTo("홍길동");
    }
    
    @Test
    void 이메일로_회원_존재_여부_확인() {
        // Given
        Member member = Member.builder()
            .email("test@example.com")
            .name("홍길동")
            .build();
        memberRepository.save(member);
        
        // When
        boolean exists = memberRepository.existsByEmail("test@example.com");
        boolean notExists = memberRepository.existsByEmail("other@example.com");
        
        // Then
        assertThat(exists).isTrue();
        assertThat(notExists).isFalse();
    }
}

    9. 트랜잭션 읽기 전용 최적화

    조회 전용 메서드에 @Transactional(readOnly = true)를 설정하면, JPA와 데이터베이스 드라이버가 여러 최적화를 수행한다. 특히 대량 조회 작업이 많은 서비스에서 성능 향상 효과가 크다.

    readOnly = true의 효과

    최적화 항목 설명
    변경 감지(Dirty Checking) 비활성화 영속성 컨텍스트가 스냅샷을 생성하지 않아 메모리와 CPU 사용량이 감소한다. 조회한 엔티티의 변경 사항을 추적할 필요가 없으므로 오버헤드가 줄어든다.
    플러시(Flush) 모드 최적화 Hibernate는 FlushMode를 MANUAL로 설정하여, 트랜잭션 커밋 시 플러시를 생략한다. 불필요한 쿼리 실행이 방지된다.
    데이터베이스 드라이버 최적화 일부 JDBC 드라이버는 readOnly 트랜잭션을 감지하여 읽기 전용 커넥션을 사용한다. Master-Slave 구조에서 Slave DB로 라우팅하여 부하를 분산할 수 있다.

    클래스 레벨 적용

    대부분의 메서드가 조회인 Service 클래스는 클래스 레벨에 @Transactional(readOnly = true)를 적용하고, 쓰기가 필요한 메서드만 @Transactional로 오버라이드하는 것이 효율적이다. 

    @Service
@RequiredArgsConstructor
@Transactional(readOnly = true)  // ✅ 클래스 레벨: 기본적으로 읽기 전용
public class MemberService {
    
    private final MemberRepository memberRepository;
    
    // readOnly = true 적용됨 (조회 전용)
    public MemberResponse getMember(Long id) {
        Member member = memberRepository.findById(id)
            .orElseThrow(() -> new MemberNotFoundException("회원을 찾을 수 없습니다."));
        return MemberResponse.from(member);
    }
    
    // readOnly = true 적용됨 (조회 전용)
    public List getAllMembers() {
        return memberRepository.findAll()
            .stream()
            .map(MemberResponse::from)
            .toList();
    }
    
    // ✅ 쓰기 작업: 메서드 레벨에서 @Transactional로 오버라이드
    @Transactional  // readOnly = false (기본값)
    public Long registerMember(MemberRegisterRequest request) {
        validateDuplicateEmail(request.getEmail());
        Member member = request.toEntity();
        Member savedMember = memberRepository.save(member);
        return savedMember.getId();
    }
    
    @Transactional  // readOnly = false
    public void updateMember(Long id, MemberUpdateRequest request) {
        Member member = memberRepository.findById(id)
            .orElseThrow(() -> new MemberNotFoundException("회원을 찾을 수 없습니다."));
        member.updateInfo(request.getName(), request.getPhoneNumber());
    }
}

    멱등성 (Idempotency)이란?

    멱등성의 의미 같은 요청을 여러 번 실행해도 결과가 동일하다는 성질이다. 수학에서 f(f(x)) = f(x)와 같은 개념이다. 조회 작업은 데이터를 변경하지 않으므로 멱등성이 보장된다. 아무리 많이 조회해도 데이터가 바뀌지 않는다.
    HTTP 메서드 멱등성 설명
    GET ✅ 보장 데이터를 조회만 한다. 100번 조회해도 결과는 동일하다.
    PUT ✅ 보장 전체 데이터를 교체한다. 같은 데이터로 100번 교체해도 결과는 동일하다.
    DELETE ✅ 보장 데이터를 삭제한다. 이미 삭제된 데이터를 또 삭제해도 결과는 동일하다(없음).
    POST ❌ 미보장 새로운 데이터를 생성한다. 100번 요청하면 100개의 데이터가 생성될 수 있다.
    PATCH ❌ 미보장 부분 수정이므로 구현에 따라 다르다. 일반적으로 멱등성을 보장하지 않는다.

    조회 메서드에 @Transactional(readOnly = true)를 설정하는 것은, 멱등성이 보장되는 작업임을 명시적으로 선언하는 것이기도 하다.

    10. API 응답 공통화

    API 응답 형식을 통일하면 클라이언트가 일관된 방식으로 데이터를 처리할 수 있다. 성공과 실패 응답 모두 동일한 구조를 가지도록 공통 응답 객체를 설계한다.

    공통 응답 포맷 설계 

    public record ApiResponse(
    int code,
    String status,
    T data
) {
    // 성공 응답 생성
    public static  ApiResponse success(HttpStatus status, T data) {
        return new ApiResponse<>(
            status.value(), 
            status.getReasonPhrase(), 
            data
        );
    }
    
    // 실패 응답 생성
    public static  ApiResponse fail(HttpStatus status, T data) {
        return new ApiResponse<>(
            status.value(), 
            status.getReasonPhrase(), 
            data
        );
    }
}
    // Controller에서 사용
@RestController
@RequestMapping("/api/members")
@RequiredArgsConstructor
public class MemberController {
    
    private final MemberService memberService;
    
    @PostMapping
    public ResponseEntity> registerMember(
        @RequestBody MemberRegisterRequest request
    ) {
        Long memberId = memberService.registerMember(request);
        ApiResponse response = ApiResponse.success(HttpStatus.CREATED, memberId);
        return ResponseEntity.status(HttpStatus.CREATED).body(response);
    }
    
    @GetMapping("/{id}")
    public ResponseEntity> getMember(
        @PathVariable Long id
    ) {
        MemberResponse member = memberService.getMember(id);
        ApiResponse response = ApiResponse.success(HttpStatus.OK, member);
        return ResponseEntity.ok(response);
    }
}

    11. Swagger를 통한 API 문서 자동화

    Swagger(OpenAPI)를 사용하면 코드에 어노테이션을 추가하는 것만으로 API 명세서를 자동으로 생성할 수 있다. 개발자는 문서를 따로 작성할 필요 없이, 코드와 문서가 항상 동기화된 상태를 유지한다.

    Swagger 의존성 추가 

    // build.gradle
dependencies {
    implementation 'org.springdoc:springdoc-openapi-starter-webmvc-ui:2.3.0'
}

    애플리케이션을 실행하면 http://localhost:8080/swagger-ui/index.html에서 API 문서를 확인할 수 있다.

    주요 어노테이션

    어노테이션 설명 사용 예시
    @Tag API를 그룹화한다. 컨트롤러 클래스에 적용한다. @Tag(name = "회원", description = "회원 관리 API")
    @Operation 개별 API 메서드를 설명한다. @Operation(summary = "회원 등록", description = "새로운 회원을 등록합니다.")
    @ApiResponse
    @ApiResponses    		 응답 코드와 메시지를 정의한다. @ApiResponse(responseCode = "201", description = "회원 등록 성공")

    @RestController
@RequestMapping("/api/members")
@RequiredArgsConstructor
@Tag(name = "회원", description = "회원 관리 API")
public class MemberController {

    @Operation(summary = "회원 등록", description = "새로운 회원을 등록합니다.")
    @ApiResponses({
        @ApiResponse(responseCode = "201", description = "회원 등록 성공"),
        @ApiResponse(responseCode = "400", description = "잘못된 요청 (중복 이메일 등)")
    })
    public ResponseEntity> registerMember(
        @RequestBody @Valid MemberRegisterRequest request
    ) {
        Long memberId = memberService.registerMember(request);
        ApiResponse response = ApiResponse.success(HttpStatus.CREATED, memberId);
        return ResponseEntity.status(HttpStatus.CREATED).body(response);
    }
    
    @GetMapping("/{id}")
    @Operation(summary = "회원 조회", description = "ID로 회원 정보를 조회합니다.")

    })

    MapStruct — DTO ↔ 엔티티 변환 자동화

    MapStruct는 컴파일 타임에 자동으로 변환 코드를 생성해주는 라이브러리이다. 리플렉션을 사용하지 않아 성능이 우수하며, 타입 안정성이 보장된다. 수동으로 작성하는 toEntity(), from() 메서드를 자동화할 수 있다. 

    // build.gradle
dependencies {
    implementation 'org.mapstruct:mapstruct:1.5.5.Final'
    annotationProcessor 'org.mapstruct:mapstruct-processor:1.5.5.Final'
}
    @Mapper(componentModel = "spring")
public interface MemberMapper {
    
    // 엔티티 → DTO
    MemberResponse toResponse(Member member);
    
    // DTO → 엔티티
    Member toEntity(MemberRegisterRequest request);
    
    // 리스트 변환
    List toResponseList(List members);
}
    // 서비스에서 사용
@Service
@RequiredArgsConstructor
public class MemberService {
    
    private final MemberRepository memberRepository;
    private final MemberMapper memberMapper;
    
    public MemberResponse getMember(Long id) {
        Member member = memberRepository.findById(id)
            .orElseThrow(() -> new MemberNotFoundException("회원을 찾을 수 없습니다."));
        
        // MapStruct가 자동으로 변환
        return memberMapper.toResponse(member);
    }
}

    MySQL JSON 필드 활용

    MySQL 5.7부터 JSON 데이터 타입을 지원한다. 정규화하기 애매한 유동적인 데이터(설정값, 메타데이터, 태그 등)를 JSON 형식으로 저장하면 테이블 구조를 단순화하고 유연성을 높일 수 있다. 

    -- 테이블 생성
CREATE TABLE member (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    email VARCHAR(255) NOT NULL,
    name VARCHAR(100),
    preferences JSON  -- JSON 필드
);
    -- JSON 데이터 삽입
INSERT INTO member (email, name, preferences) 
VALUES (
    'test@example.com', 
    '홍길동',
    '{"theme": "dark", "language": "ko", "notifications": true}'
);
    -- JSON 필드 조회
SELECT 
    email,
    preferences->>'$.theme' AS theme,
    preferences->>'$.language' AS language
FROM member
WHERE preferences->>'$.notifications' = 'true';
    JSON 필드의 장단점 장점:
    1. 스키마 유연성: 필드를 추가하거나 변경할 때 테이블 구조를 수정할 필요가 없다.
    2. 개발 속도: 프로토타입 단계에서 빠르게 기능을 추가할 수 있다.
    3. 복잡한 데이터 저장: 계층 구조나 배열 같은 복잡한 데이터를 한 필드에 저장할 수 있다.

    단점:
    1. 성능: 인덱싱이 제한적이며, 복잡한 쿼리는 느려질 수 있다.
    2. 데이터 무결성: 스키마가 없으므로 잘못된 데이터가 들어갈 위험이 있다.
    3. 정규화 위배: 중복 데이터가 발생할 수 있으며, 데이터 일관성 관리가 어렵다.

    결론: 핵심 비즈니스 데이터는 정규화된 컬럼으로 관리하고, 부가적이고 유동적인 데이터만 JSON으로 저장하는 것이 좋다.

    오늘은 실전 Spring Boot 개발에서 반드시 알아야 할 핵심 패턴과 Best Practice를 학습했다. 보안 설정 관리부터 시작하여 계층별 책임 분리, 객체 생성 패턴(빌더, 정적 팩토리 메서드), 성능 최적화(existsBy, readOnly), 테스트 전략(단위 테스트, TDD, GWT), 그리고 API 설계(공통 응답, Swagger)까지 실무에서 바로 적용할 수 있는 지식을 익혔다.
