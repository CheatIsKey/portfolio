    1. 문제 상황 — 순환 참조 (Circular Reference)

    에러 발생 현상

    애플리케이션 실행 시 Application failed to start 에러가 발생한다. 스프링 부트가 시작 단계에서 멈추고, 콘솔에 순환 참조 관련 에러 메시지가 출력된다. 

    ***************************
APPLICATION FAILED TO START
***************************

Description:

The dependencies of some of the beans in the application context form a cycle:

┌─────┐
|  scheduleService defined in file [ScheduleService.class]
↑     ↓
|  commentService defined in file [CommentService.class]
└─────┘

    원인 분석

    순환 참조는 서로 다른 두 개 이상의 빈(Bean)이 서로를 필요로 하는 상황에서 발생한다. 스프링 컨테이너는 애플리케이션 시작 시 모든 빈을 생성하고 의존성을 주입하는데, 순환 고리가 형성되면 어떤 빈을 먼저 생성해야 할지 결정할 수 없어 무한 대기 상태에 빠진다.

    순환 참조 발생 메커니즘 1. 스프링 컨테이너가 ScheduleService를 생성하려고 시도한다.
    2. ScheduleService는 생성자에 CommentService가 필요하다고 선언되어 있다.
    3. 스프링이 CommentService를 먼저 생성하러 간다.
    4. CommentService는 생성자에 ScheduleService가 필요하다고 선언되어 있다.
    5. 다시 ScheduleService로 돌아간다. (1번으로 복귀)
    6. 무한 루프에 빠져 데드락(Deadlock) 상태가 된다.
    7. 스프링은 이를 감지하고 애플리케이션 시작을 중단한다.

    실제 코드 예시 

    // ScheduleService
@Service
@RequiredArgsConstructor
public class ScheduleService {
    
    private final ScheduleRepository scheduleRepository;
    private final CommentService commentService; // ❌ CommentService를 의존
    
    public ScheduleResponse createSchedule(ScheduleCreateRequest request) {
        Schedule schedule = request.toEntity();
        Schedule savedSchedule = scheduleRepository.save(schedule);
        
        // 일정 생성 시 기본 댓글 자동 생성
        commentService.createDefaultComment(savedSchedule.getId());
        
        return ScheduleResponse.from(savedSchedule);
    }
}
    // CommentService
@Service
@RequiredArgsConstructor
public class CommentService {
    
    private final CommentRepository commentRepository;
    private final ScheduleService scheduleService; // ❌ ScheduleService를 의존
    
    public CommentResponse createComment(Long scheduleId, CommentCreateRequest request) {
        // 일정이 존재하는지 검증
        ScheduleResponse schedule = scheduleService.getSchedule(scheduleId);
        
        Comment comment = Comment.builder()
            .scheduleId(scheduleId)
            .content(request.getContent())
            .build();
            
        Comment savedComment = commentRepository.save(comment);
        return CommentResponse.from(savedComment);
    }
}
    순환 참조의 문제점 1. 애플리케이션 시작 실패: 개발 중에는 발견하기 쉽지만, 복잡한 의존성 구조에서는 찾기 어려울 수 있다.
    2. 강한 결합(Tight Coupling): 두 서비스가 서로에게 과도하게 의존하면 코드 수정이 어려워진다.
    3. 테스트 어려움: 단위 테스트 작성 시 두 서비스를 함께 모킹해야 하므로 복잡도가 증가한다.
    4. 설계 악화: 순환 참조는 잘못된 설계의 신호이다. 책임이 제대로 분리되지 않았을 가능성이 크다.

    비유로 이해하기

    금고 열쇠 비유 두 개의 금고가 있다. A 금고와 B 금고이다.

    - A 금고를 열려면 B 금고 안에 있는 열쇠가 필요하다.
    - B 금고를 열려면 A 금고 안에 있는 열쇠가 필요하다.

    결과: 두 금고 모두 영원히 열 수 없다.

    스프링의 순환 참조도 이와 정확히 같은 상황이다. ScheduleService를 만들려면 CommentService가 필요하고, CommentService를 만들려면 ScheduleService가 필요하므로 둘 다 만들 수 없는 교착 상태가 된다.

    2. 해결책 1 — @Lazy 어노테이션 (즉시 해결)

    @Lazy의 핵심 원리

    @Lazy 어노테이션은 순환 고리 중 한 곳을 끊어주는 역할을 한다. 빈을 실제로 생성하는 대신, 프록시(Proxy) 객체를 먼저 주입하여 순환 참조를 회피한다.

    프록시(Proxy)란? 프록시는 실제 객체를 감싸고 있는 껍데기(대리자)이다. 겉모습은 진짜와 똑같지만, 내부는 비어있는 가짜 객체이다.

    동작 방식:
    1. 스프링이 애플리케이션을 시작할 때, @Lazy가 붙은 쪽에는 프록시 객체를 주입한다.
    2. 이 프록시는 실제 메서드를 호출하기 전까지는 아무 일도 하지 않는다.
    3. 나중에 실제로 메서드를 호출하는 순간, 프록시가 진짜 빈(Real Bean)을 찾아와서 로직을 수행한다.

    비유: 배우의 스턴트 더블(대역)이다. 위험한 장면 전까지는 대역이 서 있다가, 실제 연기가 필요한 순간에 본인 배우로 교체되는 것과 같다.

    적용 규칙 및 우선순위

    구분 설명
    적용 위치 순환 참조 관계에서 한 쪽에만 붙이면 된다. 양쪽에 다 붙일 필요는 없다.
    우선순위 결정 주인공(Main)과 조연(Sub) 관계로 생각한다.
    - 주인공: 핵심 엔티티, 먼저 생성되어야 하는 서비스
    - 조연: 주인공에 종속적인 서비스, 보조 역할
    예시 Schedule(일정)과 Comment(댓글)의 경우
    - 주인공: ScheduleService (일정이 먼저 있어야 함)
    - 조연: CommentService (일정이 있어야 댓글을 달 수 있음)
    → CommentService에 @Lazy 적용

    코드 구현 방법

    Lombok @RequiredArgsConstructor 제거 필수 @RequiredArgsConstructor는 생성자를 자동으로 생성하기 때문에 @Lazy를 붙일 틈이 없다. 생성자를 직접 작성해야만 파라미터에 @Lazy를 적용할 수 있다. 
    // ❌ 잘못된 예시 - @RequiredArgsConstructor 사용
@Service
@RequiredArgsConstructor  // 이것 때문에 @Lazy를 붙일 수 없음
public class CommentService {
    
    private final ScheduleService scheduleService; // @Lazy를 붙일 방법이 없음
}
    // ✅ 올바른 예시 - 수동 생성자 작성
@Service
public class CommentService {
    
    private final CommentRepository commentRepository;
    private final ScheduleService scheduleService;
    
    // 생성자를 직접 작성하고, @Lazy를 파라미터 타입 앞에 붙인다
    public CommentService(
        CommentRepository commentRepository,
        @Lazy ScheduleService scheduleService  // ✅ 이 위치에 @Lazy 적용
    ) {
        this.commentRepository = commentRepository;
        this.scheduleService = scheduleService;
    }
    
    public CommentResponse createComment(Long scheduleId, CommentCreateRequest request) {
        // scheduleService를 호출하는 이 순간,
        // 프록시가 진짜 ScheduleService를 찾아와서 메서드를 실행한다
        ScheduleResponse schedule = scheduleService.getSchedule(scheduleId);
        
        Comment comment = Comment.builder()
            .scheduleId(scheduleId)
            .content(request.getContent())
            .build();
            
        Comment savedComment = commentRepository.save(comment);
        return CommentResponse.from(savedComment);
    }
}
    // ScheduleService는 @Lazy 없이 정상적으로 작성
@Service
@RequiredArgsConstructor
public class ScheduleService {
    
    private final ScheduleRepository scheduleRepository;
    private final CommentService commentService; // @Lazy 없음 (진짜 객체 주입)
    
    public ScheduleResponse getSchedule(Long id) {
        Schedule schedule = scheduleRepository.findById(id)
            .orElseThrow(() -> new ScheduleNotFoundException("일정을 찾을 수 없습니다."));
        return ScheduleResponse.from(schedule);
    }
    
    public ScheduleResponse createSchedule(ScheduleCreateRequest request) {
        Schedule schedule = request.toEntity();
        Schedule savedSchedule = scheduleRepository.save(schedule);
        
        // CommentService 메서드 호출
        commentService.createDefaultComment(savedSchedule.getId());
        
        return ScheduleResponse.from(savedSchedule);
    }
}

    @Lazy 동작 흐름

    스프링 부트 시작 시 1. ScheduleService 생성 시도
       → CommentService가 필요하다고 선언되어 있음

    2. CommentService 생성 시도
       → ScheduleService가 필요하지만, @Lazy가 붙어있음
       → 진짜 대신 프록시(가짜 껍데기)를 먼저 만들어서 주입

    3. CommentService 생성 완료 (프록시 ScheduleService를 가진 상태)

    4. ScheduleService 생성 완료 (진짜 CommentService를 가진 상태)

    5. 순환 고리가 끊어졌으므로 애플리케이션 시작 성공
    실제 메서드 호출 시 1. 클라이언트가 commentService.createComment() 호출

    2. CommentService 내부에서 scheduleService.getSchedule() 호출

    3. 이 순간 프록시가 진짜 ScheduleService를 찾아온다
       → 스프링 컨테이너에서 실제 빈을 검색
       → 프록시가 진짜 객체에게 메서드 호출을 위임

    4. 실제 로직이 실행된다

    3. 해결책 2 — 중간 목적 클래스 분리 (근본적 해결)

    설계 개선의 핵심 아이디어

    @Lazy는 당장의 에러를 해결하는 기술적 해결책이다. 하지만 순환 참조가 발생했다는 것 자체가 두 서비스가 너무 끈끈하게 엮여있다(강한 결합)는 신호이다. 근본적인 해결은 서비스를 책임별로 분리하는 것이다.

    중간 목적 클래스(Intermediate Service) 패턴 하나의 거대한 Service를 목적별로 여러 개의 작은 Service로 분리한다.

    예시: ScheduleService 분리
    - ScheduleReadService: 조회 전용 서비스 (읽기 책임)
    - ScheduleCreateService: 생성 전용 서비스 (쓰기 책임)
    - ScheduleUpdateService: 수정 전용 서비스 (수정 책임)
    - ScheduleDeleteService: 삭제 전용 서비스 (삭제 책임)

    핵심: 각 서비스는 단일 책임(Single Responsibility)만 가진다. CommentService는 전체 ScheduleService 대신 필요한 부분(예: ScheduleReadService)만 의존한다.

    기존 구조의 문제점 

    // 기존: 거대한 ScheduleService
@Service
@RequiredArgsConstructor
public class ScheduleService {
    
    private final ScheduleRepository scheduleRepository;
    private final CommentService commentService; // ❌ 순환 참조 발생
    
    // 조회 기능
    public ScheduleResponse getSchedule(Long id) { ... }
    public List getAllSchedules() { ... }
    
    // 생성 기능
    public ScheduleResponse createSchedule(ScheduleCreateRequest request) { ... }
    
    // 수정 기능
    public ScheduleResponse updateSchedule(Long id, ScheduleUpdateRequest request) { ... }
    
    // 삭제 기능
    public void deleteSchedule(Long id) { ... }
    
    // 댓글 관련 기능
    public int getCommentCount(Long scheduleId) { ... }
}
    // CommentService도 ScheduleService를 의존
@Service
@RequiredArgsConstructor
public class CommentService {
    
    private final CommentRepository commentRepository;
    private final ScheduleService scheduleService; // ❌ 순환 참조 발생
    
    public CommentResponse createComment(Long scheduleId, CommentCreateRequest request) {
        // 일정 존재 여부 검증을 위해 ScheduleService 호출
        scheduleService.getSchedule(scheduleId); // ❌ 전체 서비스를 의존
        
        // 댓글 생성 로직
        ...
    }
}
    문제점 분석 1. 과도한 의존: CommentService는 실제로는 일정 조회 기능만 필요한데, ScheduleService 전체를 의존한다.
    2. 책임 비대화: ScheduleService가 너무 많은 책임(CRUD + 댓글 관련)을 가지고 있다.
    3. 테스트 어려움: ScheduleService를 테스트하려면 CommentService도 모킹해야 하고, 그 반대도 마찬가지이다.
    4. 수정 영향 범위 확대: 한쪽을 수정하면 다른 쪽에도 영향을 준다.

    개선된 구조 — 목적별 분리 

    // 1. ScheduleReadService - 조회 전용
@Service
@RequiredArgsConstructor
@Transactional(readOnly = true) // 읽기 전용 최적화
public class ScheduleReadService {
    
    private final ScheduleRepository scheduleRepository;
    
    // 단일 조회
    public ScheduleResponse getSchedule(Long id) {
        Schedule schedule = scheduleRepository.findById(id)
            .orElseThrow(() -> new ScheduleNotFoundException("일정을 찾을 수 없습니다."));
        return ScheduleResponse.from(schedule);
    }
    
    // 전체 조회
    public List getAllSchedules() {
        return scheduleRepository.findAll()
            .stream()
            .map(ScheduleResponse::from)
            .toList();
    }
    
    // 존재 여부 확인 (CommentService가 필요한 기능)
    public boolean existsSchedule(Long id) {
        return scheduleRepository.existsById(id);
    }
    
    // 검증 (예외를 던지는 방식)
    public void validateScheduleExists(Long id) {
        if (!scheduleRepository.existsById(id)) {
            throw new ScheduleNotFoundException("일정을 찾을 수 없습니다.");
        }
    }
}
    // 2. ScheduleCreateService - 생성 전용
@Service
@RequiredArgsConstructor
public class ScheduleCreateService {
    
    private final ScheduleRepository scheduleRepository;
    
    @Transactional
    public ScheduleResponse createSchedule(ScheduleCreateRequest request) {
        Schedule schedule = request.toEntity();
        Schedule savedSchedule = scheduleRepository.save(schedule);
        return ScheduleResponse.from(savedSchedule);
    }
}
    // 3. ScheduleUpdateService - 수정 전용
@Service
@RequiredArgsConstructor
public class ScheduleUpdateService {
    
    private final ScheduleRepository scheduleRepository;
    
    @Transactional
    public ScheduleResponse updateSchedule(Long id, ScheduleUpdateRequest request) {
        Schedule schedule = scheduleRepository.findById(id)
            .orElseThrow(() -> new ScheduleNotFoundException("일정을 찾을 수 없습니다."));
        
        schedule.update(request.getTitle(), request.getContent());
        
        return ScheduleResponse.from(schedule);
    }
}
    // 4. ScheduleDeleteService - 삭제 전용
@Service
@RequiredArgsConstructor
public class ScheduleDeleteService {
    
    private final ScheduleRepository scheduleRepository;
    private final CommentRepository commentRepository;
    
    @Transactional
    public void deleteSchedule(Long id) {
        if (!scheduleRepository.existsById(id)) {
            throw new ScheduleNotFoundException("일정을 찾을 수 없습니다.");
        }
        
        // 관련 댓글 먼저 삭제
        commentRepository.deleteAllByScheduleId(id);
        
        // 일정 삭제
        scheduleRepository.deleteById(id);
    }
}
    // CommentService는 필요한 서비스만 의존
@Service
@RequiredArgsConstructor
public class CommentService {
    
    private final CommentRepository commentRepository;
    private final ScheduleReadService scheduleReadService; // ✅ 읽기 서비스만 의존
    
    @Transactional
    public CommentResponse createComment(Long scheduleId, CommentCreateRequest request) {
        // 일정 존재 여부 검증 - 필요한 기능만 사용
        scheduleReadService.validateScheduleExists(scheduleId);
        
        Comment comment = Comment.builder()
            .scheduleId(scheduleId)
            .content(request.getContent())
            .build();
            
        Comment savedComment = commentRepository.save(comment);
        return CommentResponse.from(savedComment);
    }
    
    @Transactional(readOnly = true)
    public List getCommentsBySchedule(Long scheduleId) {
        // 일정 존재 여부 검증
        scheduleReadService.validateScheduleExists(scheduleId);
        
        return commentRepository.findAllByScheduleId(scheduleId)
            .stream()
            .map(CommentResponse::from)
            .toList();
    }
}

    개선된 구조의 이점

    이점 설명
    순환 참조 완전 제거 CommentService는 ScheduleReadService만 의존한다. ScheduleReadService는 CommentService를 전혀 의존하지 않는다. 순환 고리가 애초에 형성되지 않는다.
    단일 책임 원칙 각 서비스가 하나의 명확한 책임만 가진다. ScheduleReadService는 오직 조회만, ScheduleCreateService는 오직 생성만 담당한다. 코드 이해가 쉬워지고 유지보수성이 향상된다.
    트랜잭션 최적화 읽기 전용 서비스에 @Transactional(readOnly = true)를 클래스 레벨로 적용할 수 있다. 변경 감지(Dirty Checking)가 비활성화되어 성능이 향상된다.
    테스트 용이성 각 서비스가 독립적이므로 단위 테스트 작성이 쉽다. CommentService를 테스트할 때 ScheduleReadService만 모킹하면 된다.
    확장성 새로운 요구사항이 추가되어도 기존 서비스를 수정할 필요가 없다. 새로운 중간 목적 클래스를 추가하면 된다. 예: ScheduleSearchService(검색 전용), ScheduleValidationService(검증 전용)
    병렬 개발 가능 여러 개발자가 각자 다른 서비스를 동시에 개발할 수 있다. 서로 영향을 주지 않으므로 협업이 원활하다.

    컨트롤러 계층 연결

    컨트롤러는 목적에 맞는 서비스를 주입받아 사용한다. 하나의 컨트롤러가 여러 개의 중간 목적 서비스를 의존할 수 있다. 

    @RestController
@RequestMapping("/api/schedules")
@RequiredArgsConstructor
public class ScheduleController {
    
    // 필요한 서비스만 주입
    private final ScheduleReadService scheduleReadService;
    private final ScheduleCreateService scheduleCreateService;
    private final ScheduleUpdateService scheduleUpdateService;
    private final ScheduleDeleteService scheduleDeleteService;
    
    // 조회 API
    @GetMapping("/{id}")
    public ResponseEntity> getSchedule(@PathVariable Long id) {
        ScheduleResponse response = scheduleReadService.getSchedule(id);
        return ResponseEntity.ok(ApiResponse.success(HttpStatus.OK, response));
    }
    
    // 생성 API
    @PostMapping
    public ResponseEntity> createSchedule(
        @RequestBody ScheduleCreateRequest request
    ) {
        ScheduleResponse response = scheduleCreateService.createSchedule(request);
        return ResponseEntity.status(HttpStatus.CREATED)
            .body(ApiResponse.success(HttpStatus.CREATED, response));
    }
    
    // 수정 API
    @PutMapping("/{id}")
    public ResponseEntity> updateSchedule(
        @PathVariable Long id,
        @RequestBody ScheduleUpdateRequest request
    ) {
        ScheduleResponse response = scheduleUpdateService.updateSchedule(id, request);
        return ResponseEntity.ok(ApiResponse.success(HttpStatus.OK, response));
    }
    
    // 삭제 API
    @DeleteMapping("/{id}")
    public ResponseEntity> deleteSchedule(@PathVariable Long id) {
        scheduleDeleteService.deleteSchedule(id);
        return ResponseEntity.ok(ApiResponse.success(HttpStatus.OK, null));
    }
}

    실무 적용 시 고려사항

    언제 서비스를 분리해야 하는가? 분리를 권장하는 경우:
    1. 하나의 서비스 클래스가 300줄 이상일 때
    2. 메서드가 10개 이상일 때
    3. 순환 참조가 발생했을 때
    4. 트랜잭션 설정(readOnly)이 메서드마다 다를 때
    5. 여러 팀이 동시에 개발할 때

    분리하지 않아도 되는 경우:
    1. 서비스가 단순하고 작을 때 (100줄 이하)
    2. 메서드가 5개 이하일 때
    3. 의존성이 단방향이고 명확할 때
    4. 프로토타입 단계일 때
    추가 확장 가능한 서비스 분리 예시 ScheduleValidationService: 검증 로직만 담당
    - 제목 길이 검증, 날짜 유효성 검증, 권한 검증 등

    ScheduleSearchService: 복잡한 검색 쿼리 담당
    - 키워드 검색, 날짜 범위 검색, 필터링 등

    ScheduleNotificationService: 알림 발송 담당
    - 일정 생성 시 알림, 일정 수정 시 알림 등

    핵심: 각 서비스는 응집도는 높고, 결합도는 낮게 설계한다.

    4. 기술 용어 정리

    용어 영문 설명
    빈 Bean 스프링 컨테이너가 관리하는 자바 객체이다. @Service, @Repository, @Component 등이 붙은 클래스는 빈으로 등록된다. 주의: 데이터베이스 테이블과는 전혀 관계없다.
    프록시 Proxy 실제 객체를 감싸고 있는 껍데기(대리자)이다. 겉모습은 진짜와 똑같지만, 내부 로직은 실제 호출 시점에 진짜 객체에게 위임한다. @Lazy가 적용되면 프록시 객체가 주입된다.
    목(모의 객체) Mock 테스트용 가짜 객체이다. 실제 동작을 하지 않으며, 미리 정의된 값을 반환한다. 차이점: 프록시는 실제 객체를 호출하지만, Mock은 호출하지 않고 가짜 응답만 준다.
    순환 참조 Circular Reference 두 개 이상의 빈이 서로를 의존하여 순환 고리를 형성하는 상황이다. 스프링 컨테이너가 빈 생성 순서를 결정할 수 없어 애플리케이션 시작이 실패한다.
    데드락 Deadlock 두 개 이상의 작업이 서로를 기다리며 무한 대기 상태에 빠지는 현상이다. 순환 참조는 빈 생성 과정에서 발생하는 데드락의 일종이다.
    의존성 주입 Dependency Injection 객체가 필요로 하는 의존성을 외부에서 주입받는 패턴이다. 스프링은 생성자, 필드, Setter를 통해 의존성을 주입한다. 생성자 주입이 가장 권장된다.

    5. 해결 방법 비교 — @Lazy vs 서비스 분리

    구분 @Lazy 어노테이션 중간 목적 클래스 분리
    해결 속도 즉시 (5분 이내) 시간 소요 (1~2시간)
    코드 변경량 최소 (어노테이션 1개 추가) 많음 (서비스 분리 및 리팩토링)
    근본 원인 해결 ❌ 에러만 회피 ✅ 설계 개선으로 근본 해결
    유지보수성 ⚠️ 동일 (강한 결합 유지) ✅ 향상 (느슨한 결합)
    테스트 용이성 ⚠️ 동일 (여전히 복잡) ✅ 향상 (독립적 테스트 가능)
    성능 ⚠️ 약간 느림 (프록시 오버헤드) ✅ 최적화 가능 (readOnly 분리)
    확장성 ❌ 낮음 ✅ 높음 (새 서비스 추가 용이)
    권장 시점 긴급 배포, 임시 조치 장기 프로젝트, 리팩토링 단계
    실무 전략: 단계적 접근 1단계 (긴급): @Lazy로 즉시 에러 해결하여 배포
    2단계 (단기): 기술 부채로 등록하고 리팩토링 일정 수립
    3단계 (장기): 중간 목적 클래스로 분리하여 설계 개선
    4단계 (유지): 코드 리뷰와 문서화로 재발 방지

    핵심: @Lazy는 응급처치이고, 서비스 분리는 근본 치료이다. 둘 다 알고 상황에 맞게 사용하는 것이 중요하다.

    마무리

    오늘은 Spring Boot 개발 중 자주 마주치는 순환 참조(Circular Reference) 문제를 깊이 있게 학습했다. @Lazy 어노테이션을 사용한 즉각적인 해결 방법과, 중간 목적 클래스로 서비스를 분리하는 근본적인 설계 개선 방법을 모두 익혔다.

    특히 ScheduleReadService, ScheduleCreateService처럼 책임을 명확히 분리하는 패턴은 순환 참조 해결뿐만 아니라, 트랜잭션 최적화, 테스트 용이성, 확장성 측면에서도 큰 이점을 제공한다. 이는 단일 책임 원칙(Single Responsibility Principle)과 의존성 역전 원칙(Dependency Inversion Principle)을 실천하는 좋은 예시이다.

    실무에서는 @Lazy로 긴급하게 해결한 후, 시간을 두고 서비스 분리로 리팩토링하는 단계적 접근이 효과적이다. 중요한 것은 순환 참조가 발생했다는 것 자체가 설계를 재검토해야 한다는 신호임을 인식하는 것이다.

    내일은 이 패턴을 프로젝트에 적용하며, 더 나은 설계를 위한 학습을 이어가겠다.
