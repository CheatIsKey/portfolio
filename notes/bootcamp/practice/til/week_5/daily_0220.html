    1. B-Tree / B+Tree와 쓰기 성능

    왜 데이터베이스 쓰기는 읽기보다 느린가

    대부분의 관계형 데이터베이스(MySQL, PostgreSQL 등)는 인덱스를 내부적으로 B-Tree 또는 B+Tree 자료구조로 관리한다. 이 구조 때문에 데이터 삽입(INSERT)·수정(UPDATE)·삭제(DELETE)는 데이터 조회(SELECT)에 비해 본질적으로 더 많은 연산이 필요하다.

    B-Tree / B+Tree는 이진 트리가 아니다 이진 트리(Binary Tree)는 한 노드가 자식을 최대 2개 가질 수 있다. 반면 B-Tree는 한 노드에 여러 키와 여러 자식 포인터를 담을 수 있다. 데이터 탐색은 루트 노드에서 시작해 조건에 맞는 자식 노드로 깊이(depth) 방향으로 내려가며 이루어진다. 이 방식의 시간 복잡도는 O(log N)이다.

    예를 들어, 데이터가 1,024건 있을 때 순차 탐색은 최악 1,024번 비교해야 하지만, B-Tree에서는 2의 10제곱이 1,024이므로 단 10번의 비교로 원하는 데이터를 찾을 수 있다. 이것이 인덱스가 대용량 데이터에서 강력한 이유이다.
    균형 트리(Balanced Tree)의 특성과 쓰기 비용 B-Tree는 균형 잡힌(Balanced) 트리이다. 균형이 유지되어야 O(log N)이라는 탐색 성능이 보장된다. 문제는 데이터 삽입·수정·삭제가 발생하면 이 균형이 깨질 수 있다는 점이다. 따라서 DB 엔진은 데이터 변경 이후 트리의 균형을 다시 맞추는 리밸런싱(Rebalancing) 작업을 추가로 수행해야 한다. 이 과정에서 노드 분할(Split), 병합(Merge), 키 재배치 등의 연산이 발생하고, 인덱스가 많을수록 변경해야 할 트리도 많아지기 때문에 쓰기 성능은 읽기 성능보다 느릴 수밖에 없다.

    연산별 B-Tree 동작 비교

    연산 B-Tree 내부 동작 추가 비용 발생 여부
    SELECT (읽기) 루트에서 리프 노드까지 깊이 탐색만 수행한다. 없음 (O(log N) 탐색만)
    INSERT (쓰기) 적절한 위치에 키를 삽입하고, 노드가 가득 차면 분할(Split)하여 균형을 재조정한다. 있음 (리밸런싱)
    UPDATE (쓰기) 기존 키 삭제 후 신규 키 삽입 과정을 거친다. (두 번의 트리 조작) 있음 (삭제 + 삽입)
    DELETE (쓰기) 키를 제거한 뒤, 노드가 너무 비면 병합(Merge)하여 균형을 재조정한다. 있음 (리밸런싱)

    2. 데이터베이스 이중화 (Replication)

    Master / Slave(Replica) 구조

    실제 서비스에서는 읽기(SELECT) 요청이 쓰기(INSERT/UPDATE/DELETE) 요청보다 압도적으로 많다. 이 특성을 활용한 것이 데이터베이스 이중화(Replication) 전략이다. 쓰기 전담 서버(Master)는 적게 두고, 읽기 전담 서버(Slave/Replica)를 여러 개 배치하여 읽기 부하를 분산시킨다.

    Master / Slave 역할 분담 Master DB: 모든 쓰기(INSERT, UPDATE, DELETE) 연산을 전담한다. 데이터 변경이 발생하면 변경 로그(Binary Log)를 Slave에게 전파한다.

    Slave(Replica) DB: Master로부터 복제된 데이터를 갖고 있으며, 읽기(SELECT) 요청만 처리한다. 서비스의 읽기 처리량이 늘어날수록 Slave를 추가로 늘려 대응한다.
    장애 발생 시 자동 승격 (Failover) Master DB가 장애로 다운되면, Slave들 중 하나가 새로운 Master로 자동 승격(Failover)된다. 이 과정은 Orchestrator, ProxySQL 등 HA(High Availability) 도구가 자동으로 처리하며, 서비스 중단 시간을 최소화한다. 승격된 이후에는 나머지 Slave들이 새로운 Master를 바라보도록 재설정된다.
    구분 역할 특징
    Master DB 쓰기 전담 (INSERT, UPDATE, DELETE) 단수 또는 소수 운영. 변경 내용을 Slave에게 복제한다.
    Slave DB 읽기 전담 (SELECT) 다수 운영 가능. 서비스 트래픽에 따라 수평 확장(Scale-Out)한다.
    Failover Master 장애 시 대체 Slave 중 하나가 Master로 승격된다. HA 도구가 자동으로 처리한다.

    3. 서브쿼리 심화

    비연관 서브쿼리 (Non-Correlated Subquery)

    비연관 서브쿼리는 바깥쪽(메인) 쿼리와 독립적으로 실행될 수 있는 서브쿼리이다. 안쪽 서브쿼리가 먼저 실행되어 결과를 만들고, 그 결과를 바깥쪽 메인 쿼리가 조건으로 사용한다. 서브쿼리는 단 한 번만 실행되므로, 연관 서브쿼리에 비해 성능이 좋다.

    실행 순서 ① 안쪽 서브쿼리 먼저 실행 → 결과값 반환 → ② 바깥쪽 메인 쿼리가 그 결과를 사용하여 실행. 서브쿼리는 메인 쿼리의 컬럼을 참조하지 않는다. 
    -- 비연관 서브쿼리 예시: 평균 급여보다 높은 직원 조회
SELECT employee_name, salary
FROM EMPLOYEES
WHERE salary > (SELECT AVG(salary) FROM EMPLOYEES); -- 1회만 실행됨

    연관 서브쿼리 (Correlated Subquery)

    연관 서브쿼리는 바깥쪽(메인) 쿼리의 컬럼을 참조하기 때문에, 서브쿼리 단독으로는 실행이 불가능하다. 메인 쿼리가 한 행씩 처리될 때마다, 해당 행의 값을 이용해 서브쿼리가 반복 실행된다. 즉, 메인 쿼리가 N번 실행되면 서브쿼리도 N번 실행된다.

    ⚠️ 성능 주의 메인 쿼리 결과 행이 많아질수록 서브쿼리도 그만큼 반복 실행된다. 대량의 데이터에서 연관 서브쿼리를 무분별하게 사용하면 심각한 성능 저하가 발생할 수 있다. 가능하다면 JOIN + GROUP BY로 대체하는 것을 권장한다.
    실행 순서 ① 바깥쪽 메인 쿼리가 행을 하나 읽음 → ② 그 행의 컬럼 값을 사용해 서브쿼리 실행 → ③ 메인 쿼리의 다음 행으로 이동, 반복

    아래는 사용자별 작성 게시물 수를 연관 서브쿼리로 조회하는 예시이다.

    -- 연관 서브쿼리 예시: 사용자별 게시물 수 조회
-- 메인 쿼리의 u.user_id를 서브쿼리 내부에서 참조하므로 '연관' 서브쿼리이다.
SELECT u.username,
       (SELECT COUNT(*) FROM POSTS p WHERE p.user_id = u.user_id) AS "작성 게시물 수"
FROM USERS u;

    GROUP BY + LEFT JOIN으로 대체하기

    위 연관 서브쿼리는 LEFT JOIN과 GROUP BY를 조합하면 서브쿼리 없이 동일한 결과를 가져올 수 있다. 일반적으로 이 방식이 옵티마이저가 더 효율적인 실행 계획을 수립하기 쉬워 성능이 더 우수하다. 

    -- GROUP BY + LEFT JOIN 대체 쿼리
-- 게시물이 없는 사용자도 0으로 표시하기 위해 LEFT JOIN을 사용한다.
SELECT u.username, COUNT(p.user_id) AS "작성 게시물 수"
FROM USERS u
LEFT JOIN POSTS p ON u.user_id = p.user_id
GROUP BY u.user_id, u.username;
    방식 실행 방식 성능 특성
    연관 서브쿼리 메인 쿼리 행 수만큼 서브쿼리 반복 실행 행이 많아질수록 성능 급격히 저하된다.
    JOIN + GROUP BY 한 번의 조인으로 집계 처리 옵티마이저 최적화에 유리하며 성능이 더 우수하다.

    4. 스칼라 서브쿼리 (Scalar Subquery)

    정의와 특성

    스칼라 서브쿼리는 SELECT 절에 사용되는 서브쿼리로, 반드시 단일 행(Single Row), 단일 컬럼(Single Column), 즉 값 하나만을 반환해야 한다. 메인 쿼리의 한 행이 처리될 때마다 해당 행과 관련된 값을 계산하여 새로운 컬럼처럼 결과에 추가하는 역할을 한다. JOIN을 사용하기에는 구조가 번거롭거나 복잡할 때, 또는 각 행마다 독립적으로 계산된 단일 값을 컬럼으로 표시하고 싶을 때 유용하다. 

    -- 스칼라 서브쿼리 예시: SELECT 절에서 단일 값 반환
-- u.user_id는 바깥 쿼리(USERS u)를 참조하므로 이 서브쿼리는 연관 서브쿼리이기도 하다.
SELECT
    u.username,
    (SELECT COUNT(*) FROM POSTS p WHERE p.user_id = u.user_id) AS "작성 게시물 수"
FROM USERS u;
    ⚠️ 단일 행 반환 위반 시 오류 발생 스칼라 서브쿼리가 2개 이상의 행을 반환하면 런타임 오류(Subquery returns more than 1 row)가 발생한다. 반드시 COUNT(), SUM(), MAX(), MIN() 등 집계 함수를 사용하거나, LIMIT 1을 추가하여 단일 값 반환을 보장해야 한다.

    5. 서브쿼리 종류 비교 정리

    서브쿼리 종류 사용 위치 실행 방식 성능 특성
    비연관 서브쿼리 WHERE / HAVING 절 등 메인 쿼리와 독립적으로 실행. 단 1회만 실행된다. 성능이 가장 우수하다.
    연관 서브쿼리 WHERE / HAVING 절 등 메인 쿼리의 컬럼을 참조. 메인 쿼리 행 수만큼 반복 실행된다. 대량 데이터에서 성능 저하가 심각하다.
    스칼라 서브쿼리 SELECT 절 반드시 단일 값(1행 1열)만 반환. 연관 서브쿼리의 일종이다. 성능적 이점이 크지 않다. JOIN + GROUP BY 대체를 권장한다.
    실무 전략: 서브쿼리 사용 우선순위 1순위: JOIN + GROUP BY — 옵티마이저가 최적화하기 쉽고 성능이 가장 우수하다. 대부분의 집계 쿼리는 이 방식으로 작성할 수 있다.
    2순위: 비연관 서브쿼리 — 단 한 번만 실행되므로 연관 서브쿼리에 비해 성능 부담이 적다.
    3순위: 연관/스칼라 서브쿼리 — 가독성은 높지만 반복 실행으로 인한 성능 저하 위험이 있다. 항상 EXPLAIN으로 실행 계획을 확인해야 한다.

    마무리

    오늘은 데이터베이스의 내부 구조인 B-Tree / B+Tree가 쓰기 성능에 미치는 영향을 근본 원리 수준에서 이해했다. 단순히 "쓰기가 느리다"는 사실을 아는 것을 넘어, 리밸런싱이라는 구조적 비용이 발생하는 이유를 설명할 수 있게 되었다. 인덱스를 무분별하게 추가하면 쓰기 성능이 더욱 저하되는 이유도 같은 맥락으로 이해할 수 있다.

    데이터베이스 이중화(Replication)는 B-Tree의 특성을 역이용한 아키텍처 전략이다. 읽기가 압도적으로 많다는 특성을 반영해 Master/Slave 구조로 부하를 분산시키고, 장애 시 Slave가 Master로 승격되는 Failover 메커니즘으로 가용성을 보장한다. 이처럼 데이터베이스 내부 동작을 이해하면 아키텍처 설계 결정의 근거를 명확히 할 수 있다.

    서브쿼리 학습에서는 비연관, 연관, 스칼라 세 종류의 실행 방식과 성능 차이를 비교했다. 특히 연관 서브쿼리와 스칼라 서브쿼리는 직관적인 코드 표현이 가능하지만 성능적 이점은 크지 않으므로, 실무에서는 JOIN + GROUP BY로 우선 대체를 검토하고 EXPLAIN으로 실행 계획을 반드시 검증하는 습관을 가져야 한다.
