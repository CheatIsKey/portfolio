    1. Spring Security 개요

    인증(Authentication)과 인가(Authorization)

    Spring Security는 Spring 기반 애플리케이션의 인증(Authentication)과 인가(Authorization)를 담당하는 보안 프레임워크다. 인증은 "이 사용자가 누구인가"를 확인하는 과정이고, 인가는 "인증된 사용자가 무엇을 할 수 있는가"를 결정하는 과정이다.

    기능 설명
    인증 (Authentication) 사용자가 누구인지 확인한다.
    인가 (Authorization) 인증된 사용자가 무엇을 할 수 있는지 결정한다.
    CSRF 방어 세션 기반 환경에서 위조 요청을 방어한다.
    Remember-Me 자동 로그인 기능을 제공한다.

    세션 기반 인증 vs JWT 기반 인증

    구분 세션 기반 인증 JWT 기반 인증 (이 프로젝트)
    상태 관리 서버가 세션 저장소를 관리한다. 서버는 토큰을 저장하지 않는다. (Stateless)
    확장성 다중 서버 환경에서 세션 공유가 필요하다. 어떤 서버에서도 비밀키만 있으면 검증 가능하다.
    저장 위치 쿠키에 Session ID를 저장한다. Authorization 헤더에 Bearer 토큰으로 전달한다.
    CSRF 취약점이 존재한다. 쿠키 미사용으로 CSRF 방어가 불필요하다.

    주요 인증 방식 3가지

    방식 특징 용도
    인메모리 인증 설정이 간단하지만 서버 재시작 시 모든 데이터가 삭제된다. 학습/테스트용
    JDBC 인증 Spring Security 규격의 테이블 스키마를 기반으로 DB에서 인증한다. 간단한 DB 연동
    사용자 정의 인증 UserDetailsService를 직접 구현하여 커스텀 DB와 연동한다. 실무 표준

    ROLE_ 접두사 규칙

    Spring Security의 권한 네이밍 규칙 Spring Security는 권한을 관리할 때 ROLE_ 접두사를 기준으로 동작한다. hasRole("USER")를 호출하면 내부적으로 "ROLE_USER"를 확인한다. 따라서 Enum이나 DB에 ROLE_USER, ROLE_ADMIN처럼 접두사를 포함해서 저장해야 한다.

    요청 인가(Request Authorization) 주요 옵션

    HttpSecurity.authorizeHttpRequests()의 requestMatchers() 체이닝 옵션들이다.

    http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/").permitAll()               // 누구나 접근 가능
    .requestMatchers("/admin/**").hasRole("ADMIN")  // ADMIN 권한 필요
    .requestMatchers("/my/**").hasAnyRole("ADMIN", "USER") // 여러 권한 중 하나
    .anyRequest().authenticated()                   // 나머지는 인증된 사용자만
);
    메서드 설명
    .permitAll() 누구나 접근 가능하다.
    .authenticated() 인증된 사용자만 접근 가능하다.
    .hasRole("ROLE_NAME") 특정 역할을 가진 사용자만 접근 가능하다.
    .hasAnyRole("R1", "R2") 여러 역할 중 하나라도 있으면 접근 가능하다.
    .hasAuthority("AUTH") 권한 단위로 접근을 제어한다.
    .denyAll() 모든 접근을 차단한다.
    .access(...) 커스텀 권한 검사 로직을 적용한다.

    2. JWT 개요

    JWT 구조

    JWT(JSON Web Token)는 점(.)으로 구분된 세 파트(Header · Payload · Signature)로 구성된다. 

    eyJhbGciOiJIUzUxMiIsInR5cCI6IkpXVCJ9          ← Header
.eyJzdWIiOiJ1c2VyMSIsInJvbCI6WyJST0xFX1VTRVIiXX0  ← Payload
.서명값                                              ← Signature
    파트 역할 예시
    Header 알고리즘 타입(alg), 토큰 타입(typ)을 담는다. { "alg": "HS512", "typ": "JWT" }
    Payload 실제 데이터(Claims)를 담는다. 누구나 디코딩 가능하므로 민감 정보는 넣지 않는다. { "sub": "user1", "rol": ["ROLE_USER"], "exp": ... }
    Signature Header + Payload + 비밀키로 생성한다. 위변조 감지에 사용하며 서버만 생성할 수 있다. 서버의 비밀키로 서명한 해시값

    이 프로젝트에서 사용하는 Payload 클레임

    클레임 키 설명
    sub (subject) 사용자 아이디(username)를 담는다.
    uid 커스텀 클레임: 사용자 아이디 (sub와 동일)
    rol 권한 목록을 담는다. (예: ["ROLE_USER"])
    exp 만료 시간을 Unix timestamp(ms)로 담는다.
    typ (헤더) 토큰 타입: JWT

    JWT 처리 흐름

    로그인 성공 시 — 토큰 발행 1. 사용자가 아이디/비밀번호로 로그인을 요청한다.
    2. 서버가 DB를 확인하고 사용자가 맞다면 JWT를 생성한다. Payload에 유저 ID와 권한을 담고, 서버만 아는 비밀키(Secret Key)로 Signature를 만든다.
    3. 서버가 클라이언트에게 토큰을 응답한다.
    이후 API 요청 시 — 토큰 전달 & 검증 1. 클라이언트가 요청 헤더에 토큰을 첨부한다. (Authorization: Bearer <JWT>)
    2. 서버가 헤더에서 Bearer 를 제거한 후 JWT를 추출한다.
    3. 비밀키로 Signature를 직접 계산하여 전달받은 Signature와 비교한다. 일치하면 정상 토큰이고, 불일치하면 변조된 토큰으로 거절한다.
    4. Signature가 맞으면 exp(만료시간)를 확인한다.
    5. 검증 완료 후 Payload의 권한(rol)을 보고 인가 처리를 수행한다.

    3. 인증 처리 흐름

    Spring Security 내부 인증 흐름 

    사용자 요청
    │
    ▼
Authentication Filter                 ← (1) 요청에서 로그인 정보 추출, Authentication 토큰 생성
(UsernamePasswordAuthenticationFilter)
    │
    ▼
Authentication Manager                ← (2) 토큰을 처리할 수 있는 Provider 탐색 후 위임
    │
    ▼
Authentication Provider               ← (3) 실제 인증 수행
    │
    ├──▶ UserDetailsService           ← (4-1) username으로 DB 조회 → UserDetails 반환
    │
    └──▶ PasswordEncoder              ← (4-2) 입력 비밀번호 vs DB 비밀번호 검증
    │
    ▼
SecurityContextHolder                 ← (5) 인증 성공 시 Authentication 객체 저장
    핵심 요약 흐름은 요청 → Filter → Manager → Provider → UserDetailsService → SecurityContext 순으로 이어진다. 이 중 개발자가 직접 구현해야 하는 영역은 UserDetailsService이며, 나머지는 Spring Security가 자동으로 처리한다.

    Client (POST /auth/login)
    │  username + password (JSON Body)
    ▼
LoginController
    │  UsernamePasswordAuthenticationToken 생성
    ▼
AuthenticationManager.authenticate()
    │
    ▼
CustomUserDetailsService.loadUserByUsername()
    │  DB 조회 → UserDetails 반환
    │  비밀번호 검증은 Spring Security 자동 처리
    ▼
인증 성공 → JwtTokenProvider.createToken()
    │  username + roles → JWT 생성
    ▼
Client ← JWT 응답

    이 프로젝트의 JWT 인증 요청 흐름 

    Client (GET /user/info)
    │  Authorization: Bearer <JWT>
    ▼
JwtAuthenticationFilter.doFilterInternal()
    │  헤더에서 Bearer 제거 → JWT 추출
    ▼
JwtTokenProvider.validateToken()
    │  서명 검증 + 만료 시간 확인
    ▼ (유효한 경우)
JwtTokenProvider.getAuthentication()
    │  클레임 추출 → Authentication 객체 생성
    ▼

    │  인증 정보 저장
    ▼

    │  hasRole() 등 경로별 권한 확인
    ▼ (통과 시)
Controller 처리
    │  @AuthenticationPrincipal UserDetails 활용
    ▼
Client ← 응답

    JwtProps.java

    @ConfigurationProperties를 통해 application.properties의 JWT 관련 값을 Java 객체로 자동 바인딩한다. Spring Boot 3.x에서는 record 타입으로 정의하는 것이 간결하고 권장되는 방식이다.

    ⚠️ 반드시 확인할 사항 2가지 1. @ConfigurationPropertiesScan 누락 시 빈 등록 실패: 메인 클래스에 @ConfigurationPropertiesScan을 추가하지 않으면 이 record가 빈으로 등록되지 않는다.
    2. 케밥-케이스 자동 변환: 프로퍼티의 kebab-case가 record 필드의 camelCase로 자동 매핑된다. (예: secret-key → secretKey) 
    // JwtProps.java
@ConfigurationProperties(prefix = "com.joeun.jwt")
public record JwtProps(
    String secretKey,               // com.joeun.jwt.secret-key와 매핑
    long accessTokenExpiration,     // 액세스 토큰 유효 기간 (ms)
    long refreshTokenExpiration     // 리프레시 토큰 유효 기간 (ms)
) {}

    SecurityConstants.java

    코드 여러 곳에서 반복 사용되는 문자열 리터럴(매직 스트링)을 상수로 모아 관리한다. Authorization, Bearer 같은 문자열은 오타가 발생하기 쉬우므로 반드시 상수로 정의해야 한다.

public final class SecurityConstants {

    private SecurityConstants() { } // 인스턴스화 방지

    // HTTP 헤더 관련 상수
    public static final String TOKEN_HEADER = "Authorization";
    public static final String TOKEN_PREFIX = "Bearer "; // 뒤의 공백 포함

    // JWT 토큰 속성 관련 상수

    public static final String TOKEN_ROLE_CLAIM = "rol"; // 권한 클레임 키

    // 기타
    public static final String LOGIN_URL = "/login";
}

    JwtTokenProvider.java

    JWT와 관련된 모든 핵심 기능(생성, 파싱, 검증, 인증 객체 생성)을 캡슐화한 컴포넌트다. @Component로 빈 등록하여 다른 클래스에서 주입받아 사용한다.

    JJWT 0.12.x 주요 변경 사항 (구버전과 다른 부분) 구버전 코드를 그대로 사용하면 deprecated 경고 또는 컴파일 오류가 발생한다. 아래 변경 사항을 반드시 확인한다.

    Jwts.parserBuilder() → Jwts.parser()
    .setExpiration() → .expiration() / .setSubject() → .subject()
    SignatureAlgorithm.HS512 → Jwts.SIG.HS512
    헤더 설정: .setHeaderParam() → .header().add().and()
    검증 키 설정: .setSigningKey() → .verifyWith() 
    // JwtTokenProvider.java
@Slf4j
@Component
public class JwtTokenProvider {

    private final JwtProps jwtProps;

    public JwtTokenProvider(JwtProps jwtProps) {
        this.jwtProps = jwtProps;
    }

    // 1. JWT 토큰 생성
    public String createToken(String userId, List<String> roles) {
        byte[] signingKey = jwtProps.secretKey().getBytes(StandardCharsets.UTF_8);
        SecretKey key = Keys.hmacShaKeyFor(signingKey);

        // Spring Security 내부 관리용 권한(FACTOR_PASSWORD 등)을 제외하고
        // "ROLE_"로 시작하는 값만 필터링하여 토큰에 담는다.
        List<String> filteredRoles = roles.stream()
                .filter(role -> role.startsWith("ROLE_"))
                .toList();

        return Jwts.builder()
                .header().add("typ", SecurityConstants.TOKEN_TYPE).and()
                .subject(userId)
                .claim("uid", userId)
                .claim("rol", filteredRoles)
                .expiration(new Date(System.currentTimeMillis()
                        + jwtProps.accessTokenExpiration()))
                .signWith(key, Jwts.SIG.HS512)
                .compact();
    }

    // 2. 토큰 파싱 및 클레임 추출
    public Claims getClaims(String jwt) {
        byte[] signingKey = jwtProps.secretKey().getBytes(StandardCharsets.UTF_8);
        SecretKey key = Keys.hmacShaKeyFor(signingKey);

        return Jwts.parser()
                .verifyWith(key)
                .build()
                .parseSignedClaims(jwt)
                .getPayload();
    }

    // 3. 토큰 유효성 검증
    public boolean validateToken(String jwt) {
        try {
            getClaims(jwt); // 파싱 성공 = 유효한 토큰
            return true;
        } catch (SecurityException | MalformedJwtException e) {
            log.error("잘못된 JWT 서명입니다.");
        } catch (ExpiredJwtException e) {
            log.error("만료된 JWT 토큰입니다.");
        } catch (UnsupportedJwtException e) {
            log.error("지원되지 않는 JWT 토큰입니다.");
        } catch (IllegalArgumentException e) {
            log.error("JWT 토큰이 비어있거나 잘못되었습니다.");
        }
        return false;
    }

    // 4. Spring Security 인증 객체 생성 (DB 재조회 없이 클레임만으로 처리)
    public Authentication getAuthentication(String jwt) {
        Claims claims = getClaims(jwt);

        List<String> roles = (List<String>) claims.get("rol");
        List<SimpleGrantedAuthority> authorities = roles.stream()
                .map(SimpleGrantedAuthority::new)
                .toList();

        UserDetails principal = new User(claims.getSubject(), "", authorities);
        return new UsernamePasswordAuthenticationToken(principal, jwt, authorities);
    }
}

    JwtAuthenticationFilter.java

    모든 HTTP 요청에 대해 한 번씩 실행되는 필터다. OncePerRequestFilter를 상속받아 구현한다.

    ⚠️ @Component를 붙이지 않는 이유 @Component로 등록하면 Spring의 기본 필터 체인과 Security 필터 체인 양쪽에 중복 등록되어 필터가 두 번 실행되는 문제가 발생한다. SecurityConfig에서 addFilterBefore()로 직접 등록하는 방식만 사용해야 한다. 
    // JwtAuthenticationFilter.java
@RequiredArgsConstructor
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    private final JwtTokenProvider jwtTokenProvider;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain)
            throws ServletException, IOException {

        // 1. 요청 헤더에서 JWT 토큰 추출 ("Bearer " 제거 후 7자리 슬라이싱)
        String bearerToken = request.getHeader(SecurityConstants.TOKEN_HEADER);
        String jwt = null;

        if (StringUtils.hasText(bearerToken)
                && bearerToken.startsWith(SecurityConstants.TOKEN_PREFIX)) {
            jwt = bearerToken.substring(7);
        }

        // 2. 토큰이 존재하고 유효하다면 SecurityContext에 인증 정보를 저장한다.
        if (StringUtils.hasText(jwt) && jwtTokenProvider.validateToken(jwt)) {
            Authentication authentication = jwtTokenProvider.getAuthentication(jwt);
            SecurityContextHolder.getContext().setAuthentication(authentication);
        }

        // 3. 반드시 다음 필터로 요청을 넘긴다. (없으면 요청이 중단됨)
        filterChain.doFilter(request, response);
    }
}

    SecurityConfig.java

    Spring Security의 모든 보안 설정을 정의하는 핵심 파일이다. Spring Boot 3.x에서는 WebSecurityConfigurerAdapter 상속 방식이 완전히 제거되었으므로, SecurityFilterChain 빈을 직접 등록하는 방식만 사용한다. 

    // SecurityConfig.java
@Configuration
@RequiredArgsConstructor
@EnableWebSecurity
public class SecurityConfig {

    private final JwtTokenProvider jwtTokenProvider;

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {

        http.formLogin(AbstractHttpConfigurer::disable);   // 기본 로그인 폼 비활성화
        http.httpBasic(AbstractHttpConfigurer::disable);   // HTTP 기본 인증 비활성화
        http.csrf(AbstractHttpConfigurer::disable);        // REST API + JWT 환경: CSRF 불필요
        http.sessionManagement(management ->               // 세션 미사용 (Stateless)
                management.sessionCreationPolicy(SessionCreationPolicy.STATELESS));

        http.authorizeHttpRequests(auth -> auth
                .requestMatchers("/auth/**").permitAll()        // 로그인, 회원가입 무인증 허용
                .requestMatchers("/admin/**").hasRole("ADMIN")  // 관리자 전용 경로
                .requestMatchers("/user/**").hasRole("USER")
                .anyRequest().authenticated()                   // 그 외 모든 요청은 인증 필요
        );

        // JWT 필터를 UsernamePasswordAuthenticationFilter 앞에 등록한다.
        http.addFilterBefore(new JwtAuthenticationFilter(jwtTokenProvider),
                UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }

    @Bean
    public AuthenticationManager authenticationManager(
            AuthenticationConfiguration authenticationConfiguration) throws Exception {
        return authenticationConfiguration.getAuthenticationManager();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}

    CustomUserDetailsService.java

    Spring Security가 로그인 처리 시 내부적으로 호출하는 인터페이스를 구현한다. DB에서 사용자를 조회하고 Spring Security가 이해하는 UserDetails 객체로 변환해 반환한다.

    비밀번호 검증은 직접 하지 않는다 UserDetails를 반환하기만 하면 Spring Security가 내부적으로 PasswordEncoder를 사용해 자동으로 비교한다. 개발자가 직접 matches()를 호출할 필요가 없다. 
    // CustomUserDetailsService.java
@Service
@RequiredArgsConstructor
public class CustomUserDetailsService implements UserDetailsService {

    private final UserRepository userRepository;

    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        User user = userRepository.findByUsername(username)
                .orElseThrow(() ->
                    new UsernameNotFoundException("해당 유저를 찾을 수 없습니다: " + username));

        // 암호화된 비밀번호를 그대로 넘겨야 한다. Spring Security가 직접 검증한다.
        return org.springframework.security.core.userdetails.User.builder()
                .username(user.getUsername())
                .password(user.getPassword())
                .authorities(user.getRoles().stream()
                        .map(role -> role.name())
                        .toArray(String[]::new))
                .build();
    }
}

@Entity
@Getter
@Table(name = "users")  // 'user'는 MySQL 예약어이므로 'users'로 지정한다.
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class User {

    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(unique = true)
    private String username;

    private String password; // BCrypt로 암호화된 값이 저장된다.

    @ElementCollection(fetch = FetchType.EAGER)
    @Enumerated(EnumType.STRING) // Enum의 이름을 문자열로 DB에 저장한다.
    private List<Role> roles;

    @Builder
    public User(String username, String password, List<Role> roles) {
        this.username = username;
        this.password = password;
        // roles가 null이거나 비어있으면 기본값 ROLE_USER를 할당한다.
        this.roles = (roles != null && !roles.isEmpty() ? roles : List.of(Role.ROLE_USER));
    }
}

    // LoginController.java
@Slf4j
@RestController
@RequiredArgsConstructor
public class LoginController {

    private final AuthenticationManager authenticationManager;
    private final JwtTokenProvider jwtTokenProvider;

    @PostMapping("/auth/login")
    public ResponseEntity<?> login(@RequestBody AuthenticationRequest request) {

        // 1. 아이디와 비밀번호를 담은 인증 토큰을 생성한다.
        UsernamePasswordAuthenticationToken authenticationToken =
                new UsernamePasswordAuthenticationToken(
                    request.username(), request.password());

        // 2. AuthenticationManager가 CustomUserDetailsService를 호출해 비밀번호를 검증한다.
        Authentication authentication =
                authenticationManager.authenticate(authenticationToken);

        // 3. 인증 성공 후 권한 정보를 담아 JWT를 생성한다.
        List<String> roles = authentication.getAuthorities().stream()
                .map(GrantedAuthority::getAuthority)
                .toList();

        String jwt = jwtTokenProvider.createToken(request.username(), roles);
        return ResponseEntity.ok(jwt);
    }
}

    // UserService.java
@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)
public class UserService {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;

    @Transactional
    public CreateUserResponseDto create(CreateUserRequestDto request) {
        if (userRepository.existsByUsername(request.username())) {
            throw new IllegalArgumentException("해당 유저는 이미 존재합니다.");
        }
        String encodedPassword = passwordEncoder.encode(request.password());
        User user = new User(request.username(), encodedPassword, null); // null → 기본값 ROLE_USER
        User savedUser = userRepository.save(user);
        return new CreateUserResponseDto(savedUser.getId(), savedUser.getUsername());
    }
}

    boolean existsByUsername(String username);       // 중복 가입 검사 시 사용
}

@Bean
public UserDetailsService userDetailsService(DataSource dataSource) {
    JdbcUserDetailsManager manager = new JdbcUserDetailsManager(dataSource);
    manager.setUsersByUsernameQuery(
        "SELECT user_id, user_password, active FROM my_users WHERE user_id = ?");
    manager.setAuthoritiesByUsernameQuery(
        "SELECT user_id, role_name FROM my_roles WHERE user_id = ?");
    return manager;
}

    직접 구현 방식 — 커스텀 UserDetails 확장 예시

    UserDetails를 직접 구현하면 displayName, customerId처럼 추가 정보도 자유롭게 담을 수 있다. 이것이 실무에서 직접 구현 방식을 선택하는 핵심 이유이다. 

    // 추가 정보를 담은 커스텀 UserDetails 구현 예시
@Data
public class CustomerUserDetails implements UserDetails {
    private String username;
    private String password;
    private List<SimpleGrantedAuthority> authorities;
    private String displayName;  // 추가 정보 자유롭게 정의 가능
    private int customerId;

    public CustomerUserDetails(Customer customer, List<Authority> authorities) {
        this.username = customer.getName();
        this.password = customer.getPassword();
        this.customerId = customer.getId();
        this.displayName = customer.getName();
        this.authorities = authorities.stream()
                .map(authority -> new SimpleGrantedAuthority(authority.getAuthority()))
                .toList();
    }
}

    7. 자주 발생하는 오류 & 해결법

    오류 / 증상 원인 & 해결책
    MalformedJwtException (서명 검증 실패) 비밀키가 다르거나, 헤더에서 "Bearer " 제거 없이 토큰이 전달된 경우다. substring(7) 처리를 확인한다.
    401 Unauthorized (인증 필요 경로) JWT가 없거나 만료되었다. 토큰 재발급이 필요하다.
    403 Forbidden (권한 없음) 토큰은 유효하나 해당 경로의 접근 권한이 없다. SecurityConfig의 권한 설정을 확인한다.
    JwtProps 빈 등록 실패 메인 클래스에 @ConfigurationPropertiesScan이 없다. 반드시 추가해야 한다.
    FACTOR_PASSWORD 권한이 토큰에 포함됨 createToken()에서 "ROLE_"로 시작하는 권한만 필터링하는 로직이 누락되었다.
    필터가 두 번 실행됨 JwtAuthenticationFilter에 @Component가 붙어있다. 제거 후 SecurityConfig에서만 등록해야 한다.
    user 테이블명 예약어 충돌 MySQL에서 user는 예약어다. @Table(name = "users")로 테이블명을 변경한다.
    WebSecurityConfigurerAdapter 관련 오류 Spring Boot 3.x에서 완전히 제거되었다. SecurityFilterChain 빈 방식으로 변경해야 한다.

    마무리

    오늘은 Spring Security와 JWT를 결합한 인증 시스템의 전체 구조를 처음부터 끝까지 직접 구현하며 학습했다. 단순히 어노테이션을 붙이는 수준을 넘어, Filter → Manager → Provider → UserDetailsService로 이어지는 Spring Security 내부 인증 흐름을 다이어그램으로 정리하고 각 역할의 경계를 명확히 이해할 수 있었다.

    특히 JwtAuthenticationFilter에 @Component를 붙이지 않아야 하는 이유, JJWT 0.12.x의 API 변경 사항, FACTOR_PASSWORD 권한 필터링 누락 문제처럼 실제로 마주치기 쉬운 함정들을 오류 목록으로 정리한 것이 특히 유용했다. ROLE_ 접두사 규칙, 세션리스(Stateless) 설정, CSRF 비활성화의 이유 등 각 설정이 왜 필요한지 근거를 이해하고 적용했다는 점이 오늘의 핵심 성과다.

    UserDetailsService 구현 방식 3가지(인메모리·JDBC·직접 구현)를 비교하며 실무에서 직접 구현 방식을 선택하는 이유도 명확히 이해했다. 커스텀 UserDetails를 통해 추가 정보를 자유롭게 담을 수 있다는 유연성이 핵심 이유다. 다음 단계로는 Refresh Token 발급과 Access Token 재발급 흐름을 추가 구현하며 인증 시스템을 완성해 나가겠다.
