    1. OAuth란 무엇인가

    OAuth(Open Authorization) 정의

    OAuth는 애플리케이션(서버) 간에 사용자 정보에 대한 접근 허락을 개방적으로 처리하기 위한 개방형 인증 표준이다. 사용자가 직접 자신의 자격 증명(아이디/비밀번호)을 서드파티 애플리케이션에 제공하지 않고도, 특정 리소스에 대한 제한된 접근 권한만을 위임할 수 있는 방식으로 동작한다.

    OAuth의 핵심 목적 OAuth의 목적은 리소스 서버에 저장된 사용자 정보를 클라이언트 서버에 안전하게 공유하는 것이다. 이때 사용자는 직접 리소스 서버에서 자신의 정보를 가져와 서버에 제공하지 않는다. 대신 서버(클라이언트)가 사용자를 대신하여 사용자 정보를 가져오도록 위임받는 구조로 동작한다.

    OAuth의 주요 등장인물 (구성 요소)

    구성 요소 역할 실제 예시
    사용자 (Resource Owner) 자신의 정보를 보유하고 있는 주체. 서버에게 접근 권한을 위임한다. 카카오 계정을 보유한 실제 사람
    클라이언트 서버 (Client) 사용자 정보가 필요한 우리가 만드는 서버. 리소스 서버에 접근을 요청한다. 우리가 개발하는 Spring Boot 서버
    인증 서버 (Authorization Server) 사용자를 인증하고, 클라이언트에게 인가 코드 및 액세스 토큰을 발급한다. 카카오 인증 서버, Google 인증 서버
    리소스 서버 (Resource Server) 사용자 정보(닉네임, 이메일, 프로필 사진 등)를 보관한다. 액세스 토큰으로 접근을 허용한다. 카카오 API 서버, Google API 서버

    2. OAuth 인증 8단계 흐름

    전체 흐름 다이어그램

    ▲ OAuth 인증 흐름 구조도 · 출처: 알클 ALOHA CLASS

    단계별 상세 설명

    단계 행위 주체 → 대상 설명
    1 사용자 → 클라이언트 서버 사용자가 서비스의 소셜 로그인 버튼을 클릭한다.
    2 클라이언트 서버 → 인증 서버 클라이언트 서버가 인증 서버에 인가 코드를 요청한다. 이때 API KEY, redirect_uri, client_secret 등의 정보를 함께 포함한다.
    3 사용자 → 인증 서버 인증 서버가 로그인 페이지를 사용자에게 보여주고, 사용자가 직접 로그인한다. (카카오 로그인 화면, Google 로그인 화면 등)
    4 인증 서버 → 클라이언트 서버 사용자 인증이 성공하면 인증 서버가 클라이언트 서버에 인가 코드(Authorization Code)를 발급한다. 사용자 브라우저를 통해 redirect_uri로 안전하게 전달된다.
    5 클라이언트 서버 → 인증 서버 클라이언트 서버가 발급받은 인가 코드 + API KEY + redirect_uri + client_secret을 인증 서버에 제출하며 액세스 토큰을 요청한다.
    6 인증 서버 → 클라이언트 서버 인증 서버가 요청 정보를 검증한 후 클라이언트 서버에 액세스 토큰(Access Token)을 발급한다.
    7 클라이언트 서버 → 리소스 서버 클라이언트 서버가 발급받은 액세스 토큰을 사용하여 리소스 서버에 사용자 정보를 요청한다.
    8 리소스 서버 → 클라이언트 서버 리소스 서버가 액세스 토큰을 검증한 후 닉네임, 이메일, 프로필 사진 등 사용자 정보를 응답한다.

    3. 인가 코드 (Authorization Code)

    인가 코드란 무엇인가

    인가 코드는 사용자가 인증 서버에서 로그인에 성공했을 때, 인증 서버가 클라이언트 서버에 발급해주는 위임장 역할의 코드이다. 인가 코드 자체로는 사용자 정보에 접근할 수 없으며, 이후 액세스 토큰을 발급받기 위한 중간 단계의 증표로만 사용된다.

    인가 코드의 3가지 핵심 특성 1. 보안 전달: 인가 코드는 사용자의 브라우저를 통해 안전하게 전달된다. 서버 간 직접 통신이 아니라 redirect_uri를 통해 전달되기 때문에 액세스 토큰보다 노출 위험이 낮다.

    2. 단기 사용: 인가 코드는 일회성으로 설계되어 있다. 한 번 사용하면 무효화되며, 짧은 유효 기간을 가진다.

    3. 액세스 토큰 교환용: 클라이언트가 인증 서버에 인가 코드 + API KEY + redirect_uri + client_secret을 함께 제출해야만 액세스 토큰으로 교환할 수 있다.
    ⚠️ 인가 코드와 액세스 토큰을 혼동하지 말 것 인가 코드는 "액세스 토큰을 발급받을 자격이 있다"는 위임장이고, 액세스 토큰은 "리소스 서버에 실제로 접근할 수 있는 열쇠"이다. 인가 코드만으로는 사용자 정보에 접근할 수 없다.

    4. 액세스 토큰 (Access Token)

    액세스 토큰이란 무엇인가

    액세스 토큰은 사용자 정보가 있는 리소스 서버에 접근하기 위한 열쇠이다. 클라이언트 서버가 인가 코드를 인증 서버에 제출하고 교환받는 방식으로 발급되며, 이 토큰을 리소스 서버에 제시함으로써 사용자의 정보를 요청할 수 있다.

    액세스 토큰 발급 과정 상세 클라이언트 서버는 단순히 인가 코드만 보내는 것이 아니다. 인증 서버는 아래의 모든 정보를 검증한 후에만 액세스 토큰을 발급한다.

    - 인가 코드 (Authorization Code): 사용자가 실제로 로그인했다는 증거
    - API KEY (Client ID): 어떤 애플리케이션이 요청하는지 식별
    - redirect_uri: 사전에 등록된 URI와 일치하는지 검증 (CSRF 방지)
    - client_secret: 클라이언트 서버만 알고 있는 비밀 키 (서버 신원 증명)
    구분 인가 코드 액세스 토큰
    역할 액세스 토큰을 발급받기 위한 위임장 리소스 서버에 접근하기 위한 열쇠
    발급 주체 인증 서버 → 클라이언트 서버 인증 서버 → 클라이언트 서버
    사용 대상 인증 서버에 제출 (토큰 교환) 리소스 서버에 제출 (정보 요청)
    유효 기간 매우 짧다. (수십 초 ~ 수 분) 짧다. (수 분 ~ 수 시간)
    재사용 여부 일회성. 사용 후 즉시 무효화 유효 기간 내 반복 사용 가능

    5. 사용자 정보 (Resource)

    리소스 서버가 보유하는 사용자 정보

    리소스 서버는 닉네임, 프로필 사진, 이메일 등의 사용자 정보를 보관한다. 사용자의 프로필, 이미지, 문서 또는 기타 관련 데이터를 포함할 수 있으며, 이 정보는 사용자의 계정에 연결되어 있고 해당 사용자의 권한에 따라 다양한 형태로 저장된다.

    리소스 서버의 보안 원칙 리소스 서버는 인가된 클라이언트에게만 액세스를 허용하여 사용자의 개인정보를 보호한다. 유효한 액세스 토큰을 제시한 클라이언트에게만 응답하며, 토큰이 없거나 유효하지 않으면 정보를 반환하지 않는다. 일반적으로 OAuth 2.0 프로토콜을 사용하여 클라이언트가 인가를 받고 권한을 부여받은 후에만 리소스 서버에 접근할 수 있다.

    사용자 정보 요청 ~ 응답 흐름 요약

    액세스 토큰을 통한 사용자 정보 획득 과정 1. 클라이언트 서버는 인증 서버로부터 발급받은 액세스 토큰을 보유하게 된다. 이 토큰은 사용자로부터 권한을 위임받았음을 증명한다.

    2. 클라이언트 서버가 액세스 토큰을 포함하여 리소스 서버에 사용자 정보를 요청한다.

    3. 리소스 서버는 액세스 토큰을 검증하고, 유효하다면 해당 사용자의 닉네임, 이메일, 프로필 사진 등을 클라이언트 서버에 응답한다.

    4. 클라이언트 서버는 받아온 사용자 정보를 바탕으로 자체 서비스의 회원가입 또는 로그인 처리를 수행한다.

    6. 전체 흐름 종합 정리

    단계 키워드 핵심 내용
    1 ~ 3 소셜 로그인 요청 ~ 사용자 인증 사용자가 버튼 클릭 → 인증 서버 로그인 페이지 이동 → 사용자가 직접 로그인
    4 인가 코드 발급 사용자 인증 성공 → 인증 서버가 클라이언트 서버에 인가 코드 발급 (redirect_uri 통해 전달)
    5 ~ 6 액세스 토큰 교환 클라이언트 서버가 인가 코드 + 여러 인증 정보로 액세스 토큰 요청 → 인증 서버가 검증 후 발급
    7 ~ 8 사용자 정보 획득 클라이언트 서버가 액세스 토큰으로 리소스 서버에 사용자 정보 요청 → 리소스 서버 응답
    OAuth의 보안 설계 이유 단순히 사용자 정보를 바로 전달하지 않고 인가 코드 → 액세스 토큰 → 사용자 정보의 3단계 간접 접근 방식을 사용하는 이유는 보안 때문이다. 인가 코드는 브라우저를 통해 전달되므로 노출 위험이 있지만, 그 자체만으로는 아무 권한이 없다. 실제 권한을 가진 액세스 토큰은 서버 간 직접 통신으로만 교환되기 때문에 사용자 브라우저에 노출되지 않는다. 이 설계가 OAuth가 보안적으로 강력한 핵심 이유이다.

    마무리

    오늘은 OAuth(Open Authorization)의 개념과 8단계 인증 흐름을 처음부터 끝까지 정리했다. 단순히 "소셜 로그인"이라는 편의 기능으로만 바라보던 OAuth를 이제는 인증 서버, 리소스 서버, 클라이언트 서버의 역할 분담 구조와 함께 이해하게 되었다. 특히 사용자 정보를 직접 주고받지 않고 인가 코드 → 액세스 토큰 → 사용자 정보의 3단계 간접 접근 방식을 취하는 이유가 보안 설계 때문임을 명확히 이해한 것이 오늘의 핵심 성과다.

    인가 코드는 "액세스 토큰을 발급받을 자격이 있다는 위임장"이고, 액세스 토큰은 "리소스 서버에 실제로 접근하는 열쇠"라는 역할 구분도 명확해졌다. 인가 코드는 브라우저를 통해 전달되어 노출 위험이 있지만 그 자체로는 아무 권한이 없고, 액세스 토큰은 서버 간 직접 통신으로만 교환되어 브라우저에 노출되지 않는다. 이 설계 원칙이 OAuth가 보안적으로 신뢰받는 이유임을 이해했다.
