    1. MIME 타입

    MIME란 무엇인가

    MIME(Multipurpose Internet Mail Extensions)은 원래 이메일을 보낼 때 데이터의 종류를 알려주기 위한 규격으로 시작되었다. 오늘날에는 웹에서 데이터를 주고받을 때 "이건 이런 종류의 데이터야"라고 상대방에게 알려주는 표준 이름표로 사용된다. 쉽게 말하면 파일의 신분증이다. 컴퓨터가 파일을 올바르게 식별하고 처리할 수 있도록 돕는 표준 명세이다.

    MIME 타입 설명
    image/jpeg, image/png JPEG, PNG 형식의 이미지 파일
    image/gif, image/webp GIF, WebP 형식의 이미지 파일
    video/mp4 MP4 형식의 동영상 파일
    application/json JSON 형식의 데이터
    multipart/form-data 텍스트와 파일을 함께 전송할 때 사용하는 인코딩 방식

    2. multipart/form-data와 Boundary

    왜 multipart/form-data가 필요한가

    일반적인 HTTP 요청은 텍스트 데이터만 담도록 설계되어 있다. 그러나 게시물 작성처럼 글(텍스트)과 이미지(바이너리)를 동시에 전송해야 하는 경우, 성질이 다른 두 종류의 데이터를 하나의 요청 바디에 담아야 한다. 이때 사용하는 것이 multipart/form-data 인코딩 방식이다.

    Boundary (경계선): 칸막이 역할 multipart/form-data는 서로 다른 종류의 데이터를 하나의 요청 바디에 담기 위해 Boundary(경계선)를 사용한다. Boundary는 각 데이터 파트를 구분하는 칸막이 역할을 한다. 서버는 이 Boundary를 기준으로 요청 바디를 잘라서 각 파트(글 내용, 이미지 파일 등)를 분리해 읽는다.

    예를 들어 게시물 작성 요청 하나에 본문 텍스트 1개, 이미지 파일 3개가 담겨있을 때, 서버는 Boundary 덕분에 "이건 텍스트", "이건 첫 번째 이미지", "이건 두 번째 이미지"처럼 각 파트를 명확히 구분할 수 있다.

    Spring에서 파일 수신: MultipartFile

    Spring은 multipart/form-data로 전송된 파일을 MultipartFile 인터페이스로 받는다. 단일 파일과 다중 파일 모두 지원하며, 파라미터 타입만 다르게 선언하면 된다. 

    // 단일 파일 업로드
@RequestPart MultipartFile image

// 다중 파일 업로드
@RequestPart List<MultipartFile> images
    ⚠️ @RequestBody와 @RequestPart를 함께 쓸 수 없다 글과 이미지를 함께 받을 때, formData는 글과 이미지들을 Boundary(경계선)로 나눠서 하나로 묶어 전송한다. 이때 @RequestParam과 @RequestBody를 동시에 사용할 수 없다. @RequestBody는 요청 바디 전체를 단일 JSON으로 읽으려 하기 때문에 multipart/form-data와 충돌이 발생한다.

    대신 @RequestPart를 사용해야 한다. @RequestPart는 Boundary로 나눠진 각 파트를 종류에 맞게 개별로 꺼내서 바인딩해 준다. 
    // ❌ 잘못된 방법: @RequestBody와 MultipartFile 혼용 불가
@PostMapping("/posts")
public ResponseEntity<?> createPost(
    @RequestBody PostCreateRequest request,  // JSON 바디 전체를 읽으려 함 → multipart와 충돌
    @RequestParam MultipartFile image
) { ... }

// ✅ 올바른 방법: @RequestPart로 각 파트를 분리해서 수신
@PostMapping("/posts")
public ResponseEntity<?> createPost(
    @RequestPart("request") PostCreateRequest request, // JSON 파트
    @RequestPart("image") MultipartFile image          // 파일 파트
) { ... }

    3. 파일 저장 서비스 구현

    절대 해서는 안 되는 것: 경로 하드코딩

    파일을 로컬 디스크에 저장할 때 코드 안에 경로를 직접 문자열로 적으면 절대 안 된다. 하드코딩된 경로는 개발자 PC, 테스트 서버, 운영 서버마다 디렉토리 구조가 달라 즉시 오류가 발생하고, 경로를 바꾸려면 코드를 직접 수정하고 재배포해야 한다. 반드시 application.yml에 경로를 선언하고 @Value로 주입받아 사용해야 한다. 

    # application.yml
file:
  upload:
    location: ${user.home}/instagram-clone/uploads/  # 끝에 슬래시(/) 필수
    // ❌ 절대 금지: 경로 하드코딩
String fileDir = "C:/Users/admin/uploads/"; // 다른 환경에서 즉시 오류 발생

// ✅ 올바른 방법: @Value로 외부 설정 주입
@Value("${file.upload.location}")
private String fileDir;

    파일 저장 시 필수 보안 검증 3가지

    보안 항목 설명 구현 방법
    파일명 세탁 (UUID) 원본 파일명을 그대로 사용하면 악의적인 파일명으로 서버 파일을 덮어쓰거나 노출시킬 수 있다. UUID.randomUUID()로 고유한 파일명을 새로 생성하여 저장한다.
    경로 추적 금지 (Path Traversal) 파일명에 ../../etc/passwd 같은 경로 탐색 문자가 포함되면 서버의 민감한 파일에 접근할 수 있다. UUID로 파일명을 완전히 교체하면 경로 추적 문자가 원천 차단된다.
    확장자 화이트리스트 허용되지 않은 확장자(예: .exe, .jsp)의 파일이 업로드되면 서버에서 실행될 수 있다. 허용 확장자 목록을 정의하고, MIME 타입과 확장자를 모두 검증한다.
    MIME 타입과 확장자를 둘 다 검증하는 이유 MIME 타입만 검증하면 클라이언트가 헤더를 조작해 악성 파일을 image/jpeg로 위장하여 보낼 수 있다. 확장자만 검증하면 파일 내용 자체를 바꿔 업로드하는 방식으로 우회할 수 있다. 따라서 MIME 타입(Content-Type)과 확장자 화이트리스트를 모두 검증해야 두 가지 우회 공격을 동시에 막을 수 있다.

    파일 저장 서비스 전체 구현 코드 

    @Service
public class FileStorageService {

    // 허용 확장자 목록 (화이트리스트 방식)
    private static final Set<String> ALLOWED_EXTENSIONS =
            Set.of(".jpg", ".jpeg", ".png", ".gif", ".webp");

    @Value("${file.upload.location}")
    private String fileDir;

    // 서버 시작 시 업로드 디렉토리가 없으면 자동으로 생성한다.
    @PostConstruct
    public void init() {
        File dir = new File(fileDir);
        if (!dir.exists()) {
            dir.mkdirs();
        }
    }

    public String storeFile(MultipartFile multipartFile) throws IOException {

        // 1. 원본 파일명 추출 및 빈 값 검증
        String originalFilename = multipartFile.getOriginalFilename();
        if (originalFilename == null || originalFilename.isEmpty()) {
            throw new PostException(PostErrorCode.INVALID_FILE_EXTENSION);
        }

        // 2. MIME 타입 검증: "image/"로 시작하는 타입만 허용
        String contentType = multipartFile.getContentType();
        if (contentType == null || !contentType.startsWith("image/")) {
            throw new PostException(PostErrorCode.INVALID_FILE_EXTENSION);
        }

        // 3. 확장자 화이트리스트 검증 (대소문자 무관하게 소문자로 비교)
        String lowerOriginal = originalFilename.toLowerCase();
        if (ALLOWED_EXTENSIONS.stream().noneMatch(lowerOriginal::endsWith)) {
            throw new PostException(PostErrorCode.INVALID_FILE_EXTENSION);
        }

        // 4. UUID로 새 파일명 생성 (원본명 완전 폐기 → 보안)
        int extIndex = originalFilename.lastIndexOf(".");
        String ext = (extIndex == -1) ? "" : originalFilename.substring(extIndex);
        String storeFileName = UUID.randomUUID().toString() + ext;

        // 5. 전체 파일 저장 경로 생성 및 로컬 디스크에 저장
        String fullPath = fileDir + storeFileName;
        multipartFile.transferTo(new File(fullPath));

        // 6. 클라이언트가 접근할 정적 리소스 경로로 변환하여 반환 (WebMvcConfig 활용)
        return "/img/" + storeFileName;
    }
}

    4. WebMvcConfig — 로컬 파일을 URL로 서빙하기

    WebMvcConfig가 하는 일

    파일을 로컬 디스크에 저장했다고 해서 클라이언트가 바로 그 파일에 접근할 수 있는 것은 아니다. 서버 내부의 물리적 파일 경로(/home/user/instagram-clone/uploads/abc.jpg)는 외부 클라이언트가 직접 URL로 요청할 수 없는 서버 내부 영역이기 때문이다.

    WebMvcConfig의 역할: 파일 창고에 외부 출입구를 만들어주는 것 일반적으로 Spring은 resources/static 같은 classpath 내부의 파일만 정적 리소스로 제공한다. 그런데 업로드된 파일은 classpath 바깥, 즉 서버의 로컬 디스크에 저장되어 있다.

    WebMvcConfig는 "/img/**로 URL 요청이 오면, 이 로컬 폴더에서 파일을 꺼내서 응답해줘"라고 Spring에게 알려주는 설정이다. 쉽게 말하면, 서버 안에 있는 파일 창고에 외부에서 접근할 수 있는 출입구(URL)를 만들어주는 것이다.

    이 덕분에 클라이언트(프론트엔드)는 파일 저장 서비스에서 반환받은 /img/uuid파일명.jpg 경로를 그대로 <img src="/img/uuid파일명.jpg">에 사용할 수 있게 된다. 
    @Configuration
public class WebMvcConfig implements WebMvcConfigurer {

    @Value("${file.upload.location}")
    private String fileDir;

    @Override
    public void addResourceHandlers(ResourceHandlerRegistry registry) {
        // "/img/**"로 들어오는 모든 요청을 물리적 폴더(fileDir)로 연결한다.
        // "file:" 접두어를 반드시 붙여야 로컬 파일 시스템을 가리킨다.
        // 접두어 없이 경로만 적으면 classpath(JAR 내부)를 탐색하므로 파일을 찾지 못한다.
        registry.addResourceHandler("/img/**")
                .addResourceLocations("file:" + fileDir);
    }
}
    설정 항목 설명
    addResourceHandler("/img/**") /img/로 시작하는 모든 URL 요청을 이 핸들러가 처리하도록 등록한다.
    addResourceLocations("file:" + fileDir) 요청을 로컬 디스크의 fileDir 경로에서 찾아 응답한다. file: 접두어가 없으면 classpath를 탐색하므로 반드시 붙여야 한다.

    5. 물리 저장만으로는 부족하다 — DB 연계의 필요성

    로컬 파일 저장의 한계

    파일을 로컬 디스크에 저장하는 것은 파일의 물리적 보관만 해결한다. "어떤 게시물에 어떤 이미지가 연결되어 있는지", "누가 언제 올린 파일인지"와 같은 관계와 맥락 정보는 디스크에 저장할 수 없다. 이 정보를 관리하려면 반드시 DB를 함께 활용해야 한다.

    DB에 기록해야 하는 이유 설명
    관계의 기록 몇 번 게시물에 어떤 사진이 연결되어 있는지를 외래 키(FK)로 관리한다. 이 정보 없이는 게시물과 이미지를 함께 조회할 수 없다.
    빠른 검색 특정 게시물의 이미지, 특정 사용자가 올린 이미지를 인덱스를 활용해 빠르게 조회할 수 있다.
    메타데이터 관리 원본 파일명, 파일 크기, 업로드 시각, 업로더 정보 등 파일 자체로는 알 수 없는 부가 정보를 저장한다.
    파일 업로드의 올바른 전체 처리 흐름 1단계: @RequestPart로 MultipartFile을 받아 MIME 타입과 확장자 화이트리스트 검증을 수행한다.
    2단계: UUID로 파일명을 세탁하고 로컬 디스크에 저장한다.
    3단계: 저장된 파일의 접근 경로(/img/파일명)를 DB에 기록한다. 게시물 PK, 업로더 ID, 업로드 시각 등의 메타데이터도 함께 저장한다.
    4단계: 클라이언트가 게시물을 조회할 때 DB에서 이미지 경로를 함께 응답하면, 클라이언트는 해당 경로로 WebMvcConfig를 통해 이미지를 요청한다.

    마무리

    오늘은 Spring Boot에서 파일을 안전하게 업로드하고 클라이언트에게 서빙하는 전체 과정을 학습했다. MIME 타입이 파일의 신분증 역할을 한다는 것, multipart/form-data가 Boundary(경계선)를 이용해 성질이 다른 데이터들을 하나의 요청에 묶어 전송한다는 것이 오늘 학습의 출발점이었다. 특히 @RequestBody와 MultipartFile을 동시에 사용할 수 없기 때문에 @RequestPart를 사용해야 한다는 점은 실수하기 쉬운 부분이므로 반드시 기억해야 한다.

    파일 저장 서비스 구현을 통해 경로 하드코딩 금지, UUID 파일명 세탁, MIME 타입과 확장자 이중 검증이라는 세 가지 보안 원칙을 코드로 직접 적용했다. MIME 타입만 검증하거나 확장자만 검증하면 각각을 우회할 수 있기 때문에 두 가지를 모두 검증해야 한다는 이유까지 이해한 것이 핵심이다. WebMvcConfig는 서버 내부의 파일 창고에 외부 URL 출입구를 만들어주는 역할이며, file: 접두어 없이 경로만 적으면 classpath를 탐색해 파일을 찾지 못하는 점도 꼭 기억해야 한다.

    마지막으로 물리적 파일 저장과 DB 기록이 반드시 함께 이루어져야 한다는 점을 정리했다. 디스크는 파일의 실체를 보관하고, DB는 그 파일이 누구의 것이며 어느 게시물에 속하는지의 관계와 메타데이터를 관리한다.
