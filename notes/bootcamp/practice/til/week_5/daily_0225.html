    1. 동시성과 병렬성

    동시성 (Concurrency) vs 병렬성 (Parallelism)

    동시성 (Concurrency) 오른손 하나(싱글 스레드)로 두 가지 이상의 작업을 동시에 처리하는 것이다. 실제로 동시에 실행되는 것이 아니라, 매우 빠르게 번갈아 가며 실행되어 동시에 처리되는 것처럼 보이는 방식이다.
    병렬성 (Parallelism) 왼손은 키보드를 입력하고 오른손은 마우스를 누르는 것처럼, 서로 다른 작업자(멀티 스레드)가 각자의 작업을 동시에 수행하는 것이다. 물리적으로 여러 코어가 각각 독립된 작업을 진정한 의미에서 동시에 실행하는 방식이다.
    구분 동시성 (Concurrency) 병렬성 (Parallelism)
    스레드 수 싱글 스레드 멀티 스레드
    실행 방식 빠른 교차 실행으로 동시처럼 보임 물리적으로 진짜 동시 실행
    비유 오른손 하나로 두 가지 일을 번갈아 처리 양손이 각자 다른 일을 동시에 처리

    2. Proxy 심화: @Transactional과 final의 함정

    @Transactional이 붙은 메서드를 private으로 선언하면 오류가 나는 이유

    Spring은 @Transactional을 처리하기 위해 실제 클래스를 감싸는 Proxy(대리) 객체를 만든다. 클라이언트 코드가 서비스 메서드를 호출하면, 실제로는 이 프록시 객체가 먼저 호출을 가로채서 트랜잭션을 시작한 뒤 실제 메서드를 위임 호출하는 방식으로 동작한다.

    ⚠️ private 메서드에 @Transactional을 붙이면 안 되는 이유 Spring이 @Transactional을 처리하기 위해 만드는 프록시 객체는 외부에서 메서드를 호출해야 트랜잭션 처리가 가능하다. 그런데 private 메서드는 외부에서 접근 자체가 불가능하다. 프록시가 해당 메서드에 접근할 수 없으므로 트랜잭션이 적용되지 않고, 경우에 따라 오류가 발생한다.

    엔티티 클래스를 final로 선언하면 JPA 지연 로딩이 작동하지 않는 이유

    JPA의 지연 로딩(Lazy Loading)은 연관된 엔티티를 즉시 조회하지 않고, 실제로 해당 필드에 접근하는 시점에 DB를 조회하는 최적화 기법이다. 이를 구현하기 위해 JPA는 실제 엔티티 대신 프록시(가짜) 객체를 먼저 넣어두고, 나중에 실제 데이터로 채운다.

    ⚠️ final 클래스에 지연 로딩이 작동하지 않는 이유 JPA가 프록시를 만들 때 사용하는 CGLib(코드 생성 라이브러리)는 실제 엔티티 클래스를 상속받아서 프록시를 생성한다. 그런데 final class는 Java 언어 규칙상 상속이 불가능하다. 따라서 CGLib이 프록시를 만들 수 없고, 결과적으로 지연 로딩이 작동하지 않는다. 이것이 JPA 엔티티에 final 키워드를 붙이면 안 되는 근본적인 이유이다.

    JDK Proxy vs CGLib Proxy 비교

    구분 JDK Proxy CGLib Proxy
    프록시 생성 방식 인터페이스 기반으로 프록시를 생성한다. 상속 기반으로 프록시를 생성한다.
    조건 대상 클래스가 반드시 인터페이스를 구현해야 한다. 인터페이스 없이도 사용 가능하다. 단, final 클래스는 불가능하다.
    Spring에서의 사용 인터페이스가 있는 빈에 적용 인터페이스가 없는 빈 또는 JPA 엔티티에 적용
    사용 예 @Transactional (인터페이스 기반 서비스) JPA 지연 로딩 프록시, Spring 컴포넌트 프록시

    JPA가 내부적으로 사용하는 프록시 구현체

    구현체 역할
    PersistentBag Java의 기본 컬렉션(ArrayList 등)을 하이버네이트 전용 컬렉션으로 교체하여 지연 로딩을 지원한다. @OneToMany 등의 컬렉션 필드가 실제로는 이 객체로 감싸진다.
    HibernateProxy 엔티티 클래스를 상속받아 가짜 객체를 생성한다. 실제 데이터가 필요한 시점까지 DB 조회를 미루는 지연 로딩의 핵심 구현체이다.
    내부적으로 Proxy에 의해 구현된 객체가 래핑된다 @OneToMany로 선언한 List<Comment> 필드를 꺼내보면 실제 타입은 ArrayList가 아닌 PersistentBag이다. 마찬가지로 지연 로딩으로 설정된 @ManyToOne 엔티티는 실제 타입이 HibernateProxy를 상속한 가짜 객체다. 실제 필드에 접근하는 순간 이 프록시가 DB를 조회하여 진짜 데이터로 초기화된다. 이처럼 JPA는 개발자 코드를 건드리지 않고 내부적으로 프록시 객체로 래핑하는 방식으로 지연 로딩을 투명하게 구현한다.

    3. @Transactional 실무 전략 — 외부 API 지연과 Connection Pool 고갈

    @Transactional은 save가 있다고 무조건 붙이는 게 정답이 아니다

    @Transactional이 붙은 메서드는 실행 시작부터 끝날 때까지 DB Connection을 계속 점유한다. 대부분의 간단한 저장 로직에서는 문제가 없지만, 메서드 내부에 외부 API 호출이 포함되어 있다면 이야기가 달라진다.

    ⚠️ 외부 API 지연이 Connection Pool을 고갈시키는 원인 외부 API 응답이 지연되면, 해당 메서드가 끝나지 않으므로 DB Connection이 계속 점유된 채로 대기한다. 동시 요청이 많아지면 Connection Pool의 모든 Connection이 이 대기 상태로 묶이고, 새로운 요청은 Connection을 할당받지 못해 대기하거나 타임아웃 오류가 발생한다. 결국 서비스 전체가 장애 상태에 빠진다. 
    // ❌ 문제 있는 코드: 외부 API 호출이 포함된 메서드에 @Transactional 적용
@Transactional
public void createPost(PostCreateRequest request) {
    // DB Connection 점유 시작
    Post post = postRepository.save(new Post(request.title()));

    // 외부 API 호출 → 응답 지연 시 Connection을 계속 붙잡고 있음
    String externalData = externalApiClient.fetchData(post.getId());

    post.updateWithExternalData(externalData);
    postRepository.save(post);
    // DB Connection 반납
}

    해결 전략: @Transactional 제거 + 저장 로직 분리

    해결 방법은 외부 API 호출을 포함한 메서드에서 @Transactional을 제거하고, 각 DB 저장 작업을 개별적으로 수행한 뒤 즉시 Connection을 반납하도록 분리하는 것이다. 외부 API 응답을 받은 이후의 저장 로직은 별도의 서비스 클래스에 @Transactional을 붙인 메서드로 분리하여 호출한다. 

    // ✅ 해결 코드: 외부 API 호출과 DB 저장 로직을 분리

// 저장 전담 서비스 (별도 클래스로 분리, @Transactional 적용)
@Service
public class PostSaveService {

    private final PostRepository postRepository;

    @Transactional
    public Post saveInitial(String title) {
        // DB 저장 후 즉시 Connection 반납
        return postRepository.save(new Post(title));
    }

    @Transactional
    public void updateWithExternalData(Long postId, String externalData) {
        Post post = postRepository.findById(postId).orElseThrow();
        post.updateWithExternalData(externalData);
        // DB 저장 후 즉시 Connection 반납
    }
}

// 외부 API 호출을 포함한 서비스 (@Transactional 제거)
@Service
@RequiredArgsConstructor
public class PostService {

    private final PostSaveService postSaveService;
    private final ExternalApiClient externalApiClient;

    // @Transactional 없음 → DB Connection을 점유하지 않음
    public void createPost(PostCreateRequest request) {

        // 1. 저장 후 즉시 Connection 반납
        Post post = postSaveService.saveInitial(request.title());

        // 2. 외부 API 호출 (지연되어도 DB Connection에 영향 없음)
        String externalData = externalApiClient.fetchData(post.getId());

        // 3. 응답 받은 후 저장, 다시 즉시 Connection 반납
        postSaveService.updateWithExternalData(post.getId(), externalData);
    }
}
    핵심 원칙: @Transactional의 범위는 최소화해야 한다 @Transactional이 붙은 메서드가 실행되는 동안 DB Connection은 계속 점유된다. 외부 API 호출, 파일 I/O, 네트워크 통신처럼 시간이 오래 걸릴 수 있는 작업을 트랜잭션 범위 안에 포함시키면 안 된다. 트랜잭션은 순수하게 DB 작업만 담당하도록 범위를 최소화하고, 시간이 걸리는 외부 작업은 트랜잭션 바깥에서 처리한 뒤 결과를 받아서 저장하는 구조로 설계해야 한다.

    마무리

    오늘은 Spring과 JPA 내부에서 프록시가 어떻게 동작하는지, 그리고 그 동작 방식이 코드 작성에 어떤 제약을 만드는지를 깊이 이해했다. @Transactional과 private 메서드가 충돌하는 이유, final 클래스와 JPA 지연 로딩이 충돌하는 이유 모두 결국 프록시가 상속 또는 접근이 필요하다는 동일한 원리에서 비롯된다는 것을 연결지어 이해한 것이 오늘의 핵심 성과다.

    PersistentBag과 HibernateProxy는 JPA가 개발자 코드를 건드리지 않고 내부적으로 프록시 객체를 래핑하는 방식으로 지연 로딩을 구현하는 구체적인 수단임을 이해했다. JDK Proxy와 CGLib Proxy의 차이도 "인터페이스 기반이냐, 상속 기반이냐"라는 한 가지 기준으로 명확히 정리되었다.

    @Transactional의 실무 전략에서는 외부 API 지연이 DB Connection Pool을 고갈시키는 장애 시나리오를 이해하고, 저장 로직을 별도 서비스로 분리하여 트랜잭션 범위를 최소화하는 해결 방법을 코드로 정리했다. save가 있다고 무조건 @Transactional을 붙이는 것이 정답이 아니며, 트랜잭션 범위 안에 오래 걸리는 작업을 포함시키지 않는 것이 실무에서 반드시 지켜야 할 원칙임을 명심하겠다.
