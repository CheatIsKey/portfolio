    이슈 1. CascadeType.ALL과 SoftDelete 전략 충돌

    문제 발견

    Category 엔티티에 SoftDelete 전략(@SQLRestriction + is_deleted 컬럼)을 사용하면서 자식 컬렉션에 CascadeType.ALL을 설정하고 있었다. CascadeType.ALL은 REMOVE를 포함하기 때문에, JPA가 부모 엔티티를 삭제할 때 자식들에게 실제 DELETE SQL을 전파하는 구조였다. 

    // Category.java — 문제 코드
@OneToMany(mappedBy = "parent", cascade = CascadeType.ALL)
private List<Category> children = new ArrayList<>();

    이 상태에서 categoryRepository.delete(parent)가 호출되면 아래처럼 물리 삭제가 발생한다.

    DELETE FROM category WHERE id = 1;         -- 부모 물리 삭제
DELETE FROM category WHERE parent_id = 1;  -- 자식 전체 물리 삭제!

    is_deleted = true로 표시해야 하는데, DB 행 자체가 사라지게 된다.

    접근 방법 비교

    방법 설명 채택
    방법 A CascadeType.ALL 유지, deleteWithDescendants() 엔티티 메서드로만 삭제. 삭제 경로를 서비스 메서드 하나로 제한하면 실수를 줄일 수 있다. 그러나 repository.delete()를 어디서든 호출하면 REMOVE cascade가 여전히 동작하므로, 코드로 강제할 수 없다. ❌
    방법 B CascadeType에서 REMOVE만 제거하고 PERSIST · MERGE만 유지. JPA 레벨에서 물리 삭제 전파 자체를 원천 차단한다. 어디서 어떻게 호출해도 자식이 물리 삭제되는 경로가 없어진다. 삭제 책임을 온전히 애플리케이션 코드에 귀속시킬 수 있다. ✅
    채택 이유 SoftDelete 전략에서 CascadeType.REMOVE는 의미 자체가 없다. 이 프로젝트에서 "삭제"는 isDeleted = true 세팅이지 DB 행 제거가 아니기 때문이다. 방법 A는 팀 규칙으로만 강제하는 방식이라 실수 가능성이 있다. 방법 B는 REMOVE를 아예 제거해 JPA 레벨에서 물리 삭제 경로를 막는다. 
    // Category.java

// ❌ 수정 전
@OneToMany(mappedBy = "parent", cascade = CascadeType.ALL)
private List<Category> children = new ArrayList<>();

// ✅ 수정 후
@OneToMany(mappedBy = "parent", cascade = {CascadeType.PERSIST, CascadeType.MERGE})
private List<Category> children = new ArrayList<>();
    달성한 것 CascadeType.REMOVE 제거 → 어떤 경로로도 자식이 물리 삭제될 수 없게 됨
    PERSIST / MERGE 유지 → 부모 저장 시 새 자식 저장, 부모 병합 시 자식 변경사항 반영은 정상 동작
    ⚠️ 고려할 점 @SQLRestriction("is_deleted = false")는 children 컬렉션 로딩 시에도 적용된다. 즉 소프트 삭제된 자식은 children 리스트에 보이지 않는다. 향후 복구(restore) 로직 확장 시 별도 쿼리가 필요하다.

    이슈 2. 물리 삭제 메서드 차단 누락

    문제 발견

    이슈 1에서 Cascade REMOVE를 제거해도, JpaRepository가 기본 제공하는 delete() / deleteById() / deleteAll() 메서드는 여전히 호출 가능한 상태였다. 이 메서드들은 내부적으로 EntityManager.remove()를 호출해 물리 삭제를 실행한다. 

    // 이렇게 호출하면 여전히 물리 삭제됨
categoryRepository.delete(category);
categoryRepository.deleteById(1L);

    접근 방법 비교

    방법 설명 채택
    방법 A 팀 컨벤션 문서로만 제한. "delete() 계열 메서드 사용 금지"를 문서화하는 방식이다. 코드로 강제하지 않아서 런타임 이전에 실수를 감지할 수 없다. 신규 팀원이 합류하거나 코드 리뷰가 누락되면 물리 삭제가 발생할 수 있다. ❌
    방법 B default 메서드 오버라이드로 예외를 던지는 방식이다. JpaRepository 인터페이스의 delete 계열 메서드를 오버라이드해 CustomException을 던진다. 호출 즉시 런타임 에러가 발생하므로 테스트 단계에서 반드시 발견된다. ✅
    채택 이유 팀 컨벤션은 사람에 의존한다. 오버라이드 방식은 어떤 개발자가 호출해도 즉각적으로 에러가 발생하므로 실수를 코드 레벨에서 강제로 막을 수 있다. 
    // ErrorCode.java — 추가
UNSUPPORTED_OPERATION(
    HttpStatus.BAD_REQUEST,
    "CA006",
    "물리적인 삭제는 이용할 수 없습니다."
);
    // CategoryRepository.java — 추가
@Override
default void delete(Category entity) {
    throw new CustomException(ErrorCode.UNSUPPORTED_OPERATION);
}

@Override
default void deleteById(Long id) {
    throw new CustomException(ErrorCode.UNSUPPORTED_OPERATION);
}

@Override
default void deleteAll() {
    throw new CustomException(ErrorCode.UNSUPPORTED_OPERATION);
}
    달성한 것 물리 삭제 경로를 런타임 에러로 차단 → 어디서 호출해도 즉시 감지 가능
    GlobalExceptionHandler가 CustomException을 처리하므로 400 응답으로 클라이언트에 전달됨
    deleteAll(Iterable)은 내부적으로 delete(entity) 루프이므로 간접 차단됨
    ⚠️ 고려할 점 Product는 @SoftDelete(Hibernate 기본 제공)를 사용 중이므로 deleteById() 호출 시 자동으로 UPDATE(is_deleted=true)가 실행된다. Category처럼 수동으로 막을 필요가 없다. 엔티티마다 소프트 삭제 구현 방식이 다르므로 혼동에 주의해야 한다.

    이슈 3. 카테고리 삭제 시 N+1 쿼리 문제

    문제 발견

    카테고리 삭제 시 deleteWithDescendants() 엔티티 메서드가 트리를 재귀 순회하면서 각 노드마다 children 컬렉션을 Lazy 로딩했다. findByIdWithChildren() 쿼리는 직계 자식(1단계)만 JOIN FETCH하므로, 더 깊은 노드를 순회할수록 노드 수만큼 추가 SELECT가 발생했다. 

    // CategoryService.java — 수정 전
Category category = categoryRepository
    .findByIdWithChildren(categoryId)  // 1단계만 JOIN FETCH
    .orElseThrow(...);

category.deleteWithDescendants();  // 재귀 순회 → N+1
    // Category.java — deleteWithDescendants()
public void deleteWithDescendants() {
    this.isDeleted = true;
    this.deletedAt = LocalDateTime.now();
    for (Category child : this.children) {      // children 접근 → Lazy 로딩
        if (!child.isDeleted()) {
            child.deleteWithDescendants();       // 재귀 → 또 Lazy 로딩
        }
    }
}
    -- 트리 깊이 3단계, 실제 실행 쿼리
SELECT * FROM category LEFT JOIN ... WHERE c.id = 1;  -- 1단계만 FETCH
SELECT * FROM category WHERE parent_id = 2 ...;       -- 2단계 Lazy 로딩
SELECT * FROM category WHERE parent_id = 3 ...;       -- 3단계 Lazy 로딩
-- 노드 100개 → ~100번 SELECT + ~100번 UPDATE

    접근 방법 비교

    방법 설명 채택
    방법 A JOIN FETCH 깊이를 늘리는 방식이다. JPQL에서 컬렉션을 무한 FETCH JOIN할 수 없다. 깊이를 미리 정해야 하므로 트리 구조가 변하면 쿼리를 수정해야 한다. 근본적인 해결책이 아니다. ❌
    방법 B 재귀 CTE로 하위 ID 전체를 수집한 뒤 벌크 UPDATE로 처리하는 방식이다. DB의 WITH RECURSIVE로 한 번의 SELECT에서 모든 하위 노드 ID를 가져온다. IN 절 벌크 UPDATE로 한 번에 소프트 삭제한다. 트리 깊이와 노드 수에 무관하게 항상 2번의 쿼리만 발생한다. ✅
    채택 이유 방법 A는 트리 깊이에 따라 쿼리가 무한히 늘어나는 근본 문제를 해결하지 못한다. 방법 B는 DB가 재귀 탐색을 책임지고, 애플리케이션은 ID 목록만 받아 한 번에 UPDATE한다. 구조 변경 없이 항상 2번의 쿼리로 고정된다. 
    // CategoryRepository.java — 추가

// 재귀 CTE로 자기 자신 포함 모든 하위 노드 ID 수집
@Query(value = """
    WITH RECURSIVE tree AS (
        SELECT id FROM category
        WHERE id = :rootId AND is_deleted = false
        UNION ALL
        SELECT c.id FROM category c
        INNER JOIN tree t ON c.parent_id = t.id
        WHERE c.is_deleted = false
    )
    SELECT id FROM tree
    """, nativeQuery = true)
List<Long> findAllDescendantIds(@Param("rootId") Long rootId);

// 벌크 UPDATE로 한 번에 소프트 삭제
@Modifying
@Query("UPDATE Category c SET c.isDeleted = true, c.deletedAt = :now WHERE c.id IN :ids")
void softDeleteAllByIds(@Param("ids") List<Long> ids, @Param("now") LocalDateTime now);
    // CategoryService.java

// ❌ 수정 전
Category category = categoryRepository
    .findByIdWithChildren(categoryId)
    .orElseThrow(...);

category.deleteWithDescendants();  // 재귀 N+1

return CategoryDeleteResponse.from(category);

// ✅ 수정 후
Category category = categoryRepository
    .findById(categoryId)
    .orElseThrow(...);

List<Long> allIds = categoryRepository.findAllDescendantIds(categoryId);  // SELECT 1번
LocalDateTime now = LocalDateTime.now();
categoryRepository.softDeleteAllByIds(allIds, now);                       // UPDATE 1번
    구분 기존 방식 개선 방식
    하위 노드 조회 노드 수만큼 SELECT SELECT 1번 (재귀 CTE)
    소프트 삭제 처리 노드 수만큼 UPDATE UPDATE 1번 (IN절)
    노드 100개 기준 ~200 쿼리 2 쿼리
    ⚠️ 고려할 점 WITH RECURSIVE(재귀 CTE)는 MySQL 8.0+, PostgreSQL에서 동작한다. H2를 테스트 DB로 사용하는 경우 application-test.yml에 MODE=MySQL 설정이 필요할 수 있다.

    이슈 4. getCategory() 단건 조회 N+1

    문제 발견

    getCategory() 메서드가 일반 findById()로 카테고리를 조회한 후, CategoryResponse.from()에서 getParent().getName()을 호출했다. parent는 @ManyToOne(fetch = LAZY)이므로 조회 시점에 로딩되지 않고, getName() 호출 순간 별도의 SELECT가 추가 발생했다. 

    // CategoryService.java — 수정 전
public CategoryResponse getCategory(Long categoryId) {
    Category category = categoryRepository.findById(categoryId)  // parent 없음
            .orElseThrow(...);
    return CategoryResponse.from(category);  // getParent().getName() → Lazy 로딩 발생
}
    // CategoryResponse.java
.parentCategoryName(category.getParent() != null
        ? category.getParent().getName()  // ← 이 시점에 추가 SELECT 발생
        : null)
    -- 실제 실행 쿼리 2번
SELECT * FROM category WHERE id = ?;                             -- 카테고리 조회
SELECT * FROM category WHERE id = ? AND is_deleted = false;     -- parent Lazy 로딩
    // CategoryRepository.java — 추가
@Query("SELECT c FROM Category c LEFT JOIN FETCH c.parent WHERE c.id = :id")
Optional<Category> findByIdWithParent(@Param("id") Long id);
    // CategoryService.java

// ❌ 수정 전
Category category = categoryRepository.findById(categoryId)
        .orElseThrow(...);

// ✅ 수정 후
Category category = categoryRepository.findByIdWithParent(categoryId)
        .orElseThrow(...);
// parent 이미 로딩 → getParent().getName() 추가 쿼리 없음
    달성한 것 단건 조회에서 category 1번 SELECT로 parent까지 함께 조회 → 쿼리 2번 → 1번으로 감소

    이슈 5. update() 부모 변경 시 N+1

    문제 발견

    update() 메서드에서 findById()로 카테고리를 로드한 후 changeParent()를 호출했다. changeParent() 내부에서 기존 parent의 children 컬렉션에 remove()를 호출하는데, 이 시점에 기존 parent의 children이 Lazy 로딩되어 추가 SELECT가 발생했다. 

    // Category.java — changeParent() 내부
public void changeParent(Category parent) {
    if (this.parent != null) {
        this.parent.children.remove(this);  // ← 기존 parent의 children Lazy 로딩 발생!
    }
    this.parent = parent;
    if (parent != null) {
        parent.children.add(this);
    }
}
    // CategoryService.java — 수정 전
Category category = categoryRepository.findById(categoryId)  // parent 없음, parent.children 없음
        .orElseThrow(...);

Category parentCategory = findParentCategory(request.parentCategoryId());
category.changeParent(parentCategory);  // 기존 parent.children Lazy 로딩!
    // CategoryRepository.java — 추가
@Query("""
    SELECT c FROM Category c
    LEFT JOIN FETCH c.parent p
    LEFT JOIN FETCH p.children
    WHERE c.id = :id
    """)
Optional<Category> findByIdWithParentAndChildren(@Param("id") Long id);
    // CategoryService.java

// ❌ 수정 전
Category category = categoryRepository.findById(categoryId)
        .orElseThrow(...);

// ✅ 수정 후
Category category = categoryRepository.findByIdWithParentAndChildren(categoryId)
        .orElseThrow(...);
// parent와 parent.children 이미 로딩
// → changeParent() 내부에서 추가 쿼리 없음
    달성한 것 changeParent() 내부의 this.parent.children.remove(this) 호출 시 이미 메모리에 로딩된 컬렉션을 사용하므로 추가 SELECT 없음
    ⚠️ 고려할 점 parent가 null인 루트 카테고리의 경우 LEFT JOIN FETCH p.children에서 p가 null이므로 해당 JOIN이 스킵된다. Hibernate는 이를 안전하게 처리하므로 예외 없이 정상 동작한다.

    이슈 6. delete() 응답의 deletedAt = null

    문제 발견

    이슈 3에서 softDeleteAllByIds() 벌크 UPDATE를 적용했는데, @Modifying 벌크 쿼리는 DB만 업데이트하고 영속성 컨텍스트(1차 캐시)는 갱신하지 않는다. findById()로 로드한 category 객체의 deletedAt은 여전히 null인 상태로 응답 DTO에 들어갔다. 

    // CategoryService.java — 문제 흐름
Category category = categoryRepository.findById(categoryId); // (1) deletedAt = null

List<Long> allIds = categoryRepository.findAllDescendantIds(categoryId);
categoryRepository.softDeleteAllByIds(allIds, now);          // (2) DB: deletedAt = now (반영됨)

return CategoryDeleteResponse.from(category);                // (3) category.getDeletedAt() = null !!

    접근 방법 비교

    방법 설명 채택
    방법 A @Modifying(clearAutomatically = true)를 사용하는 방식이다. 벌크 쿼리 후 1차 캐시를 전부 비워준다. 이후 엔티티를 다시 조회하면 DB에서 최신 값을 가져온다. 단, 이미 로드된 category 객체가 clear되므로 다시 findById()를 호출해야 한다. 추가 SELECT가 발생한다. ❌
    방법 B now 변수를 공유해 DTO를 직접 구성하는 방식이다. softDeleteAllByIds()에 전달한 LocalDateTime now를 그대로 DTO 생성에 사용한다. DB에 저장된 값과 동일한 시각을 응답에 담을 수 있다. 추가 SELECT 없이 1번의 트랜잭션으로 처리된다. ✅ 
    // CategoryService.java

// ❌ 수정 전
List<Long> allIds = categoryRepository.findAllDescendantIds(categoryId);
categoryRepository.softDeleteAllByIds(allIds, now);

return CategoryDeleteResponse.from(category);  // deletedAt = null !

// ✅ 수정 후
List<Long> allIds = categoryRepository.findAllDescendantIds(categoryId);
LocalDateTime now = LocalDateTime.now();
categoryRepository.softDeleteAllByIds(allIds, now);

// DB에 넣은 now를 직접 사용해 DTO 구성
return CategoryDeleteResponse.builder()
        .categoryId(category.getId())
        .categoryName(category.getName())
        .deletedAt(now)  // now 공유
        .build();
    달성한 것 DB에 저장된 deletedAt과 응답 DTO의 deletedAt이 동일한 값으로 보장됨
    추가 SELECT 없이 처리
    ⚠️ 고려할 점 @Modifying 벌크 쿼리를 사용하는 모든 곳에서 영속성 컨텍스트 불일치 문제가 발생할 수 있다. 같은 트랜잭션에서 해당 엔티티를 벌크 쿼리 이후에 다시 사용한다면 반드시 최신 값 여부를 확인해야 한다.

    이슈 7. 주문 생성 · 취소 시 재고 동시성 문제

    문제 발견

    주문 생성(add)과 주문 취소(delete) 모두 productRepository.findById()로 상품을 조회한 후 재고를 변경했다. 아무런 락이 없어서 동시에 같은 상품에 접근할 경우 Race Condition이 발생했다. 

    // OrderService.java — 수정 전 (주문 생성)
Product product = productRepository.findById(request.productId())  // 락 없음
        .orElseThrow(...);

validateProduct(product, request);
product.decreaseStock(request.quantity());  // 동시 접근 시 재고 음수 가능
    // 재고 1개인 상품에 두 명이 동시 주문 시
사용자 A: findById() → stock = 1
사용자 B: findById() → stock = 1   (A가 아직 차감 전)
사용자 A: decreaseStock(1) → stock = 0 → UPDATE
사용자 B: decreaseStock(1) → stock = -1 → UPDATE  ← 재고 음수!

    낙관적 락 vs 비관적 락

    구분 낙관적 락 (Optimistic Lock) 비관적 락 (Pessimistic Lock)
    동작 방식 @Version 컬럼으로 버전 번호를 관리한다. 저장 시점에 "내가 읽은 버전 == 현재 DB 버전"을 확인하고, 다르면 OptimisticLockException을 던진다. 조회하는 순간 SELECT ... FOR UPDATE를 날려 다른 트랜잭션이 해당 행을 건드리지 못하게 DB 레벨에서 잠근다.
    비유 구글 드라이브 문서를 두 명이 동시에 수정할 때, 나중에 저장한 사람에게 "이미 수정된 내용이 있어요" 경고를 띄우는 것. 노래방 예약 시 "이 방 내가 쓰는 동안 아무도 못 들어옴" 팻말을 걸어두는 것.
    장점 DB에 락을 걸지 않아 성능이 좋다. 충돌이 아예 일어나지 않는다. 결과가 명확하다.
    단점 충돌 시 예외가 터지므로 재시도 로직이 필요하다. 충돌이 잦으면 계속 실패한다. 조회 순간부터 락을 잡으므로 대기가 생긴다. 트래픽이 많으면 성능이 저하된다.
    비관적 락 채택 이유 재고 차감은 충돌 가능성이 높다 (인기 상품은 동시에 여러 명이 주문한다). 낙관적 락은 충돌 시 OptimisticLockException이 발생해 주문 전체를 재시도해야 하므로 사용자 경험이 저하된다. 주문이라는 행위는 실패보다 "대기 후 정확한 처리"가 더 적합하다. 
    // ProductRepository.java — 추가
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p WHERE p.id = :id")
Optional<Product> findByIdWithLock(@Param("id") Long id);
// → SELECT ... FOR UPDATE 쿼리가 실행됨
    // OrderService.java — 주문 생성(add)

// ❌ 수정 전
Product product = productRepository.findById(request.productId())
        .orElseThrow(...);

// ✅ 수정 후
Product product = productRepository.findByIdWithLock(request.productId())
        .orElseThrow(...);
    // OrderService.java — 주문 취소(delete)

// ❌ 수정 전
Product product = order.getProduct();  // Lazy 로딩, 락 없음
product.increaseStock(order.getQuantity());

// ✅ 수정 후
Product product = productRepository.findByIdWithLock(order.getProduct().getId())
        .orElseThrow(...);  // 비관적 락으로 재조회
product.increaseStock(order.getQuantity());
    비관적 락 동작 흐름 사용자 A: findByIdWithLock(1) → FOR UPDATE 락 획득, stock = 1
    사용자 B: findByIdWithLock(1) → 대기 (A가 락 보유 중)
    사용자 A: decreaseStock(1) → stock = 0 → 커밋 → 락 해제
    사용자 B: 이제 조회 가능 → stock = 0 확인 → INSUFFICIENT_STOCK 예외 → 주문 실패

    결과: 정확히 1건만 성공
    달성한 것 주문 생성(add)과 취소(delete) 모두 비관적 락 적용 → Race Condition 완전 차단
    주문 생성과 취소가 동시에 같은 상품에 접근해도 Lost Update 없음
    ⚠️ 고려할 점 추가 고려사항: ProductCommandService의 관리자 수동 재고 조정(increaseStock / decreaseStock)은 현재 락 없이 findById()를 사용한다. 관리자 전용으로 동시 호출 가능성이 낮지만, 엄밀히는 여기도 락이 필요할 수 있다.

    주의: 비관적 락은 트랜잭션이 짧아야 효과적이다. 락을 보유한 채로 오래 걸리는 작업(외부 API 호출 등)을 수행하면 다른 요청들이 장시간 대기하게 된다. 현재 add() 내부에서 resolveSession() 호출이 있으므로 트랜잭션 범위를 최대한 짧게 유지하는 것을 권장한다.

    마무리

    오늘은 이커머스 백오피스 팀프로젝트(CommercePilot)에서 실제로 발생한 7가지 이슈를 트러블슈팅하며 JPA와 Spring의 동작 원리를 더욱 깊게 이해하게 되었다. 이슈 1~2는 SoftDelete와 JPA의 물리 삭제 전파 메커니즘이 충돌하는 구조적 문제였고, 이를 해결하기 위해 JPA 레벨과 Repository 레벨에서 이중으로 물리 삭제 경로를 차단하는 방어적 설계를 적용했다. "팀 규칙으로 막는 것"과 "코드로 막는 것"의 신뢰도 차이를 몸소 확인한 경험이었다.

    이슈 3~5는 모두 Lazy 로딩으로 인한 N+1 문제였지만, 각 케이스마다 원인이 다른 형태로 나타났다. 삭제 시 재귀 순회에 의한 N+1, 단건 조회 시 DTO 변환 과정의 N+1, update 메서드 내부 도메인 로직의 N+1이 각각 달랐다. 해결책은 공통적으로 필요한 연관 데이터를 JOIN FETCH로 미리 로딩하는 것이었고, 특히 이슈 3에서는 재귀 CTE를 활용해 DB가 트리 탐색을 전담하도록 구조를 바꿔 쿼리 수를 200번에서 2번으로 줄인 것이 가장 큰 성과였다.

    이슈 6은 @Modifying 벌크 쿼리가 영속성 컨텍스트를 갱신하지 않는다는 점을 실제로 마주한 케이스였고, 이슈 7은 동시성 문제를 낙관적 락과 비관적 락 중 어떤 기준으로 선택해야 하는지 명확히 정리할 수 있는 경험이었다. 재고처럼 충돌 빈도가 높고 정확성이 중요한 상황에서는 비관적 락이 적합하다는 판단 근거를 직접 코드로 검증했다.
