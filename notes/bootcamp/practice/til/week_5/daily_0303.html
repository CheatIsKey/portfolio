    1. Filter

    필터는 본격적인 스프링 로직이 실행되기 전에 모든 요청을 한 번 걸러준다.

    Filter 구조

    Spring에서는 Filter를 직접 구현할 필요 없이, 많이 사용되는 목적에 따라 이미 구현이 되어 있고 하나를 선택해서 사용하면 된다. OncePerRequestFilter를 가장 많이 사용한다.

    Filter 종류 설명
    OncePerRequestFilter 요청 당 한 번만 실행되도록 보장한다. 인증/인가, 로깅, 트랜잭션 관리 같은 곳에서 자주 사용한다.
    CharacterEncodingFilter 모든 요청/응답에 지정한 인코딩(UTF-8 등)을 적용한다.
    HiddenHttpMethodFilter HTML form은 GET/POST만 지원한다. _method 파라미터로 PUT, DELETE 등을 매핑 가능하게 해준다.
    FormContentFilter application/x-www-form-urlencoded 데이터를 PUT, PATCH, DELETE 요청에서도 읽게 해준다.
    RequestContextFilter 현재 요청을 RequestContextHolder에 바인딩한다. @RequestScope Bean 사용 시 필요하다.
    이외 다수 이외에도 많은 필터가 존재한다.

    Filter 구현 

    @Component
public class SpringFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response, FilterChain filterChain)
            throws ServletException, IOException {

        // 요청이 들어올 때 실행되는 부분
        System.out.println("입력 테스트");

        filterChain.doFilter(request, response);   // FilterChain에 filter를 추가

        // 요청이 나갈 때 실행되는 부분
        System.out.println("출력 테스트");
    }
}
    FilterChain Filter 묶음이 있고, 필요하면 FilterChain에 Filter를 하나씩 추가하는 것이다.

    Filter 순서는 @Order 어노테이션을 통해 지정할 수 있다.

    필터 사용 이유

    번호 사용 목적 설명
    1 로깅 (요청/응답 로깅) 모든 컨트롤러 호출 전후에서 공통적으로 적용 가능하다. 장애 발생 시 어느 API에서 문제가 발생했는지 빠르게 추적할 수 있다. 성능 분석(요청 처리 시간 측정)에 활용 가능하다.
    2 공통 인증/권한 체크 컨트롤러까지 들어오기 전에 인증 실패 요청을 걸러낼 수 있다. 인증/인가 로직을 중앙 집중화하여 코드 중복을 제거한다. 미인증 요청이 내부 로직까지 전달되지 않으므로 보안상 안전하다.

    2. Spring Security

    스프링 기반 애플리케이션의 인증과 인가를 담당하는 보안 프레임워크다.

    Spring Security는 단순히 JWT만을 위한 프레임워크가 아니다. 폼 로그인, 세션 기반 인증, Basic Auth, LDAP, OAuth2, SAML 등 다양한 인증 방식을 지원한다.

    특징 설명
    Filter 기반 보안 처리 Spring Security 의존성을 추가한 이후에는 Filter 구조가 바뀐다. 기존 FilterChain에 SecurityFilterChain이 추가되고 Spring Security의 거의 모든 동작은 여기서 마무리된다.
    구조 제공 인증/인가 로직을 개발자가 직접 구현하지 않아도 되는 구조를 제공한다.
    확장 가능 다양한 인증 방식을 플러그인 형태로 확장 가능하다.

    Spring Security는 세션 방식이 기본이지만, JWT 기반 인증을 사용하면 로그인 이후 모든 요청을 커스텀 필터에서 처리하게 된다.

    Spring Security의 흐름 (세션 기반)

    단계 설명
    1 로그인 폼에서 username/password 입력
    2 UsernamePasswordAuthenticationFilter가 요청을 가로챔
    3 AuthenticationManager에게 위임 인증
    4 내부적으로 UserDetailsService에서 사용자 조회
    5 비밀번호 일치 → 인증 성공 → SecurityContext에 저장
    6 인증 실패 → 로그인 페이지로 리다이렉트

    Spring Security의 흐름 (JWT 기반)

    단계 설명
    1 사용자가 /login에 username/password 입력
    2 서버는 인증 성공 시 JWT 토큰 생성 → 클라이언트에 반환
    3 이후 모든 요청에 Authorization: Bearer {JWT} 헤더 포함
    4 커스텀 필터(JwtFilter)가 토큰 유효성 검사
    5 유효한 경우 → Authentication 생성 → SecurityContext에 저장
    6 이후 인가 절차로 넘어감

    커스텀 JwtFilter 등록 예시 

    @Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    return http
        .csrf(AbstractHttpConfigurer::disable)       // csrf 보호 비활성화
        .httpBasic(AbstractHttpConfigurer::disable)  // BasicAuthenticationFilter 비활성화
        .formLogin(AbstractHttpConfigurer::disable)  // UsernamePasswordAuthenticationFilter, DefaultLoginPageGeneratingFilter 비활성화
        .addFilterBefore(jwtFilter, SecurityContextHolderAwareRequestFilter.class)
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/login").permitAll()
            .anyRequest().authenticated()
        )
        .build();
}
    • HTTP Basic 인증 방식은 Authorization 헤더에 사용자 이름과 비밀번호를 평문으로 포함하는 방식이라, 보안의 문제로 사용하지 않는다.
    • addFilterBefore: 필터의 순서를 지정해주는 역할이다. .addFilterBefore(jwtFilter, SecurityContextHolderAwareRequestFilter.class)는 jwtFilter 뒤에 SecurityContextHolderAwareRequestFilter를 위치시키는 것이다.
    • authorizeHttpRequests: 어디에 어떻게 권한 검사를 할 것인지(인가 검사)에 대한 구체적인 내용을 작성한다.
    .authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/login").permitAll()
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/user/**").hasRole("USER")
)

    SecurityContextHolder

    현재 스레드에서 인증된 사용자(Authentication 객체)를 저장/조회하는 Spring Security의 핵심 저장소다. 즉, "지금 로그인한 사용자가 누구인지"를 저장하는 곳이다. 

    // 인증 정보 조회: 커스텀 필터나 컨트롤러, 서비스 어디서든 접근 가능
Authentication auth = SecurityContextHolder.getContext().getAuthentication();

// 인증 정보 수동 저장: JWT 사용 시 직접 저장이 필요하다.
SecurityContextHolder.getContext().setAuthentication(authentication);

    핵심 컴포넌트 정리 (JWT 인증 흐름 기준)

    컴포넌트 역할
    JwtFilter JWT 토큰 존재/유효성 검사 → 사용자 인증 처리
    JwtUtil 토큰 발급, 파싱, 검증 유틸 클래스
    SecurityContextHolder 인증 결과 저장소 (스레드 단위)
    Authentication 인증 객체 (권한 포함)
    UsernamePasswordAuthenticationToken Spring Security에서 JWT 토큰 방식을 사용하기 위한 객체
    User Spring Security에서 인증된 사용자를 표현하는 객체

    Spring Security 기본 인가 흐름

    기본 인가 흐름 SecurityContext에 저장된 Authentication을 조회 → FilterSecurityInterceptor → AccessDecisionManager → 권한 정보와 비교(접근 허용 / 거부)
    JWT 적용 코드 인가 흐름 JwtFilter: 인증된 사용자 정보를 SecurityContextHolder에 Authentication으로 저장한다.
    .authorizeHttpRequests(...): 인가 정책을 정의한다. 내부적으로 FilterSecurityInterceptor, AccessDecisionManager가 자동 구성된다.
    .authenticated(): 인가 조건(AccessDecisionManager), 인증 여부를 판단한다.

    즉, JwtFilter에서 인증을 끝내고 SecurityContext에 저장해두었기 때문에, 이후 모든 인가 과정은 Spring Security가 자동으로 처리하게 된다.

    인가 방법

    1. URL 기반 설정 — 관리가 편하다.

    http.authorizeHttpRequests()
    .requestMatchers("/admin/**").hasRole("ADMIN")
    .requestMatchers("/user/**").hasAnyRole("USER", "ADMIN")
    .anyRequest().permitAll();

    2. 메서드 기반 어노테이션 — 컨트롤러/서비스 레이어 권한 제어가 가능하다.

    SecurityConfig 클래스에 @EnableMethodSecurity(securedEnabled = true) 설정이 필요하다.

    @PreAuthorize("hasRole('ADMIN')")
public void deleteUser(Long id) { ... }

    @AuthenticationPrincipal

    이 어노테이션을 사용하면 JwtFilter에서 생성하면서 들어간 정보들을 꺼낼 수 있다. User는 내가 구현한 User 클래스가 아닌, Spring Security가 제공하는 User를 사용해야 한다. 

    @GetMapping("/me")
public ResponseEntity<?> getMyInfo(@AuthenticationPrincipal User user) {
    String username = user.getUsername();
    // JwtFilter에서 저장한 인증 정보를 꺼내 사용한다.
}

    3. 공통 관심사 — Filter / Interceptor / AOP 비교

    인증, 인가, 로깅, 트랜잭션, 예외 처리 등 여러 레이어에 반복적으로 등장하는 처리를 말한다. 핵심 로직과 분리하여 유지보수성을 높이기 위해 필요하다.

    항목 Filter Interceptor AOP
    처리 위치 DispatcherServlet 이전 컨트롤러 실행 전후 Spring Bean 메서드 전후 (Service 계층 메서드 실행 전후)
    대상 서블릿 요청 핸들러(Controller) 모든 Bean 메서드
    사용 목적 인증, 로깅, CORS 등 세션 체크, 권한 검사 로깅, 트랜잭션, 예외 처리 등
    선언 방법 Filter 구현체 HandlerInterceptor 구현체 @Aspect 클래스
    실행 순서 Filter → Interceptor → Controller → AOP → Method 순으로 진행된다.

    각 공통 관심사의 대표 사용 사례

    구분 대표 사용 사례 특징
    Filter Authorization 헤더에 담긴 JWT 토큰 유효성 검사 모든 요청의 가장 앞단에서 작동한다. Spring DispatcherServlet 이전에 실행되어야 한다. 인증 실패 시 컨트롤러까지 가지 않도록 빠르게 차단할 수 있다.
    Interceptor 로그인이 필요한 페이지 요청 시 세션 또는 인증 여부 검증 컨트롤러 진입 직전/후에 동작하여 처리 가능하다. 사용자별 세션, 권한 확인 로직 작성에 용이하다. 요청 URL 기반으로 유연하게 경로별 적용 가능하다.
    AOP Service 계층에서 비즈니스 로직 실행 시간 측정 특정 메서드 호출 전후에만 부가 기능을 넣고 싶을 때 유리하다. 핵심 로직은 건드리지 않고 공통 처리만 분리 가능하다. 다양한 메서드에 어노테이션으로 간편하게 확장할 수 있다.
    ⚠️ Spring Security 사용 시 Interceptor 사용 빈도 변화 Spring Security를 사용하고 있다면 기본 인증/인가 처리를 Spring Security Filter Chain이 담당하고 있어서 Interceptor 사용 빈도가 많이 줄어든다. 하지만 아래와 같은 일부 경우에는 여전히 Interceptor가 유용하게 사용된다.
    • 작성자 본인 확인
    • 비즈니스 조건

    HandlerInterceptor 메서드

    메서드 실행 시점
    preHandle 컨트롤러 실행 전에 동작한다. false 반환 시 흐름을 중단할 수 있다.
    postHandle 컨트롤러 실행 후, 뷰 렌더링 전에 동작한다.
    afterCompletion 응답 완료 후에 동작한다. 예외 처리도 포함한다.

    4. AOP

    AOP 핵심 용어

    용어 설명
    Aspect 공통 관심사를 담는 모듈이다. (예: 로깅, 트랜잭션, 권한 체크)
    JoinPoint 메서드 실행 지점이다.
    Advice 실행 시점에 따라 작동하는 AOP 로직이다. (@Before, @After, @Around)
    PointCut 어떤 JoinPoint에 Advice를 적용할지 정의한다. (표현식 기반)
    Weaving Advice를 적용하는 과정이다. (스프링이 알아서 해준다.)
    AOP 팁 — 어떤 것을 / 언제 / 어떻게 어떤 것을: PointCut을 통해서 대상을 지정해준다.
    언제: Advice를 통해서 실행되는 시점을 지정해준다.
    어떻게: Aspect에 정의한 메서드를 통해서 어떻게 할 것인지 지정해준다.

    AOP 어노테이션

    어노테이션 설명
    @Aspect Aspect 클래스임을 선언한다.
    @Around 메서드 실행 전후를 제어한다.
    @Before 메서드 실행 전에 수행한다.
    @AfterReturning 정상 반환 후 수행한다.
    @AfterThrowing 예외 발생 시 수행한다.
    @After 정상/예외 관계없이 실행 후 수행한다.

    AOP 활용 예시

    활용 분야 설명
    로깅 모든 서비스 실행 시 로그 기록
    트랜잭션 메서드 단위 트랜잭션 처리
    권한 검사 관리자 권한 체크
    예외 처리 공통 에러 메시지 처리
    실행 시간 측정 성능 모니터링

    AOP 코드 예시

    @Around — 메서드 실행 전후 시간을 비교 분석하는 기능

    @Aspect
@Component
@Slf4j
public class TimeCheckAop {

    // 어떤 것을: <특정 Service> 하위의 메서드
    // 언제: 메서드 실행 전후
    // 어떻게: executionTime 메서드 내용을 실행한다.

    @Around("execution(* <특정 서비스 패키지 경로>.*(..))")
    public Object executionTime(ProceedingJoinPoint joinPoint) throws Throwable {

        // 메서드 실행 전
        long start = System.currentTimeMillis();

        Object proceed = joinPoint.proceed();  // 실제 메서드 실행 → 필터의 doFilter와 유사

        // 메서드 실행 후
        long end = System.currentTimeMillis();

        log.info("[AOP] {} 실행됨 in {} ms", joinPoint.getSignature(), end - start);

        return proceed;
    }
}

    @Before — 메서드 실행 전에만 수행

    @Before("execution(* <특정 서비스 패키지 경로>.*(..))")
public void executionTime() throws Throwable {

    // 메서드 실행 전
    log.info("메서드 실행 전에만 수행");
}

    @AfterReturning — 메서드 실행 후에만 수행

    @AfterReturning("execution(* <특정 서비스 패키지 경로>.*(..))")
public void executionTime() throws Throwable {

    // 메서드 실행 후
    log.info("메서드 실행 후에만 수행");
}

    @AfterThrowing — 메서드 실행 후 예외가 터질 때만 수행

    @AfterThrowing("execution(* <특정 서비스 패키지 경로>.*(..))")
public void executionTime() throws Throwable {

    // 메서드 실행 후 예외가 터질 때만 수행
    log.info("메서드 실행 후 예외가 터져야 수행");
}

    마무리

    오늘은 Spring의 요청 처리 흐름 전반을 관통하는 공통 관심사 처리 기술인 Filter, Spring Security, Interceptor, AOP를 정리했다. Filter는 DispatcherServlet에 도달하기도 전에 모든 요청을 가로채는 가장 앞단의 보안 게이트 역할을 하며, OncePerRequestFilter를 상속하여 요청 당 한 번만 실행되도록 보장하는 방식이 실무의 표준임을 이해했다.

    Spring Security는 단순한 JWT 전용 프레임워크가 아니라 다양한 인증 방식을 지원하는 보안 프레임워크임을 명확히 했다. 세션 방식과 JWT 방식의 흐름을 단계별로 비교하면서 JWT 방식에서는 JwtFilter가 인증을 마치고 SecurityContext에 저장해두면 이후 인가는 Spring Security가 자동으로 처리한다는 흐름을 파악했다. 인가 방법으로는 URL 기반 설정과 메서드 기반 어노테이션(@PreAuthorize) 두 가지를 모두 정리했다.

    Filter → Interceptor → Controller → AOP → Method의 실행 순서를 이해하고, 각각의 처리 위치와 대표 사용 사례를 구분하는 것이 오늘의 핵심 성과다. AOP에서는 Aspect, JoinPoint, Advice, PointCut, Weaving의 개념과 @Around, @Before, @AfterReturning, @AfterThrowing, @After 어노테이션을 실제 코드 예시로 확인했다. 공통 관심사는 "어떤 것을, 언제, 어떻게" 처리할지를 PointCut과 Advice로 분리해 선언하는 방식이 핵심임을 기억하겠다.
