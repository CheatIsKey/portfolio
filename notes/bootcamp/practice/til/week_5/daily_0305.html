    내일배움캠프 Spring 백엔드 55일차 TIL — 사용자 요청부터 DB까지의 전체 흐름

    작성일 2026-03-05 · 분류: 부트캠프, TIL, Spring Boot, Filter, Security, Interceptor, ArgumentResolver, AOP

    55일차 개요

    오늘 목표

    • 사용자 요청이 DB에 닿기까지 거치는 각 계층의 역할과 순서를 한 흐름으로 정리한다.
    • Filter / Security Filter Chain / DispatcherServlet / Interceptor의 관계와 위치를 명확히 한다.
    • Argument Resolver의 역할과 WebMvcConfigurer 등록의 중요성을 이해한다.

    1. 전체 흐름도

    사용자 요청
    ↓
    1. Filter (Servlet Filter 계층 — Spring 외부)
    └ Security Filter Chain (Filter 안에 포함 — JWT 검증, 인증/인가 처리)
    ↓
    DispatcherServlet (Spring MVC 진입점 — 요청 라우팅)
    ↓
    2. Interceptor (Spring 컨텍스트 — 비즈니스 조건 검사)
    ↓
    3. Argument Resolver (파라미터 바인딩 — AuthUser 객체 생성)
    ↓
    4. Controller (요청 수신 — 서비스 호출)
    ↓
    5. Service (AOP / @Transactional — 핵심 비즈니스 로직)
    ↓
    6. Repository (JPA / Hibernate — SQL 생성 및 최적화)
    ↓
    7. Database (SQL 실행 및 결과 반환)
    ⚠️ Security Filter Chain은 Filter 안에 포함된다 Security Filter Chain은 별도의 독립 계층이 아니다. Spring Security 의존성을 추가하면 기존 Servlet FilterChain 안에 SecurityFilterChain이 하나의 필터로 삽입된다. 따라서 흐름상 "Filter → Security Filter Chain"이 아니라, Filter 계층 내부에 Security Filter Chain이 포함되는 구조다.

    또한 Filter와 Interceptor 사이에는 DispatcherServlet이 위치한다. DispatcherServlet이 요청을 받아 적절한 컨트롤러로 라우팅하는 시점에 Interceptor가 개입한다. Interceptor가 "스프링 컨텍스트 진입 이후"에서 동작하는 이유가 바로 이것이다.

    2. 각 계층별 역할

    1. Filter (Servlet Filter / Spring Security)

    항목 설명
    역할 문지기 — 가장 바깥쪽에서 모든 요청을 먼저 가로챈다.
    수행 JWT 토큰 존재 여부 확인, 화이트리스트(로그인 없이 접근 가능한 URL) 체크, CORS 설정 처리, Spring Security의 인증/인가 처리
    위치 Spring MVC(DispatcherServlet) 이전 — 스프링 컨텍스트 밖

    2. Interceptor (Spring Interceptor)

    항목 설명
    역할 관리실 — DispatcherServlet 이후, 컨트롤러 이전에서 동작한다.
    수행 로그인 여부 최종 확인, 작성자 본인 확인, 비즈니스 조건 검사
    위치 DispatcherServlet 이후 — 스프링 컨텍스트 안
    Filter vs Interceptor — 인증/인가 담당 분리 Spring Security를 사용하는 경우 JWT 검증과 인증/인가 처리는 Filter 계층(Security Filter Chain)이 담당한다. 따라서 Interceptor에서 토큰을 다시 검사하는 것은 중복이다. Interceptor는 Security가 처리하지 못하는 비즈니스 조건(작성자 본인 확인 등)을 담당하는 역할로 좁혀서 사용하는 것이 일반적이다.

    3. Argument Resolver (AuthUserArgumentResolver)

    항목 설명
    역할 웨이터 — 컨트롤러가 필요로 하는 파라미터를 미리 준비해서 전달한다.
    수행 @Auth 어노테이션 확인(supportsParameter) → 토큰에서 사용자 정보 추출 → 역할(Role: ADMIN/USER) 검증 → AuthUser 객체 생성 및 컨트롤러로 전달(resolveArgument)
    핵심 컨트롤러 코드에서 사용자 정보 추출 로직을 제거하여 책임을 분리한다.

    4. Controller

    항목 설명
    역할 주문 접수처 — 요청의 진입점이다.
    수행 Argument Resolver가 준비한 AuthUser를 받아 "누가" 요청했는지 즉시 식별하고, 서비스 계층을 호출한다.

    5. Service (AOP / @Transactional)

    항목 설명
    역할 주방장 — 핵심 비즈니스 로직을 처리한다.
    수행 트랜잭션 시작(@Transactional), 비즈니스 규칙 적용, AOP(로깅/권한 등 횡단 관심사 처리), 레포지토리 호출

    6. Repository (JPA / Hibernate)

    항목 설명
    역할 창고 관리자 — 데이터 접근을 전담한다.
    수행 Fetch Join(N+1 방지), Batch Size(컬렉션 조회 최적화) 전략을 사용하여 최적화된 SQL을 생성한다.

    7. Database

    항목 설명
    역할 실제 데이터 저장소
    수행 Repository가 생성한 최적화된 SQL을 실행하고 결과를 반환한다.

    3. 인가 & 파라미터 계층 (Argument Resolver)

    핵심: 컨트롤러의 짐을 덜어주는 곳 supportsParameter로 "내가 처리할 대상인가?"를 먼저 확인하고, resolveArgument에서 실제 객체 생성과 역할(Role) 검증까지 마친 뒤 컨트롤러에 전달한다. 컨트롤러는 전달받은 객체를 그대로 사용하기만 하면 된다. 
    @Component
@RequiredArgsConstructor
public class AuthUserArgumentResolver implements HandlerMethodArgumentResolver {

    private final JwtUtil jwtUtil;

    // 1. 처리 대상 여부 확인: @Auth 어노테이션 + AuthUser 타입인 파라미터만 처리
    @Override
    public boolean supportsParameter(MethodParameter parameter) {
        return parameter.hasParameterAnnotation(Auth.class)
            && parameter.getParameterType().equals(AuthUser.class);
    }

    // 2. 실제 객체 생성: 토큰에서 사용자 정보 추출 → 역할 검증 → AuthUser 반환
    @Override
    public Object resolveArgument(MethodParameter parameter,
                                  ModelAndViewContainer mavContainer,
                                  NativeWebRequest webRequest,
                                  WebDataBinderFactory binderFactory) {
        HttpServletRequest request = (HttpServletRequest) webRequest.getNativeRequest();
        String token = jwtUtil.getTokenFromRequest(request);
        Claims claims = jwtUtil.extractClaims(token);

        Long userId = Long.valueOf(claims.getSubject());
        String role = claims.get("auth", String.class);

        return new AuthUser(userId, UserRole.valueOf(role));
    }
}

    4. 설정 계층 (WebMvcConfigurer)

    핵심: 설계한 도구를 시스템에 장착하는 곳 아무리 잘 만든 Argument Resolver나 Interceptor도 WebMvcConfigurer에 등록하지 않으면 동작하지 않는다. addArgumentResolvers에 리졸버를, addInterceptors에 인터셉터를 반드시 등록해야 한다. 
    @Configuration
@RequiredArgsConstructor
public class WebConfig implements WebMvcConfigurer {

    private final AuthUserArgumentResolver authUserArgumentResolver;
    private final CustomInterceptor customInterceptor;

    // Argument Resolver 등록 — 빠트리면 @Auth 파라미터 바인딩이 동작하지 않는다.
    @Override
    public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
        resolvers.add(authUserArgumentResolver);
    }

    // Interceptor 등록
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(customInterceptor)
                .addPathPatterns("/admin/**");
    }
}

    마무리

    오늘은 사용자 요청이 DB에 닿기까지의 전체 흐름을 한 줄로 꿰뚫어 정리했다. 핵심은 각 계층이 왜 그 위치에 있는지를 이해하는 것이다. Filter는 Spring 외부에서 모든 요청을 먼저 가로채고, Security Filter Chain은 그 안에 포함되어 인증/인가를 처리한다. DispatcherServlet이 요청을 라우팅한 이후에야 Interceptor가 개입하며, 이 순서가 흐름도를 읽는 핵심이다.

    Argument Resolver는 컨트롤러가 사용자 정보 추출 로직을 직접 갖지 않도록 책임을 분리하는 역할을 한다. supportsParameter로 처리 대상을 결정하고, resolveArgument에서 객체 생성과 역할 검증까지 마쳐서 컨트롤러에 건네준다. WebMvcConfigurer에 등록하지 않으면 아무리 잘 만들어도 동작하지 않는다는 점도 중요하다.

    Service 계층에서 AOP와 @Transactional이 함께 동작하고, Repository에서 Fetch Join과 Batch Size 전략으로 최적화된 SQL이 생성된 뒤 DB에 전달되는 것으로 흐름이 마무리된다. 각 계층이 자신의 역할에만 집중하고, 다음 계층으로 깔끔하게 넘기는 구조가 Spring의 핵심 설계 원칙임을 다시 한 번 확인했다.
