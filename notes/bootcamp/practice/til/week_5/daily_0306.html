    1. AuthUserArgumentResolver 동작 활성화

    문제 상황

    AuthUserArgumentResolver가 구현되어 있었지만 Spring MVC에 등록되지 않아 @Auth 어노테이션이 전혀 동작하지 않는 상황이었다. @Component로 등록된 ArgumentResolver는 자동으로 Spring MVC에 연결되지 않는다. 반드시 WebMvcConfigurer.addArgumentResolvers()를 통해 명시적으로 등록해야 한다.

    핵심 원인 Spring MVC는 ArgumentResolver를 자동 탐지하지 않는다. 직접 등록하지 않으면 존재하지 않는 것과 같다. 
    @Configuration
@RequiredArgsConstructor
@EnableJpaAuditing
public class PersistenceConfig implements WebMvcConfigurer {

    private final AuthUserArgumentResolver authUserArgumentResolver;

    @Override
    public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
        resolvers.add(authUserArgumentResolver);  // 명시적 등록 필수
    }
}

    2. AuthService — 불필요한 암호화 연산 제거 (Early Return)

    문제 상황

    회원가입 시 이메일 중복 체크 이전에 비밀번호 암호화가 실행되어, 중복 이메일로 인해 요청이 실패하는 경우에도 불필요한 암호화 연산(BCrypt)이 수행되고 있었다. BCrypt는 비용이 큰 연산이므로 실패가 확정된 요청에서는 실행되어서는 안 된다.

    구분 개선 전 개선 후
    중복 이메일 실패 시 DB 조회 + 암호화 연산 모두 수행 DB 조회만 수행 후 즉시 예외 
    // 개선 전: 암호화 후 중복 체크
String encodedPassword = passwordEncoder.encode(signupRequest.getPassword()); // 비용 큰 연산 먼저
if (userRepository.existsByEmail(signupRequest.getEmail())) {
    throw new InvalidRequestException("이미 존재하는 이메일입니다.");
}

// 개선 후: 중복 체크 후 암호화 (Early Return 적용)
if (userRepository.existsByEmail(signupRequest.getEmail())) {
    throw new InvalidRequestException("이미 존재하는 이메일입니다.");  // 실패 시 즉시 종료
}
String encodedPassword = passwordEncoder.encode(signupRequest.getPassword()); // 검증 통과 후 수행

    3. WeatherClient — 응답 검증 순서 개선

    문제 상황

    HTTP 응답 body를 먼저 꺼낸 뒤 상태 코드를 검증하는 논리적으로 뒤바뀐 순서였다. 통신 자체가 실패한 상황에서도 body를 먼저 읽으려 하기 때문에 NPE 또는 의미 없는 에러가 발생할 수 있다. 

    // 개선 전: body를 먼저 꺼낸 후 상태 코드 확인 (논리적 역순)
WeatherDto[] weatherArray = responseEntity.getBody();
if (!HttpStatus.OK.equals(responseEntity.getStatusCode())) {
    throw new ServerException("날씨 데이터를 가져오는데 실패했습니다.");
}

// 개선 후: 상태 코드 확인 → body null 체크 순서로 변경
if (!HttpStatus.OK.equals(responseEntity.getStatusCode())) {
    throw new ServerException("날씨 데이터를 가져오는데 실패했습니다. 상태 코드: "
            + responseEntity.getStatusCode());  // 상태 코드 포함해 원인 명확화
}
WeatherDto[] weatherArray = responseEntity.getBody();
if (weatherArray == null || weatherArray.length == 0) {
    throw new ServerException("날씨 데이터가 없습니다.");  // 데이터 부재는 별도 처리
}
    개선 효과 HTTP 통신 실패와 데이터 부재를 별도의 예외로 구분하여 에러 원인 파악이 명확해졌다. 상태 코드를 에러 메시지에 포함시켜 디버깅 시간도 단축된다.

    4. UserService — 비밀번호 검증을 DTO로 이동 (책임 분리)

    문제 상황

    비밀번호 형식 검증 로직이 Service 계층에 있어 책임 분리 원칙을 위배하고 있었다. 형식 검증은 입력 데이터의 유효성을 확인하는 것으로, Service가 아닌 DTO 레벨에서 처리하는 것이 올바르다. 

    // 개선 전: Service에서 직접 검증 (책임 분리 위배)
if (userChangePasswordRequest.getNewPassword().length() < 8 ||
        !userChangePasswordRequest.getNewPassword().matches(".*\\d.*") ||
        !userChangePasswordRequest.getNewPassword().matches(".*[A-Z].*")) {
    throw new InvalidRequestException("새 비밀번호는 8자 이상이어야 하고, 숫자와 대문자를 포함해야 합니다.");
}
    // 개선 후: DTO에 Jakarta Validation 어노테이션으로 선언적 검증
@NotBlank
@Size(min = 8, message = "새 비밀번호는 8자 이상이어야 합니다.")
@Pattern(regexp = "^(?=.*\\d)(?=.*[A-Z]).+$",
        message = "새 비밀번호는 숫자와 대문자를 포함해야 합니다.")
private String newPassword;
    구분 개선 전 개선 후
    검증 시점 Service 계층 Controller 진입 전 (DTO)
    책임 분리 Service가 형식 검증 담당 DTO가 자기 자신을 검증
    재사용성 Service에 종속 어디서든 DTO만으로 검증 가능
    에러 응답 수동 예외 처리 @Valid로 자동 처리

    5. TodoRepository — N+1 문제 해결 (@EntityGraph)

    문제 상황

    Todo 목록을 조회할 때 각 Todo에 연관된 User를 개별 쿼리로 가져오는 N+1 문제가 발생하고 있었다. Todo가 100개라면 DB 쿼리가 101번 실행된다. 

    // 개선 전: JPQL Fetch Join
@Query("SELECT t FROM Todo t LEFT JOIN FETCH t.user ORDER BY t.modifiedAt DESC")
Page<Todo> findAllByOrderByModifiedAtDesc(Pageable pageable);

// 개선 후: @EntityGraph (동일한 성능, 더 간결한 코드)
@EntityGraph(attributePaths = {"user"})
Page<Todo> findAllByOrderByModifiedAtDesc(Pageable pageable);
    구분 N+1 발생 시 (개선 전) @EntityGraph 적용 후
    쿼리 수 1 + N번 1번 (LEFT JOIN)
    Todo 100개 기준 101번 쿼리 1번 쿼리
    @EntityGraph vs JPQL Fetch Join 둘 다 성능은 동일하게 LEFT OUTER JOIN을 실행한다. @EntityGraph는 JPQL 없이 어노테이션 한 줄로 선언할 수 있어 코드가 더 간결하고, 의도가 명확하다. 반면 JPQL Fetch Join은 복잡한 조건이 필요하거나 여러 연관 관계를 커스텀하게 제어해야 할 때 적합하다.

    6. Interceptor + AOP 관리자 API 로깅 시스템

    요구사항

    관리자 전용 API에 접근할 때마다 요청 정보를 로깅하고, 권한 없는 사용자의 접근을 차단해야 한다. Interceptor로 권한 검증과 접근 차단을 처리하고, AOP로 요청/응답 본문을 JSON으로 상세 로깅하는 방식으로 역할을 분담했다.

    구분 Interceptor AOP
    목적 권한 검증 및 접근 차단 요청/응답 상세 로그 기록
    처리 시점 Controller 진입 전 메서드 실행 전후
    로깅 내용 userId, 요청 시각, URL 요청 본문(JSON), 응답 본문(JSON), 수행시간
    실행 순서 먼저 실행 권한 통과 후 실행

    Interceptor — 권한 검증 

    @Slf4j
@Component
public class CustomInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response,
                             Object handler) throws Exception {
        String method = request.getMethod();
        String path = request.getRequestURI();

        // Admin API 여부 확인
        boolean isAdminApi = (method.equals("DELETE") && path.matches("/admin/comments/\\d+")) ||
                             (method.equals("PATCH")  && path.matches("/admin/users/\\d+"));
        if (!isAdminApi) return true;

        // 권한 검증
        String userRole = (String) request.getAttribute("userRole");
        if (userRole == null) {
            response.sendError(HttpServletResponse.SC_FORBIDDEN, "접근 권한이 없습니다.");
            return false;
        }
        UserRole role = UserRole.of(userRole);
        if (!UserRole.ADMIN.equals(role)) {
            response.sendError(HttpServletResponse.SC_FORBIDDEN, "접근 권한이 없습니다.");
            return false;
        }

        Long userId = (Long) request.getAttribute("userId");
        log.info("[관리자 기능 접근] userId = {}, requestTime = {}, url = {}",
                userId, LocalDateTime.now(), path);
        return true;
    }
}

    커스텀 어노테이션 + AOP — JSON 상세 로깅 

    // 커스텀 어노테이션
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface AdminLogging {}

// AOP Aspect
@Aspect
@Component
@Slf4j
public class Logger {

    private final ObjectMapper mapper = new ObjectMapper();

    @Around("@annotation(org.example.expert.config.AdminLogging)")
    public Object executionLogger(ProceedingJoinPoint joinPoint) throws Throwable {
        MethodSignature signature = (MethodSignature) joinPoint.getSignature();
        String className  = joinPoint.getTarget().getClass().getSimpleName();
        String methodName = joinPoint.getSignature().getName();

        String[] parameterNames = signature.getParameterNames();
        Object[] args = joinPoint.getArgs();

        Map<String, Object> paramsMap = new LinkedHashMap<>();
        for (int i = 0; i < parameterNames.length; i++) {
            paramsMap.put(parameterNames[i], args[i]);
        }

        HttpServletRequest request = ((ServletRequestAttributes)
                RequestContextHolder.getRequestAttributes()).getRequest();
        Long   userId = (Long)   request.getAttribute("userId");
        String url    = request.getRequestURI();

        log.info("[API 요청] userId = {}, url = {}, {}.{} | 요청: [{}]",
                userId, url, className, methodName, mapper.writeValueAsString(paramsMap));

        long start = System.currentTimeMillis();
        Object proceed = joinPoint.proceed();
        long end = System.currentTimeMillis();

        log.info("[API 응답] userId = {}, {}.{} | 응답: {} (수행시간: {}ms)",
                userId, className, methodName,
                mapper.writeValueAsString(proceed), (end - start));

        return proceed;
    }
}
    // Controller에 어노테이션 적용
@AdminLogging
@DeleteMapping("/admin/comments/{commentId}")
public void deleteComment(@PathVariable long commentId) {
    commentAdminService.deleteComment(commentId);
}

@AdminLogging
@PatchMapping("/admin/users/{userId}")
public void changeUserRole(@PathVariable long userId,
                           @RequestBody UserRoleChangeRequest request) {
    userAdminService.changeUserRole(userId, request);
}
    로그 출력 예시 Interceptor 로그 (권한 검증):
    [관리자 기능 접근] userId = 1, requestTime = 2026-03-06T10:30:00, url = /admin/comments/123

    AOP 로그 (JSON 상세):
    [API 요청] userId = 1, url = /admin/comments/123, CommentAdminController.deleteComment | 요청: [{"commentId":123}]
    [API 응답] userId = 1, CommentAdminController.deleteComment | 응답: null (수행시간: 45ms)

    7. ManagerService — Todo User null 체크 추가

    문제 상황

    테스트 코드에서 todo.getUser()가 null인 경우를 검증하는 시나리오가 있었지만, 실제 서비스 코드에는 null 체크가 없어 NullPointerException이 발생할 수 있는 구조였다. 테스트가 검증하려는 시나리오와 실제 코드가 일치하지 않는 상태였다. 

    @Transactional
public ManagerSaveResponse saveManager(AuthUser authUser, long todoId,
                                       ManagerSaveRequest managerSaveRequest) {
    User user = User.fromAuthUser(authUser);
    Todo todo = todoRepository.findById(todoId)
            .orElseThrow(() -> new InvalidRequestException("Todo not found"));

    // null 체크 추가 — 테스트 시나리오와 실제 코드를 일치시킨다.
    if (todo.getUser() == null) {
        throw new InvalidRequestException("todo에 등록된 user가 null일 수 없습니다.");
    }

    if (!ObjectUtils.nullSafeEquals(user.getId(), todo.getUser().getId())) {
        throw new InvalidRequestException("일정을 생성한 유저만 담당자를 지정할 수 있습니다.");
    }
    // ... 이하 로직
}
    방어적 프로그래밍 NPE를 사전에 방지하고 명확한 에러 메시지를 제공하는 것이 핵심이다. 테스트 코드가 검증하는 시나리오는 반드시 실제 코드에도 반영되어야 한다. 테스트가 통과한다고 해서 실제 코드가 안전한 것은 아니다.

    8. RestTemplate Bean 등록 (IoC/DI 원칙 준수)

    문제 상황

    WeatherClient가 생성자 내부에서 RestTemplateBuilder로 직접 RestTemplate 인스턴스를 생성하고 있었다. 이는 제어 역전(IoC) 원칙을 위배하며, 객체 생성의 책임이 Spring이 아닌 WeatherClient에 있는 구조다.

    구분 기존 방식 (내부 생성) 개선 방식 (Bean 등록)
    제어권 WeatherClient가 직접 생성/관리 스프링 컨테이너가 생성 및 관리 (IoC)
    확장성 API 클라이언트마다 빌더 코드 반복 RestTemplate 하나를 여러 클라이언트에서 공유 가능

    9. 테스트 커버리지

    과제 전반에 걸쳐 단위 테스트와 통합 테스트를 작성하고 커버리지를 측정했다. 아래는 최종 테스트 커버리지 결과다.

    마무리

    오늘 과제에서 공통적으로 느낀 점은 각 계층의 책임을 명확히 분리하는 것이 Spring 개발의 핵심이라는 것이다. 형식 검증은 DTO가, 비즈니스 규칙은 Service가, 접근 제어는 Interceptor/AOP가 담당하도록 경계를 명확히 그을 때 코드의 유지보수성과 확장성이 크게 향상된다. 특히 Early Return(암호화 연산 제거, 상태 코드 우선 확인)은 실패 케이스에서 불필요한 연산을 차단하는 간단하지만 중요한 기법이다.

    N+1 문제는 @EntityGraph 한 줄로 해결했다. JPQL Fetch Join과 성능이 동일하면서도 코드가 훨씬 간결해지는 경험을 했다. 다만 복잡한 조건이 필요한 경우에는 여전히 JPQL이 적합하다는 점을 기억해야 한다. Interceptor와 AOP의 역할 분담도 명확히 정리됐다. Interceptor는 요청이 컨트롤러에 도달하기 전 권한을 차단하고, AOP는 통과한 요청에 대해 메서드 레벨로 로그를 남긴다.

    RestTemplate을 Bean으로 등록한 것은 단순해 보이지만 IoC 원칙을 실질적으로 적용한 사례다. 객체 생성 책임을 Spring에 위임하면 설정 변경이 한 곳에서 이루어지고, 의존성 주입을 통해 테스트 시 Mock으로 교체하기도 쉬워진다. 과제 전반에서 개선한 8가지 항목 모두 "왜 이렇게 해야 하는가"에 대한 이유가 명확했고, 그것이 Spring의 설계 철학과 정확히 일치한다는 것을 확인하는 시간이었다.
