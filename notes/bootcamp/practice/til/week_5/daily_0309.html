    1. 클라이언트 · 네트워크 · 서버

    구성 요소 역할 설명
    클라이언트 요청하는 주체 사용자의 명령을 받아 서버에 요청(Request)을 보내는 기기 또는 소프트웨어다.
    네트워크 데이터의 통로 클라이언트와 서버가 데이터를 주고받을 수 있도록 연결하는 망이다. HTTP 프로토콜이라는 대화 규칙을 사용한다.
    서버 제공하는 주체 24시간 대기하며 요청이 오면 처리 후 네트워크를 통해 응답(Response)을 반환한다.
    IP 주소 클라이언트는 서버의 위치를 알아야 하고, 서버 또한 클라이언트의 위치를 알아야 한다. 서로 요청과 응답을 주고받기 위한 이 고유 주소를 IP라고 한다.

    2. 온프레미스 vs 클라우드

    구분 온프레미스 (On-Premises) 클라우드
    서버 소유 회사가 직접 구매 AWS, GCP, Azure 등 업체에서 임대
    설치/관리 직접 수행 클릭 몇 번으로 할당/반납
    네트워크 직접 구축 및 관리 관리 일부를 업체에 위임
    유연성 낮음 높음
    클라우드의 장점 5가지 초기 투자 비용 없음 · 인건비 절약 · 빠른 시작 · 확장 용이 · 실패 비용 낮음

    3. 클라우드 서비스 모델 (IaaS / PaaS / SaaS)

    모델 제공 범위 내가 관리하는 것 예시
    IaaS
    Infrastructure as a Service    		 서버, 네트워크, 스토리지 등 인프라만 빌림 OS, 런타임, 앱, 데이터 등 거의 모두 AWS EC2
    PaaS
    Platform as a Service    		 런타임, 미들웨어까지 포함한 플랫폼 제공 애플리케이션 코드 작성 및 배포 AWS Elastic Beanstalk
    SaaS
    Software as a Service    		 완성된 소프트웨어 전체 제공 툴 사용만 Gmail, Notion
    공동 책임 모델 (Shared Responsibility) 고객 책임 (Security In the Cloud) — 클라우드 "안에서" 내가 만든 것의 보안
    데이터 암호화, IAM 사용자/권한 관리, 보안그룹 설정, OS 패치, 애플리케이션 보안

    AWS 책임 (Security Of the Cloud) — 클라우드 "자체" 인프라의 보안
    물리적 데이터 센터 보안, 네트워크 인프라, 관리형 서비스 기본 보안

    IaaS에 가까울수록 고객의 보안 책임이 크고, SaaS에 가까울수록 줄어든다.

    4. AWS 기본 구조

    개념 설명
    데이터 센터 컴퓨팅 시스템 및 관련 HW 장비를 저장하는 물리적 시설이다. 모든 회사의 디지털 데이터가 여기에 저장된다.
    Account AWS 서비스를 사용하기 위한 기본 단위다. 결제, 보안, 리소스 관리의 경계가 된다.
    Region 지리적 영역(서울, 미국 동부 등)이다. 각 Region은 독립적으로 운영된다.
    Availability Zone (AZ) Region 안에 있는 개별 물리적 데이터 센터다. 하나의 Region 안에 여러 AZ가 존재한다.
    여러 AZ를 제공하는 이유 — 고가용성 (High Availability) 하나의 데이터 센터에 장애가 발생해도 다른 AZ에서 서비스를 유지하기 위함이다.
    가용성이 높다는 것은 24시간 접속 가능하고 장애가 거의 없는 상태를 의미한다. 가용성이 낮다는 것은 자주 서버가 다운되거나 접속 불가 시간이 많은 상태다.
    AWS 주요 서비스 한 줄 요약 무엇이든 올릴 수 있는 서버가 필요하면 → EC2
    데이터베이스가 필요하면 → RDS
    이미지/파일을 저장해야 하면 → S3

    5. IAM (Identity and Access Management)

    AWS에서 누가 어떤 리소스에 무엇을 할 수 있는지를 정하고 통제하는 접근 제어 시스템이다. 사람뿐만 아니라 서버에도 권한을 부여할 수 있다.

    개념 대상 설명
    User 사람이 AWS에 접근할 때 개발자가 콘솔에 로그인하거나 CLI로 AWS를 관리할 때 사용한다. 영구적인 자격증명(Credential)을 가진다.
    Role 서버/서비스에 권한 부여 EC2가 S3에 파일을 업로드하거나 GitHub Actions가 배포할 때 사용한다. 임시로 빌려 쓰는 권한 세트다.
    Policy 권한 정의 어떤 작업을 허용/거부할지 정의한다. User나 Role에 연결하는 형태로 사용한다.
    ARN 리소스 식별 AWS 리소스의 고유 주소다. 형식: arn:aws:{서비스}:...

    자격증명 (Credential)

    종류 용도 특징
    비밀번호 AWS 콘솔 로그인용 한 번 생성하면 삭제 전까지 유효(만료 없음). 1인 1계정 원칙 권장.
    Access Key
    (Access Key ID + Secret Access Key)    		 API/CLI 접근용
    ⚠️ 보안 원칙 최소 권한 원칙: 작업에 꼭 필요한 권한만 부여하고, 그 외 모든 권한은 거부한다.
    루트 사용자 보호: 계정 생성 시 만들어지는 루트 사용자는 모든 권한을 가지므로 일상적인 작업에 사용하면 안 된다.
    Access Key 주의: 영구적으로 유효하므로 유출되면 큰 사고로 이어진다. 서버/자동화에서는 User 대신 Role 사용을 권장한다.

    6. 보안 그룹 (Security Group)

    AWS 리소스에 대한 네트워크 트래픽을 제어하는 가상 방화벽이다.

    규칙 설명
    Whitelist 방식 차단 규칙이 없고 허용 규칙만 존재한다. 규칙에 없는 모든 트래픽은 자동 차단된다.
    Stateful (상태 저장) 인바운드(들어올 때) 허락된 트래픽은 아웃바운드(나갈 때) 검사하지 않는다.

    7. VPC와 네트워크 구성

    VPC(Virtual Private Cloud)는 AWS 클라우드 내에 만드는 사설 네트워크다. 다른 AWS 계정의 네트워크와 논리적으로 완벽히 격리되어 있다.

    VPC 구성 요소

    구성 요소 설명
    IP 주소 범위 VPC 생성 시 총 사용 가능한 IP 수를 정한다. 표준값 10.0.0.0/16으로 약 65,000개의 IP를 사용할 수 있다.
    Subnet (서브넷) VPC 네트워크를 관리하기 편하게 작게 쪼갠 구역이다. 하나의 서브넷은 반드시 하나의 AZ에만 속한다.
    Route Table 네트워크 트래픽이 어디로 가야 하는지 방향을 알려주는 표지판이다. 모든 서브넷은 반드시 하나의 라우팅 테이블과 연결되어야 한다.
    Internet Gateway (IGW) VPC와 외부 인터넷을 연결하는 유일한 관문이다. 하나의 VPC에 하나의 IGW만 연결 가능하다. IGW가 없으면 인터넷 접속 불가.

    Public Subnet vs Private Subnet

    라우팅 테이블에 IGW로 향하는 경로가 있는지 없는지로 결정된다.

    구분 조건 라우팅 테이블 규칙
    Public Subnet 라우팅 테이블에 IGW 경로가 있음 172.31.0.0/16 → local (VPC 내부)
    0.0.0.0/0 → igw-xxxxxxx (인터넷 연결)
    Private Subnet 라우팅 테이블에 IGW 경로가 없음 172.31.0.0/16 → local (VPC 내부 통신만 가능)

    NAT Gateway

    Private Subnet에 있는 EC2는 IGW와 연결되지 않아 외부 인터넷에 접근할 수 없다. OS 업데이트나 외부 패키지 설치가 필요할 때 NAT Gateway를 통해 Private Subnet의 EC2가 인터넷에 요청은 가능하지만, 외부에서 Private Subnet으로의 요청은 여전히 불가능하다.

    종류 설명
    Zonal NAT Gateway AZ(데이터 센터)마다 별도로 설정해야 한다.
    Regional NAT Gateway Region 전체를 커버한다. 하나만 만들면 AWS가 AZ별로 자동 확장해주므로 별도 설정 없이 사용 가능하다.

    VPC Endpoint

    VPC Endpoint가 없을 때 vs 있을 때 없을 때: VPC 내부 서버가 S3에 접근하려면 인터넷(NAT Gateway)으로 나갔다가 다시 AWS S3로 돌아와야 한다. 비용이 발생하고 속도가 느리며 보안에 취약하다.

    있을 때: VPC 내부에서 S3로 바로 연결되는 전용 터널을 뚫는다. AWS 내부 네트워크를 통해 직접 연결되므로 보안이 강력하고 속도가 빠르며 데이터 전송 비용이 무료다.

    8. EC2 (Elastic Compute Cloud)

    AWS에서 제공하는 클라우드 컴퓨팅 서비스다. 내 컴퓨터에서 작성한 코드를 아마존 데이터 센터의 원격 컴퓨터로 옮겨 실행할 수 있다.

    AMI (Amazon Machine Image)

    EC2를 시작할 때 사용하는 설치 이미지다. 운영체제, 기본 패키지/도구, 초기 설정 정보가 포함되어 있다.

    EC2 IP 주소 유형

    유형 용도 특징
    Private IP VPC 내부 통신 인스턴스 생성 시 자동 할당된다. 중지/시작해도 유지된다. 인터넷에서 접근 불가.
    Public IP 인터넷 통신 인스턴스 시작 시 자동 할당된다. 중지 → 시작 시 변경된다. Private Subnet에서는 의미 없음.
    Elastic IP (EIP) 고정 Public IP IP가 변경되지 않는 고정 Public IP다. 하지만 IGW가 특정 EIP를 가진 EC2로만 트래픽을 전달하므로 트래픽 분산이 불가능해 점차 사용이 줄고 있다.
    Elastic IP의 한계 — Load Balancer가 필요한 이유 EIP를 사용하면 IGW가 해당 EC2로만 트래픽을 전달한다. TPS(Traffic Per Second)가 EC2 한 대의 한계를 초과하면 다른 EC2로 분산하지 못하고 서버가 다운될 수 있다. 이 문제를 해결하기 위해 Load Balancer(부하 균형)가 트래픽을 여러 EC2로 분산한다.

    마무리

    오늘은 클라우드의 기본 개념부터 AWS의 핵심 구성 요소까지 전체적인 그림을 그렸다. 가장 중요하게 느낀 것은 각 계층이 왜 존재하는지를 이해하는 것이다. AZ가 여러 개인 이유는 고가용성, 서브넷을 나누는 이유도 가용성과 보안, VPC Endpoint를 쓰는 이유는 인터넷을 우회하지 않아 보안과 비용 모두 잡을 수 있기 때문이다. 모든 설계 결정에는 명확한 이유가 있다.

    IAM에서 가장 중요한 원칙은 최소 권한이다. Access Key는 영구적으로 유효하므로 유출 시 치명적이고, 서버/자동화에는 User 대신 Role을 부여해야 한다. 보안 그룹은 Whitelist 방식과 Stateful 특성 덕분에 들어오는 요청만 제어하면 나가는 응답은 자동으로 허용된다는 점도 기억해야 한다.

    EC2의 Public IP는 중지/시작 시 변경되고, Elastic IP는 고정이지만 트래픽 분산이 불가능하다. 실제 서비스 환경에서는 EIP 대신 Load Balancer를 앞에 두고 여러 EC2로 트래픽을 분산하는 구조를 사용한다. Private Subnet의 EC2가 인터넷에 요청을 보내야 할 때는 NAT Gateway를 경유하지만 외부에서 직접 접근은 차단된다는 점에서 보안과 기능 사이의 균형을 잡은 설계임을 알 수 있다.
