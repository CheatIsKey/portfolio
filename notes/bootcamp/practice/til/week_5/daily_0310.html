    58일차 개요

    오늘 목표

    • 운영 3요소(Health · Log · Config)의 개념을 이해한다.
    • Spring Boot Actuator의 엔드포인트와 노출 범위 설정을 익힌다.
    • Profile별 설정 분리로 하나의 JAR로 모든 환경을 대응하는 방법을 이해한다.
    • AWS Parameter Store에 설정값을 저장하고 연동하는 방법을 학습한다.
    • JAR 빌드 후 EC2에 배포하고 nohup으로 백그라운드 실행하는 전체 흐름을 익힌다.

    1. 운영 3요소

    요소 역할
    Health 서버 상태를 확인한다. DB, Redis 등 의존 서비스의 정상 여부를 포함한다.
    Log 문제 발생 시 원인을 추적한다. 언제, 어디서, 무슨 에러가 났는지 기록한다.
    Config 환경별 설정을 관리한다. 로컬·개발·운영 환경마다 다른 설정을 안전하게 분리한다.

    2. Spring Boot Actuator

    앱의 상태, 메트릭, 환경 정보를 HTTP 엔드포인트로 제공하는 Spring Boot 내장 모니터링 도구다. 의존성을 추가하면 /actuator/health, /actuator/info 같은 엔드포인트가 자동으로 생성된다. 

    implementation 'org.springframework.boot:spring-boot-starter-actuator'

    주요 엔드포인트

    엔드포인트 용도 민감도 기본 노출
    /actuator/health 앱 상태 확인 (DB, Redis 등 자동 감지) 🟢 낮음 ✅
    /actuator/info 앱 정보 (버전, Git 커밋 정보) 🟢 낮음 ❌
    /actuator/metrics 성능 지표 (CPU, 메모리, 요청 수) 🟡 중간 ❌
    /actuator/env 환경변수, 설정값 🔴 높음 ❌
    /actuator/configprops 모든 설정값 🔴 높음 ❌
    /actuator/heapdump JVM 메모리 덤프 🔴 높음 ❌
    /actuator/threaddump 스레드 정보 🟡 중간 ❌

    노출 범위 및 info 설정 

    # 노출할 엔드포인트 지정
management.endpoints.web.exposure.include=health,info
management.info.env.enabled=true

# /actuator/info 호출 시 나오는 정보
info.app.name=health-app
info.app.version=1.0.0

    상세 응답 설정 

    # always: DB 상태 등 상세 정보 포함 / never: 상세 정보 숨김
management.endpoint.health.show-details=always
    ⚠️ 상세 응답 주의 show-details=always로 설정하면 DB 연결 정보 등 민감한 내부 정보가 노출될 수 있다. 운영 환경에서는 가급적 never로 유지하거나, 내부 네트워크에서만 접근 가능하도록 제한해야 한다.

    3. Profile별 설정 분리

    코드는 동일하고 설정만 다르게 하면 환경별 배포가 가능해진다. 하나의 JAR 파일로 로컬, 개발, 운영 환경 모두를 대응할 수 있다.

    파일 용도
    application.properties 모든 환경에서 공통으로 적용되는 설정
    application-local.properties 로컬 개발 환경 전용 설정
    application-prod.properties 운영 환경 전용 설정
    우선순위 규칙 동일한 키가 여러 파일에 존재하면 application-{profile}.properties의 설정이 우선 적용된다. 공통 설정을 application.properties에 두고, 환경별로 덮어써야 할 값만 각 프로파일 파일에 작성하는 것이 관리하기 좋다.
    Profile 분리의 장점 코드 수정 없이 환경 변경 가능 · 비밀번호를 환경변수로 안전하게 관리 · 하나의 JAR로 모든 환경 대응

    4. AWS Parameter Store

    설정값을 안전하게 저장하고 관리하는 AWS 서비스다. DB 비밀번호처럼 코드에 하드코딩하면 안 되는 값을 외부에서 주입할 때 사용한다. 버전 관리, IAM 권한 제어를 지원하며 EC2를 재부팅해도 설정값이 유지된다.

    파라미터 생성 항목

    항목 값 설명
    Name /{앱 이름}/{profile}/{환경변수} 계층 구조로 이름을 지정한다. 예: /myapp/prod/db.password
    Type String / StringList / SecureString 값의 타입을 선택한다.
    Value 실제 설정값 저장할 값을 입력한다.
    타입 설명
    String 일반 문자열
    StringList 콤마(,)로 구분된 문자열 리스트
    SecureString KMS로 암호화된 문자열. 비밀번호, 시크릿 키 등 민감한 값에 사용한다.

    5. JAR 빌드 및 EC2 배포

    JAR 빌드 

    # 프로젝트 디렉토리에서 실행
./gradlew clean bootJar

# 빌드 결과 확인
ls -la build/libs/

    EC2 사전 세팅

    보안 그룹에서 8080 포트를 인바운드 규칙으로 추가해야 외부에서 앱에 접근할 수 있다.

    # EC2에 SSH 접속 후 Java 17 설치
sudo dnf install java-17-amazon-corretto -y

# 설치 확인
java -version
    명령어 옵션 설명 sudo: 관리자 권한으로 실행한다.
    dnf: 패키지 관리자다. 소프트웨어 설치/삭제 도구.
    -y: 설치 확인 질문에 자동으로 "yes"로 응답한다.

    IAM Role 설정

    EC2가 Parameter Store에 접근하려면 IAM Role에 AmazonSSMReadOnlyAccess 권한을 추가해야 한다. 직접 Access Key를 EC2에 저장하는 방식은 보안에 취약하므로, Role을 부여하는 방식을 사용한다.

    파일 전송 (scp) 

    # 키페어 권한 설정 (최초 1회)
chmod 400 <키페어경로>

# 로컬 → EC2 파일 전송
scp -i <키페어경로> <로컬파일> <사용자>@<EC2-IP>:~/app.jar

# EC2 → 로컬 파일 가져오기 (예: 로그 파일)
scp -i ~/.ssh/my-key.pem ec2-user@{EC2-IP}:~/app.log ./

    6. 포그라운드 vs 백그라운드 실행 (nohup)

    포그라운드 실행의 문제 

    # 포그라운드 실행 — 터미널을 닫으면 앱도 함께 종료된다
java -jar ~/app.jar --spring.profiles.active=prod
    ⚠️ 포그라운드 실행의 한계 터미널(SSH 세션)을 닫는 순간 스프링 애플리케이션 실행이 즉시 중단된다. 실제 운영 환경에서는 반드시 백그라운드 실행 방식을 사용해야 한다.

    nohup (No Hang Up) — 백그라운드 실행

    nohup은 SSH가 끊어져도 앱이 계속 실행되도록 해준다. 로그는 지정한 파일(app.log)에 기록된다. 

    # 백그라운드 실행 — SSH가 끊겨도 앱이 계속 동작한다
nohup java -jar ~/app.jar --spring.profiles.active=prod > ~/app.log 2>&1 &

# 실행 확인 — java 프로세스가 보이면 정상 실행 중
ps aux | grep java

# 로그 실시간 확인 — Ctrl+C로 종료
tail -f ~/app.log

# 앱 종료
pkill -9 -f "java -jar"
    옵션 설명
    > ~/app.log 표준 출력(stdout)을 app.log 파일로 저장한다.
    2>&1 표준 에러(stderr)도 app.log에 함께 저장한다.
    & 백그라운드에서 실행한다.
    pkill -9 -f 프로세스 이름으로 강제 종료한다.

    Parameter Store 연동 확인 

    # 앱이 Parameter Store 값을 정상적으로 읽어왔는지 확인
curl http://{EC2-IP}:8080/actuator/info

    정상적으로 연동되었다면 info.app.name, info.app.version 등 Parameter Store에 저장한 값이 응답으로 돌아온다.

    마무리

    오늘은 로컬에서 작성한 스프링 앱을 실제 EC2에 배포하는 전체 흐름을 처음부터 끝까지 직접 경험했다. 핵심은 코드와 설정을 분리하는 것이다. 하나의 JAR 파일이 Profile에 따라 전혀 다른 설정으로 동작하고, 민감한 설정값은 Parameter Store에 저장해 코드에 절대 하드코딩하지 않는다. EC2가 Parameter Store에 접근하는 권한도 Access Key가 아닌 IAM Role로 부여해 보안 원칙을 지켰다.

    Actuator는 운영 환경에서 앱 상태를 모니터링하는 강력한 도구지만, 민감도가 높은 엔드포인트를 무분별하게 노출하면 오히려 보안 취약점이 된다. health와 info만 기본 노출하고, 나머지는 필요할 때만 열거나 내부 네트워크에서만 접근하도록 제한하는 것이 올바른 방향이다. show-details=always도 운영 환경에서는 지양해야 한다.

    포그라운드 실행은 터미널이 닫히는 순간 앱도 종료된다는 치명적인 한계가 있다. nohup으로 백그라운드 실행하고 로그를 파일로 남기는 것이 기본이다. ps aux | grep java로 프로세스 확인, tail -f ~/app.log로 실시간 로그 확인, pkill -9로 종료하는 세트를 기억해두면 EC2 운영 중 대부분의 상황에 대응할 수 있다.
