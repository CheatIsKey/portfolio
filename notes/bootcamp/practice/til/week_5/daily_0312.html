    60일차 개요

    오늘 목표

    • Docker의 개념과 컨테이너 기술이 해결하는 문제를 이해한다.
    • Docker Image와 Container의 관계를 파악한다.
    • 포트 매핑과 컨테이너 네트워크 격리 원리를 이해한다.
    • Dockerfile의 핵심 명령어와 이미지 빌드 방법을 익힌다.
    • MySQL 컨테이너와 Spring Boot 컨테이너를 연결해 로컬에서 전체 스택을 실행한다.
    • host.docker.internal의 역할과 컨테이너 간 통신 방법을 이해한다.

    1. Docker란 무엇인가

    SW 개발과 배포를 도와주는 컨테이너(Container) 기술이다. 애플리케이션을 빌드, 배포, 실행하는 전 과정을 표준화된 방식으로 관리할 수 있다. Docker가 없을 때 배포 환경이 달라지면 겪는 문제들을 이미지 하나에 담아 해결한다.

    비유로 이해하는 Docker CD(디스크)에는 프로그램과 실행에 필요한 모든 것이 담겨 있다. CD가 있으면 친구 집에 가서도 똑같이 프로그램을 실행할 수 있다.

    Docker Image = CD — 실행에 필요한 모든 것을 담은 패키지
    Docker Container = 실행된 프로그램 — 이미지를 실행한 인스턴스

    Docker는 프로그램 코드뿐만 아니라 실행 환경(OS, Java 버전, 라이브러리)까지 패키징하므로 어디서 실행해도 동일한 동작을 보장한다.

    Docker가 해결하는 문제

    Docker 없이 배포 시 문제 증상
    Java 버전이 다르다 실행 불가
    라이브러리 버전이 다르다 이상하게 동작
    새벽 배포 중 휴먼 에러 장애 발생
    Docker의 해결 방법 이미지 하나에 실행 환경 전체를 담는다. 누가, 언제, 어디서 실행해도 동일하게 동작한다.

    2. 컨테이너 네트워크와 포트 매핑

    Docker Container는 자체 네트워크를 가진 격리된 환경이다. 로컬에서 컨테이너를 띄워도 컨테이너의 네트워크와 로컬 네트워크는 기본적으로 분리되어 있어, 외부(로컬)에서 컨테이너 안의 서비스에 직접 접근할 수 없다.

    포트 매핑 (Port Mapping) -p 호스트포트:컨테이너포트 옵션으로 호스트(로컬)의 포트와 컨테이너 포트를 연결한다. 이 연결이 있어야 외부에서 컨테이너 내부 서비스에 접근할 수 있다.

    예시: -p 8080:8080 — 로컬의 8080 포트로 들어온 요청을 컨테이너 8080으로 전달한다. 
    # 호스트 8080 → 컨테이너 8080으로 포트 매핑하여 실행
docker run -p 8080:8080 camp-health-app:v1.0

    환경변수 주입

    컨테이너 실행 시 -e 옵션으로 환경변수를 주입한다. 코드에 설정값을 하드코딩하지 않고 실행 시점에 외부에서 주입하는 방식이다. 

    docker run -e SPRING_PROFILES_ACTIVE=prod \
           -e SPRING_DATASOURCE_URL=jdbc:mysql://... \
           camp-health-app:v1.0

    이미지 태그 (Tag)

    Docker Image를 버전별로 분류할 때 사용한다. 이미지 빌드 시 태그를 지정하지 않으면 자동으로 latest 태그가 붙는다. 운영 환경에서는 latest 대신 명시적인 버전 태그(v1.0, v1.1 등)를 사용하는 것이 안전하다.

    3. Dockerfile

    Docker Image를 만들기 위해 필요한 내용을 정의한 파일이다. Dockerfile이 있으면 누가 빌드하더라도 동일한 이미지가 만들어진다. 

    FROM amazoncorretto:17-al2023-headless   # 수정 포인트: Java 버전에 맞는 베이스 이미지 선택
WORKDIR /app                             # 수정 포인트: 컨테이너 내부 작업 디렉토리
COPY app.jar app.jar                     # 수정 포인트: 빌드된 JAR 파일명과 일치시켜야 함
EXPOSE 8080                              # 수정 포인트: 앱이 사용하는 포트로 변경
ENTRYPOINT ["java", "-jar", "app.jar"]   # 수정 포인트: JAR 파일명과 일치시켜야 함
    명령어 하는 일 비고
    FROM 베이스 이미지를 지정한다. 모든 Dockerfile의 첫 번째 명령어다. OS + 런타임 환경이 포함된 이미지를 선택한다.
    WORKDIR 컨테이너 내부의 작업 디렉토리를 설정한다. 없으면 자동 생성한다. 이후 모든 명령어는 이 경로를 기준으로 실행된다.
    COPY 호스트의 파일을 컨테이너 이미지 안으로 복사한다. COPY 호스트경로 컨테이너경로 형식이다.
    EXPOSE 컨테이너가 사용할 포트를 문서화한다. 실제로 포트를 여는 것이 아닌 선언 용도다. 실제 오픈은 -p로 한다.
    ENTRYPOINT 컨테이너 시작 시 실행할 명령어를 정의한다. 컨테이너가 시작되면 반드시 이 명령이 실행된다.
    EXPOSE vs -p 옵션 차이 EXPOSE는 "이 포트를 쓸 예정이다"라는 문서화 선언일 뿐, 실제 포트를 열지 않는다. 외부에서 실제로 접근하려면 컨테이너 실행 시 -p 호스트포트:컨테이너포트로 명시적으로 매핑해야 한다.

    4. Docker Image 빌드 및 컨테이너 실행 전체 흐름

    Step 1 — JAR 빌드 

    ./gradlew clean bootJar

    Step 2 — Docker 이미지 빌드 

    # -t: 이미지 이름:태그 지정 / .: Dockerfile 위치 (현재 디렉토리)
docker build -t camp-health-app:v1.0 .

    Step 3 — MySQL 컨테이너 실행 

    docker run -d -p 3306:3306 \
  --name mysql-db \
  -e MYSQL_ROOT_PASSWORD=12345678 \
  -e MYSQL_DATABASE=test \
  mysql:8
    옵션 설명
    -d 백그라운드(detached) 모드로 실행한다.
    -p 3306:3306 호스트 3306 → 컨테이너 3306 포트 매핑
    --name mysql-db 컨테이너에 이름을 부여한다.
    MYSQL_ROOT_PASSWORD root 비밀번호를 설정한다.
    MYSQL_DATABASE 지정한 이름의 데이터베이스를 자동 생성한다.

    Step 4 — Spring Boot 컨테이너 실행 

    docker run -d -p 8080:8080 \
  --name my-app \
  -e SPRING_PROFILES_ACTIVE=local \
  -e SPRING_DATASOURCE_URL=jdbc:mysql://host.docker.internal:3306/test \
  -e SPRING_DATASOURCE_USERNAME=root \
  -e SPRING_DATASOURCE_PASSWORD=12345678 \
  -e SPRING_JPA_HIBERNATE_DDL_AUTO=update \
  camp-health-app:v1.0
    host.docker.internal 컨테이너 안에서 호스트 머신의 localhost를 가리키는 Docker Desktop이 제공하는 특수 주소다.

    MySQL 컨테이너와 Spring Boot 컨테이너는 각각 별도로 실행되며 격리된 네트워크를 가진다. Spring Boot 컨테이너 안에서 localhost:3306을 쓰면 컨테이너 자신의 3306을 가리키게 되어 MySQL을 찾지 못한다. host.docker.internal:3306을 사용해야 호스트 머신에서 실행 중인 MySQL 컨테이너의 포트로 올바르게 연결된다.
    환경변수 설명
    SPRING_DATASOURCE_URL spring.datasource.url 설정을 환경변수로 덮어쓴다.
    SPRING_PROFILES_ACTIVE 활성화할 Spring Profile을 지정한다.
    SPRING_JPA_HIBERNATE_DDL_AUTO DDL 자동 생성 전략을 지정한다. (update, create 등)
    ⚠️ Spring Boot 컨테이너 실행 순서 Spring Boot 컨테이너를 실행하기 전에 반드시 Docker Image를 먼저 빌드해야 한다. 이미지 없이 docker run을 실행하면 이미지를 찾을 수 없다는 오류가 발생한다. 순서: JAR 빌드 → Docker 이미지 빌드 → MySQL 컨테이너 실행 → Spring Boot 컨테이너 실행

    마무리

    Docker의 핵심 가치는 실행 환경을 코드처럼 관리한다는 것이다. Dockerfile에 베이스 이미지, 파일 복사, 실행 명령을 정의해두면 누구든 동일한 환경을 재현할 수 있다. "내 로컬에서는 되는데 서버에서 안 된다"는 상황이 원천적으로 차단된다. Java 버전 불일치, 라이브러리 충돌, 새벽 배포 휴먼 에러 같은 문제들이 이미지 하나로 해결된다.

    컨테이너 네트워크 격리와 포트 매핑의 관계를 이해하는 것이 중요하다. 컨테이너는 자체 네트워크를 가지므로 기본적으로 외부에서 접근할 수 없다. -p 호스트포트:컨테이너포트로 명시적으로 연결해야 외부 접근이 가능해진다. EXPOSE는 포트를 실제로 여는 것이 아닌 선언 용도임을 혼동하지 않아야 한다.

    MySQL과 Spring Boot를 별도 컨테이너로 실행할 때 host.docker.internal이 핵심이다. 컨테이너 안에서 localhost는 컨테이너 자신을 가리키기 때문에 호스트의 MySQL 포트로 연결되지 않는다. host.docker.internal을 사용해야 호스트 머신을 통해 MySQL 컨테이너에 올바르게 접근할 수 있다. 환경변수로 spring.datasource.url을 런타임에 주입하는 방식은 Profile 기반 설정 분리와도 잘 맞는 패턴이다.
