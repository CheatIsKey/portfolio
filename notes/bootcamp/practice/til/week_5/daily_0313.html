    작성일 2026-03-13 · 분류: 부트캠프, TIL, AWS, Spring Boot, Docker, CI/CD, ALB, Auto Scaling

    61일차 개요

    오늘 목표

    • 팀원 소개 REST API를 AWS 인프라 위에 단계별로 배포한다.
    • VPC 설계, EC2 배포, Actuator 헬스체크를 구성한다.
    • RDS 분리 + 보안 그룹 체이닝 + Parameter Store로 보안 연결을 구성한다.
    • S3 프로필 사진 업로드 / Presigned URL을 IAM Role 방식으로 구현한다.
    • Docker + GitHub Actions CI/CD로 자동 배포 파이프라인을 구성한다.
    • ALB + Auto Scaling + HTTPS 도메인으로 고가용성 아키텍처를 완성한다.

    1. 전체 시스템 아키텍처

    클라이언트의 HTTPS 요청이 Cloudflare DNS를 거쳐 ALB로 전달되고, ALB가 Private Subnet의 Auto Scaling Group 내 EC2 인스턴스들에 트래픽을 분산한다. 각 EC2는 Parameter Store에서 설정을 주입받고, RDS와 S3에 접근한다.

    단계 목표 핵심 기술
    LV 0 비용 관리 — AWS Budget 알림 설정 AWS Budgets
    LV 1 VPC 설계, EC2 배포, Actuator 헬스체크 EC2, Spring Actuator
    LV 2 RDS 분리, 보안 그룹 체이닝, Parameter Store RDS, AWS SSM
    LV 3 S3 프로필 사진 업로드 / Presigned URL 다운로드 S3, IAM Role
    LV 4 Docker 컨테이너화 + GitHub Actions CI/CD Docker, GitHub Actions
    LV 5 ALB + Auto Scaling + HTTPS 도메인 연결 ALB, ASG, ACM, Route 53

    2. LV 0 — AWS Budget 비용 관리

    클라우드 인프라를 구성하기 전에 비용 폭탄을 방지하기 위한 예산 알림을 먼저 설정한다. 월 예산 $100을 설정하고, 80% 도달 시 이메일 알림이 발송되도록 구성한다.

    AWS Budget을 먼저 설정하는 이유 클라우드는 사용한 만큼 과금된다. 설정 실수로 인한 리소스 낭비나 예상치 못한 트래픽 급증으로 청구 금액이 폭증할 수 있다. 예산 알림은 모든 AWS 작업 전 첫 번째 단계여야 한다.

    3. LV 1 — VPC 설계 및 EC2 배포

    VPC 서브넷 설계

    서브넷 종류 배치 리소스 외부 접근
    Public Subnet EC2 (초기 배포), NAT Gateway 허용 (IGW 연결)
    Private Subnet EC2 (ALB 구성 후 이동), RDS 차단 (ALB · NAT를 통해서만 통신)

    API 명세

    메서드 경로 설명
    POST /api/members 팀원 등록 (name, age, mbti)
    GET /api/members/{id} 팀원 단건 조회
    POST /api/members/{id}/profile-image 프로필 사진 업로드 (multipart/form-data)
    GET /api/members/{id}/profile-image Presigned URL 발급 (유효기간 7일)
    GET /actuator/health 서버 상태 확인
    GET /actuator/info Parameter Store에서 주입된 팀 정보

    Profile별 설정 분리 

    # application-local.yml — H2 인메모리 DB 사용 (로컬 개발용)
spring:
  datasource:
    driver-class-name: org.h2.Driver
    url: jdbc:h2:mem:testdb
    username: sa
    password:
  h2:
    console:
      enabled: true   # http://localhost:8080/h2-console 접속 가능

S3_BUCKET_NAME: test-bucket   # 로컬 테스트용 더미값
    Profile 분리 전략 spring.profiles.active=local 또는 prod 하나만 바꾸면 DB 설정 전체가 자동으로 전환된다. 로컬에서는 H2를 사용하므로 MySQL 설치 없이 바로 개발할 수 있고, 운영(prod)에서는 Parameter Store에서 RDS 접속 정보가 자동 주입된다.

    4. LV 2 — RDS 분리 · 보안 그룹 체이닝 · Parameter Store

    RDS 보안 그룹 체이닝

    RDS 인바운드 규칙에 IP 주소 대신 EC2 보안 그룹 ID만 허용한다. 이렇게 하면 EC2를 통해서만 RDS에 접근할 수 있어 외부에서 DB 포트로 직접 접속하는 것이 원천 차단된다.

    보안 그룹 체이닝의 장점 IP 기반 허용은 EC2를 교체하거나 Auto Scaling으로 새 인스턴스가 추가될 때마다 IP를 업데이트해야 한다. 보안 그룹 ID 기반 허용은 EC2가 추가·교체되어도 보안 그룹에 속해 있기만 하면 자동으로 접근이 허용된다. 관리 부담이 없어지고 실수로 인한 보안 구멍도 방지된다.

    Parameter Store 연동

    DB 접속 정보, S3 버킷명, 팀 정보 등 민감한 설정을 Parameter Store에 저장하고, Spring Boot 실행 시 자동으로 주입받는다. 코드에 비밀값을 직접 입력하지 않는다. 

    /cloud-project-app/prod/
                    ├── DB_URL          → jdbc:mysql://rds-endpoint:3306/dbname
                    ├── DB_USERNAME     → (DB 계정명)
                    ├── DB_PASSWORD     → (DB 비밀번호)
                    ├── S3_BUCKET_NAME  → (S3 버킷명)
                    ├── APP_MESSAGE     → Hello, World!
                    └── TEAM-NAME       → 2조
    // GET /actuator/info 응답 확인 — Parameter Store 값이 정상 주입된 것을 검증
{
  "app": {
    "message": "Hello, World!",
    "team-name": "2조"
  }
}

    5. LV 3 — S3 프로필 사진 업로드 / Presigned URL

    IAM Role 방식 — Access Key 미사용

    Access Key를 코드나 환경변수에 하드코딩하지 않고, EC2에 IAM Role을 직접 연결한다. EC2 인스턴스가 AWS로부터 임시 자격증명을 자동으로 발급·갱신받으므로 키 유출 위험이 없다. S3 버킷은 "모든 퍼블릭 액세스 차단"을 설정하고, EC2에 연결된 IAM Role에만 접근 권한을 부여한다.

    방식 Access Key 방식 IAM Role 방식
    자격증명 관리 개발자가 직접 생성·보관·갱신 AWS가 자동으로 발급·갱신
    유출 위험 코드·Git 실수 유출 가능 없음 (키 자체가 존재하지 않음)
    권장 환경 로컬 개발 운영 EC2 (필수)
    ⚠️ "키를 코드에 넣는 순간, 그 키는 이미 유출된 것이다" Access Key를 application-prod.yml이나 환경변수에 넣어 Git에 push하는 순간 유출 위험이 생긴다. 운영 환경의 EC2에서는 반드시 IAM Role을 사용해야 한다. 
    // GET /api/members/{id}/profile-image 응답 — 7일 유효 Presigned URL 반환
{
  "status": 200,
  "data": {
    "url": "https://s3.ap-northeast-2.amazonaws.com/bucket/uploads/uuid_filename.jpg?X-Amz-Expires=604800&..."
  }
}

    6. LV 4 — Docker 컨테이너화 + GitHub Actions CI/CD

    CI/CD 파이프라인 흐름

    개발자가 master 브랜치에 코드를 push하면 GitHub Actions가 자동으로 빌드·테스트·배포까지 수행한다. EC2에 직접 SSH 접속해서 JAR를 복사하고 재시작하는 수동 작업이 완전히 사라진다. 

    개발자 코드 Push (master 브랜치)
        │
        ▼
[GitHub Actions]
  ① Build & Test
  ② Docker 이미지 빌드
  ③ Docker Hub에 Push
        │
        ▼
[EC2 자동 배포]
  ④ docker pull (최신 이미지)
  ⑤ 기존 컨테이너 중지 → 새 컨테이너 실행
    Docker 컨테이너화의 이점 Dockerfile로 패키징하면 EC2 환경(Java 버전, 라이브러리 등)과 무관하게 동일하게 동작한다. GitHub Actions에서 빌드한 이미지를 Docker Hub에 push하고, EC2에서 pull하여 실행하는 구조이므로 어떤 EC2 인스턴스가 올라와도 동일한 이미지로 실행된다. Auto Scaling과도 자연스럽게 결합된다.

    7. LV 5 — ALB + Auto Scaling + HTTPS 도메인

    HTTPS 도메인 구성

    구성 요소 역할
    ACM (AWS Certificate Manager) SSL 인증서를 발급한다. Cloudflare DNS에 CNAME 레코드를 추가해 도메인 소유권을 검증한다.
    Cloudflare DNS 도메인(api.spring-jpa.dev)의 CNAME을 ALB DNS 주소로 연결한다.
    ALB 리스너 HTTPS(443)에 ACM 인증서를 적용한다. HTTP(80)는 HTTPS(443)로 자동 리다이렉트한다.

    ALB + Auto Scaling Group 구성

    LV 5에서 EC2가 Public Subnet에서 Private Subnet으로 이동한다. 외부에서 EC2 IP로 직접 접근하는 것이 차단되고, 모든 트래픽은 ALB를 통해서만 들어온다. Private Subnet의 EC2는 NAT Gateway를 통해 외부 인터넷(Docker Hub, Parameter Store 등)에 접근한다.

    구성 요소 역할
    ALB 트래픽을 여러 EC2에 분산한다. Target Group의 헬스체크(/actuator/health)로 정상 인스턴스만 유지한다.
    Auto Scaling Group CPU 사용량 등 지표에 따라 EC2 인스턴스를 자동으로 증설/감소한다. 새 인스턴스는 자동으로 Target Group에 등록된다.
    NAT Gateway Private Subnet의 EC2가 외부 인터넷에 요청은 보낼 수 있지만, 외부에서 직접 접근은 차단된다.
    최종 아키텍처 흐름 클라이언트 → Cloudflare DNS → ALB (HTTPS 443, ACM 인증서) → Auto Scaling Group (Private Subnet EC2) → RDS / S3 / Parameter Store

    8. 핵심 기술 의사결정 정리

    의사결정 이유
    로컬 H2 / 운영 MySQL Profile 분리 코드는 하나로 유지하면서 환경별 DB를 자동 전환한다. 로컬에서 MySQL 설치 없이 개발 가능하다.
    Access Key 대신 IAM Role 사용 키 유출 위험을 원천 제거한다. AWS가 임시 자격증명을 자동 발급·갱신하므로 관리 부담이 없다.
    Parameter Store로 민감 설정 관리 DB 비밀번호, S3 버킷명 등을 코드와 완전히 분리한다. 값 변경 시 재배포 없이 적용된다.
    Presigned URL 방식 이미지 다운로드 S3를 비공개로 유지하면서 7일 유효 임시 URL만 발급한다. 서버가 파일을 중계하지 않아 대역폭 부담이 없다.
    RDS 보안 그룹 체이닝 IP 대신 보안 그룹 ID를 허용한다. EC2가 추가·교체되어도 자동 허용되며 외부 직접 접근이 차단된다.

    마무리

    이번 과제는 단순 기능 구현을 넘어 운영 수준의 클라우드 아키텍처를 단계별로 직접 구축한 경험이었다. LV 0의 예산 알림부터 LV 5의 ALB·ASG까지, 각 단계가 전 단계의 한계를 보완하는 방식으로 발전했다. EC2 하나에서 시작해 고가용성 분산 아키텍처로 진화하는 전체 흐름을 몸으로 체감했다.

    보안 측면에서 가장 인상 깊었던 것은 IAM Role과 보안 그룹 체이닝이다. Access Key를 코드에 넣는 것이 얼마나 위험한지, 그리고 보안 그룹을 IP가 아닌 그룹 ID로 연결하면 Auto Scaling 환경에서도 별도 작업 없이 새 인스턴스가 자동으로 DB 접근 권한을 얻는다는 점이 설계의 우아함을 보여줬다. Parameter Store도 단순히 환경변수를 숨기는 것이 아니라, 재배포 없이 설정을 변경할 수 있다는 운영 유연성까지 제공한다.

    Docker + GitHub Actions CI/CD를 연결하면 코드 push 한 번으로 빌드부터 EC2 배포까지 자동화된다. ALB의 Target Group 헬스체크가 /actuator/health와 연동되어 비정상 인스턴스를 자동으로 제외하고, Auto Scaling이 트래픽에 따라 EC2를 자동으로 늘이고 줄인다. Actuator를 모니터링 엔드포인트로만 쓰는 것이 아니라 인프라 자동화의 핵심 연결 고리로 활용한다는 것이 이번 과제의 핵심 인사이트였다.
