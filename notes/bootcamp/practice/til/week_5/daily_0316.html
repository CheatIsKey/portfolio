    1. ECR (Elastic Container Registry)

    AWS에서 제공하는 Docker Image 저장소다. Docker Hub와 동일한 역할을 하지만 AWS 인프라 안에서 동작하므로 IAM으로 권한을 관리할 수 있고, EC2에서 별도 로그인 없이 이미지를 가져올 수 있어 AWS 기반 배포 흐름에 자연스럽게 통합된다.

    ECR의 기본 동작 구조 빌드 서버(GitHub Actions Runner 또는 로컬)에서 Docker Image를 빌드하여 ECR에 push한다. EC2는 ECR에서 해당 이미지를 pull하여 컨테이너로 실행한다. IAM Role이 연결된 EC2는 별도의 자격증명 없이 ECR에서 이미지를 가져올 수 있다.

    로컬에서 ECR 연동하기 

    # AWS CLI 초기 설정
aws configure
    항목 입력값
    Access Key ID IAM 사용자의 Access Key
    Secret Access Key IAM 사용자의 Secret Key
    Default region ap-northeast-2
    Default output json 
    # 현재 로그인된 AWS 자격증명 확인
aws sts get-caller-identity
    {
    "UserId": "883********7005",
    "Account": "88***********05",
    "Arn": "arn:aws:iam::8**********05:root"
}

    설정이 완료되면 ECR 콘솔 페이지에서 '푸시 명령 보기' 버튼을 누르면 인증·태그·푸시 명령이 모두 생성된다. 그대로 복사해서 실행하면 된다.

    실무에서의 이미지 태그 전략 실무에서는 Git 커밋 해시를 태그로 사용하는 경우가 많다. 커밋마다 고유한 값이므로 중복이 없고, 태그만 보면 어떤 커밋이 배포됐는지 바로 확인할 수 있어 문제 발생 시 특정 버전으로 빠르게 롤백할 수 있다. latest 태그만 사용하면 어떤 코드가 배포됐는지 추적하기 어렵다.

    2. GitHub Actions

    코드를 push하면 자동으로 빌드·테스트·배포를 실행해주는 CI/CD 도구다. 기존에 수동으로 반복하던 작업을 코드(Workflow 파일) 한 장으로 자동화할 수 있다.

    수동 배포 vs GitHub Actions 자동화

    기존 수동 배포 (매번 반복) GitHub Actions 자동화
    1. ./gradlew bootJar 실행
    2. docker build 실행
    3. docker tag 실행
    4. ECR 로그인
    5. docker push 실행
    (배포할 때마다 반복)     		코드 push 한 번으로 위 과정 전체가 자동 실행된다.

    실행 흐름 

    1. 개발자가 코드 push
2. GitHub가 .github/workflows/ 에서 Workflow 파일 발견
3. Runner (임시 서버) 자동 생성
4. Job 시작 — Step 순서대로 실행
5. 결과 리포트 (성공 or 실패)
6. Runner 삭제

    핵심 용어

    용어 설명
    Workflow 자동화 작업 전체를 정의한 파일이다. .github/workflows/ 디렉토리 안의 .yml 파일로 작성한다.
    Job Workflow 안에서 독립적으로 실행되는 작업 단위다. Job은 기본적으로 병렬 실행되며, 순서가 필요하면 needs로 의존성을 지정한다.
    Step Job 안에서 순서대로 실행되는 명령어다. 앞 Step이 실패하면 이후 Step은 실행되지 않는다.
    Runner 실제로 명령어를 실행하는 임시 서버다. GitHub가 제공하며 Job이 끝나면 삭제된다.
    ⚠️ Workflow 파일 위치 주의 반드시 프로젝트 루트의 .github/workflows/ 디렉토리 안에 .yml 파일로 작성해야 한다. 이 위치가 아니면 GitHub가 Workflow 파일을 인식하지 못해 자동 실행이 전혀 동작하지 않는다.

    변수와 비밀 관리 (vars / secrets)

    공개해도 되는 설정값은 vars에, 민감한 정보는 secrets에 저장한다. 설정 위치: GitHub → Repository → Settings → Secrets and variables → Actions

    구분 용도 예시 로그 노출
    vars 공개 가능한 설정값 AWS 리전, 앱 이름 그대로 보임
    secrets 민감한 정보 SSH 키, 토큰, 계정 ID *** 마스킹 
    steps:
  - name: Use variables
    run: |
      echo "Region: ${{ vars.AWS_REGION }}"
      echo "Account ID는 마스킹됨: ${{ secrets.AWS_ACCOUNT_ID }}"

    3. CI와 OIDC 인증

    CI (Continuous Integration)

    지속적 통합(CI)은 작업한 코드를 지속적으로 빌드·테스트하여 레포지토리에 통합(merge)하는 것이다. 코드 변경이 생길 때마다 자동으로 검증하여 문제를 조기에 발견하고, 메인 브랜치를 항상 정상 동작하는 상태로 유지한다.

    OIDC 인증 — Access Key 없이 AWS 접근

    GitHub Actions에서 AWS에 접근할 때 Access Key를 사용하면 키가 GitHub secrets에 저장되고, 유출 시 큰 사고로 이어질 수 있다. 만료 관리도 개발자가 직접 해야 한다. OIDC(OpenID Connect) 인증은 이 문제를 해결하는 방법이다.

    OIDC 인증 동작 방식 1. GitHub Actions Runner가 GitHub에 신원 증명 토큰 요청
    2. GitHub가 "이 Runner는 인증됐습니다"라는 사실을 증명하는 토큰 발급
    3. Runner가 그 토큰을 AWS에 제시
    4. AWS가 토큰을 검증하고 유효 시간 1시간의 임시 자격증명 발급
    5. Runner가 임시 자격증명으로 ECR, S3 등 AWS 서비스 접근

    Access Key가 코드나 secrets 어디에도 존재하지 않으므로 유출 자체가 불가능하다.
    구분 Access Key 방식 OIDC 방식
    자격증명 저장 GitHub secrets에 영구 보관 없음 (임시 발급·자동 소멸)
    유효 시간 삭제 전까지 영구 1시간 (자동 만료)
    유출 위험 있음 없음
    키 관리 부담 직접 갱신·교체 필요 없음
    ⚠️ GitHub Actions에서 Access Key를 사용하면 안 되는 이유 Access Key가 유출되면 IAM 권한 범위 안의 모든 AWS 리소스가 노출된다. 키 만료 관리를 잊으면 보안 허점이 된다. GitHub Actions에서 AWS 접근은 반드시 OIDC 방식을 사용해야 한다.

    4. Docker Compose

    하나의 파일로 여러 개의 컨테이너 서비스를 정의하고 한 번에 실행하는 도구다. 같은 Docker Compose 파일 안의 컨테이너들은 자동으로 같은 네트워크에 배치되어 서비스 이름으로 직접 통신할 수 있다. (네트워크 분리도 설정 가능)

    Docker Compose가 유용한 상황 주로 로컬 개발 환경에서 Spring Boot + MySQL + Redis처럼 여러 컨테이너를 한 번에 편리하게 띄울 때 사용한다. 각 서비스가 서비스 이름(예: mysql-db)으로 서로를 찾을 수 있어, 이전처럼 host.docker.internal을 쓸 필요 없이 컨테이너 간 통신이 자연스럽다.

    실무에서 Docker Compose의 한계

    상황 이유
    운영(Prod) 환경 RDS, ElastiCache처럼 AWS 관리형 서비스를 사용하기 때문에 DB·캐시를 컨테이너로 직접 띄울 필요가 없다.
    Auto Scaling 환경 여러 Spring Boot 서버는 트래픽에 따라 서로 무관하게 독립적으로 스케일링되어야 한다. Docker Compose는 단일 호스트 안에서 묶어서 실행하는 구조이므로 이 요구사항을 충족하지 못한다.
    Docker Compose 사용 범위 정리 사용하는 곳: 로컬 개발 환경 — 여러 컨테이너를 명령어 하나로 편하게 띄울 때
    사용하지 않는 곳: 운영 환경 — AWS 관리형 서비스 사용, 독립적 Auto Scaling 필요

    마무리

    오늘 학습의 핵심은 CI/CD 파이프라인에서 보안을 어떻게 지킬 것인가다. GitHub Actions에서 AWS에 접근할 때 Access Key를 secrets에 저장하는 방식은 편리해 보이지만 유출 위험과 만료 관리 부담이 남는다. OIDC는 임시 자격증명을 자동으로 발급·소멸시켜 키 자체가 존재하지 않도록 함으로써 이 문제를 근본적으로 해결한다.

    ECR은 Docker Hub와 같은 역할을 하지만 IAM으로 접근을 제어할 수 있고 AWS 내부 네트워크를 사용해 빠르게 이미지를 가져올 수 있다. 이미지 태그를 latest 대신 Git 커밋 해시로 관리하면 어떤 코드가 배포됐는지 추적이 가능하고 문제 발생 시 특정 커밋 시점으로 즉시 롤백할 수 있다. 실무에서는 배포 이력 추적이 필수이므로 태그 전략을 처음부터 올바르게 설계하는 것이 중요하다.

    Docker Compose는 로컬 개발 편의성을 크게 높여주지만 운영 환경에서의 사용은 제한적이다. AWS 관리형 서비스(RDS, ElastiCache)를 사용하면 DB를 컨테이너로 직접 띄울 이유가 없고, Auto Scaling 환경에서는 각 서비스가 독립적으로 확장되어야 하므로 단일 호스트에서 여러 컨테이너를 묶는 Docker Compose 구조와 맞지 않는다. 도구의 장점만큼 적합한 사용 범위를 명확히 아는 것이 중요하다.
