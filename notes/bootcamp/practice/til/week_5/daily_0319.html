    • 대용량 데이터에서 Pageable이 느려지는 이유와 No-Offset(커서 페이징) 방식을 파악한다.

    1. 페이지네이션 전략 — Page vs Slice vs No-Offset

    Page vs Slice

    구분 Page<T> Slice<T>
    제공 정보 전체 데이터 건수, 전체 페이지 수, 현재 페이지 데이터 현재 페이지 데이터, 다음 데이터 존재 여부(hasNext())만
    내부 쿼리 데이터 조회 쿼리 + COUNT(*) 쿼리 총 2번 데이터 조회 쿼리 1번 (LIMIT + 1)
    사용 상황 "총 1,234건, 3/24페이지"처럼 전체 페이지 수가 필요한 관리자 화면 무한 스크롤처럼 다음 데이터 존재 여부만 알면 충분한 경우
    성능 데이터가 많을수록 COUNT(*) 비용 증가 COUNT(*) 없으므로 상대적으로 빠름
    Slice가 LIMIT + 1을 사용하는 이유 10개를 보여줄 페이지를 요청받으면 실제로는 11개를 조회한다. 11개가 조회됐다면 다음 데이터가 존재하는 것이고(hasNext = true), 10개 이하라면 마지막 페이지인 것이다(hasNext = false). 클라이언트에 전달할 때는 앞 10개만 잘라서 반환한다. COUNT(*) 쿼리를 실행하지 않으면서도 다음 데이터 존재 여부를 판단하는 영리한 방법이다.

    No-Offset 방식 (커서 페이징)

    일반적인 Pageable은 OFFSET을 사용해 데이터를 건너뛴다. 10페이지를 보려면 앞 90건을 읽고 버리는 방식이므로, 데이터가 수백만 건이 넘으면 성능이 급격히 떨어진다. No-Offset(커서 페이징)은 마지막으로 읽은 데이터의 ID(커서)를 기준으로 WHERE id < :lastId처럼 그 이후 데이터만 바로 조회한다. 앞 데이터를 전혀 읽지 않으므로 데이터가 아무리 많아도 성능이 일정하게 유지된다.

    구분 Offset 방식 (Pageable) No-Offset 방식 (커서 페이징)
    쿼리 방식 LIMIT 10 OFFSET 90 — 앞 90건을 읽고 버림 WHERE id < :lastId LIMIT 10 — 커서 이후만 조회
    데이터 증가 시 성능 페이지 뒤로 갈수록 느려짐 항상 일정
    적합한 UI 페이지 번호를 직접 클릭하는 UI 무한 스크롤, "더 보기" 버튼
    단점 대용량에서 성능 급락 특정 페이지로 직접 이동 불가

    2. RestClient vs WebClient

    구분 RestClient WebClient
    방식 동기(Synchronous) 비동기(Asynchronous)
    특징 요청 → 응답을 받아야만 → 다음 로직 진행. RestTemplate의 직관적인 동기 방식을 유지하면서 WebClient의 세련된 메서드 체이닝 문법을 가져왔다. 응답을 기다리지 않고 다른 작업 가능. Mono, Flux 등 리액티브 프로그래밍 개념이 필요하다.
    가독성 직관적이고 뛰어남 리액티브 개념 없이는 디버깅이 어려움
    사용 시점 일반적인 외부 API 호출, 순차 처리가 필요한 경우 대량 I/O가 동시에 발생하는 경우, Spring WebFlux 환경
    선택 기준 일반적인 외부 API 연동(날씨, 결제, AI API 등)은 RestClient로 충분하다. WebFlux 환경이 아니라면 복잡도만 높아지는 WebClient를 굳이 사용할 필요가 없다.

    3. RestClient 설정과 외부 API 호출

    RestClientConfig — Bean 등록

    RestClient를 Bean으로 등록하면 타임아웃, 기본 헤더, 베이스 URL 등 공통 설정을 한 곳에서 관리할 수 있다. 여러 외부 API를 사용한다면 Bean 이름을 명시적으로 지정해 구분한다. 

    @Configuration
public class RestClientConfig {

    // 수정 포인트: application.yml에서 외부 API base-url과 api-key를 관리한다.
    @Value("${weather.base-url}")
    private String weatherBaseUrl;

    @Value("${weather.api-key}")
    private String weatherApiKey;

    // 수정 포인트: 여러 외부 API 클라이언트가 있다면 Bean 이름을 명시해 구분한다.
    @Bean("weatherRestClient")
    public RestClient weatherRestClient() {
        SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
        factory.setConnectTimeout(5000);  // 5초 안에 서버에 연결 안 되면 포기
        factory.setReadTimeout(10000);    // 연결 후 10초 안에 응답 안 오면 포기

        return RestClient.builder()
                .requestFactory(factory)
                .baseUrl(weatherBaseUrl)                         // Postman의 base URL 설정과 동일
                .defaultHeader("Authorization", "Bearer " + weatherApiKey) // Postman의 Headers 탭과 동일
                .defaultHeader("Content-Type", "application/json")
                .build();
    }
}

    Service — 외부 API 호출 (Postman과 대응) 

    @Service
@RequiredArgsConstructor
public class WeatherService {

    // Bean 이름으로 주입 — RestClientConfig에서 정의한 이름과 변수명을 맞춘다.
    private final RestClient weatherRestClient;

    public WeatherDto getCurrentWeather(String city) {

        // Postman의 [POST] 버튼 = .post()
        // Postman의 URL 입력창 = .uri() → baseUrl 뒤에 붙는 경로
        // Postman의 Body 탭 → raw JSON = .body(requestDto)
        // Postman의 [Send] 버튼 = .retrieve()
        // Postman의 응답 Body 창 = .body(WeatherDto.class) → JSON을 DTO로 자동 역직렬화

        WeatherRequestDto request = new WeatherRequestDto(city);

        return weatherRestClient.post()           // [POST] 메서드 선택
                .uri("/v1/current")               // 엔드포인트 경로
                .body(request)                    // 요청 Body (JSON 직렬화됨)
                .retrieve()                       // HTTP 요청 실행 (Send)
                .onStatus(HttpStatusCode::is4xxClientError, (req, res) -> {
                    // Postman에서 400~499 응답이 왔을 때의 처리
                    throw new BusinessException(ErrorCode.BAD_REQUEST);
                })
                .onStatus(HttpStatusCode::is5xxServerError, (req, res) -> {
                    // Postman에서 500~599 응답이 왔을 때의 처리
                    throw new BusinessException(ErrorCode.EXTERNAL_API_ERROR);
                })
                .body(WeatherDto.class);           // 응답 Body → DTO로 자동 역직렬화
    }
}
    Postman 작업과 RestClient 코드 대응 Postman에서 하던 작업이 RestClient 코드에서는 다음과 같이 대응된다.

    HTTP 메서드 선택(GET, POST 버튼) → .get() / .post()
    URL 입력창 → .baseUrl() + .uri()
    Headers 탭 → .defaultHeader() 또는 .header()
    Body 탭 (raw JSON) → .body(requestDto) — DTO가 JSON으로 자동 직렬화됨
    Send 버튼 → .retrieve()
    응답 Body 창 → .body(ResponseDto.class) — JSON이 DTO로 자동 역직렬화됨

    4. 외부 API DTO 설계 — Map을 쓰면 안 되는 이유

    ⚠️ Map<String, Object>로 받으면 안 되는 이유 Map이나 JSONObject로 데이터를 욱여넣고 get("contents") 형태로 꺼내 쓰는 방식은 유지보수를 포기하겠다는 것과 같다. 오타 하나만 나도 컴파일 타임에 잡히지 않고, 서버가 런타임에 NullPointerException으로 터진다. 반드시 class나 record를 활용해 계층형 DTO를 만들어야 한다. 모든 데이터를 받을 필요 없이, 필요한 필드만 선언하면 된다.

    요청 DTO 조립

    외부 API에 보낼 JSON 구조를 그대로 record나 class로 만든다. JSON의 중괄호({})는 record/class, 배열([])은 List로 대응하면 된다. 

    // Postman Body에서 보내는 JSON 구조
{
  "contents": [
    {
      "parts": [
        { "text": "AI works" }
      ]
    }
  ]
}
    // 위 JSON 구조를 그대로 record로 매핑한다.
// {} 중괄호 → record/class / [] 배열 → List
public record GeminiRequestDto(
        List<Content> contents     // "contents": [ ... ]
) {
    public record Content(
            List<Part> parts       // "parts": [ ... ]
    ) {
        public record Part(
                String text        // "text": "AI works"
        ) {}
    }
}

    응답 DTO 파싱 — @JsonIgnoreProperties

    외부 API 응답 JSON은 대부분 아주 길고 복잡하다. 전체를 DTO로 만들 필요 없이, @JsonIgnoreProperties(ignoreUnknown = true)를 붙이면 내가 선언한 필드 외의 데이터는 무시하고 필요한 것만 꺼낼 수 있다. 

    // Gemini API 응답 예시 (실제로는 훨씬 길지만, 필요한 필드만 선언)
// @JsonIgnoreProperties(ignoreUnknown = true) 덕분에 나머지 필드는 무시된다.
@JsonIgnoreProperties(ignoreUnknown = true)
public record GeminiResponse(
        List<Candidate> candidates     // 응답의 candidates 배열만 매핑
) {
    @JsonIgnoreProperties(ignoreUnknown = true)
    public record Candidate(
            Content content
    ) {
        @JsonIgnoreProperties(ignoreUnknown = true)
        public record Content(
                String role,
                List<Part> parts
        ) {
            @JsonIgnoreProperties(ignoreUnknown = true)
            public record Part(
                    String text        // 최종적으로 필요한 텍스트만
            ) {}
        }
    }
}
    외부 API DTO 설계 핵심 원칙 외부 API 문서를 볼 때는 Endpoint, Request 필수값, Response 구조 세 가지만 빠르게 확인한다.
    응답 JSON이 아무리 길고 복잡해도 @JsonIgnoreProperties(ignoreUnknown = true)로 필요한 필드만 골라서 받는다.
    Map은 절대 사용하지 않는다. 오타 하나에 런타임 장애가 발생한다.

    마무리

    오늘 가장 중요한 인사이트는 상황에 맞는 도구를 선택하는 것이다. 페이지네이션도 전체 건수가 필요한 곳은 Page, 무한 스크롤처럼 다음 데이터 존재 여부만 알면 되는 곳은 Slice가 적합하다. 대용량 환경에서는 OFFSET이 아닌 WHERE id < :lastId 방식의 커서 페이징으로 전환해야 성능을 지킬 수 있다.

    RestClient는 Postman으로 API를 테스트하는 것을 코드로 옮겨놓은 것과 같다. HTTP 메서드 선택 → URL → 헤더 → 바디 → Send → 응답 파싱의 흐름이 코드에 그대로 담겨 있어 직관적이다. 일반적인 동기 API 호출이라면 WebClient의 복잡도를 감수할 이유가 없다.

    외부 API 응답을 Map으로 받는 습관은 반드시 버려야 한다. 컴파일 타임에 오류를 잡을 수 없고 오타 하나에 런타임 장애가 발생한다. @JsonIgnoreProperties(ignoreUnknown = true)와 계층형 record를 조합하면 복잡한 외부 API 응답도 필요한 데이터만 안전하게 꺼낼 수 있다. 외부 API 문서에서 Endpoint, 필수 Request 파라미터, Response 구조 세 가지만 파악하면 DTO를 설계하는 데 충분하다.
