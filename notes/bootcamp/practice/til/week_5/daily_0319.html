    64일차 개요

    오늘 목표

    • CD(Continuous Deployment)의 개념과 전체 파이프라인 흐름을 이해한다.
    • ECR → Launch Template → Instance Refresh → ALB 헬스체크로 이어지는 무중단 배포 과정을 정리한다.
    • CDN의 개념과 CloudFront가 S3보다 빠른 이유를 이해한다.
    • AWS 리소스 삭제 시 과금 순서와 의존 관계를 파악하고 안전하게 정리하는 방법을 익힌다.

    1. CD (Continuous Deployment)

    지속적 배포(CD)는 통합된 코드를 지속적으로 릴리스하여 운영 환경에 자동으로 배포하는 것이다. CI에서 빌드·테스트가 통과되면 CD가 이어받아 새 버전을 실제 서버에 자동으로 반영한다.

    CD 파이프라인 전체 흐름 ECR에 새 Docker Image가 push되면, Launch Template을 업데이트하고 Instance Refresh로 배포한 뒤 ALB 헬스체크로 정상 동작을 검증하는 흐름으로 진행된다.

    CD 파이프라인 4단계

    단계 작업 설명
    1 Launch Template 새 버전 생성 User Data의 IMAGE_TAG를 새 커밋 SHA로 업데이트한다. 기존 버전은 그대로 유지된다.
    2 Instance Refresh 시작 ASG에 새 Launch Template 버전으로 EC2 교체를 요청한다.
    3 자동 Rolling Update 새 EC2 생성 → ALB 헬스체크 통과 → 트래픽 전환 → 기존 EC2 종료 순서로 순차 교체된다. 서비스 중단이 없다.
    4 완료 확인 Instance Refresh가 끝날 때까지 상태를 반복 조회한다. 배포 완료가 확인되면 파이프라인이 종료된다.
    커밋 SHA를 IMAGE_TAG로 사용하는 이유 커밋마다 고유한 SHA 값을 이미지 태그로 사용하면 어떤 코드가 현재 운영 중인지 즉시 추적할 수 있다. 장애 발생 시 특정 커밋 시점의 이미지로 롤백하는 것도 가능해진다. latest만 사용하면 어떤 버전이 배포됐는지 알 수 없어 운영 환경 추적이 불가능하다.

    2. CDN과 CloudFront

    CDN (Content Delivery Network)

    전 세계에 분산된 서버에 콘텐츠를 미리 복사해두고, 사용자와 가장 가까운 서버에서 응답하는 기술이다. 이미지처럼 누가 요청해도 내용이 동일한 정적 파일은 원본 서버까지 요청이 갈 필요가 없다. 사용자 근처의 캐시 서버가 대신 응답하므로 응답 속도가 크게 빨라지고 원본 서버의 부하도 줄어든다.

    AWS CloudFront

    AWS가 제공하는 CDN 서비스다. 전 세계 750개 이상 지역에 캐시 서버(엣지 로케이션)를 두고 있다. S3에 파일이 저장되어 있더라도 CloudFront를 앞에 두면 동일 지역에서도 S3 직접 접근보다 훨씬 빠르고 비용 효율적이다.

    구분 S3 직접 접근 CloudFront 경유
    응답 속도 원본 서버(특정 리전)까지 왕복 가장 가까운 엣지 로케이션에서 응답
    비용 S3 데이터 전송 비용 발생 CloudFront 데이터 전송 단가가 더 저렴
    캐싱 없음 — 매번 원본에서 가져옴 엣지에 캐시 — 동일 콘텐츠는 재사용
    HTTPS 별도 설정 필요 ACM 인증서 연동으로 간단 적용

    3. AWS 리소스 삭제 순서

    AWS 리소스는 의존 관계가 있어 순서를 지키지 않으면 삭제가 실패한다. 과금이 발생하는 리소스를 먼저 삭제하고, 다른 리소스에 의존하지 않는 리소스를 나중에 삭제한다.

    ⚠️ 삭제 전 반드시 확인 NAT Gateway가 남아있으면 VPC 삭제가 실패한다. 2단계에서 NAT Gateway 상태가 Deleted로 바뀐 것을 확인한 후 진행해야 한다. RDS 삭제 시 "Create final snapshot" 체크를 해제해야 스냅샷 비용이 추가로 발생하지 않는다.

    단계별 삭제 순서

    단계 삭제 대상 경로 및 주의사항
    1 ASG · ALB ASG 삭제 시 EC2가 자동 종료된다. EC2가 terminated 상태인지 확인 후 ALB를 삭제한다.
    2 NAT Gateway 삭제 완료까지 수 분 소요된다. 상태가 Deleted로 바뀔 때까지 기다려야 한다. 연결된 Elastic IP는 자동 해제된다.
    3 RDS "Create final snapshot" 체크 해제 → "I acknowledge" 체크 후 삭제한다. 삭제 완료까지 수 분 소요된다.
    4 Target Group · Launch Template · Security Group (3개) ALB SG, EC2 SG, RDS 자동 생성 SG를 각각 삭제한다. Security Group은 사용 중인 리소스가 없어야 삭제 가능하다.
    5 Route 53 레코드 · ACM 인증서 · Hosted Zone · 도메인 자동갱신 비활성화 레코드를 모두 삭제한 후 Hosted Zone을 삭제한다. 도메인 자동갱신도 반드시 비활성화한다.
    6 S3 버킷 · ECR 리포지토리 S3는 버킷 안 파일을 먼저 비운(Empty) 후 삭제한다. 파일이 남아있으면 버킷 삭제가 불가능하다.
    7 IAM Role · User · Policy · OIDC Provider · Parameter Store OIDC Provider는 IAM → Identity providers → token.actions.githubusercontent.com에서 삭제한다.
    8 VPC VPC 삭제 시 서브넷, 라우트 테이블, 인터넷 게이트웨이가 함께 삭제된다. NAT Gateway가 남아있으면 실패한다.
    삭제 원칙 — 과금 리소스 우선 시간당 과금이 발생하는 리소스(ASG·ALB·NAT Gateway·RDS)를 가장 먼저 삭제해야 불필요한 비용이 발생하지 않는다. 의존 관계(NAT Gateway → VPC, 파일 → S3 버킷)를 지키지 않으면 삭제가 실패하므로 반드시 순서를 따른다.

    마무리

    CD 파이프라인의 핵심은 사람이 개입하지 않아도 새 코드가 안전하게 운영 서버에 반영된다는 것이다. 커밋 SHA를 이미지 태그로 사용하고, Launch Template을 버전 관리하며, Instance Refresh로 무중단 롤링 업데이트를 진행하는 구조는 배포 이력을 추적하면서도 서비스 중단 없이 새 버전을 배포하는 가장 안정적인 방식이다. ALB 헬스체크가 새 인스턴스의 정상 여부를 검증한 후에만 트래픽이 전환되므로 배포 중 장애 가능성이 크게 줄어든다.

    CloudFront는 단순히 빠른 CDN이 아니라 S3의 퍼블릭 접근을 막으면서 CloudFront를 통해서만 파일을 제공하는 구조로 보안도 함께 강화할 수 있다. 정적 파일(이미지, JS, CSS)을 CloudFront로 제공하면 원본 서버 부하를 줄이고 전 세계 사용자에게 빠른 응답을 줄 수 있다. S3 + CloudFront 조합은 Presigned URL 방식과 함께 파일 제공의 표준 패턴으로 자리잡고 있다.

    AWS 리소스 삭제 순서는 단순히 절차의 문제가 아니라 불필요한 비용 방지와 의존 관계 오류 예방을 위한 것이다. 시간당 과금이 발생하는 ASG·ALB·NAT Gateway·RDS를 먼저 삭제하고, 의존 관계를 역순으로 해제하며 마지막에 VPC를 삭제하는 흐름을 기억해두면 실습 후 클린업 과정에서 실수를 방지할 수 있다. 도메인 자동갱신 비활성화처럼 놓치기 쉬운 항목도 반드시 체크해야 의도치 않은 과금이 생기지 않는다.
