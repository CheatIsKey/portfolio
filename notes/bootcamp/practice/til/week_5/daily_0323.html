    1. MDC (Mapped Diagnostic Context)

    수많은 요청이 동시에 들어오는 서버 환경에서, 특정 사용자의 요청이 어떤 로그들을 남겼는지 한눈에 추적하기 위해 사용한다. 각 스레드마다 별도의 저장 공간을 만들어 traceId, userId 같은 정보를 저장해두면, 그 스레드가 실행되는 동안 찍히는 모든 로그에 해당 정보가 자동으로 따라붙는다.

    단계 동작
    1. 정보 저장 요청이 들어올 때 MDC.put("traceId", "TX1234")로 저장한다.
    2. 로그 출력 logback 설정만 해두면 이후 모든 로그에 [TX1234]가 자동으로 붙는다.
    3. 정보 삭제 요청이 끝나면 반드시 MDC.clear() 또는 MDC.remove("traceId")로 비워야 한다.
    ⚠️ 스레드 풀 환경에서 MDC.remove()가 필수인 이유 톰캣(Tomcat)은 성능을 위해 스레드를 매번 새로 만들지 않고 스레드 풀(Thread Pool)에서 꺼내 재사용한다. 요청이 끝난 후 MDC.remove()를 호출하지 않으면, 같은 스레드를 재사용하는 다른 사용자의 요청에 이전 사용자의 traceId가 그대로 찍힌다. 반드시 finally 블록에서 제거해야 예외가 발생해도 항상 정리가 보장된다.

    AOP + MDC 통합 로깅

    AOP의 @Around에 MDC를 결합하면 컨트롤러 코드를 건드리지 않고도 모든 요청에 고유한 추적 ID를 자동으로 붙일 수 있다. MultipartFile, HttpServletRequest처럼 로깅에 부적합한 객체는 isLoggable()로 필터링해서 제외한다. 

    @Around("execution(* com.example.instagramclone.controller.rest.*.*(..))")
public Object logApi(ProceedingJoinPoint joinPoint) throws Throwable {

    // UUID 앞 8자리를 traceId로 사용 — 요청마다 고유한 값
    String traceId = UUID.randomUUID().toString().substring(0, 8);
    MDC.put("traceId", traceId);

    try {
        String className  = joinPoint.getTarget().getClass().getSimpleName();
        String methodName = joinPoint.getSignature().getName();

        MethodSignature signature      = (MethodSignature) joinPoint.getSignature();
        String[]        parameterNames = signature.getParameterNames();
        Object[]        args           = joinPoint.getArgs();

        // MultipartFile, HttpServletRequest 등 로깅 부적합 객체 필터링
        List<String> filteredParamNames = new ArrayList<>();
        List<Object> filteredArgs       = new ArrayList<>();
        for (int i = 0; i < args.length; i++) {
            if (!isLoggable(args[i])) continue;
            filteredParamNames.add(parameterNames != null ? parameterNames[i] : "arg" + i);
            filteredArgs.add(args[i]);
        }

        String paramsString = LogMaskingUtils.buildMaskedParamsString(
                filteredParamNames.toArray(new String[0]),
                filteredArgs.toArray()
        );

        log.info("[API 요청] {} / {} | 파라미터: [{}]", className, methodName, paramsString);

        long startTime = System.currentTimeMillis();
        Object result = joinPoint.proceed();
        long executionTime = System.currentTimeMillis() - startTime;

        log.info("[API 응답] {} / {} (수행시간: {}ms)", className, methodName, executionTime);
        return result;

    } finally {
        // 스레드 풀 재사용 시 이전 요청 데이터 오염 방지 — finally 블록 필수
        MDC.remove("traceId");
    }
}

    2. 동적 마스킹 — @Masking(type = MaskingType.XXX)

    이전에 구현한 @Masking은 모든 필드를 ******로 치환하는 단순한 방식이었다. 동적 마스킹은 @Masking 어노테이션에 타입(type) 속성을 추가하여 이름, 이메일, 비밀번호 각각에 맞는 마스킹 전략을 다르게 적용한다.

    MaskingType enum + @Masking 어노테이션 

    // 마스킹 전략 종류
public enum MaskingType {
    NAME,     // 홍길동 → 홍*동
    EMAIL,    // hong@example.com → hon***@example.com
    PASSWORD  // 무조건 ******
}

// 수정 포인트: 새 마스킹 전략이 필요하면 MaskingType에 추가하고 maskValueByType()에 case를 추가한다.
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Masking {
    MaskingType type() default MaskingType.PASSWORD; // 기본값은 PASSWORD
}

    타입별 마스킹 로직 

    // applyMaskingToObject() 내부 — 기존 field.isAnnotationPresent(Masking.class) 조건 대체
Masking masking = field.getAnnotation(Masking.class); // 어노테이션 객체를 직접 가져온다

if (masking != null && value != null) {
    // type 속성을 읽어 동적으로 마스킹 전략 적용
    String maskedValue = maskValueByType(value.toString(), masking.type());
    fieldStrings.add(field.getName() + "='" + maskedValue + "'");
} else {
    fieldStrings.add(field.getName() + "=" + value);
}

// 타입별 마스킹 비즈니스 로직
private static String maskValueByType(String value, MaskingType type) {
    return switch (type) {
        case PASSWORD -> "******";
        case EMAIL -> {
            int atIndex = value.indexOf('@');
            // 이메일 형식이 아니거나 @ 앞자리가 너무 짧으면 통째로 마스킹
            if (atIndex <= 3) yield "***" + value.substring(atIndex);
            // 앞 3글자 + *** + @도메인
            yield value.substring(0, 3) + "***" + value.substring(atIndex);
        }
        case NAME -> {
            if (value.length() <= 1) yield "*";
            if (value.length() == 2)  yield value.charAt(0) + "*";
            // 홍길동 → 홍*동 / 남궁민수 → 남**수
            yield value.charAt(0) + "*".repeat(value.length() - 2) + value.charAt(value.length() - 1);
        }
    };
}
    타입 원본 예시 마스킹 결과
    PASSWORD myP@ssw0rd ******
    EMAIL hong@example.com hon***@example.com
    NAME (3글자) 홍길동 홍*동
    NAME (4글자) 남궁민수 남**수

    DTO에 적용 예시 

    @Builder
public record SignUpRequest(
        @NotBlank(message = "사용자 이름은 필수입니다.")
        @Pattern(regexp = "^[a-z0-9._]{4,20}$", message = "...")
        @Masking(type = MaskingType.NAME)   // 로그에서 이름 형식으로 마스킹
        String username,

        @NotBlank(message = "비밀번호는 필수입니다.")
        @Pattern(regexp = "^(?=.*[A-Za-z])(?=.*\\d)(?=.*[@$!%*#?&]).{8,}$", message = "...")
        @Masking                            // type 생략 시 기본값 PASSWORD → ******
        String password,

        @NotBlank(message = "이메일 또는 전화번호는 필수입니다.")
        @Masking(type = MaskingType.EMAIL)  // 로그에서 이메일 형식으로 마스킹
        String emailOrPhone,

        @NotBlank(message = "이름은 필수입니다.")
        @Masking(type = MaskingType.NAME)
        String name
) {
}

    switch 표현식과 yield (Java 13+)

    yield vs return의 차이 return을 만나면 메서드 전체가 즉시 종료된다.
    yield를 만나면 switch 블록의 결과값만 반환하고, 메서드는 계속 실행된다.

    즉, switch 표현식 전체를 하나의 값으로 평가할 때 중괄호 블록 안에서 값을 돌려보내는 키워드가 yield다. 
    // 한 줄이면 yield 생략 가능 (화살표 오른쪽 값이 자동 반환)
String result = switch (mode) {
    case "FAST"   -> "빠름";
    case "DETAIL" -> {
        System.out.println("상세 모드 가동");
        yield "상세"; // 중괄호 안에서는 yield 필수
    }
    default -> "기본";
};

    3. JWT 단점 방어 전략 3단계

    JWT는 한 번 발급되면 서버에서 무효화하기 어렵다는 근본적인 한계가 있다. 탈취된 토큰도 만료되기 전까지는 유효하기 때문에, 아래 세 단계를 조합해 방어한다.

    단계 방어 방법 설명
    1단계 Access Token 수명 단축 Access Token의 유효기간을 짧게(5~15분) 설정한다. 탈취되더라도 피해 지속 시간을 최소화한다.
    2단계 Redis 블랙리스트 로그아웃 시 해당 Access Token을 Redis에 블랙리스트로 등록한다. Spring Security 필터가 서명 검증과 함께 Redis 블랙리스트 여부도 확인한다. TTL을 토큰의 남은 유효기간으로 설정해 Redis 메모리를 자동으로 정리한다.
    3단계 RTR (Refresh Token Rotation) Access Token 재발급 요청 시 Refresh Token도 무조건 새것으로 교체하고 기존 토큰은 폐기한다. Refresh Token이 탈취됐더라도 한 번 사용되는 순간 무효화되어 재사용이 불가능하다.
    Redis 블랙리스트가 효율적인 이유 Redis는 메모리 기반 인메모리 DB이므로 블랙리스트 조회가 매우 빠르다. 토큰의 남은 유효기간을 TTL로 설정하면, 토큰이 원래 만료될 시점에 Redis에서도 자동으로 삭제된다. 만료된 토큰은 애초에 더 이상 쓸 수 없으므로 블랙리스트에 남겨둘 이유가 없다. 메모리 낭비 없이 정확하게 관리된다.

    4. JWT 규격 — JWS vs JWE

    규격 방식 보장하는 것 내용 노출 여부
    JWS
    (JSON Web Signature)    		 서명(Signature)으로 무결성 검증 변조되지 않았음(무결성) 페이로드 내용이 Base64로 인코딩만 됨 — 누구나 디코딩해서 읽을 수 있음
    JWE
    (JSON Web Encryption)    		 강력한 암호화 알고리즘으로 페이로드 암호화 무결성 + 기밀성 비밀키가 없으면 내용을 절대 열어볼 수 없음
    실무에서 JWS를 사용하는 이유 JWE는 요청마다 무거운 암호화·복호화 연산을 수행해야 한다. 수많은 사용자가 짧은 시간에 수십 개의 API 요청을 보내는 환경에서 매번 JWE를 복호화하면 심각한 성능 저하가 발생한다.

    대부분의 JWT에는 userId, 권한, 만료 시각처럼 공개돼도 크게 문제없는 정보만 담는다. 이 경우 기밀성보다 변조 불가능함(무결성)이 더 중요하므로, 성능이 뛰어난 JWS 방식을 사용하는 것이다. JWT 페이로드에 비밀번호, 개인정보 같은 민감 데이터를 담지 않는 것이 전제 조건이다.

    마무리

    MDC는 멀티스레드 환경에서 로그 추적을 가능하게 하는 핵심 도구다. 각 요청에 고유한 traceId를 부여하면 수백 개의 동시 요청 속에서도 특정 요청의 로그 흐름을 한눈에 따라갈 수 있다. 단, 스레드 풀 환경에서 finally 블록의 MDC.remove()를 빠뜨리는 것은 데이터 오염으로 이어지므로 절대 놓치면 안 된다.

    동적 마스킹은 @Masking에 type 속성 하나를 추가한 것처럼 보이지만, 필드에 선언하는 것만으로 마스킹 전략이 자동으로 적용되어 코드 가독성과 보안이 동시에 향상된다. MaskingType에 새 전략을 추가하고 maskValueByType()에 case만 추가하면 기존 코드를 건드리지 않고 확장할 수 있다는 점에서 개방-폐쇄 원칙과도 잘 맞는 설계다.

    JWT 보안은 단일 방어로는 부족하다. 수명 단축으로 피해 범위를 제한하고, Redis 블랙리스트로 즉각적인 무효화를 지원하며, RTR로 Refresh Token 탈취 공격까지 차단하는 다층 방어가 필요하다. JWS와 JWE의 차이도 단순한 이론이 아니라 성능과 보안의 트레이드오프를 이해하는 것이다. JWT 페이로드에 민감 데이터를 담지 않는다는 원칙을 지키는 한, 성능이 뛰어난 JWS로 충분하다.
