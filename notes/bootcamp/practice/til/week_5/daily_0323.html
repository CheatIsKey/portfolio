    1. PointTransaction 엔티티 설계

    정적 팩토리 메서드 패턴 적용

    포인트 거래는 적립(earned), 사용(spend), 복구(restored), 취소(earnCanceled), 만료(expired) 등 타입마다 생성 시 필요한 파라미터가 다르다. @Builder만 사용하면 어떤 타입을 만들 때 어떤 파라미터가 필요한지 코드만 보고 파악하기 어렵다. 정적 팩토리 메서드 패턴을 적용하면 의도가 이름에 담기고 생성 규칙이 한 곳에 집중된다.

    @Builder와 정적 팩토리 메서드를 동시에 고려했다가 팩토리 메서드로 결론낸 이유 @Builder는 어떤 필드 조합으로도 객체를 만들 수 있어 유연하지만, 타입별로 반드시 채워야 하는 필드가 다른 경우에는 빌더 패턴으로 강제할 수 없다. 잘못된 조합으로 객체가 만들어지는 것을 컴파일 타임에 막을 수 없다는 뜻이다.

    정적 팩토리 메서드는 각 메서드 시그니처 자체가 어떤 데이터가 필요한지 명확히 드러내고, 내부에서만 빌더를 사용해 객체를 만들기 때문에 생성 책임이 엔티티 클래스 안에 캡슐화된다.

    트러블슈팅 정리

    문제 원인 및 해결
    클래스명을 Point로 작성 테이블명(point_transactions)과 불일치. PointTransaction으로 수정하여 테이블 매핑 명확화.
    필드명에 snake_case 혼용
    (expires_at, expiration_processed_at)    		 Java 컨벤션은 camelCase다. Hibernate가 자동으로 snake_case 컬럼명으로 변환하므로, 자바 필드명은 camelCase로 통일해야 한다.
    @NoArgsConstructor(PROTECTED)의 필요성 오해 JPA는 프록시 객체 생성과 지연 로딩을 위해 기본 생성자가 필요하다. PROTECTED로 제한하면 JPA 내부에서는 사용 가능하지만 외부에서 직접 new로 생성하는 것은 막을 수 있다. 외부 생성은 팩토리 메서드로만 허용하는 구조다.
    balanceSnapshot 필드 추가 고려 User.pointBalance가 이미 현재 잔액을 관리하고 있어 스냅샷 역할을 겸한다. 중복 데이터이므로 제거했다.
    @ManyToOne에서 Payment, Refund 직접 참조 설계 스펙 기준으로 환불도 동일한 주문 ID로 추적 가능하다. Order를 기준으로 참조하는 방식으로 변경하여 설계를 단순화했다.

    2. AOP 로깅 — 보안 문제 발견 및 해결

    문제 상황

    컨트롤러 계층 전체에 AOP 로깅을 적용했더니, 로그인·회원가입 요청의 파라미터(이메일, 비밀번호)가 로그에 그대로 노출되는 심각한 보안 문제가 발생했다.

    ⚠️ 파라미터 로깅은 항상 보안 문제를 먼저 생각해야 한다 AOP로 파라미터를 로깅할 때 인증 관련 API가 포함되어 있다면 비밀번호, 토큰, 이메일 같은 민감 정보가 로그 파일에 평문으로 기록된다. 로그 파일이 유출되거나 개발자가 실수로 외부에 노출하는 경우 치명적인 보안 사고로 이어진다.

    해결 방법 2가지

    방법 설명 채택 여부
    auth 패키지 Pointcut 제외 execution() 조건에 && !execution(* ...domain.auth..*(..))를 추가해 인증 관련 패키지를 통째로 제외한다. 1차 해결
    파라미터 로깅 자체 제거 파라미터 이름은 찍히지만 값은 안 찍히는 버그를 수정하는 과정에서 (parameterNames 대신 args를 사용해야 실제 값이 출력됨) 민감정보 노출 위험이 있으므로 파라미터 로깅 자체를 제거했다. 최종 결정 
    // 개선 전: auth 패키지까지 포함되어 비밀번호 노출
@Around("execution(* com.example.payment.domain..*Controller.*(..))")

// 개선 후: auth 패키지 제외 조건 추가
@Around("execution(* com.example.payment.domain..*Controller.*(..)) " +
        "&& !execution(* com.example.payment.domain.auth..*(..))")

    3. PointService 비즈니스 로직 구현 트러블슈팅

    구현한 메서드 목록

    메서드 동작
    earnPoints() 구매 확정 시 포인트 적립 — 등급별 적립률 계산 포함
    spendPoints() 결제 확정 시 포인트 사용 — 잔액 부족 시 예외
    restorePoints() 환불 시 사용했던 포인트 복구
    cancelEarnedPoints() 환불 시 적립 포인트 취소 + 누적 결제금액 감소 + 등급 재계산
    getPointHistory() 포인트 거래 내역 조회

    트러블슈팅 정리

    문제 원인 및 해결
    MembershipGradeService 의존 → 컴파일 에러 존재하지 않는 클래스를 참조했다. MembershipGradeRepository를 직접 주입하는 방식으로 우선 해결하고, 이후 팀 합의로 userService.getUserMembership()에 위임하는 구조로 변경했다.
    grade 변수 미정의 컴파일 에러 포인트 적립률을 계산하려면 현재 등급을 먼저 조회해야 한다. membershipGradeRepository에서 현재 등급을 조회한 후 earnRate를 계산하는 로직을 추가해서 완성했다.
    log.info에서 user.get 오타 user.getMembershipGrade()로 수정. IDE 자동완성을 맹신하지 말고 실제 메서드명을 확인해야 한다는 점을 다시 한번 상기했다.
    restorePoints()에서 orElseThrow(() -> new ) 미완성 포인트를 사용하지 않은 채로 환불되는 경우도 있으므로 사용 이력이 없는 상황이 정상 케이스다. orElseThrow() 대신 orElse(null) + null 체크 방식으로 변경했다.
    cancelPointsByRefund() 미완성 코드 메서드명이 하는 일을 명확하게 표현하지 못했다. cancelEarnedPoints()로 이름을 변경하고 로직을 완성했다.
    PointService → UserService 의존 시 순환참조 우려 UserService가 PointService를 의존하지 않는 단방향 구조임을 확인했다. 순환참조는 A→B, B→A처럼 양방향 의존이 생길 때 발생한다. 단방향이라면 문제없다.
    포인트 잔액 조회를 어느 서비스에 둘지 논의 pointBalance는 개념상 포인트 도메인의 책임이지만 User의 단순 필드 반환이기 때문에 팀 합의로 UserService로 이동했다. 도메인 설계에서 책임 경계는 개념적 원칙과 팀 컨벤션 사이의 균형으로 결정된다.
    순환참조가 발생하는 조건 순환참조는 두 클래스가 서로를 의존할 때 발생한다. A → B → A 형태로 의존 고리가 생기면 스프링이 Bean을 초기화할 수 없어 애플리케이션이 시작조차 안 된다.

    이번 케이스는 PointService → UserService이고, UserService는 PointService를 의존하지 않는다. 단방향 의존이므로 고리가 생기지 않아 문제없다. 의존 방향을 항상 단방향으로 유지하고, 양방향이 필요한 경우 이벤트나 인터페이스 분리를 통해 의존 방향을 끊어야 한다.

    마무리

    오늘 팀 프로젝트에서 포인트 도메인을 처음부터 직접 설계하면서 가장 크게 느낀 것은 이름 하나가 의도 전체를 담아야 한다는 것이다. 클래스명 Point가 PointTransaction이 되어야 하고, 메서드명 cancelPointsByRefund()가 cancelEarnedPoints()가 되어야 하는 이유가 모두 같다. 코드는 사람이 읽는 문서이기 때문에 이름이 틀리면 이해를 방해한다.

    AOP 로깅에서 보안 문제가 발생한 것은 예상치 못한 수확이었다. 파라미터를 로깅하는 것이 디버깅에 편리하다는 것만 생각하고, 인증 API가 같은 Pointcut에 포함된다는 사실을 놓쳤다. 기능을 구현할 때는 기능 자체의 정확성뿐만 아니라 그 기능이 의도치 않게 건드리는 범위도 함께 점검해야 한다. 특히 AOP처럼 범위를 넓게 잡는 기술은 더욱 그렇다.

    포인트 잔액 조회 메서드를 어느 서비스에 둘지 팀과 논의하는 과정에서, 도메인 설계의 정답은 하나가 아니라는 것을 배웠다. 개념적으로 옳은 위치와 팀이 실제로 유지보수하기 좋은 위치가 다를 수 있고, 그럴 때는 팀 합의가 원칙보다 우선될 수 있다. 중요한 것은 합의가 명확하게 문서화되고, 같은 고민이 반복되지 않도록 하는 것이다.
