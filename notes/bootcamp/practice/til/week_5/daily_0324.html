    1. 멤버십 등급 재계산 위치 논의

    포인트 환불 취소 시 누적 결제금액이 줄어들면 멤버십 등급도 재계산해야 한다. 이 로직을 어디에 두어야 하는지 세 가지 후보를 거쳐 최종 설계를 확정했다.

    후보 문제점 결과
    PointService 내부에 직접 구현 포인트 도메인이 멤버십 도메인을 직접 제어하는 책임 혼재 발생 폐기
    User 엔티티 내부로 이동 엔티티가 Repository를 의존하는 안티패턴 — 엔티티는 상태와 순수한 계산만 가져야 한다 폐기
    User 엔티티에 정책 목록을 파라미터로 전달 없음 채택
    최종 구조 — 엔티티가 정책을 파라미터로 받아 직접 계산 User.recalculateMembershipGrade(List<MembershipGrade> policies) 형태로 설계했다.

    PointService의 역할: MembershipGrade 정책 목록을 Repository에서 조회한 뒤 User에게 전달하는 것까지만 담당한다.
    User 엔티티의 역할: 전달받은 정책 목록과 자신의 누적 결제금액을 비교해 직접 등급을 계산한다.

    엔티티가 Repository를 의존하지 않으면서도 계산 로직을 엔티티 안에 캡슐화할 수 있다. PointService는 멤버십 정책의 세부 내용을 알 필요 없이 목록을 넘겨주는 역할만 하므로 책임이 명확하게 분리된다.

    2. PointRepository 쿼리 메서드 설계

    문제 원인 및 해결
    메서드명이 과도하게 길어짐
    findByTypeAndExpiresAtBeforeAndExpirationProcessedAtIsNull    		 Spring Data JPA 쿼리 메서드 네이밍은 조건이 많아질수록 가독성이 급격히 떨어진다. @Query로 JPQL을 직접 작성하는 방식으로 변경했다.
    FIFO용 쿼리와 만료 스케줄러용 쿼리를 하나로 합치려 함 두 쿼리의 목적이 다르다. FIFO용(findEarnedByUserOrderByExpiresAt)은 특정 유저를 대상으로 만료일 오름차순 조회이고, 만료 스케줄러용(findExpiredEarnedPoints)은 전체 유저 중 만료된 적립 포인트를 대상으로 한다. 목적이 다른 쿼리는 분리해야 한다.
    expiresAt Before now와 After now의 의미 혼동 Before(now)는 "만료일이 지금보다 이전 = 이미 만료된 것"을 의미한다. After(now)는 아직 유효한 포인트를 조회한다. 만료 스케줄러는 이미 만료된 것을 처리해야 하므로 Before가 올바르다.

    3. 스케줄러 구현

    구현한 스케줄러 목록

    클래스 역할
    PurchaseConfirmScheduler 결제 완료 후 7일이 지난 주문을 자동으로 구매 확정 처리한다.
    PointExpirationScheduler 매일 자정에 만료된 포인트를 소멸 처리한다.
    PointExpirationService 스케줄러에서 분리된 트랜잭션 처리 서비스 (아래 트러블슈팅 참고)

    트러블슈팅 정리

    문제 원인 및 해결
    @Transactional이 동일 클래스 내부 메서드 호출 시 무시됨 Spring AOP는 프록시 기반으로 동작한다. 같은 클래스 안에서 this.메서드()를 직접 호출하면 프록시를 우회하기 때문에 @Transactional이 적용되지 않는다. processExpiration()을 별도 클래스 PointExpirationService로 분리해서 해결했다.
    findExpiredEarnedPoints(PointTransactionType.EARNED, now) 파라미터 오류 JPQL에 EARNED 타입이 이미 하드코딩되어 있어 파라미터로 전달할 필요가 없었다. now만 전달하도록 수정했다.
    CONFIRM_AFTER_DATS 오타 CONFIRM_AFTER_DAYS로 수정. 상수명 오타는 컴파일 에러로 잡히지 않으므로 직접 눈으로 확인해야 한다.
    findByStatusAndCompletedAtBefore 필드 불일치 Order 엔티티에서 PAID 상태 전환 시 세팅되는 필드가 completedAt이 아닌 purchasedAt이었다. 실제 엔티티 필드명을 확인하고 findByStatusAndPurchasedAtBefore로 수정했다.
    order.complete() 메서드 없음 Order 엔티티에 존재하는 메서드는 complete()가 아닌 purchaseConfirmed()였다. 엔티티를 직접 열어서 실제 메서드명을 확인 후 수정했다.
    실패 로그를 log.info로 작성 실패·예외 상황은 반드시 log.error로 기록해야 로그 레벨 필터링 시 누락되지 않는다. 전체를 log.error로 통일했다.
    @Transactional이 동일 클래스 내부 호출에서 무시되는 이유 Spring의 @Transactional은 AOP 프록시를 통해 동작한다. 외부에서 Bean을 호출하면 프록시 객체가 가로채서 트랜잭션을 시작하지만, 같은 클래스 안에서 this.메서드()로 호출하면 프록시를 거치지 않고 실제 객체를 직접 호출하기 때문에 트랜잭션이 걸리지 않는다.

    해결 방법은 트랜잭션이 필요한 메서드를 별도 Bean(다른 클래스)으로 분리하는 것이다. 외부 Bean 호출이 되면 프록시가 정상적으로 개입한다. 스케줄러처럼 자기 클래스 내부에서 트랜잭션 메서드를 호출하는 패턴이 많으므로 이 원리를 반드시 기억해야 한다.

    4. FIFO 포인트 차감 전략 도입

    원장 직접 수정 방식 vs 원장 보존형 방식

    포인트는 만료일이 다른 여러 건으로 적립된다. 사용 시 만료일이 가장 이른 포인트부터 차감하는 FIFO 전략이 필요하다. 처음 제안된 방식은 적립 원장(PointTransaction)의 포인트를 직접 줄이는 방법이었지만 문제가 있었다.

    방식 문제점 결과
    원장 직접 수정
    earned.deductPoints()    		 원래 얼마를 적립받았는지 추적 불가. "언제, 얼마를 적립했다"는 사실 자체가 지워진다. 폐기
    원장 보존형
    (PointUsageDetail 별도 엔티티)    		 없음 채택
    원장 보존형 FIFO 구조 PointUsageDetail 엔티티를 별도로 설계하여 "어떤 적립 건에서 얼마를 사용했는지"를 추적한다.

    적립 원장(PointTransaction)은 절대 수정하지 않는다. "A 주문으로 500포인트 적립" 사실은 영원히 보존된다.
    사용 시 PointUsageDetail에 "이 적립 건에서 N포인트 사용"을 기록하는 방식이다.

    잔여 포인트 = 적립 원장 포인트 - 해당 원장에 연결된 PointUsageDetail 합계

    트러블슈팅 정리

    문제 원인 및 해결
    available < 0 조건 버그 available이 정확히 0일 때도 차감할 포인트가 없으므로 스킵해야 한다. available <= 0으로 수정했다. 경계값은 항상 등호 포함 여부를 명확히 따져야 한다.
    PointExpirationService를 scheduler 패키지에 배치 @Service가 붙은 클래스는 Service 계층이므로 service 패키지에 위치해야 한다. 스케줄러에서 호출한다고 해서 scheduler 패키지에 넣으면 패키지 구조의 의미가 모호해진다.
    Math.min(remainPoint, user.getPointBalance()) 의미 혼동 FIFO 미적용 시 포인트 잔액이 음수가 되는 상황을 방어하기 위한 코드였다. FIFO가 적용되면 remainPoint는 항상 pointBalance 이하이므로 이론상 불필요한 코드가 된다. 제거보다는 주석으로 맥락을 기록해두는 것이 안전하다.

    마무리

    오늘 가장 인상 깊었던 설계 결정은 멤버십 등급 재계산 위치였다. 처음에는 PointService에 직접 넣는 게 가장 쉬웠지만, 그렇게 하면 포인트 서비스가 멤버십 도메인을 제어하는 책임을 갖게 된다. 엔티티에 넣으면 Repository 의존 문제가 생긴다. 결국 "엔티티는 상태와 순수한 계산만 담당하고, 외부 데이터가 필요하면 파라미터로 받는다"는 원칙이 정답이었다. 설계는 여러 후보를 검토하면서 각각의 문제점을 드러내는 과정 자체가 중요하다.

    @Transactional이 동일 클래스 내부 호출에서 무시되는 문제는 스프링을 처음 사용할 때 가장 많이 실수하는 부분 중 하나다. 프록시 기반 AOP의 동작 원리를 이해하지 못하면 왜 트랜잭션이 안 걸리는지 원인을 찾기 어렵다. 스케줄러처럼 자기 클래스 내부에서 트랜잭션 메서드를 호출하는 패턴이 많으므로, 트랜잭션이 필요한 로직은 별도 Bean으로 분리하는 습관을 들여야 한다.

    FIFO 차감 전략에서 원장을 수정하지 않고 별도 추적 테이블을 두는 방식을 택한 것은 한 번 기록된 사실은 절대 지우지 않는다는 원칙을 따른 것이다. "언제, 얼마를 적립받았다"는 사실은 감사(audit) 목적과 고객 문의 대응에도 필수적이다. 당장 구현이 복잡해지더라도 원장을 보존하는 방식을 선택한 것이 장기적으로 옳은 결정이다.
