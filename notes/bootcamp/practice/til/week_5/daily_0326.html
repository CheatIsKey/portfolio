    작성일 2026-03-26 · 분류: 부트캠프, TIL, 팀 프로젝트, 결제 시스템, PortOne, 웹훅, 보상 트랜잭션, 스케줄러

    70일차 개요

    오늘 목표

    • 팀 프로젝트 결제 시스템의 전체 흐름을 명확하게 정리한다.
    • 결제 흐름 — 주문 생성부터 결제 검증, 상태 변경까지의 순서를 이해한다.
    • 취소/환불 흐름 — 취소 검증 → PortOne API 호출 → 상태 변경 → 웹훅 수신의 순서를 파악한다.
    • 스케줄러 흐름 — 콜백/웹훅이 유실됐을 때 PENDING 주문을 자동 복구하는 3차 방어선을 이해한다.
    • 보상 트랜잭션 — 금액 불일치, 취소 API 실패 등 예외 상황에서 데이터 정합성을 지키는 방법을 정리한다.

    1. 결제 흐름

    핵심 설계 원칙 — 클라이언트를 신뢰하지 않는다 결제 금액 검증은 반드시 백엔드가 PortOne REST API에 직접 조회해서 수행한다. 클라이언트가 보내주는 금액이나 상태를 그대로 믿으면 변조 공격에 취약하다. "클라이언트가 결제됐다고 알려줌"과 "실제로 결제됐는지 서버가 직접 확인함"은 완전히 다른 것이다.
    1. 고객이 결제 버튼 클릭
    2. 클라이언트 → 백엔드: 주문 생성 요청 POST /api/payment/orders
    3. 백엔드: DB에 PENDING 상태 주문 저장 → orderUid + amount 응답
    4. 클라이언트: PortOne JS SDK로 결제창 직접 호출 (백엔드를 거치지 않음)
    5. 고객: 결제창에서 카드 정보 입력 후 결제 완료
    6. 결제 완료 알림이 두 경로로 동시에 전달됨
    경로 동작
    경로 A — JS 콜백 클라이언트가 paymentId를 받아 백엔드에 검증 요청을 보낸다.
    경로 B — 웹훅 PortOne이 백엔드로 직접 POST를 보낸다. 클라이언트를 거치지 않는다.
    1. 백엔드: paymentId로 PortOne REST API에 직접 결제 상태 조회 (PortOne이 알려주는 게 아닌, 백엔드가 직접 가져옴)
    2. 백엔드: DB 저장 금액 vs PortOne 조회 금액 직접 비교 → 금액 일치 확인
    3. 금액 일치 → 주문 상태 PAID로 변경, 재고 차감, 이벤트 기록
    4. 백엔드 → 클라이언트: 주문 완료 응답 → 고객에게 완료 화면 표시

    2. 취소/환불 흐름

    1. 고객이 취소 버튼 클릭
    2. 클라이언트 → 백엔드: 취소 요청 POST /api/payment/cancel (paymentId + reason 전달)
    3. 백엔드 검증 3단계
    검증 항목 내용
    주문 존재 여부 해당 paymentId로 저장된 주문이 있는가
    주문 상태 확인 주문 상태가 PAID인 경우만 취소 가능 (PENDING, FAILED 상태는 취소 불가)
    본인 주문 확인 로그인 구현 이후 추가 예정
    1. 백엔드 → PortOne REST API: 취소 요청
    2. PortOne: 실제 결제 취소 처리 (테스트 환경에서는 실제 환불 없음)
    3. 백엔드: 주문 상태 CANCELED로 변경, 재고 복구, 이벤트 기록
    4. PortOne → 백엔드 웹훅: Transaction.Cancelled 수신 (취소 완료 알림)
    5. 웹훅 핸들러: 멱등성 체크 후 이미 처리됐으면 skip
    6. 백엔드 → 클라이언트: 취소 완료 응답 → 고객에게 취소 완료 화면 표시
    웹훅 멱등성(Idempotency)이 필요한 이유 콜백(경로 A)과 웹훅(경로 B)은 동시에 백엔드로 전달된다. 두 경로가 모두 정상 수신되면 동일한 결제/취소 처리 로직이 두 번 실행될 수 있다. 웹훅 핸들러에서 이미 해당 상태로 처리됐는지 확인하고 중복 처리를 skip하는 멱등성 처리가 필수다. "같은 요청을 여러 번 처리해도 결과가 동일하다"는 것이 멱등성의 의미다.

    3. 스케줄러 흐름 (3차 방어선)

    콜백과 웹훅이 모두 유실되면 결제는 완료됐지만 주문이 PENDING 상태로 영원히 방치될 수 있다. 스케줄러가 5분마다 이런 주문을 탐지하고 자동으로 복구한다.

    1. Spring @Scheduled로 5분마다 자동 실행
    2. DB에서 PENDING 상태이면서 생성된 지 10분 이상 지난 주문 목록 조회 (10분 이내는 아직 결제 진행 중일 수 있으므로 제외)
    3. 각 주문에 대해 백엔드 → PortOne REST API: paymentId로 결제 상태 직접 조회
    4. 조회 결과에 따라 분기 처리
    PortOne 조회 결과 백엔드 처리
    PAID 금액 검증 후 주문 PAID 처리, 이벤트 기록 (SCHEDULER_RECOVERED)
    FAILED 주문 FAILED 처리, 이벤트 기록
    해당 결제 없음 주문 FAILED 처리, 이벤트 기록
    PENDING 그대로 두고 다음 사이클에서 재확인
    1. 재시도 횟수가 3회를 초과한 주문은 포기 처리 후 관리자 알림 발송

    4. 보상 트랜잭션 (Compensating Transaction)

    결제 과정에서 예외 상황이 발생하면 데이터 정합성이 깨질 수 있다. 보상 트랜잭션은 이미 일어난 작업을 원상 복구하여 시스템 상태의 일관성을 유지하는 패턴이다.

    케이스 A — 결제창에서 고객이 직접 취소한 경우

    1. 고객이 결제창에서 취소 버튼 클릭
    2. PortOne JS SDK 콜백: response.code에 에러 코드가 담겨 클라이언트로 반환
    3. 클라이언트 → 백엔드: 결제 실패 알림 POST /api/payment/fail (orderUid 전달)
    4. 백엔드: 주문 상태 FAILED로 변경, 이벤트 기록 (이 시점까지 재고를 차감하지 않았으므로 재고 복구 불필요)

    케이스 B — 결제는 됐지만 금액 불일치가 감지된 경우

    1. 백엔드: 금액 불일치 감지
    2. 보상 트랜잭션 시작 — 아래 작업들이 하나의 트랜잭션으로 묶임
      • 이벤트 기록: VERIFY_FAILED
      • 백엔드 → PortOne REST API: 즉시 취소 요청
      • 이벤트 기록: AUTO_CANCELLED
      • 주문 상태 FAILED로 변경
    3. PortOne 취소 API 호출 자체가 실패한 경우
      • 주문 상태를 CANCEL_PENDING으로 저장
      • 스케줄러가 주기적으로 취소 재시도
      • 재시도 3회 초과 시 관리자 알림 발송

    케이스 C — 네트워크 단절로 콜백과 웹훅이 모두 유실된 경우

    1. 결제는 완료됐지만 백엔드가 어떤 알림도 받지 못함
    2. 주문이 PENDING 상태로 방치됨
    3. 스케줄러가 위 3. 스케줄러 흐름을 수행하여 자동 복구

    5. 전체 방어선 구조 요약

    방어선 수단 트리거 역할
    1차 클라이언트 콜백 POST /api/payment/verify 정상 결제 완료 시 즉시 검증 및 상태 변경
    2차 PortOne 웹훅 POST /api/webhook/portone 콜백이 유실됐을 때 PortOne이 직접 알림 전송 (멱등성 처리 필수)
    3차 스케줄러 5분마다 자동 실행 콜백·웹훅 둘 다 유실된 PENDING 주문 자동 복구
    보상 보상 트랜잭션 금액 불일치 감지 즉시 즉시 자동 취소 → 실패 시 CANCEL_PENDING 저장 후 재시도
    왜 이렇게 복잡한 방어선이 필요한가 결제 시스템은 돈이 오가는 곳이다. 어느 한 경로가 실패해도 다른 경로가 반드시 동일한 결과를 보장해야 한다.

    1차 방어(콜백)만 있으면 브라우저가 닫히거나 네트워크가 끊기면 결제가 완료됐지만 주문이 PENDING으로 방치된다.
    2차 방어(웹훅)를 추가하면 클라이언트 없이도 PortOne이 직접 알려준다. 그러나 웹훅도 네트워크 이슈로 유실될 수 있다.
    3차 방어(스케줄러)가 최후의 안전망 역할을 한다. 어떤 경로도 실패해도 결국 정합성이 회복된다.

    보상 트랜잭션은 이미 일어난 결제를 되돌리는 수단이다. 금액 불일치처럼 서버가 검증에 실패한 경우 자동으로 취소하고, 취소도 실패하면 상태(CANCEL_PENDING)를 저장해두고 나중에 재시도한다.

    마무리

    결제 시스템은 기능을 만드는 것보다 실패 상황을 어떻게 처리할 것인지를 설계하는 것이 훨씬 더 중요하다는 것을 이번 프로젝트를 통해 체감했다. 정상 흐름은 열 단계지만, 예외 처리를 위한 보상 트랜잭션과 스케줄러는 훨씬 더 복잡하다. 그런데 이 복잡함이 빠져 있으면 "결제는 됐는데 주문이 없다"는 치명적인 상황이 발생한다.

    클라이언트 콜백과 웹훅이 동시에 도달할 수 있다는 점도 중요한 설계 포인트였다. 두 경로가 모두 정상 동작하면 동일한 로직이 두 번 실행될 수 있다. 멱등성 처리 없이 구현하면 재고가 두 번 차감되거나 포인트가 두 번 적립되는 문제가 생긴다. 같은 요청을 여러 번 처리해도 결과가 달라지지 않아야 한다는 멱등성 원칙이 결제 시스템에서 왜 필수인지 이해했다.

    스케줄러가 PENDING 주문을 처리하기 시작하는 기준을 10분으로 잡은 이유도 중요하다. 결제창에서 고객이 한참 고민 중일 수 있기 때문에 너무 이른 시점에 FAILED로 처리하면 진행 중인 결제를 강제 종료하는 문제가 생긴다. "아직 진행 중인 것"과 "포기된 것"을 구분하는 시간 버퍼를 두는 것도 설계의 일부다.
