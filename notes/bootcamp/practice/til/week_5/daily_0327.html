    1. 단위 테스트 — Mock/Stubbing 관련

    문제 원인 및 해결
    Optional.of(null) → NPE 발생 "존재하지 않음"을 표현할 때는 Optional.of(null)이 아닌 Optional.empty()를 사용해야 한다. Optional.of()는 null을 허용하지 않는다.
    Unnecessary Stubbing 에러 서비스 코드의 실제 호출 흐름을 따라가서 실제로 호출되지 않는 stubbing을 제거했다. filter에서 걸러지는 정책의 getMembershipCode()나 confirmPayment()에서 userId 검증 실패 시 이후 코드가 실행되지 않으므로 관련 stubbing이 불필요했다.
    payment.getPaymentId() stubbing 불필요 리팩토링으로 portOnePaymentClient.getPayment(payment.getPaymentId())가 portOnePaymentClient.getPayment(paymentId)로 변경됐다. 파라미터를 직접 사용하므로 payment.getPaymentId()를 stubbing할 필요가 없어졌다.
    Unnecessary Stubbing을 방지하는 습관 테스트를 작성할 때 stubbing을 먼저 다 걸어두고 시작하면 실제로 호출되지 않는 stubbing이 남기 쉽다. 서비스 코드의 실행 흐름을 먼저 파악하고, 실제로 호출되는 메서드만 stubbing하는 것이 올바른 접근이다. Early Return이나 예외로 실행이 중단되는 지점 이후의 호출은 모두 불필요한 stubbing이 된다.

    2. 리팩토링 후 테스트 코드 수정

    PaymentService 구조 변경

    변경 내용 테스트 코드 수정 사항
    findByPaymentIdForUpdate → findByPaymentId로 변경 (락은 PaymentFinalizeService로 이동) stubbing 대상 메서드명 수정
    PaymentFinalizeService 의존성 추가 테스트 클래스에 @Mock PaymentFinalizeService paymentFinalizeService 추가
    INTERNAL 결제 확정 검증 로직 변경
    processSuccess(payment, "INTERNAL") → paymentFinalizeService.finalizeInternalPayment(paymentId)    		 verify() 검증 대상을 새 메서드로 교체

    RefundService 구조 변경

    변경 내용 테스트 코드 수정 사항
    PublicIdGenerator, RefundTransactionProcessor → RefundFinalizeService로 이동 기존 두 Mock을 제거하고 @Mock RefundFinalizeService 하나로 교체
    환불 처리 메서드 변경
    refundTransactionProcessor.processRefund() → refundFinalizeService.finalizeRefund(paymentId, amount, reason, providerRefundId)    		 verify() 검증 대상을 refundFinalizeService.finalizeRefund()로 교체
    ⚠️ 리팩토링 후 테스트 코드도 함께 수정해야 한다 프로덕션 코드를 리팩토링하면 테스트 코드도 반드시 함께 수정해야 한다. Mock 대상, stubbing 메서드명, verify 검증 대상이 모두 구체적인 클래스·메서드에 의존하기 때문에, 구조가 바뀌면 테스트가 틀린 것을 검증하고 있는 상태가 된다. 테스트가 통과하더라도 잘못된 것을 검증하고 있을 수 있으므로 리팩토링 후에는 반드시 테스트 코드도 점검해야 한다.

    3. @SpringBootTest 컨텍스트 로딩 에러

    문제 원인 및 해결
    oauth2.redirect-url PlaceholderResolutionException application-test.yml에 oauth2.redirect-url 값이 누락되어 있었다. 더미값을 추가해서 해결했다.
    webhookSignatureService Illegal base64 character 2d (-) portone.webhook.secret에 하이픈(-)이 포함된 문자열을 설정했는데, PortOne SDK가 이 값을 Base64로 디코딩하려다 실패했다. 유효한 Base64 문자열로 교체하거나 @MockitoBean으로 WebhookSignatureService를 Mock 처리하는 방식으로 해결했다.
    Base64 에러 문자 2d ASCII 코드 2d는 하이픈(-)이다. Base64 인코딩은 A-Z, a-z, 0-9, +, /, =만 허용한다. 하이픈은 URL-safe Base64에서는 허용되지만 표준 Base64에서는 유효하지 않은 문자다. 테스트용 시크릿 값을 아무 문자열이나 쓰지 않고 실제로 Base64 인코딩된 값을 사용해야 하거나, 해당 Bean을 Mock으로 대체해야 한다.

    4. 동시성 테스트 이슈

    문제 원인 및 해결
    UserPointConcurrencyTest — 검증 로직 오류
    isEqualTo(10_000L - successCount * 2_000L) 실패    		 락이 없는 환경에서 Lost Update가 발생해 실제 잔액이 예측 값과 달라진다. 동시성 테스트의 목적은 "음수가 되지 않는다"는 것을 검증하는 것이므로 isGreaterThanOrEqualTo(0L)만 남겼다.
    PaymentConcurrencyTest — executor.shutdown() 직후 READY 상태로 조회됨 shutdown()은 작업 제출을 중단할 뿐 모든 작업이 완료될 때까지 기다리지 않는다. executor.awaitTermination(10, TimeUnit.SECONDS)를 추가해 트랜잭션이 커밋될 때까지 기다린 후 조회하도록 했다.
    PaymentConcurrencyTest — 포인트 잔액 부족으로 InsufficientPointException 발생 후 롤백 테스트 유저 생성 시 포인트를 초기화하지 않았다. processSuccess() 내부에서 subtractPoint()를 호출할 때 잔액이 0이라 예외가 발생했다. user.addPoint(10_000L)를 추가해서 해결했다.
    동시성 테스트에서 검증해야 할 것 동시성 테스트는 "정확한 최종값"을 검증하는 테스트가 아니다. 락 없이 여러 스레드가 동시에 수정하면 Lost Update가 발생해 최종값이 예측 불가능해진다. 이것 자체가 문제임을 보여주는 것이 테스트의 목적이다.

    락이 있을 때는 정확한 최종값을 검증할 수 있다. 락이 없을 때는 "최소한 음수가 되지 않는다", "데드락 없이 완료된다" 같은 안전성 조건만 검증한다.

    executor.shutdown()과 awaitTermination()은 항상 함께 써야 한다. shutdown()만 호출하면 작업이 끝나기 전에 검증 코드가 실행된다.

    5. GitHub Actions CI 실패

    CI 파이프라인이 로컬에서는 통과하는데 GitHub Actions에서는 계속 실패했다.

    구분 내용
    원인 application-test.yml을 .gitignore에 추가해두어서 GitHub 원격 저장소에 파일이 없었다. CI 환경은 원격 저장소에서 코드를 체크아웃하기 때문에 해당 파일이 없어 컨텍스트 로딩에 실패했다.
    해결 application-test.yml은 전부 가짜 테스트 더미값만 포함하므로 실제 보안 정보가 없다. .gitignore에서 제외하고 커밋해서 CI 환경에서도 파일을 사용할 수 있도록 했다.
    ⚠️ gitignore와 CI 환경의 관계 .gitignore에 추가된 파일은 원격 저장소에 올라가지 않으므로 CI 환경에서도 존재하지 않는다. 테스트 설정 파일(application-test.yml)은 실제 비밀값이 없다면 .gitignore에 넣으면 안 된다. 민감한 정보는 GitHub Secrets로 관리하고, 테스트 더미값 파일은 버전 관리에 포함하는 것이 올바른 방식이다.

    6. 카카오 OAuth — account_email 권한 오류 (KOE205)

    구분 내용
    원인 카카오 account_email 권한은 비즈앱 심사를 통과해야만 활성화된다. 심사 전에는 scope에 설정 자체가 불가능하다.
    비즈앱 심사 소요 시간 영업일 3~5일. 내일 발표 일정에 맞추는 것은 불가능하다.
    해결 1 application-local.yml의 scope에서 account_email을 제거한다.
    해결 2 OAuthUserInfo.fromKakao()에서 이메일 null 처리를 추가하고, 이메일이 없을 경우 providerId@kakao.user 형태의 임시 이메일을 생성하도록 처리한다.
    카카오 OAuth 비즈앱 심사가 필요한 이유 카카오는 개인정보(이메일, 전화번호 등)를 수집하는 권한은 사업자 등록이 된 앱에만 허용한다. 일반 테스트 앱 상태에서는 profile_nickname, profile_image 같은 기본 정보만 사용할 수 있다. 프로젝트나 서비스를 실제로 출시할 때는 비즈앱 전환과 권한 심사를 미리 진행해야 한다. 발표처럼 시간이 촉박한 상황에서는 이메일 없이도 동작하는 임시 처리를 먼저 구현하고 심사를 기다리는 방식이 현실적이다.

    마무리

    오늘은 기능 구현보다 테스트와 CI 환경 안정화에 집중한 하루였다. Mock과 Stubbing은 단순히 "메서드를 가짜로 만드는 것"이 아니라 서비스 코드의 실행 흐름을 정확히 이해해야 올바르게 쓸 수 있다. 불필요한 stubbing은 테스트가 실제 코드 흐름과 다른 것을 가정한다는 신호이므로, Unnecessary Stubbing 에러는 코드를 다시 읽어봐야 한다는 알림으로 받아들여야 한다.

    동시성 테스트에서 "정확한 최종값"을 검증하려 한 것이 큰 실수였다. 락 없이 여러 스레드가 동시에 같은 데이터를 수정하면 Lost Update가 발생해 최종값은 예측 불가능하다. 동시성 테스트의 목적은 정확한 값이 아니라 안전성 조건(음수 방지, 데드락 없이 완료)을 검증하는 것이다. executor.awaitTermination()을 빠뜨린 것도 모든 스레드가 완료되기 전에 검증 코드가 실행된다는 타이밍 문제였다.

    GitHub Actions CI 실패 원인이 .gitignore에 있었다는 점은 단순하지만 중요한 교훈이다. 로컬에서 돌아간다고 CI도 돌아간다고 가정하면 안 된다. CI는 원격 저장소 기준으로 동작하므로 로컬에만 있는 파일에 의존하는 테스트는 CI에서 반드시 실패한다. 테스트 설정 파일은 실제 비밀값이 없다면 버전 관리에 포함시키고, 민감한 값은 GitHub Secrets로 분리하는 것이 원칙이다.
