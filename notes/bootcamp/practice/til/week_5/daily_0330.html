    내일배움캠프 Spring 백엔드 72일차 TIL — 데이터베이스 인덱스(Index)와 B-Tree · Index Scan 종류

    작성일 2026-03-30 · 분류: 부트캠프, TIL, Database, Index, B-Tree, 쿼리 최적화

    72일차 개요

    오늘 목표

    • Index의 개념과 역할을 이해한다.
    • 인덱스 적용 여부에 따른 검색 성능 차이를 파악한다.
    • B-Tree 구조가 인덱스에 사용되는 이유를 이해한다.
    • Index Scan 4종류(Unique Scan, Range Scan, Full Index Scan, ICP)의 동작 방식과 발생 조건을 정리한다.

    1. Index란 무엇인가

    데이터를 빠르게 찾기 위한 검색용 지도다. 단순히 가나다순으로 정렬된 목록이 아니라, 검색에 특화된 트리(Tree) 구조의 탐색 지도다. 테이블과 별도의 자료 구조로 관리되며, PRIMARY KEY와 UNIQUE 제약 조건이 걸린 컬럼에는 자동으로 생성된다.

    항목 내용
    기본 역할 원하는 데이터를 빠르게 찾도록 도와준다.
    저장 위치 테이블과 별도의 자료 구조로 관리된다.
    구조 형태 B-Tree 알고리즘을 사용한다.
    자동 생성 조건 PRIMARY KEY, UNIQUE 제약 조건이 걸린 컬럼

    인덱스 적용 여부 비교

    구분 인덱스 없음 인덱스 있음
    검색 방식 전체 테이블 탐색 (Full Table Scan) 정렬된 인덱스 탐색
    시간 복잡도 O(n) O(log n)
    I/O 비용 매우 높음 낮음
    추가 공간 없음 필요
    INSERT / UPDATE 빠름 약간 느림 (인덱스도 함께 갱신)
    ⚠️ 인덱스는 공짜가 아니다 인덱스를 추가하면 조회 성능은 향상되지만, INSERT / UPDATE / DELETE 시 인덱스도 함께 갱신해야 하므로 쓰기 성능이 저하된다. 또한 인덱스 자체가 별도의 저장 공간을 차지한다. 조회가 압도적으로 많은 컬럼에 선택적으로 적용해야 한다.

    2. B-Tree (Balanced Tree)

    데이터를 분할, 정렬, 균형 유지하는 탐색 트리다. 루트 노드를 기준으로 검색할 값이 더 크면 오른쪽, 더 작으면 왼쪽으로 이동한다. 이 구조 덕분에 전체 데이터를 다 보지 않고 절반씩 줄여가며 탐색할 수 있어 O(log n)의 시간 복잡도를 달성한다.

    B-Tree가 인덱스에 적합한 이유 B-Tree는 항상 균형(Balanced)을 유지하기 때문에 어떤 데이터를 찾더라도 탐색 깊이가 일정하다. 특정 데이터에만 편향된 탐색 경로가 생기지 않으므로 최악의 경우에도 O(log n)이 보장된다.

    또한 리프 노드(Leaf Node)끼리 연결되어 있어 범위 검색에 유리하다. 시작 위치를 찾은 뒤 옆으로 이어서 읽으면 되기 때문에 BETWEEN, >, < 같은 범위 조건이 효율적으로 처리된다.

    3. Index Scan 종류

    1. Unique Scan

    정확히 한 건의 데이터를 찾는 탐색이다. 인덱스가 PRIMARY KEY 또는 UNIQUE 제약일 때 발생하며, B-Tree에서 정확히 한 경로만 따라가면 끝나기 때문에 탐색 속도가 가장 빠르다. 불필요한 비교나 순회가 전혀 없다. 

    -- PRIMARY KEY 조건 → Unique Scan 발생
SELECT * FROM orders WHERE id = 1;

    2. Range Scan

    일정 범위(구간)의 데이터를 연속적으로 읽는 탐색이다. B-Tree의 정렬된 구조를 활용해 시작 위치를 찾은 뒤 순서대로 이어서 읽는다. 이미 정렬된 상태로 읽히기 때문에 ORDER BY를 사용해도 추가 정렬이 불필요하다. 

    -- BETWEEN, >, <, LIKE 'abc%' → Range Scan 발생
SELECT * FROM orders WHERE created_at BETWEEN '2026-01-01' AND '2026-03-31';
SELECT * FROM members WHERE name LIKE '김%';
    LIKE 조건에서 Range Scan이 되는 경우와 안 되는 경우 LIKE '김%'처럼 앞부분이 고정된 경우에는 인덱스의 시작 위치를 특정할 수 있어 Range Scan이 가능하다.
    LIKE '%김'처럼 앞부분이 와일드카드인 경우에는 시작 위치를 특정할 수 없어 Full Table Scan이 발생한다.

    3. Full Index Scan

    인덱스 전체를 처음부터 끝까지 순서대로 읽는 탐색이다. WHERE 조건이 없어 인덱스 전체를 탐색하지만, 인덱스 자체가 이미 정렬된 상태이므로 ORDER BY 시 추가 정렬이 필요 없다. Full Table Scan보다는 읽는 데이터량이 적기 때문에 I/O 비용이 낮다. 

    -- WHERE 조건 없이 정렬만 있는 경우 → Full Index Scan 발생 가능
SELECT id, name FROM members ORDER BY name;

    4. Index Condition Pushdown (ICP)

    WHERE 조건 중 일부를 인덱스 탐색 단계에서 미리 필터링하는 최적화 기법이다. 기존에는 인덱스로 후보 데이터를 찾은 뒤 테이블에서 실제 행을 읽어와 조건을 검사했다. ICP가 적용되면 인덱스 탐색 중에 조건을 먼저 검사해서 조건을 만족하지 않는 행은 테이블 접근 자체를 하지 않는다. 불필요한 디스크 I/O가 줄어들어 성능이 향상된다. 

    -- 복합 인덱스 (last_name, first_name)이 있을 때
-- last_name으로 Range Scan 후 first_name 조건을 인덱스 단계에서 미리 필터링
SELECT * FROM members WHERE last_name = '김' AND first_name LIKE '민%';
    ICP 적용 전후 비교 ICP 적용 전: 인덱스에서 last_name = '김'인 행을 모두 찾음 → 각 행에 대해 테이블 접근 → first_name LIKE '민%' 조건 검사

    ICP 적용 후: 인덱스에서 last_name = '김'인 행을 찾는 중에 first_name LIKE '민%'도 함께 검사 → 조건을 통과한 행만 테이블에 접근

    테이블 접근 횟수 자체를 줄이므로 디스크 I/O가 대폭 감소한다.

    4가지 Index Scan 요약 비교

    종류 발생 조건 탐색 범위 특징
    Unique Scan PK 또는 UNIQUE 컬럼 조건 정확히 1건 가장 빠름. 추가 순회 없음
    Range Scan BETWEEN, >, <, LIKE 'abc%' 시작 ~ 끝 구간 정렬된 순서로 읽어 ORDER BY 효율적
    Full Index Scan WHERE 없이 인덱스 컬럼 전체 조회 인덱스 전체 Full Table Scan보다 I/O 적음
    ICP 복합 인덱스 + 추가 WHERE 조건 인덱스 탐색 중 필터링 테이블 접근 횟수 감소 → I/O 절감

    마무리

    인덱스는 조회 성능을 극적으로 향상시키지만 공짜가 아니라는 것이 핵심이다. 쓰기 성능 저하와 추가 저장 공간이라는 비용이 따른다. 따라서 인덱스를 모든 컬럼에 무조건 추가하는 것이 아니라 조회 빈도가 높고 카디널리티(고유값 수)가 높은 컬럼에 선택적으로 적용해야 한다.

    B-Tree 구조를 이해하면 왜 LIKE '%김'이 인덱스를 타지 못하는지, 왜 ORDER BY가 인덱스를 사용하면 추가 정렬이 필요 없는지 자연스럽게 이해된다. B-Tree는 이미 정렬된 상태를 유지하기 때문에 범위 검색과 정렬 모두에 유리한 것이다.

    ICP는 인덱스 탐색 단계에서 조건을 미리 적용해 테이블 접근 횟수 자체를 줄이는 최적화다. 복합 인덱스를 설계할 때 ICP가 적용될 수 있는 구조를 고려하면 I/O 비용을 크게 줄일 수 있다. 실무에서 쿼리가 느릴 때 EXPLAIN으로 어떤 Index Scan이 발생하는지 확인하는 것이 쿼리 최적화의 첫 단계다.
