    오늘 목표

    • JPQL의 한계와 QueryDSL이 이를 해결하는 방식을 이해한다.
    • 동적 쿼리가 필요한 이유와 SRP와의 관계를 정리한다.
    • BooleanBuilder vs BooleanExpression의 차이와 사용 시점을 파악한다.
    • JPA 연관관계가 유발하는 문제를 이해하고 ID 기반 설계 + QueryDSL 조인으로 전환하는 방식을 익힌다.

    1. JPQL의 한계와 QueryDSL

    JPQL은 JPA에서 객체를 대상으로 쿼리를 작성하는 언어지만, 문자열 기반이라는 근본적인 한계가 있다. QueryDSL은 이 JPQL을 타입 안전하게 작성하기 위한 라이브러리이며, SQL이 아닌 JPA 위에서 동작하는 쿼리 빌더다.

    JPQL 문제점 설명
    문자열 기반 IDE 자동완성, 리팩토링이 불가능하다.
    컴파일 시 검증 불가 오타나 필드 변경 시 런타임 오류로 이어진다.
    유지보수 어려움 동적 쿼리 작성 시 가독성이 크게 저하된다.
    QueryDSL의 장점 IDE 자동완성 지원 · 컴파일 타임 검증 · 동적 쿼리 작성 간결화 · 코드 가독성 및 유지보수성 향상
    Q-Type QueryDSL 전용 클래스로, 빌드 시 자동으로 생성된다. 엔티티 클래스 User에 대해 QUser가 생성되며, QUser.user.username 같은 방식으로 필드에 타입 안전하게 접근한다. 문자열 대신 자바 코드로 쿼리를 작성하므로 오타가 컴파일 에러로 바로 잡힌다.

    2. 동적 쿼리가 필요한 이유

    동적 쿼리는 사용자가 입력한 검색 조건에 따라 쿼리를 유연하게 생성하는 방식이다. Spring Data JPA의 메서드 이름 기반 방식만으로는 여러 복합 조건(이름이 ~이거나, 이메일이 ~이면서, 역할이 ~인 경우)을 표현하기 어렵다. 

    // 메서드 이름 기반 방식의 한계 — 조건이 조금만 바뀌어도 메서드가 폭발적으로 늘어난다
List<User> findByUsernameAndEmailContainsAndRoleEnum(
    String username, String email, UserRoleEnum roleEnum
);
    동적 쿼리와 SRP (단일 책임 원칙) SRP는 "클래스마다 하나의 책임만 가져야 한다"는 원칙이며, 더 정확히는 "하나의 클래스(또는 함수)가 한 가지 이유로만 변경되어야 한다"는 의미다.

    동적 쿼리는 '검색 조건이 다양하다'는 하나의 책임(검색 기능)을 수행하는 것이므로 SRP를 준수한다. 조건 하나가 추가될 때마다 새 메서드를 만드는 것이 오히려 책임을 분산시키는 나쁜 설계다.

    3. BooleanBuilder vs BooleanExpression

    BooleanBuilder

    여러 조건을 동적으로 추가하거나 제거할 수 있는 가변 조건 컨테이너다. 조건을 하나씩 쌓아가며 만드는 방식이라 복잡한 검색 조건에 유용하다. 다만 if문이 길어지고 조건 메서드를 재사용하기 어렵다. 

    BooleanBuilder builder = new BooleanBuilder();

if (username != null) builder.and(user.username.contains(username));
if (email != null)    builder.and(user.email.contains(email));
if (role != null)     builder.and(user.role.eq(role));

queryFactory.selectFrom(user).where(builder).fetch();

    BooleanExpression

    하나의 조건(표현식)을 메서드로 정의해 조합하는 불변 표현식이다. null을 반환하면 QueryDSL이 해당 조건을 자동으로 무시하므로 별도의 null 체크 없이 코드가 깔끔해진다. 메서드 간 .and()나 .or()로 병합하며, 재사용성이 높다. 

    private BooleanExpression usernameContains(String username) {
    return username != null ? user.username.contains(username) : null;
}

private BooleanExpression emailContains(String email) {
    return email != null ? user.email.contains(email) : null;
}

private BooleanExpression roleEq(UserRoleEnum role) {
    return role != null ? user.role.eq(role) : null;
}

// null을 반환하는 조건은 QueryDSL이 자동으로 무시
queryFactory
    .selectFrom(user)
    .where(
        usernameContains(cond.getUsername()),
        emailContains(cond.getEmail()),
        roleEq(cond.getRole())
    )
    .fetch();

    비교 요약

    구분 BooleanBuilder BooleanExpression
    개념 조건을 모아두는 컨테이너 단일 조건의 표현식
    객체 성격 가변 (Mutable) 불변 (Immutable)
    null 처리 직접 제어 (if문 필요) null 반환 시 자동 무시
    where 작성 방식 .where(builder) .where(조건1, 조건2, 조건3)
    재사용성 낮음 높음
    코드 가독성 복잡함 깔끔함
    선택 기준 BooleanBuilder: 조건이 많고, 일부 조건을 동적으로 추가/삭제해야 할 때
    BooleanExpression: 조건이 고정되어 있고, null만 무시하면 될 때

    4. JPA 연관관계의 문제와 ID 기반 설계

    JPA 연관관계가 유발하는 문제들

    JPA는 객체와 테이블을 1:1로 매핑하는 ORM이므로 객체 그래프 탐색이 가능하도록 엔티티 간 연결이 필요하다. 개발자가 SQL을 직접 작성하지 않아도 객체 참조만으로 관계를 탐색할 수 있다. 하지만 이 편리함이 아래와 같은 문제들을 유발한다.

    문제 설명
    N+1 문제 연관 객체를 탐색할 때 쿼리가 N번 추가로 실행된다.
    Lazy/Eager 로딩 충돌 서비스마다 로딩 전략이 달라 일관성을 유지하기 어렵다.
    순환 참조 양방향 매핑 시 JSON 직렬화에서 무한 루프가 발생한다.
    쿼리 제어 불가 JPA가 내부적으로 어떤 SQL을 날릴지 예측하기 어렵다.
    성능 튜닝 제한 fetch join, batch size 등으로 임시 해결만 가능하다.

    해결 방향 — ID 기반 설계 + QueryDSL 조인

    JPA 연관관계 어노테이션(@ManyToOne 등) 대신 단순히 외래키 ID 값만 필드로 저장하고, 조인이 필요한 시점에만 QueryDSL로 연결하는 방식이다. 연관은 코드가 아니라 쿼리로 관리한다. 

    // Before: JPA 연관관계 방식
@Entity
public class Post {
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id")
    private User user; // 언제 로딩될지 제어하기 어렵다
}

// After: ID 기반 설계
@Entity
public class Post {
    private Long userId; // 단순 ID 값만 저장
}
    // 조인이 필요할 때만 QueryDSL로 명시적으로 연결
queryFactory
    .selectFrom(post)
    .leftJoin(user).on(post.userId.eq(user.id))
    .fetch();
    구분 JPA 연관관계 방식 QueryDSL + ID 방식
    관계 표현 @ManyToOne Long userId
    쿼리 생성 JPA가 자동으로 SQL 생성 개발자가 직접 제어
    성능 예측 어려움 완전한 예측 가능
    Lazy 로딩 프록시 객체 필요 불필요
    순환 참조 위험 있음 완전 제거
    유지보수성 낮음 매우 높음
    언제 연관관계를 쓰고 언제 ID 기반으로 쓸까 연관이 필요 없는 것은 JPA가, 조인이 필요한 것은 QueryDSL이 처리하도록 설계한다.

    단순 CRUD처럼 조인이 거의 발생하지 않는 관계라면 JPA 연관관계가 편리하다. 하지만 복잡한 조인 쿼리, 성능이 중요한 조회, N+1이 빈번하게 발생하는 관계라면 ID만 보관하고 QueryDSL에서 필요할 때만 조인하는 방식이 훨씬 예측 가능하고 유지보수하기 쉽다.

    마무리

    QueryDSL은 단순히 JPQL을 대체하는 도구가 아니라, 타입 안전성과 동적 쿼리 작성이라는 두 가지 문제를 동시에 해결한다. 문자열로 쿼리를 작성하던 방식에서 자바 코드로 쿼리를 표현하면 오타와 필드 변경에 의한 런타임 오류가 컴파일 오류로 바뀐다. 오류가 늦게 발견될수록 고치는 비용이 커지기 때문에 이 차이는 실무에서 매우 중요하다.

    BooleanBuilder와 BooleanExpression은 상황에 따라 선택한다. BooleanExpression으로 각 조건을 메서드로 쪼개면 조건 하나를 다른 쿼리에서 재사용할 수 있고, null 처리가 자동으로 되어 코드가 훨씬 깔끔해진다. where()에 여러 조건을 나열할 때 null이 있으면 그냥 무시되기 때문에 별도의 if 분기 없이도 동적 쿼리를 만들 수 있다는 점이 핵심이다.

    JPA 연관관계를 ID 기반으로 전환하는 아이디어는 처음에는 낯설게 느껴졌지만, N+1과 순환 참조 문제를 근본적으로 제거한다는 점에서 설득력이 있다. 조인이 필요할 때만 명시적으로 QueryDSL로 연결하면 어떤 SQL이 실행되는지 완전히 예측할 수 있다. JPA의 편리함이 필요한 곳에는 연관관계를, 쿼리 제어가 중요한 곳에는 ID 기반 + QueryDSL을 선택적으로 사용하는 것이 균형 잡힌 접근이다.
