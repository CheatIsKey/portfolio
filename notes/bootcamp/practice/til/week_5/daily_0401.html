    1. 캐시(Cache)란 무엇인가

    자주 사용하는 데이터를 더 빠른 저장소(메모리)에 임시로 저장해 재사용하는 기술이다. DB나 외부 API를 호출하지 않고 In-memory에서 즉시 응답할 수 있어 속도 향상, DB 부하 감소, 트래픽 절감 효과를 얻을 수 있다.

    캐시 기본 동작 구조 요청이 들어오면 캐시를 먼저 확인한다. 데이터가 있으면(Cache Hit) 즉시 응답하고, 없으면(Cache Miss) DB를 조회한 뒤 결과를 캐시에 저장하고 응답한다.

    처음 요청은 느리지만(Miss), 이후 요청은 빠르다(Hit).

    DB vs 캐시

    구분 DB 캐시
    저장 방식 디스크 기반, 영구 저장 메모리 기반, 임시 저장
    속도 상대적으로 느림 매우 빠름
    데이터 보장 항상 정확성과 정합성 유지 TTL 만료·재시작 시 소멸, 최신 보장 안 됨
    목적 데이터의 영구 보존 응답 속도 향상 및 DB 접근 횟수 감소

    캐시 유형

    구분 설명 예시 특징
    로컬 캐시 애플리케이션 내부 메모리에 저장 ConcurrentHashMap, Caffeine 빠르지만 서버 간 데이터 공유 불가
    분산 캐시 외부 서버에 저장해 여러 인스턴스가 공유 Redis, Memcached 상대적으로 느리지만 여러 서버에서 공유 가능

    2. 캐시 전략

    전략 설명 특징
    Write-through DB에 쓰는 동시에 캐시도 갱신 데이터 일관성 높음, 쓰기 속도 느림
    Write-back 캐시에 먼저 쓰고, 나중에 DB에 반영 쓰기 속도 빠름, 캐시 장애 시 데이터 유실 위험
    Cache-aside
    (Lazy Loading)    		 요청 시 캐시 확인 → 없으면 DB 조회 후 캐시 저장 가장 일반적인 방식. 첫 요청(Miss)은 느림
    Cache Invalidation 데이터 변경 시 캐시를 삭제하거나 갱신 데이터 일관성 유지의 핵심
    Cache-aside 전략 흐름 요청 → 캐시 확인 → [있음] Cache Hit → 즉시 응답
    요청 → 캐시 확인 → [없음] Cache Miss → DB 조회 → 캐시에 저장(TTL 설정) → 응답 반환

    단순하고 범용적이라 가장 많이 쓰이는 방식이다. 단, 처음 요청은 DB를 거치므로 느리다.

    3. 캐시 일관성 — TTL과 Eviction Policy

    캐시의 가장 큰 문제는 DB 값은 바뀌었는데 캐시에는 이전 값이 남아 있는 데이터 불일치다. TTL과 Eviction Policy를 활용해 이 문제를 관리한다.

    수단 설명
    TTL (Time-To-Live) 캐시 데이터가 유효한 시간이다. 만료 후 자동 삭제되어 다음 요청 시 DB에서 최신 값을 가져온다. 주기적으로 강제 최신화하는 효과가 있다.
    Eviction Policy
    (삭제 정책)    		 캐시가 꽉 찼을 때 어떤 데이터부터 제거할지 결정하는 정책이다.
    정책 설명
    LRU 최근 사용하지 않은 항목부터 제거한다 (Least Recently Used)
    LFU 사용 빈도가 낮은 항목부터 제거한다 (Least Frequently Used)
    FIFO 먼저 들어온 항목부터 제거한다 (First In First Out)
    ⚠️ 캐시 사용 시 주의사항 캐시는 항상 DB의 보조 수단이다. DB를 대체하는 것이 아니다.
    TTL이 너무 짧으면 캐시 효율이 떨어지고, 너무 길면 데이터 불일치가 발생한다.
    변경이 잦은 데이터보다 조회가 많은 데이터를 중심으로 적용해야 효과적이다.
    캐시 장애 시 DB로 fallback 처리가 반드시 필요하다.
    캐시 Key는 도메인명:ID 형식처럼 명확하게 설계해야 충돌을 방지할 수 있다.

    4. Spring Cache 추상화

    Spring이 제공하는 캐싱 추상화로, 캐시 로직을 직접 작성하지 않고 어노테이션 기반으로 처리할 수 있는 구조다. Caffeine, Redis 등 다양한 캐시 구현체를 같은 방식으로 제어할 수 있으며, 코드 변경 없이 로컬 캐시 ↔ Redis로 손쉽게 교체 가능하다. 

    @Cacheable / @CacheEvict / @CachePut
              ↓
   [ Cache Abstraction Layer ]
              ↓
          CacheManager
              ↓
     Cache (Caffeine, Redis 등)

    @Cacheable — 캐시 조회 및 저장

    메서드 결과를 캐시에 저장한다. 이후 같은 key로 호출하면 메서드를 실행하지 않고 캐시에서 바로 반환한다. value는 캐시 저장소의 이름(namespace), key는 캐시 안에서 데이터를 구분하는 고유 식별자다. 

    // value = 캐시 영역 이름, key = 캐시 안의 식별자 (SpEL 표현식 사용)
@Cacheable(value = "postCache", key = "#postId")
public Post getPost(Long postId) {
    return postRepository.findById(postId).orElseThrow();
}

// key 충돌 방지 — 같은 value 안에서 prefix로 구분
@Cacheable(value = "postCache", key = "'id:' + #postId")
@Cacheable(value = "postCache", key = "'username:' + #username")
    value와 key 설계 기준 value는 캐시 그룹(namespace)을 의미한다. "postCache"라는 하나의 영역 안에 여러 key-value 쌍이 저장된다.

    key는 캐시 안에서 데이터를 구분하는 고유 식별자다. 메서드 파라미터를 SpEL(#postId)로 참조한다.

    같은 value 안에 key가 중복되지 않도록 prefix를 붙여 구분하는 것이 안전하다. Redis에서는 user:*, post:*처럼 prefix로 분류 관리한다.

    @CacheEvict — 캐시 삭제

    데이터가 삭제될 때 캐시에서도 해당 항목을 제거한다. 캐시에 오래된 데이터가 남아 있는 일관성 문제를 방지한다. 

    @CacheEvict(value = "postCache", key = "#postId")
public void deletePost(Long postId) {
    postRepository.deleteById(postId);
}

// allEntries = true: value 안의 모든 캐시를 한 번에 삭제
@CacheEvict(value = "postCache", allEntries = true)
public void deleteAllPosts() {
    postRepository.deleteAll();
}

    @CachePut — 캐시 갱신

    데이터가 수정될 때 캐시도 함께 최신 값으로 갱신한다. @Cacheable과 달리 캐시에 데이터가 있어도 항상 메서드를 실행하고 결과로 캐시를 덮어쓴다. 

    @CachePut(value = "postCache", key = "#postId")
public Post updatePost(Long postId, PostUpdateRequest request) {
    Post post = postRepository.findById(postId).orElseThrow();
    post.update(request);
    return post; // 반환값이 캐시에 저장됨
}

    3가지 어노테이션 비교

    어노테이션 역할 메서드 실행 여부 사용 시점
    @Cacheable 캐시 조회 및 저장 Cache Hit 시 실행 안 함 조회 메서드
    @CacheEvict 캐시 삭제 항상 실행 삭제 메서드
    @CachePut 캐시 갱신 항상 실행 (결과를 캐시에 저장) 수정 메서드

    마무리

    캐시는 성능 최적화의 강력한 도구지만 항상 DB의 보조 수단이라는 원칙을 잊으면 안 된다. 캐시에 데이터가 있다고 해서 그것이 현재 DB와 일치한다는 보장이 없기 때문에, 데이터 일관성이 중요한 곳에서는 TTL과 Cache Invalidation 전략을 함께 설계해야 한다. TTL이 너무 길면 오래된 데이터가 제공되고, 너무 짧으면 캐시의 의미가 없어진다.

    Spring Cache 추상화의 가장 큰 장점은 구현체와 무관하게 동일한 어노테이션으로 제어할 수 있다는 것이다. 로컬 개발 환경에서는 Caffeine을 쓰다가 운영 환경에서 Redis로 전환할 때 비즈니스 코드를 전혀 수정하지 않아도 된다. @Cacheable은 Cache Hit 시 메서드 자체를 실행하지 않는다는 점, @CachePut은 항상 실행하고 결과를 캐시에 저장한다는 점의 차이를 명확히 이해해야 잘못 사용하는 실수를 피할 수 있다.

    캐시 Key 설계는 처음부터 신중하게 해야 한다. 같은 value 안에서 key가 충돌하면 전혀 다른 데이터가 반환되는 심각한 버그가 생긴다. 'id:' + #postId처럼 prefix를 붙이거나, Redis에서는 post:*처럼 네임스페이스를 명확히 분리하는 습관을 들여야 한다.
