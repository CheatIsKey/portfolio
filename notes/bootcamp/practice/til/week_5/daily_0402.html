    73일차 개요

    오늘 목표

    • Index의 개념과 역할, 적용 시 성능 차이를 이해한다.
    • B-Tree 구조가 인덱스에 사용되는 이유를 파악한다.
    • Index Scan 5종류의 동작 방식과 발생 조건을 정리한다.
    • 옵티마이저가 인덱스 탐색 방법을 자동으로 결정하는 원리를 이해한다.
    • EXPLAIN으로 실행 계획을 확인하는 방법을 익힌다.
    • 복합 인덱스와 Leftmost Prefix Rule을 이해한다.

    1. Index란 무엇인가

    데이터를 빠르게 찾기 위한 검색용 지도다. 단순히 가나다순으로 정렬된 목록이 아니라, 검색에 특화된 트리(Tree) 구조의 탐색 지도다. 테이블과 별도의 자료 구조로 관리되며, PRIMARY KEY와 UNIQUE 제약 조건이 걸린 컬럼에는 자동으로 생성된다.

    항목 내용
    기본 역할 원하는 데이터를 빠르게 찾도록 도와준다.
    저장 위치 테이블과 별도의 자료 구조로 관리된다.
    구조 형태 B-Tree 알고리즘을 사용한다.
    자동 생성 조건 PRIMARY KEY, UNIQUE 제약 조건이 걸린 컬럼

    인덱스 적용 여부 비교

    구분 인덱스 없음 인덱스 있음
    검색 방식 전체 테이블 탐색 (Full Table Scan) 정렬된 인덱스 탐색
    시간 복잡도 O(n) O(log n)
    I/O 비용 매우 높음 낮음
    추가 공간 없음 필요
    INSERT / UPDATE 빠름 약간 느림 (인덱스도 함께 갱신)

    2. B-Tree (Balanced Tree)

    데이터를 분할, 정렬, 균형 유지하는 탐색 트리다. 루트 노드를 기준으로 검색할 값이 더 크면 오른쪽, 더 작으면 왼쪽으로 이동한다. 항상 균형(Balanced)을 유지하기 때문에 어떤 데이터를 찾더라도 탐색 깊이가 일정하고 최악의 경우에도 O(log n)이 보장된다. 또한 리프 노드끼리 연결되어 있어 시작 위치를 찾은 뒤 옆으로 이어서 읽는 범위 검색에도 유리하다.

    3. Index Scan 종류

    1. Unique Scan

    정확히 한 건의 데이터를 찾는 탐색이다. 인덱스가 PRIMARY KEY 또는 UNIQUE 제약일 때 발생하며, B-Tree에서 정확히 한 경로만 따라가면 끝나기 때문에 탐색 속도가 가장 빠르다. 불필요한 비교나 순회가 전혀 없다.

    2. Range Scan

    일정 범위(구간)의 데이터를 연속적으로 읽는 탐색이다. B-Tree의 정렬된 구조를 활용해 시작 위치를 찾은 뒤 순서대로 이어서 읽는다. 이미 정렬된 상태로 읽히기 때문에 ORDER BY를 사용해도 추가 정렬이 필요 없다. BETWEEN, >, <, LIKE 'abc%' 등의 조건에서 발생한다.

    LIKE 조건에서 Range Scan이 가능한 경우 LIKE '김%'처럼 앞부분이 고정된 경우에는 인덱스 시작 위치를 특정할 수 있어 Range Scan이 가능하다.
    LIKE '%김'처럼 앞부분이 와일드카드인 경우에는 시작 위치를 특정할 수 없어 Full Table Scan이 발생한다.

    3. Full Index Scan

    인덱스 전체를 처음부터 끝까지 순서대로 읽는 탐색이다. WHERE 조건 없이 인덱스 컬럼 전체를 조회할 때 발생하며, 인덱스가 이미 정렬된 상태이므로 ORDER BY 시 추가 정렬이 필요 없다. Full Table Scan보다 읽는 데이터량이 적어 I/O 비용이 낮다.

    4. Index Condition Pushdown (ICP)

    WHERE 조건 중 일부를 인덱스 탐색 단계에서 미리 필터링하는 최적화 기법이다. 기존에는 인덱스로 후보 데이터를 찾은 뒤 테이블에서 실제 행을 읽어와 조건을 검사했다. ICP가 적용되면 조건을 만족하지 않는 행은 테이블 접근 자체를 하지 않으므로 불필요한 디스크 I/O가 줄어든다.

    5. Table Full Scan

    인덱스를 전혀 사용하지 않는 탐색이다. 테이블의 모든 데이터를 처음부터 끝까지 확인하므로 O(n)의 시간 복잡도를 가진다. 인덱스가 없거나 옵티마이저가 인덱스보다 Full Scan이 더 유리하다고 판단할 때 발생한다.

    5가지 Index Scan 요약 비교

    종류 발생 조건 특징
    Unique Scan PK 또는 UNIQUE 컬럼 조건 가장 빠름. 정확히 1건, 추가 순회 없음
    Range Scan BETWEEN, >, <, LIKE 'abc%' 정렬된 순서로 읽어 ORDER BY 효율적
    Full Index Scan WHERE 없이 인덱스 컬럼 전체 조회 Full Table Scan보다 I/O 적음
    ICP 복합 인덱스 + 추가 WHERE 조건 테이블 접근 횟수 감소 → I/O 절감
    Table Full Scan 인덱스 없음 또는 옵티마이저 판단 모든 행을 순회. O(n), 가장 느림

    4. 옵티마이저 (Optimizer)

    DB가 스스로 가장 효율적인 실행 방법을 고르는 두뇌다. 인덱스를 만들어 두면 어떤 Index Scan(Unique, Range 등)을 쓸지 DB가 자동으로 결정한다. 개발자는 인덱스만 잘 설계하면 된다.

    EXPLAIN으로 실행 계획 확인하기 EXPLAIN을 쿼리 앞에 붙이면 옵티마이저가 해당 SQL을 어떻게 실행할지 확인할 수 있다.

    type 필드 — 어떤 Index Scan을 사용하는지 나타낸다. const는 Unique Scan, range는 Range Scan을 의미한다.
    key 필드 — 어떤 인덱스 키를 사용하는지 나타낸다. 
    -- Unique Scan 확인 — type: const, key: PRIMARY
EXPLAIN FORMAT=TRADITIONAL SELECT * FROM users WHERE id = 1;

-- Range Scan 확인 — type: range, key: idx_age
-- B-Tree에서 20의 위치를 찾고, 그 이후 데이터를 순서대로 탐색한다.
EXPLAIN FORMAT=TRADITIONAL SELECT * FROM users WHERE age BETWEEN 20 AND 30;

    5. Index의 한계

    한계 이유
    자주 갱신되는 컬럼에는 비효율적 삽입·삭제마다 트리 구조를 재정렬해야 한다 (균형 재조정 비용).
    너무 많은 인덱스는 오히려 성능 저하 쓰기 시 모든 인덱스를 함께 업데이트해야 하므로 쓰기 성능이 저하된다.
    메모리 공간 소모 인덱스도 별도의 파일 구조로 저장되며 수 GB 이상 차지할 수 있다.
    ⚠️ 인덱스는 공짜가 아니다 인덱스를 추가할수록 조회는 빨라지지만 INSERT / UPDATE / DELETE 성능은 저하되고 저장 공간도 늘어난다. 조회 빈도가 높고 카디널리티(고유값 수)가 높은 컬럼에 선택적으로 적용해야 한다. 모든 컬럼에 인덱스를 추가하는 것은 오히려 역효과다.

    6. 복합 인덱스 (Composite Index)

    두 개 이상의 컬럼을 묶어서 하나의 인덱스로 만드는 것이다. 두 개 이상의 컬럼이 항상 함께 WHERE이나 ORDER BY에 등장할 때, 단일 인덱스 여러 개보다 복합 인덱스 하나가 더 빠르고 효율적이다.

    왼쪽 접두어 규칙 (Leftmost Prefix Rule) 복합 인덱스는 왼쪽(첫 번째 컬럼)부터 순서대로 작동한다.

    첫 번째 컬럼이 WHERE 조건에 없으면 인덱스 자체가 적용되지 않는다.

    예를 들어 (col1, col2, col3) 복합 인덱스가 있을 때:
    — WHERE에 col1, col2, col3이 모두 있으면 → 세 컬럼 모두 인덱스 활용
    — WHERE에 col1, col3만 있으면 → col2가 빠졌으므로 col3은 인덱스 없는 것처럼 동작 (col1만 활용)
    — WHERE에 col2, col3만 있으면 → 첫 번째 컬럼(col1) 없으므로 인덱스 미적용

    첫 번째 컬럼은 항상 WHERE 조건에 포함되어야 인덱스가 동작한다. 
    -- (last_name, first_name) 복합 인덱스 생성
CREATE INDEX idx_name ON users (last_name, first_name);

-- ✅ 인덱스 활용 — last_name(첫 번째)이 WHERE에 포함
SELECT * FROM users WHERE last_name = '김' AND first_name = '민준';
SELECT * FROM users WHERE last_name = '김';

-- ⚠️ 인덱스 부분 활용 — last_name만 인덱스 적용, first_name은 미적용
SELECT * FROM users WHERE last_name = '김' AND first_name LIKE '민%';

-- ❌ 인덱스 미적용 — 첫 번째 컬럼(last_name) 없음
SELECT * FROM users WHERE first_name = '민준';

    마무리

    인덱스는 데이터베이스 성능 최적화의 핵심 도구다. 하지만 인덱스를 추가하는 것 자체가 목적이 되어선 안 된다. 조회 빈도, 카디널리티, 갱신 빈도를 함께 고려해서 어떤 컬럼에, 어떤 순서로 인덱스를 걸지를 결정해야 한다. B-Tree 구조를 이해하면 Range Scan의 효율, LIKE 조건의 제한, Full Scan이 발생하는 이유가 모두 자연스럽게 연결된다.

    옵티마이저가 자동으로 실행 계획을 선택한다고 해도 개발자가 EXPLAIN으로 실제로 어떤 인덱스를 사용하는지 확인하는 습관이 중요하다. 예상과 다른 탐색 방식이 선택됐다면 인덱스 설계를 재검토해야 한다는 신호다. type: const나 type: range가 보이면 인덱스가 잘 활용되고 있는 것이고, type: ALL이 보이면 Full Table Scan이 발생하고 있다는 의미다.

    복합 인덱스는 Leftmost Prefix Rule을 이해하지 못하면 인덱스를 만들어 놓고도 효과를 전혀 보지 못하는 상황이 생긴다. 첫 번째 컬럼이 항상 WHERE에 포함되어야 한다는 원칙, 중간 컬럼이 빠지면 그 이후 컬럼은 인덱스 효과가 없다는 점을 기억하고 컬럼 순서를 신중하게 결정해야 한다.
