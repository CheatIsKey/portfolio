    77일차 개요

    오늘 목표

    • Docker가 배포 전용 도구가 아닌 개발 환경에서도 적극 활용해야 하는 이유를 이해한다.
    • Image와 Container의 차이와 관계를 명확히 파악한다.
    • Docker Hub에서 이미지를 검색하고 실행하는 흐름을 익힌다.
    • 주요 Docker 명령어를 정리한다.
    • Docker Compose로 여러 컨테이너를 한 번에 실행하는 방법을 이해한다.
    • Docker Network로 컨테이너 간 통신을 연결하는 방식을 파악한다.

    1. Docker는 배포 전용이 아니다

    ⚠️ 잘못된 인식 — "Docker는 배포할 때만 쓰는 것" Docker는 배포 시점뿐만 아니라 개발 환경 구성에도 필수적으로 활용해야 한다.

    MySQL, Redis를 로컬에 직접 설치하면 OS마다 설치 방법이 다르고, 버전 충돌이 생기며, 팀원 간 환경이 달라진다. Docker를 쓰면 명령어 한 줄로 동일한 버전의 DB를 어디서든 동일하게 실행할 수 있다. 개발부터 Docker를 활용하는 것이 올바른 방법이다.

    2. Image와 Container

    Image — 실행 환경의 설계도

    애플리케이션을 실행하는 데 필요한 운영체제, 라이브러리, 코드, 설정이 모두 들어있는 불변 패키지다. "이 환경에서 앱을 이렇게 실행하라"는 정보를 담고 있으며 읽기 전용이다. 하나의 이미지로 여러 개의 컨테이너를 동시에 생성할 수 있다.

    Container — 이미지를 실행한 인스턴스

    이미지를 실제로 실행한 결과물, 즉 동작 중인 인스턴스다. 운영체제를 새로 띄우지 않고 호스트 OS의 커널을 공유하기 때문에 가볍고 빠르게 실행된다. 컨테이너는 독립된 공간에서 실행되어 서로 간섭하지 않으며, 필요할 때 만들고 언제든지 지울 수 있다.

    Image vs Container 비유 Image는 집을 짓기 위한 설계도(Blueprint)다. 설계도 자체는 변하지 않는다.
    Container는 그 설계도로 실제로 지은 집이다. 같은 설계도로 여러 채의 집을 지을 수 있다.

    이미지는 읽기 전용이며, 컨테이너는 실행 중에 상태를 가진다. 컨테이너를 삭제해도 이미지는 그대로 남아 있어 언제든지 새 컨테이너를 만들 수 있다.
    구분 Image Container
    성격 읽기 전용 설계도 실행 중인 인스턴스
    상태 불변 (Immutable) 가변 (실행 중 상태 존재)
    관계 컨테이너 생성의 기반 이미지로부터 생성됨
    재사용 하나의 이미지로 여러 컨테이너 생성 가능 삭제하면 사라짐

    3. Docker Hub와 주요 명령어

    Docker Hub는 전 세계 개발자들이 만든 이미지를 검색하고 내려받을 수 있는 이미지 저장소다. MySQL, Redis, Nginx 같은 대부분의 오픈소스 소프트웨어 공식 이미지가 Docker Hub에 올라와 있어 직접 설치 없이 바로 사용할 수 있다.

    Docker Hub 사용 흐름 search → Docker Hub에서 원하는 이미지 검색
    pull → 이미지를 로컬로 내려받기
    run → 이미지를 실행해 컨테이너 생성
    명령어 설명
    docker run -d -p 6379:6379 --name <컨테이너명> <이미지명> 컨테이너를 백그라운드(-d)로 생성 및 실행. -p로 호스트 포트와 컨테이너 포트를 연결
    docker ps 실행 중인 컨테이너 목록 확인. STATUS가 Up이면 실행 중
    docker exec -it <CONTAINER ID> bash 실행 중인 컨테이너 내부에 접속. -it는 인터랙티브 터미널 모드
    redis-cli Redis 컨테이너 내부에서 Redis CLI 실행. Redis 명령어를 직접 입력할 수 있음
    docker stop <컨테이너명 또는 ID> 실행 중인 컨테이너 중지
    docker rm <컨테이너명 또는 ID> 컨테이너 삭제 (중지 후 삭제) 
    # Redis 컨테이너 실행 예시
docker run -d -p 6379:6379 --name my-redis redis

# 실행 중인 컨테이너 확인
docker ps

# Redis 컨테이너 내부 접속
docker exec -it my-redis bash

# Redis CLI 실행 (컨테이너 내부에서)
redis-cli

    4. Docker Compose

    여러 컨테이너(Spring, MySQL, Redis 등)를 하나의 파일에 정의하고 한 번에 실행할 수 있는 도구다. 개발 환경에서 Spring Boot 앱과 MySQL, Redis를 각각 docker run으로 따로 실행하는 대신, docker-compose.yml 한 파일로 전체 스택을 관리할 수 있다. 

    # Compose 파일에 정의된 모든 컨테이너를 백그라운드로 실행
docker compose up -d

# 모든 컨테이너 중지 및 제거
docker compose down

# 로그 확인
docker compose logs -f
    # docker-compose.yml 예시
version: "3.8"
services:
  mysql:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: password
      MYSQL_DATABASE: myapp
    ports:
      - "3306:3306"

  redis:
    image: redis:7
    ports:
      - "6379:6379"

  app:
    build: .
    ports:
      - "8080:8080"
    depends_on:
      - mysql
      - redis

    5. Docker Network

    Docker 컨테이너는 기본적으로 독립적으로 실행되어 서로를 알지 못한다. Spring Boot 컨테이너가 MySQL 컨테이너에 접근하려면 두 컨테이너를 같은 네트워크에 연결해야 한다.

    Docker Compose와 Network의 관계 docker-compose.yml에 정의된 컨테이너들은 자동으로 같은 네트워크에 배치된다. 같은 Compose 파일 안의 컨테이너끼리는 서비스 이름을 호스트명으로 사용해 통신할 수 있다.

    예를 들어 Compose 파일에서 MySQL의 서비스 이름이 mysql이라면, Spring Boot 컨테이너의 application.yml에서 spring.datasource.url: jdbc:mysql://mysql:3306/myapp처럼 서비스 이름을 그대로 호스트명으로 사용하면 된다. 
    # application.yml — Docker Compose 환경에서의 설정
spring:
  datasource:
    # 수정 포인트: 로컬 직접 실행 시 localhost, Compose 환경에서는 서비스 이름(mysql)
    url: jdbc:mysql://${MYSQL_HOST:mysql}:3306/myapp
  data:
    redis:
      # 수정 포인트: 로컬 직접 실행 시 localhost, Compose 환경에서는 서비스 이름(redis)
      host: ${SPRING_DATA_REDIS_HOST:redis}
      port: 6379
    상황 접속 호스트
    로컬에서 직접 실행 (Docker 없음) localhost
    컨테이너에서 호스트 머신의 서비스에 접근 host.docker.internal
    Compose 환경에서 같은 네트워크의 컨테이너에 접근 서비스 이름 (mysql, redis)

    마무리

    Docker를 배포 전용 도구로만 알고 있었다면 개발 환경에서의 큰 이점을 놓치고 있는 것이다. MySQL, Redis를 로컬에 직접 설치하면 팀원마다 버전이 다르고 설정이 달라 "내 환경에서는 되는데"라는 상황이 반복된다. Docker를 사용하면 docker compose up -d 한 줄로 모든 팀원이 동일한 환경을 즉시 구성할 수 있다.

    Image와 Container의 관계를 명확히 이해하는 것이 중요하다. 이미지는 불변하는 설계도이고, 컨테이너는 그 설계도로 만든 실행 인스턴스다. 컨테이너를 아무리 삭제하고 다시 만들어도 이미지가 있으면 동일한 환경을 재현할 수 있다는 것이 Docker의 핵심 가치다.

    Docker Compose에서 컨테이너끼리 서비스 이름으로 통신한다는 점도 중요하다. localhost는 컨테이너 자기 자신을 가리키기 때문에 다른 컨테이너에 접근할 수 없다. Compose 환경에서는 서비스 이름을 호스트명으로 사용해야 하며, 환경변수로 override할 수 있도록 설정하면 로컬 직접 실행과 Compose 실행 모두를 하나의 설정 파일로 처리할 수 있다.
