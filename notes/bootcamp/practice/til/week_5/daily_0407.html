    LV 1 — 필수 기능

    ① @Transactional 읽기 전용 오류 수정

    POST /todos 호출 시 Connection is read-only. Queries leading to data modification are not allowed 에러가 발생했다. TodoService 클래스 레벨에 @Transactional(readOnly = true)가 선언되어 있어, 데이터를 쓰는 saveTodo() 메서드도 읽기 전용 트랜잭션으로 실행된 것이 원인이다. 

    // 클래스 레벨: @Transactional(readOnly = true)

@Transactional  // 수정 포인트: 메서드 레벨이 클래스 레벨을 오버라이드 → 쓰기 가능
public TodoSaveResponse saveTodo(AuthUser authUser, TodoSaveRequest request) { ... }
    readOnly = true의 올바른 사용 방법 readOnly = true는 SELECT 쿼리에 대한 성능 최적화(스냅샷 생략 등)에 유용하다. 클래스 레벨에 선언하면 해당 클래스의 모든 메서드에 적용된다. INSERT / UPDATE / DELETE가 필요한 메서드에는 반드시 @Transactional을 메서드 레벨에 별도로 선언해야 한다. 메서드 레벨 어노테이션이 클래스 레벨을 오버라이드한다.

mockMvc.perform(get("/todos/{todoId}", 1L))
    .andExpect(status().isBadRequest());  // 수정 포인트: 실제 응답 코드와 일치하도록 수정

    ⑤ AOP 어드바이스 실행 시점 버그 수정

    UserAdminController.changeUserRole() 실행 전에 로그를 남겨야 하는데, AdminAccessLoggingAspect의 포인트컷이 UserController.getUser()를 대상으로 잘못 지정되어 있었다. 포인트컷 대상을 올바른 메서드로 변경했다. 

    // 수정 전 — 잘못된 대상
@Before("execution(* org.example.expert.domain.user.controller.UserController.getUser(..))")

// 수정 후 — 올바른 대상
@Before("execution(* org.example.expert.domain.user.controller.UserAdminController.changeUserRole(..))")
public void logBeforeChangeUserRole(JoinPoint joinPoint) { ... }
    AOP 어드바이스 종류와 실행 시점 @Before: 메서드 실행 전 — 접근 로그, 권한 검증 등에 사용
    @After: 메서드 실행 후 (성공/실패 무관) — 자원 정리 등에 사용
    @Around: 메서드 실행 전후 모두 제어 — 실행 시간 측정, 트랜잭션 관리 등에 사용

    이 요구사항은 "실행 전 로깅"이므로 @Before가 정확하다.

    LV 2 — 필수 기능

    ⑥ JPA Cascade — 담당자 자동 등록

    Todo 엔티티의 managers 컬렉션에 CascadeType.PERSIST를 적용했다. Todo를 저장(persist)할 때 Manager도 자동으로 함께 INSERT된다. 할 일 생성자가 자동으로 첫 번째 담당자로 등록된다. 

    // Todo.java
@OneToMany(mappedBy = "todo", cascade = CascadeType.PERSIST)
private List<Manager> managers = new ArrayList<>();

// 수정 포인트: 생성자에서 할 일 생성자를 담당자 목록에 추가
public Todo(String title, String contents, String weather, User user) {
    this.user = user;
    this.managers.add(new Manager(user, this)); // PERSIST cascade로 자동 저장됨
}
    CascadeType.PERSIST만 적용한 이유 PERSIST만 적용했으므로 Todo 삭제 시 Manager가 연쇄 삭제되지 않는다. 삭제 연쇄는 CascadeType.REMOVE 또는 orphanRemoval = true로 별도로 제어한다. 필요한 전파 타입만 명시적으로 지정해 의도치 않은 삭제를 방지하는 것이 안전하다.

    ⑦ N+1 문제 해결 — JPQL Fetch Join

    GET /todos/{todoId}/comments 호출 시 댓글 수(N)만큼 유저 정보를 개별 조회하는 N+1 문제가 있었다. 댓글이 100개면 쿼리가 101번 실행된다. JPQL에 JOIN FETCH를 추가해 댓글과 유저를 한 번의 쿼리로 가져오도록 수정했다. 

    // 수정 전: N+1 발생
List<Comment> findByTodoId(Long todoId); // 댓글 조회 후 유저마다 추가 쿼리

// 수정 후: JOIN FETCH로 한 번에 조회
@Query("SELECT c FROM Comment c JOIN FETCH c.user WHERE c.todo.id = :todoId")
List<Comment> findByTodoIdWithUser(@Param("todoId") Long todoId);

    ⑧ JPQL → QueryDSL 전환

    문자열 기반의 JPQL을 타입 안전한 QueryDSL로 전환했다. 오타나 필드 변경 시 컴파일 타임에 오류를 잡을 수 있어 유지보수성이 크게 향상된다. 

    // 기존 JPQL (문자열 기반)
// @Query("SELECT t FROM Todo t LEFT JOIN t.user WHERE t.id = :todoId")

// 변경 후 QueryDSL (타입 안전)
@Override
public Optional<Todo> findByIdWithUser(Long todoId) {
    return Optional.ofNullable(
        queryFactory.selectFrom(todo)
            .leftJoin(todo.user).fetchJoin()  // N+1 방지 + 타입 안전
            .where(todo.id.eq(todoId))
            .fetchOne()
    );
}
    구분 JPQL QueryDSL
    오류 감지 런타임 컴파일 타임
    타입 안전성 문자열 기반 Q클래스 기반
    동적 쿼리 복잡 (문자열 조합) BooleanExpression으로 간결

    ⑨ Filter/Resolver → Spring Security 전환

    기존의 커스텀 AuthFilter + ArgumentResolver 방식을 Spring Security로 전환했다. 인증(Authentication)과 인가(Authorization)를 프레임워크 레벨에서 통합하고, 경로별 권한 규칙을 SecurityConfig 한 곳에서 선언적으로 관리할 수 있게 됐다. 

    .sessionManagement(session ->
    session.sessionCreationPolicy(SessionCreationPolicy.STATELESS) // JWT → 세션 미사용
)
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/auth/**", "/actuator/**").permitAll()        // 인증 없이 허용
    .requestMatchers("/admin/**").hasAuthority("ROLE_ADMIN")        // 관리자만 허용
    .anyRequest().authenticated()                                   // 나머지는 인증 필수
)
.addFilterBefore(
    new JwtAuthenticationFilter(provider),
    UsernamePasswordAuthenticationFilter.class
);

    ⑩ QueryDSL 고급 검색 API — Projections + 페이징

    GET /todos/detail 엔드포인트에서 제목, 생성일, 담당자 닉네임을 조합한 고급 검색을 구현했다. Projections를 사용해 엔티티 전체가 아닌 필요한 필드만 반환하며, 모든 검색 조건은 선택적으로 적용된다.

    파라미터 설명
    titleKeyword제목 부분 검색 (대소문자 무시)
    startDate / endDate생성일 기간 검색
    nicknameKeyword담당자 닉네임 부분 검색
     
    queryFactory
    .select(Projections.constructor(TodoDetailResponse.class,
        todo.title,
        manager.countDistinct(),  // 담당자 수
        comment.countDistinct()   // 댓글 수
    ))
    .from(todo)
    .leftJoin(todo.managers, manager)
    .leftJoin(manager.user, user)
    .leftJoin(todo.comments, comment)
    .where(
        titleKeywordCondition(condition.titleKeyword()),
        startDateCondition(condition.startDate()),
        endDateCondition(condition.endDate()),
        managerKeywordCondition(condition.nicknameKeyword())
    )
    .groupBy(todo.id, todo.title, todo.createdAt)
    .orderBy(todo.createdAt.desc())
    .fetch();

    ⑪ Transaction 심화 — 매니저 등록 로그 독립 트랜잭션

    매니저 등록이 실패하더라도 log 테이블에는 반드시 요청 기록이 남아야 한다는 요구사항이 있었다. 같은 트랜잭션 안에 로그를 넣으면 매니저 등록 실패 시 로그도 함께 롤백된다. REQUIRES_NEW로 로그 저장을 별도 트랜잭션으로 분리해서 해결했다. 

    // LogService — 호출 측 트랜잭션과 완전히 독립된 새 트랜잭션으로 실행
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void save(Long todoId, Long requestUserId, Status result, String failReason) {
    logRepository.save(new Log(todoId, requestUserId, result, failReason));
}
    REQUIRES_NEW 동작 흐름 [ManagerService — 트랜잭션 A 시작]
      ├── 매니저 등록 로직 실행
      ├── LogService.save() 호출
      │      └── [트랜잭션 B 독립 시작 → 즉시 커밋] ← REQUIRES_NEW
      ├── 등록 성공 → 트랜잭션 A 커밋
      └── (실패 시) 트랜잭션 A 롤백
                   → 트랜잭션 B(로그)는 이미 커밋 완료

    ⑫ AWS 클라우드 배포 — EC2 · RDS · S3

    EC2: Amazon Linux 2023 인스턴스에 Docker로 애플리케이션을 실행했다. 탄력적 IP를 할당해 재시작 후에도 주소가 유지된다.

    RDS: MySQL 8.0 RDS 인스턴스를 생성했다. 보안 그룹 인바운드는 EC2 보안 그룹 ID만 허용해 외부 직접 접근을 차단했다. DB 접속 정보는 환경변수(DB_URL, DB_USERNAME, PASSWORD)로 주입한다.

    S3: S3 버킷을 생성하고(퍼블릭 액세스 차단) 프로필 이미지 업로드에 사용했다. 이미지를 직접 노출하지 않고 Presigned URL(1시간 유효)을 발급해 보안을 유지한다.

    ⑬ 대용량 데이터 처리 — 500만 건 닉네임 검색 최적화

    JPA save() 반복 호출은 건당 INSERT로 500만 건에서 매우 느리다. JdbcTemplate.batchUpdate()로 한 번에 대량 INSERT하도록 구현했다. 

    // DataInitializer.java — 500만 건 Bulk Insert
jdbcTemplate.batchUpdate(
    "INSERT INTO users (nickname, email, password, user_role) VALUES (?, ?, ?, ?)",
    new BatchPreparedStatementSetter() {
        @Override
        public void setValues(PreparedStatement ps, int i) throws SQLException {
            ps.setString(1, UUID.randomUUID().toString().substring(0, 8)); // 랜덤 닉네임
            ps.setString(2, "user" + i + "@test.com");
            ps.setString(3, encodedPassword);
            ps.setString(4, "USER");
        }
        @Override
        public int getBatchSize() { return 5_000_000; }
    }
);

    닉네임 완전 일치 검색(GET /users?nickname=xxx)의 성능을 3단계로 최적화했다.

    단계 방법
    1단계 인덱스 없음 → Full Table Scan (가장 느림)
    2단계 nickname 컬럼 인덱스 추가 → B-Tree 탐색 (대폭 향상)
    3단계 Redis 캐시 적용 (@Cacheable, TTL 10분) → 두 번째 요청부터 DB 미조회 
    // 인덱스 정의 — User 엔티티
@Table(name = "users", indexes = {
    @Index(name = "idx_user_nickname", columnList = "nickname")
})

// Redis 캐시 적용 — 동일 닉네임 두 번째 요청부터 DB 조회 없이 반환
@Cacheable(value = "users", key = "'nickname:' + #nickname")
public List<UserPageResponse> getUserByNickname(String nickname) { ... }

// RedisConfig — TTL 10분, JSON 직렬화
RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
    .entryTtl(Duration.ofMinutes(10));

    기술적 의사결정 — 왜 이 기술을 선택했나

    결정 선택 이유
    Spring Security 도입 기존 커스텀 Filter는 JWT 검증만 처리하고 권한 체크는 서비스 코드 곳곳에 흩어져 있었다. Spring Security는 인증과 인가를 프레임워크 레벨에서 통합하고, 경로별 권한 규칙을 SecurityConfig 한 곳에서 선언적으로 관리할 수 있어 유지보수성이 높아진다.
    QueryDSL 선택 JPQL은 문자열 기반이라 오타가 있어도 런타임에야 오류를 알 수 있다. QueryDSL은 Q클래스로 IDE 자동완성과 컴파일 검증이 가능하다. 제목·기간·닉네임을 조합하는 고급 검색처럼 조건 경우의 수가 많을수록 BooleanExpression의 장점이 극대화된다.
    REQUIRES_NEW로 로그 분리 같은 트랜잭션 안에 로그가 있으면 비즈니스 로직 실패 시 로그도 함께 롤백된다. REQUIRES_NEW로 분리하면 성공·실패 여부와 무관하게 로그가 항상 커밋된다. 감사(audit) 목적의 로그는 비즈니스 트랜잭션과 독립되어야 한다.
    Redis 캐시로 검색 최적화 DB 인덱스만으로도 속도가 크게 개선되지만, 동일한 닉네임 검색이 반복되면 매번 DB를 조회한다. @Cacheable로 결과를 Redis에 캐싱하면 두 번째 요청부터 DB를 거치지 않는다. TTL 10분으로 설정해 데이터 정합성과 성능 사이의 균형을 맞췄다.

    마무리

    이번 과제는 "이미 동작하는 코드를 더 잘 만드는" 연습이었다. 버그를 발견하고 수정하는 것부터 시작해서, 성능 최적화, 보안 강화, 대용량 처리, 클라우드 배포까지 단계적으로 진행했다. 가장 인상 깊었던 것은 같은 기능이라도 구현 방법에 따라 성능이 얼마나 달라지는지를 직접 수치로 확인한 부분이다. 500만 건 데이터에서 인덱스 유무, Redis 캐시 적용 여부가 체감할 수 있는 차이를 만들어냈다.

    REQUIRES_NEW로 로그 트랜잭션을 분리한 것은 단순한 기술 적용이 아니라, "비즈니스 실패와 무관하게 기록은 남아야 한다"는 설계 철학의 반영이다. 감사 로그는 시스템의 신뢰성을 보여주는 증거이므로 비즈니스 트랜잭션에 종속되면 안 된다. Spring Security로 전환한 것도 마찬가지다. 흩어진 권한 체크를 한 곳에 집중시키는 것이 단순히 코드를 줄이는 것이 아니라 변경이 발생했을 때 수정해야 할 곳을 하나로 만든다는 점에서 의미가 있다.
