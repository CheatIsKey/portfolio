    1. Redis란 무엇인가

    Remote Dictionary Server의 약자로, 데이터를 메모리(RAM)에 저장하는 초고속 Key-Value 데이터베이스다. 모든 데이터를 메모리에서 읽고 쓰기 때문에 디스크 기반 DB보다 속도가 훨씬 빠르다.

    구분 관계형 데이터베이스 (RDB) Key-Value DB (Redis)
    구조 테이블(행·열) 기반 Key와 Value 한 쌍으로 저장
    데이터 접근 SQL로 복잡한 조건 검색 가능 Key로 바로 값 조회 (단순하고 빠름)
    저장 위치 디스크 (영구 저장) 메모리 (빠르지만 휘발성)
    주요 용도 영구 데이터 저장, 관계형 모델 캐시, 세션, 실시간 데이터 처리
    예시 MySQL, PostgreSQL Redis, Memcached
    Redis가 빠른 이유 모든 데이터를 메모리(RAM)에 저장하기 때문에 디스크(SSD/HDD)를 거치는 I/O가 없다. CPU가 메모리에 직접 접근하므로 응답 속도가 μs(마이크로초) 단위다.

    디스크 기반 DB는 데이터를 읽기 위해 파일 입출력이 필요해 ms(밀리초) 단위의 지연이 생긴다. 메모리와 디스크의 속도 차이가 Redis 성능 차이의 핵심 원인이다.

    단, 서버가 꺼지면 메모리의 데이터가 사라지므로 영속성 옵션을 별도로 설정해야 한다.

    2. Redis 자료형

    Redis는 단순한 문자열 저장을 넘어 다양한 자료구조를 Value로 지원한다. 상황에 맞는 자료형을 선택하면 복잡한 로직을 Redis 명령어 하나로 처리할 수 있다.

    타입 설명 예시
    String 단일 값 저장. 가장 기본적인 타입 "user:1" → "Ravi"
    List 순서 있는 데이터. 큐/스택 구현에 유용 "queue" → [A, B, C]
    Set 중복 없는 집합 "tags" → {spring, redis}
    Hash 하나의 Key에 여러 필드-값 쌍 저장 (Map 구조) "user:1" → {name: Ravi, age: 30}
    Sorted Set 점수(score) 기반 자동 정렬된 집합. 랭킹 구현에 적합 "ranking" → {Ravi:100, Kim:200}

    데이터 영속성 옵션

    모드 설명 특징
    RDB
    (Redis Database Snapshot)    		 일정 주기마다 전체 데이터를 스냅샷으로 저장 빠름, 주기적 백업
    AOF
    (Append Only File)    		 실행된 명령어를 순차적으로 기록 높은 안정성, 장애 후 복구 가능
    Hybrid RDB + AOF 혼합 방식 Redis 7.x 이상 기본 설정

    3. RedisTemplate

    Spring이 제공하는 Redis용 클라이언트다. Java 코드로 Redis 명령어를 대신 실행해주는 도구로, 자료형별 opsFor~() 메서드로 해당 Redis 명령어를 추상화해서 사용한다.

    메서드 역할 대응 Redis 명령
    opsForValue() 단일 값 저장/조회 SET / GET
    opsForHash() 해시 구조 저장 HSET / HGET
    opsForList() 리스트 형태 저장 LPUSH / LRANGE
    opsForSet() 중복 없는 집합 SADD / SMEMBERS
    opsForZSet() 순위 정렬 집합 ZADD / ZRANGE 
    // TTL 설정 예시 — (key, value, 기간, 단위)
redisTemplate.opsForValue().set("temp:data", "123", 10, TimeUnit.MINUTES);

// 단일 값 저장 및 조회
redisTemplate.opsForValue().set("user:1", "Ravi");
String value = (String) redisTemplate.opsForValue().get("user:1");

// Key 삭제
redisTemplate.delete("user:1");

    4. Cache-Aside 패턴 구현

    데이터를 먼저 캐시에서 찾고, 없으면 DB에서 가져와 캐시에 저장하는 패턴이다. 가장 범용적으로 사용되는 캐싱 전략이다. 

    @Service
@RequiredArgsConstructor
public class PostService {

    private static final String PREFIX = "post:";
    private final PostRepository postRepository;
    private final RedisTemplate<String, Object> redisTemplate;

    public Post getPost(Long postId) {
        String key = PREFIX + postId; // 수정 포인트: 캐시 Key 네이밍 규칙 참고

        // (1) 캐시 확인 — Cache Hit 시 DB 조회 없이 즉시 반환
        Post cached = (Post) redisTemplate.opsForValue().get(key);
        if (cached != null) {
            return cached;
        }

        // (2) Cache Miss — DB 조회
        Post post = postRepository.findById(postId)
                .orElseThrow(() -> new RuntimeException("Post not found"));

        // (3) 캐시에 저장 (TTL 설정으로 일관성 관리)
        redisTemplate.opsForValue().set(key, post, 10, TimeUnit.MINUTES);
        return post;
    }

    // 데이터 수정 시 캐시 무효화 (Cache Invalidation)
    public void updatePost(Long postId, PostUpdateRequest request) {
        postRepository.findById(postId).orElseThrow().update(request);
        redisTemplate.delete(PREFIX + postId); // 수정 포인트: 캐시 삭제 후 다음 조회 시 DB에서 최신 값 로딩
    }
}

    5. 캐시 Key 네이밍 전략과 TTL 설계

    캐시 Key 네이밍 전략

    캐시 Key는 {도메인}:{리소스}:{식별자} 형식을 사용한다. Redis에서는 콜론(:)이 계층 구분자 역할을 하며, Redis GUI 도구에서 트리 구조로 시각화된다. 

    // 수정 포인트: Key 값은 상수로 선언해서 오타와 중복을 방지한다
private static final String POST_PREFIX   = "post:";
private static final String USER_PREFIX   = "user:";
private static final String SEARCH_PREFIX = "search:";

// 예시
"post:42"              // 단일 게시글
"user:7"               // 단일 유저
"search:spring:1"      // 검색 결과 (키워드 + 페이지)

    TTL 설계 원칙

    상황 TTL 권장
    변경이 거의 없는 데이터 (공지사항, 카테고리 등) 길게 설정 (수 시간 ~ 1일)
    조회가 잦고 변경이 적은 데이터 중간 (수십 분)
    변경이 잦은 데이터 짧게 설정하거나 캐시 적용 재고
    ⚠️ TTL 설정 시 주의사항 TTL이 너무 짧으면 캐시 히트율이 낮아져 캐시 효과가 없다.
    TTL이 너무 길면 DB 데이터가 바뀌어도 캐시에는 오래된 값이 남아 데이터 불일치가 발생한다.
    데이터 변경 시 반드시 캐시도 삭제(Evict)하거나 갱신(Put)해야 일관성을 유지할 수 있다.

    6. Lettuce와 커넥션 풀

    Lettuce는 Java 애플리케이션과 Redis 서버 사이를 연결해주는 클라이언트 라이브러리다. Spring Data Redis에서 기본으로 사용하는 Redis 클라이언트이다.

    커넥션 풀(Connection Pool)이 필요한 이유 Redis에 요청할 때마다 커넥션을 새로 생성하고 요청이 끝나면 소멸시키는 방식은 비효율적이다. 커넥션 생성 자체에 시간이 걸리고, 트래픽이 높을 때는 생성/소멸을 반복하다가 Redis 서버가 버티지 못한다.

    커넥션 풀은 미리 일정 개수의 커넥션을 생성해두고, 요청이 들어오면 풀에서 꺼내주고 사용이 끝나면 반납하는 방식이다. 생성/소멸 비용 없이 커넥션을 재사용하므로 성능이 안정된다. 
    spring:
  data:
    redis:
      # 수정 포인트: Docker Compose 환경에서는 서비스 이름 'redis'가 호스트명으로 사용됨
      # 환경변수 SPRING_DATA_REDIS_HOST로 override 가능
      host: ${SPRING_DATA_REDIS_HOST:redis}
      port: 6379
      lettuce:
        pool:
          max-active: 8    # 수정 포인트: 동시에 사용할 수 있는 최대 커넥션 수 (Redis 스펙과 동시 요청 수 기반으로 결정)
          max-idle: 8      # 풀에 유지할 최대 유휴 커넥션 수
          min-idle: 1      # 풀에 최소한 유지할 유휴 커넥션 수
          max-wait: -1ms   # 수정 포인트: -1은 무한 대기. 실무에서는 1000ms ~ 5000ms로 타임아웃 설정 권장
    옵션 의미 주의사항
    max-active 풀에서 동시에 사용 가능한 최대 커넥션 수 Redis 서버 스펙과 예상 동시 요청 수를 기반으로 결정
    max-idle 풀에 유지할 최대 유휴(대기 중인) 커넥션 수 보통 max-active와 동일하게 설정
    min-idle 풀에 최소한 유지할 유휴 커넥션 수 너무 낮으면 요청 급증 시 커넥션 부족 발생
    max-wait 풀이 꽉 찼을 때 커넥션을 기다리는 최대 시간 -1은 무한 대기. 실무에서는 타임아웃을 명시해야 요청이 쌓여 서버가 다운되는 것을 방지

    마무리

    Redis는 단순히 "빠른 캐시"가 아니라 String, Hash, Sorted Set 등 다양한 자료구조를 지원하는 인메모리 데이터베이스다. 어떤 자료형을 선택하느냐에 따라 랭킹 시스템, 세션 저장, 메시지 큐 등 다양한 기능을 Redis 하나로 구현할 수 있다. RedisTemplate의 opsFor~() 메서드는 각 자료형에 대응하는 Redis 명령어를 Java 코드로 추상화한 것이므로, 자료형과 명령어의 대응 관계를 이해하면 자연스럽게 사용할 수 있다.

    Cache-Aside 패턴은 Redis를 실무에서 가장 많이 쓰는 방식이다. 캐시 확인 → DB 조회 → 캐시 저장의 세 단계가 핵심이고, 데이터 수정 시 반드시 캐시를 무효화해야 한다는 점을 항상 기억해야 한다. 캐시 Key는 코드에서 상수로 관리해야 오타로 인한 버그를 방지할 수 있다.

    커넥션 풀은 Redis뿐만 아니라 DB 연결에서도 동일하게 적용되는 개념이다. 커넥션을 매번 생성하고 소멸하는 방식은 비용이 크기 때문에, 미리 생성해두고 재사용하는 풀 방식이 필수다. max-wait: -1처럼 무한 대기로 설정하면 풀이 꽉 찼을 때 요청이 쌓이다 서버가 다운될 수 있다. 실무에서는 반드시 타임아웃을 명시해야 한다.
