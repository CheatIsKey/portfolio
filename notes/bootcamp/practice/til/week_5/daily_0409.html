    1. 실시간 서비스와 HTTP의 한계

    실시간 서비스는 이벤트가 발생했을 때 지연 없이 즉시 전달되는 서비스다. 연결이 지속되어야 하고, 서버가 클라이언트의 요청 없이도 먼저 메시지를 전달할 수 있어야 한다.

    항목 HTTP 구조 실시간 서비스 요구사항
    연결 방식 요청마다 연결을 맺고 바로 끊음 (Stateless) 연결이 지속되어야 함
    통신 방향 클라이언트 → 서버 단방향 서버 → 클라이언트 Push 가능해야 함
    이벤트 반응 클라이언트가 먼저 요청해야 함 이벤트 발생 즉시 반응해야 함
    ⚠️ Polling으로 실시간을 흉내 내는 방식의 문제점 클라이언트가 일정 주기로 서버에 반복 요청하는 Polling 방식은 데이터가 없어도 요청이 발생한다. 불필요한 트래픽이 지속적으로 생기고, 실시간성이 요청 주기에 의존한다. 1초 주기로 Polling하면 최대 1초의 지연이 발생하고, 동시 사용자가 많으면 서버 부하가 폭증한다.

    2. WebSocket — 지속적인 양방향 통신 채널

    WebSocket은 HTTP의 구조적 한계를 보완하기 위해 등장한 통신 프로토콜이다. 클라이언트와 서버가 한 번 연결되면 연결을 유지하고, 양방향으로 자유롭게 메시지를 주고받는다. 서버는 클라이언트의 요청 없이도 언제든지 메시지를 전송할 수 있다.

    WebSocketConfig — 연결 진입점 설정

    WebSocketConfig는 어떤 주소로 WebSocket 연결을 받을 것인지, 어떤 Handler가 이 연결을 처리할 것인지를 결정하는 WebSocket 서버의 진입점이다. 

    @Configuration
@EnableWebSocket  // Spring에서 WebSocket 기능을 활성화
public class WebSocketConfig implements WebSocketConfigurer {

    @Override
    public void registerWebSocketHandlers(WebSocketHandlerRegistry registry) {
        registry.addHandler(chatWebSocketHandler(), "/chat")
                .setAllowedOriginPatterns("*"); // 수정 포인트: 운영에서는 허용 Origin을 제한한다
    }

    @Bean
    public WebSocketHandler chatWebSocketHandler() {
        return new ChatWebSocketHandler();
    }
}

    ChatWebSocketHandler — 실제 통신 로직

    ChatWebSocketHandler는 연결됐을 때, 메시지를 받았을 때, 연결이 끊어졌을 때 각각 무엇을 할지를 정의한다. 

    public class ChatWebSocketHandler extends TextWebSocketHandler {

    // 현재 연결된 모든 세션을 저장 — 여러 클라이언트가 동시에 연결될 수 있으므로 필요
    private final Set<WebSocketSession> sessions = new HashSet<>();

    // WebSocket 연결이 성공했을 때 자동으로 호출
    @Override
    public void afterConnectionEstablished(WebSocketSession session) throws Exception {
        sessions.add(session);
        // 수정 포인트: 서버가 클라이언트에게 먼저 메시지를 전송할 수 있다
        session.sendMessage(new TextMessage("WebSocket 연결이 완료되었습니다."));
    }

    // 클라이언트가 메시지를 보낼 때 호출
    @Override
    protected void handleTextMessage(WebSocketSession session, TextMessage message) throws Exception {
        // 현재 연결된 모든 세션에 메시지를 브로드캐스트
        for (WebSocketSession s : sessions) {
            s.sendMessage(new TextMessage("서버에서 받은 메시지: " + message.getPayload()));
        }
    }

    // WebSocket 연결이 종료되었을 때 호출
    @Override
    public void afterConnectionClosed(WebSocketSession session, CloseStatus status) throws Exception {
        sessions.remove(session); // 종료된 세션을 목록에서 제거
    }
}

    3. STOMP — WebSocket 위의 메시지 규칙

    순수 WebSocket 방식의 한계

    순수 WebSocket만 쓰면 서버가 모든 세션을 직접 관리해야 하고, 메시지를 받을 때마다 어디로 보낼지 판단하는 라우팅 로직을 직접 작성해야 한다. 채팅방 개념을 구현하려면 세션 분류 코드가 점점 복잡해진다.

    STOMP가 해결하는 것 STOMP(Simple Text Oriented Messaging Protocol)는 WebSocket 위에서 동작하는 메시지 프로토콜이다.

    WebSocket은 통신 통로를 제공한다.
    STOMP는 그 통로 위에서 메시지를 어떻게 주고받을지에 대한 규칙을 제공한다.

    메시지 전달 기준이 세션 중심 → 목적지(destination) 중심으로 전환된다. 서버는 메시지 목적지만 정의하고, 그 목적지를 구독한 클라이언트에게 자동으로 메시지가 전달된다.

    STOMP WebSocketConfig 

    @Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        registry.enableSimpleBroker("/sub");           // 서버 → 클라이언트 브로드캐스트 경로
        registry.setApplicationDestinationPrefixes("/pub"); // 클라이언트 → 서버 메시지 전송 경로
    }

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws")                    // WebSocket 연결 엔드포인트
                .setAllowedOriginPatterns("*");
    }
}

    ChatController — STOMP 메시지 처리 

    @Slf4j
@Controller  // 수정 포인트: @RestController는 HTTP 응답 반환용 → STOMP에는 @Controller 사용
@RequiredArgsConstructor
public class ChatController {

    private final SimpMessagingTemplate messagingTemplate;

    @MessageMapping("/chat.send")  // /pub/chat.send로 온 STOMP SEND 메시지를 처리
    public void sendMessage(ChatMessage message, SimpMessageHeaderAccessor headerAccessor) {
        String sessionId = headerAccessor.getSessionId();
        log.info("sessionId: {} | content: {}", sessionId, message.getContent());

        // /sub/chat을 구독한 모든 클라이언트에게 메시지 전송
        messagingTemplate.convertAndSend("/sub/chat", message);
    }
}

@Getter
@Setter
@AllArgsConstructor
public class ChatMessage {
    private String sender;   // 전송자
    private String content;  // 내용
}

    4. STOMP 인증 처리 — ChannelInterceptor

    STOMP는 HTTP와 달리 연결을 유지하기 때문에 Spring Security의 Filter에서 인증을 수행할 수 없다. 대신 ChannelInterceptor에서 CONNECT 시점에 JWT를 검증하고 사용자 정보를 세션에 저장한다.

    AuthenticatedUser — 세션 내 사용자 정보 저장소 

    // Principal 인터페이스를 구현해 Spring이 사용자를 식별할 수 있게 한다
@Getter
public class AuthenticatedUser implements Principal {

    private final User user;
    private final String name;

    public AuthenticatedUser(User user) {
        this.user = user;
        this.name = user.getName();
    }

    // 컨트롤러에서 Principal로부터 User 객체를 꺼낼 때 사용
    public static User fromPrincipal(Principal principal) {
        return ((AuthenticatedUser) principal).getUser();
    }
}

    StompAuthInterceptor — CONNECT 시점 JWT 검증 

    @Component
@RequiredArgsConstructor
public class StompAuthInterceptor implements ChannelInterceptor {

    private final JwtUtil jwtUtil;
    private final UserRepository userRepository;

    @Override
    public Message<?> preSend(Message<?> message, MessageChannel channel) {
        StompHeaderAccessor accessor =
                MessageHeaderAccessor.getAccessor(message, StompHeaderAccessor.class);

        // 수정 포인트: CONNECT 명령일 때만 JWT 검증을 수행한다
        if (StompCommand.CONNECT.equals(accessor.getCommand())) {
            String token = accessor.getFirstNativeHeader("Authorization")
                    .replace("Bearer ", "");

            Long userId = jwtUtil.getUserId(token);
            User user   = userRepository.findById(userId).orElseThrow();

            // 세션에 인증된 사용자 정보를 저장
            // 이후 같은 sessionId로 들어오는 SUBSCRIBE, SEND는 이 값을 공유한다
            accessor.setUser(new AuthenticatedUser(user));
        }

        return message;
    }
}
    accessor.setUser()가 동작하는 방식 CONNECT 시점에 accessor.setUser(new AuthenticatedUser(user))를 호출하면, 이후 같은 세션 ID로 들어오는 모든 SUBSCRIBE, SEND 요청에서 세션의 주인이 누구인지를 공유한다. 컨트롤러에서 Principal principal로 주입받으면 언제든지 현재 사용자를 꺼낼 수 있다. HTTP의 세션 쿠키와 유사하지만, WebSocket 연결 단위로 관리된다.

    5. STOMP 프레임 구조

    STOMP는 텍스트 기반 프레임 프로토콜이다. Postman으로 테스트할 때는 STOMP 프레임 원본을 Base64로 인코딩해서 전송해야 한다. 각 프레임은 명령(Command), 헤더(Header), 바디(Body), NULL 종료 문자(\0)로 구성된다.

    프레임 역할 주요 헤더
    CONNECT STOMP 통신을 시작하겠다는 핸드셰이크. 어떤 메시지 규칙을 준수할 것인지 약속한다. accept-version, host, Authorization
    SUBSCRIBE 특정 목적지(destination)의 메시지를 받겠다고 구독 등록한다. id (구독 식별자), destination (구독 경로)
    SEND 서버로 메시지를 전송한다. HTTP의 POST 요청과 유사한 역할이다. destination (전송 경로), content-type, content-length 
    -- CONNECT 프레임 (원본)
CONNECT
accept-version:1.2
host:localhost

\0

-- SUBSCRIBE 프레임 (원본)
SUBSCRIBE
id:sub-0              ← 하나의 연결에서 여러 구독을 구분하기 위한 구독 ID
destination:/sub/chat ← 이 경로로 발행되는 메시지를 수신

\0

-- SEND 프레임 (원본)
SEND
destination:/pub/chat.send  ← Spring @MessageMapping("/chat.send")에 연결
content-type:application/json
content-length:33

{"sender":"A","content":"hello"}

\0
    STOMP에서 /pub과 /sub의 역할 /pub (Application Destination Prefix): 클라이언트 → 서버로 메시지를 보낼 때 사용하는 경로다. /pub/chat.send로 SEND 프레임을 보내면 @MessageMapping("/chat.send")가 처리한다.

    /sub (Broker Destination): 서버 → 클라이언트로 메시지를 브로드캐스트하는 경로다. /sub/chat을 구독한 클라이언트들은 서버가 이 경로로 발행하는 메시지를 자동으로 수신한다.

    마무리

    실시간 서비스는 HTTP의 요청-응답 구조로는 근본적으로 구현하기 어렵다. Polling이 그 한계를 우회하려는 시도였지만, 불필요한 트래픽과 지연이라는 근본적인 문제는 해결하지 못한다. WebSocket은 이 문제를 연결 유지와 양방향 통신으로 해결하고, STOMP는 그 위에 메시지 라우팅 규칙을 얹어서 채팅방 같은 복잡한 구조를 세션 관리 코드 없이 구현할 수 있게 한다.

    STOMP 인증에서 가장 중요한 개념은 CONNECT 시점에 한 번만 인증하고 이후 메시지들이 그 세션 정보를 공유한다는 것이다. HTTP는 매 요청마다 토큰을 검증하지만, WebSocket은 연결이 지속되기 때문에 CONNECT 한 번에 인증하면 그 세션이 살아있는 동안 인증이 유지된다. 이 차이를 이해하지 못하면 ChannelInterceptor 대신 Filter에서 인증하려다 동작하지 않는 문제를 겪게 된다.

    STOMP 프레임의 구조(명령 / 헤더 / 바디 / NULL 종료)를 직접 작성해보면서 HTTP의 요청-응답과 얼마나 다른 방식으로 설계되어 있는지를 체감했다. 특히 destination 헤더 하나로 어떤 컨트롤러 메서드를 실행할지, 어떤 구독자들에게 메시지를 전달할지를 결정한다는 것이 목적지 중심 설계의 핵심이다.
