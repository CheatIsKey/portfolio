    오늘 목표

    • 동시성 문제와 경쟁 상태(Race Condition)가 발생하는 원인을 이해한다.
    • 비관적 락과 낙관적 락의 동작 방식 및 차이를 정리한다.
    • 단일 서버에서는 충분한 락이 다중 서버 환경에서 한계를 갖는 이유를 파악한다.
    • 분산 락의 필요성과 Redis SETNX 기반 구현 방법을 이해한다.
    • 락 획득 전략 3가지(즉시 실패, 재시도, 블로킹)의 차이와 사용 시점을 익힌다.

    1. 동시성과 경쟁 상태

    동시성은 하나의 데이터를 여러 사용자가 동시에 수정하거나 읽는 상황을 의미한다. DB는 요청을 독립된 트랜잭션으로 처리하기 때문에, 현재 다른 요청이 같은 데이터를 수정하고 있는지 알지 못한다. 그래서 여러 요청이 동시에 들어오면 서로의 결과를 덮어쓰는 문제가 생긴다.

    경쟁 상태 (Race Condition) 동일 자원을 누가 먼저 처리하느냐에 따라 결과가 달라지고 데이터 일관성이 깨지는 문제다.

    예를 들어 잔액이 10,000원인 계좌에서 두 요청이 동시에 5,000원씩 출금을 시도한다면, 두 요청 모두 10,000원을 읽은 뒤 각각 5,000원을 뺀 값을 저장하면 잔액이 5,000원으로 기록된다. 실제로는 둘 다 성공했으니 0원이 되어야 하는데, 한 번의 출금이 사라진 것이다.
    해결 방법 설명
    비관적 락 먼저 잠그고 다른 사용자는 대기시킨다. (선행 조치)
    낙관적 락 잠그지 않고, 나중에 충돌 여부를 확인한다. (후행 조치)
    분산 락 여러 서버 환경에서도 동일한 데이터를 하나만 수정하도록 조정한다.

    2. 비관적 락 (Pessimistic Lock)

    한 트랜잭션이 데이터를 수정하는 동안 다른 트랜잭션은 그 데이터에 접근 자체가 불가능하다. 다른 요청은 락이 해제될 때까지 대기 상태가 된다. 충돌이 발생할 것이라는 비관적 가정 하에 조회 시점부터 미리 락을 거는 선행 조치 방식이다. 

    // JPA Repository — 조회 시점에 DB Row를 잠근다
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT a FROM Account a WHERE a.id = :id")
Account findByIdForLock(@Param("id") Long id);
    ⚠️ 비관적 락 사용 시 주의사항 조회용 쿼리에는 사용하지 않는다. 불필요하게 락을 걸면 전체 성능이 저하된다.
    쓰기 작업이 동시에 발생할 가능성이 높은 구간에만 사용한다.
    트랜잭션을 짧게 유지해야 한다. 락을 오래 잡고 있으면 다른 요청들이 대기하면서 서버가 느려진다.

    3. 낙관적 락 (Optimistic Lock)

    동시에 같은 데이터를 수정할 가능성이 낮다고 낙관적으로 가정하고, 데이터를 잠그지 않고 자유롭게 수정한다. 수정이 완료되는 시점에 해당 데이터가 이미 변경됐는지를 버전(version) 필드를 통해 감지한다. 충돌이 감지되면 예외를 발생시키고 재시도 로직이 처리한다. 

    // 엔티티에 @Version 필드를 추가하면 JPA가 자동으로 관리
@Entity
public class Account {

    @Id
    private Long id;

    private Long balance;

    @Version  // 수정 포인트: 이 필드 하나로 낙관적 락이 자동 적용된다
    private Long version;
}
    @Version이 동작하는 방식 데이터를 수정할 때 JPA는 UPDATE ... WHERE id = ? AND version = ? 형태의 쿼리를 실행한다. 다른 트랜잭션이 먼저 수정해서 version 값이 바뀌었다면 UPDATE가 0건 처리되고, JPA는 이를 감지해 OptimisticLockException을 던진다. 애플리케이션 코드에서 이 예외를 잡아 재시도 로직을 구현해야 한다.
    구분 비관적 락 낙관적 락
    락 시점 조회할 때부터 잠금 수정/저장 시점에만 충돌 감지
    대기 방식 다른 요청은 락 해제까지 대기 대기 없음, 충돌 시 예외 발생
    적합한 상황 충돌이 자주 발생하는 쓰기 위주 환경 충돌이 드문 읽기 위주 환경 (게시글 수정 등)
    단점 락 대기로 인한 성능 저하 가능 충돌 시 재시도 로직 직접 구현 필요

    4. 분산 락 (Distributed Lock)

    왜 분산 락이 필요한가

    서버가 1대일 때는 비관적 락과 낙관적 락으로 충분하다. 모든 요청이 단일 서버에서 순차적으로 처리되고, Hibernate의 버전 방식과 DB의 Row Lock이 정상적으로 동작한다.

    서버가 여러 대인 환경에서는 문제가 발생한다. 서버 A와 서버 B는 서로의 상태를 알지 못한다. 메모리를 공유하지 않고, Hibernate의 영속성 컨텍스트도 각각 따로 존재한다. 어떤 서버에서 락이 걸렸는지 서로 알 수 없기 때문에 동시성 문제가 그대로 발생한다.

    락 종류별 보호 범위 비교 낙관적 락: Hibernate가 관리. UPDATE 충돌만 감지. 다중 서버에서 안전하지 않음.
    비관적 락: DB가 관리. DB Row 수정 시점만 보호. 다중 서버에서 UPDATE는 보호되나 로직 전체는 아님.
    분산 락: Redis 등 외부 저장소가 관리. 조회~비즈니스 로직~DB까지 전체 흐름을 보호. 가장 안전.

    Redis SETNX 기반 분산 락

    Redis의 SETNX(Set if Not Exists) 명령을 이용해 "아직 없는 키만" 등록할 수 있다. 이미 키가 존재하면 등록에 실패하므로, 한 번에 한 서버만 락을 획득할 수 있다.

    단계 설명
    1 SETNX lock:account:1 요청 → 성공하면 락 획득
    2 TTL(유효 시간)을 설정해 서버 장애 시 락이 자동으로 해제되도록 설정
    3 락을 획득한 서버가 안전하게 작업 수행
    4 작업 완료 후 DEL lock:account:1로 락 해제
    고려 항목 설명
    TTL 설정 서버 장애 시 락이 영원히 잠기지 않도록 반드시 설정해야 한다.
    락 주인 확인 락을 잡은 주체만 해제할 수 있도록 UUID를 값으로 저장하고 해제 전 검증한다.
    재시도 로직 락 획득 실패 시 일정 시간 대기 후 재시도하는 전략이 필요하다.

    5. 락 획득 실패 시 전략

    전략 1 — 즉시 실패 (Return Error)

    락 획득에 실패하면 바로 실패 응답을 반환한다. 출금, 주문, 결제처럼 동일 요청을 중복 실행하면 안 되는 작업이나 이벤트 쿠폰 선착순 로직에 적합하다. Redis에 부하가 없고 중복 요청을 확실히 방지하지만, 사용자가 즉시 실패 응답을 받는다.

    전략 2 — 재시도 (Retry)

    잠시 기다렸다가 다시 시도한다. 상품 재고 감소, 장바구니 → 주문 변환처럼 조금 기다려서라도 반드시 성공해야 하는 작업에 사용한다. 

    int retry = 0;
while (retry < 5) {
    boolean locked = lockService.tryLock(key, value, 5); // TTL 5초

    if (locked) break; // 락 획득 성공

    Thread.sleep(100); // 수정 포인트: 100ms 간격으로 재시도. 백오프 전략 적용 권장
    retry++;
}

if (retry == 5) {
    throw new IllegalStateException("잠시 후 다시 시도해주세요.");
}

    전략 3 — 블로킹 대기 (Blocking)

    락이 풀릴 때까지 계속 기다린다. 실무에서는 거의 사용하지 않는다. 서버 스레드를 오래 점유해 서버 부하가 증가하고, 하나의 요청이 수 초 동안 대기하면 전체 시스템에 영향을 준다.

    전략 적합한 상황 장점 단점
    즉시 실패 결제, 출금, 선착순 이벤트 Redis 부하 없음, 중복 방지 확실 사용자가 바로 실패 응답을 받음
    재시도 재고 감소, 주문 변환 늦어져도 성공 가능 재시도 과다 시 Redis 부하, 백오프 설계 필요
    블로킹 실무에서 거의 미사용 — 스레드 점유, 전체 시스템 영향

    6. 분산 락 사용 시 주의사항

    ⚠️ TTL이 너무 짧으면 발생하는 문제 작업이 끝나지 않았는데 TTL이 먼저 만료되면 락이 풀린다. 락이 풀린 사이에 다른 요청이 들어와 같은 작업을 중복 실행하는 문제가 생긴다. 작업이 완료되는 시간보다 TTL을 충분히 길게 설정해야 한다.
    ⚠️ Redis 장애 시 동시성 제어가 무너진다 Redis가 다운되면 락을 잡는 것 자체가 불가능해진다. 락이 없는 상태에서 요청이 모두 동시에 처리되어 경쟁 상태가 발생한다. Redis를 분산 락의 단일 장애점(SPOF)으로 사용할 때는 Redis 고가용성(Sentinel, Cluster) 구성을 함께 고려해야 한다.
    ⚠️ 락 남용 시 성능 저하 모든 요청이 순차적으로 처리되어 처리량(Throughput)이 크게 떨어진다. 서버 스레드가 대기하다 Timeout이 발생하고 Redis에도 부하가 생긴다. 실제로 동시성 문제가 발생할 가능성이 있는 구간에만 선택적으로 적용해야 한다.

    마무리

    동시성 문제는 코드를 아무리 잘 짜도 여러 요청이 동시에 들어오는 순간 언제든 발생할 수 있다. 중요한 것은 어떤 락을 어느 범위에 적용할지를 상황에 맞게 판단하는 것이다. 충돌이 드문 환경에서 비관적 락을 남용하면 모든 요청이 줄을 서게 되어 처리량이 크게 떨어지고, 충돌이 잦은 환경에서 낙관적 락만 쓰면 예외와 재시도가 폭발적으로 늘어난다.

    분산 락은 다중 서버 환경에서 비관적 락과 낙관적 락이 보호하지 못하는 영역을 커버한다. 비관적 락이 DB Row 수정 시점만 보호하는 것과 달리, 분산 락은 조회부터 비즈니스 로직 전체를 하나의 임계 구간으로 보호한다. 단, Redis가 단일 장애점이 되지 않도록 고가용성 구성을 함께 고려해야 하고, TTL 설정과 락 주인 검증을 빠뜨리면 오히려 더 심각한 문제가 생긴다.

    락 획득 실패 시 전략도 상황마다 다르다. 결제나 선착순처럼 중복이 절대 허용되지 않는 곳에서는 즉시 실패가 안전하고, 재고 감소처럼 조금 늦더라도 반드시 성공해야 하는 곳에서는 재시도 전략이 적합하다. 재시도를 구현할 때는 고정 간격이 아닌 지수 백오프(Exponential Backoff)를 적용해 Redis 부하를 분산하는 것이 좋다.
