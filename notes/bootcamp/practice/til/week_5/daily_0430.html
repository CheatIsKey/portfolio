    오늘 목표

    • 페이징 최적화 원칙(1:1은 조인, 1:N은 IN 쿼리 분리)을 이해한다.
    • 비정규화 컬럼(like_count)의 성능과 정합성 트레이드오프와 두 가지 해결 전략을 파악한다.
    • JdbcTemplate.batchUpdate()로 수만 건 UPDATE를 단일 네트워크 전송으로 처리하는 방법을 익힌다.

    • Spring Batch Chunk 처리 구조(Job → Step → Reader/Processor/Writer)를 이해한다.
    • Processor를 생략하고 Reader에서 집계까지 끝내는 최적화 방향과 그 트레이드오프를 파악한다.
    • 여러 Job을 @Qualifier로 구분하고 스케줄러에서 관리하는 패턴을 익힌다.

    1. 페이징 최적화 원칙

    페이징 쿼리에서 연관 데이터를 함께 가져올 때는 연관 유형에 따라 전략을 나눈다. 1:1 통계 데이터(좋아요 수, 댓글 수 등)는 QueryDSL 서브쿼리나 조인으로 한 번에 합쳐오고, 1:N 컬렉션 데이터(이미지 목록, 태그 목록 등)는 IN 쿼리로 분리해서 가져온다. 컬렉션에 Fetch Join을 사용하면 페이징이 메모리에서 처리되므로 반드시 분리해야 한다.

    2. 비정규화 정합성 — 성능과 정확성의 트레이드오프

    like_count처럼 집계 결과를 컬럼에 저장하는 비정규화는 조회 성능을 높이지만, 실제 카운트와 값이 어긋나는 정합성 문제가 언제든 생길 수 있다. 이를 해결하는 두 가지 전략이 있다.

    전략 방식 장점 단점
    주기적 배치 보정
    (Spring Batch)    		 매일 특정 시간에 실제 COUNT와 비정규화 컬럼을 비교해 차이가 나는 값만 일괄 보정 시스템 복잡도 낮음, 메인 트래픽에 거의 영향 없음 배치가 돌기 전까지 사용자에게 약간의 불일치가 보일 수 있음
    실시간 분산 락
    (Redis Redisson)    		 Redis 분산 락을 걸고 비정규화 컬럼을 원자적으로 +1 / -1 수행 거의 실시간에 가까운 정합성 보장 락 경합이 심하면 지연 발생, Redis 인프라 운영 비용 증가
    하이브리드 전략 — 실무 권장 방식 실시간성이 중요하다면 Redis를 메인으로 써서 빠른 정합성을 보장하고, 데이터 신뢰성을 위해 Spring Batch를 주기적으로 돌려 최종 정합성도 같이 챙긴다. Redis 장애나 락 누락으로 생긴 누적 오차를 배치가 자동으로 보정해준다.

    3. @Scheduled 옵션 비교

    ⚠️ @EnableScheduling을 반드시 메인 클래스에 선언해야 한다 @EnableScheduling이 없으면 @Scheduled를 아무리 붙여도 Spring이 완전히 무시한다. 배치가 실행되지 않는 원인의 상당수가 이 선언 누락이다.
    옵션 의미 예시
    fixedDelay 이전 실행이 종료된 후 N ms 뒤에 재실행 fixedDelay = 60_000 (1분)
    fixedRate 이전 실행 시작 기준 N ms 간격 (실행이 겹칠 수 있음) fixedRate = 60_000
    cron Unix cron 표현식 (초 분 시 일 월 요일) "0 0 2 * * *" = 매일 새벽 2시
    ⚠️ fixedRate 사용 시 주의 — 실행이 겹칠 수 있다 이전 실행이 아직 끝나지 않았는데 다음 실행이 시작될 수 있다. 배치 작업이 겹치면 DB에 중복 UPDATE가 나가거나 데드락이 발생할 수 있다. 배치 작업에는 fixedDelay 또는 cron을 사용하는 것이 안전하다.

    4. JdbcTemplate.batchUpdate() — 수만 건 UPDATE를 한 번에

    JPA의 save()는 영속성 컨텍스트를 거쳐 UPDATE 쿼리가 건별로 나간다. saveAll()은 트랜잭션 하나지만 쿼리 수는 동일하다. JdbcTemplate.batchUpdate()는 JDBC 드라이버 레벨에서 쿼리를 배열로 묶어 한 번에 DB로 전송한다.

    queryForList() 방식 — 타입 안전성 없음 

    // [ {post_id: 1, cnt: 3}, {post_id: 2, cnt: 1}, ... ]
List<Map<String, Object>> results = jdbcTemplate.queryForList("""
        SELECT p.id AS post_id, COUNT(pl.post_id) AS cnt
        FROM posts p LEFT JOIN post_like pl ON p.id = pl.post_id
        GROUP BY p.id
        """);

// 업데이트 파라미터 배열 준비
List<Object[]> params = results.stream()
        .map(row -> new Object[]{row.get("cnt"), row.get("post_id")})
        .toList();

// 한 번의 배치 호출 — JDBC 드라이버가 묶어서 DB에 전송
jdbcTemplate.batchUpdate(
        "UPDATE posts SET like_count = ? WHERE id = ?",
        params
);
    ⚠️ queryForList()의 타입 안전성 문제 Map<String, Object>의 키는 문자열이므로 "post_id"를 "postId"로 오타를 내면 컴파일 타임에 잡히지 않고 런타임에 NPE가 터진다. 이 문제를 해결하려면 RowMapper로 Record에 직접 매핑하는 방식으로 교체한다.

    RowMapper + Record 방식 — 타입 안전 

    // 타입 안전한 매핑용 Record 선언
record LikeCountRow(long postId, long count) {}

// ① LEFT JOIN 집계 — RowMapper로 Record에 직접 매핑
List<LikeCountRow> rows = jdbcTemplate.query("""
        SELECT p.id AS post_id, COUNT(pl.post_id) AS cnt
        FROM posts p LEFT JOIN post_like pl ON p.id = pl.post_id
        GROUP BY p.id
        """,
        (rs, rowNum) -> new LikeCountRow(rs.getLong("post_id"), rs.getLong("cnt"))
);

// ② 한 번의 배치 호출 — 오타가 나면 컴파일 에러로 즉시 감지
jdbcTemplate.batchUpdate(
        "UPDATE posts SET like_count = ? WHERE id = ?",
        rows.stream()
                .map(row -> new Object[]{row.count(), row.postId()})
                .toList()
);
    ⚠️ 전체 결과를 메모리에 올리는 문제 — OOM 위험 위 방식들은 SQL 결과 전체를 List로 메모리에 한 번에 올린다. 데이터가 수백만 건이라면 OOM(Out Of Memory)으로 서버가 다운될 수 있다. 이 문제를 해결하는 것이 Spring Batch의 Chunk 처리 방식이다.

    5. Spring Batch Chunk 처리

    의존성 및 설정 주의사항 

    // build.gradle
implementation 'org.springframework.boot:spring-boot-starter-batch'
    # application.yml
spring:
  batch:
    job:
      enabled: false              # 앱 시작 시 Job 자동 실행 방지
    jdbc:
      initialize-schema: embedded # H2에서 배치 메타 테이블 자동 생성
    ⚠️ @EnableBatchProcessing을 절대 붙이지 말자 Spring Boot 3.x에서는 @EnableBatchProcessing을 붙이면 자동 설정과 충돌해 오히려 배치가 동작하지 않는다. 아무것도 붙이지 않아야 Spring Boot의 자동 설정이 정상적으로 작동한다.

    Spring Batch 구조

    Job → Step → Chunk (Reader / Processor / Writer) Job: 전체 작업의 이름표. 여러 Step을 묶는 컨테이너다.
    Step: Job 안의 실제 처리 단위. 하나의 Job에 여러 Step이 있을 수 있다.
    Chunk: N건씩 반복 처리하는 단위. 중간에 오류가 나도 이전 Chunk는 이미 커밋되어 보존된다.

    ItemReader: DB에서 N건씩 스트리밍으로 읽어온다.
    ItemProcessor: 읽은 데이터를 가공해 Writer가 쓸 데이터로 변환한다.
    ItemWriter: 가공된 데이터를 Chunk 크기만큼 모아 DB에 한 번에 기록한다.

    Job / Step 설정 

    @Configuration
@RequiredArgsConstructor
public class LikeCountSyncJobConfig {

    private final JobRepository jobRepository;
    private final PlatformTransactionManager transactionManager;
    private final DataSource dataSource;
    private final JdbcTemplate jdbcTemplate;

    @Bean
    public Job likeCountSyncJob() {
        return new JobBuilder("likeCountSyncJob", jobRepository)
                .incrementer(new RunIdIncrementer()) // 매 실행마다 새 run.id 부여 → 중복 실행 방지
                .start(likeCountSyncStep())
                .build();
    }

    @Bean
    public Step likeCountSyncStep() {
        return new StepBuilder("likeCountSyncStep", jobRepository)
                // 수정 포인트: 1,000건씩 읽고 커밋. 중간 오류가 나도 이전 Chunk는 보존
                .<Long, Object[]>chunk(1000, transactionManager)
                .reader(postIdReader())
                .processor(likeCountProcessor())
                .writer(likeCountWriter())
                .build();
    }
}

    Reader — JdbcCursorItemReader (스트리밍) 

    @Bean
public JdbcCursorItemReader<Long> postIdReader() {
    return new JdbcCursorItemReaderBuilder<Long>()
            .name("postIdReader")
            .dataSource(dataSource)
            .sql("SELECT id FROM posts ORDER BY id")
            // 수정 포인트: RowMapper로 필요한 타입만 추출
            .rowMapper((rs, rowNum) -> rs.getLong("id"))
            .build();
}
    JdbcCursorItemReader가 OOM을 방지하는 이유 queryForList()는 결과 전체를 메모리에 올린다. JdbcCursorItemReader는 DB 커서를 열어두고 필요할 때마다 한 건씩 스트리밍으로 읽는다. 수천만 건이 있어도 Chunk 크기(예: 1,000건)만큼만 메모리에 올리므로 OOM이 발생하지 않는다.

    Processor — 건별 COUNT 쿼리 방식 

    @Bean
public ItemProcessor<Long, Object[]> likeCountProcessor() {
    return postId -> {
        // 수정 포인트: postId마다 COUNT 쿼리 1건 발생 → 쿼리 수 많음, 메모리는 안전
        Long count = jdbcTemplate.queryForObject(
                "SELECT COUNT(*) FROM post_like WHERE post_id = ?",
                Long.class, postId
        );
        return new Object[]{count, postId}; // Writer에 전달할 파라미터 배열
    };
}

    Writer — JdbcBatchItemWriter 

    @Bean
public JdbcBatchItemWriter<Object[]> likeCountWriter() {
    return new JdbcBatchItemWriterBuilder<Object[]>()
            .dataSource(dataSource)
            .sql("UPDATE posts SET like_count = ? WHERE id = ?")
            .itemPreparedStatementSetter((item, ps) -> {
                ps.setLong(1, (Long) item[0]); // like_count = ?
                ps.setLong(2, (Long) item[1]); // WHERE id = ?
            })
            .build();
}

    Processor 생략 최적화 — Reader에서 GROUP BY 집계까지

    Processor가 postId마다 COUNT 쿼리를 날리면 Chunk 수만큼 SELECT가 발생한다. Reader SQL에 GROUP BY를 포함하면 DB가 집계까지 한 번에 끝내고 Processor를 생략할 수 있다. 단, GROUP BY 집계는 DB의 임시 테이블 공간을 사용하므로 대용량 테이블에서는 DB 부하를 모니터링해야 한다. 

    // 리팩토링 버전 — Reader가 집계까지 한 번에 처리
@Bean
public JdbcCursorItemReader<Object[]> postLikeCountReader() {
    return new JdbcCursorItemReaderBuilder<Object[]>()
            .name("postLikeCountReader")
            .dataSource(dataSource)
            .sql("""
                SELECT p.id AS post_id, COUNT(pl.post_id) AS cnt
                FROM posts p LEFT JOIN post_like pl ON p.id = pl.post_id
                GROUP BY p.id
                ORDER BY p.id
                """)
            .rowMapper((rs, rowNum) -> new Object[]{
                    rs.getLong("cnt"), rs.getLong("post_id")
            })
            .build();
}

@Bean
public Step likeCountSyncStep() {
    return new StepBuilder("likeCountSyncStep", jobRepository)
            .<Object[], Object[]>chunk(1000, transactionManager)
            .reader(postLikeCountReader())
            // Processor 생략 — Reader가 이미 Writer 형식으로 데이터 제공
            .writer(likeCountWriter())
            .build();
}

    스케줄러 — 여러 Job을 @Qualifier로 구분 

    @Slf4j
@Component
@RequiredArgsConstructor
public class LikeCountScheduler {

    private final JobLauncher jobLauncher;

    // 수정 포인트: Job이 여러 개면 @Qualifier로 Bean 이름을 지정해 구분한다
    @Qualifier("likeCountSyncJob")
    private final Job likeCountSyncJob;

    @Qualifier("followerCountSyncJob")
    private final Job followerCountSyncJob;

    @Qualifier("commentCountSyncJob")
    private final Job commentCountSyncJob;

    @Scheduled(cron = "0 0 2 * * *")    // 매일 새벽 2시
    public void runLikeCountSync() {
        runJob(likeCountSyncJob, "likeCount");
    }

    @Scheduled(cron = "0 10 2 * * *")   // 새벽 2시 10분
    public void runFollowerCountSync() {
        runJob(followerCountSyncJob, "followerCount");
    }

    @Scheduled(cron = "0 20 2 * * *")   // 새벽 2시 20분
    public void runCommentCountSync() {
        runJob(commentCountSyncJob, "commentCount");
    }

    private void runJob(Job job, String label) {
        try {
            JobParameters params = new JobParametersBuilder()
                    // 수정 포인트: 같은 Job + 파라미터는 이미 실행된 것으로 판단해 스킵
                    // 매번 다른 run.id로 항상 새 실행을 허용
                    .addLong("run.id", System.currentTimeMillis())
                    .toJobParameters();
            jobLauncher.run(job, params);
        } catch (Exception e) {
            // 수정 포인트: @Scheduled 메서드에서 예외가 나면 스케줄러가 죽는다
            // try-catch로 감싸서 다음 주기가 계속 실행되도록 보장
            log.error("[Batch-{}] Job 실행 실패", label, e);
        }
    }
}

    6. @Scheduled + batchUpdate vs Spring Batch 선택 기준

    구분 @Scheduled + JdbcTemplate.batchUpdate() Spring Batch
    설정 복잡도 낮음 — 메서드 하나로 끝남 높음 — Job, Step, Reader, Writer Bean 각각 필요
    메모리 안전성 결과 전체를 메모리에 올림 → 대용량 OOM 위험 Chunk 단위 스트리밍 → OOM 없음
    실행 이력 없음 DB에 자동 기록 (성공/실패/재시작 추적 가능)
    재시작/재시도 직접 구현 필요 프레임워크 레벨에서 지원
    적합한 상황 건수가 적고 단순한 작업 수십만 건 이상, 이력 관리, 재시작 필요
    단계적 전환 전략 처음에는 @Scheduled + JdbcTemplate.batchUpdate()로 가볍게 시작한다. 건수가 늘어나거나 OOM이 우려되거나 실행 이력 관리가 필요해지면 Spring Batch로 전환한다. 처음부터 Spring Batch를 도입하면 설정 코드가 많아 오버엔지니어링이 될 수 있다.

    마무리

    비정규화 컬럼은 조회 성능의 강력한 도구지만 관리하지 않으면 데이터 신뢰성을 잃는다. Redis로 실시간에 가깝게 유지하고, Spring Batch 배치가 주기적으로 틀어진 값을 보정하는 하이브리드 전략이 실무에서 성능과 정합성을 함께 챙기는 현실적인 방법이다.

    JdbcTemplate.batchUpdate()는 JPA의 건별 UPDATE와 비교했을 때 압도적으로 빠르지만, 전체 결과를 메모리에 올린다는 제약이 있다. Spring Batch의 JdbcCursorItemReader는 이 문제를 DB 커서 스트리밍으로 해결해서 데이터가 수천만 건이어도 Chunk 크기만큼만 메모리에 올린다. OOM과 속도 사이에서 어느 쪽을 선택할지는 데이터 규모가 결정한다.

    Processor를 생략하고 Reader SQL에서 GROUP BY 집계를 끝내면 쿼리 수를 크게 줄일 수 있다. 하지만 이것은 "쿼리 횟수 감소 vs DB 집계 부하 증가"의 트레이드오프다. 어떤 방식이 더 적합한지는 데이터 규모와 DB 서버 스펙에 따라 실측으로 판단해야 한다. 설계에서 정답은 없고 상황에 맞는 선택만 있다.
