    오늘 목표

    • Redis Pub/Sub의 Publisher, Channel, Subscriber 역할을 이해한다.
    • 일반 Key-Value 방식과 Pub/Sub 방식의 차이를 비교한다.
    • Pub/Sub의 메시지 전달 흐름(구독 → 발행 → 전달 → 수신)을 정리한다.
    • 단일 서버에서 충분했던 STOMP 채팅이 Scale-Out 환경에서 한계를 갖는 이유를 파악한다.
    • Redis Pub/Sub를 중간에 넣어 다중 서버 환경에서도 메시지를 브로드캐스트하는 구조를 이해한다.

    1. Redis Pub/Sub란

    Publisher(발행자)가 메시지를 Channel(통로)에 던지면, 그 Channel을 구독한 Subscriber(구독자)들이 동시에 받아가는 1:N 메시지 전달 방식이다. 발행자는 누가 구독하고 있는지 신경 쓰지 않고 Channel에 메시지를 발행하기만 한다.

    구성 요소 역할
    Publisher (발행자) 메시지를 특정 Channel에 발행한다. 누가 구독하고 있는지 알지 못하고 관심도 없다.
    Channel (채널) 메시지의 통로다. 이름 기반으로 구독과 발행이 연결된다.
    Subscriber (구독자) 관심 있는 Channel을 구독하고, 메시지가 오면 실시간으로 수신한다.
    유튜브로 이해하는 Pub/Sub 유튜버(Publisher)가 영상을 유튜브 채널(Channel)에 업로드하면, 그 채널을 구독한 구독자들(Subscriber)에게 알림이 즉시 전달된다.

    유튜버는 구독자 명단을 직접 관리하지 않는다. 채널에 올리기만 하면 Redis(유튜브 플랫폼)가 구독자 전원에게 알린다. 이것이 Pub/Sub의 핵심 구조다.

    2. Key-Value 방식 vs Pub/Sub 방식

    특성 Key-Value (일반 Redis) Pub/Sub
    메시지 저장 저장됨 — 나중에 조회 가능 저장 안 됨 — 발행 즉시 전달 후 사라짐
    통신 방향 요청 → 응답 (1:1) 발행 → 구독 (1:N), 연결 지속
    수신자 지정 특정 키로 조회 Channel 기반 브로드캐스트
    실시간 여부 조회 시점 기준 (클라이언트가 먼저 요청해야 함) 즉시 전달 (발행되는 순간 구독자가 수신)
    주 사용 목적 캐시, 세션, 카운터 실시간 메시지 전달, 서버 간 통신
    ⚠️ Pub/Sub의 핵심 제약 — 메시지가 저장되지 않는다 Pub/Sub에서 메시지를 발행하는 시점에 구독자가 없거나 오프라인 상태이면, 해당 메시지는 그냥 사라진다. 재전송이 없고 유실된 메시지를 나중에 다시 받을 수 없다.

    메시지 유실이 허용되지 않는 경우(결제 이벤트, 주문 알림 등)에는 Pub/Sub 대신 메시지 큐(Kafka, RabbitMQ)나 Redis Stream을 사용해야 한다.

    3. Pub/Sub 전달 흐름

    Pub/Sub는 구독 → 발행 → 전달 → 수신 순서로 동작한다. 구독이 먼저 이루어진 상태에서 발행이 일어나야 메시지를 받을 수 있다.

    1. Subscriber가 SUBSCRIBE news:tech로 채널 구독을 시작한다.
    2. Publisher가 PUBLISH news:tech "새 기사"로 메시지를 발행한다.
    3. Redis가 news:tech 채널을 구독 중인 모든 Subscriber에게 즉시 전달한다.
    4. 각 Subscriber가 메시지를 수신하고 처리한다.

    Redis CLI로 직접 실습하기 

    # Docker Redis 컨테이너 접속
docker exec -it redis redis-cli

# [터미널 1] 구독자 — 채널을 구독하고 대기
SUBSCRIBE chat

# [터미널 2] 발행자 — 메시지 발행 (새 터미널에서 실행)
docker exec -it redis redis-cli
PUBLISH chat "Hello, World!"

# 터미널 1에 즉시 출력됨:
# 1) "message"
# 2) "chat"
# 3) "Hello, World!"

    4. Scale-Out 환경에서 Pub/Sub가 필요한 이유

    단일 서버 — STOMP만으로 충분하다

    서버가 1대일 때는 모든 WebSocket 연결이 같은 서버에 존재한다. 클라이언트 A가 메시지를 보내면 서버가 메모리에 있는 세션 목록을 순회하며 모든 클라이언트에게 전달할 수 있다. 

    단일 서버 환경

Client A ──┐
Client B ──┤── Server 1 (STOMP) ── 모든 세션에 브로드캐스트
Client C ──┘

Client A의 메시지 → Server 1 → Client B, Client C에게 전달 ✅

    다중 서버 (Scale-Out) — 문제 발생

    서버를 Scale-Out하면 Client A와 Client B가 서로 다른 서버에 연결될 수 있다. Server 1은 자신에게 연결된 세션만 알고 Server 2의 세션을 알지 못한다. Server 1에 메시지가 도착해도 Server 2에 연결된 Client B는 받지 못한다. 

    다중 서버 환경 — 메시지 유실 문제

Client A ── Server 1 (STOMP)
Client B ── Server 2 (STOMP)

Client A가 메시지 전송 → Server 1이 수신
Server 1은 Server 2에 연결된 Client B를 모름
→ Client B는 메시지를 받지 못함 ❌

    Redis Pub/Sub를 중간에 넣어 해결

    모든 서버가 동일한 Redis Channel을 구독하면, 어느 서버에 메시지가 도착해도 Redis가 모든 서버에 전달하고 각 서버가 자신에게 연결된 클라이언트에게 다시 전달한다. 어떤 서버에 연결된 클라이언트든 메시지를 빠짐없이 받을 수 있다. 

    Redis Pub/Sub로 해결한 다중 서버 환경

Client A ── Server 1 ── PUBLISH ──→ Redis ──→ Server 1 SUBSCRIBE ── Client A ✅
Client B ── Server 2                      └──→ Server 2 SUBSCRIBE ── Client B ✅

흐름:
1. Client A가 Server 1에 메시지 전송
2. Server 1이 Redis Channel에 PUBLISH
3. Redis가 해당 Channel을 SUBSCRIBE 중인 Server 1, Server 2 모두에 전달
4. 각 서버가 자신에게 연결된 클라이언트에게 전달
→ Client B도 메시지 수신 ✅
    Redis Pub/Sub가 해결하는 핵심 문제 서버 간 메시지 공유 문제는 서버들이 공유 메모리를 갖지 않기 때문에 발생한다. 각 서버는 자신의 메모리에 있는 WebSocket 세션만 알고, 다른 서버의 세션은 알지 못한다.

    Redis Pub/Sub는 모든 서버가 구독하는 공유 메시지 버스 역할을 한다. 어떤 서버에서 메시지를 발행하든 Redis가 구독 중인 모든 서버에 동시에 전달하므로, 서버 수가 아무리 늘어나도 메시지가 빠짐없이 전달된다.

    마무리

    Redis의 Key-Value 방식은 데이터를 저장하고 나중에 꺼내는 것이 목적이고, Pub/Sub는 데이터를 저장하지 않고 지금 이 순간 연결된 구독자들에게 즉시 전달하는 것이 목적이다. 두 방식은 전혀 다른 문제를 해결한다. Pub/Sub에서 메시지가 저장되지 않는다는 특성은 단점이 아니라 설계 철학이다. "저장 없이 즉시 전달"이 필요한 곳에 쓰고, 유실이 허용되지 않는 곳에는 적합하지 않다.

    Scale-Out 문제는 단순히 서버를 늘릴 때 생기는 부수적인 문제가 아니다. 트래픽이 증가할수록 서버를 수평 확장해야 하는데, 이 순간 단일 서버에서 완벽히 동작하던 실시간 채팅이 갑자기 깨진다. Redis Pub/Sub는 이 문제를 애플리케이션 코드를 거의 바꾸지 않고 해결할 수 있다는 점에서 실용적인 해법이다. 서버가 늘어나도 Redis Channel 구독만 추가하면 새 서버도 메시지를 빠짐없이 수신한다.
