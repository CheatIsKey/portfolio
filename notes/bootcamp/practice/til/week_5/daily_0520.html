    1. 알림 테이블 설계 — 단일 테이블 + ENUM 전략

    알림 종류(좋아요, 팔로우, 댓글, 멘션)마다 별도 테이블을 만드는 대신, 하나의 테이블에 ENUM 타입의 type 필드를 두어 관리한다. 새 알림 종류가 추가되어도 ENUM에 값 하나만 추가하면 되고, 알림 목록 조회 쿼리도 단 한 줄로 충분하다. 

    -- 단일 테이블로 모든 알림을 한 번에 조회
SELECT * FROM notifications
WHERE receiver_id = 1
ORDER BY created_at DESC
LIMIT 20;

    Notification 엔티티 

    @Entity
@Table(
    name = "notifications",
    indexes = {
        // 수정 포인트: 인덱스 순서 설계 원칙
        // ① 등호(=) 조건을 사용하는 컬럼을 앞에 배치 (receiver_id)
        // ② 그 다음 범위/필터 조건 컬럼 배치 (is_read)
        // ③ 정렬 기준 컬럼을 마지막에 배치 (created_at)
        @Index(name = "idx_notification_receiver",
               columnList = "receiver_id, created_at DESC"),
        @Index(name = "idx_notification_receiver_unread",
               columnList = "receiver_id, is_read, created_at DESC")
    }
)
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class Notification extends BaseEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 20)
    private NotificationType type;

    /** 알림을 받는 사람 */
    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "receiver_id", nullable = false)
    private Member receiver;

    /** 알림을 발생시킨 사람 */
    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "sender_id", nullable = false)
    private Member sender;

    /**
     * 알림의 대상 ID
     * - LIKE, COMMENT, MENTION → postId
     * - FOLLOW → null (팔로우는 sender만으로 충분)
     */
    @Column(name = "target_id")
    private Long targetId;

    @Column(name = "is_read", nullable = false)
    private boolean isRead = false;

    @Column(length = 200)
    private String message;

    public static Notification create(NotificationType type, Member receiver,
                                      Member sender, Long targetId, String message) {
        Notification n = new Notification();
        n.type = type;
        n.receiver = receiver;
        n.sender = sender;
        n.targetId = targetId;
        n.message = message;
        return n;
    }

    public void markAsRead() {
        this.isRead = true;
    }
}

// NotificationType.java
public enum NotificationType {
    LIKE,    // 좋아요
    FOLLOW,  // 팔로우
    COMMENT, // 댓글
    MENTION  // 멘션
}
    복합 인덱스 설계 원칙 WHERE 조건에서 등호(=) 조건을 사용하는 컬럼을 인덱스 앞에 배치하고, 범위 조건(BETWEEN, >, <)은 그 다음에, 정렬 기준은 마지막에 배치한다.

    (receiver_id, is_read, created_at) 순서로 설계한 이유:
    1. receiver_id — 등호 조건 (WHERE receiver_id = 1), 가장 앞
    2. is_read — 등호 조건 (WHERE is_read = false), 두 번째
    3. created_at — 정렬 기준 (ORDER BY created_at DESC), 마지막

    자기 자신에 대한 알림(sender.id == receiver.id) 필터링은 DB 인덱스가 아닌 Service 레이어에서 처리한다.

    2. @Async — 알림 비동기 처리

    알림 저장은 사용자의 주요 액션(좋아요, 댓글 작성)과 직접적으로 관련이 없다. 동기로 처리하면 알림 저장이 느릴 때 사용자 응답도 함께 느려진다. @Async로 알림 처리를 별도 스레드에서 수행하면 사용자에게는 빠른 응답을 주고, 알림은 뒤에서 처리할 수 있다.

    ⚠️ 기본 SimpleAsyncTaskExecutor는 절대 사용하지 않는다 Spring의 기본 @Async 실행자는 SimpleAsyncTaskExecutor로, @Async 메서드를 호출할 때마다 새 스레드를 생성하고 재사용하지 않는다. 트래픽이 몰리면 스레드가 무한히 생성되어 OOM으로 서버가 다운될 수 있다. 반드시 스레드 풀을 직접 설정해야 한다.

    AsyncConfig — 전용 스레드 풀 설정 

    // core/config/AsyncConfig.java
@Configuration
@EnableAsync // 수정 포인트: 메인 클래스 또는 설정 클래스에 반드시 선언
public class AsyncConfig implements AsyncConfigurer {

    /**
     * 알림 처리 전용 스레드 풀
     *
     * 카페 비유:
     * - corePoolSize(5)   = 기본 바리스타 5명 (평소 항상 대기)
     * - maxPoolSize(10)   = 최대 바리스타 10명 (바쁠 때 임시 투입)
     * - queueCapacity(50) = 대기 좌석 50석 (바리스타가 다 바쁘면 여기서 대기)
     */
    @Bean(name = "notificationExecutor")
    public Executor notificationExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(10);
        executor.setQueueCapacity(50);
        executor.setThreadNamePrefix("notif-"); // 로그에서 식별 편의
        // 수정 포인트: 풀+큐가 꽉 찼을 때 호출 스레드가 직접 처리 → 작업을 잃지 않음
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }

    /**
     * @Async 메서드에서 예외가 발생하면 기본적으로 조용히 사라진다.
     * AsyncUncaughtExceptionHandler로 전역 예외를 한 곳에서 처리한다.
     */
    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        return (ex, method, params) ->
                log.error("[Async 예외] method={}, message={}", method.getName(), ex.getMessage(), ex);
    }
}
    스레드 풀 동작 흐름 — CallerRunsPolicy까지 요청 1~5건: corePool(5)에서 즉시 처리
    요청 6~55건: Queue(50)에 대기 (corePool이 바쁠 때)
    요청 56~60건: maxPool(10)까지 임시 스레드 확장 (Queue가 찰 때)
    요청 61건~: CallerRunsPolicy — 풀+큐 모두 꽉 참 → 메인 스레드가 직접 처리

    CallerRunsPolicy는 작업을 버리지 않고 잠깐 느려지는 방식을 선택한다. 기본 AbortPolicy는 예외를 던지고 작업을 버리므로 알림이 유실된다.

    3. @TransactionalEventListener — 유령 알림 방지

    ⚠️ @EventListener의 유령 알림 문제 @EventListener는 트랜잭션 안에서 즉시 실행된다. 좋아요 처리가 이벤트를 발행하고 나서 롤백되더라도 알림은 이미 저장된다. 좋아요는 실패했는데 알림은 전송된 상태(유령 알림)가 발생한다.
    @TransactionalEventListener — 커밋 후에만 실행 @TransactionalEventListener는 트랜잭션이 커밋된 후에만 이벤트 핸들러를 실행한다. 비즈니스 로직이 롤백되면 이벤트 핸들러는 실행되지 않으므로 유령 알림이 발생하지 않는다.

    NotificationEventHandler — @Async + @TransactionalEventListener 조합 

    @Component
@RequiredArgsConstructor
@Slf4j
public class NotificationEventHandler {

    private final NotificationRepository notificationRepository;
    private final MemberRepository memberRepository;

    /**
     * @TransactionalEventListener: 트랜잭션 커밋 후에만 실행 → 유령 알림 방지
     * @Async("notificationExecutor"): 전용 스레드 풀에서 실행 → 빠른 응답 보장
     * @Transactional(REQUIRES_NEW): 알림 저장은 독립 트랜잭션으로 처리
     *
     * 이 세 어노테이션의 조합:
     * ① 좋아요 트랜잭션 롤백 → 이 메서드 실행 안 됨 (안전!)
     * ② 트랜잭션 커밋 → 별도 스레드에서 알림 저장 (빠름!)
     * ③ 알림 저장 실패 → 비즈니스 로직에 영향 없음 (격리!)
     */
    @Async("notificationExecutor")
    @TransactionalEventListener
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void handleNotification(NotificationEvent event) {
        // 자기 자신에 대한 알림 무시 (Service 레이어 대신 여기서 처리 가능)
        if (event.isSelfNotification()) {
            log.debug("[알림 무시] 자기 자신에 대한 {} 알림 (senderId={})",
                    event.type(), event.senderId());
            return;
        }

        try {
            Member receiver = memberRepository.getReferenceById(event.receiverId());
            Member sender   = memberRepository.getReferenceById(event.senderId());
            String message  = buildMessage(event.type(), sender);

            notificationRepository.save(
                Notification.create(event.type(), receiver, sender, event.targetId(), message)
            );

            log.info("[알림 저장] {} → {} (type={}, targetId={})",
                    sender.getUsername(), event.receiverId(), event.type(), event.targetId());

        } catch (Exception e) {
            // 수정 포인트: 알림 실패가 원본 비즈니스 로직에 영향 주면 안 됨 → catch로 격리
            log.error("[알림 저장 실패] type={}, receiver={}, sender={}: {}",
                    event.type(), event.receiverId(), event.senderId(), e.getMessage());
        }
    }

    private String buildMessage(NotificationType type, Member sender) {
        String username = sender.getUsername();
        return switch (type) {
            case LIKE    -> username + "님이 회원님의 게시물을 좋아합니다";
            case FOLLOW  -> username + "님이 회원님을 팔로우하기 시작했습니다";
            case COMMENT -> username + "님이 댓글을 남겼습니다";
            case MENTION -> username + "님이 댓글에서 회원님을 언급했습니다";
        };
    }
}

    4. 이벤트 발행 — ApplicationEventPublisher

    비즈니스 로직이 담긴 서비스에서 ApplicationEventPublisher를 주입받아 publishEvent()로 이벤트를 발행하면, 위에서 정의한 핸들러가 자동으로 수신한다. 

    @Service
@RequiredArgsConstructor
public class CommentService {

    private final ApplicationEventPublisher eventPublisher;

    public void addComment(Long loginMemberId, Long postId, CommentRequest request) {
        // ... 댓글 저장 로직 ...

        // ✅ 댓글 알림 발행 — 게시물 작성자에게
        eventPublisher.publishEvent(new NotificationEvent(
                NotificationType.COMMENT,
                post.getWriter().getId(), // receiver = 게시물 작성자
                loginMemberId,            // sender = 댓글 작성자
                postId,                   // target = 게시물
                null
        ));

        // ✅ 멘션 알림 발행 — 멘션된 각 유저에게
        for (Member mentioned : existingMembers) {
            eventPublisher.publishEvent(new NotificationEvent(
                    NotificationType.MENTION,
                    mentioned.getId(),            // receiver = 멘션된 사람
                    comment.getWriter().getId(),  // sender = 댓글 작성자
                    comment.getPost().getId(),    // target = 게시물
                    null
            ));
        }
    }
}
    Redis INCR을 @TransactionalEventListener로 처리해야 하는 이유 Redis INCR을 트랜잭션 내부에서 호출하면, 이후 DB 트랜잭션이 롤백되어도 Redis 값은 그대로 남는다. DB와 Redis 간 데이터 불일치가 발생하는 것이다.

    Redis INCR도 @TransactionalEventListener로 이동하면 트랜잭션 커밋 후에만 실행되어 DB와 Redis가 항상 같은 방향으로 움직인다.

    단, Redis는 순서 보장이 안 되기 때문에 Redis INCR은 @Async 없이 동기로 처리해야 한다. 비동기로 처리하면 여러 스레드가 동시에 INCR을 호출해 순서가 꼬일 수 있다.

    5. 알림 테이블 데이터 관리 전략

    알림은 쓰기가 매우 많은 테이블이다. 그대로 쌓아두기만 하면 SELECT 성능이 급격히 나빠진다. 아래 4가지 전략을 상황에 맞게 조합한다.

    전략 1 — TTL + 배치 삭제 

    // 수정 포인트: cron 표현식 "초 분 시 일 월 요일"
@Scheduled(cron = "0 0 3 * * *")  // 매일 새벽 3시
public void cleanOldNotifications() {
    LocalDateTime threshold = LocalDateTime.now().minusDays(30);
    notificationRepository.deleteAllByCreatedAtBefore(threshold);
}

    전략 2 — 테이블 파티셔닝 (MySQL)

    생성일자를 기준으로 월별 파티셔닝을 적용하면, 오래된 파티션을 DROP PARTITION으로 행 단위 DELETE보다 훨씬 빠르게 정리할 수 있다. 

    CREATE TABLE notifications (
    ...
) PARTITION BY RANGE (YEAR(created_at) * 100 + MONTH(created_at)) (
    PARTITION p202604 VALUES LESS THAN (202605),
    PARTITION p202605 VALUES LESS THAN (202606),
    PARTITION p202606 VALUES LESS THAN (202607)
    -- 수정 포인트: 월별로 파티션 추가
);

    전략 3 — 아카이빙

    최근 30일 알림은 MySQL에 유지하고, 오래된 알림은 S3 / DynamoDB 같은 저비용 스토리지로 이동한다. 사용자가 30일 이전 알림을 조회하려 할 때만 아카이브에서 가져오는 방식이다.

    전략 4 — Redis Sorted Set 최근 알림 캐시

    최근 N개의 알림만 Redis Sorted Set에 유지하고, 알림 목록 API가 먼저 Redis를 조회한다. Cache Miss일 때만 DB로 fallback 한다. 자주 조회되는 최근 알림에 대한 DB 부하를 대폭 줄일 수 있다.

    전략 핵심 방식 장점 단점 / 주의사항
    TTL + 배치 삭제 스케줄러로 N일 이상 된 알림 주기적 DELETE 구현 단순 대량 DELETE 시 DB 부하, 락 경합 가능
    테이블 파티셔닝 월별 파티션 생성, 오래된 파티션 DROP 삭제 속도 매우 빠름 MySQL DDL 변경 필요, 파티션 관리 운영 비용
    아카이빙 오래된 데이터를 저비용 스토리지로 이동 비용 절감, DB 용량 유지 아카이브 조회 로직 추가 구현 필요
    Redis 최근 알림 캐시 최근 N개를 Redis Sorted Set에 유지 알림 목록 조회 DB 부하 감소 Redis 장애 시 DB fallback 로직 필요

    마무리

    알림 시스템에서 핵심은 비즈니스 로직과 알림 처리를 완전히 분리하는 것이다. 알림이 실패하더라도 좋아요나 댓글 작성이 실패해서는 안 되고, 알림 저장이 느리더라도 사용자 응답이 느려서는 안 된다. @TransactionalEventListener + @Async 조합은 이 두 요구를 동시에 만족시킨다.

    @EventListener 대신 @TransactionalEventListener를 선택한 이유를 명확히 이해하는 것이 중요하다. 트랜잭션이 롤백되었는데도 알림이 전송되는 유령 알림 문제는 개발 환경에서는 드러나지 않지만 실제 서비스에서 사용자 경험을 해치는 조용한 버그가 된다. 트랜잭션 커밋 후에만 실행된다는 보장 하나가 이 클래스의 신뢰성을 결정한다.

    알림 테이블은 사용할수록 데이터가 계속 쌓이는 특성이 있다. 처음에는 배치 삭제로 단순하게 시작하고, 데이터가 급증하면 파티셔닝을 도입하고, 조회 부하가 커지면 Redis 캐시를 추가하는 식으로 단계적으로 진화시키는 것이 현실적인 접근이다.
