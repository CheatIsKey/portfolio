    1. 요청 처리 전체 흐름과 모니터링 원칙

    하나의 주문 요청이 처리되는 과정에는 스레드 풀, 분산 락, DB 커넥션 풀이 순서대로 관여한다. 어느 하나가 포화 상태가 되면 뒤의 모든 처리가 대기 상태에 빠지거나 타임아웃이 발생한다. 

    요청 도착
  ↓
[디스패처 서블릿 — 스레드 풀]
                → 유휴 스레드가 없으면 대기 or 거절
  ↓
[컨트롤러]
  ↓
[분산 락 — Redis (productId 기준)]
                → 동일 상품 동시 처리 중이면 락 획득 대기 or 즉시 실패
  ↓
[DB — 커넥션 풀]
                → 커넥션이 모두 사용 중이면 max-wait 초과 시 타임아웃
  ↓
처리 완료 → 커넥션 반납 → 스레드 반납
    감으로 조절하지 않고 모니터링으로 조절한다 스레드 풀과 DB 커넥션 풀의 크기를 임의로 크게 잡는다고 무조건 좋지 않다. 스레드가 너무 많으면 컨텍스트 스위칭 비용이 늘고, 커넥션이 너무 많으면 DB 서버가 버티지 못한다.

    APM(Application Performance Monitoring) 도구로 스레드 대기 시간, 커넥션 대기 시간, 처리량을 실측한 뒤 병목 지점을 확인하고 조절해야 한다. 적정값은 서버 스펙과 트래픽 패턴에 따라 달라지므로 항상 측정 기반으로 결정해야 한다.

    2. 재고 동시성 설계 심화 — Row 분산 저장 전략

    단순 재고 컬럼 방식의 한계

    상품 테이블의 stock 컬럼에 숫자 하나로 재고를 관리하면, 여러 요청이 동시에 들어올 때 Lost Update가 발생한다. 재고 24에서 두 요청이 동시에 2씩 차감하면 최종 재고가 22가 되어야 하는데 22가 두 번 저장된다. 또한 "언제 몇 개가 들어오고 나갔는지"를 추적할 방법이 없다.

    재고 추적 테이블 (진실의 원천)

    재고를 숫자 하나로 수정하는 대신, 입고(IN)와 출고(OUT)를 별도 Row로 기록하고 현재 재고를 SUM으로 계산하는 방식이다. 각 트랜잭션이 같은 Row를 수정하는 대신 새 Row를 INSERT하므로 동시성 충돌이 근본적으로 줄어든다. 

    재고 추적 테이블 예시 (product_id = 1)

| type | quantity | created_at          |
|------|----------|---------------------|
| IN   |   24     | 2026-01-01 09:00:00 |
| OUT  |    2     | 2026-05-01 11:00:00 |
| OUT  |    2     | 2026-05-01 11:00:01 |

현재 재고 = IN 합계(24) - OUT 합계(4) = 20
    ⚠️ 재고 추적 테이블의 한계 — Row 수 폭증 시간이 지날수록 Row 수가 폭발적으로 증가한다. 거래가 많은 상품은 수천만 Row가 쌓인다. 매 조회마다 SUM을 전체 스캔으로 계산하면 성능이 급격히 저하된다.

    대용량 환경에서는 테이블 파티셔닝(날짜 기준 분리)이나 클러스터링 전략이 필요하고, SUM을 매번 계산하는 대신 별도 스냅샷 컬럼을 유지하거나 Redis 캐싱을 결합해야 한다.

    실무 재고 동시성 전략 3가지

    전략 핵심 방식 특징
    DB Row 분산 저장 IN/OUT을 별도 Row로 기록. 현재 재고는 SUM으로 계산 이력 완전 보존, 진실의 원천 확보. Row 폭증 시 파티셔닝 필요
    분산 락 + 비정규화 + DB 락 Redis 분산 락으로 직렬화, 상품 테이블의 재고 컬럼을 낙관적/비관적 락으로 보호 구현 비교적 단순. 락 직렬화로 처리량 제한, DB 부하 집중
    Redis 원자적 카운트 + Write-back Redis가 재고를 원자적으로 관리. 변경분을 주기적으로 재고 추적 테이블에 반영 가장 빠름. Redis 장애 시 추적 테이블에서 SUM 복구 → warm-up으로 Redis 재적재
    Lua 스크립트로 원자성 보장 Redis에서 "재고 조회 → 비교 → 차감"을 Java 코드 3단계로 나눠 실행하면, 명령 사이 틈에 다른 요청이 끼어들어 재고가 음수가 될 수 있다.

    Lua 스크립트는 Redis에서 원자적으로(중단 없이) 실행되어 조회-비교-차감이 하나의 불가분한 작업이 된다. Redis 기반 재고 관리에서 원자성 보장의 핵심 수단이다.
    Redis Limit + Write-back 실무 적용 패턴 선착순 구매 한도(예: 1인 5개 제한)를 Redis Sorted Set으로 관리하고, TTL(예: 1시간)이 지나면 만료되도록 설정한다. 중요한 구매 기록은 즉시 DB 기록 테이블에 Write-back으로 백업하여 Redis가 장애를 일으켜도 데이터가 유실되지 않도록 한다.

    3. 디바운싱 (Debouncing)

    사용자가 입력하는 과정에서 매 키 입력마다 즉시 서버에 요청을 보내는 대신, 일정 시간 동안 추가 입력이 없을 때 비로소 요청을 보내는 기법이다. 주로 검색 자동완성, 유저네임 중복 체크처럼 사용자가 타이핑하는 과정에 실시간 API 호출이 필요한 경우에 적용한다.

    디바운싱이 없을 때 vs 있을 때 디바운싱 없음: 사용자가 "spring"을 입력하면 s, sp, spr, spri, sprin, spring 총 6번 API 호출 발생
    디바운싱 있음 (300ms 기준): 마지막 입력 후 300ms 동안 추가 입력이 없을 때만 호출 → API 1번

    타이핑 속도가 빠른 사용자일수록 효과가 크다. 불필요한 서버 요청을 대폭 줄여 백엔드 부하를 경감하고 사용자 경험도 개선된다. 
    // 프론트엔드 예시 — 검색 입력 디바운싱 (JavaScript)
let debounceTimer;

inputElement.addEventListener('input', (e) => {
    clearTimeout(debounceTimer); // 이전 타이머 취소

    debounceTimer = setTimeout(() => {
        // 300ms 동안 추가 입력이 없을 때만 API 호출
        fetchSearchResults(e.target.value);
    }, 300);
});

    4. 페이징 전략 — Offset vs Cursor

    페이징 전략에 절대적인 정답은 없다. 서비스의 UI/UX와 데이터 특성에 따라 선택해야 한다.

    구분 Offset 방식 Cursor 방식
    쿼리 방식 LIMIT N OFFSET M WHERE id < :cursorId LIMIT N
    성능 페이지 뒤로 갈수록 DB가 OFFSET만큼 앞 데이터를 읽고 버려야 하므로 느려짐 항상 일정 — 앞 데이터를 읽지 않음
    데이터 정합성 조회 중 삽입/삭제 시 중복 노출 또는 누락 발생 가능 ID 기준이라 중복/누락 없음
    특정 페이지 이동 가능 (3페이지, 10페이지 직접 이동) 불가능 (순차 탐색만 가능)
    적합한 UI 번호 기반 페이지 네비게이션 무한 스크롤, 피드 형태
    선택 기준 번호 기반 페이징이 필요한 서비스 (관리자 페이지, 게시판 목록 등): Offset 방식
    무한 스크롤이나 피드 형태의 UI (SNS 피드, 뉴스피드, 채팅 내역 등): Cursor 방식

    5. LIKE '%keyword%' 한계와 전문 검색 엔진

    DB에서 LIKE '%keyword%'로 검색하면 인덱스를 타지 못해 Full Table Scan이 발생한다. 앞부분이 와일드카드(%)로 시작하면 B-Tree 인덱스의 시작 위치를 특정할 수 없기 때문이다. 데이터가 많을수록 성능이 급격히 저하된다.

    ⚠️ LIKE '%keyword%'가 인덱스를 타지 못하는 이유 B-Tree 인덱스는 왼쪽부터 시작하는 접두사 기반으로 탐색한다. LIKE 'spring%'처럼 앞부분이 고정된 경우에는 Range Scan이 가능하다. 하지만 LIKE '%keyword%'처럼 앞부분이 와일드카드인 경우에는 어디서부터 탐색을 시작해야 할지 알 수 없어 인덱스를 전혀 사용하지 못하고 Full Scan이 발생한다.
    Elasticsearch로 해결하는 이유 Elasticsearch는 데이터를 저장할 때 텍스트를 단어(토큰) 단위로 분리하고 역색인(Inverted Index)을 만든다. "spring boot tutorial"을 저장하면 ["spring", "boot", "tutorial"] 각각에 대한 인덱스가 생성된다.

    이후 "boot" 검색 시 B-Tree처럼 전체를 스캔하지 않고, 역색인에서 "boot"가 포함된 문서를 즉시 찾아낸다. 데이터가 아무리 많아도 검색 성능이 일정하고, 형태소 분석, 유사어 검색, 관련도 랭킹 같은 고급 기능도 지원한다.

    단순 조건 검색은 DB + 인덱스로 충분하다. 텍스트 본문 검색(Full-Text Search)이 필요한 경우 Elasticsearch 같은 전문 검색 엔진이 필요하다.
    구분 DB LIKE 검색 Elasticsearch
    인덱스 활용 LIKE '%keyword%'는 Full Scan 역색인(Inverted Index)으로 즉시 탐색
    성능 데이터 증가에 따라 급격히 저하 데이터가 많아도 일정
    검색 기능 패턴 매칭만 가능 형태소 분석, 유사어, 관련도 랭킹 지원
    구축 비용 추가 인프라 불필요 별도 클러스터 운영 비용 발생
    적합한 상황 단순 조건 검색, 데이터량이 적은 경우 상품명/게시글 본문 검색, 자동완성, 대용량 텍스트 검색

    마무리

    오늘 공통으로 흐르는 주제는 "당장 동작하는 것"과 "규모가 커졌을 때도 동작하는 것"의 차이였다. 단순 재고 컬럼은 요청이 적을 때는 잘 동작하지만 동시 요청이 폭증하는 순간 데이터가 사라진다. 재고 추적 테이블은 정합성을 보장하지만 Row가 쌓일수록 파티셔닝 전략이 필요해진다. 어느 설계도 "이게 정답"이라고 말할 수 없고, 현재 트래픽과 미래 규모를 고려한 선택이다.

    디바운싱은 프론트엔드 기술이지만 백엔드 엔지니어가 반드시 알아야 하는 개념이다. 클라이언트가 매 입력마다 API를 호출하면 서버는 불필요한 요청을 수십 배 더 처리해야 한다. 백엔드에서 Rate Limiting을 적용하는 것도 방어책이지만, 프론트에서 디바운싱으로 애초에 요청 수를 줄이는 것이 더 근본적인 해결이다.

    LIKE '%keyword%'가 Full Scan을 유발한다는 사실을 알면, 검색 기능을 설계할 때 Elasticsearch를 처음부터 고려하거나 최소한 언제 도입할지를 미리 계획할 수 있다. DB가 잘하는 것과 검색 엔진이 잘하는 것은 다르다. 역할에 맞는 도구를 선택하는 것이 좋은 설계다.
