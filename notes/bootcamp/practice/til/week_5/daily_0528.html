    87일차 개요

    오늘 목표

    • Redis ZSet(Sorted Set)의 구조와 자동 정렬 원리를 이해한다.
    • 핵심 명령어(ZADD, ZINCRBY, ZREVRANGE)의 사용법을 익힌다.
    • 누적 데이터가 아닌 최근 데이터 기반 실시간 랭킹을 구현하기 위한 Time Window 개념을 파악한다.
    • ZUNIONSTORE로 여러 날짜의 ZSet을 합산해 슬라이딩 윈도우 랭킹을 구성하는 방법을 정리한다.

    1. Redis ZSet (Sorted Set) 이란

    점수(score)에 따라 자동으로 정렬되는 집합 자료구조다. 값을 넣는 순간 이미 정렬된 형태로 저장되기 때문에, 조회할 때 별도의 정렬 연산이 필요 없다. 실시간으로 순위 변동이 잦은 데이터 처리에 압도적으로 유리하다.

    구성 요소 설명
    key ZSet 자체의 이름. 어떤 랭킹인지를 나타낸다. (예: category_rank)
    value (member) ZSet 안에 저장되는 실제 데이터. (예: 카테고리명, 상품 ID)
    score value를 정렬하는 기준이 되는 숫자. 높을수록 상위에 위치한다.
    ZSet이 실시간 랭킹에 압도적으로 유리한 이유 DB에서 ORDER BY COUNT DESC LIMIT 5로 TOP 5를 뽑으려면 매번 집계 연산이 필요하고, 데이터가 많을수록 쿼리가 느려진다.

    ZSet은 데이터를 저장하는 순간 이미 정렬된 상태로 유지된다. 점수가 변경되면 순위가 즉시 반영되고, TOP N 조회도 O(log N) 수준으로 빠르다. 별도 ORDER BY가 필요 없고 순위 계산 비용이 거의 없다.

    2. ZSet 핵심 명령어

    ZADD — 값과 점수 저장

    ZSet에 value와 score를 함께 저장한다. 같은 value가 이미 있으면 score를 덮어쓴다. 

    # 형식: ZADD key score value
ZADD category_rank 1 "A"   # A는 점수 1
ZADD category_rank 2 "B"   # B는 점수 2
# 점수가 높은 값이 자동으로 상위에 정렬됨

    ZINCRBY — 점수 증가

    사용자가 클릭, 조회, 검색 등을 수행할 때 해당 항목의 점수를 올린다. 점수가 변경되는 즉시 순위도 자동으로 반영된다. 

    # 형식: ZINCRBY key increment value
ZINCRBY category_rank 1 "A"
# A의 점수가 1만큼 증가 → 순위가 즉시 재정렬됨

    ZREVRANGE — 상위 N개 조회

    점수가 높은 순서(내림차순)대로 데이터를 가져온다. 인덱스는 0부터 시작하므로 TOP 3은 0~2, TOP 5는 0~4다. 

    # 형식: ZREVRANGE key 시작인덱스 종료인덱스 [WITHSCORES]
ZREVRANGE category_rank 0 2 WITHSCORES   # TOP 3 조회
ZREVRANGE category_rank 0 4 WITHSCORES   # TOP 5 조회

# 응답 예시 (WITHSCORES 포함)
# 1) "B"
# 2) "3"    ← score
# 3) "A"
# 4) "2"    ← score
    명령어 역할 대표 사용 시점
    ZADD value + score 저장 / 갱신 초기 데이터 등록, 집계 결과 일괄 입력
    ZINCRBY 기존 score에 점수 누적 클릭, 검색, 조회 발생 시 실시간 카운트 증가
    ZREVRANGE 점수 높은 순으로 TOP N 조회 랭킹 목록 API 응답

    3. Time Window — 실시간 랭킹의 핵심 개념

    ⚠️ 누적 데이터만 쓰면 실시간 랭킹이 되지 않는다 Key 하나에 계속 ZINCRBY를 누적하면, 오래전부터 쌓인 데이터가 반영되어 과거에 잠깐 유행했던 항목이 영원히 상위를 차지하게 된다. "지금 인기 있는 것"이 아니라 "역대 가장 많이 클릭된 것"이 나온다.

    실시간 랭킹은 최근 N일(또는 N시간)의 데이터만 반영해야 한다. 이를 구현하는 방법이 날짜(또는 시간) 단위로 Key를 분리하는 Time Window 패턴이다. 

    # 날짜별로 Key를 분리해서 데이터 저장
ZADD category_rank:2026-05-26 3 "스포츠"
ZADD category_rank:2026-05-27 5 "패션"
ZADD category_rank:2026-05-27 2 "스포츠"
ZADD category_rank:2026-05-28 4 "스포츠"
ZADD category_rank:2026-05-28 1 "전자제품"

# 이렇게 하면 하루치 데이터를 독립적으로 관리할 수 있다.

    4. ZUNIONSTORE — 여러 날짜의 ZSet 합산

    날짜별로 분리된 ZSet들을 합산해서 최근 N일 랭킹을 만드는 명령어다. 같은 value의 score를 합산(AGGREGATE SUM)하여 새로운 ZSet을 생성한다. 

    # 형식: ZUNIONSTORE 새로운키 집합개수 집합목록... [AGGREGATE SUM]

# 최근 3일치 데이터를 합산해서 새로운 ZSet 생성
ZUNIONSTORE category_rank:recent3 3 \
    category_rank:2026-05-26 \
    category_rank:2026-05-27 \
    category_rank:2026-05-28 \
    AGGREGATE SUM

# 파라미터 설명
# category_rank:recent3  → 새로 생성될 ZSet 이름
# 3                       → 합칠 ZSet의 개수
# 3개의 날짜 키           → 합산 대상 ZSet 목록
# AGGREGATE SUM           → 같은 value의 score를 더하는 방식으로 합산
    # 최근 3일 합산 결과로 TOP 3 조회
ZREVRANGE category_rank:recent3 0 2 WITHSCORES

# 결과 예시
# 1) "스포츠"     → 3 + 2 + 4 = 9
# 2) "패션"       → 5
# 3) "전자제품"   → 1
    Time Window + ZUNIONSTORE 전체 흐름 Step 1 — 데이터 쌓기: 사용자 이벤트(클릭, 검색 등) 발생 시 오늘 날짜 Key에 ZINCRBY
    ZINCRBY category_rank:2026-05-28 1 "스포츠"

    Step 2 — 합산 ZSet 생성: 주기적(예: 1시간마다 스케줄러)으로 최근 N일치 키를 ZUNIONSTORE로 합산
    ZUNIONSTORE category_rank:recent3 3 ... AGGREGATE SUM

    Step 3 — 랭킹 조회: API 요청 시 합산 ZSet에서 TOP N을 ZREVRANGE로 조회
    ZREVRANGE category_rank:recent3 0 4 WITHSCORES

    Step 4 — 오래된 Key TTL 설정: 날짜 Key에 TTL을 걸어 자동으로 만료되도록 설정해 메모리를 관리한다.
    EXPIRE category_rank:2026-05-26 604800 (7일 후 자동 삭제)

    Spring에서 ZSet 사용 예시 

    @Service
@RequiredArgsConstructor
public class RankingService {

    private final RedisTemplate<String, String> redisTemplate;
    private static final String RANK_KEY_PREFIX = "category_rank:";

    // 카테고리 클릭 시 오늘 날짜 Key에 점수 증가
    public void incrementScore(String category) {
        String todayKey = RANK_KEY_PREFIX + LocalDate.now();
        redisTemplate.opsForZSet().incrementScore(todayKey, category, 1);
    }

    // 최근 3일 합산 ZSet에서 TOP N 조회
    public Set<ZSetOperations.TypedTuple<String>> getTopN(int n) {
        String recentKey = RANK_KEY_PREFIX + "recent3";
        // 수정 포인트: 0부터 시작, n-1까지 (TOP N)
        return redisTemplate.opsForZSet().reverseRangeWithScores(recentKey, 0, n - 1);
    }

    // 스케줄러에서 주기적으로 최근 3일치 합산 ZSet 갱신
    @Scheduled(cron = "0 0 * * * *") // 매 정시마다
    public void refreshRecentRanking() {
        String recentKey = RANK_KEY_PREFIX + "recent3";
        List<String> keys = List.of(
                RANK_KEY_PREFIX + LocalDate.now(),
                RANK_KEY_PREFIX + LocalDate.now().minusDays(1),
                RANK_KEY_PREFIX + LocalDate.now().minusDays(2)
        );

        // ZUNIONSTORE 실행 후 TTL 설정 (25시간 — 하루치 여유)
        redisTemplate.opsForZSet().unionAndStore(keys.get(0), keys.subList(1, 3), recentKey);
        redisTemplate.expire(recentKey, Duration.ofHours(25));
    }
}

    마무리

    ZSet은 단순히 "빠른 정렬"을 제공하는 게 아니라, 삽입 시점에 정렬이 완료된다는 것이 핵심이다. DB에서 매번 GROUP BY와 ORDER BY로 집계하는 것과 비교하면, ZSet은 조회 시 정렬 비용이 없어 트래픽이 높아도 응답 속도가 일정하다. 실시간으로 순위가 바뀌는 서비스(검색어 랭킹, 상품 인기도, 게임 점수판 등)에서 ZSet이 사실상 표준인 이유다.

    Time Window 패턴이 없으면 실시간 랭킹이 아니라 역대 누적 랭킹이 된다. 날짜별로 Key를 분리하는 것만으로 "오늘의 인기 카테고리", "이번 주 인기 상품" 같은 기간 기반 랭킹을 자연스럽게 구현할 수 있다. ZUNIONSTORE로 여러 날짜의 ZSet을 합산하면 슬라이딩 윈도우가 완성된다.

    날짜 Key에 TTL을 설정해 오래된 데이터가 자동으로 만료되도록 하는 것도 중요하다. TTL 없이 날짜 Key를 계속 만들면 Redis 메모리가 계속 증가해 결국 장애로 이어진다. 데이터를 쌓는 것만큼 지우는 전략도 함께 설계해야 한다.
