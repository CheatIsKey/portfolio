    오늘 목표

    • 피드 페이징에서 "게시물마다 댓글 수를 함께 보여주는" 문제가 왜 까다로운지 이해한다.
    • Batch Mapping(IN + GROUP BY) 방식의 동작 흐름과 트레이드오프를 파악한다.
    • Correlated Subquery(상관 서브쿼리) 방식의 동작 흐름과 트레이드오프를 파악한다.
    • 두 방식을 비교해 상황에 맞는 선택 기준을 익힌다.
    • QueryDSL transform()이 보일러플레이트 코드를 얼마나 줄여주는지 이해한다.

    1. 문제 상황 — 피드에서 댓글 수를 보여주려면

    인스타그램 피드처럼 게시물 카드에 댓글 수를 함께 표시하는 화면을 만든다고 가정해보자. 게시물 20개를 한 페이지에 보여줄 때, 각 게시물의 댓글 수를 가져오는 방법이 문제다.

    ⚠️ 가장 나쁜 방법 — 루프 안에서 COUNT 쿼리를 N번 날리기 (N+1 문제) 게시물 20개를 가져온 뒤 for문을 돌며 게시물마다 SELECT COUNT(*) FROM comments WHERE post_id = ?를 실행하면 쿼리가 1(게시물 조회) + 20(댓글 수 조회) = 21번 발생한다. 100개면 101번, 1,000개면 1,001번으로 선형적으로 증가한다. 이것이 N+1 문제다.

    이 문제를 해결하는 대표적인 두 가지 전략이 Batch Mapping과 Correlated Subquery다.

    2. Batch Mapping (IN + GROUP BY)

    한 번의 쿼리로 여러 게시물의 댓글 수를 한 방에 가져오는 방법이다. 먼저 게시물 목록을 조회해 ID를 추출하고, 그 ID 목록을 IN 절에 넣어 댓글 수를 일괄 집계한다. 

    // Batch Mapping — QueryDSL 구현 (transform 미사용 버전)
@Override
public Map<Long, Long> countCommentsByPostIds(List<Long> postIds) {
    if (postIds == null || postIds.isEmpty()) return Collections.emptyMap();

    // 쿼리 2번: IN 절 + GROUP BY로 postId별 댓글 수를 한 번에 집계
    // 인스타그램 정책: 피드 카드에는 원댓글 수만 노출 (대댓글 제외) → parent IS NULL
    List<Tuple> results = queryFactory
            .select(comment.post.id, comment.count())
            .from(comment)
            .where(
                comment.post.id.in(postIds),
                comment.parent.isNull()    // 원댓글만 카운트
            )
            .groupBy(comment.post.id)
            .fetch();

    // DB 결과(List<Tuple>)를 WAS 메모리에서 Map으로 변환 — 보일러플레이트
    return results.stream()
            .collect(Collectors.toMap(
                    tuple -> tuple.get(comment.post.id),
                    tuple -> tuple.get(comment.count())
            ));
}

// 서비스에서 사용
public List<FeedResponse> getFeed(Pageable pageable) {
    List<Post> posts = postRepository.findAll(pageable).getContent();
    List<Long> postIds = posts.stream().map(Post::getId).toList();

    // 한 번의 쿼리로 N개의 게시물 댓글 수를 Map으로 받아옴
    Map<Long, Long> commentCountMap = commentRepository.countCommentsByPostIds(postIds);

    return posts.stream()
            .map(post -> FeedResponse.of(
                    post,
                    commentCountMap.getOrDefault(post.getId(), 0L) // 댓글 없으면 0
            ))
            .toList();
}
    Batch Mapping의 특징 — "DB는 단순하게, 조립은 WAS에서" 서브쿼리가 없어 쿼리 자체가 가볍다. DB 옵티마이저가 실행 계획을 세우기 쉽고, 인덱스를 잘 탄다면 매우 빠르고 안정적이다.

    단점은 네트워크 라운드트립이 2회 발생하고, WAS 서버 메모리에서 ID 추출과 Map 병합 코드를 직접 작성해야 해서 코드가 길어진다.

    3. Correlated Subquery (상관 서브쿼리)

    쿼리 한 번에 게시물 조회와 댓글 수 집계를 동시에 끝내는 방법이다. SELECT 절 안에 서브쿼리를 넣어 각 게시물 행마다 댓글 수를 함께 계산한다. 

    // Correlated Subquery — QueryDSL 구현
@Override
public List<FeedRow> findFeedWithCommentCount(Pageable pageable) {
    // 한 번의 쿼리로 게시물 + 댓글 수를 함께 조회
    // 단, SELECT 절의 서브쿼리는 게시물 1건마다 내부적으로 1번씩 실행됨
    return queryFactory
            .select(Projections.constructor(FeedRow.class,
                    post.id,
                    post.content,
                    // 수정 포인트: 상관 서브쿼리 — post.id가 바깥 쿼리의 현재 행을 참조
                    JPAExpressions
                            .select(comment.count())
                            .from(comment)
                            .where(
                                comment.post.id.eq(post.id),
                                comment.parent.isNull()
                            )
            ))
            .from(post)
            .orderBy(post.id.desc())
            .offset(pageable.getOffset())
            .limit(pageable.getPageSize())
            .fetch();
}
    ⚠️ 상관 서브쿼리의 성능 함정 — 인덱스가 없으면 N+1보다 나쁠 수 있다 SELECT 절에 들어가는 상관 서브쿼리는 조회되는 게시물 행마다 내부적으로 한 번씩 실행된다. 게시물 20개면 DB 내부에서 댓글 COUNT 쿼리가 20번 돌아간다.

    단, 이 실행은 DB 내부에서 이루어지므로 네트워크 왕복이 1번이고, comments.post_id에 인덱스가 걸려 있다면 각 COUNT가 매우 빠르다. 하지만 인덱스가 없거나 데이터가 수천만 건에 달한다면 DB CPU 부하가 Batch Mapping보다 심해질 수 있다.

    4. 두 방식 비교

    선택 기준 Batch Mapping 추천: 데이터가 많거나(수백만 건+), 성능이 최우선이거나, DB 부하를 예측 가능하게 유지하고 싶을 때
    Correlated Subquery 추천: 데이터 규모가 크지 않고, 코드 가독성이 우선이며, comments.post_id에 인덱스가 확실히 걸려 있을 때

    인덱스 유무, 데이터 규모, 팀의 코드 컨벤션을 종합해서 결정한다. 어느 쪽도 절대적으로 옳은 답은 없다.

    5. QueryDSL transform() — 보일러플레이트를 없애는 마법

    Batch Mapping에서 가장 귀찮은 부분이 List<Tuple>을 Map으로 변환하는 코드다. transform()은 이 과정을 QueryDSL이 내부적으로 처리해준다. 

    // transform() 사용 버전 — 완성본
@Override
public Map<Long, Long> countCommentsByPostIds(List<Long> postIds) {
    if (postIds == null || postIds.isEmpty()) {
        return Collections.emptyMap();
    }

    // SELECT post_id, COUNT(*) FROM comments
    // WHERE post_id IN (...) AND parent_id IS NULL
    // GROUP BY post_id
    //
    // transform()이 DB 조회 결과를 순회하며 Map<postId, count> 형태로 자동 변환
    Map<Long, Long> result = queryFactory
            .from(comment)
            .where(
                comment.post.id.in(postIds),
                comment.parent.isNull()   // 원댓글만 집계
            )
            .groupBy(comment.post.id)
            .transform(groupBy(comment.post.id).as(comment.id.count()));
            // 수정 포인트: comment.count()는 Long을 반환하지 않을 수 있어
            //             comment.id.count()를 사용하는 것이 더 안전하다

    return result != null ? result : Collections.emptyMap();
}
    transform()이 하는 일 transform(groupBy(키).as(값))은 QueryDSL이 DB 조회 결과를 내부적으로 순회하며 자바 Map 객체로 한 번에 매핑해준다.

    .fetch()로 List<Tuple>을 받은 뒤 stream().collect(Collectors.toMap(...))으로 수동 변환하는 보일러플레이트 코드를 완전히 없애준다.

    단, transform()은 JPQL로 변환되지 않고 메모리에서 처리된다는 점을 알아두어야 한다. 쿼리 실행 후 결과를 Java 메모리에서 그룹핑하는 방식이므로, 결과 행이 매우 많으면 메모리 사용량이 늘어날 수 있다.

    6. 전체 흐름 한눈에 보기

    마무리

    피드 페이징에서 통계 집계(댓글 수, 좋아요 수)를 붙이는 문제는 어떤 방법을 선택하든 트레이드오프가 존재한다. 중요한 것은 "N+1 문제를 반드시 피한다"는 원칙 하에 상황에 맞는 방법을 선택하는 것이다.

    Batch Mapping은 코드가 길어지는 대신 DB에 단순하고 예측 가능한 부하를 준다. 데이터 규모가 크고 성능이 중요할수록 Batch Mapping이 유리하다. Correlated Subquery는 코드가 깔끔하지만 인덱스 설계가 받쳐주지 않으면 오히려 더 느려질 수 있다.

    transform()은 Batch Mapping의 가장 큰 단점인 보일러플레이트 코드를 제거해준다. List<Tuple>을 받아 수동으로 Map으로 변환하는 코드를 직접 작성하지 않아도 된다. 단, transform()은 DB가 아닌 WAS 메모리에서 처리되므로 결과 행이 매우 많을 때는 메모리 사용량을 의식해야 한다.
