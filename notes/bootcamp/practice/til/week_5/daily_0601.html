    1. Kafka란 — 해결하는 핵심 문제 3가지

    Kafka는 이벤트를 순서대로 저장해두는 분산 메시지 스트리밍 플랫폼이다. Consumer가 저장된 메시지를 하나씩 꺼내 처리하므로, 이벤트를 실시간으로 바로 처리하지 않아도 된다. API 서버와 처리 로직 사이의 중간 저장소(버퍼) 역할을 한다.

    문제 1 — 버퍼 역할: API 서버가 죽지 않는다 

    Kafka 없이                              Kafka 있이
사용자 요청 → 서버 → DB/Redis (동기)    a. 요청 받음 → Kafka에 저장 → 즉시 응답 반환
                                        b. Consumer가 준비되면 Kafka에서 꺼내 처리

모든 요청을 바로 처리해야 하므로        API 서버는 무거운 일을 직접 하지 않으므로
트래픽 증가 시 빠르게 한계 도달          트래픽이 몰려도 버틸 수 있음

    문제 2 — 비동기 처리: 데이터 유실이 없다

    Redis가 잠시 느려지거나, DB가 막히거나, 서버가 재배포되더라도 이벤트는 Kafka에 안전하게 저장되어 있어 Consumer가 재시작 후 이어서 처리할 수 있다. Kafka는 메시지가 안전하게 처리되었다고 Commit이 완료되어야 다음 메시지를 읽으며, 처리된 메시지도 즉시 삭제되지 않고 설정된 보관 기간(retention)까지 유지된다.

    문제 3 — 파티션 기반 병렬 처리: 처리량이 폭발적으로 증가한다

    Kafka는 Partition이라는 병렬 처리 단위를 제공한다. 파티션 수만큼 Consumer가 동시에 메시지를 처리할 수 있으므로 수평 확장이 간단하다. 

    파티션 3개  →  Consumer 3개가 동시 처리  →  처리량 3배
파티션 10개 →  Consumer 10개가 동시 처리 →  처리량 10배
    Kafka 한 줄 요약 트래픽이 갑자기 폭발해도 데이터를 절대 잃지 않고, 이벤트를 안전하게 모아서 나중에 여러 Consumer가 병렬로 처리할 수 있게 해주는 중간 저장소다.

    2. 핵심 구성 요소

    구성 요소 역할
    Producer Kafka로 메시지를 보내는 역할. API 서버가 Producer가 된다. 이벤트를 생성해 Topic에 전달한다.
    Consumer Kafka에서 메시지를 읽어 처리하는 역할. Kafka에 쌓인 이벤트를 하나씩 꺼내 처리한다.
    Topic 메시지를 종류별로 분리하는 폴더. 이벤트 종류마다 별도 Topic을 만든다.
    Partition Topic을 여러 칸으로 나눈 병렬 처리 단위. 파티션이 많을수록 동시 처리량이 증가한다.
    Consumer Group 여러 Consumer를 팀으로 묶어 파티션을 나눠 처리한다. 어떤 Consumer가 어떤 파티션을 담당할지 자동 조율한다.
    Offset Consumer가 어디까지 읽었는지를 나타내는 번호. 재시작 후에도 이어서 읽을 수 있다.
    Broker 메시지를 저장하는 Kafka 서버. Producer가 보낸 데이터를 Topic/Partition 형태로 내부에 저장한다.
    Partition과 Consumer 수의 최적 관계 파티션 수 = Consumer 수일 때 가장 효율적이다.

    Consumer가 더 많은 경우: 남는 Consumer는 아무 일도 하지 않고 대기한다 (낭비)
    Consumer가 더 적은 경우: 1개의 Consumer가 2개 이상의 파티션을 담당해 처리 속도가 저하된다

    또한 하나의 파티션은 한 시점에 오직 하나의 Consumer만 처리할 수 있다. 파티션 수가 최대 병렬 처리 Consumer 수의 상한선이 된다. 파티션은 늘릴 수 있지만, 이미 존재하는 파티션을 줄일 수는 없으므로 초기 설계에 주의한다.

    Spring Kafka — Producer / Consumer 기본 예시 

    // Producer — 이벤트를 Topic에 전송
@Service
@RequiredArgsConstructor
public class OrderEventProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public void sendOrderEvent(Long orderId) {
        // 수정 포인트: Topic 이름과 메시지 형식을 실제 도메인에 맞게 교체
        kafkaTemplate.send("order-events", String.valueOf(orderId));
    }
}

// Consumer — Topic에서 메시지를 읽어 처리
@Component
public class OrderEventConsumer {

    // groupId: Consumer Group 이름. 같은 Group 내 Consumer끼리 파티션을 나눠 처리
    @KafkaListener(topics = "order-events", groupId = "order-consumer-group")
    public void handle(String message) {
        // 수정 포인트: 메시지 역직렬화 및 비즈니스 로직 처리
        Long orderId = Long.parseLong(message);
        // ... 주문 처리 로직
    }
}

    3. Kafka 클러스터 — 고가용성 보장

    단일 브로커 구조는 장애에 취약하고 성능에 한계가 있어 실서비스에 적합하지 않다. Kafka 클러스터는 여러 브로커가 하나의 Kafka처럼 동작하는 구조로, 파티션을 여러 브로커에 분산해 병렬 처리 성능을 높이고 장애 시 자동으로 역할을 인계한다.

    Replica — Leader와 Follower

    각 Partition은 원본(Leader)과 복제본(Follower)으로 구성된다. Producer와 Consumer는 항상 Leader와 통신하고, Leader에 장애가 발생하면 Follower 중 하나가 자동으로 Leader로 승격된다.

    역할 설명
    Leader 모든 read/write 요청을 처리한다. Producer와 Consumer가 실제로 통신하는 대상이다. 장애 발생 시 새로운 Leader가 자동 선출된다.
    Follower Leader의 데이터를 그대로 복제한다. Leader 장애 시 자동으로 Leader로 승격된다. Follower 장애 복구 후에는 Leader 데이터를 다시 동기화한다.
    KRaft Controller — Kafka 내부 조율 시스템 Controller는 브로커 생사 확인, Partition Leader 관리, Failover 시 새 Leader 선출, 메타데이터 관리를 담당한다.

    과거에는 ZooKeeper라는 외부 시스템이 이 역할을 했으나, KRaft(Kafka Raft)는 이 기능을 Kafka 내부에 직접 포함시켰다. Raft 합의 알고리즘 기반으로 안정적인 Leader 선출을 지원하며, 외부 의존성을 제거해 운영이 단순해졌다.

    4. Offset과 Commit

    개념 설명
    Offset Consumer가 작업을 어디까지 수행했는지 나타내는 번호. 파티션 내 메시지 위치를 식별한다.
    Commit 이 번호(Offset)까지 읽었다는 것을 Kafka에 기록하는 행위. 서버 재시작 후에도 이어서 읽을 수 있게 된다.

    자동 Commit vs 수동 Commit

    Spring Kafka는 @KafkaListener 뒤에서 Listener Container가 메시지를 가져와 메서드에 전달하고, 기본적으로 적절한 시점에 자동으로 Offset을 Commit한다. 처리 성공 여부와 무관하게 Commit되면 데이터 유실 위험이 있으므로, 중요한 처리에는 수동 Commit을 사용한다. 

    // 수동 Commit 예시 — 처리가 완전히 끝난 후에만 Commit
@KafkaListener(
    topics = "order-events",
    groupId = "order-consumer-group",
    // 수정 포인트: 수동 Commit을 사용하려면 containerFactory에서 ackMode를 MANUAL로 설정
    containerFactory = "manualAckContainerFactory"
)
public void handle(String message, Acknowledgment ack) {
    try {
        Long orderId = Long.parseLong(message);
        // 비즈니스 로직 처리
        processOrder(orderId);

        // 처리가 성공적으로 끝난 경우에만 Offset Commit
        ack.acknowledge();

    } catch (Exception e) {
        // 예외 발생 시 ack.acknowledge()를 호출하지 않으므로
        // 이 메시지는 재처리 대상이 됨
        log.error("주문 처리 실패: {}", message, e);
    }
}

    5. 에러 처리 — DefaultErrorHandler와 DLT

    에러 처리 흐름 

    [1] Kafka Consumer 스레드가 메시지를 poll()로 가져옴
  ↓
[2] @KafkaListener 메서드 호출
  ↓
[3] 비즈니스 로직 실행 중 예외 발생
  ↓
[4] DefaultErrorHandler로 제어권 이동
  ↓
[5] DefaultErrorHandler 결정
    - 재시도할지?
    - 잠깐 기다렸다가 재시도할지? (BackOff)
    - 더 이상 안 되겠다 → 포기(recover)할지?
    - 포기라면 DLT로 보낼지?
  ↓
[6] 최종 결과
    - 재시도 중  → 같은 메시지 다시 리스너에 전달
    - 포기 결정 → Offset Commit + DLT 전송(옵션)
    DefaultErrorHandler 기본 동작 동일 메시지를 10번까지 재시도한다. (백오프 0ms, 연속 시도)
    10번 실패하면 에러 로그를 남기고 해당 레코드를 포기하면서 Offset을 Commit한다.

    Kafka는 멈추지 않고 다음 메시지로 넘어간다. 포기한 메시지가 사라지지 않도록 DLT로 전송하면 나중에 원인을 분석하고 재처리할 수 있다.

    DLT와 재시도 불필요 예외 설정 

    // ErrorHandler 커스터마이징 — DLT 설정 + 재시도 불필요 예외 지정
@Bean
public DefaultErrorHandler errorHandler(KafkaOperations<String, Object> template) {

    // 수정 포인트: 재시도 간격(100ms), 최대 재시도 횟수(3회)
    BackOff backOff = new FixedBackOff(100L, 3L);

    // DLT로 실패 메시지를 전송하는 DeadLetterPublishingRecoverer
    // 원본 Topic명-dlt 형태로 자동 전송됨 (예: order-events → order-events-dlt)
    DeadLetterPublishingRecoverer recoverer =
            new DeadLetterPublishingRecoverer(template);

    DefaultErrorHandler handler = new DefaultErrorHandler(recoverer, backOff);

    // 수정 포인트: 재시도해도 의미없는 예외는 즉시 포기하고 DLT로 보냄
    // 예) 역직렬화 오류, 비즈니스 규칙 위반 등 재시도해도 동일하게 실패하는 경우
    handler.addNotRetryableExceptions(
            IllegalArgumentException.class,  // 잘못된 메시지 형식
            InvalidDataException.class       // 유효하지 않은 데이터
    );

    return handler;
}

// DLT Consumer — 실패한 메시지 확인 및 처리
@KafkaListener(topics = "order-events-dlt", groupId = "order-dlt-group")
public void handleDeadLetter(String message) {
    // 수정 포인트: 운영자 알림, 별도 저장, 수동 재처리 등 판단
    log.warn("[DLT] 처리 실패 메시지 수신: {}", message);
}
    ⚠️ DLT 없이 메시지를 포기하면 데이터가 사라진다 DefaultErrorHandler가 재시도를 포기할 때 DLT 설정이 없으면 해당 메시지는 그냥 버려진다. 왜 실패했는지, 어떤 데이터였는지 추적이 불가능해진다. 실서비스에서는 반드시 DLT를 설정하고, DLT를 주기적으로 모니터링하는 프로세스를 만들어야 한다.

    6. Lag — Consumer 처리 상태 모니터링

    Lag는 Topic에 쌓여있는 메시지 중 해당 Consumer Group이 아직 읽지 못한 메시지 수다. Consumer의 처리 속도가 메시지 발행 속도를 따라가고 있는지를 나타내는 핵심 지표다.

    Lag 상태 의미 대응
    0에 가까움 새로운 메시지가 거의 실시간으로 처리되고 있다. 정상 상태. 유지
    점점 증가 Consumer 처리 속도보다 메시지가 더 빠르게 쌓이고 있다. 병목 신호. Consumer 수 증가 / 처리 로직 최적화 / 메시지 생산량 조절
    급격히 증가 Consumer가 다운됐거나 처리 오류가 반복되고 있다. Consumer 상태 확인 / DLT 모니터링 / 긴급 대응
    Redis vs Kafka — 언제 어떤 것을 쓰는가 Redis: 조회가 빠른 것 — 캐시, 세션, 실시간 카운트, 순위 조회
    Kafka: 저장 / 재처리 / 순서 보장이 필요한 것 — 이벤트 로그, 비동기 처리, 대량 메시지 병렬 처리

    Kafka는 실시간으로 이벤트를 처리하는 것이 아니다. 이벤트를 저장해두었다가 순서대로 안전하게 처리할 수 있도록 도와주는 구조임을 기억해야 한다.

    마무리

    Kafka의 핵심 가치는 API 서버와 처리 로직의 분리다. 서버가 모든 것을 동기로 처리하면 트래픽이 몰릴 때 가장 느린 작업(DB 저장, 외부 API 호출 등)이 전체 응답 시간을 결정한다. Kafka를 중간에 두면 API 서버는 이벤트를 Kafka에 던지고 바로 응답을 반환하며, 실제 처리는 Consumer가 자신의 속도에 맞게 처리한다.

    Partition과 Consumer Group의 관계를 이해하는 것이 Kafka 성능 설계의 핵심이다. 파티션 수가 최대 병렬 처리 한계를 결정하고, Consumer 수를 파티션 수에 맞추어야 낭비 없이 효율적으로 동작한다. 파티션은 늘릴 수 있지만 줄일 수 없으므로 초기 설계에 충분한 여유를 두어야 한다.

    에러 처리에서 DLT는 선택이 아니라 필수다. 재시도를 포기한 메시지가 사라지면 어떤 데이터가 처리되지 않았는지 알 수 없다. DLT를 설정해두면 실패한 메시지를 나중에 분석하고 재처리할 수 있다. Lag 모니터링과 DLT 모니터링을 함께 운영하면 Kafka 기반 시스템의 건강 상태를 실시간으로 파악할 수 있다.
