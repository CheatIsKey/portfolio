    Git & GitHub — 이론 정리

    작성일 2025-10-25 · 분류: GitHub

    개요: Git과 GitHub는 다르다

    Git 로컬 도구

    분산 버전 관리 시스템(DVCS). 프로젝트 스냅샷(커밋)을 로컬에 저장/관리한다. .git 폴더가 저장소의 핵심이며, 파일의 생성/수정/삭제 이력을 스냅샷으로 남긴다.

    GitHub 원격 호스팅

    Git 저장소를 원격 서버에 호스팅하는 서비스. 협업, 코드 리뷰(PR), 이슈/위키, 액션 등 기능을 제공한다. 로컬이 망가져도 원격에 백업된 이력을 복구할 수 있다.

    정정: “웹페이지는 프론트/백엔드 각각 서버가 1대씩 꼭 필요”한 건 아님. 하나의 서버에서 정적 파일과 API를 함께 제공할 수도 있고, 반대로 프론트(정적 호스팅)와 백엔드(API)를 분리 배포할 수도 있다.

    로컬 저장소 시작과 스냅샷

    # 1) 저장소 초기화 (현재 폴더에 .git 생성)
git init

# 2) 변경 파일들을 스테이징(기록 후보)에 올리기
git add .          # 전체
git add src/app.js # 특정 파일

# 3) 커밋(스냅샷 저장) — 메시지는 현재 변경의 의도를 간결히
git commit -m "회원가입 API 최초 구현"

    정정: git init은 기존 이력을 “초기화해서 지운다”가 아니라, 해당 폴더를 Git 저장소로 만들고 .git 메타데이터를 만든다(이미 저장소라면 재초기화 메시지).

    기본 동작을 CRUD 관점으로 보기

    파일 CRUD
    • Create: 파일 생성 → git add → git commit
    • Read: git status (작업/스테이징 상태), git log (커밋 이력)
    • Update: 파일 수정 → git add → git commit
    • Delete: git rm → git commit
    자주 쓰는 확인 명령
    git status           # 변경/스테이징 상태
git log --oneline    # 짧은 커밋 이력
git diff             # 변경 내용 비교
git show <커밋해시>  # 특정 커밋 상세

    주의: rebase는 “기록 삭제”가 아니라 역사를 재작성한다. 잘못 쓰면 참조되지 않는 커밋이 생길 수 있으니, 공유 후 브랜치에서 강제 푸시는 신중히.

    브랜치 CRUD

    1. Create — 새로운 브랜치 만들기 
      git branch feat/login            # 생성만
git checkout -b feat/login       # 생성 + 이동 (또는 git switch -c feat/login)
    2. Read — 목록/차이/이력 보기 
      git branch                       # 로컬 브랜치 목록
git log --oneline --decorate --graph --all
git diff main feat/login         # 두 브랜치 비교
    3. Update — 병합(merge) 
      # 1) 병합 대상이 되는 브랜치로 이동
git checkout main

# 2) 병합 수행 (충돌 없으면 바로 완료, 있으면 파일을 수정하여 해결)
git merge feat/login

# 3) 충돌이 났다면: 파일 수정 → git add → git commit (merge commit 완료)

      겹치는 라인을 동시에 수정했을 때 충돌이 난다. 항상 나는 건 아니다.

    4. Delete — 브랜치 삭제 
      git branch -d feat/login   # 병합 후 안전 삭제
git branch -D feat/login   # 강제 삭제 (병합 전)

    원격과 GitHub

    # 1) 원격 저장소 연결 (보통 'origin'으로 명명)
git remote add origin https://github.com/<USER>/<REPO>.git

# 2) 최초 푸시 (업스트림 설정)
git push -u origin main

# 3) 이후에는 간단히
git push
git pull
    Push / Pull
    • commit: 로컬 스냅샷 저장
    • push: 로컬 → 원격 업로드
    • pull: 원격 → 로컬 가져오기(fetch + merge)
    원격 브랜치 삭제
    git push origin --delete feat/login

    원격 이름(예: origin)을 쓰는 게 일반적이야. URL을 직접 써서 푸시할 수도 있지만 권장 패턴은 아님.

    README.md: 저장소 최상단에 있는 문서로, 프로젝트 개요/사용법/배포/라이선스를 정리해 리포지터리 첫 화면에 노출한다.

    자주 하는 오해 교정

    • Git = 라벨/태그? → 아니고, 스냅샷 기반 버전 관리 시스템이다. 태그(tag)는 커밋에 붙이는 참조 기능 중 하나일 뿐.
    • 기록을 덮어씌운다? → 커밋은 새 스냅샷을 추가하는 개념. 과거 이력은 남아 복구가 가능하다(역사 재작성 명령 제외).
    • 브랜치 병합은 항상 충돌 → 아님. 변경 라인이 겹칠 때만 충돌한다. 그 외엔 fast-forward나 자동 병합됨.

    요약

    • Git은 로컬 버전관리, GitHub는 원격 호스팅/협업 플랫폼.
    • add → commit → push 흐름으로 기록을 남기고 공유한다.
    • 브랜치로 안전하게 기능을 분리, merge로 통합.
    • rebase는 역사 재작성 — 팀 공유 브랜치에선 신중히.
