    Git과 GitHub의 개념

    Git 로컬 버전관리

    Git은 버전 관리 시스템이다. 프로젝트의 변경 이력을 기록하고 관리한다. .git 폴더 안에 모든 버전 정보가 저장되며, 파일의 생성, 수정, 삭제를 모두 기록한다.

    GitHub 원격 저장소

    GitHub는 Git으로 관리되는 프로젝트를 인터넷 서버에 저장하는 공간이다. 여러 사용자가 협업하고 코드를 공유하며, 프로젝트 이력을 안전하게 보관할 수 있다.

    Git은 로컬 컴퓨터에서 작동하고, GitHub는 원격 서버에 저장된 Git 저장소를 관리한다. 두 시스템을 함께 사용하면 버전 관리와 협업을 동시에 수행할 수 있다.

    로컬 저장소 생성과 커밋

    # 저장소 초기화 (.git 폴더 생성)
git init

# 파일 등록 (스테이징)
git add .

# 변경 내용 저장 (커밋)
git commit -m "초기 커밋"

    git init은 폴더를 Git 저장소로 만든다. 기존 파일이 삭제되거나 초기화되지 않는다. git add는 파일의 변경 내용을 임시로 기록하며, git commit은 메시지와 함께 변경 이력을 저장한다.

    파일 기록의 CRUD

    파일 단위 작업
    • Create: 파일 생성 후 git add → git commit
    • Read: git status로 현재 상태 확인, git log로 이력 조회
    • Update: 파일 수정 후 git add → git commit
    • Delete: git rm 파일명 → git commit
    상태 확인 명령
    git status         # 현재 상태 확인
git log --oneline  # 커밋 이력 한 줄 요약
git diff           # 변경 내용 비교
git show <커밋해시>  # 특정 커밋 상세 보기

    커밋(commit)은 변경 이력을 남기는 단위이다. 이전 기록은 삭제되지 않고 새로운 버전이 누적된다.

    브랜치의 생성과 관리

    1. 브랜치 생성 
      git branch login         # 브랜치 생성
git checkout -b login     # 생성 후 이동 (또는 git switch -c login)
    2. 브랜치 조회 
      git branch               # 브랜치 목록 조회
git diff main login       # 브랜치 간 변경 비교
    3. 병합 
      git checkout main         # 기준 브랜치로 이동
git merge login           # login 브랜치를 main에 병합

      병합 중 충돌이 발생하면 충돌 파일을 수정한 뒤 git add와 git commit을 다시 수행한다.

    4. 브랜치 삭제 
      git branch -d login       # 병합된 브랜치 삭제
git branch -D login       # 병합되지 않은 브랜치 강제 삭제

    원격 저장소와 push

    # 원격 저장소 등록
git remote add origin https://github.com/사용자명/저장소명.git

# 커밋 업로드
git push -u origin main

# 이후에는 간단히
git push
git pull

    원격 저장소 이름은 보통 origin으로 지정한다. push는 로컬 저장소의 커밋을 원격 저장소로 업로드하며, pull은 원격 저장소의 최신 커밋을 로컬로 가져온다.

    # 원격 브랜치 삭제
git push origin --delete 브랜치명

    README.md 파일은 프로젝트의 설명서 역할을 한다. 저장소의 대표 문서로, 첫 화면에 자동으로 표시된다.

    기억해야 할 개념

    • Git은 스냅샷 단위로 버전을 관리한다.
    • GitHub는 원격 저장소이자 협업 플랫폼이다.
    • add → commit → push 순서로 변경 내용을 기록하고 업로드한다.
    • branch를 사용하면 기능 단위로 안전하게 코드를 관리할 수 있다.
    • merge 시 코드 충돌이 발생하면 직접 수정하고 다시 커밋한다.
