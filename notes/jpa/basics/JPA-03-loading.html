    프록시와 로딩 전략, 값 타입

    작성일 2025-08-03 · 분류: Spring Boot, JPA

    프록시 동작과 == 비교

    JPA 프록시는 엔티티를 상속하며, 영속성 컨텍스트(1차 캐시)를 먼저 확인하고 없으면 DB를 통해 초기화된다. == 비교에서는 “처음 조회된 대표 인스턴스”로 동일성이 맞춰진다. 즉, 먼저 프록시→나중에 엔티티를 조회해도 둘 다 프록시 인스턴스로 정규화되고, 먼저 엔티티→나중에 프록시를 조회하면 둘 다 실제 엔티티 인스턴스로 정규화되어 ==가 true가 된다.

    즉시 로딩(EAGER)과 N+1 문제

    EAGER는 편해 보이지만 화면마다 불필요한 연관 로딩으로 N+1 문제가 빈번하다. ToOne(@ManyToOne, @OneToOne)은 항상 LAZY를 명시하고, 필요한 경우에만 튜닝(페치 조인, EntityGraph, 배치 페치, DTO)으로 가져온다.

    N+1 해결 전략

    대표 솔루션과 주의점은 다음과 같다.

    • Fetch Join: 한 번에 가져오되, 컬렉션 페치 조인은 페이징 금지(Row 뻥튀기 주의).
    • @EntityGraph: 특정 조회 메서드에 필요한 연관만 동적으로 로딩.
    • Batch Fetch: hibernate.default_batch_fetch_size로 LAZY를 묶음 로딩.
    • DTO 프로젝션: 화면 전용 쿼리로 필요한 필드만 선별.

    영속성 전이(CASCADE)와 고아 객체

    부모가 자식 생명주기를 독점할 때만 cascade = ALL과 orphanRemoval = true를 함께 사용한다. 다른 곳에서 참조하는 엔티티에 전파하면 위험.

    값 타입 — 기본 값 타입

    기본 값 타입(String, 기본형/Wrapper, Enum 등)은 엔티티 식별자가 없고 엔티티와 독립 저장되지 않는다. 바뀌면 새로운 값으로 치환하는 관점으로 다룬다.

    값 타입 — 임베디드 & 불변 설계

    임베디드 타입은 객체 묶음을 값으로 내장한다. 사이드 이펙트를 줄이기 위해 불변으로 설계하고 교체(새 인스턴스 생성)로 변경을 표현한다. 동등성은 equals()로 비교한다.

    값 타입 컬렉션

    @ElementCollection은 별도 테이블에 값들을 저장한다. 변경 시 “통삭제 → 재삽입” 패턴이므로 쓰기 비용과 이력 관리에 유의하자. 가능하면 엔티티로 승격하여 식별자와 생명주기를 분리한다.

    체크리스트

    • 프록시/엔티티 순서와 무관하게 ==는 동일 인스턴스로 정규화된다.
    • ToOne은 항상 LAZY 명시(OneToOne 포함).
    • N+1은 EntityGraph/배치 페치/DTO로 해결, 컬렉션 페치 조인에 페이징 금지.
    • cascade + orphanRemoval은 독점 소유 구조에만 적용.
    • 값 타입은 불변·치환 전략 + equals() 비교.
    • 값 타입 컬렉션은 통삭제 특성·쓰기 비용 고려. 필요 시 엔티티 승격.
