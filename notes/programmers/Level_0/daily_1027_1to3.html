    문제 1 — 약수 구하기

    문제 링크

    프로그래머스 문제 보기

    코드 (IDE 스크린샷)

    해설

    정수 n의 모든 약수를 오름차순의 정수 배열로 반환하는 문제다. 이 풀이에서는 자바의 IntStream을 사용해 1부터 n까지의 구간을 순회하며 n % i == 0인 값만 걸러내어 int[]로 변환한다. (문제의 기본 전제는 양의 정수 n이다.)

    • IntStream.rangeClosed(1, n)으로 1..n 구간의 정수를 생성한다.
    • .filter(i -> n % i == 0)로 n을 나누어 떨어뜨리는 값(약수)만 통과시킨다.
    • .toArray()로 결과를 int[]로 수집한다. (입력 구간이 오름차순이므로 정렬 추가 작업이 필요 없다.)

    예시

    • n = 24 ⟶ [1, 2, 3, 4, 6, 8, 12, 24]
    • n = 13 (소수) ⟶ [1, 13]
    • n = 1 ⟶ [1]

    시간 복잡도

    1부터 n까지 한 번씩만 검사하므로 O(n)이다.

    공간 복잡도

    출력 배열의 크기를 k(약수의 개수)라 하면 결과 저장에 O(k)가 필요하다. 스트림 파이프라인 자체의 추가 공간은 상수(O(1)) 수준이다.

    주의

    • rangeClosed(1, n)는 n < 1일 때 빈 스트림을 만들어 결과가 빈 배열이 될 수 있다. 이 경우는 문제 조건 밖이므로 별도 방어 로직이 필요하다.
    • 스트림을 사용하므로 for 루프 대비 미세한 오버헤드가 있을 수 있으나, 본 문제 규모에서는 무시 가능하다.
    대안 아이디어
    • 루프 기반 구현: for (int i = 1; i <= n; i++)로 순회하며 약수를 리스트에 담고, 마지막에 배열로 변환하면 된다. 동작/복잡도는 동일.
    • √n 최적화: i = 1..⌊√n⌋만 검사하면서 i가 약수면 i와 n/i를 함께 수집해 평균적으로 검사 횟수를 절감(O(√n))할 수 있다. 단, 결과를 오름차순으로 만들려면 마지막에 정렬 또는 두 컨테이너 병합이 필요하다.
    • 박싱/언박싱 회피: 현재 풀이처럼 IntStream을 쓰면 Stream<Integer> 대비 박싱 오버헤드가 없다.

    문제 2 — 한 번만 등장한 문자

    문제 링크

    프로그래머스 문제 보기

    코드 (IDE 스크린샷)

    해설

    문자열 s에서 한 번만 등장하는 문자를 찾아 오름차순으로 이어붙인 문자열을 반환하는 문제다. 문자는 모두 소문자('a' ~ 'z')로 제한되어 있으며, 등장 횟수를 집계하여 1회 등장한 문자만 결과에 포함한다.

    • int[] cnt = new int[26]; — 알파벳 개수(26)만큼의 카운트 배열을 준비한다.
    • for (char c : s.toCharArray()) cnt[c - 'a']++; — 각 문자의 빈도를 1씩 증가시킨다.
    • StringBuilder sb = new StringBuilder(); — 결과 문자열을 효율적으로 구성하기 위한 버퍼다.
    • for (int i = 0; i < 26; i++) 반복하며 cnt[i] == 1인 문자를 sb에 추가한다.
    • 마지막에 sb.toString()을 반환한다. (루프가 'a' → 'z' 순서로 진행되므로 결과는 자동으로 정렬된 형태다.)

    예시

    • s = "abcabcadc" ⟶ "d"
    • s = "abdc" ⟶ "abcd"
    • s = "hello" ⟶ "eho" (각 문자의 등장 횟수: h=1, e=1, l=2, o=1)

    시간 복잡도

    문자열 길이를 N이라 할 때, 한 번 순회하며 빈도를 세고(O(N)), 이후 26개의 문자를 확인하므로 전체 복잡도는 O(N)이다.

    공간 복잡도

    카운트 배열의 크기는 상수(O(26) = O(1))이며, 입력 문자열과 출력 문자열 외의 추가 메모리는 거의 사용되지 않는다.

    주의

    • 입력 문자열은 소문자만 포함한다고 가정한다. (대문자나 숫자 등이 포함되면 c - 'a' 계산이 잘못된다.)
    • 결과가 빈 문자열("")일 수도 있다. (모든 문자가 중복 등장할 때)
    • 문자열 길이가 매우 길어도 카운트 배열은 고정 크기이므로 메모리 사용량이 일정하다.
    • StringBuilder는 문자열 덧붙임(+)보다 효율적이다.
    대안 아이디어
    • 스트림 활용: s.chars()로 문자 스트림을 생성해 빈도수를 Collectors.groupingBy로 세고, 1회 등장 문자만 필터링해 sorted() 후 collect로 이어붙일 수 있다. (단, 코드 길이는 길어지고 속도는 약간 느릴 수 있다.)
    • HashMap 사용: 알파벳 제한이 없거나 유니코드 문자까지 처리해야 한다면, Map<Character, Integer>로 일반화할 수 있다.

    문제 3 — 인덱스 바꾸기

    문제 링크

    프로그래머스 문제 보기

    코드 (IDE 스크린샷)

    해설

    문자열 my_string에서 주어진 두 인덱스 num1, num2의 문자를 서로 교환(swap)한 뒤, 새로운 문자열로 반환하는 문제다. 문자열은 불변(immutable)이므로 직접 수정이 불가능하므로, char[] 배열로 변환 후 교환한 뒤 다시 문자열로 변환한다.

    • char[] array = my_string.toCharArray(); — 문자열을 문자 배열로 변환한다.
    • char tmp = array[num1]; — num1 위치의 문자를 임시 변수에 저장한다.
    • array[num1] = array[num2]; array[num2] = tmp; — 두 문자의 위치를 교환한다.
    • return new String(array); — 수정된 문자 배열을 문자열로 변환해 반환한다.

    예시

    • my_string = "hello", num1 = 1, num2 = 2 ⟶ "hlelo"
    • my_string = "I love you", num1 = 3, num2 = 6 ⟶ "I loev you"
    • my_string = "abcde", num1 = 0, num2 = 4 ⟶ "ebcda"

    시간 복잡도

    문자열 길이를 N이라 할 때, toCharArray()가 O(N), 교환 연산은 O(1), 새로운 문자열 생성도 O(N)이므로 전체 복잡도는 O(N)이다.

    공간 복잡도

    문자 배열이 입력 문자열 길이만큼 추가로 필요하므로 O(N)이다.

    주의

    • num1과 num2는 반드시 0 ≤ num < my_string.length() 범위에 있어야 한다. (그렇지 않으면 ArrayIndexOutOfBoundsException 발생)
    • 두 인덱스가 같으면 교환 전후가 동일하므로 원본 문자열이 그대로 반환된다.
    • 문자열이 불변이므로 원본 my_string 자체는 변경되지 않는다.
    대안 아이디어
    • StringBuilder 사용: StringBuilder sb = new StringBuilder(my_string); 이후 charAt()과 setCharAt()으로 교환하면 배열 변환 없이도 처리 가능하다. (성능 유사)
    • 스트림 방식: IntStream.range(0, my_string.length())로 인덱스를 순회하며 교환된 위치의 문자를 조건식으로 처리해 Collectors.joining()으로 합칠 수 있다. 하지만 코드 복잡도가 높아 권장되지 않는다.
