    문제 1 — 영어가 싫어요

    문제 링크

    프로그래머스 문제 보기

    코드 (IDE 스크린샷)

    해설

    영단어로 이어 붙인 문자열 numbers를 대응하는 숫자들로 치환해 하나의 정수(long)로 변환하는 문제다. 이 풀이에서는 자바의 Map<String,Integer>로 "zero"부터 "nine"까지의 매핑을 준비하고, 각 키에 대해 String#replace로 일괄 치환한 뒤 최종 결과를 Long.parseLong으로 숫자로 변환한다.

    • Map.ofEntries(...)로 "zero"⟶0 … "nine"⟶9 매핑을 생성한다.
    • for (String s : map.keySet()) 루프로 각 영단어 s에 대해 numbers = numbers.replace(s, String.valueOf(map.get(s)))로 치환한다.
    • 모든 영단어가 숫자 문자로 바뀐 후 Long.parseLong(numbers)로 정수로 변환한다.

    예시

    • "onefourzerosixseven" ⟶ 치환 ⟶ "14067" ⟶ 14067L
    • "zerozero" ⟶ "00" ⟶ 0L
    • "nineeightseven" ⟶ "987" ⟶ 987L

    시간 복잡도

    영단어 종류는 상수(10)개이므로 각 replace가 문자열 길이 L에 선형으로 동작한다고 보면 총합은 O(10·L) = O(L)다.

    공간 복잡도

    치환 과정에서 생성되는 중간 문자열에 대해 입력 길이 L에 비례하는 추가 공간이 사용되므로 O(L)이다. 매핑 테이블은 상수 크기(O(1))다.

    주의

    • 치환 안전성: 키들은 서로가 서로의 부분 문자열이 아니며(예: "one"이 "two"에 포함되지 않음), 입력은 영단어를 바로 이어 붙인 형태이므로 순차 치환이 의미를 깨뜨리지 않는다.
    • 예외: numbers가 빈 문자열이거나 형식이 잘못되면 NumberFormatException이 발생할 수 있다. (문제 조건에서는 정상 입력만 주어진다.)
    • 범위: 결과 자릿수가 길 수 있어 int 범위를 넘길 수 있으므로 long으로 파싱하는 것이 안전하다.
    대안 아이디어
    • 버퍼 스캔: 한 글자씩 읽어 StringBuilder에 누적하고, 누적 버퍼가 매핑 키와 일치할 때마다 해당 숫자를 결과에 붙이는 방식. 한 번의 선형 스캔으로 해결(O(L)), 중간 문자열 생성이 줄어든다.
    • 정규식 단일 패스: 패턴 "zero|one|two|...|nine"로 Matcher#replaceAll과 람다(자바 9+ replaceAll(Function))를 사용해 매치마다 숫자로 치환.
    • Trie(접두사 트리): 모든 영단어를 Trie에 넣고 입력을 좌→우로 탐색하며 매치가 끝날 때 숫자를 기록. 키 집합이 더 커지는 일반화 상황에서 유리하다.

    문제 2 — 대문자와 소문자

    문제 링크

    프로그래머스 문제 보기

    코드 (IDE 스크린샷)

    해설

    문자열 my_string의 모든 알파벳 대소문자를 서로 반전시켜 결과 문자열을 반환하는 문제다. 이 풀이에서는 String#chars()로 문자를 코드 포인트 단위로 순회하며, 각 문자의 범위('a'..'z', 'A'..'Z')에 따라 대소문자를 바꾼 뒤 다시 문자열로 합친다.

    • my_string.chars()로 입력 문자열을 IntStream으로 변환한다.
    • .map(ch -> ...)에서 각 문자 ch를 검사하여:
      • 'a' ≤ ch ≤ 'z'이면 Character.toUpperCase(ch)로 대문자로 변환,
      • 'A' ≤ ch ≤ 'Z'이면 Character.toLowerCase(ch)로 소문자로 변환,
      • 그 외(숫자, 공백 등)는 그대로 둔다.
    • .mapToObj(ch -> String.valueOf((char) ch))로 다시 문자로 캐스팅한 뒤,
    • .collect(Collectors.joining())으로 전체를 하나의 문자열로 합친다.

    예시

    • "cccCCC" ⟶ "CCCccc"
    • "abCdEfghIJ" ⟶ "ABcDeFGHij"
    • "123aBc!" ⟶ "123AbC!"

    시간 복잡도

    문자열의 길이를 n이라 하면, 각 문자를 한 번씩만 순회하므로 O(n)이다.

    공간 복잡도

    결과 문자열을 새로 생성하므로 O(n)의 추가 공간이 필요하다. 스트림 파이프라인 자체의 부수 공간은 상수(O(1)) 수준이다.

    주의

    • 비영문자 처리: 숫자, 공백, 특수문자 등은 그대로 유지된다.
    • 문자 범위: chars()는 UTF-16 코드 단위로 순회하므로, 알파벳 외 다국어 문자(예: 한글, 일본어)는 변환 대상이 아니다.
    • 대소문자 판정을 아스키 코드 범위로 직접 비교하여 Character.isLowerCase, isUpperCase보다 약간 더 빠르다.
    대안 아이디어
    • Character 클래스 활용: Character.isLowerCase(ch), Character.isUpperCase(ch)로 명확하게 판별할 수 있다.
    • toCharArray() 기반 루프: 스트림 대신 for (char c : my_string.toCharArray()) 루프로 순회하며 StringBuilder에 변환 문자를 추가하면 성능상 이점이 있다.
    • 비스트림 버전은 오토박싱/언박싱이 없어 대규모 문자열에서 미세한 속도 향상이 가능하다.

    문제 3 — 암호 해독

    문제 링크

    프로그래머스 문제 보기

    코드 (IDE 스크린샷)

    해설

    문자열 cipher에서 일정한 간격(code)마다 문자를 추출하여 이어 붙인 결과 문자열을 반환하는 문제다. 이 풀이에서는 for 루프를 사용하여 인덱스가 code - 1, 2·code - 1, 3·code - 1 …인 문자들을 순서대로 StringBuilder에 추가한다.

    • StringBuilder sb = new StringBuilder();로 결과를 누적할 버퍼를 생성한다.
    • for (int i = code - 1; i < cipher.length(); i += code)로 주어진 간격마다 인덱스를 증가시키며 순회한다.
    • sb.append(cipher.charAt(i));로 해당 위치의 문자를 추가한다.
    • 루프 종료 후 sb.toString()으로 결과 문자열을 반환한다.

    예시

    • cipher = "dfjardstddetckdaccccdegk", code = 4 ⟶ "attack"
    • cipher = "pfqallllabwaoclk", code = 2 ⟶ "fallback"
    • cipher = "abcde", code = 3 ⟶ "c"

    시간 복잡도

    문자열 길이를 n, 간격을 k라 하면 n/k개 문자를 추출하므로 시간 복잡도는 O(n/k), 즉 O(n) 수준이다.

    공간 복잡도

    결과 문자열의 길이를 m = ⌊n/k⌋라 할 때 O(m)의 추가 공간이 필요하다. StringBuilder는 문자열 결합 시 불필요한 객체 생성을 방지하므로 효율적이다.

    주의

    • 인덱스 시작: 문제에서 문자는 1부터 시작하지만, 자바 문자열 인덱스는 0부터이므로 code - 1에서 시작해야 한다.
    • 유효성 검사: code가 1보다 작거나 cipher보다 크면 결과가 빈 문자열이 될 수 있다. (문제 조건에서는 정상 입력이 보장됨)
    • StringBuilder 사용
      • : 문자열을 반복 결합할 때 + 연산보다 효율적이다.
    대안 아이디어
    • 스트림 방식: IntStream.range(code - 1, cipher.length())에서 .filter(i -> (i - code + 1) % code == 0)으로 간격 필터링 후 .mapToObj(cipher::charAt)과 Collectors.joining()으로 결합할 수 있다.
    • 슬라이스 연산: 루프 대신 while문 또는 재귀로 i += code 이동을 수행할 수도 있으나 복잡도는 동일하다.
