    문제 1 — 이진수 더하기

    문제 링크

    프로그래머스 문제 보기

    코드 (IDE 스크린샷)

    해설

    두 이진 문자열 bin1, bin2를 각각 2진수로 해석해 정수로 변환한 뒤, 합을 다시 2진 문자열로 바꿔 반환하는 풐다. 자바 표준 라이브러리의 진법 변환 API를 그대로 활용해 구현을 단순화한다.

    • Integer.parseInt(bin1, 2) / Integer.parseInt(bin2, 2)로 각 문자열을 2진수 기준 정수(int)로 파싱한다.
    • 두 정수를 더한다: num1 + num2.
    • Integer.toBinaryString(sum)으로 합을 2진 문자열로 변환해 반환한다. 이 메서드는 선행 0을 자동으로 제거하므로 별도 처리 없이 요구 형식과 일치한다.

    예시

    • bin1 = "10", bin2 = "11" ⟶ 합: 5 ⟶ 반환: "101"
    • bin1 = "0", bin2 = "0" ⟶ 합: 0 ⟶ 반환: "0"
    • bin1 = "1010", bin2 = "111" ⟶ 합: 17 ⟶ 반환: "10001"

    시간 복잡도

    길이를 각각 N, M이라 하면 파싱과 변환이 각 자리(문자)를 한 번씩 훑으므로 전체적으로 O(N + M)이다.

    공간 복잡도

    입력/출력을 제외한 추가 사용 공간은 상수 수준이므로 O(1)이다.

    주의

    • 오버플로 가능성: Integer.parseInt(..., 2)는 32비트 부호 있는 정수 범위 (-231 ~ 231-1)를 넘는 값을 파싱할 수 없다. 즉, 이진 문자열이 너무 길거나(대략 31비트 초과의 양수) 두 수의 합이 int 범위를 넘으면 예외/잘못된 결과가 날 수 있다. 문제 제약이 int 범위 내라면 안전하지만, 그 외에는 아래 대안을 고려한다.
    • 선행 0: 결과에 선행 0이 필요 없다면(일반적인 요구사항) 현재 방식이 적합하다. 선행 0을 유지해야 하는 요구가 있다면 별도 처리 필요.
    대안 아이디어
    • BigInteger 사용(임의 정밀)
      import java.math.BigInteger;

      class Solution {
        public String solution(String bin1, String bin2) {
          var a = new BigInteger(bin1, 2);
          var b = new BigInteger(bin2, 2);
          return a.add(b).toString(2);
        }
      }
      입력 길이에 제한 없이 안전하다. 복잡도는 자리수 기준 O(N + M).
    • 수동 자리올림(캐리) 덧셈 — 문자열만으로 처리(외부 진법 변환 없이).
      class Solution {
        public String solution(String bin1, String bin2) {
          int i = bin1.length() - 1, j = bin2.length() - 1, carry = 0;
          var sb = new StringBuilder();
          while (i >= 0 || j >= 0 || carry != 0) {
            int x = (i >= 0) ? bin1.charAt(i--) - '0' : 0;
            int y = (j >= 0) ? bin2.charAt(j--) - '0' : 0;
            int sum = x + y + carry;
            sb.append(sum & 1);
            carry = sum >> 1;
          }
          return sb.reverse().toString();
        }
      }
      임의 길이에서도 O(N + M)로 동작하며 라이브러리 범위 제한이 없다.
    • long 사용(입력이 63비트 이내일 때만)
      long a = Long.parseLong(bin1, 2);
      long b = Long.parseLong(bin2, 2);
      return Long.toBinaryString(a + b);
      합이 long 범위를 넘지 않는다는 전제에서만 안전.

    문제 2 — 치킨 쿠폰

    문제 링크

    프로그래머스 문제 보기

    코드 (IDE 스크린샷)

    해설

    서비스 치킨 주문 수 chicken이 주어질 때, 10마리 주문마다 1마리씩 쿠폰 보상 치킨을 받을 수 있다고 할 때, 받을 수 있는 총 서비스 치킨 수를 구하는 문제다. 단, 서비스 치킨도 주문 치킨처럼 쿠폰을 제공하므로 재귀적(반복적) 교환이 가능하다.

    제시된 풀이에서는 “치킨 한 마리당 1장 쿠폰, 10장마다 1마리 서비스” 규칙을 다음과 같이 치킨 수를 9씩 줄이며 서비스 치킨 수를 센다. 즉, 10마리를 주문하면 1마리 서비스(총 11마리)가 되므로, “10마리를 줄일 때마다 9마리만 실제 소비”한다는 성질을 이용한다.

    • while (chicken / 10 != 0) — 쿠폰으로 교환 가능한 동안 반복한다.
    • cnt++ — 서비스 치킨을 한 마리 추가한다.
    • chicken -= 9 — 실제 쿠폰 교환 후 남는 쿠폰/치킨 수를 갱신한다. (즉, 치킨 한 마리를 새로 얻지만 쿠폰은 10장 사용 → 순감 9장)
    • 반복 종료 후 누적된 cnt가 총 서비스 치킨 수가 된다.

    예시

    • chicken = 100
      ⟶ 100 → 91 → 82 → 73 → 64 → 55 → 46 → 37 → 28 → 19 → 10 → 1 → 종료
      총 11회 교환 → 반환: 11
    • chicken = 1081
      ⟶ 총 서비스 치킨 120 (문제 예시 결과와 일치)

    시간 복잡도

    매 반복에서 치킨 수가 9 감소하므로 반복 횟수는 O(chicken / 9) 수준이다. chicken ≤ 106이라면 충분히 빠르다.

    공간 복잡도

    누적 카운터와 입력 변수를 제외하면 별도 자료구조를 사용하지 않으므로 O(1).

    주의

    • 정수 연산 정확성: chicken / 10 비교 대신 chicken >= 10을 써도 된다. (명시적으로 “쿠폰 10장 이상” 조건 표현 가능)
    • 직관성: 이 풀이의 핵심은 “10마리마다 1마리 서비스 → 9마리만 실제 소모” 관계를 수식화한 것. 따라서 간단한 수식으로도 동일 결과를 구할 수 있다.
    대안 아이디어
    • 정석적인 쿠폰 시뮬레이션 (더 직관적)
      class Solution {
        public int solution(int chicken) {
          int coupon = chicken;
          int service = 0;
          while (coupon >= 10) {
            int newChick = coupon / 10;
            service += newChick;
            coupon = newChick + (coupon % 10);
          }
          return service;
        }
      }
      쿠폰/치킨 교환 과정을 직접 모사한다. 복잡도는 동일 O(chicken).
    • 수학적 식으로 단순화:
      서비스 치킨 총합은 chicken / 9의 내림값과 동일하다. (int answer = chicken / 9; 단, chicken % 9 == 0이면 1 줄여야 함은 없음)

    문제 3 — 로그인 성공?

    문제 링크

    프로그래머스 문제 보기

    코드 (IDE 스크린샷)

    해설

    사용자 입력 정보 id_pw와 회원 데이터베이스 db가 주어졌을 때, 입력한 ID와 비밀번호의 일치 여부를 판별하여 로그인 결과를 문자열로 반환하는 문제다. 조건은 다음 세 가지 중 하나에 해당한다:

    • ID와 PW 모두 일치 → "login"
    • ID는 존재하지만 PW 불일치 → "wrong pw"
    • ID 자체가 존재하지 않음 → "fail"

    제시된 풀이에서는 데이터베이스를 순차 탐색하며 ID를 찾고, 비밀번호를 검증하는 구조로 구현했다.

    • String id = id_pw[0], String pw = id_pw[1] — 입력 계정 정보를 분리한다.
    • for문으로 db의 각 행을 순회하며 db[i][0]이 id와 일치하는지 확인한다.
    • 일치하는 ID를 찾으면, db[i][1]과 pw 비교:
      • 같으면 즉시 "login" 반환
      • 다르면 즉시 "wrong pw" 반환
    • 끝까지 탐색했는데 일치하는 ID가 없다면 "fail" 반환

    예시

    • id_pw = ["meosseugi", "1234"], db = [["rardss", "123"], ["yyoom", "1234"], ["meosseugi", "1234"]]
      → ID/PW 모두 일치 ⟶ "login"
    • id_pw = ["programmer01", "15789"], db = [["programmer02", "111111"], ["programmer01", "114514"]]
      → ID 일치, PW 불일치 ⟶ "wrong pw"
    • id_pw = ["rabbit04", "98761"], db = [["rabbit00", "11111"], ["rabbit01", "22222"]]
      → ID 없음 ⟶ "fail"

    시간 복잡도

    데이터베이스 크기를 N이라 하면, 최악의 경우 모든 항목을 순회하므로 O(N). 각 비교 연산은 문자열 비교로 O(L)(L: 평균 문자열 길이)이지만, 일반적으로 매우 작아 상수 시간으로 간주 가능하다.

    공간 복잡도

    입력 외 추가 공간은 없으므로 O(1).

    주의

    • 순차 탐색: 데이터베이스가 커질 경우 비효율적이므로, 실제 서비스에서는 HashMap 사용이 권장된다.
    • 즉시 반환 구조: 일치 여부가 판명되는 즉시 결과를 반환하므로 불필요한 반복을 수행하지 않는다.
    • 비밀번호 비교: equals()를 반드시 사용해야 한다. == 비교는 참조 비교로 오작동한다.
    대안 아이디어
    • HashMap 사용 — 탐색 효율 개선
      import java.util.*;

      class Solution {
        public String solution(String[] id_pw, String[][] db) {
          Map<String, String> map = new HashMap<>();
          for (String[] pair : db) map.put(pair[0], pair[1]);
          String id = id_pw[0], pw = id_pw[1];
          if (!map.containsKey(id)) return "fail";
          return map.get(id).equals(pw) ? "login" : "wrong pw";
        }
      }
      해시 탐색을 사용하므로 평균 O(1) 시간에 결과를 얻을 수 있다.
    • 스트림 API 버전 — 간결하지만 효율은 동일
      import java.util.*;
      import java.util.stream.*;

      class Solution {
        public String solution(String[] id_pw, String[][] db) {
          return Arrays.stream(db)
            .filter(arr -> arr[0].equals(id_pw[0]))
            .findFirst()
            .map(arr -> arr[1].equals(id_pw[1]) ? "login" : "wrong pw")
            .orElse("fail");
        }
      }
