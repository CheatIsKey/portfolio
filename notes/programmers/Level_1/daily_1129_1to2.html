    문제 1 — 카드 뭉치

    문제 링크

    프로그래머스 문제 보기

    코드 (IDE 스크린샷)

    해설

    카드 뭉치 (프로그래머스 159994) 문제는
    두 개의 카드 더미 cards1, cards2가 주어졌을 때,
    이 두 더미의 맨 앞에서부터 카드들을 순서대로 사용해서
    목표 단어 배열 goal을 정확한 순서로 만들 수 있는지 판별하는 문제다.

    규칙은 다음과 같다.

    • 각 카드 뭉치에서 카드는 위(맨 앞)에서부터만 사용할 수 있다.
      • 사용한 카드는 버려지고, 다시 되돌릴 수 없다.
    • goal의 단어들을 왼쪽에서 오른쪽 순서대로 만들어야 한다.
    • 현재 만들어야 할 단어가
      • cards1의 맨 앞 카드와 같으면 → cards1에서 한 장 사용
      • 아니고 cards2의 맨 앞 카드와 같으면 → cards2에서 한 장 사용
      • 둘 다 아니면 → 더 이상 진행 불가, 실패

    결국 goal 배열을 순서대로 훑으면서,
    각 단어를 두 카드 뭉치의 “앞에서부터만” 가져올 수 있는지 보는 간단한 그리디 + 두 포인터 문제다.

    전략 개요

    • 각 카드 뭉치에 대해 “현재 맨 위 카드 위치”를 가리키는 인덱스를 둔다.
      • cards1Idx, cards2Idx
      • 둘 다 처음에는 0 (맨 앞 카드)에서 시작한다.
    • goal을 왼쪽부터 하나씩 보면서:
      • 현재 만들고 싶은 단어를 s라고 하면,
      • 먼저 cards1의 현재 카드(cards1[cards1Idx])와 비교
      • 일치하면 → cards1Idx를 한 칸 증가 (그 카드 사용)
      • 아니면 cards2의 현재 카드와 비교해서 일치하면 → cards2Idx 증가
      • 둘 다 아니면 → 더 이상 이 단어를 만들 수 없으므로 실패
    • 끝까지 실패 없이 모든 goal의 단어를 처리했다면 성공이다.

    핵심 로직

    • 인덱스 및 상태 변수 초기화 
      public String solution(String[] cards1, String[] cards2, String[] goal) {
          int cards1Idx = 0;
          int cards2Idx = 0;
          boolean ok = true;
      • cards1Idx: cards1에서 아직 사용하지 않은 맨 앞 카드의 위치.
      • cards2Idx: cards2에서 아직 사용하지 않은 맨 앞 카드의 위치.
      • ok: 끝까지 조건을 만족하는지 여부.
    • goal 배열 순회 
          for (String s : goal) {
              if (cards1Idx < cards1.length && cards1[cards1Idx].equals(s)) {
                  cards1Idx++;
              } else if (cards2Idx < cards2.length && cards2[cards2Idx].equals(s)) {
                  cards2Idx++;
              } else {
                  ok = false;
                  break;
              }
          }
      • for (String s : goal):
        • 필요한 단어들을 순서대로 처리한다.
      • 먼저 cards1에서 꺼낼 수 있는지 확인:
        • cards1Idx < cards1.length → 아직 카드가 남아 있어야 하고,
        • cards1[cards1Idx].equals(s) → 현재 카드가 필요한 단어와 같아야 한다.
        • 둘 다 만족 시 → cards1Idx++로 해당 카드를 사용 처리.
      • 그렇지 않다면 cards2 확인:
        • 조건은 동일하게 길이 체크 + 문자열 비교.
        • 만약 cards2의 현재 카드가 s와 같으면 → cards2Idx++.
      • 둘 다 매칭되지 않으면:
        • 어느 카드 뭉치에서도 현재 단어를 꺼낼 수 없다는 뜻이므로 → 실패.
        • ok = false로 두고 루프를 종료한다.
    • 결과 반환 
          return ok ? "Yes" : "No";
      }
      • 끝까지 문제 없이 단어를 만들 수 있었다면 ok == true → "Yes" 반환.
      • 중간에 한 번이라도 막혔다면 ok == false → "No" 반환.
      • 문제에서 요구하는 결과 문자열("Yes"/"No") 그대로 반환한다.

    시간 복잡도

    goal 배열의 길이를 G라고 하면:

    • 각 goal 원소를 한 번씩만 확인하고,
    • 각 원소마다 cards1, cards2의 현재 인덱스에서 상수 시간 비교만 수행한다.

    따라서 전체 시간 복잡도는 O(G)이다.
    두 카드 뭉치의 총 길이 합도 사실상 G와 비슷한 수준이므로 매우 효율적이다.

    공간 복잡도

    • 추가로 사용하는 것은 정수 인덱스 두 개와 불리언 하나뿐이다.
    • 입력 배열 외의 추가 메모리는 상수 개뿐이므로, 공간 복잡도는 O(1)이다.

    주의

    • 순서 유지:
      • 각 카드 뭉치에서 중간 카드를 건너뛰고 뒤의 카드를 쓸 수 없다.
      • 반드시 “맨 앞” 카드부터 하나씩 순서대로만 사용할 수 있다는 점이 중요하다.
    • 인덱스 범위 체크:
      • 항상 cards1Idx < cards1.length, cards2Idx < cards2.length를 먼저 확인한 뒤
        배열에 접근해야 한다.
      • 코드에서 두 조건을 &&로 묶어 순서대로 검사하기 때문에 인덱스 범위를 벗어나지 않는다.
    • 비교 순서:
      • 항상 cards1를 먼저 보고, 안 되면 cards2를 본다.
      • 문제에서 어느 카드 뭉치를 먼저 써야 한다는 제한은 없지만,
        현재 구현처럼 “cards1 먼저, 안 되면 cards2”로 고정해도
        가능한 경우만 잘 찾아내면 되므로 정답에 영향을 주지 않는다.

    포인트 요약

    • 각 카드 뭉치를 큐처럼 앞에서부터만 꺼낼 수 있다고 생각한다.
    • goal을 순서대로 보면서, 현재 단어가 cards1 앞 또는 cards2 앞과 일치하는지 확인한다.
    • 둘 중 어디에서도 꺼낼 수 없으면 즉시 실패하고 "No"를 반환한다.
    • 끝까지 문제없이 진행되면 "Yes"를 반환한다.
    • 두 인덱스와 한 개의 불리언만으로 해결 가능한 간단한 그리디 + 두 포인터 문제다.

    문제 2 — 둘만의 암호

    문제 링크

    프로그래머스 문제 보기

    코드 (IDE 스크린샷)

    해설

    둘만의 암호 (프로그래머스 155652) 문제는
    소문자 알파벳으로 이루어진 문자열 s가 있을 때,
    특정 알파벳들(skip)을 건너뛰면서 각 문자를 index만큼 앞으로 이동시켜
    새로운 문자열을 만드는 문제다.

    규칙을 정리하면:

    • 알파벳은 a ~ z 26글자를 원형으로 연결한 것처럼 생각한다.
      • z 다음은 다시 a로 돌아간다.
    • skip 문자열에 포함된 알파벳은 모두 건너뛰는 대상이다.
      • 이 알파벳들은 암호를 만들 때 셀 수 없다.
    • s의 각 문자마다:
      • 알파벳을 한 칸씩 앞으로 이동시키되,
      • skip에 포함되지 않은 알파벳을 정확히 index번 만나면 거기서 멈춘다.
      • 그 멈춘 알파벳이 해당 문자의 치환 결과가 된다.

    예를 들어, s = "aukks", skip = "wbqd", index = 5라면
    각 문자마다 건너뛰는 과정을 거쳐 새로운 문자열을 만들어야 한다.

    전략 개요

    • 건너뛰어야 하는 알파벳을 빠르게 확인하기 위해,
      • 크기 26의 boolean 배열 ban을 만든다.
      • skip에 포함된 알파벳에 해당하는 인덱스를 true로 표시한다.
    • s의 각 문자에 대해:
      • 현재 위치를 알파벳 인덱스(0~25)로 바꾼 뒤,
      • 원형 배열(모듈러 연산)로 한 칸씩 이동하면서 skip에 포함되지 않은 글자를 index번 찾을 때까지 전진한다.
      • 마지막으로 도달한 위치를 다시 문자로 바꿔 결과 문자열에 붙인다.

    핵심 로직

    • skip 문자 전처리 
      boolean[] ban = new boolean[26];
      for (char c : skip.toCharArray()) {
          ban[c - 'a'] = true;
      }
      • ban[0] → 'a', ban[1] → 'b', … , ban[25] → 'z'.
      • skip에 포함된 문자는 true로 표시해, 이후에 “이 알파벳은 세면 안 된다”는 것을 빠르게 알 수 있다.
    • 각 문자 치환 
      StringBuilder sb = new StringBuilder();
      
      for (char c : s.toCharArray()) {
          int pos = c - 'a';
          int cnt = 0;
      
          while (cnt < index) {
              pos = (pos + 1) % 26;
      
              if (!ban[pos]) {
                  cnt++;
              }
          }
      
          char ch = (char)('a' + pos);
          sb.append(ch);
      }
      • pos = c - 'a':
        • 현재 문자를 0~25 범위의 인덱스로 변환한다.
      • cnt:
        • 건너뛰지 않고 실제로 세어야 하는 알파벳을 몇 번 거쳤는지 카운트하는 변수.
      • while (cnt < index):
        • 현재 문자에서 시작해, 유효한 알파벳을 index번 만날 때까지 계속 전진한다.
      • pos = (pos + 1) % 26;:
        • 알파벳을 한 칸 앞으로 이동한다.
        • % 26 덕분에 z에서 한 칸 더 가면 다시 a로 순환한다.
      • if (!ban[pos]) { cnt++; }:
        • 현재 위치의 알파벳이 skip에 포함되지 않은 경우에만 유효한 한 칸으로 본다.
        • skip에 포함된 알파벳이면 무시하고 계속 다음으로 넘어간다.
      • 반복이 끝났을 때:
        • cnt == index 상태이고, pos에는 최종 도착 알파벳의 인덱스가 들어 있다.
        • ch = (char)('a' + pos)로 다시 문자로 변환해 결과에 추가한다.
    • 최종 결과 반환 
      return sb.toString();
      • 모든 문자를 치환한 뒤, StringBuilder에 쌓인 문자열을 반환한다.

    시간 복잡도

    문자열 s의 길이를 n이라고 하면:

    • 각 문자마다 while 반복문에서 최대 index개의 유효 알파벳을 만날 때까지 전진한다.
    • 전진 과정에서 skip 때문에 조금 더 많은 회전을 할 수 있지만, 알파벳은 26개뿐이므로 최대 한 문자당 상수 배 수준이다.

    따라서 전체 시간 복잡도는 대략 O(n × index) (또는 최악 기준 O(26 × n × index)) 정도이며,
    문제의 제한 내에서는 충분히 빠르게 동작한다.

    공간 복잡도

    • boolean[26] 배열 하나와 StringBuilder만 추가로 사용한다.
    • 추가 공간은 알파벳 개수에 비례하는 O(1) 수준이다.

    주의

    • skip 처리:
      • skip에 있는 문자들은 알파벳을 전진할 때 칸 수에 포함되면 안 된다.
      • 그래서 전진할 때마다 ban[pos]를 보고, false일 때만 cnt를 증가시킨다.
    • 원형 순환:
      • pos = (pos + 1) % 26로 처리해 z에서 다시 a로 자연스럽게 돌아가도록 한다.
      • 인덱스 범위를 벗어나지 않도록 % 26이 필수다.
    • 소문자 전제:
      • 문제에서 입력이 소문자라고 가정하므로 'a' 기준으로 인덱싱한다.
      • 만약 대문자까지 섞였다면 Character.toLowerCase 처리가 필요했겠지만, 이 문제에서는 필요 없다.

    포인트 요약

    • skip에 포함된 알파벳을 boolean[26]에 표시해서 O(1)로 건너뛴다.
    • s의 각 문자마다, 알파벳을 원형으로 한 칸씩 전진하면서
      건너뛰지 않는 알파벳을 index번 만날 때까지 반복한다.
    • 최종 도착한 알파벳을 결과 문자열에 붙이고, 이를 모든 문자에 대해 반복한다.
    • 알파벳 인덱싱(c - 'a', 'a' + pos)과 모듈러 연산만 정확히 처리하면 구현은 간단하다.
