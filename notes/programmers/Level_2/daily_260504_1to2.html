    문제 1 — 연속된 부분 수열의 합

    문제 링크

    프로그래머스 문제 보기

    코드 (IDE 스크린샷)

    해설

    연속된 부분 수열의 합 (프로그래머스)는
    비내림차순 정수 배열에서 연속된 부분 수열의 합이 k가 되는 구간 중 길이가 가장 짧은 구간의 시작·끝 인덱스를 구하는 문제다.
    두 포인터(슬라이딩 윈도우)로 합이 k인 구간을 탐색하며 최단 길이를 갱신한다.

    • 배열이 비내림차순이므로 윈도우를 오른쪽으로 확장하면 합이 단조 증가하고, 왼쪽을 줄이면 합이 감소한다. 이 성질 덕분에 두 포인터가 유효하게 동작한다.
    • 합이 k를 초과하면 왼쪽 포인터를 오른쪽으로 이동하여 합을 줄이고, 합이 k와 같아지면 현재 구간 길이를 최솟값과 비교한다.
    • 길이가 같은 구간이 여러 개일 경우 먼저 등장한 구간을 유지한다.

    전략 개요

    • 핵심 알고리즘: 두 포인터 (슬라이딩 윈도우)
    • 선택 이유: 배열이 비내림차순이므로 구간 합이 단조적으로 변화한다. 두 포인터로 윈도우를 조절하면 모든 유효 구간을 O(N)에 탐색할 수 있어, 완전 탐색 O(N²)보다 훨씬 효율적이다.
    • 세부 전략:
      • end를 오른쪽으로 이동하며 total에 현재 원소를 더한다.
      • total이 k를 초과하면 start를 오른쪽으로 이동하며 total을 줄인다.
      • total == k이면 현재 구간 길이를 최솟값과 비교하여 더 짧으면 answer를 갱신한다.

    코드가 하는 일

    • 1) 초기화 
      int[] answer = new int[2];
      int length = Integer.MAX_VALUE;
      int total = 0;
      int start = 0;
      int end = 0;
      • 현재까지 찾은 최단 구간 길이 length를 Integer.MAX_VALUE로 초기화하여 첫 번째 유효 구간이 항상 갱신되도록 한다. start와 end는 윈도우의 왼쪽·오른쪽 경계다.
    • 2) 두 포인터 탐색 
      while (end < sequence.length) {
          total += sequence[end];
      
          while (total > k && start <= end) {
              total -= sequence[start];
              start++;
          }
      
          if (total == k) {
              int curLength = end - start + 1;
      
              if (curLength < length) {
                  length = curLength;
                  answer[0] = start;
                  answer[1] = end;
              }
          }
      
          end++;
      }
      • total += sequence[end]: 오른쪽 포인터를 확장하며 윈도우 합에 현재 원소를 추가한다.
      • total > k인 동안 start를 오른쪽으로 이동하며 합을 줄인다. start <= end 조건으로 윈도우가 역전되지 않도록 보호한다.
      • total == k이면 현재 구간 길이 end - start + 1을 최솟값과 비교하여 더 짧을 때만 answer를 갱신한다. 동일한 길이의 구간은 갱신하지 않으므로 먼저 등장한 구간이 유지된다.
    • 3) 결과 반환 
      return answer;
      • 합이 k가 되는 가장 짧은 구간의 시작·끝 인덱스를 담은 배열을 반환한다.

    시간 · 공간 복잡도

    • 시간 복잡도
      • O(N): start와 end 모두 왼쪽에서 오른쪽으로만 이동하므로 각 원소는 최대 두 번(추가, 제거) 처리된다. 전체 O(N)이다.
    • 공간 복잡도
      • O(1): answer 배열(크기 2)과 포인터 변수들만 사용하므로 입력 크기와 무관한 상수 공간만 필요하다.

    문제 2 — 과제 진행하기

    문제 링크

    프로그래머스 문제 보기

    코드 (IDE 스크린샷)

    해설

    과제 진행하기 (프로그래머스)는
    과제 목록이 주어질 때, 시작 시각 순서대로 과제를 진행하되 새 과제가 시작되면 현재 과제를 중단하고, 남은 시간에 중단된 과제를 역순으로 처리하여 완료 순서를 구하는 문제다.
    중단된 과제는 스택으로 관리하여 가장 최근에 중단된 것부터 재개한다.

    • 과제를 시작 시각 오름차순으로 정렬한 뒤, 인접한 두 과제의 시작 시각 차이로 현재 과제에 주어진 가용 시간을 계산한다.
    • 가용 시간이 현재 과제의 소요 시간보다 짧으면 과제를 중단하고 스택에 push한다.
    • 가용 시간이 남으면 현재 과제를 완료한 뒤 남은 시간으로 스택의 과제를 순서대로 재개한다.
    • 마지막 과제와 스택에 남은 과제는 순서대로 완료 처리한다.

    전략 개요

    • 핵심 알고리즘: 정렬 + 스택을 이용한 과제 중단·재개 시뮬레이션
    • 선택 이유: 중단된 과제는 가장 최근에 중단된 것부터 재개되므로 LIFO 구조인 스택이 적합하다. 시작 시각 기준으로 정렬하면 인접한 두 과제의 시각 차이로 가용 시간을 O(1)에 계산할 수 있어, 전체 O(N log N)에 해결된다.
    • 세부 전략:
      • 과제를 시작 시각 오름차순으로 정렬한다.
      • 인접한 두 과제의 시작 시각 차이 sub를 계산하여 현재 과제의 소요 시간과 비교한다.
      • sub == playtime이면 현재 과제 완료, sub < playtime이면 남은 시간을 줄여 스택에 push, sub > playtime이면 완료 후 남은 시간으로 스택 과제를 순서대로 재개한다.
      • 마지막 과제를 완료 처리한 뒤 스택에 남은 과제를 순서대로 완료한다.

    코드가 하는 일

    • 1) Plan 클래스 및 정렬 
      static class Plan {
          String name;
          int start;
          int playtime;
      
          public Plan(String name, String start, String playtime) {
              this.name = name;
              this.start = convertTime(start);
              this.playtime = Integer.parseInt(playtime);
          }
      
          private int convertTime(String time) {
              String[] split = time.split(":");
              return Integer.parseInt(split[0]) * 60 + Integer.parseInt(split[1]);
          }
      }
      
      planList.sort((a, b) -> a.start - b.start);
      • 시각 문자열 "HH:MM"을 분 단위 정수로 변환하여 시각 차이를 정수 연산으로 처리한다. 과제 목록을 시작 시각 오름차순으로 정렬한다.
    • 2) 과제 진행 시뮬레이션 
      for (int i = 0; i < planList.size() - 1; i++) {
          Plan current = planList.get(i);
          Plan next = planList.get(i + 1);
      
          int sub = next.start - current.start;
      
          if (sub == current.playtime) {
              answer.add(current.name);
          } else if (sub < current.playtime) {
              current.playtime -= sub;
              stack.push(current);
          } else {
              answer.add(current.name);
              int gap = sub - current.playtime;
      
              while (!stack.isEmpty() && gap > 0) {
                  Plan paused = stack.peek();
      
                  if (paused.playtime <= gap) {
                      gap -= paused.playtime;
                      answer.add(stack.pop().name);
                  } else {
                      paused.playtime -= gap;
                      gap = 0;
                  }
              }
          }
      }
      • sub == playtime: 가용 시간이 정확히 맞아 현재 과제를 완료한다.
      • sub < playtime: 다음 과제가 일찍 시작되므로 현재 과제를 중단하고 남은 소요 시간을 갱신한 뒤 스택에 push한다.
      • sub > playtime: 현재 과제를 완료한 뒤 남은 시간 gap으로 스택 최상단 과제부터 재개한다. 재개한 과제의 소요 시간이 gap 이하이면 완료하고 다음 스택 과제를 확인하며, gap이 부족하면 소요 시간을 줄이고 중단한다.
    • 3) 마지막 과제 및 스택 처리 
      answer.add(planList.get(planList.size() - 1).name);
      
      while (!stack.isEmpty()) {
          answer.add(stack.pop().name);
      }
      • 마지막 과제는 다음 과제가 없으므로 항상 완료된다. 이후 스택에 남은 과제를 LIFO 순서로 순서대로 완료 처리한다.

    시간 · 공간 복잡도

    • 시간 복잡도
      • O(N log N): 과제 정렬에 O(N log N), 이후 순회와 스택 처리에 각 과제가 최대 한 번씩 push·pop되므로 O(N)이다. 전체 O(N log N)이다.
    • 공간 복잡도
      • O(N): 과제 목록 리스트, 스택, 결과 리스트 모두 최대 N개의 항목을 저장하므로 전체 O(N)이다.
