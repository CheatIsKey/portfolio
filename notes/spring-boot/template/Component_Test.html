    템플릿 — 단위 테스트 (Mockito) · 통합 테스트 (MockMvc)

    분류: 템플릿, Spring Boot, JUnit 5, Mockito, MockMvc, 단위 테스트, 통합 테스트

    이 템플릿의 구성 두 가지 테스트 계층을 제공한다.

    케이스 1 — 단위 테스트 (Mockito) — 외부 의존성을 Mock으로 격리하고 서비스 비즈니스 로직만 검증
    케이스 2 — 통합 테스트 (MockMvc) — 실제 Spring Context를 띄우고 HTTP 요청부터 DB까지 전 계층 검증

    케이스 1. 단위 테스트 — Mockito + @Nested

    외부 의존성(Repository, PasswordEncoder 등)을 Mock으로 격리하고 서비스 로직만 순수하게 검증한다. Spring Context를 띄우지 않으므로 실행 속도가 매우 빠르다.

    핵심 어노테이션 정리

    어노테이션 / 클래스 역할
    @ExtendWith(MockitoExtension.class) Mockito를 JUnit 5에 통합. Spring Context 없이 Mock 주입을 활성화한다.
    @InjectMocks 실제 테스트 대상 클래스. @Mock으로 선언된 객체들이 자동으로 주입된다.
    @Mock 가짜(Mock) 객체 생성. 실제 구현체 없이 동작을 stubbing할 수 있다.
    @Nested 테스트를 메서드 단위로 그룹화. 관련 테스트를 시각적으로 묶어 가독성을 높인다.
    given(...).willReturn(...) 특정 메서드 호출 시 반환값을 미리 지정하는 stubbing이다.
    then(...).should(...) 특정 메서드가 실제로 호출됐는지 검증하는 행동 검증(Behavior Verification)이다.
    assertThatThrownBy() 예외 발생 검증. 예외 타입과 메시지를 함께 확인할 수 있다.

    단위 테스트 전체 구조

    수정 포인트 MemberService, MemberRepository, PasswordEncoder를 실제 테스트 대상 서비스와 의존성으로 교체한다.
    SignUpRequest, MemberException, MemberErrorCode를 프로젝트 실제 클래스로 교체한다.
    @Nested 클래스명과 각 @DisplayName을 테스트 목적에 맞게 수정한다. 
    @ExtendWith(MockitoExtension.class) // Spring Context 없이 Mockito만 사용
class MemberServiceTest {

    // 수정 포인트: 테스트 대상 서비스 — @Mock 객체들이 자동 주입됨
    @InjectMocks
    private MemberService memberService;

    // 수정 포인트: 테스트 대상 서비스가 의존하는 클래스들을 Mock으로 선언
    @Mock
    private MemberRepository memberRepository;

    @Mock
    private PasswordEncoder passwordEncoder;

    // ================================================================
    // @Nested — 메서드 단위로 테스트를 그룹화해 가독성을 높인다
    // ================================================================

    @Nested
    @DisplayName("signUp() 로직 테스트")
    class SignUpTests {

        // ── 실패 케이스: 예외 발생 검증 패턴 ──────────────────────────

        @Test
        @DisplayName("회원가입 실패 - 이메일 중복")
        void signUp_fail_duplicate_email() {
            // given: 요청 데이터 준비
            SignUpRequest request = SignUpRequest.builder()
                    .username("new_user")
                    .password("password!23")
                    .emailOrPhone("test@test.com")
                    .name("New User")
                    .build();

            // 수정 포인트: 이메일 중복 확인 Mock — true를 반환해 중복 상태를 시뮬레이션
            given(memberRepository.existsByEmail("test@test.com")).willReturn(true);

            // then: 예외 타입과 메시지를 함께 검증
            assertThatThrownBy(() -> memberService.signUp(request))
                    .isInstanceOf(MemberException.class)
                    .hasMessage(MemberErrorCode.DUPLICATE_EMAIL.getMessage());

            // 예외가 터진 이후에는 아래 로직이 실행되지 않아야 한다 (Early Return 검증)
            then(passwordEncoder).shouldHaveNoInteractions();           // 비밀번호 암호화 미실행
            then(memberRepository).should(never()).save(any(Member.class)); // save 미호출
        }

        @Test
        @DisplayName("회원가입 실패 - 전화번호 중복")
        void signUp_fail_duplicate_phone() {
            SignUpRequest request = SignUpRequest.builder()
                    .username("new_user")
                    .password("password!23")
                    .emailOrPhone("01012345678")
                    .name("New User")
                    .build();

            given(memberRepository.existsByPhone("01012345678")).willReturn(true);

            assertThatThrownBy(() -> memberService.signUp(request))
                    .isInstanceOf(MemberException.class)
                    .hasMessage(MemberErrorCode.DUPLICATE_PHONE.getMessage());

            then(passwordEncoder).shouldHaveNoInteractions();
            then(memberRepository).should(never()).save(any(Member.class));
        }

        @Test
        @DisplayName("회원가입 실패 - 유저네임 중복")
        void signUp_fail_duplicate_username() {
            SignUpRequest request = SignUpRequest.builder()
                    .username("existing_user")
                    .password("password!23")
                    .emailOrPhone("test@test.com")
                    .name("New User")
                    .build();

            // 이메일은 통과(false), 유저네임은 중복(true) 시뮬레이션
            given(memberRepository.existsByEmail("test@test.com")).willReturn(false);
            given(memberRepository.existsByUsername("existing_user")).willReturn(true);

            assertThatThrownBy(() -> memberService.signUp(request))
                    .isInstanceOf(MemberException.class)
                    .hasMessage(MemberErrorCode.DUPLICATE_USERNAME.getMessage());

            then(passwordEncoder).shouldHaveNoInteractions();
            then(memberRepository).should(never()).save(any(Member.class));
        }

        // ── 성공 케이스: 메서드 호출 여부 검증 패턴 ───────────────────

        @Test
        @DisplayName("회원가입 성공 - 전화번호 가입")
        void signUp_success_with_phone() {
            SignUpRequest request = SignUpRequest.builder()
                    .username("new_user")
                    .password("password!23")
                    .emailOrPhone("01012345678")
                    .name("New User")
                    .build();

            given(memberRepository.existsByPhone("01012345678")).willReturn(false);
            given(memberRepository.existsByUsername("new_user")).willReturn(false);
            given(passwordEncoder.encode(request.password())).willReturn("encoded!");

            memberService.signUp(request);

            // save()가 실제로 한 번 호출됐는지 행동 검증
            then(memberRepository).should().save(any(Member.class));
        }

        @Test
        @DisplayName("회원가입 성공 - 이메일 가입, 비밀번호 암호화 및 영속화 검증")
        void signUp_success() {
            SignUpRequest request = SignUpRequest.builder()
                    .username("new_user")
                    .password("password!23")
                    .emailOrPhone("test@test.com")
                    .name("New User")
                    .build();
            String encodedPassword = "encoded_password!23";

            given(memberRepository.existsByEmail("test@test.com")).willReturn(false);
            given(memberRepository.existsByUsername("new_user")).willReturn(false);
            given(passwordEncoder.encode(request.password())).willReturn(encodedPassword);

            memberService.signUp(request);

            // 수정 포인트: 비밀번호 암호화가 정확한 값으로 호출됐는지 검증
            then(passwordEncoder).should().encode(eq("password!23"));
            then(memberRepository).should().save(any(Member.class));
        }
    }

    // ================================================================
    // 단순 조회 + 예외 분기 검증 패턴
    // ================================================================

    @Nested
    @DisplayName("checkDuplicate() 로직 테스트")
    class CheckDuplicateTests {

        @Test
        @DisplayName("checkDuplicate - username 중복 → false 반환")
        void checkDuplicate_username() {
            given(memberRepository.existsByUsername("testuser")).willReturn(true);
            boolean result = memberService.checkDuplicate("username", "testuser");
            // 수정 포인트: 사용 가능(not duplicate) = true, 사용 불가(duplicate) = false
            assertThat(result).isFalse();
        }

        @Test
        @DisplayName("checkDuplicate - email 사용 가능 → true 반환")
        void checkDuplicate_email() {
            given(memberRepository.existsByEmail("test@test.com")).willReturn(false);
            boolean result = memberService.checkDuplicate("email", "test@test.com");
            assertThat(result).isTrue();
        }

        @Test
        @DisplayName("checkDuplicate - phone 사용 가능 → true 반환")
        void checkDuplicate_phone() {
            given(memberRepository.existsByPhone("01012345678")).willReturn(false);
            boolean result = memberService.checkDuplicate("phone", "01012345678");
            assertThat(result).isTrue();
        }

        @Test
        @DisplayName("checkDuplicate - 지원하지 않는 타입 → 예외 발생")
        void checkDuplicate_invalid_type() {
            // 수정 포인트: 잘못된 입력에 대한 예외 코드를 실제 프로젝트 에러 코드로 교체
            assertThatThrownBy(() -> memberService.checkDuplicate("invalid", "value"))
                    .isInstanceOf(MemberException.class)
                    .hasMessage(CommonErrorCode.INVALID_INPUT_VALUE.getMessage());
        }
    }

    // ================================================================
    // Optional 결과 반환 + orElseThrow 검증 패턴
    // ================================================================

    @Nested
    @DisplayName("getMemberById() 로직 테스트")
    class GetMemberByIdTests {

        @Test
        @DisplayName("getMemberById - 성공")
        void getMemberById_success() {
            // 수정 포인트: 테스트용 Member 객체를 실제 엔티티 빌더로 생성
            Member mockMember = Member.builder().username("test").build();
            given(memberRepository.findById(1L)).willReturn(Optional.of(mockMember));

            Member result = memberService.getMemberById(1L);

            assertThat(result.getUsername()).isEqualTo("test");
        }

        @Test
        @DisplayName("getMemberById - 실패 (회원 없음)")
        void getMemberById_not_found() {
            // Optional.empty()로 존재하지 않는 데이터를 시뮬레이션
            given(memberRepository.findById(999L)).willReturn(Optional.empty());

            assertThatThrownBy(() -> memberService.getMemberById(999L))
                    .isInstanceOf(MemberException.class)
                    .hasMessage(MemberErrorCode.MEMBER_NOT_FOUND.getMessage());
        }
    }
}

    케이스 2. 통합 테스트 — @SpringBootTest + MockMvc

    실제 Spring Context를 전부 띄우고 HTTP 요청부터 DB까지 전 계층을 실제와 동일한 환경에서 검증한다. 단위 테스트로는 잡을 수 없는 계층 간 연동 문제(URL 매핑, 직렬화/역직렬화, 필터, 유효성 검사 등)를 검증한다.

    핵심 어노테이션 정리

    어노테이션 / 클래스 역할
    @SpringBootTest 전체 Spring Application Context 로딩. 실제 운영과 동일한 Bean 구성이 활성화된다.
    @AutoConfigureMockMvc MockMvc를 자동으로 설정해 HTTP 요청을 실제 서버 없이 시뮬레이션한다.
    @Transactional 각 테스트 메서드가 끝나면 DB 변경사항을 자동으로 롤백해 테스트 간 데이터 오염을 방지한다.
    @BeforeEach 각 테스트 실행 전 공통으로 수행할 준비 로직(테스트 데이터 삽입 등)을 정의한다.
    MockMvc.perform() 실제 HTTP 요청을 시뮬레이션한다. GET, POST, PUT, DELETE 모두 가능하다.
    jsonPath() 응답 JSON의 특정 필드 값을 검증한다. $.data.username처럼 경로 표현식을 사용한다.

    통합 테스트 전체 구조

    수정 포인트 AuthController, MemberService를 실제 테스트 대상 컨트롤러와 서비스로 교체한다.
    @BeforeEach의 사전 데이터 삽입 로직을 테스트 시나리오에 맞게 수정한다.
    API 경로(/api/auth/signup), 응답 필드($.data.username), 에러 코드(M002, C001)를 실제 프로젝트 명세로 교체한다. 
    @SpringBootTest       // 전체 Spring Context 로딩
@AutoConfigureMockMvc // MockMvc 자동 설정
@Transactional        // 수정 포인트: 각 테스트 후 DB 롤백 보장 — 테스트 데이터가 다음 테스트에 영향 안 줌
class AuthControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Autowired
    private ObjectMapper objectMapper; // Java 객체 → JSON 직렬화

    // 수정 포인트: 사전 데이터 준비에 사용할 실제 서비스 주입
    @Autowired
    private MemberService memberService;

    // ================================================================
    // 테스트 사전 준비 — 각 테스트 메서드 실행 전 자동 호출
    // ================================================================

    @BeforeEach
    void setUp() {
        // 수정 포인트: 테스트 시나리오에 필요한 사전 데이터를 서비스 레이어를 통해 삽입
        // Mock이 아닌 실제 서비스를 호출하므로 실제 DB에 데이터가 들어간다
        // @Transactional 덕분에 테스트가 끝나면 자동으로 롤백된다
        SignUpRequest signUpRequest = SignUpRequest.builder()
                .username("integration_user")
                .emailOrPhone("inter@test.com")
                .password("password!23")
                .name("Integration Test")
                .build();
        memberService.signUp(signUpRequest);
    }

    // ================================================================
    // 성공 케이스 — HTTP 상태코드 + 응답 Body 필드 검증 패턴
    // ================================================================

    @Test
    @DisplayName("회원가입 API 성공 흐름 검증 (201 Created)")
    void signup_api_success() throws Exception {
        // given: 정상적인 요청 데이터 준비
        SignUpRequest request = SignUpRequest.builder()
                .username("new_signup_user")
                .password("Password123!")          // 수정 포인트: @Valid 유효성 검사를 통과할 값으로 설정
                .emailOrPhone("newuser@test.com")
                .name("New User")
                .build();
        String content = objectMapper.writeValueAsString(request); // DTO → JSON 문자열 변환

        // when & then: HTTP 요청 전송 및 응답 검증
        mockMvc.perform(post("/api/auth/signup")            // 수정 포인트: 실제 API 경로로 교체
                        .contentType(MediaType.APPLICATION_JSON)
                        .content(content))
                .andExpect(status().isCreated())                            // 201 Created 검증
                .andExpect(jsonPath("$.success").value(true))
                .andExpect(jsonPath("$.data.username").value("new_signup_user")) // 수정 포인트: 응답 필드명
                .andExpect(jsonPath("$.data.message").value("회원가입이 완료되었습니다."));
    }

    // ================================================================
    // 실패 케이스 1 — 비즈니스 로직 예외 검증 패턴 (중복 등)
    // ================================================================

    @Test
    @DisplayName("회원가입 API 실패 검증 - 중복 가입 시도 (400 Bad Request)")
    void signup_api_fail_duplicate() throws Exception {
        // given: setUp()에서 이미 생성한 "integration_user"와 동일한 username 사용
        SignUpRequest request = SignUpRequest.builder()
                .username("integration_user")     // 수정 포인트: @BeforeEach에서 삽입한 중복 값 사용
                .password("ValidPass123!")
                .emailOrPhone("other_email@test.com")
                .name("Duplicate User")
                .build();
        String content = objectMapper.writeValueAsString(request);

        mockMvc.perform(post("/api/auth/signup")
                        .contentType(MediaType.APPLICATION_JSON)
                        .content(content))
                .andExpect(status().isBadRequest())                     // 400 Bad Request
                .andExpect(jsonPath("$.success").value(false))
                .andExpect(jsonPath("$.error.status").value(400))
                .andExpect(jsonPath("$.error.code").value("M002"))      // 수정 포인트: 실제 에러 코드로 교체
                .andExpect(jsonPath("$.error.message").value("이미 존재하는 사용자 이름입니다."))
                .andExpect(jsonPath("$.error.path").value("/api/auth/signup"));
    }

    // ================================================================
    // 실패 케이스 2 — @Valid 유효성 검사 실패 패턴
    // ================================================================

    @Test
    @DisplayName("회원가입 API 실패 검증 - 유효성 검사 실패 (@Valid)")
    void signup_api_fail_validation() throws Exception {
        // given: @Valid 유효성 검사를 의도적으로 통과하지 못하는 불량 데이터
        SignUpRequest request = SignUpRequest.builder()
                .username("usr")                // 수정 포인트: 길이 제한 미달 값으로 교체
                .password("1234")               // 특수문자/대문자 규칙 불만족
                .emailOrPhone("invalid_test.com") // 이메일 형식 불만족
                .name("")                       // @NotBlank 불만족
                .build();
        String content = objectMapper.writeValueAsString(request);

        mockMvc.perform(post("/api/auth/signup")
                        .contentType(MediaType.APPLICATION_JSON)
                        .content(content))
                .andExpect(status().isBadRequest())
                .andExpect(jsonPath("$.success").value(false))
                .andExpect(jsonPath("$.error.status").value(400))
                .andExpect(jsonPath("$.error.code").value("C001"))  // 수정 포인트: CommonErrorCode 코드로 교체
                .andExpect(jsonPath("$.error.path").value("/api/auth/signup"));
    }

    // ================================================================
    // 추가 패턴: 인증이 필요한 API 검증 (401 Unauthorized)
    // ================================================================

    @Test
    @DisplayName("인증 필요 API — 미인증 요청 시 401 반환")
    void authenticated_api_without_auth() throws Exception {
        // 수정 포인트: 인증이 필요한 API 경로로 교체
        mockMvc.perform(get("/api/members/me")
                        .contentType(MediaType.APPLICATION_JSON))
                .andExpect(status().isUnauthorized()); // 401 Unauthorized
    }
}

    단위 테스트 vs 통합 테스트 비교

    구분 단위 테스트 통합 테스트
    주요 어노테이션 @ExtendWith(MockitoExtension.class) @SpringBootTest + @AutoConfigureMockMvc
    Spring Context 사용 안 함 (빠름) 전체 로딩 (느리지만 실제와 동일)
    의존성 처리 @Mock으로 가짜 객체 사용 실제 Bean 사용 (DB, 필터 포함)
    검증 대상 서비스 비즈니스 로직 (분기, 예외, 메서드 호출 여부) 전 계층 연동 (URL 매핑, 직렬화, 유효성, 인증)
    DB 격리 DB 접근 없음 (Mock으로 대체) @Transactional로 테스트 후 자동 롤백
    실행 속도 매우 빠름 (ms 단위) 느림 (Context 로딩 수 초)
    적합한 상황 로직 분기, Early Return, 예외 발생 조건 검증 API 명세, 에러 코드, 인증/인가, 유효성 검사 검증
    단위 테스트와 통합 테스트를 모두 작성해야 하는 이유 단위 테스트만 있으면 서비스 로직은 완벽해도 컨트롤러 URL이 틀렸거나, 응답 JSON 필드명이 다르거나, 필터에서 걸리는 문제를 잡지 못한다.

    통합 테스트만 있으면 실행이 너무 느리고, 어떤 계층에서 문제가 생겼는지 원인을 좁히기 어렵다.

    서비스 비즈니스 로직 → 단위 테스트로 빠르게 검증
    API 계약(URL, 상태코드, 응답 포맷, 에러 코드) → 통합 테스트로 확인
    두 계층을 함께 작성하는 것이 가장 효과적인 테스트 전략이다.
