    코드 템플릿 — 전역 예외 처리기 (GlobalExceptionHandler)

    분류: 템플릿, Spring Boot, 예외 처리, @RestControllerAdvice, ApiResponse

    개요

    모든 컨트롤러에서 발생하는 예외를 한 곳에서 일관성 있게 처리하는 전역 예외 처리기 템플릿이다. 커스텀 예외, Jakarta Validation 검증 실패, 그 외 예상치 못한 예외를 각각 구분하여 처리하며, 에러 코드 체계와 통일된 응답 포맷(ApiResponse)을 함께 제공한다.

    전역 예외 처리기가 필요한 이유 예외 처리를 각 컨트롤러에서 개별적으로 하면 응답 포맷이 제각각이 되고 중복 코드가 발생한다. @RestControllerAdvice로 한 곳에 모아두면 응답 형식의 일관성이 보장되고, 에러 코드를 추가하거나 로깅 방식을 변경할 때 단 한 파일만 수정하면 된다.

    파일 구성

    파일명 종류 역할
    GlobalExceptionHandler.java @RestControllerAdvice 모든 예외를 수신하여 처리하고 통일된 응답을 반환한다.
    ErrorCode.java enum HTTP 상태 코드, 에러 코드 문자열, 에러 메시지를 묶어 관리한다.
    CustomException.java RuntimeException 상속 ErrorCode를 담아 던지는 커스텀 예외 클래스다.
    ApiResponse.java record 성공/실패 공통 응답 포맷이다. 모든 API 응답은 이 형식을 따른다.
    ErrorResponse.java record (@Builder) 에러 메시지와 필드 검증 오류 목록을 담는 응답 바디다.
    FieldError.java record @Valid 검증 실패 시 각 필드의 오류 정보를 담는다.

    1. ErrorCode.java

    HTTP 상태 코드, 에러 코드 문자열, 메시지를 하나의 enum 상수로 묶어 관리한다. 에러 코드 체계는 카테고리 문자 + 3자리 숫자 형식을 사용한다. 

    @Getter
public enum ErrorCode {

    // 수정 포인트: 도메인에 맞게 카테고리와 코드를 추가한다.
    //   공통 에러 코드 (A###)
    INVALID_INPUT_VALUE(HttpStatus.BAD_REQUEST,  "A001", "입력값이 올바르지 않습니다."),

    //   사용자 관련 에러 코드 (U###)
    NOT_FOUND_USER(HttpStatus.NOT_FOUND,         "U001", "해당 유저는 존재하지 않습니다."),

    //   파일 관련 에러 코드 (F###)
    FILE_UPLOAD_FAIL(HttpStatus.BAD_REQUEST,     "F001", "파일 업로드를 실패했습니다."),

    //   프로필 관련 에러 코드 (P###)
    PROFILE_NOT_FOUND(HttpStatus.NOT_FOUND,      "P001", "프로필을 찾을 수 없습니다.");

    private final HttpStatus status;
    private final String code;
    private final String message;

    ErrorCode(HttpStatus status, String code, String message) {
        this.status = status;
        this.code   = code;
        this.message = message;
    }
}
    카테고리 접두사 예시
    공통 A### 입력값 오류, 인증 실패 등
    사용자 U### 유저 미존재, 비밀번호 불일치 등
    파일 F### 업로드 실패, 확장자 불가 등
    프로필 P### 프로필 미존재 등

    2. CustomException.java

    ErrorCode를 담아 던지는 커스텀 예외 클래스다. 비즈니스 규칙 위반 시 이 예외를 던지면 GlobalExceptionHandler가 받아 처리한다. 

    @Getter
public class CustomException extends RuntimeException {

    private final ErrorCode errorCode;

    public CustomException(ErrorCode errorCode) {
        super(errorCode.getMessage());
        this.errorCode = errorCode;
    }
}

// 사용 예시 (Service 계층)
public User findUser(Long userId) {
    return userRepository.findById(userId)
            .orElseThrow(() -> new CustomException(ErrorCode.NOT_FOUND_USER));
}

    3. FieldError.java

    @Valid 검증 실패 시 스프링이 생성하는 FieldError 객체를 응답 바디에 담을 수 있는 형태로 변환하는 record다. 어떤 필드에서, 어떤 값이, 왜 거부됐는지를 클라이언트에 전달한다. 

    public record FieldError(
        String field,          // 검증 실패한 필드명
        String rejectedValue,  // 거부된 입력값
        String message         // 검증 실패 메시지 (@NotBlank의 message 속성 등)
) {
    // Spring의 FieldError → 응답용 FieldError 변환
    public static FieldError from(org.springframework.validation.FieldError error) {
        return new FieldError(
                error.getField(),
                error.getRejectedValue() != null ? error.getRejectedValue().toString() : null,
                error.getDefaultMessage()
        );
    }
}

    4. ErrorResponse.java

    에러 메시지와 필드 검증 오류 목록을 담는 응답 바디다. 단순 에러 메시지만 있는 경우와, @Valid 검증 실패로 여러 필드 오류가 있는 경우를 각각 from()과 of()로 구분해 생성한다. 

    @Builder
public record ErrorResponse(
        String message,
        List<FieldError> fieldErrors  // @Valid 검증 실패 시에만 채워진다. 그 외에는 null
) {
    // 단순 에러 메시지만 있을 때
    public static ErrorResponse from(String message) {
        return ErrorResponse.builder()
                .message(message)
                .build();
    }

    // 필드 검증 오류 목록이 있을 때
    public static ErrorResponse of(String message, List<FieldError> fieldErrors) {
        return ErrorResponse.builder()
                .message(message)
                .fieldErrors(fieldErrors)
                .build();
    }
}

    5. ApiResponse.java

    성공·실패 모든 API 응답이 따르는 공통 응답 포맷이다. 성공 시에는 success(), 실패 시에는 fail()로 생성한다. 

    public record ApiResponse<T>(
        boolean success,
        String code,           // 성공: HTTP 상태코드 문자열 / 실패: ErrorCode.code
        T data,                // 성공: 응답 데이터 / 실패: ErrorResponse
        LocalDateTime timestamp
) {
    // 응답 데이터가 있는 성공
    public static <T> ApiResponse<T> success(HttpStatus status, T data) {
        return new ApiResponse<>(true, String.valueOf(status.value()), data, LocalDateTime.now());
    }

    // 응답 데이터가 없는 성공 (예: 삭제, 수정)
    public static <T> ApiResponse<T> success(HttpStatus status) {
        return new ApiResponse<>(true, String.valueOf(status.value()), null, LocalDateTime.now());
    }

    // 단순 에러 응답
    public static ApiResponse<ErrorResponse> fail(ErrorCode errorCode) {
        return new ApiResponse<>(false, errorCode.getCode(),
                ErrorResponse.from(errorCode.getMessage()), LocalDateTime.now());
    }

    // 필드 검증 오류 포함 에러 응답
    public static ApiResponse<ErrorResponse> fail(ErrorCode errorCode, List<FieldError> fieldErrors) {
        return new ApiResponse<>(false, errorCode.getCode(),
                ErrorResponse.of(errorCode.getMessage(), fieldErrors), LocalDateTime.now());
    }
}

    6. GlobalExceptionHandler.java

    모든 예외를 수신하여 처리하는 핵심 클래스다. @ExceptionHandler에 예외 타입을 지정하면 해당 예외 발생 시 자동으로 해당 메서드가 호출된다. 

    @Slf4j
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 커스텀 예외 처리 — 비즈니스 로직에서 throw new CustomException(ErrorCode.XXX) 로 던진 경우
    @ExceptionHandler(CustomException.class)
    public ResponseEntity<ApiResponse<ErrorResponse>> handleException(CustomException exception) {
        log.error("[API - ERROR] 발생 원인: ", exception);
        ErrorCode errorCode = exception.getErrorCode();
        return ResponseEntity.status(errorCode.getStatus())
                .body(ApiResponse.fail(errorCode));
    }

    // @Valid 검증 실패 처리 — DTO에 @Valid 적용 시 검증 실패하면 자동 발생
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ApiResponse<ErrorResponse>> handleMethodArgumentNotValid(
            MethodArgumentNotValidException exception) {
        log.error("[API - ERROR] 발생 원인: ", exception);
        ErrorCode errorCode = ErrorCode.INVALID_INPUT_VALUE;
        List<FieldError> fieldErrors = exception.getBindingResult()
                .getFieldErrors()
                .stream()
                .map(FieldError::from)
                .toList();
        return ResponseEntity.status(errorCode.getStatus())
                .body(ApiResponse.fail(errorCode, fieldErrors));
    }

    // 수정 포인트: 추가로 처리할 예외가 있으면 @ExceptionHandler 메서드를 아래에 추가한다.
    // 예시 — 예상치 못한 모든 예외를 최후 방어선으로 처리
    // @ExceptionHandler(Exception.class)
    // public ResponseEntity<ApiResponse<ErrorResponse>> handleAll(Exception exception) {
    //     log.error("[API - ERROR] 예상치 못한 오류: ", exception);
    //     ErrorCode errorCode = ErrorCode.INTERNAL_SERVER_ERROR; // 수정 포인트: ErrorCode에 추가 필요
    //     return ResponseEntity.status(errorCode.getStatus())
    //             .body(ApiResponse.fail(errorCode));
    // }
}
    ⚠️ @ExceptionHandler 우선순위 예외 클래스 계층 구조에서 더 구체적인 예외 타입을 먼저 선언해야 한다. 예를 들어 Exception.class 핸들러를 가장 위에 두면, 모든 예외가 거기서 처리되어 CustomException이나 MethodArgumentNotValidException 핸들러가 무시된다. 구체적인 타입 → 추상적인 타입 순서로 선언한다.

    8. 주요 수정 포인트 요약

    파일 수정 포인트
    ErrorCode.java 도메인별 에러 코드를 추가한다. 카테고리 접두사 규칙(A###, U### 등)을 유지한다.
    GlobalExceptionHandler.java 추가로 처리할 예외 타입이 있으면 @ExceptionHandler 메서드를 아래에 추가한다. Exception.class 최후 방어선 핸들러는 주석 해제 후 INTERNAL_SERVER_ERROR ErrorCode를 추가해 활성화한다.
    ApiResponse.java 응답 포맷을 변경해야 한다면 record 필드를 수정한다. success()/fail() 정적 팩토리 메서드도 함께 수정한다.
    ErrorResponse.java 에러 응답에 추가 정보(예: stackTrace, requestId)가 필요하면 필드를 추가하고 빌더 메서드를 수정한다.
    컨트롤러에서의 사용 패턴 
    @PostMapping("/members")
public ResponseEntity<ApiResponse<MemberResponse>> createMember(
        @Valid @RequestBody MemberCreateRequest request) {  // 수정 포인트: 검증 대상 DTO에 @Valid 적용
    MemberResponse response = memberService.create(request);
    return ResponseEntity.status(HttpStatus.CREATED)
            .body(ApiResponse.success(HttpStatus.CREATED, response));
}
