    템플릿 — Interceptor · ArgumentResolver

    분류: 템플릿, Spring Boot, Interceptor, ArgumentResolver, 로깅, 인가, 세션

    이 템플릿의 구성 두 가지 독립적인 케이스를 제공한다.

    케이스 1 — 커스텀 어노테이션(@AdminLogging) + HandlerInterceptor를 이용한 관리자 권한 검증 + 접근 로그 기록
    케이스 2 — 커스텀 어노테이션(@LoginUser) + HandlerMethodArgumentResolver를 이용한 세션 기반 인증 객체 자동 주입

    케이스 1. 관리자 권한 검증 + 접근 로그 (Interceptor)

    특정 경로에 대해 요청이 컨트롤러에 도달하기 전에 관리자 권한을 검증하고, 통과한 요청에 한해 접근 로그를 기록한다. preHandle()이 false를 반환하면 이후 컨트롤러 로직이 실행되지 않는다.

    파일 구성

    파일 역할 수정 포인트
    AdminLogging.java 마커 어노테이션 어노테이션 이름, @Target 범위
    CustomInterceptor.java 권한 검증 + 접근 로그 경로 조건, getAttribute 키 이름, 역할 비교 로직
    WebConfig.java 인터셉터 등록 addPathPatterns 경로

    AdminLogging.java

    인터셉터를 메서드 단위로 식별하고 싶을 때 붙이는 마커 어노테이션이다. 단순히 addPathPatterns로 경로만 제어한다면 이 어노테이션 없이도 동작한다.

    수정 포인트 어노테이션 이름(AdminLogging)을 프로젝트 목적에 맞게 변경한다.
    클래스 레벨에도 적용하려면 @Target에 ElementType.TYPE을 추가한다. 
    @Target(ElementType.METHOD)          // 메서드에만 적용 가능
@Retention(RetentionPolicy.RUNTIME)  // 런타임까지 어노테이션 정보 유지
public @interface AdminLogging {
    // 필요 시 속성 추가 가능 — 예: String value() default "";
}

    CustomInterceptor.java

    수정 포인트 isTargetRequest 조건의 HTTP 메서드와 경로 패턴을 프로젝트 요구사항에 맞게 교체한다.
    request.getAttribute("userId")와 request.getAttribute("userRole")은 JWT 필터 등 앞단에서 setAttribute()로 값을 넣어두어야 정상 동작한다.
    UserRole.of()는 프로젝트의 실제 역할 변환 메서드로 대체한다. 
    @Slf4j
@Component
public class CustomInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler)
            throws Exception {

        String method = request.getMethod();      // 요청 HTTP 메서드 (GET, POST, DELETE 등)
        String path   = request.getRequestURI();  // 요청 경로 (/admin/comments/1 등)

        // 수정 포인트: 로깅/검증을 적용할 경로 조건 — 해당하지 않으면 바로 통과
        boolean isTargetRequest =
                (method.equals("DELETE") && path.matches("/admin/comments/\\d+")) ||
                (method.equals("PATCH")  && path.matches("/admin/users/\\d+"));

        if (!isTargetRequest) {
            return true; // 대상 경로가 아니면 다음 단계로 통과
        }

        // 수정 포인트: JWT 필터 등 앞단에서 setAttribute()로 넣어둔 값을 꺼낸다.
        Long     userId = (Long)   request.getAttribute("userId");
        UserRole role   = UserRole.of((String) request.getAttribute("userRole"));

        // 관리자 권한 검증 — 권한이 없으면 403 응답 후 요청 중단
        if (role == null || !UserRole.ADMIN.equals(role)) {
            response.sendError(HttpServletResponse.SC_FORBIDDEN, "접근 권한이 없습니다.");
            return false;
        }

        // 관리자 접근 로그 기록
        log.info("[관리자 기능 접근] userId = {}, requestTime = {}, url = {}",
                userId, LocalDateTime.now(), path);

        return true; // 검증 통과 → 컨트롤러 실행 허용
    }
}

    WebConfig.java (케이스 1) 

    @Configuration
@RequiredArgsConstructor
public class WebConfig implements WebMvcConfigurer {

    private final CustomInterceptor customInterceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(customInterceptor)
                .addPathPatterns("/admin/comments/*")  // 수정 포인트: 적용할 경로
                .addPathPatterns("/admin/users/*");
                // .excludePathPatterns("/admin/health")  // 제외할 경로가 있으면 추가
    }
}

    케이스 2. 세션 기반 인증 객체 자동 주입 (ArgumentResolver)

    컨트롤러 메서드 파라미터에 @LoginUser를 붙이면 HandlerMethodArgumentResolver가 세션에서 인증 정보를 꺼내 자동으로 주입한다. 컨트롤러마다 반복하던 세션 null 체크 코드가 사라지고, 메서드 시그니처만 봐도 인증 여부가 명확히 드러난다.

    ArgumentResolver가 해결하는 문제 Before: 모든 컨트롤러 메서드마다 HttpSession session = request.getSession(false); if (session == null) throw ...; 코드가 반복된다.

    After: @LoginUser CurrentUser user 파라미터 하나로 세션 조회 + null 체크 + 객체 변환이 한 번에 처리된다. 컨트롤러는 비즈니스 로직에만 집중할 수 있다.

    파일 구성

    파일 역할 수정 포인트
    LoginUser.java 파라미터 마커 어노테이션 어노테이션 이름
    LoginUserArgumentResolver.java 세션 조회 + 인증 객체 반환 세션 키, 반환 타입(CurrentUser), 예외 코드
    WebConfig.java ArgumentResolver 등록 등록 대상 Resolver

    @LoginUser 어노테이션 

    @Target(ElementType.PARAMETER)       // 메서드 파라미터에만 적용 가능
@Retention(RetentionPolicy.RUNTIME)  // 런타임까지 어노테이션 정보 유지
public @interface LoginUser {
}

    LoginUserArgumentResolver.java

    수정 포인트 supportsParameter()에서 검사하는 타입(CurrentUser)을 프로젝트의 실제 인증 DTO 클래스로 교체한다.
    session.getAttribute("loginUser")의 세션 키를 로그인 처리 시 session.setAttribute()에 사용한 키와 반드시 일치시킨다.
    Admin과 Customer처럼 역할별로 세션 키가 다르다면 resolveArgument()에서 분기 처리를 추가한다.
    예외 클래스(CustomException, ErrorCode.UNAUTHORIZED)를 프로젝트 예외 체계로 교체한다. 
    @Component
public class LoginUserArgumentResolver implements HandlerMethodArgumentResolver {

    /**
     * 이 Resolver가 처리할 파라미터 조건 — @LoginUser 어노테이션이 붙어 있고,
     * 파라미터 타입이 CurrentUser(또는 그 하위 타입)일 때만 동작한다.
     */
    @Override
    public boolean supportsParameter(MethodParameter parameter) {
        boolean hasLoginUserAnnotation = parameter.hasParameterAnnotation(LoginUser.class);
        // 수정 포인트: CurrentUser를 프로젝트 실제 인증 DTO 타입으로 교체한다.
        boolean hasCurrentUserType = CurrentUser.class.isAssignableFrom(parameter.getParameterType());
        return hasLoginUserAnnotation && hasCurrentUserType;
    }

    /**
     * supportsParameter()가 true를 반환한 파라미터에 실제 주입할 객체를 만들어 반환한다.
     * 세션 조회 + null 체크 + 객체 변환 로직이 여기에 집중된다.
     */
    @Override
    public Object resolveArgument(MethodParameter parameter,
                                  ModelAndViewContainer mavContainer,
                                  NativeWebRequest webRequest,
                                  WebDataBinderFactory binderFactory) throws Exception {

        HttpServletRequest request = (HttpServletRequest) webRequest.getNativeRequest();
        HttpSession session = request.getSession(false); // false: 없으면 새로 만들지 않음

        if (session == null) {
            throw new CustomException(ErrorCode.UNAUTHORIZED);
        }

        // 수정 포인트: 로그인 처리 시 session.setAttribute()에 사용한 키 이름과 일치시킨다.
        Object loginUser = session.getAttribute("loginUser");
        if (loginUser == null) {
            throw new CustomException(ErrorCode.UNAUTHORIZED);
        }

        // 수정 포인트: 세션 객체를 CurrentUser 형태로 변환해서 반환한다.
        // 세션에 이미 CurrentUser 객체를 저장했다면 캐스팅만 해서 바로 반환 가능하다.
        // 예) return new CurrentUser(((Admin) loginUser).getId(), Role.ADMIN);
        return loginUser;
    }
}

    컨트롤러 적용 예시 

    // Before: 컨트롤러마다 세션 null 체크 반복
@PostMapping
public ResponseEntity<ApiResponse<OrderCreateResponse>> createOrder(
        HttpServletRequest request,
        @Valid @RequestBody OrderCreateRequest orderRequest) {

    HttpSession session = request.getSession(false);
    if (session == null) throw new CustomException(ErrorCode.UNAUTHORIZED);
    CurrentUser user = (CurrentUser) session.getAttribute("loginUser");
    if (user == null) throw new CustomException(ErrorCode.UNAUTHORIZED);

    return ResponseEntity.status(HttpStatus.CREATED)
            .body(ApiResponse.success(HttpStatus.CREATED, orderService.add(user, orderRequest)));
}

// After: @LoginUser 하나로 세션 조회 + null 체크 + 변환이 자동 처리됨
@PostMapping
public ResponseEntity<ApiResponse<OrderCreateResponse>> createOrder(
        @LoginUser CurrentUser user,                  // ArgumentResolver가 세션에서 자동 주입
        @Valid @RequestBody OrderCreateRequest request) {

    return ResponseEntity.status(HttpStatus.CREATED)
            .body(ApiResponse.success(HttpStatus.CREATED, orderService.add(user, request)));
}

    두 케이스를 함께 사용하는 경우

    인터셉터(케이스 1)와 ArgumentResolver(케이스 2)는 독립적으로 동작하므로 함께 사용할 수 있다. 하나의 WebConfig에 모두 등록하면 된다. 

    @Configuration
@RequiredArgsConstructor
public class WebConfig implements WebMvcConfigurer {

    private final CustomInterceptor customInterceptor;
    private final LoginUserArgumentResolver loginUserArgumentResolver;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(customInterceptor)
                .addPathPatterns("/admin/comments/*")
                .addPathPatterns("/admin/users/*");
    }

    @Override
    public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
        resolvers.add(loginUserArgumentResolver);
    }
}
    Interceptor vs ArgumentResolver — 역할 구분과 실행 순서 Interceptor (케이스 1): 요청이 컨트롤러에 도달하기 전 단계에서 동작한다. 권한 검증, 접근 차단, 접근 로그처럼 요청 전체에 걸친 횡단 관심사를 처리한다.

    ArgumentResolver (케이스 2): 컨트롤러 메서드를 실행하는 시점에 파라미터를 준비한다. 세션이나 JWT에서 꺼낸 인증 객체를 메서드 파라미터로 자동 주입한다.

    실행 순서: Interceptor → ArgumentResolver → Controller
