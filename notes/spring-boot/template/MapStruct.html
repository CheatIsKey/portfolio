    템플릿 — MapStruct 매퍼

    분류: 템플릿, Spring Boot, MapStruct, DTO 변환, 엔티티 매핑

    이 템플릿의 구성 MapStruct를 사용한 엔티티 ↔ DTO 변환 패턴 3가지를 제공한다.

    패턴 1 — 필드명 동일: 자동 매핑 (코드 없음)
    패턴 2 — 필드명 다름: @Mapping(source, target)으로 명시
    패턴 3 — List 변환: 단건 메서드를 선언하면 MapStruct가 자동으로 List 변환 생성
    패턴 4 — 커스텀 로직: default 메서드로 직접 구현
    서비스 사용법 — 생성된 Bean을 주입해서 호출하는 방법

    1. 의존성 추가

    수정 포인트 Lombok과 MapStruct를 함께 사용할 때 annotationProcessorPaths 순서가 중요하다. Lombok이 MapStruct보다 먼저 와야 한다. 순서가 바뀌면 MapStruct가 Lombok이 생성한 생성자/getter를 인식하지 못해 컴파일 오류가 발생한다. 
    // build.gradle

dependencies {
    // MapStruct 런타임 — 어노테이션만 포함 (구현체는 컴파일 타임에 생성됨)
    implementation 'org.mapstruct:mapstruct:1.5.5.Final'

    // MapStruct 어노테이션 프로세서 — 컴파일 타임에 MapperImpl 자동 생성
    annotationProcessor 'org.mapstruct:mapstruct-processor:1.5.5.Final'

    // Lombok과 함께 사용할 때 필수 — 반드시 Lombok 다음에 위치
    // 수정 포인트: Lombok을 사용하지 않는다면 아래 두 줄 제거
    annotationProcessor 'org.projectlombok:lombok'
    annotationProcessor 'org.projectlombok:lombok-mapstruct-binding:0.2.0'
}

    2. 매퍼 인터페이스 설계

    MapStruct의 동작 원리 MapStruct는 컴파일 타임에 인터페이스를 분석해 PostMapperImpl 구현체를 자동 생성한다. 런타임 리플렉션이 없으므로 ModelMapper보다 훨씬 빠르고, 생성된 코드를 직접 확인할 수 있다.

    componentModel = "spring"을 설정하면 생성된 구현체가 Spring Bean으로 자동 등록되어 @RequiredArgsConstructor로 바로 주입받을 수 있다.

    파일 구성

    파일 역할 수정 포인트
    PostMapper.java 매핑 규칙을 선언하는 인터페이스 클래스명, 필드명 매핑, default 메서드 로직
    PostMapperImpl.java 컴파일 타임에 자동 생성된 구현체 (직접 수정 금지) 수정 불필요 — 인터페이스를 수정하면 재생성됨
    PostService.java 생성된 Bean을 주입받아 사용 주입받는 Mapper 타입, 호출 메서드명

    PostMapper.java — 완성본 

    /**
 * Post 도메인 MapStruct 매퍼.
 *
 * [componentModel = "spring"]
 * 컴파일 시 PostMapperImpl을 생성하고 Spring Bean으로 등록한다.
 * → PostService에서 @RequiredArgsConstructor로 바로 주입받아 사용 가능.
 *
 * 수정 포인트: 클래스명(PostMapper)과 변환 대상 엔티티/DTO를 실제 도메인으로 교체한다.
 */
@Mapper(componentModel = "spring")
public interface PostMapper {

    // ================================================================
    // 패턴 1. 필드명이 같은 경우 — 아무것도 선언하지 않아도 자동 매핑
    // ================================================================
    // 예시: PostImage.id → PostImageResponse.id (이름이 같으면 MapStruct가 알아서 연결)
    // 아래 @Mapping은 이름이 다른 필드만 명시적으로 선언한 것이다.

    // ================================================================
    // 패턴 2. 필드명이 다른 경우 — @Mapping으로 방향을 명시
    // ================================================================

    /**
     * PostImage(Entity) → PostImageResponse(DTO) 단건 변환.
     *
     * 수정 포인트:
     *   - source: 엔티티의 실제 필드명
     *   - target: DTO의 실제 필드명
     *   - 필드명이 같으면 해당 @Mapping 줄 제거
     *   - 매핑이 필요 없는 필드(무시)는 @Mapping(target = "fieldName", ignore = true) 사용
     */
    @Mapping(source = "imgOrder", target = "imageOrder") // Entity 필드명 → DTO 필드명
    PostImageResponse toImageResponse(PostImage postImage);

    // ================================================================
    // 패턴 3. List 변환 — 단건 메서드를 선언하면 MapStruct가 자동으로 List 변환 생성
    // ================================================================

    /**
     * List<PostImage> → List<PostImageResponse> 변환.
     *
     * 별도 구현 없이 선언만 해도 된다.
     * MapStruct가 위에서 선언한 toImageResponse()를 내부적으로 for문 돌려서 자동 생성한다.
     * Stream().map().toList() 노가다가 완전히 사라진다.
     *
     * 수정 포인트: 단건 변환 메서드(toImageResponse)가 인터페이스에 선언되어 있어야 한다.
     */
    List<PostImageResponse> toImageResponses(List<PostImage> postImages);

    // ================================================================
    // 패턴 4. 커스텀 로직 — default 메서드로 직접 구현
    // ================================================================

    /**
     * Post + List<PostImage> → PostResponse 변환.
     *
     * MapStruct가 자동으로 처리하기 어려운 경우
     * (여러 파라미터 조합, 기본값 세팅, 중첩 변환 호출 등)에 사용한다.
     *
     * 수정 포인트:
     *   - PostResponse 생성자 파라미터를 실제 DTO 구조에 맞게 수정한다.
     *   - 기본값 세팅 로직(LikeStatusResponse.empty(), 임시 댓글 수 0 등)을 실제 요구사항에 맞게 교체한다.
     *   - 내부에서 위에 선언한 toImageResponses()처럼 다른 MapStruct 메서드를 자유롭게 호출할 수 있다.
     */
    default PostResponse toResponse(Post post, List<PostImage> images) {
        if (post == null) {
            return null;
        }
        return new PostResponse(
                post.getId(),
                post.getContent(),
                post.getWriter().getUsername(),         // 수정 포인트: 연관 엔티티 접근 시 N+1 주의
                post.getWriter().getProfileImageUrl(),
                toImageResponses(images),               // 위에서 선언한 List 변환 메서드 호출
                post.getCreatedAt(),
                LikeStatusResponse.empty(),             // 수정 포인트: 복잡한 기본값 처리 로직
                0                                       // 수정 포인트: 임시값 — 실제 조회 로직으로 교체
        );
    }
}

    3. 서비스에서 사용하기

    @Service
@RequiredArgsConstructor
@Transactional(readOnly = true)
public class PostService {

    private final PostRepository postRepository;
    private final PostImageRepository postImageRepository;

    // 수정 포인트: MapStruct가 컴파일 타임에 생성한 PostMapperImpl이 Bean으로 등록된다.
    // 인터페이스 타입(PostMapper)으로 주입받으면 된다.
    private final PostMapper postMapper;

    public PostResponse getPost(Long postId) {
        Post post = postRepository.findById(postId).orElseThrow();
        List<PostImage> images = postImageRepository.findByPost(post);

        // Stream().map(PostResponse::from).toList() 대신 한 줄로 처리
        return postMapper.toResponse(post, images);
    }

    public List<PostImageResponse> getImages(List<PostImage> postImages) {
        // List 변환도 한 줄 — for문이나 Stream 불필요
        return postMapper.toImageResponses(postImages);
    }
}

    4. 자주 쓰는 @Mapping 옵션 정리

    옵션 설명 및 사용 예시
    @Mapping(source = "A", target = "B") 필드명이 다를 때 매핑 방향을 명시한다.
    source: 엔티티의 필드명 / target: DTO의 필드명
    @Mapping(target = "field", ignore = true) 특정 필드를 변환에서 제외한다. 보안상 노출하면 안 되는 필드나 직접 세팅할 필드에 사용한다.
    @Mapping(target = "field", constant = "ACTIVE") 항상 고정값을 할당한다. enum 기본값이나 상수 세팅에 유용하다.
    @Mapping(target = "field", expression = "java(…)") Java 표현식을 직접 작성한다. 간단한 로직을 인라인으로 처리할 때 사용한다.
    예: expression = "java(entity.getPrice() * 1.1)"
    @Mapping(source = "a.b", target = "c") 중첩 필드에 접근한다. post.writer.username처럼 깊은 경로를 직접 지정할 수 있다.
    예: source = "writer.username", target = "authorName"
    @BeanMapping(nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.IGNORE) 부분 업데이트(PATCH)에 사용한다. 소스가 null인 필드는 타겟에 덮어쓰지 않는다.

    중첩 필드 + ignore 활용 예시 

    @Mapper(componentModel = "spring")
public interface UserMapper {

    // 수정 포인트: 중첩 필드(writer.username)를 단일 필드(authorName)로 매핑
    @Mapping(source = "writer.username",        target = "authorName")
    @Mapping(source = "writer.profileImageUrl", target = "profileUrl")
    @Mapping(target  = "password",              ignore = true)  // 보안 필드 제외
    @Mapping(target  = "status",                constant = "ACTIVE") // 고정값 설정
    UserResponse toResponse(User user);

    // 부분 업데이트(PATCH) — null인 필드는 덮어쓰지 않음
    @BeanMapping(nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.IGNORE)
    void updateFromRequest(UserUpdateRequest request, @MappingTarget User user);
}
    @MappingTarget — 기존 객체를 업데이트할 때 @MappingTarget을 붙인 파라미터는 새 객체를 생성하지 않고 기존 객체의 필드를 수정한다.
    PATCH 요청처럼 변경된 필드만 엔티티에 반영할 때 사용한다. @BeanMapping(nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.IGNORE)와 함께 쓰면 요청에서 null로 온 필드는 엔티티를 덮어쓰지 않는다.

    5. MapStruct vs 직접 변환 비교

    구분 직접 변환 (정적 팩토리 메서드) MapStruct
    코드량 변환 메서드를 DTO마다 직접 작성 필드명이 같으면 선언만 해도 자동 생성
    리스트 변환 .stream().map().toList() 반복 작성 메서드 선언 하나로 자동 처리
    필드 변경 시 변환 메서드를 수동으로 찾아 수정 컴파일 오류로 즉시 감지
    커스텀 로직 메서드 안에 자유롭게 작성 default 메서드로 직접 구현
    런타임 성능 빠름 빠름 (리플렉션 없음, 컴파일 타임 코드 생성)
    적합한 상황 변환 로직이 복잡하고 커스텀이 많은 경우 필드 수가 많고 변환 패턴이 단순 반복인 경우
    ⚠️ MapStruct 사용 시 주의사항 N+1 주의: default 메서드 안에서 post.getWriter().getUsername()처럼 연관 엔티티에 접근하면 Lazy Loading이 발생한다. 매퍼를 호출하기 전에 Fetch Join으로 연관 데이터를 미리 로딩해야 한다.

    컴파일 후 확인: MapStruct 관련 오류는 컴파일 타임에 발생한다. IDE에서 Build Project를 실행해야 오류가 표시된다. build/generated/sources/annotationProcessor에서 생성된 코드를 직접 확인할 수 있다.

    Lombok 순서: annotationProcessorPaths에서 Lombok이 MapStruct보다 반드시 먼저 와야 한다.
