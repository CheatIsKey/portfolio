    코드 템플릿 — 로컬 파일 업로드 (MultipartFile)

    분류: 템플릿, Spring Boot, MultipartFile, 파일 업로드, WebMvcConfigurer

    개요

    MultipartFile로 전달된 이미지 파일을 서버 로컬 디스크에 저장하고, 정적 리소스 경로를 통해 클라이언트가 URL로 접근할 수 있도록 제공하는 템플릿이다. 단일 파일 업로드, 다중 파일 업로드, JSON 데이터와 파일을 함께 받는 @RequestPart 패턴을 모두 포함한다.

    전체 처리 흐름 1. 클라이언트가 multipart/form-data로 파일 전송
    2. 컨트롤러가 MultipartFile을 받아 FileStore에 위임
    3. FileStore: 확장자·MIME 타입 검증 → UUID 파일명 생성 → 로컬 디스크에 저장
    4. WebMvcConfig: /img/** 요청을 로컬 파일 경로로 연결
    5. 클라이언트가 반환된 /img/{uuid}.jpg URL로 이미지를 조회

    파일 구성

    파일명 종류 역할
    application.yml 설정 파일 로컬 저장 경로 및 멀티파트 파일 크기 제한을 설정한다.
    FileStore.java @Component 파일 검증, UUID 파일명 생성, 로컬 디스크 저장을 담당한다.
    WebMvcConfig.java @Configuration /img/** URL 요청을 로컬 파일 경로로 연결하는 정적 리소스 핸들러를 등록한다.
    PostController.java @RestController 단일·다중 파일 업로드, @RequestPart 복합 요청 처리 예시를 담는다.

    1. application.yml

    파일 저장 경로와 멀티파트 크기 제한을 설정한다. ${user.home}은 JVM이 실행 중인 운영체제의 홈 디렉토리로 자동 치환된다.

    # 수정 포인트: 저장 경로 변경 시 아래 location 값을 수정한다.
file:
  upload:
    location: ${user.home}/instagram-clone/uploads/

spring:
  servlet:
    multipart:
      max-file-size: 10MB      # 수정 포인트: 파일 1개당 최대 허용 용량
      max-request-size: 50MB   # 수정 포인트: 요청 1번당 (파일 여러 개 합산) 최대 허용 용량
    ⚠️ 경로 끝 슬래시 필수 file.upload.location 값의 끝에 반드시 슬래시(/)를 붙여야 한다. 없으면 fileDir + storeFileName을 연결할 때 경로 구분자가 누락되어 파일이 엉뚱한 경로에 저장된다.

    2. FileStore.java

    파일 저장 로직을 컨트롤러·서비스에서 분리한 컴포넌트다. @PostConstruct로 애플리케이션 시작 시 저장 디렉토리가 없으면 자동으로 생성하고, storeFile()이 검증 → 파일명 생성 → 저장을 순서대로 처리한다. 

    @Component
public class FileStore {

    // 수정 포인트: 허용할 이미지 확장자 목록을 프로젝트에 맞게 수정한다.
    private static final Set<String> ALLOWED_EXTENSIONS =
            Set.of(".jpg", ".jpeg", ".png", ".gif", ".webp");

    @Value("${file.upload.location}")
    private String fileDir;

    // 앱 시작 시 저장 디렉토리가 없으면 자동 생성
    @PostConstruct
    public void init() {
        File dir = new File(fileDir);
        if (!dir.exists()) {
            dir.mkdirs();
        }
    }

    public String storeFile(MultipartFile multipartFile) throws IOException {

        // 1. 원본 파일명 추출
        String originalFilename = multipartFile.getOriginalFilename();
        if (originalFilename == null || originalFilename.isEmpty()) {
            // 수정 포인트: 프로젝트 예외 클래스에 맞게 변경한다.
            throw new PostException(PostErrorCode.INVALID_FILE_EXTENSION);
        }

        // 2. MIME 타입 검증 — 이미지가 아니면 거부
        String contentType = multipartFile.getContentType();
        if (contentType == null || !contentType.startsWith("image/")) {
            throw new PostException(PostErrorCode.INVALID_FILE_EXTENSION);
        }

        // 3. 확장자 화이트리스트 검증
        String lowerOriginal = originalFilename.toLowerCase();
        if (ALLOWED_EXTENSIONS.stream().noneMatch(lowerOriginal::endsWith)) {
            throw new PostException(PostErrorCode.INVALID_FILE_EXTENSION);
        }

        // 4. UUID 기반 고유 파일명 생성 (원본 확장자 유지)
        int extIndex = originalFilename.lastIndexOf(".");
        String ext = (extIndex == -1) ? "" : originalFilename.substring(extIndex);
        String storeFileName = UUID.randomUUID() + ext;

        // 5. 로컬 디스크에 저장
        String fullPath = fileDir + storeFileName;
        multipartFile.transferTo(new File(fullPath));

        // 6. 클라이언트 접근용 정적 리소스 경로 반환
        // 수정 포인트: WebMvcConfig의 addResourceHandler 경로와 반드시 일치해야 한다.
        return "/img/" + storeFileName;
    }
}
    검증 단계 확인 내용 이유
    원본 파일명 null 체크 getOriginalFilename() null·빈 문자열 여부 파일명이 없으면 확장자를 추출할 수 없다.
    MIME 타입 검증 getContentType()이 image/로 시작하는지 확인 Content-Type 조작으로 실행 파일을 업로드하는 공격을 차단한다.
    확장자 화이트리스트 파일명이 허용된 확장자로 끝나는지 확인 MIME 타입과 이중으로 검증하여 보안을 강화한다.
    UUID 파일명을 사용하는 이유 원본 파일명을 그대로 저장하면 동명의 파일이 덮어써진다. 또한 원본 파일명에 ../ 같은 경로 탐색 문자가 포함되면 디렉토리 트래버설 공격이 가능하다. UUID.randomUUID()로 생성한 고유한 파일명은 충돌과 보안 문제를 동시에 해결한다.

    3. WebMvcConfig.java

    /img/** URL 패턴의 요청을 로컬 파일 시스템 경로로 연결하는 정적 리소스 핸들러를 등록한다. 이 설정이 없으면 FileStore가 저장한 파일에 URL로 접근할 수 없다. 

    @Configuration
public class WebMvcConfig implements WebMvcConfigurer {

    @Value("${file.upload.location}")
    private String fileDir;

    @Override
    public void addResourceHandlers(ResourceHandlerRegistry registry) {
        // 수정 포인트: addResourceHandler 경로를 바꾸면 FileStore.storeFile()의 반환 경로도 함께 수정한다.
        registry.addResourceHandler("/img/**")
                // "file:" 접두어 필수 — 이것이 없으면 클래스패스에서 찾으려 해 파일을 찾지 못한다.
                .addResourceLocations("file:" + fileDir);
    }
}
    ⚠️ "file:" 접두어 필수 addResourceLocations()에 file: 접두어를 반드시 붙여야 한다. 없으면 Spring이 로컬 파일 시스템이 아닌 클래스패스에서 리소스를 찾으려 하여 이미지를 찾지 못한다. 올바른 형식: "file:" + fileDir → 예: file:/Users/user/uploads/

    4. PostController.java — 업로드 패턴 3가지

    Pattern 1 — 단일 파일 업로드 (@RequestParam) 

    // 단일 파일 1개를 받는다. 키 이름("file")은 클라이언트와 협의한다.
@PostMapping("/test/upload/single")
public String testSingleUpload(@RequestParam("file") MultipartFile file) throws IOException {
    String savedPath = fileStore.storeFile(file);
    return "단일 파일 업로드 성공! 저장 경로: " + savedPath;
}

    Pattern 2 — 다중 파일 업로드 (@RequestParam + List) 

    // 동일한 키 이름("files")으로 여러 파일을 받는다.
@PostMapping("/test/upload/multi")
public String testMultiUpload(@RequestParam("files") List<MultipartFile> files) throws IOException {
    StringBuilder result = new StringBuilder("다중 파일 업로드 성공!\n");
    for (MultipartFile file : files) {
        result.append("- ").append(fileStore.storeFile(file)).append("\n");
    }
    return result.toString();
}

    Pattern 3 — JSON + 파일 복합 요청 (@RequestPart)

    JSON 데이터(feed)와 파일(images)을 하나의 요청에서 분리하여 각각 받는다. 프론트엔드에서 각 파트에 이름을 지정해서 전송하며, 컨트롤러에서는 @RequestPart의 value로 해당 파트를 가져온다. 

    // 수정 포인트: "feed", "images" 파트 이름은 클라이언트와 반드시 동일하게 맞춰야 한다.
@PostMapping
public ResponseEntity<ApiResponse<Void>> createPost(
        @RequestPart("feed") PostCreateRequest request,                                   // JSON 데이터
        @RequestPart(value = "images", required = false) List<MultipartFile> images,     // 파일 (선택)
        @SessionAttribute(name = AuthConstants.SESSION_KEY, required = false)
            SessionUser sessionUser) throws IOException {

    if (sessionUser == null) {
        throw new MemberException(MemberErrorCode.UNAUTHORIZED_ACCESS);
    }

    postService.create(request, images, sessionUser.id());

    return ResponseEntity.status(HttpStatus.CREATED)
            .body(ApiResponse.success(null));
}
    어노테이션 역할 사용 상황
    @RequestParam 단순 파일 파라미터 바인딩 파일만 전송하는 단순한 업로드
    @RequestPart 멀티파트 요청의 특정 파트 바인딩 JSON 데이터 + 파일을 함께 전송하는 복합 요청

    Case 1 — 단일 파일 업로드 성공

    Case 2 — 다중 파일 업로드 성공

    POST /api/posts/test/upload/multi

    다중 파일 업로드 성공!

    6. 주요 수정 포인트 요약

    파일 수정 포인트
    application.yml file.upload.location의 경로를 프로젝트에 맞게 변경한다. max-file-size, max-request-size도 서비스 정책에 맞게 조정한다.
    FileStore.java ALLOWED_EXTENSIONS에 허용할 확장자를 추가·제거한다. 예외 클래스(PostException, PostErrorCode)를 프로젝트 예외 체계에 맞게 교체한다.
    WebMvcConfig.java URL 패턴(/img/**)을 변경하면 FileStore.storeFile()의 반환 경로도 함께 수정한다. 두 값은 항상 일치해야 한다.
    PostController.java @RequestPart의 파트 이름("feed", "images")을 클라이언트와 협의한 이름으로 변경한다. 인증 방식이 세션이 아닌 JWT라면 @SessionAttribute 대신 AuthUser ArgumentResolver로 교체한다.
    로컬 저장소 vs S3 선택 기준 로컬 저장소(이 템플릿): 단일 서버, 소규모 프로젝트, 빠른 프로토타이핑에 적합하다. 서버를 교체하거나 재시작하면 파일이 유실되고 여러 EC2 인스턴스가 파일을 공유하지 못한다.

    S3 (운영 환경 권장): 다중 서버, Auto Scaling 환경에서는 파일을 S3에 저장해야 어떤 인스턴스에서도 동일한 파일에 접근할 수 있다. Presigned URL로 다운로드를 처리하면 서버 대역폭 부담도 없어진다.
