    템플릿 — 무한 스크롤 페이징 (Slice · Cursor 기반)

    분류: 템플릿, Spring Boot, JPA, Slice, Cursor Paging, No-Offset, 무한 스크롤

    이 템플릿의 구성 두 가지 무한 스크롤 구현 방식을 제공한다.

    케이스 1 — Slice<T> 기반 무한 스크롤 (OFFSET 방식) + Batch Size로 N+1 해결
    케이스 2 — Cursor(No-Offset) 기반 페이징 — 대용량 데이터에서 성능 저하 없는 방식

    케이스 1. Slice 기반 무한 스크롤 (OFFSET 방식)

    Slice<T>는 Page<T>와 달리 COUNT(*) 쿼리를 실행하지 않는다. 내부적으로 LIMIT + 1로 다음 데이터 존재 여부(hasNext)만 확인한다. "다음 페이지가 있는가"만 알면 되는 무한 스크롤에 적합하다.

    파일 구성

    파일 역할 수정 포인트
    FeedResponse.java 무한 스크롤 공통 응답 DTO 제네릭 그대로 재사용 가능
    PostController.java 페이지/사이즈 파라미터 수신 인증 방식, 정렬 기준 컬럼, 기본값
    PostService.java Slice 조회 + DTO 변환 Repository 메서드명, DTO 변환 로직
    PostRepository.java JPQL Fetch Join + Slice 반환 JOIN FETCH 대상, 엔티티 클래스명
    application.yml Batch Size 설정 크기 값 (데이터 규모에 따라 조정)

    FeedResponse.java — 공통 응답 DTO 

    // 수정 포인트: 제네릭으로 설계되어 있어 어떤 DTO도 담을 수 있다. 그대로 재사용 가능
public record FeedResponse<T>(
        boolean hasNext,  // 다음 페이지 존재 여부 (클라이언트가 "더 보기" 버튼을 표시할지 결정)
        List<T> feedList  // 현재 페이지의 실제 데이터 목록
) {
    public static <T> FeedResponse<T> of(boolean hasNext, List<T> feedList) {
        return new FeedResponse<>(hasNext, feedList);
    }
}

    PostController.java

    수정 포인트 인증 방식(@SessionAttribute)을 프로젝트의 실제 인증 체계(JWT @LoginUser 등)로 교체한다.
    정렬 기준 컬럼("id")을 실제 정렬이 필요한 컬럼으로 변경한다.
    defaultValue의 page/size 값을 서비스 요구사항에 맞게 조정한다. 
    @RestController
@RequestMapping("/posts")
@RequiredArgsConstructor
public class PostController {

    private final PostService postService;

    @GetMapping
    public ResponseEntity<ApiResponse<FeedResponse<PostResponse>>> getFeed(
            @RequestParam(name = "page", defaultValue = "1") int page,   // 수정 포인트: 기본 페이지
            @RequestParam(name = "size", defaultValue = "5") int size,   // 수정 포인트: 페이지당 데이터 수
            // 수정 포인트: 인증 방식을 프로젝트에 맞게 교체 (JWT @LoginUser 등)
            @SessionAttribute(name = AuthConstants.SESSION_KEY, required = false) SessionUser sessionUser) {

        if (sessionUser == null) {
            throw new MemberException(MemberErrorCode.UNAUTHORIZED_ACCESS);
        }

        Pageable pageable = PageRequest.of(
                page - 1,  // 클라이언트는 1페이지부터 시작하므로 -1 보정
                size,
                Sort.by(Sort.Direction.DESC, "id")  // 수정 포인트: 최신 데이터 우선 정렬 기준
        );

        FeedResponse<PostResponse> response = postService.getFeed(pageable);
        return ResponseEntity.ok(ApiResponse.success(response));
    }
}

    PostService.java 

    @Service
@RequiredArgsConstructor
@Transactional(readOnly = true)
public class PostService {

    private final PostRepository postRepository;

    public FeedResponse<PostResponse> getFeed(Pageable pageable) {
        // 1. Slice로 조회 — Fetch Join으로 N+1 해결 & LIMIT 페이징 동작
        Slice<Post> postSlice = postRepository.findAllWithImages(pageable);

        // 2. .getContent()로 실제 데이터 리스트만 추출 후 DTO 변환
        // 수정 포인트: PostResponse::from을 실제 DTO 변환 메서드로 교체
        List<PostResponse> feedList = postSlice.getContent().stream()
                .map(PostResponse::from)
                .toList();

        // 3. hasNext()로 다음 페이지 여부를 안전하게 전달
        return FeedResponse.of(postSlice.hasNext(), feedList);
    }
}

    PostRepository.java

    수정 포인트 JOIN FETCH 대상(p.writer, p.images)을 실제 엔티티 연관관계로 교체한다.
    컬렉션(@OneToMany)에 Fetch Join을 적용하면 페이징이 메모리에서 처리되는 경고가 발생한다. 이 경우 케이스 1 하단의 Batch Size 방식으로 전환한다. 
    public interface PostRepository extends JpaRepository<Post, Long> {

    // 수정 포인트: @ManyToOne 연관관계는 Fetch Join + Slice가 함께 동작한다.
    // @OneToMany 컬렉션을 Fetch Join하면 페이징이 인메모리로 처리되는 문제 발생
    // → 컬렉션은 Fetch Join을 제거하고 Batch Size로 처리 (아래 application.yml 참고)
    @Query("""
        SELECT p FROM Post p
        JOIN FETCH p.writer
        JOIN FETCH p.images
        """)
    Slice<Post> findAllWithImages(Pageable pageable);
}

    Batch Size 설정 — 컬렉션 N+1 해결

    컬렉션(@OneToMany)에 Fetch Join을 적용하면 페이징이 메모리에서 처리되어 위험하다. 이 경우 Fetch Join을 포기하고 Hibernate의 default_batch_fetch_size를 설정한다. Hibernate가 자식 데이터를 IN (id1, id2, ...) 쿼리로 묶어서 가져오므로 N+1을 막을 수 있다. 

    spring:
  jpa:
    properties:
      hibernate:
        # 수정 포인트: 한 번에 묶어서 조회할 ID 개수. page size와 맞추는 것을 권장
        default_batch_fetch_size: 100
    Fetch Join vs Batch Size 선택 기준 @ManyToOne (단일 연관): Fetch Join 사용 가능. Slice 페이징과 함께 정상 동작한다.
    @OneToMany (컬렉션): Fetch Join 사용 시 페이징이 인메모리로 처리되어 전체 데이터를 메모리에 올린다. Fetch Join을 포기하고 Batch Size를 설정하면 Hibernate가 IN 쿼리로 자식을 묶어서 가져온다. 쿼리 수: 부모 1번 + 자식 종류당 1번(IN 쿼리)으로 최적화된다.

    케이스 2. Cursor 기반 페이징 (No-Offset)

    OFFSET 방식은 10페이지를 보려면 앞 90건을 읽고 버리는 구조다. 페이지 뒤로 갈수록 쿼리가 느려지고, 데이터가 삽입/삭제되면 중복이나 누락이 생긴다. Cursor 기반 페이징은 마지막으로 읽은 데이터의 ID(cursorId)를 기준으로 그 이후 데이터만 바로 조회하므로 데이터가 아무리 많아도 항상 빠르다.

    파일 구성

    파일 역할 수정 포인트
    CursorFeedResponse.java 커서 정보를 포함한 응답 DTO 제네릭 그대로 재사용 가능
    PostController.java cursorId 파라미터 수신 인증 방식, 기본 size 값
    PostService.java cursorId 분기 처리 + Slice 조회 첫 페이지 처리 방식, DTO 변환
    PostRepository.java No-Offset 쿼리 메서드 정렬 기준 컬럼, 조건 방향(Lt/Gt)

    CursorFeedResponse.java — 커서 포함 응답 DTO 

    // 수정 포인트: 클라이언트는 nextCursorId를 저장해두고 다음 요청 시 cursorId로 전달한다
public record CursorFeedResponse<T>(
        boolean hasNext,     // 다음 데이터 존재 여부
        Long nextCursorId,   // 다음 요청 시 사용할 커서 ID (마지막 데이터의 ID)
        List<T> feedList
) {
    public static <T> CursorFeedResponse<T> of(Slice<?> slice, Long nextCursorId, List<T> feedList) {
        return new CursorFeedResponse<>(slice.hasNext(), nextCursorId, feedList);
    }
}

    PostController.java — cursorId 파라미터 수신 

    @RestController
@RequestMapping("/posts")
@RequiredArgsConstructor
public class PostController {

    private final PostService postService;

    @GetMapping
    public ResponseEntity<ApiResponse<CursorFeedResponse<PostResponse>>> getFeed(
            // 수정 포인트: 첫 요청에는 cursorId가 없으므로 required = false
            @RequestParam(name = "cursorId", required = false) Long cursorId,
            @RequestParam(name = "size", defaultValue = "5") int size,
            // 수정 포인트: 인증 방식을 프로젝트에 맞게 교체
            @SessionAttribute(name = AuthConstants.SESSION_KEY, required = false) SessionUser sessionUser) {

        if (sessionUser == null) {
            throw new MemberException(MemberErrorCode.UNAUTHORIZED_ACCESS);
        }

        // Cursor 방식에서는 정렬이 쿼리 조건에 포함되므로 Sort 설정이 불필요한 경우도 있다
        Pageable pageable = PageRequest.of(0, size);

        CursorFeedResponse<PostResponse> response = postService.getFeedByCursor(cursorId, pageable);
        return ResponseEntity.ok(ApiResponse.success(response));
    }
}

    PostService.java — cursorId 분기 처리

    수정 포인트 첫 페이지(cursorId = null)와 이후 페이지(cursorId 존재)를 반드시 분기 처리해야 한다.
    Long.MAX_VALUE를 전달하는 방식 대신, Repository 메서드를 분리해서 처리하는 것이 더 명확하다. 
    @Service
@RequiredArgsConstructor
@Transactional(readOnly = true)
public class PostService {

    private final PostRepository postRepository;

    public CursorFeedResponse<PostResponse> getFeedByCursor(Long cursorId, Pageable pageable) {
        Slice<Post> postSlice;

        if (cursorId == null) {
            // 수정 포인트: 첫 페이지 — 커서 없이 가장 최신 데이터부터 조회
            postSlice = postRepository.findAllByOrderByIdDesc(pageable);
        } else {
            // 이후 페이지 — 마지막으로 읽은 ID보다 작은(이전) 데이터 조회
            postSlice = postRepository.findByIdLessThanOrderByIdDesc(cursorId, pageable);
        }

        List<PostResponse> feedList = postSlice.getContent().stream()
                .map(PostResponse::from)
                .toList();

        // 다음 커서 ID = 현재 응답 중 가장 마지막(가장 오래된) 데이터의 ID
        Long nextCursorId = feedList.isEmpty()
                ? null
                : postSlice.getContent().get(postSlice.getContent().size() - 1).getId();

        return CursorFeedResponse.of(postSlice, nextCursorId, feedList);
    }
}

    PostRepository.java — No-Offset 쿼리 메서드 

    public interface PostRepository extends JpaRepository<Post, Long> {

    // 첫 페이지 — 커서 없이 최신 순 조회
    // 수정 포인트: 정렬 기준 컬럼이 id가 아니라면 메서드명과 쿼리를 함께 수정
    Slice<Post> findAllByOrderByIdDesc(Pageable pageable);

    // 이후 페이지 — cursorId보다 작은 id를 가진 데이터만 조회 (OFFSET 없이 동작)
    // WHERE id < :cursorId ORDER BY id DESC LIMIT :size
    Slice<Post> findByIdLessThanOrderByIdDesc(Long cursorId, Pageable pageable);

    // 수정 포인트: Fetch Join이 필요하면 @Query로 명시적으로 작성한다
    @Query("""
        SELECT p FROM Post p
        JOIN FETCH p.writer
        WHERE p.id < :cursorId
        ORDER BY p.id DESC
        """)
    Slice<Post> findByIdLessThanWithWriter(@Param("cursorId") Long cursorId, Pageable pageable);
}
    Cursor 페이징의 클라이언트-서버 통신 흐름 첫 요청: GET /posts?size=5 → cursorId 없음 → 최신 5건 반환 + nextCursorId = 마지막 글 ID

    두 번째 요청: GET /posts?cursorId=98&size=5 → id < 98인 데이터 5건 반환 + nextCursorId = 93

    세 번째 요청: GET /posts?cursorId=93&size=5 → id < 93인 데이터 5건 반환

    hasNext = false가 되면 클라이언트가 "더 보기" 버튼을 숨긴다. OFFSET과 달리 앞 데이터를 읽고 버리지 않으므로 데이터가 아무리 많아도 쿼리 성능이 일정하다.

    두 방식 비교 요약

    구분 Slice (OFFSET 방식) Cursor (No-Offset 방식)
    쿼리 방식 LIMIT N OFFSET M WHERE id < :cursorId LIMIT N
    성능 페이지 뒤로 갈수록 느려짐 (앞 데이터를 읽고 버림) 항상 일정 (앞 데이터를 읽지 않음)
    데이터 중복/누락 조회 중 삽입/삭제 시 발생 가능 ID 기준이라 발생하지 않음
    구현 복잡도 낮음 (page/size 파라미터만) 높음 (cursorId 분기 처리 필요)
    특정 페이지 이동 가능 불가능 (순차 탐색만 가능)
    적합한 상황 데이터가 적거나 중간에 임의 페이지로 이동이 필요한 경우 수백만 건 이상 대용량, 무한 스크롤 UI
