    1. 테스트 코드

    테스트 코드의 이점

    번호 이점
    1 버그 조기 발견
    2 리팩토링 자신감 확보
    3 협업 효율 극대화
    테스트 코드 핵심 원칙 테스트 코드는 시간을 절약하고 신뢰를 높인다.
    테스트는 QA 팀의 일이 아니라, 개발자의 자기 방어막이다.
    테스트 코드는 독립적으로 실행되어야 한다.

    2. JUnit

    Java 진영의 대표적인 단위 테스트 프레임워크다. 테스트 코드를 작성하고 자동으로 실행/검증할 수 있도록 돕는 도구다.

    JUnit 어노테이션

    어노테이션 설명
    @Test 테스트 메서드 표시
    @BeforeEach 각 테스트 실행 전 초기화
    @AfterEach 각 테스트 실행 후 초기화
    @BeforeAll 전체 테스트 시작 전 1회 실행 (static 필요)
    @AfterAll 전체 테스트 끝난 후 1회 실행 (static 필요)
    @DisplayName 테스트 이름 커스텀 (출력 시 보기 좋게)
    @Disabled 특정 테스트 임시 비활성화
    실행 순서 @BeforeEach → @Test → @AfterEach

    단언문 (Assertions)

    테스트에서 기대하는 결과와 실제 결과를 비교하는 코드다.

    메서드 설명
    assertEquals(expected, actual) 값이 같은지 확인한다.
    assertTrue(condition) 조건이 참인지 확인한다.
    assertFalse(condition) 조건이 거짓인지 확인한다.
    assertThrows(Exception.class, () -> 실행코드) 특정 예외 발생을 검증한다.
    assertAll() 여러 검증을 묶어서 실행한다.
    assertNotNull(object) 객체가 null이 아닌지 확인한다.

    3. 단위 테스트와 Mock 객체

    단위 테스트란 단위(Unit): 프로그램의 가장 작은 실행 단위 (메서드, 클래스)
    단위 테스트(Unit Test): 이 한 단위가 의도대로 동작하는지를 독립적으로 검증하는 것

    단위 테스트의 핵심은 독립성이다. 단위 테스트는 빠르고 정확해야 한다. 하지만 DB, 외부 API, 파일 시스템 등에 의존하면 테스트 속도가 느려지고 실패 원인도 외부 요인과 섞여서 불명확해진다.
    → 이 문제를 해결하기 위해 Mock 객체가 등장했다.

    Mock 객체

    실제 객체처럼 동작하지만, 테스트를 위해 가짜로 만든 객체다. 단위 테스트에서는 외부 의존성을 제거하기 위해 Mock 객체를 사용한다.

    이점 설명
    속도 향상 DB, 네트워크 없이 빠르게 테스트 가능하다.
    독립성 확보 특정 클래스만 검증한다. (외부 영향 없음)
    예측 가능성 Mock의 동작을 직접 지정할 수 있다.
    안정성 향상 외부 장애나 환경 문제로 인한 테스트 실패를 방지한다.

    Mockito 주요 어노테이션

    어노테이션 / 메서드 설명
    @Mock 가짜 객체를 생성한다.
    @InjectMocks Mock을 주입받는 테스트 대상 클래스다.
    @ExtendWith(MockitoExtension.class) Mockito 환경 통합이다. 테스트 클래스에 달아준다.
    when(...).thenReturn(...) Mock 동작을 정의한다.
    verify(..., times(n)) 메서드 호출 횟수를 검증한다.

    Mockito 사용 패턴

    when().thenReturn() — 반환 값 고정

    when(postRepository.save(any(Post.class))).thenReturn(<반환 값>);

    Mock 객체를 통해 특정 메서드를 실행해야 할 때, 묻지도 따지지도 않고 반환 값을 그대로 돌려주라는 것이다. 여기서 any(~.class)는 특정 class 타입이 들어오면 무조건 반환 값을 그대로 돌려주라는 것이다.

    Optional.empty() — 실패 케이스 테스트

    when(postRepository.findById(any(Post.class))).thenReturn(Optional.empty());

    실패 케이스로 예외가 제대로 던져지는지 테스트하려면, Optional.empty()로 빈 값을 이용하면 된다.

    assertThatThrownBy() — 예외 검증

    assertThatThrownBy(() -> postService.getPostById(1))
    .isInstanceOf(예외 클래스)
    .hasMessageContaining(예외 메시지);

    예외가 터졌을 때 확인하기 위해 사용한다. 특정 예외 클래스로 어떤 메시지를 던졌는지까지 테스트가 가능하다.

    verify() — 메서드 호출 횟수 검증

    verify(postRepository, times(횟수)).메서드(any(Post.class));

    특정 클래스의 메서드가 몇 번 실행됐는지 검증할 수 있다.

    4. ReflectionTestUtils

    실제 코드는 건들지 않으면서, 객체의 값을 임의적으로 세팅해주는 테스트 유틸이다. 

    // 필드 값 주입 (@Value 주입 대체에 유용)
ReflectionTestUtils.setField(jwtUtil, "secretKey", SECRET_KEY);

// 필드 값 조회 (메인 코드에서 사용하던 객체를 꺼내 쓸 수 있다.)
JwtParser parser = (JwtParser) ReflectionTestUtils.getField(jwtUtil, "parser");
    ⚠️ NPE 주의 ReflectionTestUtils.getField()로 조회한 객체는 NPE 위험이 있다. 다음 라인에 assert parser != null;을 추가해서 NPE 케이스를 피해야 한다.

    ReflectionTestUtils 활용 예시 — JwtUtilTest 

    class JwtUtilTest {
    private JwtUtil jwtUtil;
    private static final String SECRET_KEY = "mysecretkeymysecretkeymysecretkeymysecretkey";

    @BeforeEach
    void setUp() {
        jwtUtil = new JwtUtil();

        // 테스트용 secret key (Base64 인코딩된 256bit 키)
        String testSecretKey = "c2VjdXJldGVzdGtleW15c2VjdXJldGVzdGtleW15c2VjdXJldGVzdGtleQ==";

        // Reflection을 이용해 @Value 주입 대체
        ReflectionTestUtils.setField(jwtUtil, "secretKeyString", testSecretKey);

        // @PostConstruct 수동 호출
        jwtUtil.init();
    }

    @Test
    @DisplayName("JWT 토큰 생성 시 username과 role 정보가 정상적으로 포함된다")
    void generateToken_정상생성_Claims확인() {
        // Given
        String username = "ravi";
        UserRoleEnum role = UserRoleEnum.ADMIN;

        // When
        String tokenWithPrefix = jwtUtil.generateToken(username, role);
        String pureToken = tokenWithPrefix.substring(JwtUtil.BEARER_PREFIX.length());

        // Then
        assertThat(tokenWithPrefix).startsWith("Bearer ");
        Claims claims = jwtUtil.getParser()
                .parseSignedClaims(pureToken)
                .getPayload();

        assertThat(claims.get("username", String.class)).isEqualTo(username);
        assertThat(claims.get("auth", String.class)).isEqualTo(role.name());
        assertThat(claims.getExpiration()).isAfter(new java.util.Date());
    }
}

    5. Fixture 클래스

    테스트 코드에서 사용할 가짜 객체는 별도의 Fixture 클래스를 도입하는 게 좋다. 여러 테스트 클래스에서 반복적으로 사용하는 객체 생성 코드를 한 곳으로 모아 관리할 수 있다. 

    public class UserFixture {
    public static final String DEFAULT_USERNAME = "user";
    public static final String DEFAULT_PASSWORD = "1234";

    public static User createUser() {
        return new User(DEFAULT_USERNAME, DEFAULT_PASSWORD);
    }
}

    6. 통합 테스트

    애플리케이션의 여러 구성 요소를 실제로 묶어 실행해 시스템 전체의 동작 흐름을 검증하는 테스트다. @SpringBootTest 어노테이션을 테스트 클래스에 붙여주고 여러 계층을 묶어서 테스트한다. Mock을 사용하지 않고 실제 DB를 사용하면서 전체 흐름을 검증한다.

    어노테이션 설명
    @SpringBootTest 스프링 애플리케이션 전체 컨텍스트를 로드한다.
    @Transactional 테스트 후 DB 변경사항을 자동 롤백한다.
    @AutoConfigureMockMvc MockMvc를 자동 설정한다. Postman 역할 / Controller 테스트용이다.

    통합 테스트 코드 예시 

    @SpringBootTest
@Transactional
@AutoConfigureMockMvc
class PostControllerTest {

    @Autowired
    private MockMvc mockMvc;  // Postman의 메서드, URL, HEADER, BODY 등 설정을 대신해준다.

    @Autowired
    private JwtUtil jwtUtil;

    @Autowired
    private UserRepository userRepository;

    private String token;

    @BeforeEach
    void setup() {
        // filter를 먼저 통과해야 한다.
        // jwt 토큰을 생성하기 위해 user를 생성하고 저장
        User user = UserFixture.createUser();
        userRepository.save(user);

        token = jwtUtil.generateToken(user.getUsername(), user.getRole());
    }

    @Test
    @DisplayName("POST /api/post - 게시글 생성 요청 성공")
    void createPost_통합테스트_success() {
        String requestBody = """
                {
                    "content" : "통합테스트"
                }
                """;

        mockMvc.perform(post("/api/post")
                .contentType(MediaType.APPLICATION_JSON)
                .header("Authorization", token)
                .content(requestBody))
            .andExpect(status().isOk())    // 응답코드가 200인지 검증
            .andExpect(jsonPath("$.content").value("통합테스트"))
            .andExpect(jsonPath("$.username").value(DEFAULT_USERNAME));
    }
}
    MockMvc란 MockMvc는 실제 서버를 띄우지 않고도 HTTP 요청을 시뮬레이션할 수 있는 도구다. Postman처럼 메서드, URL, Header, Body 등을 설정해 컨트롤러를 테스트할 수 있다. @AutoConfigureMockMvc를 선언하면 @Autowired로 바로 주입해서 사용할 수 있다.

    단위 테스트 vs 통합 테스트 비교

    항목 단위 테스트 통합 테스트
    범위 메서드, 클래스 단위 여러 계층(Controller~DB) 전체 흐름
    의존성 Mock으로 대체 실제 Bean, 실제 DB 사용
    속도 빠름 느림 (전체 컨텍스트 로드)
    핵심 어노테이션 @ExtendWith(MockitoExtension.class) @SpringBootTest
    DB 롤백 해당 없음 @Transactional로 자동 롤백

    마무리

    오늘은 테스트 코드의 이점과 핵심 원칙에서 시작해 JUnit, Mockito, ReflectionTestUtils, Fixture 패턴, 통합 테스트까지 테스트 코드 전반을 정리했다. 테스트는 QA 팀의 일이 아니라 개발자의 자기 방어막이라는 인식이 중요하며, 그 시작은 독립적으로 실행되는 단위 테스트를 작성하는 습관에서 온다.

    단위 테스트에서 DB, 외부 API에 의존하면 속도도 느려지고 실패 원인도 불명확해진다. 이 문제를 해결하는 것이 Mockito의 @Mock과 when().thenReturn() 패턴이다. assertThatThrownBy()로 예외 케이스까지 검증하고, verify()로 메서드 호출 횟수까지 확인하는 방식을 통해 테스트의 신뢰도를 높일 수 있다. ReflectionTestUtils는 @Value로 주입되는 필드처럼 테스트 환경에서 직접 세팅하기 어려운 값을 실제 코드를 건드리지 않고 주입할 수 있어 유용하다.

    통합 테스트는 단위 테스트가 검증하지 못하는 계층 간 연동과 전체 흐름을 검증한다. @SpringBootTest로 전체 컨텍스트를 로드하고, MockMvc로 실제 HTTP 요청처럼 컨트롤러를 테스트하며, @Transactional로 테스트가 끝나면 DB를 자동 롤백하는 구조를 갖춘다. Fixture 클래스로 테스트용 객체 생성 코드를 한 곳에서 관리하면 여러 테스트 클래스에서 재사용할 수 있어 유지보수성이 높아진다.
