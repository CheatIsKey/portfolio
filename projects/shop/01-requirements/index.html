요구사항 분석

유스케이스·명세서·시퀀스 다이어그램을 통해 기능적 요구를 정리합니다. 클래스/시스템 다이어그램은 설계 챕터에 포함됩니다.

프로젝트 범위

  • 고객: 상품 검색/조회, 장바구니, 주문 생성·취소, 주문 내역 조회
  • 관리자: 상품/카테고리 등록·관리, 사용자/문의 관리
  • 품질: N+1 방지, 권한 분리, 안정적 페이징, 로그 상관관계 확보

유스케이스 다이어그램

고객·관리자 액터와 로그인 관계를 포함한 상위 흐름을 한눈에 보여줍니다.

유스케이스 명세서 (22)

인증/회원

로그인·로그아웃, 회원정보수정, 마이페이지, 관리자 사용자 관리 등 인증과 계정 관련 기능.

카탈로그/검색/상품

카테고리·키워드 검색, 상품 목록과 상세 조회.

장바구니

수량 0 → 삭제, 재고 초과 → 자동 보정 + 안내 메시지. “담긴 수량 + 추가 수량 ≤ 재고” 보장을 서버에서 강제.

주문

주문 생성 후 리다이렉트, 목록/상세 조회, 조건 충족 시 취소. 배송 완료된 주문은 취소 불가.

문의/고객센터

문의 작성·조회·답변·종결. 관리자 전용 조회도 포함.

상품/카테고리 관리

상품 등록, 카테고리 등록/삭제. 정적 팩토리와 연관관계 편의 메서드로 일관성 유지.

시퀀스 다이어그램 (7)

인증/회원

로그인 시 세션 발급, 사용자 권한 검증 및 내비게이션 분기.

카탈로그/장바구니

상품 탐색→장바구니 담기/수량 검증/총액 반영.

주문

장바구니 → 주문 생성 → 상세 확인(리다이렉트).

상품/관리자

관리자 전용 상품 등록, 사용자 관리 플로우.

고객센터

문의 작성→답변→종결.

요구사항 총정리

기능 요구

  • 고객: 상품 탐색·검색, 장바구니, 주문 생성/취소/조회, 마이페이지, 문의 등록
  • 관리자: 상품/카테고리 관리, 회원 조회, 문의 관리

비즈니스 규칙

  • 장바구니: 수량 0 → 삭제, 재고 초과 → 보정 + 안내 메시지
  • 주문: 배송 완료(COMP) 상태면 취소 불가

품질/운영 요구

  • N+1 방지(fetch join/EntityGraph), 권한 분리, 안정적 페이징
  • POST 후 Redirect, Lazy 컬렉션은 서비스에서 DTO로 완비
  • 로그 상관관계, P6Spy SQL 로깅

남은 과제

  • 주문 상세 API 개선, 금액 Formatter 적용
  • Address 정적 팩토리 도입 및 파라미터 순서 통일
