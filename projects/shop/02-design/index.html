섹션 목차

    설계

    클래스 다이어그램(도메인 모델)과 시스템 다이어그램(런타임·스토리지·배포)을 정리합니다. 요구/명세/시퀀스는 01장에서, 테스트/운영 이슈는 04~05장에서 다룹니다.

    클래스 다이어그램

    엔티티 개요(필드/연관/카디널리티)

    엔티티핵심 필드연관/카디널리티비고
    Userid, name, phone, userid, password, address: Address, role: UserRoleOrder(1..*), SupportTicket(1..*)역할: CLIENT | ADMIN
    Orderid, orderDate: Date, status: OrderStatusUser(1), OrderItem(*), Delivery(1)상태: ORDER | CANCEL
    OrderItemid, orderPrice: int, count: intOrder(1), Item(1)합계 계산은 도메인 메서드
    Deliveryid, address: Address, status: DeliveryStatusOrder(1)상태: READY | COMP
    Itemid, name, price, stockQuantity, imageUrl, detail: ItemDetail(0..1)CategoryItem(*), ItemImage(*), OrderItem(*)이미지 다중, 상세 0..1
    ItemImageid, url, fileName, sortOrderItem(1)정렬 순서
    ItemDetailid, description: Text, spec: TextItem(1)상세/스펙 별도 테이블
    Categoryid, name, parent, childListCategory(자기참조 1..*), CategoryItem(*)계층 구조
    CategoryItemidCategory(1) ↔ Item(1)다대다 매핑 엔티티
    SupportTicketid, title, category: TicketCategory, content, isPrivate, status: TicketStatus, createdAtUser(1), SupportReply(*), Attachment(*)카테고리/상태 ENUM
    SupportReplyid, content, isStaffReply, createdAtSupportTicket(1), User(1)사용자/관리자 구분
    Attachmentid, fileName, uri, mimeType, sizeSupportTicket(1)첨부파일 메타
    Address (Value)city, state, street, zipOrder/Delivery에 임베드값 타입

    열거형(ENUM)

    UserRole

    CLIENT, ADMIN

    OrderStatus

    ORDER, CANCEL

    DeliveryStatus

    READY, COMP

    TicketCategory

    ORDER, PAYMENT, DELIVERY, ITEM, ACCOUNT, OTHER

    TicketStatus

    OPEN, ANSWERED, CLOSED

    연관 및 제약(다이어그램 기준)

    • Order–Delivery는 1:1, FK는 orders.delivery_id.
    • Category는 자기참조 트리(부모 1 : 자식 N).
    • Item–Category는 CategoryItem을 통한 다대다.
    • Item은 ItemImage 다중(1:N), ItemDetail 단일(0..1).
    • SupportTicket는 SupportReply, Attachment를 1:N으로 가짐.

    조회 전략(요약/상세 분리)

    • 주문 목록: 배송만 로딩(요약 화면).
    • 주문 상세: 배송 + 주문아이템 + 상품까지 로딩(N+1 방지).
    • 권장: 파생 쿼리로 표현 불가한 로딩은 @EntityGraph 또는 JPQL fetch join.

    시스템 다이어그램

    Cloud Run(앱) ↔ Cloud SQL(DB) ↔ GCS(이미지) 중심의 구성. 빌드/마이그레이션은 Cloud Build + Flyway.

    아키텍처 결정(요지)

    ADR-01

    주문–배송 1:1: FK를 orders.delivery_id에 두고, 양방향 편의 메서드로 정합성 보장.

    ADR-02

    조회 최적화: 화면 별 fetch 구분(요약/상세), N+1 방지.

    ADR-03

    PRG 패턴: POST 후 Redirect(중복 제출/화이트라벨 방지).

    ADR-04

    권한/세션: changeSessionId() 도입, ADMIN/USER 경로 분리.

    ADR-05

    고객센터 서브도메인: Ticket/Reply/Attachment를 독립 집합으로 설계(ENUM 중심 상태 전이).
