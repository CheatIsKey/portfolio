섹션 목차

    설계

    도메인 모델(클래스 다이어그램)과 런타임/스토리지/배포 토폴로지(시스템 다이어그램)를 정리합니다. 요구/명세/시퀀스는 01장에서, 테스트/운영 이슈는 04~05장에서 다룹니다.

    클래스 다이어그램

    도메인 요약

    • Order ↔ Delivery (1:1): FK는 orders.delivery_id에 귀속(필요 시 NULL, ON DELETE SET NULL).
    • 연관 일관성: Order#setDelivery(), Delivery#assignOrder()로 양방향 동기화 보호.
    • OrderItem: 가격×수량 합산을 도메인 메서드로 보유(getTotalPrice()), Order#totalPrice()에서 총합.
    • Category–Item: 중간 매핑 엔티티로 다대다 관리, 정적 팩토리/편의 메서드로 일관성 유지.
    • 엔티티 설계: setter 최소화, 정적 팩토리/생성자 보호, 지연 로딩 기본.

    조회 전략(요약/상세 분리)

    • 주문 목록(마이페이지): 배송만 fetch(요약 화면).
    • 주문 상세: 배송 + 주문아이템 + 상품까지 fetch로 N+1 방지.
    • 파생 쿼리 한계: withItems류는 파싱 불가 → @EntityGraph 또는 JPQL fetch join 사용.

    시스템 다이어그램

    Cloud Run(앱) ↔ Cloud SQL(DB) ↔ GCS(이미지) 중심의 단순/안정 구성을 채택합니다. 빌드/마이그레이션은 Cloud Build + Flyway.

    운영 구성(핵심)

    • 프로필: SPRING_PROFILES_ACTIVE=cloud, SPRING_JPA_OPEN_IN_VIEW=false.
    • DB: Cloud SQL(PostgreSQL). Flyway로 이관/검증(필요 시 일시 완화 플래그로 부팅 안정화 후 제거).
    • 스토리지: GCS에 이미지 업로드(버킷/권한/URL 저장).
    • 빌드·배포: Cloud Build에서 컨테이너 이미지 태그 → Cloud Run 배포, 환경변수로 커넥션/버킷/업로드 경로 주입.
    • 관측성: P6Spy SQL 로깅 + 요청 상관관계 로깅.

    아키텍처 결정(요지)

    • ADR-01 · 주문–배송 매핑: 1:1을 orders.delivery_id로 귀속. 도메인 메서드로 양방향 일관성 보장.
    • ADR-02 · 조회 최적화: 화면 별 fetch 구분(요약=배송만, 상세=배송+아이템+상품), N+1 방지 원칙.
    • ADR-03 · PRG 패턴: POST 성공 후 Redirect를 강제(화이트라벨/중복 제출 방지).
    • ADR-04 · 보안: 로그인 성공 시 changeSessionId() 적용, ADMIN/USER 경로 분리 및 인터셉터.
    • ADR-05 · 마이그레이션: Flyway로 스키마 이관, 과거 FK 구조 정리 및 히스토리 관리.

    데이터 접근 설계

    • 리포지토리: 요약/상세 전용 메서드 분리. 파생 쿼리로 표현 불가한 연관 로딩은 @EntityGraph 또는 JPQL.
    • 서비스: 뷰 모델(DTO)로 변환 후 전달. 뷰에서 지연 컬렉션 직접 순회 금지.

    릴리스 & 인프라

    • 설정: DB 접속(유저/패스/DB명), Cloud SQL 연결명, 버킷명, 업로드 디렉토리 등 환경변수 관리.
